<div class="site-inner"><div class="content-sidebar-wrap"><main class="content" id="genesis-content"><div class="breadcrumb full_width_center_breadcrumb"><span class="breadcrumb-link-wrap" itemprop="itemListElement" itemscope itemtype="https://schema.org/ListItem"><a href="https://www.aon.com/cyber-solutions/" itemprop="item"><span itemprop="name">Home</span></a></span> &#8594; <span class="breadcrumb-link-wrap" itemprop="itemListElement" itemscope itemtype="https://schema.org/ListItem"><a href="https://www.aon.com/cyber-solutions/aon_cyber_labs/" itemprop="item"><span itemprop="name">Aonâ€™s Cyber Labs</span></a></span> &#8594; Cobalt Strike Configuration Extractor and Parser</div><script src='https://www.google.com/recaptcha/api.js'></script>
  <div class="full_width_center content_only"> 

    <p>With each passing month and each new ransomware case, the consultants at <a href="https://www.aon.com/cyber-solutions/solutions/breach-response/" target="_blank" rel="noopener">Stroz Friedberg Incident Response</a> are finding <a href="https://www.cobaltstrike.com/help-beacon" target="_blank" rel="noopener">Cobalt Strike Beacons</a> to be the norm for persistence, lateral movement, and exfiltration rather than the exception.Â  Finding command and control (&#8220;C2&#8221;) servers and shutting off those connections is imperative before starting the rebuilding phase to help ensure threat actors can&#8217;t regain entry into the network.Â  Due to the large volume of Beacons in our casework, and given the opportunity to dive deep into this advanced post-exploitation framework, we set out to write a high-quality library and associated set of command line tools for extracting and parsing configuration data from Cobalt Strike Beacons to aid in accelerating response times.Â  Today we are open sourcing <a href="https://github.com/strozfriedberg/cobaltstrike-config-extractor" target="_blank" rel="noopener">the library and scripts</a>, as well as extensive <a href="https://strozfriedberg.github.io/cobaltstrike-config-extractor/" target="_blank" rel="noopener">Beacon and library documentation</a>, for the broader DFIR community to use and collaborate on with us!</p>
<h2>Getting Started</h2>
<p>The <a href="https://github.com/strozfriedberg/cobaltstrike-config-extractor" target="_blank" rel="noopener">README</a> contains detailed instructions for using pip to install the package from <a href="https://pypi.org/project/libcsce/">PyPI</a>.Â  There are two command line scripts that come bundled with the package:</p>
<ol>
<li>csce: Parse Beacon configuration data to JSON</li>
<li>list-cs-settings: Attempt to discover, extract, and parse all possible Beacon configuration data by brute force</li>
</ol>
<p>&nbsp;</p>
<p>The first script, csce (Cobalt Strike Configuration Extractor), is intended for daily use to extract and parse Beacon configuration data and is the one most will likely be interested in.Â  list-cs-settings is designed for those who want to conduct research on Beacon configurations by attempting to detect setting types by brute force.Â  Both the library and (thus) the scripts currently support parsing Beacon PE files (EXE/DLL) and memory dumps from systems running a Beacon.Â  As an example, you can parse a Beacon DLL sample using csce like this:</p>
<p><code class="EnlighterJSRAW" data-enlighter-language="generic">&gt; csce --pretty path/to/beacon.{exe,dll,bin,dmp}</code></p>
<p>This will pretty-print Beacon configuration data as JSON (assuming the input file is a Beacon) in a structure that closely mimics the <a href="https://cobaltstrike.com/help-malleable-c2" target="_blank" rel="noopener">Malleable C2 Profile</a> of the Team Server the Beacon was generated from.Â  The output shown below is from an HTTP Beacon we recovered during one of our cases (with all sensitive IOCs redacted/replaced):</p>

<p>Beacons do not contain full Malleable C2 Profiles for space and OPSEC reasons, but as is shown above critical sections can be reconstructed from Beacon configuration data including C2 IP addresses and domains, beaconing behavior and payload settings, and more.Â  These data points can be very important during the containment phase of an investigation.Â  For Beacon-specific settings that aren&#8217;t specified in the Malleable C2 Profile, we did our best to group items together where it makes sense (i.e., the ssh section).</p>
<h2>Why Use It?</h2>
<p>There are few primary reasons why we recommend the use of this library and these scripts over other open source options:</p>
<ol>
<li>Fairly well tested against generated samples and samples discovered in the wild.</li>
<li>Relatively fast and will likely get faster over time.</li>
<li>To our knowledge, the only library/scripts that attempt to reconstruct the Malleable C2 Profile from Beacon configuration data.</li>
</ol>
<p>&nbsp;</p>
<h2>Where are the Tests?</h2>
<p>You might be thinking &#8220;they said it&#8217;s fairly well tested, but I don&#8217;t see any test samples in the GitHub repository. What Gives?&#8221;Â  Many of the samples we use for testing were gathered from client environments, and thus we cannot publish them to GitHub.Â  Moreover, we think it&#8217;s generally a bad idea to publish malware to a repository others may download to computers not configured for malware sandboxing.Â  However, we will likely add unit tests for the core parser logic in the future, and we currently test all the examples present in the documentation for correctness (shoutout to <a href="https://docs.pytest.org/en/6.2.x/doctest.html" target="_blank" rel="noopener">doctest</a>).</p>
<h2>Our Hope and Vision</h2>
<p>These tools have helped speed up our own investigations, and our hope is that by sharing them with the community they will do the same for others.Â  Our long-term vision is to work collectively with DFIR practitioners and researchers to help make these tools the best available for parsing Cobalt Strike Beacons, and raise the bar for threat actors going forward.</p>
