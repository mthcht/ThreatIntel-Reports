<h3 class='post-title entry-title' itemprop='name'>
Cyber Heist Attribution
</h3>
<div class='post-header'>
<div class='post-header-line-1'>
</div>
</div>
<div class='post-body entry-content' id='post-body-5119779489792068637' itemprop='articleBody'>
<em>Written by Sergei Shevchenko and Adrian Nish</em><br />
<h5>
BACKGROUND</h5>
<br />
Attributing a single cyber-attack is a hard task and often impossible. However, when multiple attacks are conducted over long periods of time, they leave a trail of digital evidence. Piecing this together into a campaign can help investigators to see the bigger picture, and even hint at who may be behind the attacks.<br />
<a name='more'></a><br />
Our <a href="http://baesystemsai.blogspot.com/2016/04/two-bytes-to-951m.html" target="_blank">research</a> into malware used on SWIFT based systems running in banks has turned up multiple bespoke tools used by a set of attackers.  What initially looked to be an isolated incident at one Asian bank turned out to be part of a wider campaign. This led to the identification of a commercial bank in Vietnam that also appears to have been targeted in a similar fashion using tailored malware, but based off a common code-base.<br />
<br />
In the bank malware cases we know of, the coders used a unique file wipe-out function. This implementation was so distinctive that it further drew our attention &#8211; and so we began to look for other instances of code which had used the same function. Using disassembled machine opcodes (with masked out dynamic virtual addresses) we generated signatures to scan a large malware corpus.<br />
<br />
Our initial search turned up an additional sample which implemented the same wipe-out function.<br />
<br />
This sample was uploaded from a user in the US on 4th March 2016:<br />

<h5>
ANALYSIS</h5>
<h5>
The msoutc.exe functionality</h5>
<br />
<code>msoutc.exe</code> accepts a number of parameters passed with the command line. When executed, it checks if there is another instance of itself already running on a system, by attempting to create a mutex called:<br />
<br />
<div style="background: #f2f2f2; border: solid #a6a6a6 1.0pt; padding: 2pt 5pt 2pt 5pt;">
<code>
Global\FwtSqmSession106839323_S-1-5-20
</code></div>
<br />
If the mutex creation attempt fails, returning the 'already exists' error code, the bot will quit and delete itself by executing a self-delete batch file. The bot then copies itself into a temporary directory as:<br />
<br />
<div style="background: #f2f2f2; border: solid #a6a6a6 1.0pt; padding: 2pt 5pt 2pt 5pt;">
<code>
%TEMP%\sysman\svchost.exe
</code></div>
<br />
Next, it recreates its configuration file <code>wmplog09c.sqm</code> in the same directory. As in case with the malware described in our <a href="http://baesystemsai.blogspot.com/2016/04/two-bytes-to-951m.html" target="_blank">blogpost</a>, it will also install itself as a system service.<br />
<br />
The service will be called <em>"Indman"</em>, <em>"Indexing Manager"</em>, and described as <em>"Provides fast indexing service"</em>.<br />
<br />
The malware keeps its logs within an encrypted file <code>wmplog15r.sqm</code> and/or <code>wmplog21t.sqm</code>, located in the same directory. The logged messages are encrypted with a key:<br />
<br />
<div style="background: #f2f2f2; border: solid #a6a6a6 1.0pt; padding: 2pt 5pt 2pt 5pt;">
<code>
y@s!11yid60u7f!07ou74n001
</code></div>
<br />
Back in 2015, PwC <a href="http://pwc.blogs.com/cyber_security_updates/2015/01/destructive-malware.html" target="_blank">reported</a> a malware with an identical encryption key as above and a very similar unique mutex name.  The <code>msoutc.exe</code> bot matches the description of other tools from the same toolkit, as described in the US CERT Alert TA14-353A:<br />

<br />
If the configuration file is missing, the bot initiates a default configuration and saves it to the <code>wmplog09c.sqm</code> file, located in the same directory. The default parameters specify 127.0.0.1 (local host) and port 443 as the default address for the C&amp;C server.<br />
<br />
In this case, the bot expects the C&amp;C server's IP address and port number to be passed as the command line parameters. These parameters are then used to update the configuration file.<br />
<br />
Since the configuration file is not available for analysis, the C&amp;C IP remains unknown, but the port number is likely to be 443, same as the default value.<br />
<br />
Every 5 minutes it initiates a communication to the remote C&amp;C. All communications are encrypted. The beacon signal it sends contains the local IP address. The data it receives from the C&amp;C is first decrypted, and then used to update its own executable and the <code>wmplog09c.sqm</code> configuration file. The updated executable may have additional functionality that is missing in the analysed sample.<br />
<br />
<h5>
The msoutc.exe 'wipe-out' and 'file-delete' functions</h5>
<br />
When the malware reaches out to the C&amp;C server, it may receive a special instruction to terminate itself. In that case, it uninstalls its service, deletes its configuration and log files, then quits and deletes itself by executing a self-delete batch file that has the following format:<br />
<br />
<div style="background-color: #3d4146; color: silver; font-family: monospace;">
.data:0040A400&nbsp;echoOffD1Del&nbsp;db&nbsp;'<span style="color: white;">@echo off</span>',<span style="color: white;">0Dh</span>,<span style="color: white;">0Ah</span><br />
.data:0040A400&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;db&nbsp;'<span style="color: white;">:D1</span>',<span style="color: white;">0Dh</span>,<span style="color: white;">0Ah</span><br />
.data:0040A400&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;db&nbsp;'<span style="color: white;">del /a %1</span>',<span style="color: white;">0Dh</span>,<span style="color: white;">0Ah</span><br />
.data:0040A400&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;db&nbsp;'<span style="color: white;">if exist %1 goto D1</span>',<span style="color: white;">0Dh</span>,<span style="color: white;">0Ah</span><br />
.data:0040A400&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;db&nbsp;'<span style="color: white;">del /a %0</span>',<span style="color: white;">0</span>
</div>
<br />
The batch file format is identical to the one previously <a href="https://www.operationblockbuster.com/wp-content/uploads/2016/02/Operation-Blockbuster-Report.pdf" target="_blank">reported</a> by Novetta.<br />
<br />
In order to delete its configuration and log files, the bot calls an internal function that wipes these files out so that their contents cannot be forensically restored.<br />
<br />
The implementation of this function is very unique - it involves complete filling of the file with the random data in order to occupy all associated disk sectors, before the file is deleted. The file-delete function itself is also unique &#8211; the file is first renamed into a temporary file with a random name, and that temporary file is also deleted.<br />
<br />
<a href="https://blogger.googleusercontent.com/img/b/R29vZ2xl/AVvXsEhlGoMjgMPilSHxCFNMYbqCkZ-JIh3gzBBR5UmoTBf9QDkVGX5vVA17xjyY1CwaFfV5POK8DvXuqhraZdPLdWxJSkOMCGgXOhTS1rm70EkdT0XqCmSkWtDLyw2yExcusySdllhUOyeHBoXr/s1600/scheme.png" imageanchor="1"><img border="0" src="https://blogger.googleusercontent.com/img/b/R29vZ2xl/AVvXsEhlGoMjgMPilSHxCFNMYbqCkZ-JIh3gzBBR5UmoTBf9QDkVGX5vVA17xjyY1CwaFfV5POK8DvXuqhraZdPLdWxJSkOMCGgXOhTS1rm70EkdT0XqCmSkWtDLyw2yExcusySdllhUOyeHBoXr/s640/scheme.png" /></a><br />
<br />

<br />
The first instruction is to assign <code>0x1020</code> to <code>EAX</code> - this instruction is represented with an opcode <code>B8 20 10 00 00</code>, and it stays the same across the compared samples.<br />
<br />
The second instruction is to call a function located at the address calculated as the address of the next instruction (<code>push ebx</code>) plus <code>0x4EA96</code>. Depending on a sample, the compiler may place <em>__alloca_probe()</em> function at a different offset within the executable, hence the <code>0x4EA96</code> offset is dynamic and the opcode bytes <code><span style="background: #3d4146; border: 1px #3d4146 solid; color: white; display: inline;">96 EA 04</span></code> (representing <code>0x4EA96</code>) should be ignored in our comparison.<br />
<br />
The API <em>GetTickCount()</em> is called with the last instruction, but again, the pointer into the API is stored in this particular sample at the virtual address of <code>0x45F04C</code>. In a modified sample with a different image base, the virtual address of this pointer will also be different. Hence, the bytes <code><span style="background: #3d4146; border: 1px #3d4146 solid; color: white; display: inline;">4C F0 45</span></code> (representing <code>0x45F04C</code>) should also be masked out (ignored).<br />
<br />
Using this principle, full comparison of this function's implementation can now be done.<br />
<br />

<h5>
ATTRIBUTION</h5>
<br />
So what else do we know about <code>msoutc.exe</code>? It turns out that this malware exhibits the same unique characteristics, such as mutex names and encryption keys, as other tools from a larger toolkit <a href="https://www.us-cert.gov/ncas/alerts/TA14-353A" target="_blank">described</a> in US-CERT Alert TA14-353A.<br />
<br />
The US-CERT alert mentions "a major entertainment company" and is widely believed to describe the toolkit used to conduct destructive cyber-attack which took place in late 2014. Further details of this same toolkit were disclosed in the 'Op Blockbuster' <a href="http://www.operationblockbuster.com/wp-content/uploads/2016/02/Operation-Blockbuster-Report.pdf" target="_blank">report</a> in February 2016.<br />
<br />
<code>msoutc.exe</code> matches the description of the 'Sierra Charlie' variants in their report.  From their analysis this is described as a spreader type of malware, presumably used to gain a foothold on multiple devices within a target environment before launching further actions.<br />
<br />

This was done in order to make sure the deleted file could not be forensically restored as all the original contents of the hard disk sectors associated with the file would be lost.<br />

<h5>
CONCLUSIONS</h5>
<br />
The overlaps between these samples provide strong links for the same coder being behind the recent bank heist cases and a wider known campaign stretching back almost a decade.<br />
<br />
It is possible that this particular file-delete function exists as shared code, distributed between multiple coders who look to achieve similar results. However, we have noted that this code isn't publically available or present in any other software after searching through tens of millions of files. The unique decision to move and rename the file before deletion after overwriting is unusual, and not a common step we would expect to see when implementing this capability.<br />
<br />
A motivated and talented coder could implement the same function and choose to compile their project in Visual Studio 6.0 to achieve the same binary result for the purposes of misattribution. We cannot prove for certain that this is not the case, and further investigation of command infrastructure and related tools is needed before any definitive statements on attribution can be made.<br />
<br />
Whilst there are possibilities that exist which may lead to alternative hypotheses, these are unlikely and as such, we believe that the same coder is central to these attacks.  Who the coder is, who they work for, and what their motivation is for conducting these attacks cannot be determined from the digital evidence alone. However, this adds a significant lead to the investigation.
