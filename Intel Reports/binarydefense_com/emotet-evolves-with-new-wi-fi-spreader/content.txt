            </h1>
                        <div class="hero__authors">

                    <div class="grid">
                        <div class="c-5:13 richtext">
                            <p>Emotet is a highly sophisticated trojan that typically also
serves as a loader for other malware. A key functionality of Emotet is its
ability to deliver custom modules or plugins that are suited for specific
tasks, including stealing Outlook contacts, or spreading over a LAN. Recently,
Binary Defense has identified a new loader type that takes advantage of the
wlanAPI interface to enumerate all Wi-Fi networks in the area, and then
attempts to spread to these networks, infecting all devices that it can access
in the process (as seen in Figure 1). </p>



<figure class="wp-block-image size-large"><img src="https://www.binarydefense.com/wp-content/uploads/2023/04/wifispreader-Picture1-1024x577.png" alt="class=" class="wp-image-24482" /></figure>



<p><em>Figure 1 Wi-Fi spreader overview</em></p>



<h4 class="wp-block-heading">Emotet’s Protocol</h4>



<p>Before getting into the analysis, Emotet’s Protocol should be explained. The protocol is based on <a href="https://developers.google.com/protocol-buffers">Google’s Protobufs</a> to serialize data sent to and from the server. The documentation for Protobufs shows that data is defined through the use of protocol buffer message types in a .proto file. While <a href="https://www.binarydefense.com/threat_watch/emotet-returns-for-a-new-year-of-spam/">Emotet</a> uses several of these “messages”, the one this analysis will be focusing on is the Deliverable message, which is used when the server sends a response containing data to be loaded/executed. The message is as follows:</p>



<p><em>message Deliverable {</em><br /><em>    required int32 ID = 1;</em><br /><em>     required int32 executeFlag = 2;</em><br /><em>     required bytes blob = 3;</em><br /><em>}</em></p>



<p>In the above protobuf message, ID is the module ID, blob is
the binary data, and executeFlag determines how the binary loaded. The
executeFlag field can be one of the following:</p>



<ul>
<li>1: Reserved for payloads and standalone executables, like Trickbot. Drops in C:ProgramData and executes.</li>



<li>2: Like Type 1, but duplicates user's token.</li>



<li>3: Loads the binary into memory. Typically used by modules, as they are mainly DLLs which can be easily loaded into memory.</li>
</ul>



<span class="hs-cta-wrapper" id="hs-cta-wrapper-be60c510-00dc-418e-93ea-7a9b20f1dcbb"><span class="hs-cta-node hs-cta-be60c510-00dc-418e-93ea-7a9b20f1dcbb" id="hs-cta-be60c510-00dc-418e-93ea-7a9b20f1dcbb"><a href="https://cta-redirect.hubspot.com/cta/redirect/3412420/be60c510-00dc-418e-93ea-7a9b20f1dcbb" target="_blank" rel="noreferrer noopener"><img class="hs-cta-img" id="hs-cta-img-be60c510-00dc-418e-93ea-7a9b20f1dcbb" style="border-width:0px;" height="200" width="600" src="https://www.binarydefense.com/wp-content/uploads/2023/04/be60c510-00dc-418e-93ea-7a9b20f1dcbb.png" alt="Sign Up for Threat Watch Daily Cybersecurity News" /></a></span></span>



<h4 class="wp-block-heading">Initial Binary</h4>



<p>The initial binary for the Wi-Fi spreader arrives on a system with module ID: 5079 and executeFlag: 1, which means it will be dropped into <strong>C:ProgramData</strong> before executing, like loader updates or other malware, like <a href="https://www.binarydefense.com/trickbot-ono-new-tricks/">TrickBot</a>. This initial binary is a self-extracting RAR file which contains the 2 binaries used for Wi-Fi spreading. </p>



<figure class="wp-block-image size-large"><img src="https://www.binarydefense.com/wp-content/uploads/2023/04/wifispreader_Picture2-1024x544.png" alt="class=" class="wp-image-24483" /></figure>



<p><em>Figure 2 RAR Extraction</em></p>



<p>The self-extracting RAR contains 2 files: service.exe and
worm.exe. Worm.exe is configured as the setup file, as seen in Figure 2, for
the self-extracting RAR, which means worm.exe will execute automatically once
the RAR file has unpacked itself.</p>



<h4 class="wp-block-heading">Worm.exe</h4>



<p>Worm.exe is the main executable used for spreading. This
executable has a timestamp of 04/16/2018 and was first submitted to <a href="https://www.virustotal.com/gui/home/upload">VirusTotal</a>
on 05/04/2018. The executable with this timestamp contained a hard-coded IP
address of a <a href="https://whatis.techtarget.com/definition/command-and-control-server-CC-server">Command and Control (C2) server</a> that was used by
Emotet. This hints that this Wi-Fi spreading behavior has been running unnoticed
for close to two years. This may be in part due to how infrequently the binary
is dropped. Based on our records, 01/23/2020 was the first time that Binary
Defense observed this file being delivered by Emotet, despite having data going
back to when Emotet first came back in late August of 2019. </p>



<h4 class="wp-block-heading">Operation of Worm.exe</h4>



<p>Upon startup of Worm.exe, the first action it takes is to
copy the service.exe string to a variable that will be used during file
spreading. Next, it steps into the main loop and immediately begins profiling
the wireless network using wlanAPI.dll calls in order to spread to any networks
it can access, as seen in Figure 3. </p>



<figure class="wp-block-image size-large"><img src="https://www.binarydefense.com/wp-content/uploads/2023/04/wifispreader-Picture3-1024x944.png" alt="A screenshot of a computer  Description automatically generated" class="wp-image-24484" /></figure>



<p><em>Figure 3 Worm.exe Start</em></p>



<p>The use of purely wlanAPI.dll calls for network profiling
makes sense; it is one of the libraries used by Native Wi-Fi to manage wireless
network profiles and wireless network connections. A possibly unintended
consequence of this is that researchers running this malware in VMs/automated
sandboxes will not see any of worm.exe’s spreading behavior if a Wi-Fi card is
not present. This can be bypassed partially by starting the WLAN auto config
service in Windows 10; however, this only allows opening a handle to interface
with. The subsequent calls will crash the program at that point, so the bypass is
not recommended.</p>



<h4 class="wp-block-heading">Worm Information Gathering</h4>



<p>As seen in Figure 3, once a handle has been obtained,
WlanEnumInterfaces is called. This function enumerates all Wi-Fi devices
currently enabled on the local computer, which it returns in a series of
structures. These structures contain all the information relating to the Wi-Fi
device, including the device’s GUID and description.</p>



<p>Using the first available Wi-Fi device,
WlanGetAvailableNetworkList is called to obtain a list of all available
networks. Interestingly, this has the dwFlag parameter set to
WLAN_AVAILABLE_NETWORK_INCLUDE_ALL_ADHOC_PROFILES, which is an invalid
parameter on Windows XP with SP3 and Wireless LAN API for Windows XP with SP2,
meaning this malware cannot properly run on Windows XP.</p>



<p>Next, the malware begins profiling the network, saving the following information to a buffer:</p>



<p> <em>SSID:               %s</em><br /><em>SIGNAL:             %d</em><br /><em>SECURITY:           [WPA|WPA2|UNKNOWN|WEP|OPEN]</em><br /><em>encryption:         [UNKNOWN|WEP104|CCMP|TKIP|WEP40|NONE]</em><br /><em>Note: [Current Connecting| OR</em><br /><em>      WLAN_AVAILABLE_NETWORK_HAS_PROFILE| OR</em><br /><em>      WLAN_AVAILABLE_NETWORK_CONSOLE_USER_PROFILE]</em></p>



<p>This information is gathered for every available network in
the list of networks. Then, a massive switch case used to handle connection
notifications is passed to the Wi-Fi device using WlanRegisterNotification.
Finally, the network authentication method is obtained and encryption is parsed
again. This can be one of the following:</p>



<ul>
<li>WPA2PSK</li>



<li>WPAPSK</li>



<li>UNKNOWN</li>



<li>WEP</li>



<li>OPEN</li>
</ul>



<p>The encryption method can be one of the following:</p>



<ul>
<li>TKIP</li>



<li>NOEN (None misspelled)</li>



<li>WEP</li>



<li>AES</li>
</ul>



<figure class="wp-block-image size-large"><img src="https://www.binarydefense.com/wp-content/uploads/2023/04/wifispread_Picture4-1024x857.png" alt="class=" class="wp-image-24485" /></figure>



<p><em>Figure 4 Worm Generating Network Profile</em></p>



<h4 class="wp-block-heading">Worm Connection Brute-Forcing</h4>



<p>As seen in Figure 4, once each the information for each network has been obtained, the malware moves into the connection, brute-forcing loops. The first step is to zero out two important flags and then to use the data obtained to fill in the below template that will be used to create the Network Profiles:</p>



<p><em>&lt;?xml version="1.0" encoding="UTF-8"?&gt;<br />     &lt;WLANProfile &gt;      &lt;name&gt;%s&lt;/name&gt;<br />     &lt;SSIDConfig&gt;<br />          &lt;SSID&gt;<br />               &lt;name&gt;%s&lt;/name&gt;<br />          &lt;/SSID&gt;<br />     &lt;/SSIDConfig&gt;<br />     &lt;connectionType&gt;ESS&lt;/connectionType<br />&lt;connectionMode&gt;auto&lt;/connectionMode&gt;<br />     &lt;MSM&gt;<br />          &lt;security&gt;<br />               &lt;authEncryption&gt;<br />                     &lt;authentication&gt;%s&lt;/authentication&gt;<br />                     &lt;encryption&gt;%s&lt;/encryption&gt;<br />                     &lt;useOneX&gt;false&lt;/useOneX&gt;</em><br />  <em>             &lt;/authEncryption&gt;<br />               &lt;sharedKey&gt;<br />                     &lt;keyType&gt;passPhrase&lt;/keyType&gt;<br />                     &lt;protected&gt;false&lt;/protected&gt;<br />                     &lt;keyMaterial&gt;%s&lt;/keyMaterial&gt;<br />               &lt;/sharedKey&gt;<br />          &lt;/security&gt;<br />     &lt;/MSM&gt;<br />&lt;/WLANProfile&gt; </em></p>



<p>In the keyMaterial field, a password obtained from one of two
internal password lists is used. Next, the profile is set and a connection is
attempted.</p>



<ul>
<li>If the connection results in
“wlan_notification_acm_network_available”, NetworkAvailableFlag is set to 1.</li>



<li>If the connection results in
“wlan_notification_msm_connected”, NetworkAvailableFlag and
NetworkConnectedFlag are both set to 1.</li>



<li>If the connection is not successful,
NetworkAvailableFlag and NetworkConnectedFlag are both set to 0, and the function
loops after moving to the next password in the password list. </li>
</ul>



<p>If the connection is successful, the malware sleeps for 14 seconds before sending an HTTP POST to its Command and Control (C2) server at 87.106.37[.]146 on port 8080. The resource requested is a static, hard-coded value:<br /><em>87.106.37[.]146:8080/230238982BSBYKDDH938473938HDUI33/index.php</em></p>



<p>The data contained in the POST is:<br /><em>c=&lt;WirelessNetworkConnected&gt;:&lt;Password&gt;</em></p>



<p>Once a connection is established with the Wi-Fi network,
worm.exe begins enumerating users and attempting to brute-force passwords for
all users on the network.</p>



<figure class="wp-block-image size-large"><img src="https://www.binarydefense.com/wp-content/uploads/2023/04/wifispreader-Picture5-860x1024.png" alt="class=" class="wp-image-24486" /></figure>



<p>            <em>Figure 5 Enumerating all non-hidden shares</em></p>



<h4 class="wp-block-heading">Worm User Brute-Forcing</h4>



<p>With the infected victim now connected to a new network, the
malware begins enumerating all non-hidden shares on the network, as seen in
Figure 5. Once shares have been discovered, the malware attempts to connect to
the IPC$ share for the network resources. Using IPC$, it attempts to enumerate
all users connected to the network resource. Using the second password list
contained in the malware, the malware attempts to then brute-force its way into
all users enumerated, saving each successful attempt to 2 buffers: one for the
username and one for the password.</p>



<p>If it is unable to guess passwords for any users, it pivots
to attempting to brute-force the “Administrator” account for the network
resource. </p>



<p>If either of these brute-force attempts are successful, it
then moves onto the spreader function.</p>



<h4 class="wp-block-heading">Spreader</h4>



<figure class="wp-block-image size-large"><img src="https://www.binarydefense.com/wp-content/uploads/2023/04/wifispreader-Picture6-857x1024.png" alt="class=" class="wp-image-24487" /></figure>



<p><em>Figure 6 Connecting and
generating “my.exe”</em></p>



<p>With buffers containing either a list of all usernames
successfully brute-forced and their passwords, or the Administrator account and
its password, worm.exe can now begin spreading service.exe to other systems. </p>



<p>The malware first attempts to gain access to the C$\ share
for the connected network resource. This gives it access to the C drive of the
specified username. From there, it drops service.exe as my.exe in C:\, as seen
in Figure 6. Additionally, it adds a new service with the following information
using the recently dropped my.exe.</p>



<p><em>Binary Path Name: C:\my.exe</em><br /><em>Desired Access: SERVICE_ALL_ACCESS</em><br /><em>Display Name: WinDefService</em><br /><em>Service Name: Windows Defender System Service</em></p>



<p>It then starts this service, executing service.exe as my.exe
on a remote system.</p>



<h4 class="wp-block-heading">Service.exe</h4>



<p>Service.exe is the infected payload installed on remote
systems by worm.exe. This binary has a PE timestamp of 01/23/2020, which was
the date it was first found by Binary Defense. This binary is installed as a
service called WinDefService by worm.exe.</p>



<h4 class="wp-block-heading">Main Thread</h4>



<p>Service.exe’s startup is simple. It has a ServiceMain setup, which calls StartServiceCtrlDispatcher to connect the main thread of the service process to the service control manager, meaning the main thread is run when the service is executed. </p>



<p>In the main thread, service.exe starts a new thread, which
serves two purposes:</p>



<ol>
<li>Communicate back to a new C2 when the service is
installed.</li>



<li>Drop and execute the Emotet binary embedded in
service.exe.</li>
</ol>



<h4 class="wp-block-heading"><br />Service Communication</h4>



<p>The first action service.exe takes is to open a connection to
45.79.223[.]161:443 and send a request to /09FGR20HEU738LDF007E848F715BVE.php.
Although the connection to the server uses port 443, which is normally used for
Transport Layer Security (TLS) encrypted communications, the connection is unencrypted
HTTP. When opening this connection, service.exe sets the following flags:</p>



<ul>
<li>0x80000000 - INTERNET_FLAG_RELOAD</li>



<li>0x04000000 - INTERNET_FLAG_NO_CACHE_WRITE</li>



<li>0x00040000 - INTERNET_FLAG_NO_AUTH</li>



<li>0x00008000 -
INTERNET_FLAG_IGNORE_REDIRECT_TO_HTTP</li>



<li>0x00004000 - INTERNET_FLAG_IGNORE_REDIRECT_TO_HTTPS</li>



<li>0x00002000 -
INTERNET_FLAG_IGNORE_CERT_DATE_INVALID</li>



<li>0x00001000 -
INTERNET_FLAG_IGNORE_CERT_CN_INVALID</li>



<li>0x00000400 - INTERNET_FLAG_HYPERLINK</li>



<li>0x00000200 - INTERNET_FLAG_NO_UI</li>



<li>0x00000100 - CACHE_ENTRY_ACCTIME_FC</li>
</ul>



<p> The resource requested
is hardcoded and static. This IP is currently an active Emotet Command and
Control server (C2). The data contained in the request is:</p>



<p><em>“c=installed”</em></p>



<p>This notifies the server that service.exe has been properly
installed. As both this request and the request sent by worm.exe are not sent
over TLS/SSL, a <a href="https://suricata.readthedocs.io/en/suricata-4.1.4/what-is-suricata.html">Suricata rule</a> is included in this analysis.</p>



<figure class="wp-block-image size-large"><img src="https://www.binarydefense.com/wp-content/uploads/2023/04/wifispreader-Picture7.png" alt="class=" class="wp-image-24488" /></figure>



<p><em>Figure 7. Flags used to create the new Emotet process</em></p>



<h4 class="wp-block-heading">Emotet Process</h4>



<p>After service.exe has installed itself and communicated back
to the C2, it begins dropping the embedded Emotet executable. First, it obtains
the path to %TEMP%, and concatenates “setup.exe” to the end of it. Next, it enters
a function which locates the embedded executable by specifying a hard-coded
buffer offset. Once the embedded executable is located, it creates a new file
at %Temp%\setup.exe and writes the embedded executable to this file. It then
calls CreateProcess in order to create a new process running the Emotet
Executable as seen in Figure 7.</p>



<h4 class="wp-block-heading">Closing</h4>



<p>With this newly discovered loader-type used by Emotet, a new
threat vector is introduced to Emotet’s capabilities. Previously thought to
only spread through malspam and infected networks, Emotet can use this
loader-type to spread through nearby wireless networks if the networks use
insecure passwords. Binary Defense’s analysts recommend using strong passwords
to secure wireless networks so that malware like Emotet cannot gain
unauthorized access to the network. Detection strategies for this threat
include active monitoring of endpoints for new services being installed and
investigating suspicious services or any processes running from temporary
folders and user profile application data folders. Network monitoring is also
an effective detection, since the communications are unencrypted and there are
recognizable patterns that identify the malware message content. </p>
