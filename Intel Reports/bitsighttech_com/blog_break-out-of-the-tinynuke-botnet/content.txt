            <div class="clearfix text-formatted field field--name-body field--type-text-with-summary field--label-hidden field__item">
                          <p><strong>New Tinynuke variant with a DGA<span> in the wild</span></strong></p>

<p>&nbsp;</p>

<h2><span>Summary</span></h2>

<p dir="ltr"><span>Tinynuke, or Nukebot malware, is a trojan able to perform man in the browser attacks against modern web browsers and equipped with the most common features needed by a bank trojan (e.g. Webinjects, Socks proxy, VNC, Remote command execution). This malware was in the spotlight in 2017 after the complete bot source code was leaked in March by someone claiming to be the author of the malware.</span></p>
<!--more-->

<p><span>The leaked source contained a fully working bot, builder and botnet control panel and, as we have seen in past, leaks of working malware usually lead to that malware being used increasingly and new variants and adaptations of it start to emerge.</span></p>

<p><span>We at Bitsight constantly monitor the internet for new threats that may affect the security of organizations worldwide. A few months ago we first noticed an unknown DGA showing up in our network traffic analysis system that we could not identify. After some research we managed to identify the malware behind it, and found a malware operation targeting users in the United Kingdom and Canada using a new variant of Tinynuke with some improvements, such as the use of a DGA (Domain Generation Algorithm) for it’s command and control channel or asymmetric cryptography for C2 authentication.</span></p>

<h2><span>Malware distribution</span></h2>

<p><span>The <a data-entity-substitution="canonical" data-entity-type="node" data-entity-uuid="92be95e6-24bc-4dab-a986-eda833d0df71" href="/blog/what-is-malware">malware</a> is being distributed through fake websites (fake product website or fake blogs) promoted through social networking and advertising, that contain links to malicious software installers.</span></p>

<p><span>The following is an example of one of these websites being used to distribute a trojanized pdf reader software installer.</span></p>

<p><span><img alt="tinynuke-blog-1.png" height="488" src="https://cdn2.hubspot.net/hubfs/277648/images/blog/tinynuke-blog-1.png" width="730" loading="lazy"></span></p>

<p><span>Another interesting example is a fake cryptocurrency security advice blog that was set-up to make available a set of fake installers of known tools that can be used to make a PC more secure. This fake blog was reported to be promoted through facebook advertising by a user who found it suspicious. He wrote about it on his blog (see references). The following image shows this website:</span></p>

<p><span><img alt="tinynuke-blog-site.png" caption="false" data-constrained="true" src="https://cdn2.hubspot.net/hubfs/277648/images/blog/tinynuke-blog-site.png" title="tinynuke-blog-site.png" width="575"></span></p>

<p><span>The software installers contain both the trojan and a legitimate software installer and are, in most cases, digitally signed by a company called “AGM 1980 Limited” with a valid certificate signed by Comodo CA.</span></p>

<p><span>Once the user executes the fake installer:</span></p>

<ul>
	<li><span>Requests administrative privileges to continue with the installation;</span></li>
	<li><span>Performs several checks to verify if it is running on a sandboxed environment (e.g. number of CPU’s, Installed RAM, Virtualization software);</span></li>
	<li><span>If the installer is running on a sandbox it proceeds with the legitimate tool installer and does not install the malware;</span></li>
	<li><span>If the installer is not on a sandbox it installs the malware and then proceeds with the legitimate installer;</span></li>
</ul>

<p><span>After the trojan is installed and running, the user’s system is compromised and it’s data at risk.</span></p>

<h2><span>Affected countries</span></h2>

<p><span>In order to gather some data about the geographies targeted by the malware we have sinkholed it’s DGA domains and observed that there are currently at least 2 different botnets, each using a different TLD on the DGA and each targeting users in a specific country. In particular, one botnet seems to be targeting users in Canada and other users in the United Kingdom.</span></p>

<p><img alt="tinynuke-blog-3.png" height="247" src="https://cdn2.hubspot.net/hubfs/277648/images/blog/tinynuke-blog-3.png" width="316" loading="lazy"></p>

<p><img alt="tinynuke-blog-4.png" height="241" src="https://cdn2.hubspot.net/hubfs/277648/images/blog/tinynuke-blog-4.png" width="320" loading="lazy"></p>

<p>&nbsp;</p>

<h2><span>Webinjects and targets</span></h2>

<p><span>Looking at the webinject configuration we found that besides a few test injects there were also specific injects for two targets. Amazon and Lloyds bank, as shown in the following excerpt (the full webinjects are listed in the IOC’s section):</span></p>

<p><span><img alt="tinynuke-blog-5.png" height="334" src="https://cdn2.hubspot.net/hubfs/277648/images/blog/tinynuke-blog-5.png" width="639" loading="lazy"></span></p>

<p><span>These injects include additional javascript from domains that were registered using the same email address that was used to register the hardcoded domains and the active DGA domains, indicating that these are possibly being developed by the same person or group.</span></p>

<div  class="block-content block-content--type-text block-content--7721 block-content--view-mode-full container text-wysiwyg">

      <div class="text-wysiwyg__body">
      
            <div class="clearfix text-formatted field field--name-body field--type-text-with-summary field--label-hidden field__item">
                          <p><span>It is also worth noting that besides these two webinjects, the normal bot behaviour is to collect HTTP POST request data on IE, Firefox and Chrome for all hosts not listed in the fg_blacklist object in the config. The following blacklist was found active:</span></p>
<p><span><img src="https://cdn2.hubspot.net/hubfs/277648/images/blog/tinynuke-blog-6.png" alt="tinynuke-blog-6.png" width="639" height="100" loading="lazy"></span></p>
<h2><span>Malware variant</span></h2>
<p><span>The malware is mostly the same as the TinyNuke bot that was leaked. However, there are some new features, of which the following are the most relevant:</span></p>
<strong>DGA</strong> 
<p><span>Perhaps the most interesting modification is the introduction of a DGA. This DGA increases the resiliency of the botnet against takedown requests and is triggered when the hardcoded C2 is unreachable.</span></p>
<p><span>Its implementation is very simple, it consists of the md5 of a number concatenated with the current day. The following is an example implementation of the DGA in python:</span></p>
<p><span><img src="https://cdn2.hubspot.net/hubfs/277648/images/blog/tinynuke-blog-8.png" alt="tinynuke-blog-8.png" width="432" height="602" loading="lazy"></span></p>
<strong>C2 authentication using public RSA key</strong>
<p><span>The bot uses asymmetric cryptography to authenticate the C2, as a measure to prevent botnet takeover. To do this, the malware:</span></p>
<ul>
<li><span> </span><span>Generates a random 8 byte long string and calculates it’s md5 value;</span></li>
<li><span> </span><span>Encrypts this value using the C2 RSA Public key that is hard coded on the bot;</span></li>
<li><span> </span><span>Checks if the md5 of the send value is present in the response to the ping command;</span></li>
<li><span> </span><span>If it is, the malware stores this C2 as the active C2 in a global variable and communicates with it directly on further interactions;</span></li>
</ul>
<p><span>The following public key was found in use:</span></p>
<p><img src="https://cdn2.hubspot.net/hubfs/277648/images/blog/tinynuke-blog-7.png" alt="tinynuke-blog-7.png" width="469" height="118" loading="lazy"></p>
<strong>C2 Proxy layer</strong>
<p><span>The bot has been modified to support what appears to be a proxy layer between the C2 and the infected bots. The HTTP requests now have the following format:</span></p>
<p><span><img src="https://cdn2.hubspot.net/hubfs/277648/images/blog/tinynuke-blog-9.png" alt="tinynuke-blog-9.png" width="625" height="36" loading="lazy"></span></p>
<strong>Sandbox/malware analysis detection</strong>
<p><span>The new version includes a few self protection features not present in the leaked source. These are the same that exist in the fake software installers used to drop the malware:</span></p>
<ul>
<li><span> </span><span>Check for sandbox related usernames by looking for the strings:</span><span></span></li>
</ul>
<p><span><img src="https://cdn2.hubspot.net/hubfs/277648/images/blog/tinynuke-blog-10.png" alt="tinynuke-blog-10.png" width="435" height="35" loading="lazy"></span></p>
<ul>
<li><span>Check physical memory on the device is higher than 1Gb;</span></li>
<li><span>Check if the system has more than one CPU;</span></li>
<li><span> </span><span>Check if the Sleep command is being bypassed by calling Sleep and checking if the process actually sleeps;</span></li>
<li><span> </span><span>Check for VMWare tools registry keys;</span></li>
<li><span> </span><span>Check for VirtualBox guest additions files;</span></li>
<li><span> </span><span>Check if the user is connected through remote desktop;</span></li>
</ul>
<h2><span>Conclusion</span></h2>
<p><span>This is an interesting evolution on the TinyNuke malware family and we will continue to monitor this threat as it progresses.</span></p>
