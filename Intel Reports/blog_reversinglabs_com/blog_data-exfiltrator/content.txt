<div id="hs_cos_wrapper_module_1577203992502740" class="hs_cos_wrapper hs_cos_wrapper_widget hs_cos_wrapper_type_module" style="" data-hs-cos-general-type="widget" data-hs-cos-type="module">
<h2 class="post-header__secondary-head blog__post-secondary-h2">A New Tactic for Ransomware Adversaries</h2>
</div>
</div>
<div class="blog-author__short">

<div id="hs_cos_wrapper_module_1705062190107298" class="hs_cos_wrapper hs_cos_wrapper_widget hs_cos_wrapper_type_module" style="" data-hs-cos-general-type="widget" data-hs-cos-type="module"></div>
<div class="blog__post-content html-content">
<span id="hs_cos_wrapper_post_body" class="hs_cos_wrapper hs_cos_wrapper_meta_field hs_cos_wrapper_type_rich_text" style="" data-hs-cos-general-type="meta_field" data-hs-cos-type="rich_text"><h2><img src="https://www.reversinglabs.com/hs-fs/hubfs/Blog/data-exfiltrator-blog.jpg?width=700&amp;name=data-exfiltrator-blog.jpg" alt="Data Exfiltrator" loading="lazy" style="width: 700px;" width="700" srcset="https://www.reversinglabs.com/hs-fs/hubfs/Blog/data-exfiltrator-blog.jpg?width=350&amp;name=data-exfiltrator-blog.jpg 350w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/data-exfiltrator-blog.jpg?width=700&amp;name=data-exfiltrator-blog.jpg 700w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/data-exfiltrator-blog.jpg?width=1050&amp;name=data-exfiltrator-blog.jpg 1050w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/data-exfiltrator-blog.jpg?width=1400&amp;name=data-exfiltrator-blog.jpg 1400w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/data-exfiltrator-blog.jpg?width=1750&amp;name=data-exfiltrator-blog.jpg 1750w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/data-exfiltrator-blog.jpg?width=2100&amp;name=data-exfiltrator-blog.jpg 2100w" sizes="(max-width: 700px) 100vw, 700px"><br><span style="font-weight: bold;">Summary</span></h2>
<p>Over the past year, a major change in tactics employed by ransomware adversaries is to exfiltrate data from the victim's environment. The data then serves as the material for an extortion threat on top of the ransom for encrypted data. This additional tactic became a trend followed by most major ransomware families early last year, 2020 <a href="#references" rel="noopener"><sup>1</sup></a>.To support this tactic, some ransomware operators have added a specific type of malware to perform this exfiltration to their intrusion set<a href="#references" rel="noopener"><sup> 2</sup></a>. The five samples analyzed here perform this type of data exfiltration. They upload a set of files from the victim's computer to command and control servers hosted on IP addresses 51.81.153[.]212, 51.161.82[.]135, and 51.77.110[.]6. All of these IP addresses are owned by OVH SAS, a French hosting company. The malware follows the exfiltration with a single line PowerShell command that stops the malware's running process and then deletes the malware file that was executed. The malware has a type of anti-analysis behavior called "Relocate API Code" according to the Malware Behavior Catalog's <a href="#references" rel="noopener"><sup>3</sup></a> categorization <a href="#references" rel="noopener"><sup>4</sup></a>. The malware reads a copy of system DLLs into memory and resolves imports from there. This causes a problem for debuggers such as x64dbg <a href="#references" rel="noopener"><sup>5</sup></a>.<br><br>Interestingly, these files share code with an earlier malware sample with completely different capabilities. This earlier file has been observed alongside TrickBot, CobaltStrike, and ransomware <a href="#references" rel="noopener"><sup>6</sup></a>. This earlier malware additionally uses the same anti-analysis technique, but does not exfiltrate data. It has the capability to download a CobaltStrike beacon and execute it <a href="#references" rel="noopener"><sup>7</sup></a>. In addition to this overlap in code and behavior, the command and control (C2) infrastructure domains are registered via the same registrar. Also, the C2 IP addresses are owned by the same hosting company, OVH.</p>
<!--more-->
<h2 style="font-weight: bold;">Anti-analysis Trick</h2>
<p style="font-weight: bold;">Relocate API Code</p>
<p>The first behavior one observes when running these samples in a sandbox or in a debugger is that at the point where the imports are resolved an exception is raised. Debugging past this point is not possible without circumvention of an anti-analysis trick. This circumvention starts by examining the first encoded string the samples decode. Encoded strings are found in two general forms in these samples. First is with all the rest of the strings in the file in the .data section. Some of these can be seen in Figure 1 with one example highlighted.</p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/data_exfiltrator_fig1_encoded_string.jpg" class="lightbox-image"><img src="https://www.reversinglabs.com/hubfs/Blog/data_exfiltrator_fig1_encoded_string.jpg" alt="Encoded Strings" style="width: 1400px;"></a>
<p style="text-align: center;"><br><span style="font-weight: bold; font-size: 16px;">Figure 1: Encoded Strings</span></p>
<p>Notice the "mOrxxxx" characters that trail each of the encoded strings. These trailing characters will be examined below.<br><br>The other location where encoded strings are found is split up character-by-character as stack strings. Each byte is moved one-by-one to a location on the stack before the decoding operation occurs. The first string of this type is in the function that resolves imports from kernel32.dll. This encoded string is shown in Figure 2.</p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/data_exfiltrator_fig2_kernel32_dll.jpg" class="lightbox-image"><img src="https://www.reversinglabs.com/hubfs/Blog/data_exfiltrator_fig2_kernel32_dll.jpg" alt="First Encoded String" style="width: 1400px;"></a>
<p style="text-align: center;"><br><span style="font-weight: bold; font-size: 16px;">Figure 2: First Encoded String</span></p>
<p>This first string decodes to "C:\Windows\System32\kernel32.dll". This path is then used to read the DLL from the filesystem into memory. Imports are then resolved against this copy of the DLL rather than the system DLL. The function calls to copy the DLL are shown in Figure 3.</p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/data_exfiltrator_fig3_copy_dll.jpg" class="lightbox-image"><img src="https://www.reversinglabs.com/hubfs/Blog/data_exfiltrator_fig3_copy_dll.jpg" alt="Copy DLL Function" style="width: 1400px;"></a>
<p style="text-align: center;"><br><span style="font-weight: bold; font-size: 16px;">Figure 3: Copy DLL Function</span></p>
<p>In the debugger's environment, this read fails with an exception which then prevents the imports from being properly resolved <a href="#references" rel="noopener"><sup>8</sup></a>. A detailed explanation of what's happening here can be found on OALabs YouTube channel <a href="#references" rel="noopener"><sup>9</sup></a>.To circumvent this trick, one can create a copy of the DLLs on the filesystem and change their names along with the decoded path strings. This way the DLLs can be read correctly and the imports properly resolved. This change can be done on the fly in the debugger after the strings are decoded. An example of changing this on the fly using the filename kernel33.dll is shown in Figure 4.</p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/data_exfiltrator_fig4_kernel33_dll.jpg" class="lightbox-image"><img src="https://www.reversinglabs.com/hubfs/Blog/data_exfiltrator_fig4_kernel33_dll.jpg" alt="Change DLL Name" style="width: 1400px;"></a>
<p style="text-align: center;"><br><span style="font-weight: bold; font-size: 16px;">Figure 4: Change DLL Name</span></p>
<p>Alternatively, the single encoded byte needed for this change can be modified in the sample with a hex editor to make this change permanent. This also makes restarting the analysis in the debugger less annoying. This byte difference is highlighted using HexFiend's <a href="#references" rel="noopener"><sup>10</sup></a> comparison function and can be seen in Figure 5.</p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/data_exfiltrator_fig5_compare_patch.jpg" class="lightbox-image"><img src="https://www.reversinglabs.com/hubfs/Blog/data_exfiltrator_fig5_compare_patch.jpg" alt="Original Compared to Patch" style="width: 1400px;"></a>
<p style="text-align: center;"><br><span style="font-weight: bold; font-size: 16px;">Figure 5: Original Compared to Patch</span></p>
<p>After the DLL path has been decoded, the various imports from that DLL are then decoded from similar stack strings. One example with LoadLibraryW is shown in Figure 6. Again, please note the trailing "mOrxxxx" string immediately after the encoded bytes of the "LoadLibraryW" string.</p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/data_exfiltrator_fig6_decode_plus.jpg" class="lightbox-image"><img src="https://www.reversinglabs.com/hubfs/Blog/data_exfiltrator_fig6_decode_plus.jpg" alt="Encoded LoadLibraryW with Trailing Additional String" style="width: 1400px;"></a>
<p style="text-align: center;"><br><span style="font-weight: bold; font-size: 16px;">Figure 6: Encoded LoadLibraryW with Trailing Additional String</span></p>
<p>Once the import strings are decoded, a custom implementation of GetProcAddress is used on the copy of kernel32.dll to resolve the imports. The results of this process can be seen in Figure 7.</p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/data_exfiltrator_fig7_after_import.jpg" class="lightbox-image"><img src="https://www.reversinglabs.com/hubfs/Blog/data_exfiltrator_fig7_after_import.jpg" alt="After Imports Resolved" style="width: 1400px;"></a>
<p style="text-align: center;"><br><span style="font-weight: bold; font-size: 16px;">Figure 7: After Imports Resolved</span></p>
<p>Examining the exports for these samples shows a DLL name "Input.exe" as well as one exported function "bsearch". These exports are shown in Figure 8.</p>

<p>The data exfiltration process starts by enumerating the logical drives that are available on the victim's computer. This is determined using a call to GetLogicalDriveStringsW and is shown in Figure 23.</p>

<p>For each of these logical drives, a function is called that walks the file system searching for targeted files and exfiltrating them. This walk function interestingly is recursive. This recursion is shown in Figure 24.</p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/data_exfiltrator_fig24_recursive_walk.jpg" class="lightbox-image"><img src="https://www.reversinglabs.com/hubfs/Blog/data_exfiltrator_fig24_recursive_walk.jpg" alt="Walk Function Recursion" style="width: 1400px;"></a>

<h2 style="font-weight: bold;">Related Malware</h2>
<p>One function in particular that is found in each of the three samples analyzed above is the string decoding function. Encoding and decoding functions are a good area of code to examine closely and to hunt for in malware repositories. This type of function can be stable across samples and tends to be reused by an adversary in multiple campaigns even if the capabilities of the malware are radically different. Building a YARA rule based on this function reveals an older malware sample <a href="#references" rel="noopener"><sup>21</sup></a> which was blogged about by researchers at Walmart <a href="#references" rel="noopener"><sup>22</sup></a>. The results of a retrohunt in the Titanium Platform using this YARA rule is shown in Figure 39. The result shown in orange is a false positive. This file is a copy of one of the other actual malware files but appears to have been modified by a researcher.</p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/data_exfiltrator_fig39_retrohunt_results.jpg" class="lightbox-image"><img src="https://www.reversinglabs.com/hubfs/Blog/data_exfiltrator_fig39_retrohunt_results.jpg" alt="Retrohunt Results" style="width: 1400px;"></a>

Both the standard string form as well as the stack strings are found in this sample. However, this older sample is a CobaltStrike beacon loader rather than a data exfiltrator. The YARA rule for detecting this code overlap is provided at the end of the blog. As opposed to earlier YARA rules that I have written, I read the advice from Marc Ochsenmeier on Twitter about adding comments with the meaning of opcodes in YARA rules meant for sharing <a href="#references" rel="noopener"><sup>23</sup></a>. This rule and future rules will include the assembly as well as the bytecode strings.<br><br>In addition to this older malware sample that is definitely related, a hunt for other malware that contains variations of the string "mOrxxxx" reveals a multitude of potentially related malware samples.
