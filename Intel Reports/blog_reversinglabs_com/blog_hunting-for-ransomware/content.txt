<div id="hs_cos_wrapper_module_1577203992502740" class="hs_cos_wrapper hs_cos_wrapper_widget hs_cos_wrapper_type_module" style="" data-hs-cos-general-type="widget" data-hs-cos-type="module">
<h2 class="post-header__secondary-head blog__post-secondary-h2">Getting Ahead of Ryuk attacks using YARA rules</h2>
</div>

<div class="blog__post-content html-content">
<span id="hs_cos_wrapper_post_body" class="hs_cos_wrapper hs_cos_wrapper_meta_field hs_cos_wrapper_type_rich_text" style="" data-hs-cos-general-type="meta_field" data-hs-cos-type="rich_text"><p style="text-align: left;"><img src="https://www.reversinglabs.com/hs-fs/hubfs/blog-hunting-for-ransomware.jpg?width=700&amp;name=blog-hunting-for-ransomware.jpg" alt="RYUK - Hunting for Ransomware" style="width: 700px;" width="700" srcset="https://www.reversinglabs.com/hs-fs/hubfs/blog-hunting-for-ransomware.jpg?width=350&amp;name=blog-hunting-for-ransomware.jpg 350w, https://www.reversinglabs.com/hs-fs/hubfs/blog-hunting-for-ransomware.jpg?width=700&amp;name=blog-hunting-for-ransomware.jpg 700w, https://www.reversinglabs.com/hs-fs/hubfs/blog-hunting-for-ransomware.jpg?width=1050&amp;name=blog-hunting-for-ransomware.jpg 1050w, https://www.reversinglabs.com/hs-fs/hubfs/blog-hunting-for-ransomware.jpg?width=1400&amp;name=blog-hunting-for-ransomware.jpg 1400w, https://www.reversinglabs.com/hs-fs/hubfs/blog-hunting-for-ransomware.jpg?width=1750&amp;name=blog-hunting-for-ransomware.jpg 1750w, https://www.reversinglabs.com/hs-fs/hubfs/blog-hunting-for-ransomware.jpg?width=2100&amp;name=blog-hunting-for-ransomware.jpg 2100w" sizes="(max-width: 700px) 100vw, 700px"></p>
<h2 style="font-weight: bold;">November Update:</h2>
<p>Here’s your opportunity to hear directly from Rob Simmons, Threat Researcher involved in #Ryuk ransomware research.</p>
<h6>Join us to learn:</h6>
<ul>
<li>The current state of Ransomware and how it is becoming more targeted</li>
<li>How to use the A1000 to hunt for threats using YARA</li>
<li>How to bring new visibility about file risks into your SOC process</li>
<li>How to apply this new intelligence on Ryuk to actively update your defenses</li>
</ul>
<p>Register for our November 17 webinar here:<br><a href="https://reversinglabs.zoom.us/webinar/register/6215881027977/WN_X6tAd0-NTeSRllyjtEQS0g" rel="noopener" target="_blank">https://reversinglabs.zoom.us/webinar/register/6215881027977/WN_X6tAd0-NTeSRllyjtEQS0g</a><br><br></p>
<p style="text-align: left;">Many ransomware families have changed their tactics and victim-targeting in recent years. Rather than indiscriminate attacks against anyone they’re able to infect, they have moved to a process called “big game hunting”. The motivation underlying this change of tactics is to increase the potential payout by targeting an organization rather than an individual. The adversary performs extensive reconnaissance on the target to determine what they may be able to pay. Rather than small ransom demands in thousands of dollars, by targeting businesses, they are aiming for payouts in the hundreds of thousands to millions of dollars. <br><br>One malware family in particular, Ryuk <a href="#_ftn1" name="_ftnref1" rel=" noopener"><sup>[1]</sup></a>, has been attributed to the GRIM SPIDER <sup><a href="#_ftn2" name="_ftnref2" rel=" noopener">[2]</a></sup> threat actor group. According to malpedia.io, this group has been operating the Ryuk ransomware since August of 2018 <sup><a href="#_ftn3" name="_ftnref3" rel=" noopener">[3]</a></sup>. In recent months, a staged attack dubbed “triple threat” <sup><a href="#_ftn4" name="_ftnref4" rel=" noopener">[4]</a></sup> has emerged with the initial access to the network achieved by the Emotet <sup><a href="#_ftn5" name="_ftnref5" rel=" noopener">[5]</a></sup> malware family. Once initial access is achieved, the next stage, TrickBot <sup><a href="#_ftn6" name="_ftnref6" rel=" noopener">[6]</a></sup>, delivered inside the target organization. TrickBot has capabilities to steal credentials and to move laterally within the organization’s network. The third stage of the attack is to execute Ryuk ransomware on as many workstations and servers as possible via the lateral movement of TrickBot.</p>
<!--more-->
<p style="text-align: left;">To hunt for and identify Ryuk samples, many YARA <sup><a href="#_ftn7" name="_ftnref7" rel=" noopener">[7]</a></sup> rules search for strings that are hard-coded in the sample. However, this type of strings-based rule may be prone to false positives. An excellent conference talk that includes this topic given by Lauren Pierce at ShmooCon 2017 should be watched for more information about this concept. <sup><a href="#_ftn8" name="_ftnref8" rel=" noopener">[8]</a></sup> Rather than hunting for these hard-coded strings, one should be hunting for code patterns in the sample. Rules of this type do more damage to the adversary’s intrusion set according to David Bianco’s Pyramid of Pain. <sup><a href="#_ftn9" name="_ftnref9" rel=" noopener">[9]</a></sup> More painful code changes are needed to avoid detection by this paradigm of YARA rule. Here, we examine a single algorithm that Ryuk uses in the latest 64bit variant to generate a random string. This string is part of the filename that Ryuk uses when dropping a copy of itself during the installation phase of intrusion.</p>
<p style="text-align: left;">Looking at the execution of the Ryuk sample <sup><a href="#_ftn10" name="_ftnref10" rel=" noopener">[10]</a></sup> in x64dbg, <sup><a href="#_ftn11" name="_ftnref11" rel=" noopener">[11]</a></sup> we see that the first step taken is to gather entropy from the tick count of the victim’s computer. In Figure 1, we see the library function call to GetTickCount to gather this randomness.</p>
<a href="https://cdn2.hubspot.net/hubfs/3375217/Blog/hunting_for_ransomware_01.jpg" class="lightbox-image"><img src="https://www.reversinglabs.com/hs-fs/hubfs/Blog/hunting_for_ransomware_01.jpg?width=1400&amp;name=hunting_for_ransomware_01.jpg" alt="Figure 1: Entropy Input From Tick Count" style="width: 1400px;" width="1400" srcset="https://www.reversinglabs.com/hs-fs/hubfs/Blog/hunting_for_ransomware_01.jpg?width=700&amp;name=hunting_for_ransomware_01.jpg 700w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/hunting_for_ransomware_01.jpg?width=1400&amp;name=hunting_for_ransomware_01.jpg 1400w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/hunting_for_ransomware_01.jpg?width=2100&amp;name=hunting_for_ransomware_01.jpg 2100w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/hunting_for_ransomware_01.jpg?width=2800&amp;name=hunting_for_ransomware_01.jpg 2800w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/hunting_for_ransomware_01.jpg?width=3500&amp;name=hunting_for_ransomware_01.jpg 3500w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/hunting_for_ransomware_01.jpg?width=4200&amp;name=hunting_for_ransomware_01.jpg 4200w" sizes="(max-width: 1400px) 100vw, 1400px"></a>
<p style="text-align: center;"><br><span style="font-size: 16px;"><strong>Figure 1: Entropy Input From Tick Count</strong></span></p>
<p>According to Microsoft’s documentation, GetTickCount returns “the number of milliseconds that have elapsed since the system was started.” <sup><a href="#_ftn12" name="_ftnref12" rel=" noopener">[12]</a></sup> The function called immediately after is a C library function, srand. This function takes a seed value and initializes the random number generator. The srand and rand functions’ identities were detected using Ghidra’s <sup><a href="#_ftn13" name="_ftnref13" rel=" noopener">[13]</a></sup> function signatures during its code analysis process.<br><br>The random number generator initialized by srand is subsequently used by rand function calls to generate random data. The goal of generating this data is to produce a random string. However, not all the bytes of randomness generated can be used in a filename, so a subsequent function checks the output to verify that the generated byte is an alphabet character and therefore valid for a filename. This function has been labelled as “isalpha” in Figure 2.</p>

<p>If the byte fails this test, execution jumps back to the rand function and a new random byte is generated. This loop continues until all the characters are alphabet characters, and the generated string is therefore usable as a filename.<br><br>To write an effective YARA rule for detecting this algorithm, first we examine the srand function and find a hexadecimal string that can be used to match the function. Figure 3 shows the srand function in the debugger’s disassembler.</p>

E8 [4] 89 }</span></strong><br><br>The hexadecimal string seen above identifies the srand function, but leaves room for the destination registers to change and still match the function. <sup><a href="#_ftn14" name="_ftnref14" rel=" noopener">[14]</a></sup> These wildcards are represented as “??”. The four byte jump “[4]” allows for the address of the “__acrt_getptd” function to change locations.<br><br>Next we repeat the same process by examining the rand function in the debugger.</p>

<p>Following the same methodology to write a signature as above, destination registers except for the opcode “movsxd” are replaced with wildcards. Then the location of the pointer from the “mov” instruction is replaced with a four byte jump. The resulting hexadecimal string is as follows:<br><br><strong><span style="font-family: 'courier new', courier;">$isalpha = { 40 53 48 83 EC ?? 48 63 D9 8B 05 [4] 85 C0 74 4E }</span></strong><br><br>Armed with the locations of the three functions, we return to analyzing the code that is used to call them. This code can be split into two separate opcode signatures. This will allow for variation in the code between these two code snippets thereby still identifying this adversary code even if it changes slightly as the ransomware code is developed for new variants of Ryuk.</p>

<p>By following the process of allowing for variation in destination registers as well as the location of the called functions, the following two opcode signatures are developed:<br><br><strong><span style="font-family: 'courier new', courier;">$op1 = { E8 [4] 49 8B ?? E8 [4] ?? }</span></strong><br><strong><span style="font-family: 'courier new', courier;">$op2 = { 03 ?? 8B ?? E8 [4] 85 C0 74 ?? }</span></strong><br><br>Now that we have signatures for the functions that are called as well as signatures for the code that calls them, we tie these together by comparing the bytes found in the opcode that calls the function with the location of the called function. This is done by using YARA condition statements to calculate the locations. This first condition statement verifies that the first opcode calls the srand function:</p>
<p><strong><span style="font-family: 'courier new', courier;">uint32(@op1 + 1) + @op1 + 5 == @srand</span></strong><br><br>This condition verifies that the first opcode then calls the rand function:<br><br><strong><span style="font-family: 'courier new', courier;">uint32(@op1 + 9) + @op1 + 13 == @rand</span></strong><br><br>And this condition verifies that the second opcode calls the isalpha function:<br><br><strong><span style="font-family: 'courier new', courier;">uint32(@op2 + 5) + @op2 + 9 == @isalpha</span></strong><br><br>The last instruction seen in Figure 8 is a jump that leads back to the rand function to generate a new byte of random data if the previous byte was not an alphabet character. The following condition reflects this jump and allows for the landing address of the jump to change based on differences at compile time or code changes between the two opcode snippets.<br><br><strong><span style="font-family: 'courier new', courier;">@op2 + 5 + int8(@op2 + 12) == @op1</span></strong><br><br>Now that we have a fully formed YARA rule, we can hunt for samples of Ryuk using the Titanium Platform that are related to the one that we started with. The complete YARA rule is provided at the bottom.</p>
