    <h1 class="text-center">GPlayed Trojan - .Net playing with Google Market</h1>

    <div class="text-center m-1">
        <div class="post-author">
            <span>By </span>
                <a href="https://blog.talosintelligence.com/author/vitor-ventura/">Vitor Ventura</a>
        </div>

        <br/>

    <section class="post-content-wrapper mt-5">
        <div class="post-content">
            <p>This blog post is authored by<a href="https://twitter.com/_vventura">Vitor Ventura</a>.</p><h3 id="introduction">Introduction</h3><p>In a world where everything is always connected, and mobile devices are involved in individuals' day-to-day lives more and more often, malicious actors are seeing increased opportunities to attack these devices. Cisco Talos has identified the latest attempt to penetrate mobile devices â€” a new Android trojan that we have dubbed "GPlayed." This is a trojan with many built-in capabilities. At the same time, it's extremely flexible, making it a very effective tool for malicious actors. The sample we analyzed uses an icon very similar to Google Apps, with the label "Google Play Marketplace" to disguise itself.</p><figure class="kg-card kg-image-card"><img src="https://blog.talosintelligence.com/content/images/-kN38WqRfbdk/W6Ifxr-wYeI/AAAAAAAAAK4/JTOaH8cvN4Ym8V6HIj3jYggxkr6J21RaACK4BGAYYCw/s640/image14.png" class="kg-image" alt loading="lazy"></figure><p><br> The malicious application is on the left-hand side.</p><p>What makes this malware extremely powerful is the capability to adapt after it's deployed. In order to achieve this adaptability, the operator has the capability to remotely load plugins, inject scripts and even compile new .NET code that can be executed. Our analysis indicates that this trojan is in its testing stage but given its potential, every mobile user should be aware of GPlayed. Mobile developers have recently begun eschewing traditional app stores and instead want to deliver their software directly through their own means. But GPlayed is an example of where this can go wrong, especially if a mobile user is not aware of how to distinguish a fake app versus a real one. <br></p><h3 id="trojan-architecture-and-capabilities">Trojan architecture and capabilities</h3><p>This malware is written in .NET using the Xamarin environment for mobile applications. The main DLL is called "Reznov.DLL." This DLL contains one root class called "eClient," which is the core of the trojan. The imports reveal the use of a second DLL called "eCommon.dll." We determined that the "eCommon" file contains support code and structures that are platform independent. The main DLL also contains eClient subclasses that implement some of the native capabilities.</p><p>The package certificate is issued under the package name, which also resembles the name of the main DLL name.</p><figure class="kg-card kg-image-card"><img src="https://blog.talosintelligence.com/content/images/-WawcRxTnliU/W6IgP5nanjI/AAAAAAAAALM/bVDUMbwKgMIcDTNAk_ZN76iPrJGaT86FQCK4BGAYYCw/s640/image5.png" class="kg-image" alt loading="lazy"></figure><p><br> Certificate information <br> The Android package is named "verReznov.Coampany." The application uses the label "Installer" and its name is "android.app.Application." <br></p><figure class="kg-card kg-image-card"><img src="https://blog.talosintelligence.com/content/images/-6U4pSkQHgzY/W6IgsiMsnQI/AAAAAAAAALY/lnD3snBCKnEG9iPgyXVnKU2V2EHdopUGQCK4BGAYYCw/s640/image2.png" class="kg-image" alt loading="lazy"></figure><p><br> Package permissions</p><p>The trojan declares numerous permissions in the manifest, from which we should highlight the BIND_DEVICE_ADMIN, which provides nearly full control of the device to the trojan.</p><p>This trojan is highly evolved in its design. It has modular architecture implemented in the form of plugins, or it can receive new .NET source code, which will be compiled on the device in runtime.</p><figure class="kg-card kg-image-card"><img src="https://blog.talosintelligence.com/content/images/-_lTYk6uID68/W6Ihj8dGO_I/AAAAAAAAALk/1DfqQb0x0VIRVXyM3sv68xbvJn5B-ZzEACK4BGAYYCw/s640/image9.png" class="kg-image" alt loading="lazy"></figure><p><br> Initialization of the compiler object</p><p>The plugins can be added in runtime, or they can be added as a package resource at packaging time. This means that the authors or the operators can add capabilities without the need to recompile and upgrade the trojan package on the device.</p><figure class="kg-card kg-image-card"><img src="https://blog.talosintelligence.com/content/images/-1ESJhmo3-tU/W6IhnBg5gcI/AAAAAAAAALs/Bs6JOvt6BaQXteQ4yOrkgbQiUxfredPkQCK4BGAYYCw/s640/image11.jpg" class="kg-image" alt loading="lazy"></figure><p><br> Trojan native capabilities</p><p>This is a full-fledged trojan with capabilities ranging from those of a banking trojan to a full spying trojan. This means that the malware can do anything from harvest the user's banking credentials, to monitoring the device's location. There are several indicators (see section "trojan activity" below) that it is in its last stages of development, but it has the potential to be a serious threat. <br></p><h3 id="trojan-details">Trojan details</h3><p>Upon boot, the trojan will start by populating a shared preferences file with the configuration it has on its internal structures. Afterward, it will start several timers to execute different tasks. The first timer will be fired on the configured interval (20 seconds in this case), pinging the command and control (C2) server. The response can either be a simple "OK," or can be a request to perform some action on the device. The second timer will run every five seconds and it will try to enable the WiFi if it's disabled. The third timer will fire every 10 seconds and will attempt to register the device into the C2 and register wake-up locks on the system to control the device's status.</p><p>During the trojan registration stage, the trojan exfiltrates private information such as the phone's model, IMEI, phone number and country. It will also report the version of Android that the phone is running and any additional capabilities.</p><figure class="kg-card kg-image-card"><img src="https://blog.talosintelligence.com/content/images/-5ZU9E99_hUk/W6IiQwSpFcI/AAAAAAAAAL8/GB66zdj4UHkz8Sq3jSXBK0PQVEPWCVMGwCK4BGAYYCw/s640/image1.png" class="kg-image" alt loading="lazy"></figure><p><br> Device registration <br> This is the last of the three main timers that are created. The trojan will register the SMS handler, which will forward the contents and the sender of all of the SMS messages on the phone to the C2.</p><p>The final step in the trojan's initialization is the escalation and maintenance of privileges in the device. This is done both by requesting admin privileges on the device and asking the user to allow the application to access the device's settings.</p><figure class="kg-card kg-image-card"><img src="https://blog.talosintelligence.com/content/images/-QO05rPtaX2g/W6Iiei93dLI/AAAAAAAAAME/83lZjUZLw-U5KcwVuDBZYgb89jX6Cpi8wCK4BGAYYCw/s400/image8.png" class="kg-image" alt loading="lazy"></figure><figure class="kg-card kg-image-card"><img src="https://blog.talosintelligence.com/content/images/-dcs8piIexRE/W6IihNEJHpI/AAAAAAAAAMQ/MTmEOLUtRtAqGT4cTdVX_TwbHQeeZDQ1QCK4BGAYYCw/s400/image7.png" class="kg-image" alt loading="lazy"></figure><p><br> Privilege escalation requests</p><p>The screens asking for the user's approval won't close unless the user approves the privilege escalation. If the user closes the windows, they will appear again due to the timer configuration.</p><p>After the installation of the trojan, it will wait randomly between three and five minutes to activate one of the native capabilities â€” these are implemented on the eClient subclass called "GoogleCC." This class will open a WebView with a Google-themed page asking for payment in order to use the Google services. This will take the user through several steps until it collects all the necessary credit card information, which will be checked online and exfiltrated to the C2. During this process, an amount of money, configured by the malicious operator, is requested to the user.</p><figure class="kg-card kg-image-card"><img src="https://blog.talosintelligence.com/content/images/-HnAQWMSLvxM/W6Ii6PhYY3I/AAAAAAAAAMg/ySse5lhbX685ODhj-Ke4-oA2f_2XqK-sQCK4BGAYYCw/s320/image3.png" class="kg-image" alt loading="lazy"></figure><figure class="kg-card kg-image-card"><img src="https://blog.talosintelligence.com/content/images/-N3jKNMkplrk/W6Ii82sewQI/AAAAAAAAAMo/WJVigc3OIy4ZB8XQMxZ3fub19u15OctCACK4BGAYYCw/s320/image12.png" class="kg-image" alt loading="lazy"></figure><figure class="kg-card kg-image-card"><img src="https://blog.talosintelligence.com/content/images/-Uy_l-XRpzcw/W6IjDc3khnI/AAAAAAAAAMw/xQza_u4yn8c4IHSWxQwzaaO4SPfqzxuTgCK4BGAYYCw/s320/image4.png" class="kg-image" alt loading="lazy"></figure><p><br> Steps to request the user's credit card information<br></p><p>In our sample configuration, the request for the views above cannot be canceled or removed from the screen â€” behaving just like a screen lock that won't be disabled without providing credit card information.</p><p>All communication with the C2 is done over HTTP. It will use either a standard web request or it will write data into a web socket if the first method fails. The C2 can also use WebSocket as a backup communication channel.</p><p>Before sending any data to the C2 using the trojan attempts to disguise its data, the data is serialized using JSON, which is then encoded in Base64. However, the trojan replaces the '=' by 'AAAZZZXXX', the '+' by '|' and the '/' by '.' to disguise the Base64.</p><figure class="kg-card kg-image-card"><img src="https://blog.talosintelligence.com/content/images/-ICC-_c4UTmk/W6IkG9BXQhI/AAAAAAAAANA/ZSB6AHU1Y9g1BPFAl2kiqY2NXWT2tIxLgCK4BGAYYCw/s640/image10.png" class="kg-image" alt loading="lazy"></figure><p><br> Request encoding process <br> The HTTP requests follow the format below, while on the WebSocket only the query data is written.</p><p>&lt;server path&gt;?q=&lt;IMEI&gt;-&lt;REQUEST CODE&gt;:&lt;Obfuscated Base64 encoded data&gt;</p><p>As is common with trojans, the communication is always initiated by the trojan on the device to the C2. The request codes are actually replies to the C2 action requests, which are actually called "responses." There are 27 response codes that the C2 can use to make requests to the trojan, which pretty much match what's listed in the capabilities section.</p><!--kg-card-begin: html--><table><tbody><tr><td><ul><li> Error</li><li> Registration</li><li> Ok</li><li> Empty</li><li> SendSMS</li><li> RequestGoogleCC</li><li> Wipe</li><li> OpenBrowser</li><li> SendUSSD</li></ul>
</td><td><ul><li> RequestSMSList</li><li> RequestAppList</li><li> RequestLocation</li><li> ShowNotification</li><li> SetLockPassword</li><li> LockNow</li><li> MuteSound</li><li> LoadScript</li><li> LoadPlugin</li></ul>

</td></tr></tbody></table><!--kg-card-end: html--><p>Another feature of this trojan is the ability to register injects, which are JavaScript snippets of code. These will be executed in a WebView object created by the trojan. This gives the operators the capability to trick the user into accessing any site while stealing the user's cookies or forging form fields, like account numbers or phone numbers. <br></p><h3 id="trojan-activity-at-the-time-of-the-writing-of-this-post-all-urls-see-ioc-section-found-on-the-sample-were-inactive-and-it-does-not-seem-to-be-widespread-there-are-some-indicators-that-this-sample-is-just-a-test-sample-on-its-final-stages-of-development-there-are-several-strings-and-labels-still-mentioning-test-or-testcc-even-the-url-used-for-the-credit-card-data-exfiltration-is-named-testcc-php-">Trojan activity At the time of the writing of this post, all URLs (see IOC section) found on the sample were inactive, and it does not seem to be widespread. There are some indicators that this sample is just a test sample on its final stages of development. There are several strings and labels still mentioning 'test' or 'testcc' â€” even the URL used for the credit card data exfiltration is named "testcc.php."</h3><figure class="kg-card kg-image-card"><img src="https://blog.talosintelligence.com/content/images/-db47Gub0HG8/W6ImA7P8ZBI/AAAAAAAAANM/tsML-GC5OXA5ZHkIxXC2ojEdn3vbViaNgCK4BGAYYCw/s640/image13.png" class="kg-image" alt loading="lazy"></figure><p><br> Debug information on logcat</p><p>Another indicator is the amount of debugging information the trojan is still generating â€” a production-level trojan would keep its logging to a minimum.</p><p>The only sample was found on public repositories and almost seemed to indicate a test run to determine the detection ratio of the sample. We have observed this trojan being submitted to public antivirus testing platforms, once as a package and once for each DLL to determine the detection ratio. The sample analyzed was targeted at Russian-speaking users, as most of the user interaction pages are written in Russian. However, given the way the trojan is built, it is highly customizable, meaning that adapting it to a different language would be extremely easy. The wide range of capabilities doesn't limit this trojan to a specific malicious activity like a banking trojan or a ransomware. This makes it impossible to create a target profile. <br></p><h3 id="conclusion">Conclusion</h3><p>This trojan shows a new path for threats to evolve. Having the ability to move code from desktops to mobile platforms with no effort, like the eCommon.DLL demonstrates that malicious actors can create hybrid threats faster and with fewer resources involved than ever before. This trojan's design and implementation is of an uncommonly high level, making it a dangerous threat. These kinds of threats will become more common, as more and more companies decide to publish their software directly to consumers.</p><p>There have been several recent examples of companies choosing to release their software directly to consumers, bypassing traditional storefronts. The average user might not have the necessary skills to distinguish legitimate sites from malicious ones. We've seen that this has been the case for many years with spear-phishing campaigns on desktop and mobile platforms, so, unfortunately, it doesn't seem that this will change any time soon.
