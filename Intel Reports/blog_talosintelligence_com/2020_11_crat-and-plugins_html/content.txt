    <h1 class="text-center">CRAT wants to plunder your endpoints</h1>

    <div class="text-center m-1">
        <div class="post-author">
            <span>By </span>
                <a href="https://blog.talosintelligence.com/author/asheer-malhotra/">Asheer Malhotra</a>

        <br/>
        <time class="post-datetime" datetime="November 12, 2020 08:18">
            Thursday, November 12, 2020 08:18
        </time>

        <div class="m-3">
                            <a href="/category/rat/" class="category primary-category">
                                RAT
                            </a>
                                                    <a href="/category/threat-spotlight/" class="category primary-category">

    <section class="post-content-wrapper mt-5">
        <div class="post-content">
            <ul><li>Cisco Talos has observed a new version of a remote access trojan (RAT) family known as CRAT.</li><li>Apart from the prebuilt RAT capabilities, the malware can download and deploy additional malicious plugins on the infected endpoint.</li><li>One of the plugins is a ransomware known as "Hansom."</li><li>CRAT has been attributed to the Lazarus APT Group in the past.</li><li>The RAT consists of multiple obfuscation techniques to hide strings, API names, command and control (C2) URLs and instrumental functions, along with static detection evasion.</li><li>The attack also employs a multitude of anti-infection checks to evade sandbox based detection systems. </li></ul><h2 id="what-s-new">What's new? </h2><p>Cisco Talos has recently discovered a new version of the CRAT malware family. This version consists of multiple RAT capabilities, additional plugins and a variety of detection-evasion techniques. In the past, CRAT has been attributed to the Lazarus Group, the malicious threat actors behind multiple cyber campaigns, including attacks against the entertainment sector.</p><p>Indicators and tactics, techniques and procedures (TTPs) discovered by this investigation resemble those of the Lazarus Group.</p><h2 id="how-did-it-work">How did it work? </h2><p>The attack consists of a highly modular malware that can function as a standalone RAT and download and activate additional malicious plugins from its C2 servers. Cisco Talos has discovered multiple plugins so far, consisting of ransomware, screen-capture, clipboard monitoring and keylogger components.</p><h2 id="so-what">So what? </h2><p>This attack demonstrates how the adversary operates an attack that:</p><ul><li>Uses obfuscation and extensive evasion techniques to hide its malicious indicators.</li><li>Has evolved across versions to achieve effectiveness of their attack.</li><li>Employs a highly modular plugin framework to selectively infect targeted endpoints.</li><li>Most importantly, it deploys RAT malware to ransack the endpoint, followed by deployment of ransomware to either extort money or burn infrastructure of targeted entities. </li></ul><h2 id="distribution-vector">Distribution vector </h2><p>The first version of CRAT has been known to be distributed via <a href="https://otx.alienvault.com/pulse/5e8edeec3c1da2b0f89707aa">malicious HWPs</a>. The HWPs masquerade as a COVID-19 themed document pertaining to an infectious disease management support group from South Korea. The HWPs consisted of an exploit for <a href="https://nvd.nist.gov/vuln/detail/CVE-2017-8291">CVE-2017-8291</a> used to activate malicious shellcode. The shellcode would then download and execute CRATv1 on the infected endpoint.<br> The distribution vector of the new version of CRAT (v2) is currently unknown. However, it is highly likely the attackers may have re-used a maldoc-based infection vector to spread CRATv2 as well.</p><figure class="kg-card kg-image-card"><img src="https://blog.talosintelligence.com/content/images/-h9Kz3ui3m1M/X6tpxmMR-RI/AAAAAAAAAL8/Nd9KzZbz3U0zee59xb36ikPhuDnu8f3ZgCLcBGAsYHQ/s16000/image4.png" class="kg-image" alt loading="lazy"></figure><p>Malicious HWP document - downloader for CRATv1</p><figure class="kg-card kg-image-card"><img src="https://1.bp.blogspot.com/-OtYC72gqxa8/X6tqCf1CtuI/AAAAAAAAAMI/X8oXoLjTmHkh_Y_mXV9_nmNlZx8g1qayACLcBGAsYHQ/s16000/image14.png" class="kg-image" alt loading="lazy"></figure><p> Shellcode executing PowerShell to download and activate CRATv1 via regsvr32.</p><h2 id="obfuscation">Obfuscation</h2><p>Before we analyze the attack it is important to understand the extensive measures taken by the attackers to conceal the RAT executables. The RAT is highly obfuscated in terms of:</p><ul><li>String Obfuscation: Almost all of the strings are obfuscated using a four-byte XOR key and subsequent base64 encoding. String obfuscations are used to thwart string-based static malware detection signatures.</li><li>API Resolution: All APIs used are resolved dynamically. The process of selecting the API to call is also a cumbersome algorithm (overkill) involving index tables, switch cases and API name deobfuscation. This technique is used to make analysis cumbersome for an analyst by hiding API call sequences.</li><li>Runtime Code Patching: The malware consists of a select few instrumental subroutines that are decoded, executed and then patched/encoded again (during runtime). On-the-fly patching of subroutines/functions may be aimed towards evading detection mechanisms that scan process memory to identify malicious strings and code.</li></ul><h3 id="why-the-occurrence-of-multiple-obfuscations">WHY THE OCCURRENCE OF MULTIPLE OBFUSCATIONS?</h3><p>The use of multiple obfuscations signifies the attackers confidence in selective obfuscation rather than the use of packers as a means of evasion. Many detection systems look for the presence of a packer using techniques such as entropy analysis, Import API analyses, etc. Selective obfuscation of code and strings prevents these systems from detecting the malware solely on the basis of the obfuscations.<br><br>It is also likely that this group of attackers employs a modular build system that obfuscates different aspects of malware. Different modules can thus be used in conjunction to produce a combination of obfuscations for the same malware.</p><h3 id="masquerading-as-legitimate">MASQUERADING AS LEGITIMATE</h3><p>The attackers have also used file names and export API names in the CRAT DLLs to masquerade the RAT as a benign application's library. Some examples of the exported function names are:</p><ul><li>ConfigChrome</li><li>DownloadChrome</li><li>GetChromeVersion</li><li>InstallChrome</li><li>UninstallChrome</li><li>UpdateChrome</li><li>InstallFirefox</li><li>SaveMicrosoftEdge</li><li>ExtractMicrosoftEdge</li><li>ExtractMicrosoftWord</li></ul><h2 id="anti-infection-checks">Anti-Infection Checks</h2><p>The implant performs checks to verify its execution on an allowed endpoint. The implant uses a variety of criteria to identify a blocklisted system by utilizing:</p><ul><li>Process name blocklists</li><li>Network adapter name blocklists</li><li>MAC Address blocklists</li><li>Analysis tool names blocklists</li></ul><p>The implant also checks for indicators that the process is being debugged (including CheckRemoteDebuggerPresent). If the infected endpoint fails any of the anti-infection checks, the implant quits execution. The blocklists are provided at the end of this post.</p><h2 id="finding-the-right-process-sweet-home-alabama">Finding the right process — Sweet Home Alabama</h2><p>The CRAT DLL ensures it resides in the desired process on the infected endpoint. Depending on the MS Windows OS version, it will inject and reflectively-load itself into the right process. The processes trojanized by CRAT v2 are:<br><br>Win10:</p><ul><li>sihost.exe</li><li>taskhostw.exe</li><li>ApplicationFrameHost.exe</li><li>svchost.exe</li></ul><p><br>Win 6.1:</p><ul><li>dwm.exe</li><li>taskhostex.exe</li><li>svchost.exe</li><li>notepad.exe</li></ul><p>Other Win 6.x:</p><ul><li>dwm.exe</li><li>taskhost.exe</li><li>svchost.exe</li><li>notepad.exe</li></ul><p>OS agnostic injection fall back process: explorer.exe</p><h3 id="communication-mechanisms">Communication mechanisms</h3><p>Before detailing CRAT's capabilities, it is important to illustrate the communication mechanism used by CRAT to talk to its C2 servers.<br>Both versions of CRAT use HTTP to communicate with the C2 servers. The data sent to the C2 is in the form of URL-encoded form data and may consist of 3 types of requests:</p><ul><li>Login: Login/check-in with the C2 to register the infected endpoint.</li><li>Question: Request a command code to execute a malicious RAT functionality on the endpoint.</li><li>Answer: Respond to the C2's command with the output of the executed RAT functionality.</li></ul><p>The HTTP GET/POST requests are sent to the C2 URLs appended with a 'timestamp' value: http(s)://&lt;C2_url&gt;?ts=&lt;epoch_time&gt;_&lt;rand&gt;</p><h3 id="exfiltration-mechanism">EXFILTRATION MECHANISM</h3><p>CRAT uses the following algorithm to encode the data before sending it out via the HTTP POST request:</p><ol><li>Generate a random four-byte value (DWORD).</li><li>XOR the data to be exfiltrated with the DWORD.</li><li>Base64-encode (std alphabet) the XOR key and the XORed data to achieve the final form of the data.</li><li>Exfiltrate the encoded data as URL encoded form data.</li></ol><p>The format of the response to the C2 is:<br><br>code=answer&amp;token=&lt;token&gt;&amp;content=&lt;base64_encoded_data&gt;frags=&lt;fragment_count&gt;&amp;limit=10<br><br>Where:</p><ul><li>code = type of the communication being done. Values include:</li><li>login = login request to C2 from implant</li><li>question = request command to be processed</li><li>answer = response to command executed</li><li>token = a token value sent by the C2 as a response to the login (session identifier)</li><li>content = the actual base64-encoded + XORed data being exfiltrated</li><li>frags = number of fragments of data being POSTed to the C2</li><li>limit = retry count in case of failure to send</li></ul><p>The C2 server will respond with a JSON containing command codes and supporting data to be recognized by the implant to execute a corresponding RAT capability.</p><h3 id="rat-capabilities">RAT capabilities</h3><p>The CRAT malware consists of multiple RAT capabilities that allow it to be highly versatile and dangerous. CRAT executes these capabilities by receiving command codes and corresponding data from the C2 in the form of JSONs communicated over HTTP.</p><p>Apart from the introduction of a wide variety of new RAT capabilities in CRAT version 2 (vs CRAT version 1), the biggest update is the spin-off of key RAT capabilities into plugin modules. These plugins (DLLs) are downloaded on-the-fly by CRATv2 and injected into specific processes running on the infected endpoint.<br><br>The following capabilities have been observed in CRATv2.</p><h3 id="collect-system-information">COLLECT SYSTEM INFORMATION</h3><p>Collects the following system information:</p><ul><li>Installed AntiVirus software names</li><li>Installed FirewallProduct names</li><li>Domain Names:</li><li>Netbios domain name</li><li>DNS domain name</li><li>Domain forest name</li><li>File version number from DLLs embedded Version Information</li><li>Path to the system folder</li><li>Flag if the current user has administrative privileges</li></ul><figure class="kg-card kg-image-card"><img src="https://blog.talosintelligence.com/content/images/-wn7wj6dkFyk/X6tqReWKm2I/AAAAAAAAAMM/usz_TFbK7W4p-1j1dH5tUnbiszg35KT5wCLcBGAsYHQ/s16000/image13.png" class="kg-image" alt loading="lazy"></figure><p>Implant checking for ADMIN privileges </p><ul><li>TCP/IP enabled MAC Addresses using WMI query:</li></ul><!--kg-card-begin: html--><blockquote><pre><code>wmic PATH Win32_NetworkAdapterConfiguration WHERE IPEnabled=TRUE GET MACAddress.</code></pre>

</blockquote><!--kg-card-end: html--><figure class="kg-card kg-image-card"><img src="https://blog.talosintelligence.com/content/images/-yIYbxDBUqq4/X6tr3TzYNII/AAAAAAAAAMk/kkLFMZARLVYYLU1QK11rNwIKnf-fHtXPQCLcBGAsYHQ/s16000/image8.png" class="kg-image" alt loading="lazy"></figure><p>Implant replacing strings in the output of the WMIC command to obtain only the MAC Addresses</p><h3 id="enumerate-drives-and-gather-file-size-information">ENUMERATE DRIVES AND GATHER FILE SIZE INFORMATION</h3><p>Collects size information about all files and folders on the infected endpoint except:</p><ul><li>Recycle Bin</li><li>%windir%</li></ul><p>The data gathered is arranged into a specific format:</p><pre><code>
&lt;Drive_Letter&gt;\t&lt;Disk_Type&gt;\r\n
[DIR]\t&lt;Folder_Path&gt;
\t&lt;File_Name&gt;&lt;spaces&gt;&lt;size_in_bytes&gt; bytes</code></pre><p>Where Disk Type =</p><ul><li>Local Disk</li><li>CD ROM</li><li>Removable Disk</li><li>Remote Disk</li><li>RAM Disk</li><li>Unknown Disk</li></ul><p>E.g.<br></p><figure class="kg-card kg-image-card"><img src="https://1.bp.blogspot.com/-iA2cNRnA7pw/X6tsA9fou1I/AAAAAAAAAMo/YogoBKRXqPQMUeVsS-RkwgA1uH_UBxRSwCLcBGAsYHQ/s16000/image2.png" class="kg-image" alt loading="lazy"></figure><p>Snippet of drive and file size information gathered by the implant.</p><h3 id="enumerate-drives-using-dir">ENUMERATE DRIVES USING DIR</h3><p>Enumerate drives using the 'dir' command:<br>cmd /c "dir &lt;drive_name&gt; /s &gt;&gt; %temp%\&lt;custom_prefix&gt;error.log"<br><br>Where custom_prefix = file name prefix specified by the C2<br>E.g. cmd /c "dir C:\ /s &gt;&gt; C:\Users\&lt;username&gt;\AppData\Local\Temp\BLAHerror.log"</p><figure class="kg-card kg-image-card"><img src="https://1.bp.blogspot.com/-YVWpOzgd884/X6tsha4OTXI/AAAAAAAAAM0/ZKAVYo6drb0uOh5ifantTIdsTfjgYr7AQCLcBGAsYHQ/s16000/image3.png" class="kg-image" alt loading="lazy"></figure><p>Implant constructing the dir cmd to be executed.</p><p>The log file is then rolled up into a ".rar" archive using a pre-existing installation of a RAR archiver program using the command:<br>&lt;rar_utility&gt; a -k -r -s -ibck -m5 &lt;output_rar_filepath&gt; &lt;input_log_filepath&gt;<br><br>The RAR archive is then read to memory and exfiltrated to the C2, followed by deletion of the .rar and log files.</p><h3 id="read-and-write-file">READ AND WRITE FILE</h3><p>CRAT has the capability to read the contents of the file specified by the C2 and exfiltrate these to the C2. The write file capability consists of writing the data received from the C2 to a temporary file in the %temp% folder: %temp%\&lt;temp_name&gt;.tmp<br><br>The implant will also timestomp the tmp file to either a hardcoded value or copy the filetimes from a system file such as %windir%\system32\user32.dll.</p><h3 id="execute-commands">EXECUTE COMMANDS</h3><p>Execute commands with two variations of the functionality:</p><ul><li>Simply execute the command on the endpoint.</li><li>Execute a command and send output of command to C2.</li></ul><h3 id="reverse-shell">REVERSE SHELL</h3><p>Open up a reverse command shell for arbitrary command execution using Windows pipes.</p><h3 id="set-query-times">SET QUERY TIMES</h3><p>This capability allows the C2 to specify delays between consecutive command queries from the implant to the C2 for:</p><ul><li>Receiving commands from the C2.</li><li>Receiving arbitrary commands to execute in the reverse shell.</li></ul><h3 id="steal-browser-passwords-chromepass">STEAL BROWSER PASSWORDS: CHROMEPASS</h3><p>Execute chromepass.exe with the /stext switch to obtain the usernames and passwords stored in Google Chrome. The credentials are dumped to a text file which is read and exfiltrated to the C2.<br><br>Command format: chromepass.exe /stext &lt;output_filepath&gt;</p><h3 id="file-explorer">FILE EXPLORER</h3><p>CRAT contains a RAT functionality that implements a custom-built File Explorer sub-module.<br><br>The C2 specifies the parent functionality command code and an additional sub-command code to specify the sub-capability to be executed in the file explorer. The file explorer sub-module contains the following sub-capabilities:</p><ul><li>Gather free and total disk space information for each drive on the system.</li><li>Enumerate file listings (recursively) in a given directory.</li><li>Move files across locations.</li><li>Find and remove files and folders.</li><li>Create RAR archive files from files specified.</li><li>Create RAR archives and exfiltrate file contents to C2.</li><li>Execute arbitrary commands specified by C2 on the endpoint.</li></ul><h3 id="plugin-administration">PLUGIN ADMINISTRATION</h3><p>Downloads and installs a malicious plugin from the C2. The plugin is placed in a standard directory specified by the C2, and locations may include:</p><ul><li>%PUBLIC%\Documents</li><li>%USERPROFILE%\Downloads</li><li>%PUBLIC%\Music</li><li>%PUBLIC%\Pictures</li><li>%PUBLIC%\Videos</li><li>%windir%</li><li>%windir%\system32</li><li>%APPDATA% (%USERPROFILE%\AppData\Roaming)<br><br>CRAT can also administer the malicious plugins by performing the following actions:<br></li><li>For EXE based plugins:</li><li>Execute plugin as an independent process.</li><li>For DLL plugins:</li><li>Activate using regsvr32.exe</li><li>Activate using rundll32.exe</li><li>Reflective load into its own process</li><li>Reflectively load into a target process</li><li>Delete files related to plugins</li></ul><h3 id="uninstall-crat">UNINSTALL CRAT</h3><p>The RAT comes with an uninstallation routine built into it as well. This routine can be triggered in response to a command issued by the C2 or if the endpoint fails its anti-infection checks. The uninstallation routine consists of:</p><ul><li>Deleting .exe and .lnk files associated with the RAT.</li><li>Sending the uninstall flag to the communications pipe.</li><li>Delete associated plugin files (EXEs and DLLs) from the endpoint.</li></ul><h3 id="additional-capabilities">ADDITIONAL CAPABILITIES</h3><p>In addition to the RAT capabilities, CRAT also consists of additional functionalities that are implemented as part of independent threads in the infected process:</p><ul><li>Send Heartbeat Thread: This will send four bytes of data periodically to a named pipe.</li><li>Upload File Thread: This thread will periodically send the contents of a file to the C2 server over HTTP. The contents of either the Keylogger or Clipboard monitor files are read and sent to the C2.</li><li>Upload Screen Thread: This thread reads and sends the contents of another file to the C2 server over HTTP. Most likely the content of screenshots taken by the Screen Recorder plugin.</li></ul><h2 id="cratv2-plugins">CRATv2 plugins</h2><p>With version 2, the attackers have evolved CRAT into a modular RAT with the ability to download and activate additional malicious plugins (DLLs) on the infected endpoint. The first version of CRAT had these plugin capabilities implemented within the RAT. Cisco Talos has discovered these capabilities so far:</p><ul><li>Screen capture plugins</li><li>Clipboard monitor plugins</li><li>Keylogger plugins</li><li>Ransomware</li></ul><h3 id="screen-capture-plugin">SCREEN CAPTURE PLUGIN</h3><p>The screen capture plugin will capture the current foreground window (every second or so) and save the screenshot to a .tmp file in a hardcoded directory. The screenshot is saved in TIFF file format. Since TIFF is widely supported by scanning and faxing systems, the malicious screenshots may be mis-identified as document scans by a forensic analyst instead of malicious screen captures.<br><br>The saved screenshot file is also XORed using a variable length XOR key hardcoded into the plugin as saved to add a further layer of obfuscation.<br></p><figure class="kg-card kg-image-card"><img src="https://1.bp.blogspot.com/-27v585afNOo/X6tstAEV9xI/AAAAAAAAAM4/0ewCWQVMl5MdXUL6Jg9NLz75kOUO3sM8wCLcBGAsYHQ/s16000/image15.png" class="kg-image" alt loading="lazy"></figure><p>Sample screenshot captured by the plugin</p><h3 id="clipboard-monitor-plugin">CLIPBOARD MONITOR PLUGIN</h3><p>This plugin will read the clipboard data and write the contents of the clipboard to a log file. The format of the clipboard data logged is:<br>\r\n[YYYY-mm-DD HH:MM:SS]\r\n&lt;clipboard_data&gt;<br><br>The data is stored in a seemingly benign file location such as: %localappdata%\Google\Chrome\Application\Update.chk<br></p><figure class="kg-card kg-image-card"><img src="https://1.bp.blogspot.com/-eda6Dh6iPPg/X6ts3NvltfI/AAAAAAAAANA/P3ulOqV16Ecqj6JJkqnqdAtGAzCCo5qegCLcBGAsYHQ/s16000/image1.png" class="kg-image" alt loading="lazy"></figure><p>Clipboard monitor implant getting clipboard data.</p><h3 id="keylogger-plugin">KEYLOGGER PLUGIN</h3><p>The keylogger plugin monitors the state of all the alphanumeric keys pressed. It also logs the following keys:</p><ul><li>Ctrl</li><li>Shift</li><li>Menu</li><li>Alt</li><li>Enter</li><li>Tab</li><li>Backspace</li><li>Capslock</li><li>Delete</li><li>End</li><li>Home</li><li>Up</li><li>Down</li><li>Left</li><li>Right</li><li>Function keys</li></ul><p>The keystrokes are logged to another seemingly benign file on the endpoint such as:<br>%localappdata%\Google\Chrome\Application\Update.cert<br><br>The format of the log file is:<br>\r\n[&lt;Window_Name&gt; - YYYY-mm-DD HH:MM:SS]\r\n&lt;keylogger_data&gt;<br>\r\n[YYYY-mm-DD HH:MM:SS]\r\n&lt;keylogger_data&gt;<br><br>Example:<br><br>[Untitled - Notepad - 2020-06-05 01:02:03]<br>These keystrokes are being recorded[ENTER]<br><br>[Blah - 2020-06-05 01:02:03]<br>[CTRL] + c<br></p><figure class="kg-card kg-image-card"><img src="https://1.bp.blogspot.com/-Xx7g3OAby78/X6ttDh8pOrI/AAAAAAAAANI/96g7KrdtagQ2Hd_4O9blDZufgJnxPchdACLcBGAsYHQ/s16000/image5.png" class="kg-image" alt loading="lazy"></figure><p>Keywords used to log keys being pressed.</p><h2 id="ransomware-plugin-hansom">Ransomware plugin — Hansom</h2><p>We also discovered a ransomware plugin that encrypts specific file extensions on the infected endpoint. Traditionally ransomware encrypts files on the infected endpoint using a combination of asymmetric and symmetric encryption. This plugin is different.<br><br>It locks files into individually created archives using randomly generated passwords. The passwords are then encrypted using an embedded public key (part of an embedded pub certificate). Once the infected user gets access to the private key (after the ransom has been paid), the private key can be plugged into the accompanying decryptor .exe that decrypts the archive passwords and subsequently unpacks the original files from the archive.<br><br>The plugin performs the following housekeeping actions before it begins encrypting files on the endpoint:</p><ul><li>Terminate specific application processes.</li><li>Suppress Windows Defender notifications by setting registry value: HKLM\SOFTWARE\Policies\Microsoft\Windows Defender\UX Configuration | Notification_Suppress = 0x1</li><li>Terminate Windows Defender process "MsMpEng.exe" specifically.</li><li>Disable task manager via registry: HKCU\Software\Microsoft\Windows\CurrentVersion\Policies\System | DisableTaskMgr = 0x1</li><li>Setup persistence for self via registry and regsvr32: HKCU\Software\Microsoft\Windows\CurrentVersion\Run | HANSOM = regsvr32.exe /s &lt;filepath_to_ransomware_dll&gt;</li></ul><h3 id="targeted-file-extensions">TARGETED FILE EXTENSIONS</h3><p>The following file extensions are targeted by the ransomware plugin. The ransomware excludes certain file and folder names from encryption to ensure the stability of the Operating System (listed in the IOCs section).</p><ul><li>doc,docx,ppt,pps,pptx,xls,xlsx,mdb,chm</li><li>txt,log,pdf,hwp,hst,ods,odt,rtf,csv,hwpml,show,eps,epub,eml</li><li>htm,html,js,css,vbs,php,jsp,xml,asp,aspx</li><li>asm,java,c,cpp,cs,h,m,3ds,max,dwg,cad</li><li>rar,zip,alz,tar,tgz,gz,7z,iso,bz2,bzip2,arj,arc,ace,xz,lz</li><li>jpg,jpeg,png,bmp,tiff,gif,psd,ai,fla,ppm,xbm</li><li>exe,dll,jar,war,bat,py,pl,apk,msi,ocx,cmd</li><li>db,frm,myd,myi,mdf,sdf,dbf,sql,sqlite3,dat</li><li>cer,crt,der,p7b,p7c,p12,pfx,pem,csr,key</li><li>ini,inf,reg,bak,ldf,config,conf,cka,au3</li></ul><h3 id="ransomware-encryption-algorithm">RANSOMWARE ENCRYPTION ALGORITHM</h3><p>The plugin uses the following sequence of steps to encrypt/lock files on the infected endpoint:</p><ol><li>Create a password by generating random 0x48 bytes and base64 encoding them. Then take the target file and add it to a password-protected RAR archive using command:<br>"C:\Users\&lt;username&gt;\AppData\Roaming\WinRar\Rar.exe" a -y -ep -k -r -s -ibck -df -m0 -hp&lt;password&gt; -ri1:250 "&lt;target_filepath&gt;.rar" "&lt;target_filepath&gt;" <br>This command also deletes the target file once it has been added to the archive. <br></li><li>After the target file has been archived, the random password used for the creation of the RAR archive is encrypted using an embedded RSA public key.<br></li><li>The encrypted password blob is appended to the archive along with a marker indicating that the file has been encrypted successfully. The marker used is:<br>48 61 6E 73 6F 6D 32 30 30 38 20 20 07 29 Hansom2008 .)<br></li><li>Then the RAR archive is renamed (via file move) to the name of the original file.<br>E.g. C:\blah\abc.txt.rar is renamed back to C:\blah\abc.txt<br></li><li>A ransom note is dropped in the target file's directory named HANSOM_README.txt informing the user of the ransomware infection.<br></li></ol><figure class="kg-card kg-image-card"><img src="https://1.bp.blogspot.com/-86Sj1_F9lKY/X6ttNct-UQI/AAAAAAAAANQ/u_M0kKNle3kqkoliB201VGJ31Ss1ewTvACLcBGAsYHQ/s16000/image6.png" family. This section provides an overview of these components.</p><h3 id="exe-loaders">EXE LOADERS</h3><p>The EXE based loader serves two primary purposes:</p><ol><li>Establish persistence for itself using a scheduled task on the infected endpoint. Usually scheduled to run once every few minutes. The loader may also obtain persistence using a registry RUN key.</li><li>Perform anti-infection checks and inject the CRAT DLL into a target process.</li><li>The CRAT DLL isn't dropped or downloaded by the .exe based loader to the endpoint.</li><li>It simply reads the DLL from a hardcoded filepath, un-XORs it using a 4-byte key and the resulting DLL is reflectively injected into a specified target process.</li></ol><h3 id="variant-2">VARIANT #2</h3><p><strong>The latest variant of the EXE loader contains the following capabilities/upgrades:</strong></p><ol><li>Establish persistence for itself using two mechanisms:</li><li>Create and run a malicious service on the endpoint to run itself.</li><li>Create a malicious LNK for itself which is persisted via a malicious RUN registry key value.</li><li>Read an implant (most likely CRATv2) into memory and reflectively load it into the loader's process space.</li></ol><h3 id="dll-loaders">DLL LOADERS</h3><p>Apart from EXE based loaders, this CRAT also uses DLL based loaders. The DLL loaders carry out the same function of decoding the CRAT DLL and reflectively injecting it into a specified process.<br><br>The differences are in the persistence techniques used:</p><ul><li>The DLL loader copies itself to a specified directory and creates a LNK that activates the loader via rundll32.exe</li><li>The LNK file is then persisted across reboots via the registry RUN key: HKCU\Software\Microsoft\windows\CurrentVersion\Run | &lt;Value_Name&gt; = &lt;LNK_filepath&gt;</li></ul><h3 id="variant-2-1">VARIANT #2</h3><p><strong>Cisco Talos discovered another variant of the DLL-based loader that acts as a packager as well. Depending on the option specified, the loader will either:</strong></p><ul><li>Activate CRAT:</li><li>Activate unpacked CRAT DLL:</li><li>Reflectively inject an existing CRAT DLL into a hardcoded process.</li><li>Unpack and activate:</li><li>Extract components of the attack from a RAR archive and execute an accompanying EXE file (Most likely used to load the CRAT DLL).</li><li>The extraction isn't carried out directly. This variant will create a VBS file to run rar.exe to extract CRAT components. Syntax used:</li></ul><blockquote>Set s = Wscript.CreateObject("Wscript.Shell"): s.Run """%s"" x -y -p%s ""%s"" ""%s""", 0, TRUE: s.Run """%s\%s""", 0: Set s = Nothing</blockquote><blockquote>E.g.</blockquote><blockquote>Set s = Wscript.CreateObject("Wscript.Shell"): s.Run """C:\Users\&lt;username&gt;\AppData\Roaming\WinRar\Rar.exe"" x -y -p[password] ""&lt;path_to_rar_file&gt;\&lt;filename&gt;.rar"" ""&lt;target_dir&gt;""", 0, TRUE: s.Run """&lt;target_dir&gt;\&lt;extracted_exe&gt;.exe""", 0: Set s = Nothing<br></blockquote><ul><li>Just like the previous DLL loader, this variant will create an LNK file to run the VBS script upon reboot by setting up registry key: HKCU\Software\Microsoft\windows\CurrentVersion\Run | &lt;Value_Name&gt; = &lt;LNK_filepath&gt;<br></li><li>Package CRAT: This variant of the loader also has the capability to create an archive from a specified file on disk. This functionality may be used to create deployment packages for another part of the infection chain. Syntax used:</li></ul><blockquote>"&lt;path_to&gt;\Rar.exe" a -k -r -s -ibck -m5 -df -ep -hp[password] "&lt;filepath_of_rar_archive_to_be_created&gt;.rar" "&lt;file_to_be_archived&gt;"</blockquote><figure class="kg-card kg-image-card"><img src="https://1.bp.blogspot.com/-9wRTc4wJbx0/X6ttg6BKDaI/AAAAAAAAANg/6gj8mopTtoUsl02rbIuhfDVHdw7_pHj-gCLcBGAsYHQ/s16000/image12.png" class="kg-image" alt loading="lazy"></figure><p>Infection chain of CRATv2</p><h2 id="evolution-timeline-of-crat-and-components">Evolution Timeline of CRAT and Components</h2><p>The earliest versions of CRAT discovered were compiled in April 2020. Although <a href="https://blog.alyac.co.kr/2896">prior reporting</a> has linked <a href="https://blog.alyac.co.kr/3018">CRAT to Lazarus</a>, there has been no solid evidence to back these claims. There are however some aspects of the attack that resemble Lazarus' practices:</p><ul><li><strong>Code Reuse</strong> - CRAT uses the same HTTP wrapper library used by other Lazarus implants such as Wild Positron and Rising Sun. Although this may be a weak attribution link, it does make sense for the group to reuse code they've had success with in the past.<br></li><li><strong>Functionality Duplication</strong> - Lazarus tends to mix-and-match RAT capabilities and peripheral functionalities to stitch together variations of the same implant. That is, different obfuscations mechanisms, communication techniques and RAT capabilities are assembled together to create variations of the implant. This may cause duplication of certain functionalities. In the case of CRAT, the occurrence of the same capability like 'execution of arbitrary commands' multiple times as independent RAT commands and also as part of the File Explorer sub-module indicates a factory-based build approach for the implants. Although a weak link, this process is still in sync with Lazarus' build paradigms.</li></ul><p></p><ul><li><strong>Ransomware Plugin</strong> - Lazarus has been known to distribute <a href="https://securelist.com/lazarus-on-the-hunt-for-big-game/97757/">ransomware</a> as a means of sponsoring their malicious activities. The BTC addresses used by the ransomware plugin in the case of CRAT (Hansom Ransomware) do not currently hold any funds. Thus, it is unclear whether this module is in fact ransomware or a pseudo-ransomware (posing as ransomware but meant to burn endpoints).<br></li><li><strong>Wordpress C2s</strong> - CRAT uses Wordpress based websites to act as its C2 servers. This practice is in sync with Lazarus' TTPs of <a href="https://blog.alyac.co.kr/3018">using Wordpress websites</a> to place C2 modules that act as intermediaries between the implants and real C2 servers.<br></li><li><strong>Past Attribution</strong> - CRAT has been <a href="https://blog.alyac.co.kr/2896">attributed to Lazarus</a> in the past by the security community<br></li><li><strong>Targets</strong> - Based on the lures observed so far by Cisco Talos, CRAT aims to infect Korean-speaking entities. This may also be an indicator of Lazarus targeting specific parties of interest.</li></ul><p></p><h2 id="conclusion">Conclusion</h2><p>This investigation illustrates the continued use and evolution of the CRAT implant. CRAT started out as a nascent RAT with limited capabilities, which evolved to introduce a wide variety of malicious capabilities including a full-fledged File Explorer.<br><br>The latest iteration of CRAT now has the capability to download and deploy arbitrary plugins on the infected endpoint. These plugins have capabilities ranging from spying to encrypting and holding the user's data hostage. The ransomware plugin in particular has the potential to cause extensive loss of services, time and money to affected organizations.<br><br>The continued evolution of loaders used to deploy CRAT also indicates that the attackers are actively developing novel ways of infecting their targets. The extensive use of obfuscation, patching and anti-infection techniques indicates that the attackers have taken great care towards evading detection systems. Thus, while static and network-based detection is important, it should be complimented with system behavior analysis and endpoint protections.</p><p></p><h2 id="coverage">Coverage</h2><p>Ways our customers can detect and block this threat are listed below.</p><figure class="kg-card kg-image-card"><a href="https://1.bp.blogspot.com/-kf1lAu7_46E/X6tt343lUTI/AAAAAAAAAN0/J6lWZvjpyzszGQ2zQXfj040F8Uqqx1LyQCLcBGAsYHQ/s320/image11.png"><img src="https://1.bp.blogspot.com/-kf1lAu7_46E/X6tt343lUTI/AAAAAAAAAN0/J6lWZvjpyzszGQ2zQXfj040F8Uqqx1LyQCLcBGAsYHQ/s16000/image11.png" class="kg-image" alt loading="lazy"></a></figure><p>Advanced Malware Protection (<a href="https://www.cisco.com/c/en/us/products/security/advanced-malware-protection">AMP</a>) is ideally suited to prevent the execution of the malware detailed in this post.
