    <h1 class="text-center">BeaverTail and OtterCookie evolve with a new Javascript module</h1>

    <div class="text-center m-1">
        <div class="post-author">
            <span>By </span>
                <a href="https://blog.talosintelligence.com/author/vanja-svajcer/">Vanja Svajcer</a>,

        <time class="post-datetime" datetime="October 16, 2025 06:00">
            Thursday, October 16, 2025 06:00

        <div class="m-3">

    <section class="post-content-wrapper mt-5">
        <div class="post-content">
            <ul><li>Cisco Talos has uncovered a new attack linked to <a href="https://blog.talosintelligence.com/python-version-of-golangghost-rat/" rel="noreferrer noopener"><u>Famous Chollima</u></a>, a threat group aligned with North Korea (DPRK). This group is known for impersonating hiring organizations to target job seekers, tricking them into installing information-stealing malware to obtain cryptocurrency and user credentials.&nbsp;</li><li>In this incident, although the organization was not directly targeted, one of its systems was compromised-likely because a user was deceived by a fake job offer and installed a trojanized Node.js application called "Chessfi."&nbsp;</li><li>The malicious software was distributed via a Node.js package named "node-nvm-ssh" on the official NPM repository.</li><li>Famous Chollima often uses two malicious tools, BeaverTail and OtterCookie, which started as separate but complementary programs. Recent campaigns have seen their functions merging, and Talos has identified a new module for keylogging and taking screenshots.&nbsp;</li><li>While searching for related threats, Talos also found a malicious VS Code extension containing BeaverTail and OtterCookie code. Although attribution to Famous Chollima is not certain, this suggests the group may be testing new methods for delivering their malware.&nbsp;</li></ul><h2 id="introduction">Introduction&nbsp;</h2><p>In a <a href="https://blog.talosintelligence.com/python-version-of-golangghost-rat/" rel="noreferrer noopener"><u>previous Cisco Talos blog post</u></a>, we described one side of the <a href="https://unit42.paloaltonetworks.com/north-korean-threat-actors-lure-tech-job-seekers-as-fake-recruiters/" rel="noreferrer noopener"><u>Contagious Interview</u></a> (Deceptive Development) campaigns, where the threat actor utilized fake employment websites, ClickFix social engineering techniques and payload variants of credential and cryptocurrency remote access trojans (RATs) known as GolangGhost and PylangGhost.&nbsp;</p><p>&nbsp;Talos is actively monitoring other clusters of these campaigns, which are attributed to the threat actor group Famous Chollima, a subgroup of Lazarus, and aligned with the economic interests of DPRK. This post discusses some of the tactics, techniques and procedures (TTPs) and changes in tooling developed over time by another large cluster of Contagious Interview activities. These campaigns center around tools known as BeaverTail and OtterCookie.&nbsp;&nbsp;</p><p>&nbsp;Famous Chollima frequently uses BeaverTail and OtterCookie, with many individual sub-clusters of activities installing <a href="https://any.run/cybersecurity-blog/invisibleferret-malware-analysis/" rel="noreferrer noopener"><u>InvisibleFerret</u></a>, a Python based modular payload. Although BeaverTail and OtterCookie originated as separate-but-complementary entities, their functionality in some recent campaigns started to merge, along with the inclusion of new functional OtterCookie modules.&nbsp;</p><p>&nbsp;Talos detected a Famous Chollima campaign in an organization headquartered in Sri Lanka. The organization was not deliberately targeted by the attackers, but it had one of the systems on the network infected. It is likely that a user fell for a fake job offer instructing them to install a trojanised Node.js application called Chessfi as a part of a fake job interview process.&nbsp;</p><p>Once Talos conducted the initial analysis, we realized that the tools used to conduct it had characteristics of BeaverTail and of OtterCookie, blurring the distinction between the two. The code also contained some additional functionality we have not previously encountered.&nbsp;&nbsp;</p><h2 id="beavertail-and-ottercookie-combine">BeaverTail and OtterCookie combine&nbsp;</h2><p>This blog focuses on OtterCookie modules and will not provide a deep dive into well-known BeaverTail and OtterCookie functionality. While some of these modules are already known, at least one was not previously documented. The examples we show are already deobfuscated, and with the help of an LLM, the function and variable names are replaced by names that correspond to their actual functionality.&nbsp;</p><h3 id="keylogging-and-screenshotting-module">Keylogging and screenshotting module&nbsp;</h3><p>Talos encountered a keylogging and screenshotting module in this campaign that has not been previously documented. We were able to find earlier OtterCookie samples containing the module that were uploaded to VirusTotal in April 2025.&nbsp;&nbsp;</p><p>The keylogging module uses the packages “node-global-key-listener" for keylogging, “screenshot-desktop” for taking desktop screenshots and “sharp” for converting the captured screenshots into web-friendly image formats.&nbsp;</p><p>The module configures the packages to listen for keystrokes and periodically takes a screenshot of the current desktop session to upload them to the OtterCookie command and control (C2) server.&nbsp;</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.talosintelligence.com/content/images/2025/10/image002.png" class="kg-image" alt="" loading="lazy" width="664" height="664" srcset="https://blog.talosintelligence.com/content/images/size/w600/2025/10/image002.png 600w, https://blog.talosintelligence.com/content/images/2025/10/image002.png 664w"><figcaption><span style="white-space: pre-wrap;">Figure 1. The keylogger listens for the keyboard and mouse key presses and saves them into a file.&nbsp;</span></figcaption></figure><p>The keystrokes are saved in the user’s temporary sub-folder windows-cache with the file name “1.tmp” and screenshots are saved in the same sub-folder with the file name “2.jpeg”. While the keylogger runs in a loop and flushes the buffer every second, a screenshot is taken every four seconds.&nbsp;&nbsp;</p><p>Talos also discovered one instance of the module where the clipboard monitoring was included in the module code, extending its functionality to stealing clipboard content.&nbsp;&nbsp;</p><p>The keylogging data and the captured screenshots are uploaded to the OtterCookie C2 server at a specific TCP port 1478, using the URL “hxxp[://]172[.]86[.]88[.]188:1478/upload”.&nbsp;</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.talosintelligence.com/content/images/2025/10/image004.png" class="kg-image" alt="" loading="lazy" width="823" height="766" srcset="https://blog.talosintelligence.com/content/images/size/w600/2025/10/image004.png 600w, https://blog.talosintelligence.com/content/images/2025/10/image004.png 823w" sizes="(min-width: 720px) 720px"><figcaption><span style="white-space: pre-wrap;">Figure 2. Keystrokes saved as “1.tmp” and screenshots as “2.jpeg”, then uploaded to C2 server.&nbsp;</span></figcaption></figure><h3 id="ottercookie-vs-code-extension">OtterCookie VS Code extension&nbsp;</h3><p>During the search for similar samples on VirusTotal, Talos discovered a recently-uploaded VS Code extension, which may attempt to run OtterCookie if installed in the victim’s editor environment. The extension is a fake employment onboarding helper, supposedly allowing the user to track and manage candidate tests.&nbsp;</p><p>While Talos cannot attribute this VS Code extension to Famous Chollima with high confidence, this may indicate that the threat actor is experimenting with different delivery vectors. The extension could also be a result of experimentation from another actor, possibly even a researcher, who is not associated with Famous Chollima, as this stands out from their usual TTPs.&nbsp;</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.talosintelligence.com/content/images/2025/10/image006.png" class="kg-image" alt="" loading="lazy" width="722" height="240" srcset="https://blog.talosintelligence.com/content/images/size/w600/2025/10/image006.png 600w, https://blog.talosintelligence.com/content/images/2025/10/image006.png 722w" sizes="(min-width: 720px) 720px"><figcaption><span style="white-space: pre-wrap;">Figure 3. VS Code extension configuration pretends to be Mercer Onboarding Helper but contains OtterCookie code.&nbsp;</span></figcaption></figure><h3 id="other-ottercookie-modules">Other OtterCookie modules&nbsp;</h3><p>The OtterCookie section of code starts with the definition of a JSON object that contains configuration values such as unique campaign ID and C2 server IP address. The OtterCookie portion of the code constructs additional modules from strings, which are executed as child processes. In the attack we analyzed, we observed three modules, but we also found one additional module while hunting for similar samples in our repositories and on VirusTotal.&nbsp;&nbsp;</p><p><strong>Remote shell module&nbsp;</strong></p><p>The first module is fundamental for OtterCookie and begins with the detection of the infected system platform and a virtual machine check, followed by reporting the collected user and host information to the OtterCookie C2 server.&nbsp;</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.talosintelligence.com/content/images/2025/10/image008.png" class="kg-image" alt="" loading="lazy" width="753" height="333" srcset="https://blog.talosintelligence.com/content/images/size/w600/2025/10/image008.png 600w, https://blog.talosintelligence.com/content/images/2025/10/image008.png 753w" sizes="(min-width: 720px) 720px"><figcaption><span style="white-space: pre-wrap;">Figure 4. Main Ottercookie module starts with machine checking and includes virtual machines check.&nbsp;</span></figcaption></figure><p>Once the system information is submitted, the module installs the “socket.io-client” package, which is used to connect to a specific port on the OtterCookie C2 server to wait for the commands and execute them in a loop. socket.io-client first uses HTTP and then switches to WebSocket protocol to communicate with the server, which we observed listening on the TCP port 1418.&nbsp;&nbsp;</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.talosintelligence.com/content/images/2025/10/image010.png" class="kg-image" alt="" loading="lazy" width="905" height="195" srcset="https://blog.talosintelligence.com/content/images/size/w600/2025/10/image010.png 600w, https://blog.talosintelligence.com/content/images/2025/10/image010.png 905w" sizes="(min-width: 720px) 720px"><figcaption><span style="white-space: pre-wrap;">Figure 5. socket.io-client package used for communication with C2 server.&nbsp;</span></figcaption></figure><p>Finally, depending on the operating system, this module periodically checks the clipboard content using the commands “pbpaste” on macOS or “powershell Get-Clipboard” on Windows. It sends the clipboard content to the C2 server URL specifically used for logging OtterCookie activities at “hxxp[://]172[.]86[.]88[.]188/api/service/makelog”.&nbsp;</p><p><strong>File uploading module&nbsp;</strong></p><p>This module enumerates all drives and traverses the file system in order to find files to be uploaded to the OtterCookie C2 IP address at a specific port and URL (in this case, “hxxp[://]172[.]86[.]88[.]188:1476/upload”).&nbsp;&nbsp;</p><p>This module contains a list of folder and file names to be excluded from the search, and another list with target file name extensions and file name search patterns to select files to be uploaded.&nbsp;&nbsp;</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.talosintelligence.com/content/images/2025/10/image012.png" class="kg-image" alt="" loading="lazy" width="942" height="309" srcset="https://blog.talosintelligence.com/content/images/size/w600/2025/10/image012.png 600w, https://blog.talosintelligence.com/content/images/2025/10/image012.png 942w" sizes="(min-width: 720px) 720px"><figcaption><span style="white-space: pre-wrap;">Figure 6. The list of excluded folders and patterns for files uploaded to C2.&nbsp;</span></figcaption></figure><p>The “interesting” file list contains the following search patterns:&nbsp;<br>&nbsp;<br><mark>"*.env*", "*metamask*", "*phantom*", "*bitcoin*", "*btc*", "*Trust*", "*phrase*", "*secret*", "*phase*", "*credential", "*profile*", "*account*", "*mnemonic*", "*seed*", "*recovery*", "*backup*", "*address*", "*keypair*", "*wallet*", "*my*", "*screenshot*", "*.doc", "*.docx", "*.pdf", "*.md", "*.rtf", "*.odt", "*.xls", "*.xlsx", "*.txt", "*.ini", "*.secret", "*.json", "*.ts", "*.js", "*.csv"</mark>&nbsp;</p><p><strong>Cryptocurrency extensions stealer module&nbsp;</strong></p><p>While not present in the campaign Talos analyzed, this module was found while looking for similar files on VirusTotal. In addition to the targeting of cryptocurrency browser extensions by the BeaverTail code, this OtterCookie module targets extensions from a list that partially overlaps with the list of cryptocurrency wallet extensions from the BeaverTail part of the payload.&nbsp;</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.talosintelligence.com/content/images/2025/10/image014.jpg" class="kg-image" alt="" loading="lazy" width="1228" height="1429" srcset="https://blog.talosintelligence.com/content/images/size/w600/2025/10/image014.jpg 600w, https://blog.talosintelligence.com/content/images/size/w1000/2025/10/image014.jpg 1000w, https://blog.talosintelligence.com/content/images/2025/10/image014.jpg 1228w" sizes="(min-width: 720px) 720px"><figcaption><span style="white-space: pre-wrap;">Table 1. Cryptocurrency modules targeted by OtterCookie.&nbsp;</span></figcaption></figure><p>The cryptocurrency module targets Google Chrome and Brave browsers. If any extensions are found in any of the browser profiles, the extension files as well as the saved Login and Web data are uploaded to a C2 server URL. In the discovered sample Talos found, the uploading C2 URL was “hxxp[://]138[.]201[.]50[.]5:5961/upload”.&nbsp;</p><h2 id="ottercookie-evolution">OtterCookie evolution&nbsp;</h2><p>OtterCookie malware samples were <a href="https://archive.ph/WWAAG" rel="noreferrer noopener"><u>first observed</u></a> by NTT Security Holdings around November 2024, leading to a blog article published in December 2024. However, it is believed that the malware has been in use since approximately September 2024. The motivation for using the name OtterCookie seems to come from the early samples that used content of HTTP response cookies to transfer the malicious code executed by the response handler. This remote code loading functionality evolved over time to include additional functionality.&nbsp;&nbsp;</p><p>However, in April 2025, Talos started seeing additional modules included within the OtterCookie code and the usage of the C2 server, mostly for downloading a simple OtterCookie configuration and uploading stolen data.&nbsp;</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.talosintelligence.com/content/images/2025/10/data-src-image-d9a8b0af-ca69-4005-b48c-439709a8a759.png" class="kg-image" alt="" loading="lazy" width="936" height="759" srcset="https://blog.talosintelligence.com/content/images/size/w600/2025/10/data-src-image-d9a8b0af-ca69-4005-b48c-439709a8a759.png 600w, https://blog.talosintelligence.com/content/images/2025/10/data-src-image-d9a8b0af-ca69-4005-b48c-439709a8a759.png 936w" sizes="(min-width: 720px) 720px"><figcaption><span style="white-space: pre-wrap;">Figure 7. OtterCookie modules evolution timeline.</span></figcaption></figure><p>OtterCookie evolved from the initial basic data-gathering capabilities to more modular design for data theft and remote command execution techniques. The modules are stored within OtterCookie strings and executed on the fly.&nbsp;</p><p>The earliest versions, corresponding to what NTT researchers <a href="https://archive.ph/vE8C8" rel="noreferrer noopener"><u>refer to as v1</u></a>, contain code for remote command execution (RCE) and use a socket.IO package to communicate with a C2 server. Over time, OtterCookie modules evolved by adding code to steal and upload files, with the end goal of stealing cryptocurrency wallets from a list of hardcoded browser extensions and saved browser credentials. Targeted browsers include Brave, Google Chrome, Opera and Mozilla Firefox.&nbsp;</p><p>The next iteration, referred to as v2, included a clipboard stealing code using the Clipboardy package to send clipboard contents to the remote server. This version also handles the loading of Javascript code from the server slightly differently. Instead of evaluating the returned header cookie as v1, the server generates an error which gets handled by the error handler on the client side. The error handler simply passes the error response data to the eval function, where it gets executed. The loader code is small and easy to miss, and along with the risk of false positive detections, this may be why the detection of the OtterCookie loaders on VirusTotal is not very successful.&nbsp;</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.talosintelligence.com/content/images/2025/10/image016.png" class="kg-image" alt="" loading="lazy" width="866" height="446" srcset="https://blog.talosintelligence.com/content/images/size/w600/2025/10/image016.png 600w, https://blog.talosintelligence.com/content/images/2025/10/image016.png 866w" sizes="(min-width: 720px) 720px"><figcaption><span style="white-space: pre-wrap;">Figure 8. C2 server generates an error but the code is still executed by OtterCookie.&nbsp;</span></figcaption></figure><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.talosintelligence.com/content/images/2025/10/image018.png" class="kg-image" alt="" loading="lazy" width="849" height="246" srcset="https://blog.talosintelligence.com/content/images/size/w600/2025/10/image018.png 600w, https://blog.talosintelligence.com/content/images/2025/10/image018.png 849w" sizes="(min-width: 720px) 720px"><figcaption><span style="white-space: pre-wrap;">Figure 9. OtterCookie loader error handler evaluates the response data.&nbsp;</span></figcaption></figure><p>The v3 variant, observed in February 2025, includes a function to send specific files (documents, image files and cryptocurrency-related files) to the C2 server. OtterCookie v4, observed since April 2025, includes a virtual environment detection code to help attackers discern logs from sandbox environments from those of actual infections, indicating a focus on evading analysis. The code also contains some anti-debugging and anti-logging functionality.&nbsp;&nbsp;</p><p>The v4 variant improves on the previous version’s code and updates the clipboard content-stealing method. It no longer uses the Clipboardy library and instead it uses standard macOS or Windows commands for retrieving clipboard content.&nbsp;</p><p>It is important to note that over time the difference between BeaverTail and OtterCookie became blurred and in some attacks their code was merged into a single tool. &nbsp;</p><h3 id="ottercookie-v5">OtterCookie v5&nbsp;</h3><p>The campaign Talos observed in August 2025 uses the most recent version of OtterCookie, which we call v5, demonstrated by the addition of a keylogging module. The keylogging module contains code to capture screenshots, which are uploaded to the C2 server together with keyboard keystrokes.&nbsp;</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.talosintelligence.com/content/images/2025/10/data-src-image-0f582648-aa48-4b35-9bdb-2e1c219a174a.png" class="kg-image" alt="" loading="lazy" width="936" height="519" srcset="https://blog.talosintelligence.com/content/images/size/w600/2025/10/data-src-image-0f582648-aa48-4b35-9bdb-2e1c219a174a.png 600w, https://blog.talosintelligence.com/content/images/2025/10/data-src-image-0f582648-aa48-4b35-9bdb-2e1c219a174a.png 936w" sizes="(min-width: 720px) 720px"><figcaption><span style="white-space: pre-wrap;">Figure 10. Node-nvm-ssh infection path.&nbsp;</span></figcaption></figure><p>The initial infection vector was a modified Chessfi application hosted on Bitbucket. ChessFi is a web3-based multiplayer chess platform where players can challenge each other and bet cryptocurrency on the outcome of their matches. The choice of a cryptocurrency-related application to lure victims is consistent with previous reporting of <a href="https://www.welivesecurity.com/en/eset-research/deceptivedevelopment-targets-freelance-developers/" rel="noreferrer noopener"><u>Famous Chollima targeting</u></a>.&nbsp;&nbsp;</p><p>The first sign of the attack was the user installing the source code of the application. Based on the folder name of the project, we assess with moderate confidence that the victim was approached by the threat actor through the freelance marketplace site Fiverr, which is consistent with the previous reporting. While hunting for similar samples we have also discovered code repositories that were uploaded for the victim as attachments to Discord conversations.&nbsp;&nbsp;&nbsp;</p><p>The infection process started with the victim running Git to clone the repository:&nbsp;</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.talosintelligence.com/content/images/2025/10/image021.png" class="kg-image" alt="" loading="lazy" width="544" height="50"><figcaption><span style="white-space: pre-wrap;">Figure 11. The initial infection vector.&nbsp;</span></figcaption></figure><p>The Development section of the application’s readme document gives instructions to developers on how to install and run the project. After cloning the repository, it states that the users should run npm install to install dependencies, which, in this campaign, also included a malicious npm package named “node-nvm-ssh".&nbsp;</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.talosintelligence.com/content/images/2025/10/image023.png" class="kg-image" alt="" loading="lazy" width="560" height="257"><figcaption><span style="white-space: pre-wrap;">Figure 12. Modified application installation steps.&nbsp;</span></figcaption></figure><p>During the installation of dependencies, the malicious package is downloaded from the repository and installed. The npm installer parses the package.json file of the malicious package and finds instructions to run commands after the installation. This is executed by parsing the “postinstall” value of the JSON object named “scripts”. At the first glance, it seems like the postinstall scripts are there to run tests, transpile TypeScript files to Java script and possibly run other test scripts.&nbsp;</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.talosintelligence.com/content/images/2025/10/image025.png" class="kg-image" alt="" loading="lazy" width="838" height="240" srcset="https://blog.talosintelligence.com/content/images/size/w600/2025/10/image025.png 600w, https://blog.talosintelligence.com/content/images/2025/10/image025.png 838w" sizes="(min-width: 720px) 720px"><figcaption><span style="white-space: pre-wrap;">Figure 13. Malicious package.json file contains the instruction that will cause the malicious code to run.&nbsp;</span></figcaption></figure><p>However, the package.json module installation instruction “npm run skip” causes npm to call the command node test/fixtures/eval specified in the value “skip”. The default node.js loading conventions will try loading a number of file names if none of them are specifically mentioned, one of them being index.js.&nbsp;</p><p>The test/fixtures/eval/index.js content contains code to spawn a child process using the file “test/fixtures/eval/node_modules/file15.js”.&nbsp;</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.talosintelligence.com/content/images/2025/10/image027.png" class="kg-image" alt="" loading="lazy" width="764" height="108" srcset="https://blog.talosintelligence.com/content/images/size/w600/2025/10/image027.png 600w, https://blog.talosintelligence.com/content/images/2025/10/image027.png 764w" sizes="(min-width: 720px) 720px"><figcaption><span style="white-space: pre-wrap;">Figure 14. index.js spawning a child process to execute file15.js.&nbsp;</span></figcaption></figure><p>Eventually, file15.js loads the file test.list, which is the final payload. This somewhat complex process to reach the payload code makes it quite difficult for an unsuspecting software engineer to discover that the installation of the Chessfi application will eventually lead to execution of malicious code.&nbsp;&nbsp;&nbsp;</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.talosintelligence.com/content/images/2025/10/image029.png" class="kg-image" alt="" loading="lazy" width="619" height="198" srcset="https://blog.talosintelligence.com/content/images/size/w600/2025/10/image029.png 600w, https://blog.talosintelligence.com/content/images/2025/10/image029.png 619w"><figcaption><span style="white-space: pre-wrap;">Figure 15. file15.js reads and calls eval on the content of the file test.list.</span></figcaption></figure><p>With test.list we have finally reached the last piece of the puzzle of how the malicious code is run. The test.list file is over 100KB long and obfuscated using <a href="http://obfuscator.io/" rel="noreferrer noopener"><u>Obfuscator.io</u></a>. Thankfully, the obfuscation in this case is not configured to make the analysis very difficult and with the help of the deobfuscator and an LLM, Talos was able to deobfuscate most of its functionality, revealing a combination of BeaverTail and OtterCookie.&nbsp;</p><h2 id="standard-beavertail-functionality">Standard BeaverTail functionality&nbsp;</h2><p>There seem to be two distinguishable parts in the code. The first is associated with BeaverTail, including enumeration of various browser profiles and extensions as well as the download of a Python distribution and Python client payload from the C2 server “23.227.202[.]244” using the common BeaverTail/InvisibleFerret TCP port 1224. The second part of the code is associated with OtterCookie.&nbsp;</p><p>The BeaverTail portion starts with a function that disables the console logging, moving toward loading the required modules and calling functions in order to steal data from a list of browser extensions, cryptocurrency wallets and browser credentials storage.&nbsp;&nbsp;&nbsp;</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.talosintelligence.com/content/images/2025/10/data-src-image-81d62751-2b12-4642-89e6-ed6ce535b2cf.png" class="kg-image" alt="" loading="lazy" width="883" height="936" srcset="https://blog.talosintelligence.com/content/images/size/w600/2025/10/data-src-image-81d62751-2b12-4642-89e6-ed6ce535b2cf.png 600w, https://blog.talosintelligence.com/content/images/2025/10/data-src-image-81d62751-2b12-4642-89e6-ed6ce535b2cf.png 883w" sizes="(min-width: 720px) 720px"><figcaption><span style="white-space: pre-wrap;">Table 2. Targeted BeaverTail cryptocurrency browser extensions.</span></figcaption></figure><h2 id="beavertail-evolution">BeaverTail evolution&nbsp;</h2><p>BeaverTail has been observed since at least May 2023, and originally was a relatively small downloader component, designed to be included with Node.js based Javascript applications. BeaverTail was also used in supply chain attacks affecting packages in the <a href="https://www.aikido.dev/blog/malware-hiding-in-plain-sight-spying-on-north-korean-hackers" rel="noreferrer noopener"><u>NPM package repository</u></a>, which was extensively covered in the <a href="https://socket.dev/blog/contagious-interview-campaign-escalates-67-malicious-npm-packages" rel="noreferrer noopener"><u>previous research</u></a> and it is outside of the scope of this post.&nbsp;</p><p>From the beginning, BeaverTail supported Windows, Linux and macOS, taking advantage of the fact that Node.js applications can be run on different operating system platforms.&nbsp;&nbsp;</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.talosintelligence.com/content/images/2025/10/image032.png" class="kg-image" alt="" loading="lazy" width="1123" height="501" srcset="https://blog.talosintelligence.com/content/images/size/w600/2025/10/image032.png 600w, https://blog.talosintelligence.com/content/images/size/w1000/2025/10/image032.png 1000w, https://blog.talosintelligence.com/content/images/2025/10/image032.png 1123w" sizes="(min-width: 720px) 720px"><figcaption><span style="white-space: pre-wrap;">Figure 16. Early BeaverTail OS platform check.&nbsp;</span></figcaption></figure><p>The other major functionalities within BeaverTail are the download of InvisibleFerret Python stealer payload modules and installation of a remote access module, typically an AnyDesk client, which would allow the attacker to take over the control of the infected machine remotely. Information stealing and remote access have remained recurring BeaverTail operational techniques over time.&nbsp;</p><p>Soon after the initial samples were discovered in June 2023, BeaverTail started to use simple base64 encoding of strings and renaming of variables to make the detection and analysis more difficult. This also included a scheme used to encode the C2 URL as a shuffled string whose slices are base64 decoded individually and then concatenated in a correct order to generate the final URL.&nbsp;&nbsp;</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.talosintelligence.com/content/images/2025/10/image034.png" class="kg-image" alt="" loading="lazy" width="850" height="217" srcset="https://blog.talosintelligence.com/content/images/size/w600/2025/10/image034.png 600w, https://blog.talosintelligence.com/content/images/2025/10/image034.png 850w" sizes="(min-width: 720px) 720px"><figcaption><span style="white-space: pre-wrap;">Figure 17. C2 URL encoding scheme used from early BeaverTail variants until the present.&nbsp;</span></figcaption></figure><p>Although BeaverTail is typically written in Javascript, Talos has also discovered several Javascript C2 IP server addresses. These were shared with C++ compiled binary variants created <a href="https://blog.talosintelligence.com/lazarus-magicrat/" rel="noreferrer noopener"><u>with the help of the Qt framework</u></a>.&nbsp;</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.talosintelligence.com/content/images/2025/10/image036.png" class="kg-image" alt="" loading="lazy" width="703" height="468" srcset="https://blog.talosintelligence.com/content/images/size/w600/2025/10/image036.png 600w, https://blog.talosintelligence.com/content/images/2025/10/image036.png 703w"><figcaption><span style="white-space: pre-wrap;">Figure 18. Qt based BeaverTail setting a Qthread parameters.&nbsp;</span></figcaption></figure><p>From the early beginnings in mid-2023, to the last quarter of 2024. BeaverTail C2 URL patterns stabilized around the most commonly-used TCP ports 1224 and 1244, rather than the port 3306 used by early variants. It seems that the threat actors quickly realized that most Windows installations do not come with preinstalled Python interpreters as Linux distributions and macOS. To tackle this issue, they included code which installs a Python distribution, typically from the “/pdown” URL path, required to run Python InvisibleFerret modules. This TTP remains until today.&nbsp;</p><p>In terms of detection evasion, Famous Chollima are using several methods to obfuscate code, most frequently utilzing different configurations of the free Javascript tool <a href="http://obfuscator.io/" rel="noreferrer noopener"><u>Obfuscator.io</u></a> which does make the analysis and especially detection of the malicious code more challenging.&nbsp;&nbsp;</p><p>In addition to obfuscating the Javascript code they also regularly use various modes of XOR-based obfuscation of downloaded modules. XORed Python InvisibleFerret modules start with a unique user based string assignment followed by a reversed base64 encoded string, which contains the final Python module’s code that can also be XORed for obfuscation.&nbsp;</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.talosintelligence.com/content/images/2025/10/image038.png" class="kg-image" alt="" loading="lazy" width="557" height="505"><figcaption><span style="white-space: pre-wrap;">Figure 19. A typical InvisibleFerret self-decoding Python module.&nbsp;</span></figcaption></figure><p>Thankfully, by using the combination of a <a href="https://obf-io.deobfuscate.io/" rel="noreferrer noopener"><u>deobfuscating tool</u></a> and an LLM to rename the variables and base64 decode encoded strings it is possible to analyse new samples with relative ease. However, the operational tempo of groups attributed to Famous Chollima is high and the detection of completely new samples and code on VirusTotal remains unreliable, allowing threat actors enough time to successfully attack some victims.&nbsp;</p><h2 id="beavertail-ottercookie-and-invisibleferret-functional-overlaps">BeaverTail, OtterCookie and InvisibleFerret functional overlaps</h2><p>All additional modules present in OtterCookie code correspond well to the functionality that is traditionally associated with InvisibleFerret and its Python-based modules, as well as some parts of the BeaverTail code.
