            <h2 id="summary-and-background">Summary and&nbsp;background</h2><p>Google Cloud Platform (GCP)&nbsp;<a href="https://cloud.google.com/functions#documentation" rel="noopener noreferrer nofollow">Cloud Functions</a>&nbsp;are event-triggered, serverless functions that automatically scale and execute code in response to specific events like Hypertext Transfer Protocol (HTTP) requests or data changes. Tenable Research published an&nbsp;<a href="https://www.tenable.com/blog/confusedfunction-a-privilege-escalation-vulnerability-impacting-gcp-cloud-functions" rel="noopener noreferrer nofollow">article</a>&nbsp;discussing a vulnerability they discovered within GCP’s Cloud Functions serverless compute service and its Cloud Build continuous integration and continuous delivery or deployment (CI/CD) pipeline service.</p><p>“When a GCP user creates or updates a Cloud Function, a multi-step backend process is triggered,” Tenable author Liv Matan writes.&nbsp;“This process, among other things, attaches a&nbsp;<a href="https://cloud.google.com/build/docs/cloud-build-service-account" rel="noopener noreferrer nofollow">default Cloud Build service account</a>&nbsp;to the Cloud Build instance that is created as part of the function’s deployment.”&nbsp;This default Cloud Build Service&nbsp;Account&nbsp;(SA) previously gave users excessive&nbsp;Cloud&nbsp;Function permissions. An attacker who&nbsp;has gained the ability to create or update a&nbsp;cloud&nbsp;function could utilize the function’s deployment process to escalate privileges to the default Cloud Build service account or assign a higher privileged SA. Google has since partially addressed Tenable’s discovery to ensure the default Cloud Build service account no longer provides users with excessive permissions.</p><p>Based on Tenable’s research, Cisco Talos conducted a series of offensive tests within Cisco’s Google Cloud Platform (GCP) to identify additional threats that may affect customer&nbsp;environments.</p><p>During its research, Talos discovered that the technique&nbsp;Tenable&nbsp;identified could be adapted to perform other malicious activities. id="conclusion-and-defense-summary">Conclusion and Defense Summary</h2><h3 id="google%E2%80%99s-response">Google’s Response</h3><p>As described in Tenable’s article, Google responded to their research by creating a remediation patch. This update altered the default behavior of Cloud Build and the default Cloud Build&nbsp;SA. Additionally, new organization policies were released to give organizations full control over which&nbsp;SA&nbsp;Cloud Build uses by default. While Google has implemented this remediation, Cloud Build services can still be used to execute non-privileged commands as a means of enumerating an environment.</p><h3 id="mitigation-summary">Mitigation Summary</h3><p>The most effective mitigation strategy to protect your environment from&nbsp;similar threat actor&nbsp;behavior is to ensure that all&nbsp;SAs&nbsp;within your cloud environment adhere to the principle of least privilege and that no legacy cloud&nbsp;SAs&nbsp;are still in use. Ensure that all cloud services and dependencies are up&nbsp;to&nbsp;date with the latest security patches. If legacy&nbsp;SAs&nbsp;are present, replace them with least-privilege&nbsp;SAs.&nbsp;</p><p>Additionally, users with access to Cloud Functions should not have IAM permissions to the services included in the function’s orchestration.</p><h2 id="threat-hunting-recommendations">Threat Hunting Recommendations</h2><ol><li><strong>Audit and&nbsp;monitor&nbsp;SA&nbsp;permissions</strong>: Regularly audit and monitor&nbsp;SA&nbsp;permissions, with a particular focus on the default Cloud Build&nbsp;SA. Adhere to the principle of least privilege by removing any excessive permissions that are not essential for the&nbsp;SA’s&nbsp;operations.</li><li><strong>Alert&nbsp;setup for Cloud Functions</strong>: Establish alerts for any unusual or unauthorized creation or modification of Cloud Functions. Identify potentially malicious activities where an attacker may be attempting to exploit function deployments for privilege escalation.</li><li><strong>Inspect&nbsp;network&nbsp;traffic</strong>: Analyze network traffic for unusual patterns or connections that might indicate data exfiltration attempts. Pay attention to data being sent to unknown or unauthorized external endpoints, such as those using Ngrok or similar tunneling services.</li><li><strong>Verify NPM&nbsp;package&nbsp;integrity</strong>: Ensure the integrity and authenticity of NPM packages used within Cloud Functions. Prevent the execution of malicious scripts embedded in package.json files that could facilitate environment enumeration or other malicious activities.</li><li><strong>Detect&nbsp;environment&nbsp;enumeration</strong>: Detect and respond to signs of environment enumeration, such as ICMP discovery or system information gathering.</li></ol>
