    <h1 class="text-center">“Good enough” emulation: Fuzzing a single thread to uncover vulnerabilities</h1>

    <div class="text-center m-1">
        <div class="post-author">
            <span>By </span>
                <a href="https://blog.talosintelligence.com/author/kelly/">Kelly Patterson</a>
        </div>

        <br/>
        <time class="post-datetime" datetime="February 18, 2026 06:00">
            Wednesday, February 18, 2026 06:00
        </time>

        <div class="m-3">
                        <a href="https://blog.talosintelligence.com/category/vulnerability-deep-dive/" class="category primary-category">
                            Vulnerability Deep Dive
                        </a>
                            </div>
    </div>

    <section class="post-content-wrapper mt-5">
        <div class="post-content" id="post-content">
            <ul><li>A Cisco Talos researcher worked around the limitations of hardware-level Code Read-out Protection (RDP) on the&nbsp;Socomec&nbsp;DIRIS M-70 gateway by pivoting from physical debugging to a "good enough" emulation approach.&nbsp;</li><li>By focusing on emulating only the single thread responsible for Modbus protocol handling rather than the entire system, the author&nbsp;demonstrates&nbsp;how a streamlined emulation strategy can effectively surface vulnerabilities in complex industrial&nbsp;Internet of Things (IoT)&nbsp;devices.&nbsp;</li><li>The post highlights the integration of the Unicorn Engine and AFL for coverage-guided fuzzing, as well as the use of the&nbsp;Qiling&nbsp;framework to visualize code coverage and perform root cause analysis on crashes.&nbsp;</li><li>This research led to the discovery of six CVEs related to denial-of-service vulnerabilities, all of which have been patched by the manufacturer through Cisco’s Coordinated Disclosure Policy.&nbsp;</li></ul><hr><p>This blog describes&nbsp;efforts at emulating functionality of the&nbsp;Socomec&nbsp;DIRIS M-70 gateway to discover vulnerabilities. In vulnerability research, knowing which tool to use for the job at hand is crucial. This post will highlight multiple emulation tools and approaches used, detail the benefits and drawbacks of each, and&nbsp;reveal&nbsp;how a "good enough" approach can really pay off.</p><h2 id="project-background">Project background&nbsp;</h2><p>The M-70 gateway&nbsp;facilitates&nbsp;data communication over both RS485 and Ethernet networks, supporting a wide array of industrial communication protocols, including Modbus RTU, Modbus TCP, BACnet IP, and SNMP (v1, v2,&nbsp;and&nbsp;v3). This gateway is vital for energy management in sectors like critical infrastructure, data centers, healthcare, and the general energy sector. However, as an industrial&nbsp;Internet-of-Things&nbsp;(IIoT) device, vulnerabilities in the M-70 or similar gateways can lead to severe consequences, including operational disruption, financial losses, and manipulation of industrial processes. These risks are severe, especially in critical infrastructure where a compromised gateway could lead to widespread outages or equipment damage.</p><p>This large attack surface, the impact of vulnerabilities, and the fact that&nbsp;the M-70 gateway runs&nbsp;the real-time&nbsp;operating system&nbsp;(RTOS)&nbsp;µC/OS-III,&nbsp;made it&nbsp;an attractive research target.&nbsp;There was an expectation that prior familiarity with this RTOS, gained through&nbsp;<a href="https://blog.talosintelligence.com/fuzzing-uc-os-protocol-stacks/" rel="noreferrer noopener"><u>previous work</u></a>, would offer an advantage in understanding the device’s intricacies.</p><h2 id="why-emulate-the-debugging-roadblock">Why emulate? The&nbsp;debugging roadblock&nbsp;</h2><p>Having insight into the system is critical to performing root cause analysis of any&nbsp;discovered&nbsp;vulnerabilities. Ideally, one would have real hardware and the ability to debug the software running on that hardware.&nbsp;The presence of an unpopulated JTAG header on the board was an exciting&nbsp;initial&nbsp;find.</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.talosintelligence.com/content/images/2026/02/jtag_header_fig1.png" class="kg-image" alt="" loading="lazy" width="298" height="308"><figcaption><span style="white-space: pre-wrap;">Figure 1. Unpopulated JTAG header.</span></figcaption></figure><p>However,&nbsp;the presence of a&nbsp;JTAG&nbsp;header&nbsp;does not always&nbsp;guarantee&nbsp;debug access. There are a variety of reasons for&nbsp;this, but&nbsp;in the case of the M-70 gateway,&nbsp;code read-out protection (RDP)&nbsp;Level 1&nbsp;is enabled. This is a feature of STM32 microcontrollers, which provides flash memory protection. There are&nbsp;three&nbsp;possible levels&nbsp;(0&nbsp;–&nbsp;2) of this protection. Level&nbsp;1&nbsp;prevents flash memory reads while debugger access is detected&nbsp;(e.g.,&nbsp;JTAG). When attached via JTAG, no access to Flash memory&nbsp;is&nbsp;permitted,&nbsp;essentially preventing&nbsp;debugging of&nbsp;the running software.&nbsp;The intention behind this&nbsp;protection&nbsp;is to prevent third parties (like&nbsp;myself) from dumping the contents of flash via JTAG.</p><p>This&nbsp;was&nbsp;bad news.&nbsp;It was not possible to step through the code processing malicious network messages to&nbsp;determine&nbsp;the cause of device disruption.&nbsp;The&nbsp;address for the&nbsp;<code>$pc</code>&nbsp;register&nbsp;(see Figure 2)&nbsp;indicates&nbsp;that the MCU has entered a core lock-up state.</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.talosintelligence.com/content/images/2026/02/rdp_output_fig2.png" class="kg-image" alt="" loading="lazy" width="536" height="532"><figcaption><span style="white-space: pre-wrap;">Figure 2. RDP Level 1 debug output.</span></figcaption></figure><p>In this project, two significant opportunities arose&nbsp;regarding&nbsp;code and memory access.&nbsp;First, an unencrypted firmware update file was available, providing the code that would be written to flash and&nbsp;eliminating&nbsp;the need to read it directly from memory.&nbsp;The second is that the ability to access SRAM while a debugger is attached is allowed with RDP&nbsp;Level 1 enabled&nbsp;(see Figure 2).&nbsp;This made it&nbsp;feasible&nbsp;to dump the contents of SRAM during the device’s execution&nbsp;and capture a&nbsp;snapshot of dynamic data.</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.talosintelligence.com/content/images/2026/02/dumpram_fig3.png" class="kg-image" alt="" loading="lazy" width="555" height="87"><figcaption><span style="white-space: pre-wrap;">Figure 3. RAM dumping script.</span></figcaption></figure><p>While it was not possible to have fine-grained control over the&nbsp;processor’s&nbsp;state when dumping the SRAM contents, some influence could be exerted (e.g., opening a TCP connection with the device before dumping the SRAM contents). The objects and data created&nbsp;as a result of&nbsp;this connection would be present when the CPU was halted for the SRAM dump.</p><h2 id="emulating-with-unicorn">Emulating with Unicorn&nbsp;</h2><p>Emulation is one solution to this inability to debug the software natively.&nbsp;If the processing code of interest can be emulated, it is possible to gain visibility into the effects of a malicious message on the state of the M-70. When emulating software,&nbsp;it’s important to recognize that the emulated code might not behave exactly like it would on the physical device. Full system emulation aims to mitigate this by mimicking device behavior as closely as possible, but it requires deep knowledge of system internals and significant development to accurately emulate peripherals.&nbsp;The focus for this project was on vulnerabilities within the Modbus protocol handling code, which ran in a single thread of the M-70 application.&nbsp;Rather than spending the time required for full system emulation, the decision was made to emulate only the Modbus thread.&nbsp;Admittedly, emulating this single thread would&nbsp;not be true to the device’s real-world operation. However,&nbsp;this&nbsp;deliberate time trade-off&nbsp;was made with the&nbsp;hope that it would still be “good enough” to find vulnerabilities in the Modbus protocol handling code.&nbsp;</p><p>The first step in this process involved&nbsp;utilizing&nbsp;the&nbsp;<a href="https://github.com/unicorn-engine/unicorn" rel="noreferrer noopener"><u>Unicorn Engine</u></a>, a powerful CPU emulation framework supporting various architectures. It&nbsp;provided&nbsp;the core capability to run the Modbus&nbsp;thread’s&nbsp;code in a controlled software environment where I could then inspect the system state when processing network data.&nbsp;</p><p>The emulator was implemented with an entry point in the Modbus processing thread, positioned after network data had been received.&nbsp;Before emulating this&nbsp;code, the argument registers&nbsp;<code>$r2</code>&nbsp;and&nbsp;<code>$r3</code>&nbsp;which originally&nbsp;contained&nbsp;a pointer&nbsp;to network data and&nbsp;its&nbsp;length&nbsp;were&nbsp;modified&nbsp;to reference data originating from the emulator, along with&nbsp;it’s&nbsp;corresponding length. Once the argument registers were updated,&nbsp;emulation could begin and continue&nbsp;until that thread&nbsp;returned from the message processing function.</p><h2 id="the-need-to-fuzz">The need to fuzz&nbsp;</h2><p>Manual inspection of network processing code is sometimes sufficient; however, this Modbus thread supports over 700 unique message types, defined by supported register values and something referred to as service identifier. The combination of these two values within a Modbus message influenced the code path of data processing, and with so many code paths to investigate, automation was clearly necessary.</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.talosintelligence.com/content/images/2026/02/binja_register_vals_fig4.png" class="kg-image" alt="" loading="lazy" width="1347" height="347" srcset="https://blog.talosintelligence.com/content/images/size/w600/2026/02/binja_register_vals_fig4.png 600w, https://blog.talosintelligence.com/content/images/size/w1000/2026/02/binja_register_vals_fig4.png 1000w, https://blog.talosintelligence.com/content/images/2026/02/binja_register_vals_fig4.png 1347w" sizes="(min-width: 720px) 720px"><figcaption><span style="white-space: pre-wrap;">Figure 4. Register values and service&nbsp;identifier.</span></figcaption></figure><p>Unicorn’s AFL integration made it simple to fuzz&nbsp;using the&nbsp;emulator, automatically exploring these many execution paths.&nbsp;AFL uses coverage-guided test case generation to maximize the number of different code paths explored.&nbsp;This tool&nbsp;provided precisely the type of automation that was necessary. It was simple integrating AFL fuzzing into&nbsp;the&nbsp;Unicorn&nbsp;script,&nbsp;requiring only the addition of the&nbsp;<code>place_input_callback</code>&nbsp;function and&nbsp;a&nbsp;call&nbsp;to&nbsp;<code>unicorn_afl_fuzz</code>&nbsp;(see Figure 5).&nbsp;</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.talosintelligence.com/content/images/2026/02/unicorn_afl_fig5.png" class="kg-image" alt="" loading="lazy" width="848" height="402" srcset="https://blog.talosintelligence.com/content/images/size/w600/2026/02/unicorn_afl_fig5.png 600w, https://blog.talosintelligence.com/content/images/2026/02/unicorn_afl_fig5.png 848w" sizes="(min-width: 720px) 720px"><figcaption><span style="white-space: pre-wrap;">Figure 5. Unicorn AFL integration.</span></figcaption></figure><h3 id="triage-and-debugging">Triage and debugging&nbsp;</h3><p>With fuzzing&nbsp;came crashes, and the next step&nbsp;was&nbsp;to triage those crashes to perform root cause analysis. Typically, a debugger would be the go-to tool for this job; however,&nbsp;because execution was performed through emulation, GDB&nbsp;didn’t&nbsp;"just work" out of the box.&nbsp;A tool compatible with the Unicorn framework’s internal CPU representation was&nbsp;required. Conveniently, a tool called&nbsp;<a href="https://github.com/bet4it/udbserver" rel="noreferrer noopener"><u>udbserver</u></a>&nbsp;does exactly that.&nbsp;Udbserver&nbsp;is a plugin for the Unicorn engine that enables&nbsp;debugging of&nbsp;Unicorn emulated code within GDB. This tool worked as advertised and allowed&nbsp;remote&nbsp;GDB&nbsp;connections&nbsp;to&nbsp;the&nbsp;emulated code. There is only one line&nbsp;required&nbsp;to add&nbsp;udbserver&nbsp;support to a unicorn emulator: <code>udbserver.udbserver(mu,1234,0x80fede0)</code> before calling <code>emu_start</code>.</p><h2 id="qiling-framework-adding-code-coverage-to-the-mix">Qiling&nbsp;framework: Adding code coverage to the mix&nbsp;</h2><p>Observing code coverage visually is another important part of any fuzzing campaign. It helps&nbsp;identify&nbsp;unexplored paths and provides insights for root cause analysis by comparing coverage between test cases.&nbsp;The need for this feature prompted an investigation into the&nbsp;Qiling&nbsp;framework.&nbsp;Described as a full system emulator, it also supports debugging and code coverage output.&nbsp;Could&nbsp;Qiling&nbsp;be used to emulate&nbsp;only&nbsp;a&nbsp;single thread&nbsp;rather than the&nbsp;whole system?&nbsp;It would be wonderful to&nbsp;benefit&nbsp;from its features without having to spend the time&nbsp;to implement&nbsp;full system emulation.</p><p>The&nbsp;<a href="https://github.com/qilingframework/qiling" rel="noreferrer noopener"><u>Qiling framework</u></a>&nbsp;is based on Unicorn, so&nbsp;it was likely that the&nbsp;Unicorn script could be easily ported to&nbsp;Qiling.&nbsp;Figure 6 shows&nbsp;the&nbsp;API&nbsp;changes between&nbsp;unicorn&nbsp;engine and the&nbsp;Qiling&nbsp;framework.</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.talosintelligence.com/content/images/2026/02/unicorn_to_qiling_fig6.png" class="kg-image" alt="" loading="lazy" width="969" height="121" srcset="https://blog.talosintelligence.com/content/images/size/w600/2026/02/unicorn_to_qiling_fig6.png 600w, https://blog.talosintelligence.com/content/images/2026/02/unicorn_to_qiling_fig6.png 969w" sizes="(min-width: 720px) 720px"><figcaption><span style="white-space: pre-wrap;">Figure 6. Unicorn to&nbsp;Qiling&nbsp;API&nbsp;changes.</span></figcaption></figure><p>It&nbsp;wasn’t&nbsp;clear from existing examples in the&nbsp;Qiling&nbsp;codebase if single thread emulation was possible.&nbsp;After some investigation and&nbsp;some small modifications to two components called the blob loader and the blob&nbsp;OS&nbsp;it&nbsp;became&nbsp;feasible&nbsp;to emulate just this single thread rather than the&nbsp;whole system.&nbsp;Those&nbsp;code&nbsp;changes have been integrated into the development branch of&nbsp;Qiling&nbsp;on&nbsp;GitHub. Also, a little bit of monkey&nbsp;patching was&nbsp;required&nbsp;for&nbsp;my emulation script&nbsp;in order to&nbsp;output the coverage data in the correct way so that it&nbsp;contains&nbsp;accurate&nbsp;metadata for use in visualization tools like&nbsp;<a href="https://github.com/ForAllSecure/bncov" rel="noreferrer noopener"><u>bncov</u></a>&nbsp;or&nbsp;<a href="https://github.com/gaasedelen/lighthouse" rel="noreferrer noopener"><u>Lighthouse</u></a>. You can see an example of this in action in the&nbsp;<a href="https://github.com/qilingframework/qiling/blob/fe462b8313aa46d84cac5a0f7b207382fb25a37b/examples/hello_arm_blob_raw.py" rel="noreferrer noopener"><u>Qiling repository</u></a>.</p><p>This code coverage feature turned out to be more useful than originally expected.&nbsp;Code coverage data from multiple test inputs was compared to&nbsp;identify&nbsp;points at which their execution paths diverged. This approach&nbsp;facilitated&nbsp;rapid identification of the root causes of the crashes generated by AFL.</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.talosintelligence.com/content/images/2026/02/bncov_fig7.png" class="kg-image" alt="" loading="lazy" width="1292" height="647" srcset="https://blog.talosintelligence.com/content/images/size/w600/2026/02/bncov_fig7.png 600w, https://blog.talosintelligence.com/content/images/size/w1000/2026/02/bncov_fig7.png 1000w, https://blog.talosintelligence.com/content/images/2026/02/bncov_fig7.png 1292w" sizes="(min-width: 720px) 720px"><figcaption><span style="white-space: pre-wrap;">Figure 7. Code coverage visualization with&nbsp;bncov.</span></figcaption></figure><h2 id="vulnerability-highlight">Vulnerability highlight&nbsp;</h2><p>This fuzzing campaign led to the discovery of multiple Modbus messages that would cause a denial of service within the device and resulted in&nbsp;six&nbsp;CVEs. You can read those vulnerability reports here:&nbsp;<a href="https://talosintelligence.com/vulnerability_reports/TALOS-2025-2248" rel="noreferrer noopener"><u>TALOS-2025-2248</u></a>&nbsp;(CVE-2025-54848&nbsp;–&nbsp;&nbsp;CVE-2025-54851),&nbsp;<a href="https://talosintelligence.com/vulnerability_reports/TALOS-2025-2251" rel="noreferrer noopener"><u>TALOS-2025-2251</u></a>&nbsp;(CVE-2025-55221,&nbsp;CVE-2025-55222).</p><p>All&nbsp;the discussed vulnerabilities&nbsp;have been reported to the manufacturers&nbsp;in accordance with&nbsp;Cisco’s&nbsp;<a href="https://sec.cloudapps.cisco.com/security/center/resources/vendor_vulnerability_policy.html" rel="noreferrer noopener"><u>Coordinated Disclosure Policy</u></a>. Each of these vulnerabilities in the affected products has been patched by the corresponding manufacturer.</p><p>For&nbsp;SNORT®&nbsp;coverage that can detect the exploitation of these vulnerabilities, download the latest rulesets from&nbsp;<a href="https://snort.org/" rel="noreferrer noopener"><u>Snort.org</u></a>.</p><h2 id="conclusion">Conclusion&nbsp;</h2><p>In the future,&nbsp;Qiling&nbsp;will be my go-to&nbsp;for&nbsp;from&nbsp;the start of&nbsp;an emulation&nbsp;project. The high-level features of debugging and code coverage really make this a stand-out tool. However, if all you need is the ability to debug your scripts,&nbsp;udbserver&nbsp;is an easy solution that you can use with your Unicorn scripts as-is.&nbsp;Remember,&nbsp;"good enough" emulation is&nbsp;sometimes&nbsp;all that&nbsp;is needed to achieve impactful&nbsp;vulnerability&nbsp;discovery.&nbsp;</p>

    const currentWidth = window.innerWidth;  
  
    if (coverageTable && sidebar && postContent) {

        if (subscribeButton) {

        } else {
          // No subscribe button found, prepend to top of sidebar
          sidebar.prepend(coverageTable);
        }
      }
    }
  }
