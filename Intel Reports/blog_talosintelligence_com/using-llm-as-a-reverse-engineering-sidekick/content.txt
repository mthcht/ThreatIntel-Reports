    <h1 class="text-center">Using LLMs as a reverse engineering sidekick</h1>

    <div class="text-center m-1">
        <div class="post-author">
            <span>By </span>
                <a href="https://blog.talosintelligence.com/author/guilherme-venere/">Guilherme Venere</a>
        </div>

        <br/>
        <time class="post-datetime" datetime="July 31, 2025 06:00">
            Thursday, July 31, 2025 06:00
        </time>

        <div class="m-3">
                        <a href="https://blog.talosintelligence.com/category/on-the-radar/" class="category primary-category">
                            On The Radar
                        </a>
                            </div>
    </div>

    <section class="post-content-wrapper mt-5">
        <div class="post-content">
            <ul><li>This research explores how large language models (LLMs) can complement, rather than replace, the efforts of malware analysts in the complex field of reverse engineering.&nbsp;</li><li>LLMs may serve as powerful assistants to streamline workflows, enhance efficiency, and provide actionable insights during malware analysis.&nbsp;</li><li>We will showcase practical applications of LLMs in conjunction with essential tools like Model Context Protocol (MCP) frameworks and industry-standard disassemblers and decompilers, such as IDA Pro and Ghidra.&nbsp;</li><li>Readers will gain insights into which models and tools are best suited for common challenges in malware analysis and how these tools can accelerate the identification and understanding of unknown malicious files.&nbsp;</li><li>We also show how some common hurdles faced when using LLMs may influence the results, like cost increases due to tool usage and limitations of input context size in local models.</li></ul><hr><h2 id="talos-suggested-approach">Talos' suggested approach&nbsp;</h2><p>As the adoption of LLMs accelerates across industries, concerns about their potential to replace human expertise have become widespread. However, rather than viewing it as a threat to human expertise, we can consider LLMs as powerful tools to help malware researchers in our work.&nbsp;</p><p>We seek to show with this research that even by using low-cost tools and hardware, a malware researcher can take advantage of this technology to improve their work.&nbsp;&nbsp;</p><p>This blog covers the different choices of client applications available to interact with LLMs and disassemblers, the features to consider when choosing the best language model and the available plugins to integrate these applications into a solid framework to help during a malware analysis session.&nbsp;</p><p>For our tests, we decided to use a setup composed of a MCP server which implements integration with IDA-PRO and a MCP client based on VSCode. With this stack, we show how MCP servers can be used to help the language model execute tasks based on user input or in reaction to information found in the malicious code.&nbsp;</p><p>This blog also provides a step-by-step guide on how to set up your environment to achieve a basic setup to use an LLM with a local model running on your GPU.&nbsp;&nbsp;</p><h2 id="introduction-to-mcp">Introduction to MCP&nbsp;</h2><p>The <a href="https://modelcontextprotocol.io/introduction" rel="noreferrer noopener"><u>Model Context Protocol</u></a> (MCP) is an open protocol that standardizes how applications provide context to LLM clients and models. Tools and data sources made available by MCP servers provide the context, and the LLM model can choose which tool or data source to access based on user request or autonomously select it during runtime.&nbsp;&nbsp;</p><p>MCP servers implement tools using code and a description instructing the LLM on how to use each tool. The code can implement any kind of task, be it accessing API integration, file or network access, or any other automation necessary.</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.talosintelligence.com/content/images/2025/07/llm-flow.jpg" class="kg-image" alt="" loading="lazy" width="2000" height="1260" srcset="https://blog.talosintelligence.com/content/images/size/w600/2025/07/llm-flow.jpg 600w, https://blog.talosintelligence.com/content/images/size/w1000/2025/07/llm-flow.jpg 1000w, https://blog.talosintelligence.com/content/images/size/w1600/2025/07/llm-flow.jpg 1600w, https://blog.talosintelligence.com/content/images/2025/07/llm-flow.jpg 2262w" sizes="(min-width: 720px) 720px"><figcaption><span style="white-space: pre-wrap;">Figure 1. Diagram showing how an MCP server connects to other components in a typical setup.</span></figcaption></figure><p>These components can all be installed on the same or separate machines as needed. For example, the local LLM model may run on a separate server with better GPUs while IDA Pro and MCP server may be on a different machine with more restricted access, due to it being used to handle malware.&nbsp;&nbsp;</p><h2 id="choosing-the-right-tools-for-the-trade">Choosing the right tools for the trade&nbsp;</h2><p>A user interacts with an MCP server through an MCP client, which serves as the main interface to query the LLM model, exchange data with MCP servers and display this data back to the user. These clients can be any application which supports the MCP protocol, such as the <a href="https://claude.ai/download" rel="noreferrer noopener"><u>Claude.AI Desktop</u></a> or <a href="https://code.visualstudio.com/" rel="noreferrer noopener"><u>Visual Studio Code</u></a> (VSCode), using any of the MCP client extensions available in their marketplace.&nbsp;&nbsp;</p><p>For this blog, we use VSCode with the Cline MCP client extensions, but any of the many extensions currently available can be used. The most popular ones are:&nbsp;</p><ul><li>Cline: <a href="https://cline.bot/" rel="noreferrer noopener"><u>https://cline.bot/</u></a>&nbsp;</li><li>Roo Code: <a href="https://roocode.com/" rel="noreferrer noopener"><u>https://roocode.com/</u></a>&nbsp;</li><li>Copilot MCP: <a href="https://marketplace.visualstudio.com/items?itemName=AutomataLabs.copilot-mcp" rel="noreferrer noopener"><u>https://marketplace.visualstudio.com/items?itemName=AutomataLabs.copilot-mcp</u></a>&nbsp;</li></ul><p>For local implementations, the inference engine imposes another choice to make. There are several open-source engines with different levels of performance and usage complexity. Some of them are Python frameworks like <a href="https://docs.vllm.ai/en/v0.7.3/index.html" rel="noreferrer noopener"><u>vLLM,</u></a> others are C++ with Python-bindings and can be deployed on full servers that can be contacted via a REST API like <a href="https://github.com/ggml-org/llama.cpp?tab=readme-ov-file" rel="noreferrer noopener"><u>LLama.CPP</u></a> or Ollama. They all have advantages and disadvantages and an analysis between them is beyond the scope of this article. For our experiments we decided to use Ollama, mainly for its simplicity of use.&nbsp;</p><h3 id="model-selection-criteria">Model selection criteria&nbsp;</h3><p>The next component needed is an LLM. These can be either a cloud-based model or a locally running model. Most of the MCP clients support a wide range of cloud-based services and have pre-configured settings for them. For locally running models, using the <a href="https://ollama.com/" rel="noreferrer noopener"><u>Ollama</u></a> inference engine with one of their supported models is one of the most compatible solutions.&nbsp;</p><p>When choosing which model to use with MCP servers, some features need to be taken into consideration. This is due to the way MCP client interacts with the model and the MCP servers.&nbsp;</p><p>First, the model must support prompts with structured instructions. This is how the client will inform the model about what MCP tools are available, what they are used for and the template syntax on how to use them.&nbsp;</p><p>The model must also support large input <a href="https://www.ibm.com/think/topics/context-window" rel="noreferrer noopener"><u>context windows</u></a>, so it's able to parse large code bases and follow up analysis. This is needed so the client can pass all the necessary information to the model in the same query.&nbsp;&nbsp;</p><p>When the MCP client sends a query to the model, it puts into a single prompt all the information provided by the MCP server about what tools are available and how each one is used, generic instructions optimized for the model, and the prompt entered by the user. If it is a follow-up query, any output from previous commands is also added to the prompt, like the full decompilation of a function or a list of strings. As a result, this can quickly make the prompt reach the maximum input context the model or the inference engine supports.&nbsp;</p><p>This is less of a problem for cloud-based models, as it mainly influences the price of each request, and more of a problem for local models using the Ollama inference engine, which may truncate the prompt and provide invalid responses since it lacks all the context.&nbsp;&nbsp;</p><p>Due to these restrictions, models that are not trained specifically to handle tool usage or structured instructions may not be ideal to use with MCP servers and may cause hallucinations during code analysis.&nbsp;</p><p>For our study, we use the Ollama inference engine with <a href="https://ollama.com/library/devstral" rel="noreferrer noopener"><u>Devstral:24b</u></a> as a local LLM option and Antrophic's <a href="https://www.anthropic.com/news/claude-3-7-sonnet" rel="noreferrer noopener"><u>Claude 3.7 Sonnet</u></a> (<em>claude-3-7-sonnet-20250219</em>) as cloud-based model, as these models are specifically targeted for tools use and code analysis. Alternative models may be used in case of hardware restrictions, cost effectiveness, or user preference.&nbsp;&nbsp;</p><p>Some other points that need to be considered when choosing a local vs cloud-based model are:&nbsp;</p><ul><li><strong>Cost</strong>: Cloud-based models tend to charge for API access based on the number of tokens used in input queries and their responses. Analyzing large files with follow-up questions while keeping the context may quickly increase the cost associated with each prompt. On the other hand, local models tend to be slower and use the GPUs at maximum power, increasing the hidden cost of energy used to keep the machine up for the extension of the analysis. Local models also have an upfront cost of acquiring the necessary hardware to run the model, which does not exist for cloud-based models.&nbsp;</li><li><strong>Privacy and confidentiality:</strong> when using a cloud-based service, it is assumed that all information about the file being analyzed will be sent to the cloud LLM provider. Depending on the type of file being analyzed, this could break confidentiality rules imposed by employers or customers requesting the analysis. &nbsp;</li><li><strong>Speed</strong>: LLMs are processor- and memory-intensive applications. While running an LLM on a local single-GPU machine may have advantages in terms of confidentiality and cost, it is a much slower process than running the same on a cloud-based service. The same analysis which may take minutes to run on a cloud-based LLM may take several hours to run on a local LLM. Adding to this problem, once the model and context exceed the available GPU memory, the LLM may switch to using the CPU instead of (or in addition to) the GPU. This makes the process even slower.&nbsp;</li></ul><h3 id="ida-pro-mcp-servers">IDA Pro MCP servers&nbsp;</h3><p>The MCP server is usually composed of two parts, one of which is the plugin that runs inside the disassembler and performs the actions requested by the user, like renaming functions and variables, getting the disassembly or decompilation of a function or any other tasks implemented by the plugin. These functions are implemented as remote procedure calls (RPCs) which are made available to the client via the MCP server as âtools.â&nbsp;</p><p>The second component is the MCP server which will interact with the plugin and wait for instructions from the MCP client. This server uses the <a href="https://html.spec.whatwg.org/multipage/server-sent-events.html#server-sent-eventsvvvvv" rel="noreferrer noopener"><u>Server Sent Events (SSE)</u></a> protocol to communicate with the client and expose the tools configured in the plugin. Each MCP server will implement their own set of tools, so it's up to the user to choose the ones that best adapt to their needs. Below is a partial list of existing MCP servers for use with IDA Pro or Ghidra:&nbsp;</p><ul><li><a href="https://github.com/LaurieWired/GhidraMCP" rel="noreferrer noopener"><u>https://github.com/LaurieWired/GhidraMCP</u></a>&nbsp;</li><li><a href="https://github.com/suidpit/ghidra-mcp" rel="noreferrer noopener"><u>https://github.com/suidpit/ghidra-mcp</u></a>&nbsp;&nbsp;</li><li><a href="https://github.com/mrexodia/ida-pro-mcp" rel="noreferrer noopener"><u>https://github.com/mrexodia/ida-pro-mcp</u></a>&nbsp;</li><li><a href="https://plugins.hex-rays.com/mxiris-reverse-engineering/ida-mcp-server" rel="noreferrer noopener"><u>https://plugins.hex-rays.com/mxiris-reverse-engineering/ida-mcp-server</u></a>&nbsp;</li><li><a href="https://github.com/fdrechsler/mcp-server-idapro" rel="noreferrer noopener"><u>https://github.com/fdrechsler/mcp-server-idapro</u></a>&nbsp;</li><li><a href="https://github.com/taida957789/ida-mcp-server-plugin" rel="noreferrer noopener"><u>https://github.com/taida957789/ida-mcp-server-plugin</u></a>&nbsp;</li><li><a href="https://github.com/JusticeRage/Gepetto" rel="noreferrer noopener"><u>https://github.com/JusticeRage/Gepetto</u></a>&nbsp;</li></ul><p>To summarize how we set up our environment for this research, this is what we are using:&nbsp;</p><ul><li>Local model with Ollama and Devstral&nbsp;</li><li>Cloud-based LLM using Anthropic's Claude Sonnet 3.7&nbsp;</li><li><a href="https://github.com/mrexodia/ida-pro-mcp" rel="noreferrer noopener"><u>MCP Server by Duncan Ogilvie (Mrexodia)</u></a> with its plugin configured in IDA Pro&nbsp;</li><li>Installed and configured Cline plugin in VSCode&nbsp;</li></ul><p>For our analysis, the setup utilized was a Desktop machine running on an AMD Ryzen 7 9800X3D at 4.7ghz, 64Gb DDR5 6000 RAM, 3Tb SSD and an NVIDIA 5070 Ti 16Gb GPU, which can be considered a medium/high end consumer desktop.&nbsp;</p><h2 id="using-llm-to-analyze-a-real-world-sample">Using LLM to analyze a real-world sample&nbsp;</h2><p>Using the setup previously defined, let's analyze a malicious file and compare the results between human analysis, local LLM and cloud-based LLM. The sample we are going to use is in fact an old <a href="https://malpedia.caad.fkie.fraunhofer.de/details/win.icedid" rel="noreferrer noopener"><u>IcedID</u></a> sample which has been extensively reversed and documented using <a href="https://hex-rays.com/lumina" rel="noreferrer noopener"><u>Hex-Rays's Lumina database</u></a>. This way we have a common base to compare the results from different LLMs.&nbsp;&nbsp;</p><p>This may imply that these analysis results may have gone into the training data of both models and may influence the results, but our intention is to compare how both LLMs fare against each other.&nbsp;</p><ul><li>7412945177641e9b9b27e601eeda32fda02eae10d0247035039b9174dcc01d12&nbsp;</li></ul><p>This is a small sample, with about 37 defined functions, which will be important to consider when analyzing the results. A bigger sample with hundreds of functions will have a different impact on the analysis as we will discuss it later.&nbsp;&nbsp;</p><h3 id="creating-the-prompts">Creating the prompts&nbsp;</h3><p>Perhaps the most important step in making efficient use of an LLM is defining the right prompts to query the model. A good prompt not only will give you a better result in the analysis but may as well make the query less expensive and faster to finish while reducing the chances of hallucinations.&nbsp;</p><p>For this example, we used a prompt template suggested by <a href="https://github.com/mrexodia/ida-pro-mcp?tab=readme-ov-file#prompt-engineering" rel="noreferrer noopener"><u><em>Mrexodia </em>on </u></a><u>his Github page</u>, with some improvements and changes to cover different types of analysis after some experimentation. Our intention was to use the exact same prompt for both local and cloud-based LLM, and cover three different situations which may be common during malware analysis:&nbsp;</p><ul><li>Top-down analysis from the program entry point&nbsp;</li><li>Figuring out what a specific function does and how it is used by the application&nbsp;</li><li>Analyse all unknown functions and rename them to make it easy to understand the rest of the code&nbsp;</li></ul><p>To cover these examples, we created three prompts (Figures 2 â 4), which were used for all cases discussed.&nbsp;&nbsp;</p><p>Please note that these prompts are not optimized and most definitely can be further improved. However, in this case, all three prompts needed to be generic enough to be used in all our cases without changes to create a fair comparison.&nbsp;&nbsp;</p><p>Another important note is that we do not explicitly say in the prompt that the file was malicious, as this caused the model to assume every function was malicious, and that caused a bias in their analysis.&nbsp;</p><pre><code>Your task is to analyze an unknown file which is currently open in IDA Pro. This application also has limitations that may prevent it from being widely adopted, such as:&nbsp;</p><ul><li><strong>Usage cost</strong>: The use of language models is usually associated with high cost per token input by the user or generated by the model. When analyzing complex malware with hundreds of functions, this cost can compound quickly and make it impractical as a solution. This is especially problematic with the use of MCP tools since they add their own content with instructions on how to use the tools to the user prompt. They also include any text extracted from the analysis like strings and disassembly/decompilation listing, and when all of this is put together, a simple query can cost several hundreds of thousands of tokens.&nbsp;</li><li><strong>Time overhead</strong>: The analyst using these tools needs to consider not only the time taken waiting for a response, but also the time taken creating the most efficient prompts to maximize results while reducing costs. A local model may reduce costs but also add considerable time to output results.&nbsp;</li><li><a href="https://blog.virustotal.com/2025/06/what-17845-github-repos-taught-us-about.html" rel="noreferrer noopener"><strong><u>Malicious tools</u></strong></a>: Since MCP servers are a somewhat recent technology used in a field that moves very fast, there is a proliferation of tools and applications that are available to users looking to use LLMs with their work. There is a variety of MCP clients and as many marketplaces where people can upload their MCP servers for anyone to use. Users need to be aware and take careful consideration of which tool they are using, as these applications may intentionally or unintentionally contain malicious code, which may compromise the researchers using them.&nbsp;&nbsp;</li><li><a href="https://www.prompt.security/blog/top-10-mcp-security-risks" rel="noreferrer noopener"><strong><u>Vulnerabilities</u></strong></a>: As is true with other computer applications, MCP protocol and LLMs have attack surfaces that <a href="https://vulnerablemcp.info/" rel="noreferrer noopener"><u>can be exploited</u></a> in many ways to compromise the data being generated or the actual user's machine. <a href="https://genai.owasp.org/llmrisk/llm01-prompt-injection/" rel="noreferrer noopener"><u>Prompt injection</u></a>, MCP tool poisoning, tool privilege abuse and other techniques may be used to compromise the generated data, exfiltrate sensitive data, expose confidential code or tokens and even execute code remotely.&nbsp;</li></ul><h2 id="ida-pro-mcp-server-installation">IDA Pro MCP server installation&nbsp;</h2><p>Now that we've seen how useful a local model can be in helping the analyst perform reverse engineering on malicious files, let's learn how to set up the environment.&nbsp;&nbsp;</p><p>The process of installing and configuring an MCP server to use with your disassembler may vary depending on the software stack you plan to use, but to give readers an idea of how the process works and discuss some caveats that may impact the results, we will review the installation of an MCP server integrated with IDA Pro. To make sure IDA Pro and the MCP server are using the same version, you need to ensure IDA is using the same Python version you have on your default path.&nbsp;&nbsp;</p><p>This is done using a tool available in the IDA Pro installation folder. In our case we are using IDA Pro 8.5 so the tool is located at "c:\Program Files\IDA Pro 8.5\idapyswitch.exe". Running the tool will give you a list of currently installed Python versions and let you choose the one IDA will use:&nbsp;</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.talosintelligence.com/content/images/2025/07/2a5c25bb-b747-4b02-86fd-253d986f2f68.PNG" class="kg-image" alt="" loading="lazy" width="995" height="293" srcset="https://blog.talosintelligence.com/content/images/size/w600/2025/07/2a5c25bb-b747-4b02-86fd-253d986f2f68.PNG 600w, https://blog.talosintelligence.com/content/images/2025/07/2a5c25bb-b747-4b02-86fd-253d986f2f68.PNG 995w" sizes="(min-width: 720px) 720px"><figcaption><span style="white-space: pre-wrap;">Figure 9. Example usage of the IDApyswitch tool to choose the correct python version to use in IDA.</span></figcaption></figure><p>Once IDA is using the same Python version needed by the MCP Server, you can install the server following the process from their Github page:&nbsp;</p><p><code>pip uninstall ida-pro-mcp<br>pip install git+https://github.com/mrexodia/ida-pro-mcp<br>ida-pro-mcp --install</code></p><p>This will install the required modules and copy the IDA Pro plugin to its proper location. The final step is to run the MCP server. This step is necessary if you're using Cline or Roo Code inside VSCode, as these tools don't work well with the server running directly by the client. To run the MCP server, use the command in Figure 10 and ensure the terminal where it is running stays open for the duration of your analysis session.&nbsp;</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.talosintelligence.com/content/images/2025/07/43dcf2dc-1d92-4a0d-ad37-9a492c8af022.PNG" class="kg-image" alt="" loading="lazy" width="667" height="119" srcset="https://blog.talosintelligence.com/content/images/size/w600/2025/07/43dcf2dc-1d92-4a0d-ad37-9a492c8af022.PNG 600w, https://blog.talosintelligence.com/content/images/2025/07/43dcf2dc-1d92-4a0d-ad37-9a492c8af022.PNG 667w"><figcaption><span style="white-space: pre-wrap;">Figure 10. Example command to run the MCP server from command line.&nbsp;</span></figcaption></figure><p>&nbsp;The last step is to ensure the MCP plugin is working and running inside IDA. Once you have a file open in the disassembler, you can start the MCP Plugin component by going to âEdit-&gt;Plugins-&gt;MCPâ or using the hotkey âControl+Alt+M.â You should see a message informing you the server started successfully in the IDA output window.&nbsp;</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.talosintelligence.com/content/images/2025/07/f06c5e69-4362-47bb-89ca-daf576525ba1.PNG" class="kg-image" alt="" loading="lazy" width="372" height="43"><figcaption><span style="white-space: pre-wrap;">Figure 11. Message showing the successful start of the MCP plugin in IDA.</span></figcaption></figure><h3 id="install-and-configure-mcp-client">&nbsp;Install and configure MCP client&nbsp;</h3><p>The last step is to install the MCP client, which will work as a connector to all other components. The installation process will vary depending on which tools you choose, but in this case, we will use VSCode with the Cline extension. In VSCode, go to the Extensions Marketplace and search for Cline. Click on the Install button and wait for the installation to finish.&nbsp;</p><p>A new tab will appear in the left tab menu with the Cline extension icon. This interface is how you are going to interact with the IDA Pro MCP Server and the LLM.&nbsp;</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.talosintelligence.com/content/images/2025/07/cline.png" class="kg-image" alt="" loading="lazy" width="654" height="587" srcset="https://blog.talosintelligence.com/content/images/size/w600/2025/07/cline.png 600w, https://blog.talosintelligence.com/content/images/2025/07/cline.png 654w"><figcaption><span style="white-space: pre-wrap;">Figure 12. Cline extension main interface.&nbsp;</span></figcaption></figure><p>Clicking on the gear icon in the top right menu takes you to the Configuration page, where you must configure the LLM model you're using. Once you choose the API provider, you need to configure the base URL and model name at least, but cloud-based models will require an API key, as well. The settings in Figure 13 are for a local model running on Ollama.&nbsp;</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.talosintelligence.com/content/images/2025/07/ollama_config.png" class="kg-image" alt="" loading="lazy" width="910" height="575" srcset="https://blog.talosintelligence.com/content/images/size/w600/2025/07/ollama_config.png 600w, https://blog.talosintelligence.com/content/images/2025/07/ollama_config.png 910w" sizes="(min-width: 720px) 720px"><figcaption><span style="white-space: pre-wrap;">Figure 13. Cline extension LLM configuration page.</span></figcaption></figure><p>The next step is to configure the MCP server in Cline. This is how Cline will know which servers are available and what tools each server provides. These settings are found in the icon with three bars at the top right menu.&nbsp;</p><p>The MCP server can be configured in the âRemote Serversâ tab, or you can directly edit the configuration file. The example in Figure 14 shows both options to configure the MCP server we set up in the previous step:&nbsp;</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.talosintelligence.com/content/images/2025/07/mcp_config.png" class="kg-image" alt="" loading="lazy" width="1218" height="416" srcset="https://blog.talosintelligence.com/content/images/size/w600/2025/07/mcp_config.png 600w, https://blog.talosintelligence.com/content/images/size/w1000/2025/07/mcp_config.png 1000w, https://blog.talosintelligence.com/content/images/2025/07/mcp_config.png 1218w" sizes="(min-width: 720px) 720px"><figcaption><span style="white-space: pre-wrap;">Figure 14. Cline extension MCP server configuration.&nbsp;</span></figcaption></figure><p>Once this is done, you should see something similar to Figure 15 on your âInstalledâ tab, indicating that Cline is able to talk to the MCP Server and the IDA Pro plugin.&nbsp;</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.talosintelligence.com/content/images/2025/07/7b70fa6b-928a-4a46-a180-1a5305ac0271.PNG" class="kg-image" alt="" loading="lazy" width="735" height="822" srcset="https://blog.talosintelligence.com/content/images/size/w600/2025/07/7b70fa6b-928a-4a46-a180-1a5305ac0271.PNG 600w, https://blog.talosintelligence.com/content/images/2025/07/7b70fa6b-928a-4a46-a180-1a5305ac0271.PNG 735w" sizes="(min-width: 720px) 720px"><figcaption><span style="white-space: pre-wrap;">Figure 15. MCP server list of available tools.</span></figcaption></figure><p>&nbsp;Now that everything is set up, you may begin to use the Cline chat window to query the IDA Pro database currently open and perform analysis.&nbsp;</p><h2 id="coverage">Coverage&nbsp;</h2><p>Ways our customers can detect and block this threat are listed below.&nbsp;</p><figure class="kg-card kg-image-card"><img src="https://blog.talosintelligence.com/content/images/2025/07/endpoint-firewall-analytics-web.jpg" class="kg-image" alt="" loading="lazy" width="2000" height="404" srcset="https://blog.talosintelligence.com/content/images/size/w600/2025/07/endpoint-firewall-analytics-web.jpg 600w, https://blog.talosintelligence.com/content/images/size/w1000/2025/07/endpoint-firewall-analytics-web.jpg 1000w, https://blog.talosintelligence.com/content/images/size/w1600/2025/07/endpoint-firewall-analytics-web.jpg 1600w, https://blog.talosintelligence.com/content/images/size/w2400/2025/07/endpoint-firewall-analytics-web.jpg 2400w" sizes="(min-width: 720px) 720px"></figure><p><a href="https://www.cisco.com/c/en/us/products/security/amp-for-endpoints/index.html" rel="noreferrer noopener">Cisco Secure Endpoint</a> (formerly AMP for Endpoints) is ideally suited to prevent the execution of the malware detailed in this post.
