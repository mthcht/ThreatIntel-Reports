        <h1>Interception: Dissecting BokBot’s “Man in the Browser”</h1>
        
        <div class="publish_info">
            <p>March 21, 2019</p>
            <span>|</span>
            <a href="/en-us/blog/author.shaun-hurley-and-james-scalise/" title="Posts by shaun.hurley.and.james.scalise" rel="author">
                shaun.hurley.and.james.scalise</a>
            <span>|</span>
                
            <a href="/en-us/blog/category.from-the-front-lines/" title="From The Front Lines">From The Front Lines</a>
        </div>
        <div class="post_image">
            <img width="1060" height="698" src="https://assets.crowdstrike.com/is/image/crowdstrikeinc/BokBot-blog-image-1" class="attachment-post-thumbnail size-post-thumbnail wp-post-image" alt="" decoding="async" fetchpriority="high" srcset="/content/dam/crowdstrike/www/en-us/wp/2019/03/BokBot-blog-image-1.png 530w,https://assets.crowdstrike.com/is/image/crowdstrikeinc/BokBot-blog-image-1-300x198 300w" sizes="(max-width: 1060px) 100vw, 1060px"/>

<div class="cmp-text" id="text-550a135bb3" data-cmp-name="cmp-text-" data-cmp-is="text" data-target-location="false">
    
       <span><h2><span style="	font-weight: 400;
">BokBot: Proxy Module</span></h2>
<p><span style="	font-weight: 400;
">This article is a continuation of CrowdStrike’s recent blog, “</span><a href="/en-us/blog/digging-into-bokbots-core-module/"><span style="	font-weight: 400;
">Digging Into BokBot’s Core Module</span></a><span style="	font-weight: 400;
">,” and provides a detailed analysis of the inner workings of the BokBot proxy module — </span><b>a complex piece of code written to trick victims into sending sensitive information to a command and control (C2) server.</b></p>
<h2><span style="	font-weight: 400;
">Overview</span></h2>
<p><span style="	font-weight: 400;
">The BokBot banking Trojan —</span> also known as IcedID — was first observed in April 2017 and CrowdStrike has been tracking this threat ever since. BokBot has been used against financial institutions worldwide and is able to augment its capabilities by retrieving several modules, including one that runs a local proxy server. This proxy module is able to intercept and potentially manipulate web traffic with the goal of facilitating financial fraud. <span style="	font-weight: 400;
">The BokBot core module downloads the proxy module, injects it into a spawned svchost child process, and the proxy module initializes itself in the target process. The following is a step-by-step analysis of how this process unfolds.</span></p>
<h2><span style="	font-weight: 400;
">Module Initialization</span></h2>
<p><span style="	font-weight: 400;
">Prior to standing up the proxy server thread, this module goes through an initialization process. Most of it is similar to the other BokBot modules — </span><a href="/en-us/blog/digging-into-bokbots-core-module/"><span style="	font-weight: 400;
">covered in the previous blog</span></a><span style="	font-weight: 400;
"> — and includes making errors less noisy, building a list of C2s, and setting up named events for communicating with the parent process. The following steps are taken prior to intercepting any traffic from a victim’s browser.</span></p>
<h3><span style="	font-weight: 400;
">Webinject Updates</span></h3>
<p><span style="	font-weight: 400;
">User-mode asynchronous procedure call (APC) objects are used to trigger update events for the webinject data that exists in process memory. During initialization, separate user APC objects are sent to the APC queue, one for each of the webinject DAT files (</span><a href="/en-us/blog/digging-into-bokbots-core-module/"><span style="	font-weight: 400;
">see previous blog</span></a><span style="	font-weight: 400;
">). Each DAT file is decoded and stored in process memory, and will be passed as a parameter to the APC callback function. Once the APC queue is processed, the web configs will be parsed and loaded into process memory.</span></p>
<h3><span style="	font-weight: 400;
">C2 Communication Thread</span></h3>
<p><span style="	font-weight: 400;
">The communication thread is signaled whenever there is collected data to be sent back to the C2. A signal event occurs when the injected malicious javascript sends a specific type of request to the proxy server (see: Browser Perspective section) . The data that is sent back can consist of harvested personal information, snapshots or proxy-related errors.</span></p>
<h3><span style="	font-weight: 400;
">Proxy Server Initialization</span></h3>
<p><span style="	font-weight: 400;
">The proxy server is bound to 127.0.0.1 on TCP port 57391. After the listener is set, a Windows Socket API (WSA) event handler is registered using this socket to handle all connect/send/receive network requests.</span></p>
<h3><span style="	font-weight: 400;
">SSL Certificates</span></h3>
<p><span style="	font-weight: 400;
">In order to perform a <a href="/epp-101/man-in-the-middle-mitm-attacks.html">man-in-the-middle (MITM) attack</a> on SSL connections, the proxy server needs to generate an SSL certificate, and insert it into the cert store. The certificate is created by calling CertCreateSelfSignCertificate, using the following hard-coded distinguished name (DN) values:</span> <img class="alignnone wp-image-22851 size-full" src="/content/dam/crowdstrike/www/en-us/wp/2019/03/code-1.png" width="672" height="100"/> <span style="	font-weight: 400;
">A temporary cert store is created in memory and will eventually be written out to the following location, with a filename that is generated using the Bot ID:</span></p>
<ul><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">C:\Users\jules\AppData\Local\Temp\D38D667F.tmp</span></li></ul>
<p><span style="	font-weight: 400;
">The certificate store contains an SSL certificate for the webinject and C2 servers.</span> <img class="alignnone wp-image-22852 size-full" src="/content/dam/crowdstrike/www/en-us/wp/2019/03/code-2.png" width="672" height="315"/> <span style="	font-weight: 400;
">Once these steps have been completed, the proxy server is ready to intercept requests from the browser, but the browser has not been reconfigured to point to the proxy server.</span></p>
<h2><span style="	font-weight: 400;
">Proxy Connections</span></h2>
<p><span style="	font-weight: 400;
">Now, whenever the browser attempts to connect to a website, that request is hijacked and first processed by the proxy server. This section covers how the connection states are managed, how SSL MITM works, and what actions are taken by the proxy server.</span></p>
<h3><span style="	font-weight: 400;
">Managing Connection State</span></h3>
<p><span style="	font-weight: 400;
">All connections are managed by a series of data structures tied to WSA callback events that keep track of internal communication with the client, external communication with the target website, and to ensure the integrity of all the requests handled by the proxy server.</span></p>
<div class="wp-caption" id="attachment_22854"><img class="wp-image-22854 size-full" src="/content/dam/crowdstrike/www/en-us/wp/2019/03/Figure-1-ProxyConnectionState.png" alt="Proxy Connection State Architecture" width="672" height="397"/>Figure 1: Proxy Connection State Architecture</div>
<p><span style="	font-weight: 400;
">The architecture diagram in Figure 1 summarizes the layout of relations between the proxy server components used to maintain the state of a proxy request.</span></p>
<h3><span style="	font-weight: 400;
">SSL Man in the Middle</span></h3>
<p><span style="	font-weight: 400;
">Any SSL request, where the URL matches a URL from the list of webinject targets, is hijacked by the proxy inserting itself into the communication using its own SSL certificate. The proxy server receives the request from the victim and establishes an SSL connection using the proxy server’s SSL certificate. After that, the proxy server sends the request to the target website, and establishes an SSL connection between the proxy and the target website. The response from the target is decrypted, and then encrypted using the proxy’s certificate. This response is sent to the victim, where the traffic will be decrypted using the proxy’s certificate.</span></p>
<p>An SSL context data-structure, similar to Figure 1, is used to maintain the state of the SSL traffic. <span style="	font-weight: 400;
">See the “Ensuring Valid Certificates” section to understand how the browser is hooked to ensure that it sees these certificates as valid.</span></p>
<h2><span style="	font-weight: 400;
">Proxy Action</span></h2>
<p><span style="	font-weight: 400;
">If the requested URL hostname does not match one of the webinjects, then the proxy server passes the HTTP requests/responses between the infected host and the web server. However, if a URL matches one of the many websites targeted by the webinjects, then additional action is taken (see “Traffic Manipulation Proxy Perspective”).</span></p>
<h2><span style="	font-weight: 400;
">Redirecting Browsers to the Proxy</span></h2>
<p><span style="	font-weight: 400;
">After the proxy server has been initialized, any currently executing browser process will have to be configured to use the proxy. To do this, BokBot injects code into the browser process. The injected code adds hooks into key functions, allowing it to hijack browser traffic.</span></p>
<h3><span style="	font-weight: 400;
">Browser Process Selection</span></h3>
<p><span style="	font-weight: 400;
">A looping thread is spawned to iterate over all of the currently executing process names. To get a list of current processes, a list of </span><code><span style="	font-weight: 400;
">BASIC_PROCESS_INFORMATION</span></code><span style="	font-weight: 400;
"> structures is generated using </span><code><span style="	font-weight: 400;
">ZwQuerySystemInformation</span></code><span style="	font-weight: 400;
">.</span></p>
<h4><span style="	font-weight: 400;
">Process Identification</span></h4>
<p><span style="	font-weight: 400;
">Once the list is generated, the module will attempt to identify browser processes. A hash of the process name is used to identify whether the process is one of the specific browser processes listed in Table 1.</span></p>
<div class="wp-caption" id="attachment_22855"><img class="wp-image-22855 size-full" src="/content/dam/crowdstrike/www/en-us/wp/2019/03/Table-1.png" width="672" height="192"/>Table 1: Browser Identification by Hash</div>
<p><span style="	font-weight: 400;
">The hash is generated using a custom method, and then XORed with a constant value that varies between samples, so that the hash values differ between campaigns.</span></p>
<h4><span style="	font-weight: 400;
">Proxy Configured Check</span></h4>
<p><span style="	font-weight: 400;
">Once the browser has been identified an additional check is made to determine if this browser process has already had the proxy configured by BokBot. To keep track of every browser process that has been injected into, a linked list data structure is created (Figure 2). The entire list is walked, checking to see if both the process ID and the process creation time exist in the list. If the browser is not in the list, then a new list item is created that contains the target process ID and target process creation time.</span></p>
<div class="wp-caption" id="attachment_22856"><img class="wp-image-22856 size-full" src="/content/dam/crowdstrike/www/en-us/wp/2019/03/Figure2-proxied-browser-process-list.png" alt="Linked List of Browser Processes using the BokBot Proxy" width="672" height="80"/>Figure 2: Linked List of Browsers Processes Using the Proxy</div>
<p><span style="	font-weight: 400;
">An additional check is made by attempting to open up a named event. After the code is injected, a named event is created by the browser process. The naming scheme is generated in a similar manner as was discussed in the previous blog about BokBot’s main module, however, the process ID is appended to the end of the name. The module calls </span>OpenEvent and if the returned error code is <code style="	font-size: 14.0px;
">ERROR_FILE_NOT_FOUND</code>, the injection code continues.</p>
<h3><span style="	font-weight: 400;
">Opening the Process and Additional Checks</span></h3>
<p><code><span style="	font-weight: 400;
">OpenProcess</span></code><span style="	font-weight: 400;
"> is called to open a handle to the browser process. BokBot checks to see if the process is WOW64, and if so, different procedures are used that will yield the same result.</span> <span style="	font-weight: 400;
">To be thorough, another check is made to determine if the process has already been configured:</span></p>
<ul><li style="	font-weight: 400;
"><code><span style="	font-weight: 400;
">ZwQueryInformationProcess</span></code><span style="	font-weight: 400;
"> and </span><code><span style="	font-weight: 400;
">ReadProcessMemory</span></code><span style="	font-weight: 400;
"> are used to get the process environment strings.</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Each string is checked to see if this exists: </span><code><span style="	font-weight: 400;
">v313235373937&#61;true</span></code><ul><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">The </span><code><span style="	font-weight: 400;
">3132353739</span></code><span style="	font-weight: 400;
"> is an ascii representation of the BotID.</span></li></ul>
</li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">No injection occurs if this environment string exists.</span></li></ul>
<h3><span style="	font-weight: 400;
">Browser Process Injection and Code Execution</span></h3>
<p><span style="	font-weight: 400;
">The proxy configuration code is injected in the same manner as BokBot’s child process injection method (</span><a href="/en-us/blog/digging-into-bokbots-core-module/"><span style="	font-weight: 400;
">see the blog on BokBot’s main module</span></a><span style="	font-weight: 400;
">), except for two things:</span><code><span style="	font-weight: 400;
">OpenProcess</span></code><span style="	font-weight: 400;
"> is called to connect to the process, and a hook is added for </span><code><span style="	font-weight: 400;
">ZwWaitForSingleObject</span></code><span style="	font-weight: 400;
">. This means that as soon as the browser executes </span><code><span style="	font-weight: 400;
">ZwWaitForSingleObject</span></code><span style="	font-weight: 400;
">, the injected code executes:</span></p>
<p>The hook is removed, the proxy is configured and the <code><span style="	font-weight: 400;
">ZwWaitForSingleObject</span></code><span style="	font-weight: 400;
"> call is completed to maintain process execution.</span></p>
<h4><span style="	font-weight: 400;
">Context Structure</span></h4>
<p><span style="	font-weight: 400;
">The context data structure is injected into the browser process and provides the proxy configuration code with the necessary information to properly configure the proxy.</span></p>
<h3><span style="	font-weight: 400;
">Browser Proxy Configuration Code</span></h3>
<p><span style="	font-weight: 400;
">The “configuration code” is actually a series of procedure hooks that are used to insert the proxy module into the communication channel:</span></p>
<ul><li style="	font-weight: 400;
"><code><span style="	font-weight: 400;
">CertVerifyCertificateChainPolicy</span></code></li><li style="	font-weight: 400;
"><code><span style="	font-weight: 400;
">CertGetCertificateChain</span></code></li><li style="	font-weight: 400;
"><code><span style="	font-weight: 400;
">connect (ws2_32.dll)</span></code></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Browser specific functions:</span><ul><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Internet Explorer: </span><code><span style="	font-weight: 400;
">MSAFD_ConnectEx </span></code></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Firefox: </span><code><span style="	font-weight: 400;
">SSL_AuthCertificateHook</span></code></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Chrome: <code>ws2_32.WSAEventSelect</code></span></li></ul>
</li></ul>
<h4><span style="	font-weight: 400;
">Inserting the Hooks</span></h4>
<p><span style="	font-weight: 400;
">Most of the hooks are inserted by walking the export table of the target module, hashing the procedure name, and then comparing that hash against a static value. If these hashes match, then the hook can be placed.</span></p>
<div class="wp-caption" id="attachment_22857"><img class="wp-image-22857 size-full" src="/content/dam/crowdstrike/www/en-us/wp/2019/03/Table-2.png" width="672" height="216"/>Table 2: Hash for Hooked Function Names</div>
<p><span style="	font-weight: 400;
">To get the address for </span><code><span style="	font-weight: 400;
">MSAFD_ConnectEx</span></code><span style="	font-weight: 400;
">, a different method is used and is discussed in the “Internet Explorer: MSAEFD_ConnectEx” section.</span></p>

<h5><span style="	font-weight: 400;

<h5><span style="	font-weight: 400;

<h4><span style="	font-weight: 400;

<p><span style="	font-weight: 400;
">Once browser traffic is redirected to the proxy, the malware must prevent browser certificate errors from informing the user that requests are being intercepted. To ensure that the certificates are both verified and trusted, two procedures are hooked: </span><code><span style="	font-weight: 400;

<h5><span style="	font-weight: 400;

<p>Each one of these <code><span style="	font-weight: 400;

<h2><span style="	font-weight: 400;
">Traffic Manipulation: Proxy Perspective</span></h2>
<p><span style="	font-weight: 400;
">BokBot’s proxy module relies on traffic manipulation to steal a victim’s sensitive information. Web traffic generated by a victim’s browser is matched against a list of target URLs (webinjects), and if matched, the proxy takes one of the following actions: redirect to a decoy website (webfake); scrape the page; screenshot the page; or ignore. In addition, the response to the request is matched to determine if either HTML or Javascript is injected into the page and served back to the victim.</span></p>
<div class="wp-caption" id="attachment_22867"><img class="size-full wp-image-22867" src="/content/dam/crowdstrike/www/en-us/wp/2019/03/Figrure3-WebfakeProcess.png" alt="BokBot WebFake Process Overview" width="672" height="414"/>Figure 3: BokBot WebFake Process Overview</div>

<p><span style="	font-weight: 400;

During the initialization process, this structured data is broken into multiple lists for each of the webinject categories. The webinject category lists are built out of a series of webinject types. Once parsed, each element has the following structure:</span><img class="alignnone size-full wp-image-22868" src="/content/dam/crowdstrike/www/en-us/wp/2019/03/code-8.png" width="672" height="461"/></p>
<h3><span style="	font-weight: 400;
">Page Scraping</span></h3>
<p><span style="	font-weight: 400;
">The page-scraping processing functions perform a match on either the URL or on the HTML body to determine if the webpage’s information should be scraped and sent to the C2. Page scraping targets banking account display pages to grab the target’s account information.</span></p>
<h4><span style="	font-weight: 400;
">Types 33, 34</span></h4>
<p><span style="	font-weight: 400;
">Either an exact URL string or a regex is used to match the victim’s requested URL. Once a match is made, both the HTML body and the matched URL are sent to the C2. Each of the targeted URLs are related to pages that contain the victim’s account information, for example:</span></p>

<h4><span style="	font-weight: 400;
">Type 64</span></h4>
<p><span style="	font-weight: 400;
">This scans the webpage body looking for text (“Account Balance,” “Current Balance,” etc.) related to account balances, security challenge questions and other personal data. If located, the URL, the HTML body and the matched text are compressed and sent to the C2.</span></p>
<h3><span style="	font-weight: 400;
">Screenshots</span></h3>
<p><span style="	font-weight: 400;
">Screenshots are generated when a URL matches on a Type 49 inject. Screenshots are taken using the Windows GDI&#43; API. A bitmap is generated, written to a tmp file, read into a buffer, compressed, and then sent to the C2. Bitmaps are written to the local temp directory under AppData and the filename is a unique string of alpha characters.</span></p>
<ul><li style="	font-weight: 400;
"><span style="	font-weight: 400;

<p><span style="	font-weight: 400;
">Although no concrete examples came with the webinjects of the BokBot versions used to generate this report, it is believed that these screenshots will contain sensitive details related to commercial and banking accounts.</span></p>
<h3><span style="	font-weight: 400;
">Code Injection</span></h3>
<p><span style="	font-weight: 400;
">Code injection works by either matching</span></p>
<p>a URL, or doing a match and replace on an HTML element.</p>
<h4><span style="	font-weight: 400;
">Types 17 and 19</span></h4>
<p><span style="	font-weight: 400;
">Types 17 and 19 are used to hide elements within a page, grab form data, inject code, replace code, and to make the webfake experience more believable for the victim. The difference between these two types is that Type 17 doesn’t rely on regex to match and replace HTML and Javascript.</span><img class="alignnone size-full wp-image-22870" src="/content/dam/crowdstrike/www/en-us/wp/2019/03/code-9.png" width="672" height="124"/></p>
<h4><span style="	font-weight: 400;
">Type 81</span></h4>
<p><span style="	font-weight: 400;
">Type 81 is a list of URLs that are ignored entirely by the proxy server. The purpose is to avoid complications that could come up when injecting into advertisements and chat or email clients. Rather than deal with handling this code, the requests and responses are forwarded to the victim or the target by the proxy, without modification.</span></p>
<h3><span style="	font-weight: 400;
">Webfake Phishing Site</span></h3>
<p><span style="	font-weight: 400;
">BokBot uses the webinjects to create a replica of the original target website. These replica websites are called webfakes. URIs are rewritten to forward the traffic to the webfake site. This section covers each of the injection types used to redirect traffic to a webfake. The web browser is not aware that traffic is being redirected to the webfake.</span></p>
<div class="wp-caption" id="attachment_22872"><img class="wp-image-22872 size-full" src="/content/dam/crowdstrike/www/en-us/wp/2019/03/Figure-4-WebFake-Example.png" alt="BokBot WebFake Example" width="672" height="290"/>Figure 4: BokBot WebFake Example</div>
<h4><span style="	font-weight: 400;
">Type 50</span></h4>
<p><span style="	font-weight: 400;

Whenever a match occurs, the URL is rewritten to point toward the webfake site, and the request is sent to the replica page. The response to that request is parsed by the proxy server and sent back to the victim.</span><img class="alignnone size-full wp-image-22874" src="/content/dam/crowdstrike/www/en-us/wp/2019/03/code-10-type50.png" width="672" height="96"/></p>
<h4><span style="	font-weight: 400;
">Type 51</span></h4>
<p><span style="	font-weight: 400;
">To match individual icon/gif/bitmap/other names, regex match and replace is leveraged:</span></p>
<ul><li style="	font-weight: 400;
"><span style="	font-weight: 400;

"><span style="	font-weight: 400;

</li></ul>
<p><span style="	font-weight: 400;

<h4><span style="	font-weight: 400;
">Type 52</span></h4>
<p><span style="	font-weight: 400;
">Type 52 requests appear to check a URL for a substring and then extract that data, but the BokBot samples used to build this report did not contain a Type 52 rewrite.</span></p>
<h4><span style="	font-weight: 400;
">URL Rewrite Bypass</span></h4>
<p><span style="	font-weight: 400;
">It would become an issue if the same webfake process is used every time a victim attempts to log into a targeted website. Once the victim’s information has been collected, there is no longer a need to rewrite URLs to point towards the webfake site. Instead, the request should be sent, unmodified, to the real website. To ensure this happens, BokBot, maintains a list of subkeys located under the HKCU\Software\Classes\CLSID registry key.</span> <span style="	font-weight: 400;
">For this to work, a unique name needs to be generated for each of the target sites. Each of the inject types included in this category contains a Webfake Template field, for example:</span> <img class="alignnone size-full wp-image-22876" src="/content/dam/crowdstrike/www/en-us/wp/2019/03/code-13.png" width="672" height="86"/> <span style="	font-weight: 400;
">The highlighted value is used as a seed to generate a unique registry key name (covered in more detail in the “Verification Requests” section). Each of the target sites contains either a value similar to the one highlighted above, or a simple identifier that uses the URL hostname (e.g., “amazon”).</span></p>
<p><span style="	font-weight: 400;
">If the registry subkey exists, the proxy server sends a request to the C2 to determine if the request should be rewritten. If the response from the C2 is anything other than one of the two bytes — “0x2D” or “0x2B” — the legitimate URL will not be rewritten to point to the webfake site.</span></p>
<h3><span style="	font-weight: 400;

<p><span style="	font-weight: 400;

<h4><span style="	font-weight: 400;
">Types 97 and 98</span></h4>
<p><span style="	font-weight: 400;

<h4><span style="	font-weight: 400;

<p>These requests are either HTTP POST or HTTP GET requests that contain the information collected from the malicious javascript code.</p>

">main.js</span></code><span style="	font-weight: 400;
"> code when the redirected website responds to the request. Now all of the necessary components are in place to collect the target’s account information.</span> <span style="	font-weight: 400;
">Once form data is collected, Validation Type 100 requests are sent and processed by the proxy server. These requests are POST requests with the form data contained within the request body. Once processed by the proxy, the form data will be sent to the C2 by signaling the C2 communication thread.</span> <img class="alignnone size-full wp-image-22885" src="/content/dam/crowdstrike/www/en-us/wp/2019/03/code-21.png" width="672" height="49"/> <span style="	font-weight: 400;
">The request parameters can be parsed by referring back to the “Proxy C2 Communication” section.</span> <span style="	font-weight: 400;
">There are multiple variations of how webinjects are handled by the proxy server. However, this example is sufficient to showcase the entire workflow.</span></p>
<h2><span style="	font-weight: 400;
">Traffic Manipulation: Browser Perspective</span></h2>
<p><span style="	font-weight: 400;
">Due to the multitude of variations for the injected javascript API, and rather than attempting to abstract and present the high-level processes each inject has in common, this section will focus only on continuing the cashanalyzer.com example introduced in the previous section. This analysis can be used to understand how the other injects work.</span> <span style="	font-weight: 400;
">The first couple of sections will cover how the site interacts with the victim, the proxy and the C2. After that is covered, the “Putting it All Together” section will connect the separate pieces.</span></p>
<h4><span style="	font-weight: 400;
">Bot API: Core Javascript Module</span></h4>
<p><span style="	font-weight: 400;
">As previously mentioned, the</span></p>
<p><code><span style="	font-weight: 400;
">main.js</span></code><span style="	font-weight: 400;
"> file contains the Bot API code. This code provides everything necessary to interact with the proxy, the C2 and the webpage HTML elements, to ensure that the victim has a seamless experience while entering the account information.</span></p>
<h4><span style="	font-weight: 400;
">Token States</span></h4>
<p><span style="	font-weight: 400;
">A token state is a numerical indicator used to identify what action the Bot API should take next. There are five token states:</span></p>
<div class="wp-caption" id="attachment_22886"><img class="size-full wp-image-22886" src="/content/dam/crowdstrike/www/en-us/wp/2019/03/Table-5.png" alt="BokBot Token States" width="672" height="204"/>Table 5: Token States</div>
<p><span style="	font-weight: 400;
">As an example, if there is some sort of error, the token state is set to either 3 or 4, and this will trigger the logic in the javascript to call the viewBlock method (next section) and load the “Site Down for Maintenance” page.</span></p>
<h4><span style="	font-weight: 400;
">Page View Architecture</span></h4>
<p><span style="	font-weight: 400;
">In order to ensure that the correct information is collected in the proper order, the Bot API relies on a traditional view/controller architecture. In this case, the controller is the attacker-controlled webfake site. The Bot API client notifies the C2 that it is ready to receive a command, the server responds with a command, and this command is used to load the next page view.</span></p>
<div class="wp-caption" id="attachment_22887"><img class="size-full wp-image-22887" src="/content/dam/crowdstrike/www/en-us/wp/2019/03/Table-6.png" alt="BokBot Page View Commands" width="672" height="426"/>Table 6: Page View Commands</div>
<p><span style="	font-weight: 400;
">Once the command is received by the Bot API, the CSS display field of the relevant HTML element is modified to display this code block. A typical session using the webfake will take the following steps:</span></p>
<ol><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Login page view is loaded</span><ol><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Victim enters the account information and submits the form</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">A wait page is displayed as the login form is processed</span></li></ol>
</li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Security token page view is loaded</span><ol><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Victim enters the security token</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">A wait page is displayed as the security token form is processed</span></li></ol>
</li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">First name/last name/phone number page view is loaded</span><ol><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Victim enters this information</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Another wait page is displayed as this form is processed</span></li></ol>
</li></ol>
<p><span style="	font-weight: 400;
">An example of this will be covered in the “Putting it All Together” section below.</span></p>
<h4><span style="	font-weight: 400;
">Wait Pages</span></h4>
<p><span style="	font-weight: 400;
">After the victim attempts to log in to the website, there is an idle period as the proxy forwards the request to the webfake site, and that site forwards the request to the legitimate website.</span></p>
<p>To avoid causing any concern, the Bot API displays a wait page.</p>
<div class="wp-caption" id="attachment_22888"><img class="size-full wp-image-22888" src="/content/dam/crowdstrike/www/en-us/wp/2019/03/Figure-5-Post-login-wait.png" alt="Example of a BokBot Wait Page" width="672" height="497"/>Figure 5: Post-Login Wait Page</div>
<p><span style="	font-weight: 400;
">There are multiple versions of this site, one for each step of the account information collection process: login information; security token information; security question answers; and the form that collects the victim’s name and address.</span></p>
<h4><span style="	font-weight: 400;
">Communication: Bot API Requests to the C2</span></h4>
<p><span style="	font-weight: 400;
">Commands are forwarded to the C2 on behalf of the Bot API by the proxy server using Validation Type 100 requests. Contained within these requests is a base64-encoded string. This string decodes to a series of parameters in JSON format.</span> <img class="alignnone size-full wp-image-22889" src="/content/dam/crowdstrike/www/en-us/wp/2019/03/code-22.png" width="672" height="287"/> <span style="	font-weight: 400;
">Table 7 describes the meaning of each of these parameters. In this case, the Bot API is notifying the C2 that the page has not been initialized (invalid).</span></p>
<div class="wp-caption" id="attachment_22890"><img class="size-full wp-image-22890" src="/content/dam/crowdstrike/www/en-us/wp/2019/03/Table-7.png" alt="BokBot URL Parameters for Bot API C2 Communication" width="672" height="434"/>Table 7: URL Parameters for Bot API C2 Communication</div>
<p><span style="	font-weight: 400;
">Heartbeat (Type 1) requests are sent out at regular intervals by the Bot API. These requests are used as a means to maintain constant communication with the C2 and provide a medium for the Bot API to receive commands.</span></p>
<h4><span style="	font-weight: 400;
">Communication: C2 Responses to the Bot API</span></h4>
<p><span style="	font-weight: 400;
">Responses from the C2 contain commands specifying what actions the Bot API should take, based on the initial request. After each form submission, a command is received from the C2 telling the Bot API which page to load next. These responses are also base64-encoded JSON.</span> <img class="alignnone size-full wp-image-22891" src="/content/dam/crowdstrike/www/en-us/wp/2019/03/code-23.png" width="672" height="51"/> <span style="	font-weight: 400;
">In this case, the command ID is “99” and it is telling the Bot API to reload the original site. This will reset all of the forms, clear any site blocks and load the original page. Table 8 covers the various fields of the C2 response:</span></p>
<div class="wp-caption" id="attachment_22893"><img class="size-full wp-image-22893" src="/content/dam/crowdstrike/www/en-us/wp/2019/03/Table-8.png" width="672" height="321"/>Table 8: C2 Response Fields</div>
<p><span style="	font-weight: 400;
">As can be seen, the </span><code><span style="	font-weight: 400;
">id</span></code><span style="	font-weight: 400;
"> field lines up with the view commands table (Table 6) covered in the “Page View Architecture” section.</span></p>
<h4><span style="	font-weight: 400;
">Site Maintenance Page</span></h4>
<p><span style="	font-weight: 400;
">Whenever the token state of the Bot API is set to “3” or “4,” an error page stating that there are technical issues is displayed (Figure 6). The token is set to one of those two states whenever there is an issue communicating with the C2, the webfake, the proxy server, or if an error occurs when the webfake communicates with the real website.</span></p>
<div class="wp-caption" id="attachment_22894"><img class=" wp-image-22894" src="/content/dam/crowdstrike/www/en-us/wp/2019/03/Figure-6-Proxy-Stie-Down.png" alt="Example of a BokBot Site Maintenance Page" width="658" height="319"/>Figure 6: BokBot Proxy “Site Down for Maintenance” Page</div>
<p><span style="	font-weight: 400;
">The fix date is generated by adding one hour to the current time. After displaying the website, this timestamp is set in the registry by sending a Type 96 Verification request. Once the timer expires, a Type 98 Verification request is sent to delete the block time from the registry.</span></p>
<h4><span style="	font-weight: 400;
">Logging</span></h4>
<p><span style="	font-weight: 400;
">The Bot API is constantly sending logging data back to the C2 as base64-encoded JSON. The following is the decoded JSON object:</span> <img class="alignnone size-full wp-image-22895" src="/content/dam/crowdstrike/www/en-us/wp/2019/03/code-24.png" width="672" height="141"/> <span style="	font-weight: 400;
">This chunk of data has a series of </span><code><span style="	font-weight: 400;
">key:value</span></code><span style="	font-weight: 400;
"> pairs, where the h key contains the log, as a </span><code><span style="	font-weight: 400;
">value</span></code><span style="	font-weight: 400;
">. The logs are descriptive and useful for the malicious actor.</span></p>
<h3><span style="	font-weight: 400;
">Putting it All Together</span></h3>
<h4><span style="	font-weight: 400;
">Initial Browser Request</span></h4>
<p><span style="	font-weight: 400;
">The victim’s browser sends a request for cashanlyzer.com, and that request is intercepted by the proxy, rewritten, sent to the webfake site, and the response is sent from the proxy to the victim. The page content also contains the following javascript code to load a javascript file from the attacker-controlled site:</span> <img class="alignnone size-full wp-image-22897" src="/content/dam/crowdstrike/www/en-us/wp/2019/03/code-25.png" width="672" height="131"/> <span style="	font-weight: 400;
">This </span><code><span style="	font-weight: 400;
">main.js</span></code><span style="	font-weight: 400;
"> request is forwarded by the proxy and has the following code injected into the response:</span> <img class="alignnone size-full wp-image-22898" src="/content/dam/crowdstrike/www/en-us/wp/2019/03/code-26.png" width="672" height="74"/> <span style="	font-weight: 400;
">This code will call the </span><code><span style="	font-weight: 400;
">String.prototype.init</span></code><span style="	font-weight: 400;
"> method that is defined in </span><code><span style="	font-weight: 400;
">main.js</span></code><span style="	font-weight: 400;
">. In this case, the init method contains the malicious Bot API javascript module, and will end up calling the main function to initialize and load the webfake site.</span></p>
<div class="wp-caption" id="attachment_22899"><img class=" wp-image-22899" src="/content/dam/crowdstrike/www/en-us/wp/2019/03/Figure-7-cashanalyzer-fake-login.png" alt="Example of a BokBot Webfake Login Page" width="680" height="264"/>Figure 7: Cashanalyzer Fake Login Page</div>
<p><span style="	font-weight: 400;
">After checks are performed to determine if a block page needs to be loaded, the Bot API calls the viewStep method and passes it the string “login.” The string passed to this method is the element </span><code><span style="	font-weight: 400;
">id</span></code><span style="	font-weight: 400;
"> of the HTML &lt;div&gt; object that contains the login form.</span> <img class="alignnone size-full wp-image-22900" src="/content/dam/crowdstrike/www/en-us/wp/2019/03/code-27.png" width="672" height="139"/> <span style="	font-weight: 400;
">This display field of this element will be switched from </span><code><span style="	font-weight: 400;
">none</span></code><span style="	font-weight: 400;
"> to </span><code><span style="	font-weight: 400;
">block</span></code><span style="	font-weight: 400;
"> and the login page is displayed (Figure 7). Now the victim can enter the login information and click “continue.” All of the data entered is placed into a C2 request structure and sent to the webfake site.</span> <img class="alignnone size-full wp-image-22901" src="/content/dam/crowdstrike/www/en-us/wp/2019/03/code-28.png" alt="BokBot data in C2 request structure code" width="672" height="203"/> <span style="	font-weight: 400;
">The first highlighted section above is the user account credentials, and the second is the logging information. This information notifies whoever is on the other end that the login view was loaded, and that the last view is the wait page.</span> <span style="	font-weight: 400;
">The wait page that the victim is seeing churns as the Bot API continuously sends heartbeat requests, waiting for the response from the webfake site.</span> <img class="alignnone size-full wp-image-22902" src="/content/dam/crowdstrike/www/en-us/wp/2019/03/code-29.png" alt="BokBot API heartbeat request code" width="672" height="92"/> <span style="	font-weight: 400;
">After processing is finished, the webserver responds with a command to disable the login page, and make the next page viewable. In this case, the command is to load the user contact information form.</span> <img class="alignnone size-full wp-image-22903" src="/content/dam/crowdstrike/www/en-us/wp/2019/03/code-30.png" alt="BokBot Proxy command code to load user contact info form" width="672" height="53"/> <span style="	font-weight: 400;
">The rest of the steps are self explanatory: The subsequent actions consist of forms leading to a wait time, followed by additional forms until the last step is reached.</span></p>
<h4><span style="	font-weight: 400;
">Wrapping Up the Process</span></h4>
<p><span style="	font-weight: 400;
">Once all of the information has been collected, the Bot API sends a Validation Type 96 request to the proxy server: However, instead of a timestamp, the value being set in the registry is the string “true,” as highlighted below.</span> <img class="alignnone size-full wp-image-22904" src="/content/dam/crowdstrike/www/en-us/wp/2019/03/code-31.png" width="672" height="369"/> <span style="	font-weight: 400;
">This request instructs the proxy server to create the ‘</span><code><span style="	font-weight: 400;
">HKCU\Software\Classes\CLSID\{7570CC99-D32B-6883-1375-9D2881583EFB)</span></code>&#39;<span style="	font-weight: 400;
"> registry key with the (<code>Default</code>)value, set to a four-byte binary value.</span></p>
<p>Once this is set, any further attempt to access bypasses any attempt to rewrite the URL (inject Types 50, 51, 52), and will load the legitimate website. <span style="	font-weight: 400;
">To load the legitimate website, the page is refreshed and now the victim will be able to login to the legitimate cashanalyzer.com website. Of course, if any of the other webinjects match on the site, other actions will be taken (collecting account balance data, etc.).</span></p>
