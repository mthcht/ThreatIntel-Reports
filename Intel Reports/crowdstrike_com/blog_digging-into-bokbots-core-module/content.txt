<div class="cmp-text" id="text-c66b0d0dee" data-cmp-name="cmp-text-" data-cmp-is="text" data-target-location="false">
    
       <span><h2><span style="	font-weight: 400;
">Introduction</span></h2>
<p>BokBot, developed and operated by the actor named LUNAR SPIDER, was first observed in 2017 and the CrowdStrike’s Falcon<sup>®</sup> Overwatch™ and Falcon Intelligence™ teams have analyzed these infections to ensure customers are both protected and informed. Recently, BokBot infections have become more prevalent due to distribution campaigns through the Emotet malware, which is associated with MUMMY SPIDER. The BokBot malware provides robust functionality, such as:</p>

<h2><span style="	font-weight: 400;
">BokBot Container Execution</span></h2>
<p><span style="	font-weight: 400;
">BokBot comes packed inside a crypter. The crypter goes through several stages before finally unpacking the BokBot binary and injecting it into svchost.exe. Here is a quick rundown of the different stages:</span></p>
<ul><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Stage 1 (crypter)</span><ul><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Decode stage 2 and execute</span></li></ul>
</li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Stage 2 (crypter)</span><ul><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Decodes shellcode and execute</span></li></ul>
</li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Stage 3 (shellcode)</span><ul><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Hollows out the base process image</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Decodes the core process injection PE</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Overwrites the base process image with the core process injection PE</span></li></ul>
</li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Stage 4 (process injection)</span><ul><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Executes the process injection code</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Launches svchost.exe child process</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Injects BokBot as a headless PE image into the child process</span></li></ul>
</li></ul>
<p><span style="	font-weight: 400;
">All of the behaviors relevant to the CrowdStrike® Falcon platform occur in stage 4. The primary focus for the following section is the unique method in which BokBot is injected into the child process.</span></p>
<h3><span style="	font-weight: 400;
">Process Injection</span></h3>
<p><span style="	font-weight: 400;
">In order to bypass antivirus (AV) detections for process hollowing, BokBot hooks several Windows API functions, executes the hooking code, and then removes the hook.</span></p>
<h4><span style="	font-weight: 400;
">Simulating Process Hollowing</span></h4>
<p><span style="	font-weight: 400;
">In order to simulate process hollowing, the </span><code><span style="	font-weight: 400;
">ZwCreateUserProcess </span></code><span style="	font-weight: 400;
">routine is hooked. BokBot calls </span><code><span style="	font-weight: 400;
">ZwProtectVirtualMemory </span></code><span style="	font-weight: 400;
">to modify the permissions of the routine to </span><code><span style="	font-weight: 400;
">PAGE_READWRITE</span></code><span style="	font-weight: 400;
">. Next, the first five opcodes (bytes) are replaced with the opcodes for a </span>JMP <code style="	font-size: 14.0px;
">&lt;address of hooking code&gt;</code> instruction.</p>
<p>Permissions are restored, and then <code style="	font-size: 14.0px;
">CreateProcessA </code>is called.</p>
<p> </p>
<p><img class="wp-image-22066 " src="/content/dam/crowdstrike/www/en-us/wp/2019/01/Figure-1-Bok-Bot-j.jpg" alt="" width="836" height="414"/>Figure 1: Hooking <code>ZwCreateUserProcess</code> <span style="	font-weight: 400;
">Once </span><code><span style="	font-weight: 400;
">CreateProcessA </span></code><span style="	font-weight: 400;
">is called, a function call chain leads to calling </span><code><span style="	font-weight: 400;
">ZwCreateUserProcess </span></code><span style="	font-weight: 400;
">and then the hooking code, as shown in Figure 1. At this point, no process has been created.</span> <span style="	font-weight: 400;
">The hooking code will complete the creation of the child process by removing the hook from the </span><code><span style="	font-weight: 400;
">ZwCreateUserprocess </span></code><span style="	font-weight: 400;
">routine, and then the non-hooked </span><code><span style="	font-weight: 400;
">ZwCreateUserProcess </span></code><span style="	font-weight: 400;
">procedure is called. This will create the child process, but execution doesn’t begin until </span><code><span style="	font-weight: 400;
">CreateProcessInternal </span></code><span style="	font-weight: 400;
">returns. The rest of the hook routine will decode and inject the embedded BokBot binary into the child svchost.exe process.</span></p>
<p>Code Injection</p>
<p><span style="	font-weight: 400;
">Prior to injecting the code, the BokBot PE is decompressed and loaded into the local process memory. Once loaded, the following Windows procedures are used to allocate and write to the svchost child process:</span> <span style="	font-weight: 400;
"><img class="alignnone wp-image-22076" src="/content/dam/crowdstrike/www/en-us/wp/2019/01/ZwallocateVM-1024x157.jpg" alt="" width="811" height="124"/></span> <span style="	font-weight: 400;
">After the main BokBot module has been written to the child process, the steps to execute the BokBot code will begin.</span></p>
<h4><span style="	font-weight: 400;
">Code Execution</span></h4>
<p><span style="	font-weight: 400;
">BokBot uses a novel technique to get the code to execute inside of the child process. Using the same APIs as earlier, the dropper hooks </span><code><span style="	font-weight: 400;
">RtlExitUserProcess</span></code><span style="	font-weight: 400;
"> in the child process. Since svchost.exe is launched without arguments, it will terminate immediately. As the process attempts to exit, it will call the hooked </span><code><span style="	font-weight: 400;
">RtlExitUserProcess</span></code><span style="	font-weight: 400;
">, thus executing the BokBot payload.</span></p>
<div class="wp-caption" id="attachment_22069"><img class="wp-image-22069 " src="/content/dam/crowdstrike/www/en-us/wp/2019/01/Figure-2-Bok-Bot-rev.jpg" alt="" width="733" height="499"/><center>Figure 2: Executing BokBot with <code>RtlExitUserProcess</code> Hook</center>
</div>
<p><span style="	font-weight: 400;
">There is one more task for the hooking routine to complete before </span><code><span style="	font-weight: 400;
">CreateProcessInternalW</span></code><span style="	font-weight: 400;
"> resumes execution.</span></p>
<h4><span style="	font-weight: 400;
">Injecting a Context Data Structure</span></h4>
<p><span style="	font-weight: 400;
">After the BokBot payload is injected into the child process, a context data structure is written to the child process. This context contains all of the data necessary to ensure that BokBot’s main module is able to execute without issue:</span></p>
<ul><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Windows </span>Procedure Addresses<ul><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">ntdll.ZwAllocateVirtualMemory</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">ntdll.ZwWriteVirtualMemory</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">ntdll.ZwProtectVirtualMemory</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">ntdll.ZwWaitForSingleObject</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">ntdll.LdrLoadDll</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">ntdll.LdrGetProcedureAddress</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">ntdll.RtlExitUserProcess</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">ntdll.ZwCreateUserProcess</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">ntdll.RtlDecompressBuffer</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">ntdll.ZwFlushInstructionCache</span></li></ul>
</li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Load address for payload</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Path to the dropper binary</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">C2 URLs</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Project ID</span></li></ul>
<p><span style="	font-weight: 400;
">This data is collected throughout the lifetime of the dropper process. In addition, a similar structure will be written to the child processes of BokBot as it downloads and execute modules.</span> <span style="	font-weight: 400;
">After injection, </span><code><span style="	font-weight: 400;
">CreateProcessInternalW</span></code><span style="	font-weight: 400;
"> resumes, and the dropper process exits. BokBot’s main module starts the initialization phase.</span></p>
<h2><span style="	font-weight: 400;
">BokBot Initialization</span></h2>
<p><span style="	font-weight: 400;
">Prior to executing the primary loop to communicate with the C2, BokBot goes through several initialization steps to prepare itself for C2 communication. Initialization occurs in the following steps:</span></p>
<ul><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Remove the RtlExitUserProcess hook</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Create a memory-mapped file to store logging data</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Execute BokBot as the logged-on user (if the current process is running as System)</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Suppress error windows</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Collect System information</span><ul><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Windows version information</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">User SID</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Member of a domain</span></li></ul>
</li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Unique ID generation</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Prevent multiple executions</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Install BokBot on the host</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Inject existing downloaded modules into child processes</span></li></ul>
<p><span style="	font-weight: 400;
">Some of these steps are covered in more details inside the following sections.</span></p>
<h3><span style="	font-weight: 400;
">Silence Errors</span></h3>
<p><span style="	font-weight: 400;
">To prevent error windows from informing the victim of an issue, BokBot sets the error mode of the process to 0x8007, which corresponds to the following:</span> <span style="	font-weight: 400;
"><img class="alignnone wp-image-22077" src="/content/dam/crowdstrike/www/en-us/wp/2019/01/SilenceErrors-1024x180.jpg" alt="" width="825" height="145"/></span> <span style="	font-weight: 400;
">This will disable most error notices that are generated when a process crashes.</span></p>
<h3><span style="	font-weight: 400;
">Generating Unique IDs</span></h3>
<p><span style="	font-weight: 400;
">BokBot uses several unique IDS that are generated earlier on during process execution. These values are passed to the C2 (command and control), used as a key for RC4, and passed to child processes.</span></p>
<h4><span style="	font-weight: 400;
">Project ID</span></h4>
<p><span style="	font-weight: 400;
">In addition to injecting the main BokBot module into svchost, the dropper also injects a chunk of binary data that provides context for BokBot to execute, including the Project ID. These unique Project ID values appear to be used to identify infections that correspond to distribution campaigns. The Project ID is a four-byte value.</span></p>
<h4><span style="	font-weight: 400;
">Bot ID</span></h4>
<p><span style="	font-weight: 400;
">Bot IDs are unique to specific instances for a user on an infected host. The value is used as an encryption key and as a seed in the generation of the unique values that BokBot needs for a variety of purposes, such as the generation of pseudo-random strings for file and event names. This will be discussed further in subsequent sections.</span> <span style="	font-weight: 400;
">The Bot ID is generated in one of the two following ways:</span></p>
<ul><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Security ID (SID) of the account name</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">System time in a file time format</span></li></ul>
<p><span style="	font-weight: 400;
">Since both values are 64-bit, no matter which method is used, the value is split into two 32-bit chunks and XORed.</span></p>
<h4><span style="	font-weight: 400;
">ID Hash</span></h4>
<p><span style="	font-weight: 400;
">In addition to this Bot ID, a simple hash is generated that can be used to verify the validity of both the Bot ID and the Project ID. This hash is generated using the Bot ID and the Project ID, in the following manner:</span> <img class="alignnone wp-image-22078" src="/content/dam/crowdstrike/www/en-us/wp/2019/01/ID_Hash-1024x220.jpg" alt="" width="874" height="188"/> This value will be passed along with the Project ID and the Bot ID as part of the C2 URL parameters. If this request is invalid, infected hosts will not receive any instructions from the C2.</p>
<h2><span style="	font-weight: 400;
">C2 Hostname Initialization</span></h2>
<p><span style="	font-weight: 400;
">Bokbot contains an encoded list of C2 hostnames that were provided as part of the context data structure that was injected by the dropper. The C2 list within that structure is decoded using a key that was also provided by the context, and then re-encoded using a new key that was generated using an rdtsc instruction, and stored as an array of pointers.</span></p>
<h2><span style="	font-weight: 400;
">Prevent Multiple Executions</span></h2>
<p><span style="	font-weight: 400;
">A unique global named event is generated using the Bot ID. A successful call to </span><code><span style="	font-weight: 400;
">CreateEvent</span></code><span style="	font-weight: 400;
"> is proceeded with a call to </span><code><span style="	font-weight: 400;
">GetLastError</span></code><span style="	font-weight: 400;
">. If the malware is already executing, the last error is </span><code><span style="	font-weight: 400;
">ERROR_ALREADY_EXISTS</span></code><span style="	font-weight: 400;
">, and the process exits.</span></p>
<h2><span style="	font-weight: 400;
">Installation</span></h2>
<p><span style="	font-weight: 400;
">During installation, the BokBot dropper binary is written to an installation directory, and a scheduled task is created for persistence.</span> <span style="	font-weight: 400;
">The installation directory is created in the following root directory:</span></p>
<ul><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">c:\ProgramData</span></li></ul>
<p><span style="	font-weight: 400;
">The installation directory name is unique and generated using the Bot ID. Once the directory is created, the original dropper file is renamed (also using the Bot ID as a seed) and written to the directory. Because the Bot ID is based on system information, using it as a seed ensures that the malware will always generate the same installation path and filename on a particular host.</span> <span style="	font-weight: 400;
">After generating the installation directory name, BokBot needs to generate a filename for the BokBot binary that is going to be written to that directory. The following Python code reproduces the algorithm that BokBot uses to generate the filename, and various other strings.</span> <span style="	font-weight: 400;

"><span style="	font-weight: 400;
">Update BokBot</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Write to a file</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Download and execute a binary</span></li></ul>
<p><span style="	font-weight: 400;
">The commands that use the C2 URL hostnames send a d URL parameter, such as the following example:</span> <img class="alignnone wp-image-22093" src="/content/dam/crowdstrike/www/en-us/wp/2019/01/C2-url-hostname.png" alt="" width="788" height="120"/> <span style="	font-weight: 400;
">This value is typically set to 0; the file to download is specified by the g parameters.</span></p>
<h2><span style="	font-weight: 400;
">Modules and DAT Files</span></h2>
<p><span style="	font-weight: 400;
">All data received from the C2 that needs to persist between reboots is written out as a DAT file on the infected machine. These files include:</span></p>
<ul><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Web inject configuration</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">C2 configuration</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">External modules</span></li></ul>
<p><span style="	font-weight: 400;
">Each file is encrypted and decrypted as needed by either the main module or the child module, using the Bot ID as the key. Each module is given a unique tag.</span></p>
<h3><span style="	font-weight: 400;
">Unique Tag Generation</span></h3>
<p><span style="	font-weight: 400;
">BokBot assigns unique tag values for injected processes, downloaded modules, and the downloaded DAT files. These tags are a convenient method for the executing BokBot process to identify external process resources. Tag generation is simple:</span></p>
<ul><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">18 – Web injects configuration file, statically defined in the binary</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">19 – Reporting configuration file, statically defined in the binary</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">20 – C2 configuration file, statically defined in the binary</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">33-46 – Downloaded modules to be injected into child processes</span><ul><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Assigned as needed in an incremental fashion</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Not necessarily a unique tag for what the module does</span></li></ul>
</li></ul>
<p><span style="	font-weight: 400;
">During analysis of BokBot, these values will come up on a regular basis, including values to generate a unique filename, as described later.</span></p>
<h2><span style="	font-weight: 400;
">Downloading DAT Files</span></h2>
<p><span style="	font-weight: 400;
">As previously mentioned, DAT files are downloaded based on commands sent from the C2. Once the command is received from the C2, a command handler specific to this command is called to process the request. In response, the infected machine notifies the C2 with the command that it is ready to receive an RC4-encrypted blob</span> <span style="	font-weight: 400;
">from the C2. Figure 3 illustrates the process of commands that download configuration files and modules.</span></p>
<div class="wp-caption" id="attachment_22094"><img class="wp-image-22094 size-full" src="/content/dam/crowdstrike/www/en-us/wp/2019/01/Figure-3-Bok-Bot.jpg" alt="" width="741" height="348"/><center>Figure 3: C2 Command to Trigger DAT File Download</center>
</div>
<p><span style="	font-weight: 400;
">An eight-byte RC4 key is prepended to the data buffer. Prior to writing the BLOB to a file, BokBot decrypts the file, and then re-encrypts it using a new RC4 key based on the Bot ID.</span></p>
<h3><span style="	font-weight: 400;
">Write to a File</span></h3>
<p><span style="	font-weight: 400;
">BokBot creates a new directory under </span><code><span style="	font-weight: 400;
">C:\ProgramData</span></code><span style="	font-weight: 400;
"> to store the DAT files. The directory name is generated using the string generation algorithm described previously. DAT file names are generated using the unique tag value. This value is run through a string generation algorithm (also dependent on the Bot ID), which returns a unique filename for the DAT file.</span></p>
<div class="wp-caption" id="attachment_22095"><img class="wp-image-22095 size-full" src="/content/dam/crowdstrike/www/en-us/wp/2019/01/Table-4-Bok-Bot.jpg" alt="" width="816" height="280"/><center>Table 4: Example of BokBot DAT Files Written During Analysis</center>
</div>
<p><span style="	font-weight: 400;
">Table 4 references all of the DAT files that were written during the testing process used for writing this blog. In this case, the installation directory is C:\ProgramData\yyyyyyyyiu\.</span> <span style="	font-weight: 400;
">These DAT files are further handled based on the specified type, depending on whether it is an executable module or a data file.</span></p>
<h3><span style="	font-weight: 400;
">Executable Module</span></h3>
<p><span style="	font-weight: 400;
">BokBot has several executable modules that can be downloaded and injected into a svchost.exe child process. Once the relevant DAT file is decoded using RC4, no additional decoding or decompression is necessary for the executable module DAT files. The executable module header contains information necessary to ID the module:</span> <img class="alignnone wp-image-22096" src="/content/dam/crowdstrike/www/en-us/wp/2019/01/module-ID.png" alt="" width="727" height="211"/> <span style="	font-weight: 400;
">The rest of the file contains data necessary to load and execute the module, including the various portions of a PE file along with a custom PE header.</span></p>
<h4><span style="	font-weight: 400;
">Module Injection and Execution</span></h4>
<p><span style="	font-weight: 400;
">Executable modules are injected with a technique similar to the dropper, minus the hook of <code>ZwCreateUserProcess</code>, and the child process start is suspended (</span><code><span style="	font-weight: 400;
">CREATE_SUSPENDED</span></code><span style="	font-weight: 400;
">). It’s a little closer to traditional process migration with the addition of the </span><code><span style="	font-weight: 400;
">RtlExitUserProcess</span></code><span style="	font-weight: 400;
"> hook.</span></p>
<h5><span style="	font-weight: 400;
">PE Image Loading</span></h5>
<p><span style="	font-weight: 400;
">Because there is no standard PE header, the DAT file has to contain all of the relevant information (virtual sizes, relocations, etc.) to properly map this binary into the child process. This data is part of the header of the DAT file. BokBot builds the binary in local process memory prior to injecting it into the child process.</span></p>
<h5><span style="	font-weight: 400;
">Injection</span></h5>
<p><span style="	font-weight: 400;
">Injection uses the same APIs as the dropper: ZwAllocateVirtualMemory, ZwWriteVirtualMemory, ZwProtectVirtualMemory. After injection the process is resumed using ResumeThread.</span></p>
<h5><span style="	font-weight: 400;
">Execution Context Injection</span></h5>
<p><span style="	font-weight: 400;
">Once again, an execution context structure is written to the child process, prior to execution. Some of the information contained in this context includes:</span></p>
<ul><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Bot ID</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Project ID</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">C2 hostnames</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">A URL path format string</span></li></ul>
<p><span style="	font-weight: 400;
">This keeps everything consistent between the parent and child process. No new unique identifiers need to be generated, all of the encryption keys are going to be the same: same hostnames, and even the same URL path. Consistency between parent and child is necessary for the messages sent between the two, using inter-process communication (IPC).</span> <span style="	font-weight: 400;
">After a module is injected into a child process, the first four bytes of the decrypted DAT file are added to an array, used by BokBot to identify which modules are currently executing.</span></p>
<h3><span style="	font-weight: 400;
">Data Files</span></h3>
<p><span style="	font-weight: 400;
">The other DAT files contain data necessary to either communicate with a C2, or related to web injection. Essentially, these files provide whatever additional data the main BokBot process and the executable modules require to accomplish their job.</span></p>
<h4><span style="	font-weight: 400;
">Config File</span></h4>
<p><span style="	font-weight: 400;
">The config file contains all of the data necessary for the BokBot main module to maintain communication with the C2. Once the file is decrypted using the process-specific RC4 key, no additional decompression or decryption is necessary.</span></p>
<h5><span style="	font-weight: 400;
">Signature Verification</span></h5>
<p><span style="	font-weight: 400;
">Each config file comes with a digital signature block, used to verify the integrity of the C2 hostname data. The signature is verified based on the signature verification method outlined in the obfuscations section. The following is an example C2 configuration, with the signature block in red:</span><img class="alignnone size-full wp-image-22099" src="/content/dam/crowdstrike/www/en-us/wp/2019/01/sig-verification.png" alt="" width="836" height="529"/></p>
<h4><span style="	font-weight: 400;
">Web Inject Files</span></h4>
<p><span style="	font-weight: 400;
">There are multiple web inject files. One contains all of the target URL and hostname data, and the second contains regex patterns, as well as the code to inject. These files are both RC4-encrypted and compressed.</span> <img class="alignnone wp-image-22100" src="/content/dam/crowdstrike/www/en-us/wp/2019/01/web-inject.png" alt="" width="709" height="182"/> <span style="	font-weight: 400;
">These files are not parsed by the main BokBot binary, but rather by the intercepting proxy module. The </span><code><span style="	font-weight: 400;
">zeus</span></code><span style="	font-weight: 400;
"> file magic is verified, a buffer is allocated, and then the files are decompressed.</span> <span style="	font-weight: 400;
">A forthcoming blog post on the proxy module will cover decompression and usage of the web injection configuration files.</span></p>
<h2><span style="	font-weight: 400;
">Communication with Child Processes</span></h2>
<p><span style="	font-weight: 400;
">Memory-mapped files and events are used by BokBot to communicate with all child processes that contain an injected module. Through the process of leveraging named events with </span><code><span style="	font-weight: 400;
">CreateEvent, OpenEvent</span></code><span style="	font-weight: 400;
"><code>,</code> and <code>OpenFileMapping</code>, the BokBot main module is able to provide additional information to these child processes.</span></p>
<h3><span style="	font-weight: 400;
">Shared Module Log</span></h3>
<p><span style="	font-weight: 400;
">Modules write to the same shared memory-mapped file. The memory-mapped file is created using a shared name between the parent and child processes. Each process that can generate this name can use it to open the memory-mapped file, and to write data to the shared modules log. Further details are covered in the next section, and specific data written will be covered in the separate module descriptions below. The main module is responsible for clearing the log and sending the data to the C2.</span><img class="alignnone wp-image-22101" src="/content/dam/crowdstrike/www/en-us/wp/2019/01/shared-module.png" alt="" width="709" height="195"/></p>
<h3><span style="	font-weight: 400;
">Module-Specific Communication</span></h3>
<p><span style="	font-weight: 400;
">BokBot’s main module often needs to issue commands to the child processes that contain injected module code. The commands can trigger an update of module-specific data, or instruct the module to perform a specific function, such as harvest data from Outlook. Figure 4 outlines this process, although it will be further explained in the subsequent sections.</span> <img class="wp-image-22102 size-full" src="/content/dam/crowdstrike/www/en-us/wp/2019/01/Figure-4-Bok-Bot.jpg" alt="" width="821" height="609"/>Figure 4: BokBot Communication Between Parent and Child Processes</p>
<h4><span style="	font-weight: 400;
">Event Name Generation</span></h4>
<p><span style="	font-weight: 400;
">In order for the BokBot main modules and the child process to communicate with events, unique names need to be generated and must consistent across all of the processes. Table 5 illustrates BokBot’s approach.</span></p>
<div class="wp-caption" id="attachment_22103"><img class="wp-image-22103 size-full" src="/content/dam/crowdstrike/www/en-us/wp/2019/01/Table-5-Bok-Bot.jpg" alt="" width="757" height="370"/><center>Table 5: Event Name Structure</center>
</div>
<p><img class="alignnone wp-image-22104" src="/content/dam/crowdstrike/www/en-us/wp/2019/01/event-name-structure.png" alt="" width="723" height="127"/><span style="	font-weight: 400;
">These events will be used by the parent and child processes to exchange data.</span></p>
<h4><span style="	font-weight: 400;
">BokBot Main Module</span></h4>
<p><span style="	font-weight: 400;
">This process has the ability to communicate with all of the children of the injected modules. These communication all revolve around commands generated by the C2. Once a command that requires notification of an executable module child process is initiated, a named </span><code><span style="	font-weight: 400;
">Q</span></code><span style="	font-weight: 400;
"> event is opened to ensure that the child process is ready to receive the data. If this </span><code><span style="	font-weight: 400;
">Q</span></code><span style="	font-weight: 400;
"> event does not exist, then the child process has not been started. BokBot injects the target module into a child process, and loops a check to see if the event can be opened.</span> <span style="	font-weight: 400;
">Once the </span><code><span style="	font-weight: 400;
">Q</span></code><span style="	font-weight: 400;
"> event has been successfully opened, BokBot creates a new named </span><code><span style="	font-weight: 400;
">R</span></code><span style="	font-weight: 400;
"> event, creates a memory-mapped file (named </span><code><span style="	font-weight: 400;
">M</span></code><span style="	font-weight: 400;
"> event), writes data to the file, signals the open </span><code><span style="	font-weight: 400;
">Q</span></code><span style="	font-weight: 400;
"> event, and waits for a response from the child process. After the child clears the </span><code><span style="	font-weight: 400;
">R</span></code><span style="	font-weight: 400;
"> event, the memory-mapped file is unmapped, and all handles are closed.</span></p>
<h4><span style="	font-weight: 400;
">BokBot Executable Module</span></h4>
<p><span style="	font-weight: 400;
">After initialization, the child process will create a named </span><code><span style="	font-weight: 400;
">Q</span></code><span style="	font-weight: 400;
"> event and wait until it is signaled by the parent process. Once signaled, the named </span><code><span style="	font-weight: 400;
">R</span></code><span style="	font-weight: 400;
"> event is opened, and the data in the memory-mapped file is processed.</span></p>
<h5><span style="	font-weight: 400;
">Data from the BokBot Parent</span></h5>
<p><span style="	font-weight: 400;
">BokBot’s main module writes some contextual information to the injected module, telling it to perform specific actions. These actions change based on the module receiving the data. The following commands are consistent between modules, but the actions performed may vary:</span></p>
<ul><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">0xFF00: Process exit with a 0x1122 code</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">0xFF01: Check web injects or no operation</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">0xFF02: Update C2 hostnames</span></li></ul>
<p><span style="	font-weight: 400;
">In addition to a command, relevant data associated with a command is also processed based on whatever instruction the command tells the injected module to accomplish.</span> <span style="	font-weight: 400;
">After the task assigned by the parent process has completed, the memory mapped file is unmapped, the </span><code><span style="	font-weight: 400;
">R</span></code><span style="	font-weight: 400;
"> event is signaled, and all other open events are closed.</span></p>
<h2><span style="	font-weight: 400;
">Obfuscations and TamperProofing</span></h2>
<p><span style="	font-weight: 400;
">Bokbot uses several methods to obfuscate analysis:</span></p>
<ul><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">String obfuscation</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Encrypted DAT files from the server</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Signature verification</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Polymorphism</span></li></ul>
<h3><span style="	font-weight: 400;
">String Obfuscation</span></h3>
<p><span style="	font-weight: 400;
">To make analysis more difficult, significant strings have been XOR encoded using a shifting key algorithm. All encoded strings have the following structure:</span> <img class="alignnone wp-image-22105" src="/content/dam/crowdstrike/www/en-us/wp/2019/01/string-obfuscation.png" alt="" width="786" height="116"/> <span style="	font-weight: 400;
">Here is the algorithm to decode the string (Python):</span><img class="alignnone wp-image-22106" src="/content/dam/crowdstrike/www/en-us/wp/2019/01/string-structure.png" alt="" width="837" height="439"/></p>
<h3><span style="	font-weight: 400;
">Signature Verification</span></h3>
<p><span style="	font-weight: 400;
">Signature verification occurs under two circumstances: updated C2 urls, and updated BokBot binary. In both cases, the process is the same. The verification function receives two things: a 128-byte signature to verify, and the data to verify.</span> <span style="	font-weight: 400;
">First, BokBot creates an MD5 hash of the data requiring verification. Next, an RSA public key embedded in the executing binary is importing via </span><code><span style="	font-weight: 400;
">CryptImportKey</span></code><span style="	font-weight: 400;
">. Once the hash is generated and the key imported, </span><code><span style="	font-weight: 400;
">CryptVerifySignature</span></code><span style="	font-weight: 400;
"> is used to verify the signature. This may be an attempt to prevent some third party from taking over or otherwise disrupting the botnet.</span></p>
<h3><span style="	font-weight: 400;
">Polymorphism</span></h3>
<p><span style="	font-weight: 400;
">Everytime BokBot is installed, prior to it being written to the install directory, the .text section of the binary is modified with junk data and the virtual size is updated. A new checksum is generated to replace the current checksum.</span></p>

">Without preventions enabled the customer will still be notified of the malicious activity, but no action will be taken to prevent the behavior.</span></p>

Turning on process blocking in Falcon Prevent kills the BokBot infection at the parent svchost process. Looking at the process tree in the Falcon UI with process blocking enabled, shows an analyst that the svchost process was prevented. The block message (see Figure 7) that occurs with this preventative action explains why this process was terminated.</span></p>

<h2><span style="	font-weight: 400;
">In Summary</span></h2>
<p><span style="	font-weight: 400;
">BokBot is a powerful banking trojan that provides attackers with a robust feature set. One of the more unique features of BokBot is the method in which it uses to communicate with its child modules. Additional blog posts for BokBot are coming that will contain more information for the downloaded modules.</span></p>
