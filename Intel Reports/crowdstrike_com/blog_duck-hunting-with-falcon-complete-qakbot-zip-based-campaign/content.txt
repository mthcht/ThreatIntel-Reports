       <span><span style="	font-weight: 400;
">This blog is Part 2</span><span style="	font-weight: 400;
"> of a three-part blog series detailing the reemergence and evolution of QakBot in the spring and summer of 2020. In this installment we cover analysis of the QakBot ZIP-based delivery campaign, particularly one example that exhibited a tactical breakdown by the threat actor with a botched downloader. In addition, the </span><a href="/en-us/endpoint-security-products/falcon-complete/"><span style="	font-weight: 400;
">CrowdStrike</span><span style="	font-weight: 400;
">®</span><span style="	font-weight: 400;
"> Falcon Complete™</span></a><span style="	font-weight: 400;
"> team will cover dynamic analysis of two experimental campaigns, one of which also includes an additional stage-two malware, Zloader.<p> </p></span>
<h2><span style="	font-weight: 400;
">Threat Background and Context</span></h2>
<span style="	font-weight: 400;
">As discussed in </span><a href="/en-us/blog/duck-hunting-with-falcon-complete-analyzing-a-fowl-banking-trojan-part-1/"><span style="	font-weight: 400;
">Part 1</span></a><span style="	font-weight: 400;
">, QakBot is an eCrime banking trojan that has the potential to severely impact an organization’s ability to operate. QakBot has the ability to spread laterally throughout a network utilizing a worm-like functionality through brute forcing network shares, brute forcing Active Directory user group accounts or via server message block (SMB) exploitation.<p> </p></span>

<span style="	font-weight: 400;
">QakBot also employs a robust set of anti-analysis features to evade detection and frustrate analysis. Despite these protections, the </span><a href="/en-us/endpoint-security-products/falcon-platform/"><span style="	font-weight: 400;
">CrowdStrike Falcon®</span><span style="	font-weight: 400;
">®</span><span style="	font-weight: 400;
"> platform</span></a><span style="	font-weight: 400;
"> detects and prevents this </span><a href="/en-us/epp-101/malware/"><span style="	font-weight: 400;
">malware</span></a><span style="	font-weight: 400;
"> from completing its execution chain. </span><span style="	font-weight: 400;
">
</span>

<h2><span style="	font-weight: 400;
">Failed Campaigns with “Broken” Downloader — Late April</span></h2>
<span style="	font-weight: 400;
">In early-to-mid-April 2020, Falcon Complete identified an attempted, unsuccessful QakBot campaign. This was possibly an example of a failed development cycle during the threat actor’s retooling efforts.<p> </p></span>

<span style="	font-weight: 400;
">This campaign diverged from the tactics, techniques and procedures (TTPs) observed in the prior, DOC-based campaign. Instead, the delivery tactic includes a .ZIP attachment containing a malicious Visual Basic Script (VBS) dropper; however, due to a failure in error handling within the .VBS, the actor was unable to download and write the payload to disk successfully.</span>

<div class="wp-caption" id="attachment_27049"><a href="/content/dam/crowdstrike/www/en-us/wp/2020/10/Fig-2-QakBot-Part2.png" target="_blank" rel="noopener noreferrer"><img class="size-full wp-image-27049" src="/content/dam/crowdstrike/www/en-us/wp/2020/10/Fig-2-QakBot-Part2-1.png" alt="" width="679" height="177"/></a> Figure 2. The portion of the dropper responsible for writing QakBot to disk (click image to enlarge)</div>

<span style="	font-weight: 400;
">In this instance, the VBS dropper would reach out to an array of sites, beginning with hxxp&lt;:&gt;//millionsawesomeproducts&lt;.&gt;com, in an attempt to download </span><em><span style="	font-weight: 400;
">444444.png </span></em><span style="	font-weight: 400;
">from the distribution server — which is actually a Windows PE file — and would subsequently be written to disk with the name </span><em><span style="	font-weight: 400;
">PaintHelper.exe.</span></em>

<div class="wp-caption" id="attachment_27050"><a href="/content/dam/crowdstrike/www/en-us/wp/2020/10/Fig-3-QuakBot-Part2.png" target="_blank" rel="noopener noreferrer"><img class="size-full wp-image-27050" src="/content/dam/crowdstrike/www/en-us/wp/2020/10/Fig-3-QuakBot-Part2-1.png" alt="" width="1247" height="206"/></a> Figure 3. The malware distribution sites associated with the failed campaign (click image to enlarge)</div>

<span style="	font-weight: 400;
">If a 404 error was returned, the script would move on to the next site in the array, and attempt to download the malware and place it into the user’s </span><em><span style="	font-weight: 400;
">%TEMP%</span></em><span style="	font-weight: 400;
"> directory. The script would then proceed along the kill chain with a persistence mechanism in the form of a scheduled task with a GUID-based naming convention.</span><span style="	font-weight: 400;
"><p> </p></span>

<span style="	font-weight: 400;
">The site in question did not have a DNS record published — and thus, no HTTP response was returned because no connection could be made. Due to an apparent failure in error handling in the QakBot dropper, the script would fail to infect the host and write a zero-byte, innocuous </span><em><span style="	font-weight: 400;
">PaintHelper.exe </span><span style="	font-weight: 400;
">to</span><span style="	font-weight: 400;
"> %TEMP%.</span></em>

<span style="	font-weight: 400;
">It was discovered that via spoofing a DNS response and the subsequent 404 from the site, the QakBot loader would move on to the next site and successfully write its payload — </span><em><span style="	font-weight: 400;
">PaintHelper.exe.</span></em>

<span style="	font-weight: 400;
">Additionally, the dropper would encode the antivirus product in use, the current OS version and other system information into a Base64 string, and utilize GET strings to inform the malware distribution servers of this information, as shown in Figure 4.</span>

<div class="wp-caption" id="attachment_27051"><a href="/content/dam/crowdstrike/www/en-us/wp/2020/10/Fig-4-QakBot-combo-Part2.png" target="_blank" rel="noopener noreferrer"><img class="size-full wp-image-27051" src="/content/dam/crowdstrike/www/en-us/wp/2020/10/Fig-4-QakBot-combo-Part2-1.png" alt="" width="1458" height="646"/></a> Figure 4. Spoofing the DNS response and subsequent 404 to un-break the dropper; decoded Base64 sent as a GET parameter to the server (click image to enlarge)</div>

<span style="	font-weight: 400;
">This appears to be an implementation of hashbusting — a method of obfuscation in which a malware sample is subtly changed on the fly so each sample has a different checksum. As a result, the SHA256 hash of each payload downloaded from the sites in question appeared to be unique. However, the SSDEEP fuzzy hash of this sample was as follows:</span>

<em><span style="	font-weight: 400;
">6144:y2la96gEZbXtD/uY/HmJV8cc0em/wnXPKYGvZxYney3brNLFDPMTJYhr64Fgw:y2JvZbJYRwnXPKvZxYn7hLFPMdV4Fgw</span></em>

<span style="	font-weight: 400;
">Unfortunately, the actors behind QakBot quickly recovered, and a new campaign was launched the week of May 25, 2020, with no such mistakes and far more success. Despite this success, the operators continued development and experimentation with additional delivery tactics as their campaigns persisted. </span>
<h2><span style="	font-weight: 400;
">QakBot Experiments with PicturesViewer.dll and Secondary Payloads<p> </p><p> </p></span></h2>
<span style="	font-weight: 400;
">The following dynamic analysis was conducted on two iterations of QakBot observed and blocked within client environments. During mid-June, the actors behind the QakBot malware experimented with two disparate tactics that each only lasted one day.<p> </p></span>
<ul><li><span style="	font-weight: 400;
">The first anomalous TTP payload was on June 11, 2020. QakBot would drop a Windows PE named </span><em><span style="	font-weight: 400;
">PicturesViewer.dll,</span></em><span style="	font-weight: 400;
"> which differs from the </span><em><span style="	font-weight: 400;
">PicturesViewer.exe</span></em><span style="	font-weight: 400;
"> seen in prior weeks and noted in the previous section.</span></li><li><span style="	font-weight: 400;
">The following day on June 12, QakBot downloaded and executed an additional Windows PE named senate.m4a. Once executed, this binary would install additional malware from the </span><span style="	font-weight: 400;
">ZeuS family</span><span style="	font-weight: 400;
"> known as Zloader, or Zbot.<p> </p></span><span style="	font-weight: 400;
">
</span></li></ul>
<strong>Please Note: Some of the examples in the following scenario have CrowdStrike Falcon® configured with DETECTIONS ONLY and PREVENTIONS off for illustrative purposes. A properly configured Falcon instance as noted above would prevent the activity presented here.</strong>
<h2><span style="	font-weight: 400;

<span style="	font-weight: 400;
">On June 11, QakBot reemerged with new tactics. </span><span style="	font-weight: 400;
">Beginning at approximately 13:47 UTC, </span><span style="	font-weight: 400;

"> <p> </p> MSIExec, spawning multiple Cmd.exe processes and Explorer injection</span></li><li><span style="	font-weight: 400;
"> <p> </p> Commands run by the C2:</span></li></ol>

<h2><span style="	font-weight: 400;
">Dynamic Analysis of QakBot — June 12</span></h2>
<span style="	font-weight: 400;
">On June 12, QakBot continued its evolution. The delivery method of a .ZIP file to malicious .VBS was the same, but this time QakBot also dropped a Zloader payload on its victim. </span><span style="	font-weight: 400;
">Beginning around 14:24 UTC, </span><span style="	font-weight: 400;
">Falcon Complete observed </span><span style="	font-weight: 400;
">QakBot threat actors using a new .VBS payload. Once the user invoked this script, the process tree is as follows.</span>
<ol><li><span style="	font-weight: 400;
"> Wscript.exe</span></li><li><span style="	font-weight: 400;
"> WmiPrivSe.exe</span></li><li><span style="	font-weight: 400;

"> MSIExec, spawning multiple Cmd.exe processes, DNS requests, and writing a few interesting files to disk</span></li><li><span style="	font-weight: 400;
"> Commands run by the C2:</span></li></ol>
<ul><li style="	list-style-type: none;
">
<ul><li style="	list-style-type: none;
">
<ul><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Cmd.exe /c net view /all /domain</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Cmd.exe /c net view /all</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Cmd.exe /c net config workstation</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;

<span style="	font-weight: 400;
">Looking closer at Rundll32.exe, we can see that it’s executing a senate.m4a, DllRegisterServer. </span><span style="	font-weight: 400;
">Further analysis shows senate.m4a is actually a Zloader.dll. At the time of this writing, we are no longer seeing the senate.m4a Zloader being distributed by the QakBot .VBS delivery method.</span>

<span style="	font-weight: 400;
">The name of the value will either be randomly chosen or a GUID, targeting a PE in the %APPDATA%\Roaming\Microsoft\ folder.</span>
<h2><span style="	font-weight: 400;
">Conclusion</span></h2>
<span style="	font-weight: 400;
">As we have seen, QakBot employs a robust set of anti-analysis features and has recently surged in its operational volume within the threat landscape.<p> </p></span>

<span style="	font-weight: 400;
">This blog provided an in-depth analysis of a botched QakBot downloader along with dynamic analysis of experimental payloads that include a secondary malware stage: Zloader. The threat actors behind QakBot, tracked as MALLARD SPIDER, have demonstrated the ability to rapidly retool, implement anti-analysis techniques and develop methods of advanced obfuscation in a short period.<p> </p></span>
