<h2><span style="	font-weight: 400;
">The Evolution of GuLoader</span></h2>
<p><span style="	font-weight: 400;
">GuLoader was first spotted in 2019 as a file downloader that was used to distribute remote access trojans (RATs) such as AgentTesla, FormBook, Nanocore, NETWIRE and the Parallax RAT.</span></p>
<p><span style="	font-weight: 400;
">These early versions of GuLoader were distributed via spam email campaigns containing archived attachments containing the malware in executable form. In 2020, CrowdStrike </span><a href="/en-us/blog/guloader-malware-analysis/"><span style="	font-weight: 400;
">published a detailed analysis of GuLoader</span></a><span style="	font-weight: 400;
"> in which a significant number of DJB2 hash values were mapped, revealing some of the APIs abused by the malware.</span></p>
<p><span style="	font-weight: 400;
">Recent variants started using an updated delivery mechanism where the payload is delivered via a Visual Basic Script (VBS) file. GuLoader also started employing advanced anti-analysis techniques to evade detection, such as anti-debug, anti-sandbox, anti-VM and anti-detection to make analysis difficult.</span></p>
<p><span style="	font-weight: 400;
">By analyzing the new GuLoader samples, we’re able to reveal and understand every anti-analysis and evasion technique being employed and paint a complete picture of the malware’s behavior.</span></p>
<h2><span style="	font-weight: 400;
">GuLoader’s Multistage Deployment</span></h2>
<p><span style="	font-weight: 400;
">The recent GuLoader sample exhibits a multistage deployment:</span></p>
<ul><li style="	font-weight: 400;
"><b>The first stage </b><span style="	font-weight: 400;
">involves using a VBS dropper file to drop a second-stage packed payload into a registry key. It then uses a PowerShell script</span> <span style="	font-weight: 400;
">to execute and unpack the second stage payload from the registry key within memory. </span></li><li style="	font-weight: 400;
"><b>The</b> <b>second stage</b><span style="	font-weight: 400;
"> payload performs all anti-analysis routines (described below), creates a Windows process (e.g., an ieinstal.exe) and injects the same shellcode into the new process.</span></li><li style="	font-weight: 400;
"><b>The third stage</b><span style="	font-weight: 400;
"> reimplements all the anti-analysis techniques, downloads the final payload from a remote server and executes it on the victim’s machine.</span></li></ul>
<p><span style="	font-weight: 400;
">Existing public research on </span><a href="/en-us/blog/guloader-malware-analysis/"><span style="	font-weight: 400;
">GuLoader’s multistage deployment</span></a><span style="	font-weight: 400;
"> has extensively covered a wide range of anti-evasion techniques and behaviors. We used this as a starting point to further our analysis.</span></p>
<h3><span style="	font-weight: 400;
">VBScript</span></h3>
<p><span style="	font-weight: 400;
">The VBScript contains two main arrays. One of them is the shellcode present in hex format that will be injected into memory and the second is a Base64-encrypted PowerShell script.</span> <span style="	font-weight: 400;
">For persistence, this shellcode is then added to the Registry Key (HKEY_CURRENT_USER\SOFTWARE\TYMPANIESI) by the VBScript.</span> <span style="	font-weight: 400;
">Another variable “MEDITABU” with Base64 content is being merged and after decoding it forms a PowerShell script.</span></p>
<h3><span style="	font-weight: 400;
">PowerShell Script</span></h3>
<p><span style="	font-weight: 400;
">The PowerShell script adds a Microsoft .NET class to a PowerShell session using Add-Type -typedefinition. It then reads the shellcode from the registry entry created by the VBScript and loads the shellcode into the virtually allocated memory space using the API ZwAllocateVirtualMemory and RtlMoveMemory. After that, the shellcode is launched inside powershell.exe using the API Callwindowprocw function.</span></p>
<p><span style="	font-weight: 400;
">The first 40 bytes of the shellcode are executable assembly instructions and the remaining bytes are encrypted. The first 40 bytes are responsible for decrypting the remaining part of the code, then the execution flow jumps into the decrypted part. During the next stage, the shellcode replaces the first 40 hex bytes with a no operation (NOP) instruction. This is done to avoid re-debugging of the code.</span></p>
<p><span style="	font-weight: 400;
">Initially, the shellcode traverses the process environment block (PEB) structure and fetches Ntdll to tap into exported NTAPI functions. The rest of the DLLs are loaded by using LoadLibrary. As </span><a href="/en-us/blog/guloader-malware-analysis/"><span style="	font-weight: 400;
">previously covered by CrowdStrike</span></a><span style="	font-weight: 400;
">, GuLoader uses the DJB2 algorithm to load APIs. The assembly code for DJB2 traverses through export functions of the required DLLs one by one, calculates the DJB2 hashes for each export API and then compares those with the hardcoded hash value. We covered the DJB2 algorithm </span><a href="/en-us/blog/guloader-malware-analysis/"><span style="	font-weight: 400;
">in a previous blog post.</span></a></p>
<h2><span style="	font-weight: 400;
">Anti-Analysis Techniques</span></h2>
<p><span style="	font-weight: 400;
">The shellcode employs several anti-analysis and anti-debugging tricks at every step of execution, throwing an error message if the shellcode detects any known analysis of debugging mechanisms.</span></p>
<h3><span style="	font-weight: 400;
">Anti-Debugging</span></h3>
<p><b>GuLoader uses a vectored exception handler (VEH) </b><span style="	font-weight: 400;
">to throw off researchers and make disassembly and debugging difficult by disrupting the normal flow of code execution to point the control flow to incorrect paths, raising exceptions that jump to other instructions. To add the exception, the shellcode uses the RtlAddVectoredExcepitionHandler API function.</span></p>
<div class="wp-caption" id="attachment_145060"><a href="/content/dam/crowdstrike/www/en-us/wp/2022/12/GuLoader1.png" target="_blank" rel="noopener noreferrer"><img class="wp-image-145060 size-full" src="/content/dam/crowdstrike/www/en-us/wp/2022/12/GuLoader1.png" alt="" width="977" height="856"/></a>Figure 1. Vector Exception Handler function (click to enlarge)</div>
<p><span style="	font-weight: 400;
">GuLoader performs a series of anti-debugging and anti-disassembling checks to detect the presence of breakpoints, usually associated with researchers analyzing its code execution flow.</span></p>
<p><span style="	font-weight: 400;
">For example, it extracts information from EXCEPTION_RECORD when it hits INT3 (0xCC) instruction and then it checks ExceptionCode from it. To determine if the VEH routine has been triggered by an INT3 instruction, it will check if the value matches 0x80000003 (e.g., EXCEPTION_BREAKPOINT). It then retrieves the DR registers from CONTEXT_RECORD to check if there are any HARDWARE breakpoints and it also checks the EIP (Extended Instruction Pointer) to see if it is equal to 0xCC. Looking for software breakpoints, GuLoader also checks for the presence of other 0xCC instructions in code and terminates execution if found (shown in Figure 1). If everything is as expected, the malware performs an XOR operation on the next byte after EIP and then replaces the EIP on CONTEXT with the new value, ensuring the execution flow will reach the correct address.</span></p>
<p><span style="	font-weight: 400;
">To bypass this check, to automatically jump to the next real address and avoid the VEH routine, we can use the following statement, inside the “command window” present at the bottom of x32 debugger, when the debugger reaches INT3 instruction (here the XOR value inside VEH was 0x40; it may be different in other samples):</span></p>
<p><code>eip&#61;((ReadByte(eip&#43;1)^0x40)&#43;eip)</code></p>
<p><span style="	font-weight: 400;
">To avoid step-by-step replacement in the debugger, the following script can also be used to patch all of the INT3 instructions by replacing them with a JUMP to the real execution flow (copy the script below in the x32dbg “Script” tab):</span></p>
<pre>
<code>call loop
loop:
mov $a, 0
findasm “int3”
cmp $result, $a
je exit
mov $temp, ref.addr($a)
mov $i, 0x40
xor &lt;$temp&#43;1&gt;, $i
sub &lt;$temp&#43;1&gt;, 2
1:&lt;$temp&gt; &#61; 0xEB
jmp loop

exit:
ret</code>
</pre>
<p><b>Breakpoint checks on APIs</b><span style="	font-weight: 400;
"> are performed before calling every API and if found, the shellcode terminates.</span></p>
<p><b>Using the NtsetInformationThread API</b><span style="	font-weight: 400;
"> is also an anti-debugging technique. The DJB2 algorithm loads the NtsetInformationThread API and passes the second parameter as 11 (corresponding to ThreadHideFromDebugger), which will crash the process when it runs from inside a debugger.</span></p>
<p><b>Anti-debugging via NtQueryInformationProcess </b><span style="	font-weight: 400;
">enables GuLoader to check the presence of a remote debugger in its process. Our sample leverages the NtQueryInformationProcess API, by specifying ProcessDebugPort (0x7) as the second parameter. The loader checks for non-zero return values, which means the process is being debugged.</span></p>
<p><b>Anti-debugging via DbgBreakPoint and DbgUiRemoteBreakin</b><span style="	font-weight: 400;
"> allows GuLoader to patch two APIs in memory by leveraging the NtProtectVirtualMemory API to mark it writable for their addresses:</span></p>
<ul><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">DbgBreakPoint — by replacing with a NOP instruction</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">DbgUiRemoteBreakin — by replacing with a random instruction</span></li></ul>

<h3><span style="	font-weight: 400;
">Anti-Virtual Machine</span></h3>
<p><span style="	font-weight: 400;
">What’s different from previously analyzed GuLoader variants is that this shellcode performs </span><b>memory scanning for VMware-related string checks on every memory page </b><span style="	font-weight: 400;
">from the entire process memory. GuLoader uses NtQueryVirtualMemory API to scan the entire memory of the process to check if there are any Virtual Machine (VM)-related strings present.</span></p>
<div class="wp-caption" id="attachment_145061"><a href="/content/dam/crowdstrike/www/en-us/wp/2022/12/GuLoader2.png" target="_blank" rel="noopener noreferrer"><img class="wp-image-145061 size-full" src="/content/dam/crowdstrike/www/en-us/wp/2022/12/GuLoader2.png" alt="" width="977" height="668"/></a>Figure 2. Pseudocode showing how the Virtual Memory is scanned and how the Djb2 algorithm is used (click to enlarge)</div>
<p><span style="	font-weight: 400;
">This is implemented by calling API NtQueryVirtualMemory with the handle 0xffffffff (current process) to iteratively retrieve the base address of every page. The fourth parameter of this API is the MemoryInformation structure which contains information about a range of pages in the virtual address space of a process. A similar technique has been implemented in the past on </span><a href="https://github.com/hfiref0x/VMDE/blob/master/src/vmde/detect.c#L679"><span style="	font-weight: 400;
">VMDE </span></a><span style="	font-weight: 400;
">project by hfiref0x, in which the author searches in memory for “Sandboxie” artifacts strings; in this case GuLoader searches for virtualization software traces.</span></p>
<p><span style="	font-weight: 400;
">I</span>f it finds any of the DJB2 values for a series of strings (i.e., VMSwitchUserControlClass, VM3DService Hidden Window, VMDisplayChangeControlClass, vmtoolsdControlWndClass, etc.), the shellcode throws an error message that it is running under a virtual environment and then terminates execution.</p>
<p><b>Using CPUID and rdtsc</b> is a very common anti-debugging trick that involves using the read time-stamp counter (rdtsc) instruction to determine how many CPU ticks took place since the processor was reset. This is used as a timing check comparing the time required to execute two rdtsc instructions and then calling the CPUID instruction with EAX &#61;1 to retrieve the process information, returning the output in the ECX registry. If the thirty-first bit of ECX is set, it is used as that the shellcode is running inside a potentially hostile environment (virtual machine).</p>
<p>The <b>use of the EnumWindows function</b> is also a popular anti-VM technique generally used to enumerate all top-level windows on the screen by passing the handle to each window. This API is used in the shellcode, counting the number of open windows inside the callback function. If the number is lower than 12, it will call the API TerminateProcess.</p>
<p><b>Enumerating device drivers</b> also falls under the anti-VM category. GuLoader uses EnumDeviceDriver from (psapi.dll) and checks the presence of specific drivers and triggers an error if found. Shellcode fetches and calls two APIs from its DJB2 values — DADA7345 and CDAFD506, respectively EnumDeviceDriver and GetDeviceDriverBasename — to enumerate driver names. Every enumerated driver name’s DJB2 hash value is calculated. These hash values are then compared with hard-coded DJB2 hash values, which are actually VM-related device drivers.</p>

<p><b>Scanning and enumerating installed software</b><span style="	font-weight: 400;
"> is a technique GuLoader uses to check for virtualization software installed as part of its anti-sandbox/anti-VM mechanism by loading APIs that match the DJB2 hash values 55fbd1cd (MsiGetProductInfoA) and AD5448 (MsiEnumProductsA). The shellcode enumerates the products using API MsiGetProductInfoA and checks if they match with a list of known software.</span></p>
<p><b>Service enumeration using the OpenScManager API</b><span style="	font-weight: 400;
"> establishes a connection to the service control manager on the machine and opens the specified service control manager database. It then enumerates service control manager database services using EnumServicesStatusA</span><b>.</b></p>
<h3><span style="	font-weight: 400;
">Process Hollowing</span></h3>
<p><a href="https://attack.mitre.org/techniques/T1055/012/"><span style="	font-weight: 400;
">Process hollowing</span></a><span style="	font-weight: 400;
"> is a technique of executing arbitrary code in the address space of a separate live process by creating a process in a suspended state then unmapping/hollowing its memory, which can then be replaced with malicious code. In this case, the malware does not unmap an already mapped section on the remote process, but tries to add a new section and write the injected shellcode into it.</span></p>
<p><span style="	font-weight: 400;
">The below steps are followed for injection:</span></p>
<p><span style="	font-weight: 400;
">1. The shellcode first creates a suspended process by calling CreateProcessInternal.</span></p>
<p><span style="	font-weight: 400;
">2. It then calls</span> <span style="	font-weight: 400;
"><code>NtOpenFile</code></span> <span style="	font-weight: 400;
">on</span> <span style="	font-weight: 400;
"><code>\\??\\C:\\Windows\\syswow64\\iertutil.dll</code>.</span></p>
<p><span style="	font-weight: 400;
">3. It does</span> <span style="	font-weight: 400;
"><code>NtCreateSection</code> on that file, where it will inject its malicious shellcode.</span></p>
<p><span style="	font-weight: 400;
">4. It then maps that section via NtMapViewofSection on the suspended process. </span><b>If this injection technique fails, it uses the following redundancy method:</b></p>
<p>a. <code>NtAllocateVirtualMemory</code> by invoking the inline assembly instructions (without calling ntdll.dll, to bypass AV/EDR User Mode hooks) of that function, using the following assembly stub:</p>
<pre>
<code>mov eax,18                           
mov edx,ntdll.77178850       
call edx                           
ret 18  </code>

</pre>
<p><span style="	font-weight: 400;
">It uses</span> <span style="	font-weight: 400;
"><code>NtWriteProcessMemory</code> to copy the same shellcode onto that virtually allocated address.</span> </p>
<p><span style="	font-weight: 400;
">5. It then uses API</span> <span style="	font-weight: 400;
"><code>NtGetContextThread</code> on remote thread of suspended process, by specifying the following flags to retrieve the registry values of that thread:</span></p>
<p> <span style="	font-weight: 400;
">i. CONTEXT_CONTROL to retrieve the registers ESP, EIP, FLAGS, BP</span> </p>
<p><span style="	font-weight: 400;
">ii. CONTEXT_INTEGER to retrieve the registers AX, BX, CX, DX, SI, DI</span> </p>
<p><span style="	font-weight: 400;
">iii. CONTEXT_SEGMENTS to retrieve the registers DS, ES, FS, GS</span> </p>
<p><span style="	font-weight: 400;
">6. The retrieved</span> <span style="	font-weight: 400;
"><code>CONTEXT</code></span> <span style="	font-weight: 400;
">is used to manipulate registers by calling the NTAPI</span> <span style="	font-weight: 400;
"><code>NtSetContextThread</code></span> <span style="	font-weight: 400;
">to set the <code>EAX</code> register to the address of shellcode (EIP points to</span> <span style="	font-weight: 400;
"><code>RtlUserThreadStart</code>, which will jump to new</span> <span style="	font-weight: 400;
"><code>EAX</code>).</span></p>
<p> <span style="	font-weight: 400;
">7. Finally, the malware calls the native API</span> <span style="	font-weight: 400;
"><code>NtResumeThread</code></span> <span style="	font-weight: 400;
">to resume the process and execute the shellcode in the new process.</span></p>

It loads the API to manage the internet connection and connects to a URL to download the first level of payload, which then gets decrypted by the shellcode and results in dropping the Remcos malware.</span></p>

<h2><span style="	font-weight: 400;
">How the CrowdStrike Falcon Platform Protects Against GuLoader</span></h2>
<div class="wp-caption" id="attachment_145062"><a href="/content/dam/crowdstrike/www/en-us/wp/2022/12/GuLoader3.png" target="_blank" rel="noopener noreferrer"><img class="wp-image-145062 size-full" src="/content/dam/crowdstrike/www/en-us/wp/2022/12/GuLoader3.png" alt="" width="977" height="402"/></a>Click to enlarge</div>
<p><span style="	font-weight: 400;
">GuLoader remains a dangerous threat that’s been constantly evolving with new methods to evade detection. The Falcon platform uses behavior-based detection capabilities to automatically detect and prevent GuLoader early in the attack chain by identifying the initial VBScript loader, preventing its execution.</span></p>
<p> <span style="	font-weight: 400;
">This recent analysis performed by CrowdStrike on GuLoader now offers a complete picture of all the DJB2 hash values used for APIs. This type of threat research enables CrowdStrike to leverage expert human intelligence and augment its machine learning and behavior-based detection capabilities to stop breaches.</span></p>
