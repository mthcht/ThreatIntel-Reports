        <h1>Critical Hit: How DoppelPaymer Hunts and Kills Windows Processes</h1>
        
        <div class="publish_info">
            <p>December 07, 2021</p>
            <span>|</span>
            <a href="/en-us/blog/author.shaun-hurley/" title="Posts by Shaun Hurley" rel="author">
                Shaun Hurley</a>
            <span>|</span>
                
            <a href="/en-us/blog/category.counter-adversary-operations/" title="Threat Hunting &amp; Intel">Threat Hunting &amp; Intel</a>
        </div>
        <div class="post_image">
            <img width="1060" height="698" src="https://assets.crowdstrike.com/is/image/crowdstrikeinc/Blog_1060x698-8-us?wid=1060&hei=698&fmt=png-alpha&qlt=95,0&resMode=sharp2&op_usm=3.0,0.3,2,0" class="attachment-post-thumbnail size-post-thumbnail wp-post-image" alt="" decoding="async" fetchpriority="high" srcset="/content/dam/crowdstrike/www/en-us/wp/2024/12/Blog_1060x698-8-us.jpeg 1060w" sizes="(max-width: 1060px) 100vw, 1060px"/>
        </div>
    

    
</div>

    <div style="display: none;">
        
    </div>


    
</div>
<div class="text text--blog-content aem-GridColumn aem-GridColumn--default--12">



<div class="cmp-text" id="text-bef7d2ec3c" data-cmp-name="cmp-text-" data-cmp-is="text" data-target-location="false">
    
       <span><span style="	font-weight: 400;
">In a July 2019 blog post about </span><a href="/en-us/blog/doppelpaymer-ransomware-and-dridex-2/"><span style="	font-weight: 400;
">DoppelPaymer</span></a><span style="	font-weight: 400;
">, Crowdstrike Intelligence reported that ProcessHacker was being hijacked to kill a list of targeted processes and gain access, delivering a “critical hit.” Although the blog is now a couple of years old, the hijacking technique is interesting enough to dig into its implementation.</span>

<span style="	font-weight: 400;
">The hijack occurs when ProcessHacker loads a malicious stager DLL designed to exploit legitimate behavior. Once the process has been hijacked, the stager DLL is able to terminate processes, including those protected by </span><a href="/en-us/blog/evolution-protected-processes-part-1-pass-hash-mitigations-windows-81/"><span style="	font-weight: 400;
">Protected Process Light (PPL)</span></a><span style="	font-weight: 400;
">. To accomplish this task, it leverages ProcessHacker’s kernel driver, KProcessHacker, that has been registered under the service name KProcessHacker3. This blog delves into the details about how DoppelPaymer hijacks ProcessHacker and exploits KProcessHacker to kill a list of processes, including both antivirus (AV) and endpoint detection and response (EDR) applications.</span>

<div class="wp-caption" id="attachment_50630"><img class="wp-image-50630 size-full" src="/content/dam/crowdstrike/www/en-us/wp/2021/12/Picture1-3.png" alt="" width="2031" height="1575"/> Figure 1. Architecture diagram to kill a target process</div>
<h2><span style="	font-weight: 400;
">Launching Process Hacker</span></h2>
<span style="	font-weight: 400;
">To start ProcessHacker, DoppelPaymer writes the ProcessHacker executable, the KProcessHacker driver, and the malicious stager DLL into a subdirectory of </span><code><span style="	font-weight: 400;
">%APPDATA%</span></code><span style="	font-weight: 400;
">. Both the subdirectory name and the file names for the executable and driver are a unique string of alphanumeric characters. Once those two files have been written, one of the DLLs loaded by ProcessHacker has to be hijacked using a technique called “DLL search order hijacking.”</span>
<h3><span style="	font-weight: 400;
">DLL Search Order Hijacking</span></h3>
<span style="	font-weight: 400;
">Similar to Dridex, DoppelPaymer uses DLL search order hijacking to exploit the DLL loading behavior for Windows processes. As the operating system PE loader loads a binary, it also needs to load the DLL files required for the PE to function. By default, MS Windows has a specific path it takes when looking for the DLL files to load. Windows checks for Windows system DLLs in the same directory as the target binary before it checks the Windows system directories. A malicious process, in this case DoppelPaymer, can drop a malicious version of a DLL in that directory and it will be loaded by the target application.<p> </p></span>

<span style="	font-weight: 400;
">To determine which DLL to hijack, DoppelPaymer walks the module name list in the Import Address Table (IAT) of the ProcessHacker binary. Each name is hashed with a CRC32 algorithm and compared against a hardcoded list of hashes (Table 1), and if a match occurs, the name is added to a list data structure. A random number generator is used to pick one of the three names out of the list.</span>
<table class="orange"><tbody><tr class="top-row bg-orange"><td><strong>CRC32</strong></td><td><strong>Filename</strong></td></tr><tr><td><code>0xd8946922</code></td><td><code>VERSION.dll</code></td></tr><tr><td><code>0x020da855</code></td><td><code>WINSTA.dll</code></td></tr><tr><td><code>0x3c55abe2</code></td><td><code>UxTheme.dll</code></td></tr></tbody></table>
<p style="	text-align: center;
">Table 1. Mapping DLL name to CRC32 hash for search order hijacking</p>

The file is written to the same subdirectory as the ProcessHacker executable with the same file name as the hijacked DLL.<p> </p></span>

<h2><span style="	font-weight: 400;
">Loading the Stager DLL</span></h2>
<span style="	font-weight: 400;

<ul><li style="	font-weight: 400;
"><span style="	font-weight: 400;

"><span style="	font-weight: 400;

"><span style="	font-weight: 400;
">The IPC objects necessary for DoppelPaymer to communicate with the stager DLL need to be duplicated</span></li></ul>
<span style="	font-weight: 400;
">After all four of these steps have been successfully completed, the stager DLL can start killing target processes provided by DoppelPaymer.</span>
<h3><span style="	font-weight: 400;
">Reaching ProcessHacker’s Code Entrypoint Address</span></h3>
<span style="	font-weight: 400;
">Once the process starts to load the stager DLL, the malicious code will start to execute, but if control isn’t passed back to the OS to finish loading ProcessHacker, it will not be usable by DoppelPaymer. The loading process completes when the entry-point address of ProcessHacker is reached. To determine when the entry point is reached, the stager DLL will overwrite the entry point of ProcessHacker with the code in Figure 3.</span>
<pre><code>.rdata:10006120             mov     eax, <strong>94A351BBh</strong>

<span style="	font-weight: 400;

<h3><span style="	font-weight: 400;
">Kernel Verification of the KProcessHacker Client</span></h3>
<span style="	font-weight: 400;
">Every time an IOCTL is sent to the KProcessHacker service, it is checked to ensure that the caller is a verified KProcessHacker client that is allowed to communicate with the service. All attempts to communicate with KProcessHacker are validated using an IOCTL request key that is generated by sending a </span><code><span style="	font-weight: 400;
">KPH_RETRIEVEKEY</span></code><span style="	font-weight: 400;
"> request from the user-mode process. The importance of this key is discussed in the “KProcessHacker IOCTL Request Keys and APC&#39;&#39; section below. Attached to the </span><code><span style="	font-weight: 400;
">KPH_RETRIEVEKEY</span></code><span style="	font-weight: 400;
"> request is an Asynchronous Procedure Call (APC) routine, </span><code><span style="	font-weight: 400;
">KphpWithKeyApcRoutine</span></code><span style="	font-weight: 400;
">, which will be executed upon completion.</span>

<span style="	font-weight: 400;
">KProcessHacker initializes a </span><code><span style="	font-weight: 400;
">KPH_CLIENT</span></code><span style="	font-weight: 400;

">KphDispatchCreate</span></code><span style="	font-weight: 400;

<h2><span style="	font-weight: 400;
">Killing Blocklisted Applications</span></h2>
<span style="	font-weight: 400;
">Once DoppelPaymer receives the signaled event, it starts enumerating both service and process names, and hashes them with the CRC32 algorithm. These hashes are compared against a list of blocklisted hashes in DoppelPaymer’s process memory. The complete list was covered in the previous DoppelPaymer blog post. This section discusses what happens when an application matches one of the blocklisted items.</span><span style="	font-weight: 400;
"><p> </p></span>

<span style="	font-weight: 400;
">DoppelPaymer writes the process ID associated with the service, along with a command to the mapped section object, </span><code><span style="	font-weight: 400;
">0x124</span></code><span style="	font-weight: 400;
">. </span><span style="	font-weight: 400;
">The command will tell the stager DLL which steps to take.</span>
<pre><code>AntiAV {
     &#43;0x00   Command
     &#43;0x08   Process ID
     &#43;0x10   errorCodeResponse   // Response code from ProcessHacker
}</code></pre>
<p style="	text-align: center;
">Figure 7. Blocklisted process information written to IPC section</p>
<span style="	font-weight: 400;
">Process termination occurs in two steps: a process is opened, then it is killed. The first command sent, </span><code><span style="	font-weight: 400;
">1</span></code><span style="	font-weight: 400;
">, will tell the stager DLL to open a handle to the process. Table 4 contains a list of valid commands.</span>

<h3><span style="	font-weight: 400;
">KProcessHacker IOCTL Request Keys and APC<p> </p></span></h3>
<span style="	font-weight: 400;
">Certain IOCTL requests to the KProcessHacker service require the verification of an IOCTL<p> </p> request key. To ensure that the key cannot be tampered with, the key is generated by the driver and stored in the </span><code><span style="	font-weight: 400;
">KPH_CLIENT</span></code><span style="	font-weight: 400;
"> structure. The following IOCTL requests require a key:</span>
<ul><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">KPH_OPENPROCESS</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">KPH_OPENPROCESSTOKEN</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">KPH_TERMINATEPROCESS</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">KPH_READVIRTUALMEMORY</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">KPH_OPENTHREAD</span></li></ul>
<span style="	font-weight: 400;
">Prior to making any of these requests, ProcessHacker has to send a </span><code><span style="	font-weight: 400;
">KPH_RETRIEVEKEY</span></code><span style="	font-weight: 400;
"> request using </span><code><span style="	font-weight: 400;
">NtDeviceIoControlFile</span></code><span style="	font-weight: 400;
">. Along with this request, the user-mode address of an APC routine, </span><code><span style="	font-weight: 400;
">KphpWithKeyApcRoutine</span></code><span style="	font-weight: 400;
">, and the user-mode address of a function called by the APC are sent as parameters. This routine to be called by the APC will end up making one of the IOCTL requests mentioned in the above bulleted list.</span>
<pre><code>NtDeviceIoControlFile(
     PhKphHandle,
     NULL,
     <strong>KphpWithKeyApcRoutine</strong>,  // Called after NtDeviceIoControlFile 
                             //  returns
     NULL,
     &amp;context.Iosb,          // Receives the status code
     <strong>KPH_RETRIEVEKEY</strong>,        // IOCTL
     &amp;input,                 // Parameters passed to IOCTL
     sizeof(input),
     NULL,
     0
);</code></pre>
<span style="	font-weight: 400;
">The </span><code><span style="	font-weight: 400;
">KPH_RETRIEVEKEY</span></code><span style="	font-weight: 400;
"> request is handled by </span><code><span style="	font-weight: 400;
">KphRetrieveKeyViaApc</span></code><span style="	font-weight: 400;
">. Prior to generating the request key, several checks are performed to ensure that the client (ProcessHacker, in this case) making the request is verified and that the APC parameter is valid:</span>
<ul><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Ensure the client has been verified by checking the </span><code><span style="	font-weight: 400;
">KPH_CLIENT-&gt;VerificationSucceeded</span></code><span style="	font-weight: 400;
"> field</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Ensure that the process information for the client matches what was set during the verification process<p> </p></span>
<ul><li style="	font-weight: 400;
"><code><span style="	font-weight: 400;
">KPH_CLIENT-&gt;VerifiedProcess</span></code></li><li style="	font-weight: 400;
"><code><span style="	font-weight: 400;
">KPH_CLIENT-&gt;VerifiedProcessId</span></code></li></ul>
</li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Ensure that the instruction address of the APC routine falls within the executable section of the verified client</span></li></ul>
<span style="	font-weight: 400;
">Once those checks are passed, a request key is generated and stored in the </span><code><span style="	font-weight: 400;
">KPH_CLIENT</span></code><span style="	font-weight: 400;
"> structure. This key will also be passed as argument to the APC routine. So now, both the client and the server have independent copies of the same request key. The APC routine, </span><code><span style="	font-weight: 400;
">KphpWithKeyApcRoutine</span></code><span style="	font-weight: 400;
">, executes.<p> </p><p> </p><p> </p></span>

<span style="	font-weight: 400;
">As noted earlier, the APC routine receives a function pointer that will be used to execute a specific action (kill process, open process, etc.). To restrict which requests can be made via this APC routine, it makes sure that only the following functions can be called from the APC:</span>
<ul><li style="	font-weight: 400;
"><span style="	font-weight: 400;
"><code>KphpGetL1KeyContinuation</code><p> </p></span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
"><code>KphpOpenProcessContinuation</code><p> </p></span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
"><code>KphpOpenProcessTokenContinuation</code><p> </p></span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
"><code>KphpTerminateProcessContinuation</code><p> </p></span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
"><code>KphpReadVirtualMemoryUnsafeContinuation</code><p> </p></span></li><li style="	font-weight: 400;
"><code><span style="	font-weight: 400;
">KphpOpenThreadContinuation</span></code></li></ul>
<span style="	font-weight: 400;
">This check prevents DLLs from being injected into ProcessHacker and leveraging the </span><code><span style="	font-weight: 400;
">KphpWithKeyApcRoutine</span></code><span style="	font-weight: 400;
"> APC as a method to execute its own routines under the guise of being a valid </span><code><span style="	font-weight: 400;
">KPH_CLIENT</span></code><span style="	font-weight: 400;
">. Once this check passes, the function passed to the APC is called and the client copy of the request key is passed to that function.</span>

<span style="	font-weight: 400;
">Both this check and the checks made in the </span><code><span style="	font-weight: 400;
">KphRetrieveKeyViaApc</span></code><span style="	font-weight: 400;
"> procedure pose a challenge for the stager DLL. The author of the stager DLL scraps the original </span><code><span style="	font-weight: 400;
">KphpWithKeyApcRoutine</span></code><span style="	font-weight: 400;
"> routine and passes their own APC routine, </span><code><span style="	font-weight: 400;
">StagerAPCRoutine</span></code><span style="	font-weight: 400;
">. The code for this is written directly after ProcessHacker’s overwritten entry-point code. This bypasses both of the function pointer checks and passes the checks performed by </span><code><span style="	font-weight: 400;
">KphRetrieveKeyViaApc</span></code><span style="	font-weight: 400;
">.</span>
<pre><code>000000013f5b2f2c    jmp  cs:CallFunctionPointerRoutine
000000013f5b2f3a    push    rax
000000013f5b2f3b    retn</code></pre>
<p style="	text-align: center;
">Figure 8. StagerAPCRoutine</p>
<span style="	font-weight: 400;
">The </span><code><span style="	font-weight: 400;
">StagerAPCRoutine</span></code><span style="	font-weight: 400;
"> (Figure 8) shortcuts the entire process and jumps directly to a procedure, </span><code><span style="	font-weight: 400;
">CallFunctionPointerRoutine</span></code><span style="	font-weight: 400;
">,</span><span style="	font-weight: 400;
"> that calls the DLL stager versions of </span><code><span style="	font-weight: 400;
">KphpOpenProcessContinuation</span></code><span style="	font-weight: 400;
"> and </span><code><span style="	font-weight: 400;
">KphpTerminateProcessContinuation</span></code><span style="	font-weight: 400;
"> procedures and passes the client copy of the request key as a parameter.</span>
<h3><span style="	font-weight: 400;
">Opening a Process Handle</span></h3>
<span style="	font-weight: 400;
">As outlined in the previous section, to open a process handle, a </span><code><span style="	font-weight: 400;
">KPH_RETRIEVEKEY</span></code><span style="	font-weight: 400;
"> request is sent to the KProcessHacker service. Along with this request, the </span><code><span style="	font-weight: 400;
">StagerAPCRoutine</span></code><span style="	font-weight: 400;
"> address and the address of the function called by the APC open a process, </span><code><span style="	font-weight: 400;
">StagerOpenProcess</span></code><span style="	font-weight: 400;
">. A new request key is generated, saved to </span><code><span style="	font-weight: 400;
">KPH_CLIENT</span></code><span style="	font-weight: 400;
"> and passed to </span><code><span style="	font-weight: 400;
">StagerAPCRoutine</span></code><span style="	font-weight: 400;
">. Once everything has been validated, the </span><code><span style="	font-weight: 400;
">StagerAPCRoutine</span></code><span style="	font-weight: 400;
"> calls </span><code><span style="	font-weight: 400;
">StagerOpenProcess</span></code><span style="	font-weight: 400;
"> where a </span><code><span style="	font-weight: 400;
">KPH_OPENPROCESS</span></code><span style="	font-weight: 400;
"> request is sent to the KProcessHacker service. Both the client copy of the request key and the process ID of the target are sent with the request.</span>

<div class="wp-caption" id="attachment_50632"><img class="size-full wp-image-50632" src="/content/dam/crowdstrike/www/en-us/wp/2021/12/Picture1-4.png" alt="" width="1820" height="1643"/> Figure 9. Process diagram to open a process</div>

<span style="	font-weight: 400;
">The service handles this request by calling </span><code><span style="	font-weight: 400;
">KpiOpenProcess</span></code><span style="	font-weight: 400;
">. Before a handle to the process can be opened, the client’s request key is validated by calling </span><code><span style="	font-weight: 400;
">KphValidateKey</span></code><span style="	font-weight: 400;
">, where the client copy of the key is compared against the copy stored in </span><code><span style="	font-weight: 400;
">KPH_CLIENT</span></code><span style="	font-weight: 400;
">. If these match, a handle to the target process is opened.</span>

<code><span style="	font-weight: 400;
">PsLookupProcessByProcessId</span></code><span style="	font-weight: 400;
"> is called to get a pointer to the process object in kernel memory. That pointer is used to open a handle to the object by calling </span><code><span style="	font-weight: 400;
">ObOpenObjectByPointer</span></code><span style="	font-weight: 400;
">. This handle can now be referenced by the stager DLL.<p> </p></span>

<span style="	font-weight: 400;
">The stager DLL signals the </span><code><span style="	font-weight: 400;
">0x120</span></code><span style="	font-weight: 400;
"> event handle, notifying DoppelPaymer that a handle has successfully been opened to the target process. Now the process can be killed.</span>
<h3><span style="	font-weight: 400;
">Killing a Process</span></h3>
<span style="	font-weight: 400;
">DoppelPaymer verifies that the process was successfully opened, and then takes the appropriate action. If an error occurred, it continues checking for blocklisted applications; otherwise, another notification is sent, this time with the command </span><code><span style="	font-weight: 400;
">2</span></code><span style="	font-weight: 400;
"> to terminate the process.</span>

<span style="	font-weight: 400;
">Terminating a process follows the same procedure as opening a process with one difference: The </span><code><span style="	font-weight: 400;
">StagerKillProcess</span></code><span style="	font-weight: 400;
"> function pointer is passed to the </span><code><span style="	font-weight: 400;
">StagerAPCRoutine</span></code><span style="	font-weight: 400;
">. The </span><code><span style="	font-weight: 400;
">StagerKillProcess</span></code><span style="	font-weight: 400;
"> function sends a </span><code><span style="	font-weight: 400;
">KPH_TERMINATEPROCESS</span></code><span style="	font-weight: 400;
"> request to the KProcessHacker service. This is handled by the </span><code><span style="	font-weight: 400;
">KpiTerminateProcess</span></code><span style="	font-weight: 400;
"> kernel-mode function. The request key is validated before process termination can occur. The target process is reopened to get a kernel handle, and </span><code><span style="	font-weight: 400;
">ZwTerminateProcess</span></code><span style="	font-weight: 400;
"> is called to kill the process. Note that using this procedure ignores PPL, so even protected processes will be killed.</span>
<h2><span style="	font-weight: 400;
">Conclusion</span></h2>
<span style="	font-weight: 400;
">DoppelPaymer’s usage of ProcessHacker to kill AV services is part of a larger trend of various actors leveraging legitimate tools to disable AV/EDR functionality. DoppelPaymer’s method is a testament to how innovative malware authors can be when it comes to neutralizing the defenses of their target.<p> </p></span>
