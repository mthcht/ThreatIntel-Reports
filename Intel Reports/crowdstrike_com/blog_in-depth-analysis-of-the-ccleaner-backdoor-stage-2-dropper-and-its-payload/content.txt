       <span><p> </p>
<h2><span style="	font-weight: 400;
">Overview</span></h2>
<p><span style="	font-weight: 400;
">Recently, CrowdStrike® analyzed the backdoor embedded in the legitimate PC cleaning utility CCleaner version 5.33, as reported in the blog post </span><a href="/en-us/blog/protecting-software-supply-chain-deep-insights-ccleaner-backdoor/"><span style="	font-weight: 400;
">Protecting the Software Supply Chain: Deep Insights into the CCleaner Backdoor</span></a><span style="	font-weight: 400;
">. This was an example of using an organization’s supply chain infrastructure as an infection vector, a trend that has been on the rise in 2017 as discussed in another recent post, </span><a href="/en-us/blog/software-supply-chain-attacks-rise-undermining-customer-trust/"><span style="	font-weight: 400;
">Software Supply Chain Attacks on the Rise, Undermining Customer Trust</span></a><span style="	font-weight: 400;
">. In addition, CrowdStrike Falcon® Intelligence™ reported on the backdoor previously and discussed the possibility of the infrastructure being tied to a Chinese nexus.</span> <span style="	font-weight: 400;
">Additionally, CrowdStrike Falcon® Intelligence also discussed the technical details of the Stage 1 and Stage 2 backdoors with analysis showing that the original backdoor was the first stage in a multi-stage infection chain, meant to download a dropper (Stage 2) that was only deployed to specific targets. Stage 2 drops either a 32-bit or 64-bit binary, depending on the system architecture and is responsible for decrypting the actual payload embedded in a registry key. This payload attains the C2 address via a variety of steps, and downloads an unknown binary which is Stage 3.</span> <span style="	font-weight: 400;
">This post provides an in-depth analysis of the Stage 2 dropper; the subsequent payload and the steps that are taken to calculate the C2 IP address in order to download the next stage binary.</span></p>
<h2><span style="	font-weight: 400;
">Technical Analysis</span></h2>
<h2><span style="	font-weight: 400;
">Stage 2 Dropper</span></h2>
<p><span style="	font-weight: 400;
">The following information describes the Stage 2 dropper that pertains to the CCleaner embedded malware:</span> <b>Size: </b><span style="	font-weight: 400;
">175616 </span><b>SHA256:<p> </p>
<p> </p>
<p> </p>
<p> </p>
<p> </p>
</b><span style="	font-weight: 400;
">DC9B5E8AA6EC86DB8AF0A7AA897CA61DB3E5F3D2E0942E319074DB1AACCFDC83 </span><b>Compiled:</b><span style="	font-weight: 400;
"> Tue, Sep 12 2017, 8:44:58 — 32 Bit DLL</span> <span style="	font-weight: 400;
">Once executed, the dropper calls </span><b>IsWow64Process</b><span style="	font-weight: 400;
"> to determine if it&#39;s being run in a 64-bit environment. Depending on the result, it will drop a 32-bit or 64-bit binary on the system. The binary is embedded within the malware itself, and it is zlib compressed. The dropper will zlib inflate itself and drop onto the victim computer. The dropper also performs system checks by accessing the USER_SHARED_DATA of its own process and querying the </span><b>NtMajorVersion</b><span style="	font-weight: 400;
"> value to determine if the system is running Windows XP. The output determines the location of the dropped binary.</span></p>
<ul><li>If XP x86:<ul><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">location is C:\Windows\System32\spool\prtprocs\\w32x86\\localspl.dll</span></li></ul>
</li></ul>
<ul><li>If XP x64:<ul><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">location is C:\Windows\System32\spool\x64\localspl.dll</span></li></ul>
</li></ul>
<ul><li>If Windows 7 or higher:<ul><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">location is C:\Windows\System32\TSMSISrv.dll</span></li></ul>
</li></ul>
<h2><span style="	font-weight: 400;

<h3><span style="	font-weight: 400;

<p><b>Full path on victim machine (Windows 7 or higher):</b><span style="	font-weight: 400;
"> C:\Windows\System32\TSMSISrv.dll </span><b>Full path on victim machine (Windows XP):</b><span style="	font-weight: 400;
"> C:\Windows\system32\spool\prtprocs\w32x86\localspl.dll </span><b>Size: </b><span style="	font-weight: 400;
">173568 </span><b>SHA256:</b><span style="	font-weight: 400;
"> 07FB252D2E853A9B1B32F30EDE411F2EFBB9F01E4A7782DB5EACF3F55CF34902 </span><b>Compiled:</b><span style="	font-weight: 400;
"> Wed, Apr 22 2015, 18:20:39 — 32 Bit DLL </span><b>Version:</b><span style="	font-weight: 400;
"> 2, 0, 4, 23 </span><b>File</b> <b>Description:</b><span style="	font-weight: 400;
"> VirtCDRDrv Module </span><b>Internal</b> <b>Name</b><span style="	font-weight: 400;
">: VirtCDRDrv </span><b>Original</b> <b>Filename:</b><span style="	font-weight: 400;
"> VirtCDRDrv.dll </span><b>Product</b> <b>Name:</b><span style="	font-weight: 400;
"> VirtCDRDrv Module</span></p>
<h3><span style="	font-weight: 400;
">64-bit</span></h3>
<p><b>Full path on victim machine (Windows 7 or higher):</b><span style="	font-weight: 400;
"> C:\Windows\System32\TSMSISrv.dll </span><b>Full path on victim machine (Windows XP):</b><span style="	font-weight: 400;
"> C:\Windows\system32\spool\prtprocs\x64\localspl.dll </span><b>Size:</b><span style="	font-weight: 400;
"> 81408 </span><b>SHA256:</b><span style="	font-weight: 400;
"> 128ACA58BE325174F0220BD7CA6030E4E206B4378796E82DA460055733BB6F4F </span><b>Compiled:</b><span style="	font-weight: 400;
"> Tue, Apr 19 2011, 0:09:20 — 64 Bit DLL </span><b>Version:</b><span style="	font-weight: 400;
"> 2.2.0.65 </span><b>File</b> <b>Description:</b><span style="	font-weight: 400;
"> Symantec Extended File Attributes </span><b>Internal</b> <b>Name:</b><span style="	font-weight: 400;
"> SymEFA </span><b>Original</b> <b>Filename:</b><span style="	font-weight: 400;
"> EFACIi64.dll </span><b>Product</b> <b>Name:</b><span style="	font-weight: 400;
"> EFA</span> <span style="	font-weight: 400;
">It is important to note that both TSMSiSrv.dll and localspl.dll are actually the names of legitimate Microsoft Windows libraries. TSMSiSrv.dll&#39;s official description is &#34;Windows Installer Coordinator for Remote Desktop Session Host Server&#34; and it is loaded by the service &#34;SessionEnv&#34; that is the Remote Desktop Configuration service. According to MSDN, localspl.dll is a &#34;Local Print Provider&#34; and handles all print jobs directed to printers that are managed from the local server. This file is loaded by the service &#34;Spooler&#34; that is used for printing services.</span> <span style="	font-weight: 400;
">After dropping the file, the dropper modifies its date/time stamp so that it matches that of C:\Windows\System32\msvcrt.dll.</span> <span style="	font-weight: 400;
">Next, the dropper adds the following registry keys. (Note: This is specific to a 32-bit environment. Certain value such as file size will change if the malware is running in a 64-bit environment.)</span></p>

<h2><span style="	font-weight: 400;
">Leveraging Legitimate Services</span></h2>
<p><span style="	font-weight: 400;
">The dropper leverages an existing Microsoft Windows service to load the malware. Once the registries have been added, the dropper calls a function to modify and restart an existing service. If executing in Windows 7 or higher, it calls </span><b>OpenServiceA</b><span style="	font-weight: 400;
"> on the existing service, &#34;SessionEnv&#34; — a service for Remote Desktop Configuration — and changes its configuration by calling </span><b>ChangeServiceConfigA</b><span style="	font-weight: 400;
"> with the following parameters:</span></p>
<ul><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">hService &#61; Service Handle</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">ServiceType &#61; SERVICE_KERNEL_DRIVER|SERVICE_FILE_SYSTEM_DRIVER|SERVICE_ADAPTER|</span><br />
<span style="	padding-left: 88.0px;
">SERVICE_RECOGNIZER_DRIVER|SERVICE_WIN32_OWN_PROCESS|SERVICE_WIN32_SHARE_PROCESS|</span><br />
<span style="	padding-left: 88.0px;
">SERVICE_INTERACTIVE_PROCESS|FFFFFEC0</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">StartType &#61; SERVICE_AUTO_START</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">ErrorControl &#61; SERVICE_NO_CHANGE</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">BinaryPathName &#61; NULL</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">LoadOrderGroup &#61; NULL</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">pTagId &#61; NULL</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;

"><span style="	font-weight: 400;

"><span style="	font-weight: 400;

"><span style="	font-weight: 400;
">DisplayName &#61; NULL</span></li></ul>
<p><span style="	font-weight: 400;
">This ensures that the service will auto-start upon system reboot (i.e., a persistence mechanism). The dropper then restarts the service, which invokes the legitimate windows library &#34;SessEnv.dll&#34; located in C:\Windows\system32. It is important to note that SessEnv.dll is loaded in the process svchost.exe. Analysis shows that it attempts to load the legitimate library %SystemRoot%\system32\TSMSISrv.dll by calling </span><b>LoadLibrary</b><span style="	font-weight: 400;
"> on it to call the functions </span><i><span style="	font-weight: 400;
">StartComponent</span></i><span style="	font-weight: 400;
">, </span><i><span style="	font-weight: 400;
">StopComponent</span></i><span style="	font-weight: 400;
">, </span><i><span style="	font-weight: 400;
">OnSessionChange</span></i><span style="	font-weight: 400;
">, and </span><i><span style="	font-weight: 400;

<h2><span style="	font-weight: 400;
">File Modifications</span></h2>
<p><span style="	font-weight: 400;
">As mentioned earlier, Stage 2 drops either a 32-bit or 64-bit binary on the victim system. Similar to the Stage 1 dropper, which was a modified version of the legitimate utility CCleaner, the 32-bit and 64-bit binaries are modified versions of VirtCDRDrv.dll (a module developed by Corel, Inc.) and EFACLi64.dll (a module developed by Symantec), respectively. However, it should be noted that unlike the trojanized version of CCleaner, these files are NOT signed.</span></p>
<h3><span style="	font-weight: 400;
">VirtCDRDrv.dll 32-bit</span></h3>
<p><span style="	font-weight: 400;
">Analysis shows that the </span><b>__security_init_cookie </b><span style="	font-weight: 400;
">function of the file has been modified. Normally this function is used as mitigation against buffer overflows; however, in this case, a few extra instructions have been added to the end of the function to initialize a global variable. The image below displays the difference between the clean VirtCDRDrv.dll (labeled primary on the left), and the trojanized one on the right.</span> <img src="/content/dam/crowdstrike/www/en-us/wp/2017/11/ccleaner-pt2/img-2.png"/> <span style="	font-weight: 400;
">The primary difference is the set of instructions at the end of the </span><b>__security_init_cookie</b><span style="	font-weight: 400;
"> function. Specifically, these instructions insert a memory address in a global pointer; the memory address is the image base address &#43; 0x1C22E, inserted in the global variable located at image base address &#43; 0x2ACA4 as seen below.</span> <img src="/content/dam/crowdstrike/www/en-us/wp/2017/11/ccleaner-pt2/img-3.png"/> <span style="	font-weight: 400;
">Once the </span><b><code>__security_init_cookie</code></b><span style="	font-weight: 400;
"> function is done, the </span><b><code>__DLLMainCRTStartup</code></b><span style="	font-weight: 400;
"> function is called, which then makes the call to the function located at the memory address that was inserted into the global variable. This function is responsible for the core functionality of the dropped file. It should be noted that the malicious function is called prior to the entry point of the binary being reached.</span></p>
<h3><span style="	font-weight: 400;
">EFACIi64.dll 64-bit</span></h3>
<p><span style="	font-weight: 400;
">The 32-bit binary and the 64-bit dropped file have been modified in the same manner. As seen in the image below, the only difference in the </span><b><code>__security_init_cookie</code></b><span style="	font-weight: 400;
"> function between the legitimate utility (on the left) and the trojanized version is a jmp instruction at the end of the function.</span> <img src="/content/dam/crowdstrike/www/en-us/wp/2017/11/ccleaner-pt2/img-4.png"/> <span style="	font-weight: 400;

Similar to the 32-bit binary, the malicious function at image base address &#43; 0x12891 is called before the entry point is reached and is responsible for the core functionality of the malware.</span></p>
<h2><span style="	font-weight: 400;
">Dropped Binary</span></h2>
<p><span style="	font-weight: 400;
">Once loaded by the service, the binary reads the registries created earlier by the dropper. It allocates a block of memory and reads the data blob from HKLM\SOFTWARE\Microsoft\Windows NT\CurrentVersion\WbemPerf\002. In addition, it also reads the first 2 DWORDS from HKLM\SOFTWARE\Microsoft\Windows NT\CurrentVersion\WbemPerf\002, XORs them with the value 0x0xF3289317 and prepends them to the data blob. Together, this structure forms a shellcode appended by obfuscated data.</span></p>
