<div class="cmp-text" id="text-5cac6425e8" data-cmp-name="cmp-text-" data-cmp-is="text" data-target-location="false">
    
       <span><p><span style="	font-weight: 400;
">The Advanced Research Team at CrowdStrike Intelligence discovered multiple vulnerabilities affecting libvncclient. In some widely used desktop environments, such as GNOME, these vulnerabilities can be triggered in a one-click fashion.</span></p>
<h2><span style="	font-weight: 400;
">Introduction</span></h2>
<p><span style="	font-weight: 400;
">Client-side exploitation has become a crucial component of many attackers&#39; toolkits. In the desktop space, exploiting browsers is considered to be one of the most impactful capabilities, but due to continuous hardening measures and wide adoption of sandboxing, it is also one of the most complex. However, other components of a typical desktop environment have not been subject to the same scrutiny and can therefore pose risks that go unnoticed. Sparked by our own observations of applications helpfully spawning applications at the click of a link, we decided to investigate the security posture of a typical Linux desktop environment.</span></p>

We therefore decided to take a closer look at </span><a href="https://github.com/LibVNC/libvncserver"><span style="	font-weight: 400;

<h2><span style="	font-weight: 400;
">Code Auditing</span></h2>
<p><a href="https://libvnc.github.io"><span style="	font-weight: 400;
">Libvncclient</span></a><span style="	font-weight: 400;
"> implements the </span><a href="https://datatracker.ietf.org/doc/html/rfc6143"><span style="	font-weight: 400;
">VNC protocol</span></a><span style="	font-weight: 400;
">, which is based on the concept of a remote frame buffer. It offers functionality to transfer keystrokes and mouse inputs from the client to the server while relaying the server’s graphical desktop as framebuffer updates to the client. The original specification of VNC’s underlying remote framebuffer protocol (RFB) was released in 1998 and was relatively simple. However, over the years, various implementations added further features to optimize different aspects of the protocol, many of them aimed at reducing the amount of data that needs to be transferred for framebuffer updates. On the flip side, these optimizations naturally also resulted in more complexity and therefore an increased </span><a href="https://cve.mitre.org/cgi-bin/cvekey.cgi?keyword&#61;vnc"><span style="	font-weight: 400;
">attack surface</span></a><span style="	font-weight: 400;
">.</span> <span style="	font-weight: 400;
">Two of these optimizations originate from the </span><a href="https://www.uvnc.com"><span style="	font-weight: 400;
">UltraVNC</span></a><span style="	font-weight: 400;
"> and </span><a href="https://www.tightvnc.com"><span style="	font-weight: 400;
">TightVNC</span></a><span style="	font-weight: 400;
"> protocol implementations</span></p>
<p> </p>
<p>and use custom encoding schemes. For compatibility reasons, both of these schemes are also supported by libvncclient. As outlined in the following sections, a manual code audit uncovered two distinct memory corruption vulnerabilities in the handling of these encoding schemes.</p>
<p> </p>
<h3><span style="	font-weight: 400;
">Heap Buffer Overflow in Ultra Encoding</span></h3>
<p><span style="	font-weight: 400;
">Messages from the VNC server to the client are handled by the </span><code><b>HandleRFBServerMessage()</b></code><span style="	font-weight: 400;
"> function. That function first reads the message type and then continues to process the rest of the message accordingly. Framebuffer updates are transmitted in messages of type </span><code><b>rfbFramebufferUpdate</b></code><span style="	font-weight: 400;
">. These messages will then contain information about the number of rectangular framebuffer updates contained in the message, and for each rectangle, the data encoding (e.g., </span><code><b>rfbEncodingUltra</b></code><span style="	font-weight: 400;
"> for UltraVNC encoding). The following source code excerpt from libvncclient shows the function </span><code><b>HandleUltraBPP()</b></code><span style="	font-weight: 400;
">, which implements the processing of UltraVNC-encoded framebuffer updates:</span></p>
<pre>
<code><b>static rfbBool

  int toRead&#61;0;

</pre>
<p><span style="	font-weight: 400;
">The string </span><code><b>BPP</b></code><span style="	font-weight: 400;
"> in the function name will be expanded by the C preprocessor into three different functions for handling frame buffer updates with 8, 16 or 32 bits of color information per pixel. <pre>
<code><b>&lt;...&gt;

<pre>
<code><b>&lt;...&gt;

<p><span style="	font-weight: 400;
">A malicious VNC server is able to fully control the value of the </span><code><b>toRead</b></code> <span style="	font-weight: 400;
">variable, including negative values. As a result, </span><code><b>ultra_buffer</b></code><span style="	font-weight: 400;
"> can be overflowed by first sending a frame buffer update that leads to an allocation of a certain size, followed by a second update that sets </span><code><b>toRead</b></code><span style="	font-weight: 400;

<p> </p>
<p><span style="	font-weight: 400;
">As </span><code><b>ReadFromRFBServer()</b></code> <span style="	font-weight: 400;
">expects an unsigned integer value, the implicit type conversion ultimately leads to a </span><code><b>read()</b></code><span style="	font-weight: 400;

">an overly large </span><code><b>count</b></code><span style="	font-weight: 400;
"> value. The exact amount of data that is overwritten can be controlled by closing the underlying socket connection early.</span></p>
<p> </p>

<h2><span style="	font-weight: 400;

<p><span style="	font-weight: 400;
">In addition to manually auditing some of the libvncclient code base, we also made use of </span><a href="https://github.com/AFLplusplus/AFLplusplus"><span style="	font-weight: 400;

<h2><i><span style="	font-weight: 400;
">Exploitability</span></i></h2>
<p><i><span style="	font-weight: 400;
">While the outlined vulnerabilities allow to achieve an arbitrary write primitive and to fully control the instruction pointer, the presence of address space layout randomization (ASLR) makes exploitation difficult in a default Ubuntu x64 desktop environment. This is due to the fact that Remmina itself is compiled as a position-independent executable and is therefore similar to libraries loaded at a randomized location. Thus, an attacker has no fixed addresses to rely on. However, in case one of the vulnerabilities can be turned into an information leak, or by finding a separate bug allowing to disclose memory addresses, the outlined vulnerabilities would immediately allow for remote code execution.</span></i></p>
<h2><i><span style="	font-weight: 400;
">Conclusion</span></i></h2>
<p><i><span style="	font-weight: 400;
">While Linux distributions have begun to provide </span><a href="https://ubuntu.com/core/docs/security-and-sandboxing"><span style="	font-weight: 400;
">application-specific isolation mechanisms</span></a><span style="	font-weight: 400;
"> and have also started to integrate </span><a href="https://gitlab.gnome.org/GNOME/tracker-miners/-/blob/master/src/libtracker-miners-common/tracker-seccomp.c"><span style="	font-weight: 400;
">sandboxing techniques in critical components</span></a><span style="	font-weight: 400;
"> that expose complex attack surfaces, many other parts of typical desktop environments have not been hardened in a similar fashion and remain comparably easy targets for exploitation. In the case presented here, we were able to identify several memory corruption bugs that can be triggered with minimal user interaction due to vulnerable URL handlers.</span></i></p>
<p><i> </i></p>
<p><i> <span style="	font-weight: 400;
">Some of these were also identified through fuzzing, and we’ll document our general methodology in a subsequent blog post. The research is by far not exhaustive and should only be considered to serve as an example of the extensive attack surface exposed by custom URL handlers.</span> <span style="	font-weight: 400;
">It is worth noting that Linux isn’t the only platform that exposes this type of attack surface. In fact, the recently disclosed MSHTML remote code execution vulnerability (CVE-2021-40444 — for more information, see the </span><a href="/en-us/blog/patch-tuesday-analysis-september-2021/"><span style="	font-weight: 400;
">September 2021 Patch Tuesday</span></a><span style="	font-weight: 400;
"> update) on Windows operating systems was also primarily enabled by the platform supporting and automatically adding custom protocol handlers for known file types.</span> <span style="	font-weight: 400;
">Finally, we would like to thank Christian Beier of the libvnc project for working with us and developing patches for the bugs.</span><h4><span style="	font-weight: 400;
