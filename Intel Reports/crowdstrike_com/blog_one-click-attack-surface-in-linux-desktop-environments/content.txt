        <h1>Scheming with URLs: One-Click Attack Surface in Linux Desktop Environments</h1>
        
        <div class="publish_info">
            <p>November 09, 2021</p>
            <span>|</span>
            <a href="/en-us/blog/author.lukas-kupczyk---max-julian-hofmann/" title="Posts by Lukas Kupczyk - Max Julian Hofmann" rel="author">
                Lukas Kupczyk - Max Julian Hofmann</a>
            <span>|</span>
                
            <a href="/en-us/blog/category.counter-adversary-operations/" title="Counter Adversary Operations">Counter Adversary Operations</a>
        </div>
        <div class="post_image">
            <img width="1060" height="698" src="https://assets.crowdstrike.com/is/image/crowdstrikeinc/Blog_1060x698-12-us" class="attachment-post-thumbnail size-post-thumbnail wp-post-image" alt="" decoding="async" fetchpriority="high" srcset="/content/dam/crowdstrike/www/en-us/wp/2024/12/Blog_1060x698-12-us.jpeg 1060w" sizes="(max-width: 1060px) 100vw, 1060px"/>
        </div>
    

    
</div>

    <div style="display: none;">
        
    </div>


    
</div>
<div class="text text--blog-content aem-GridColumn aem-GridColumn--default--12">

  

<div class="cmp-text" id="text-5cac6425e8" data-cmp-name="cmp-text-" data-cmp-is="text" data-target-location="false">
    
       <span><p><span style="	font-weight: 400;
">The Advanced Research Team at CrowdStrike Intelligence discovered multiple vulnerabilities affecting libvncclient. In some widely used desktop environments, such as GNOME, these vulnerabilities can be triggered in a one-click fashion.</span></p>
<h2><span style="	font-weight: 400;
">Introduction</span></h2>
<p><span style="	font-weight: 400;
">Client-side exploitation has become a crucial component of many attackers&#39; toolkits. In the desktop space, exploiting browsers is considered to be one of the most impactful capabilities, but due to continuous hardening measures and wide adoption of sandboxing, it is also one of the most complex. However, other components of a typical desktop environment have not been subject to the same scrutiny and can therefore pose risks that go unnoticed. Sparked by our own observations of applications helpfully spawning applications at the click of a link, we decided to investigate the security posture of a typical Linux desktop environment.</span></p>
<h2><span style="	font-weight: 400;
">URL Scheme Handlers</span></h2>
<p><span style="	font-weight: 400;
">Whether the desktop starts an application when the user clicks on a link (e.g., in an email or an instant message) depends on whether a handler application is registered for a URL scheme. In a stock Ubuntu 21.04 desktop based on Gnome, there are various applications registered as handlers for specific URL schemes. A full list of these handlers can be retrieved by grepping the </span><b><code>mimeinfo.cache</code></b><span style="	font-weight: 400;
"> file as follows:</span></p>
<pre>
<b><code>$ grep x-scheme-handler /usr/share/applications/mimeinfo.cache 
x-scheme-handler/apt&#61;apturl.desktop;
x-scheme-handler/chrome&#61;firefox.desktop;
x-scheme-handler/ftp&#61;firefox.desktop;
x-scheme-handler/ghelp&#61;yelp.desktop;
x-scheme-handler/help&#61;yelp.desktop;
x-scheme-handler/http&#61;firefox.desktop;
x-scheme-handler/https&#61;firefox.desktop;
x-scheme-handler/icy&#61;org.gnome.Totem.desktop;
x-scheme-handler/icyx&#61;org.gnome.Totem.desktop;
x-scheme-handler/info&#61;yelp.desktop;
x-scheme-handler/magnet&#61;transmission-gtk.desktop;
x-scheme-handler/mailto&#61;thunderbird.desktop;
x-scheme-handler/man&#61;yelp.desktop;
x-scheme-handler/mms&#61;org.gnome.Totem.desktop;
x-scheme-handler/mmsh&#61;org.gnome.Totem.desktop;
x-scheme-handler/net&#61;org.gnome.Totem.desktop;
x-scheme-handler/pnm&#61;org.gnome.Totem.desktop;
x-scheme-handler/rdp&#61;org.remmina.Remmina.desktop;remmina-file.desktop;
x-scheme-handler/remmina&#61;org.remmina.Remmina.desktop;remmina-file.desktop;
x-scheme-handler/rtmp&#61;org.gnome.Totem.desktop;
x-scheme-handler/rtp&#61;org.gnome.Totem.desktop;
x-scheme-handler/rtsp&#61;org.gnome.Totem.desktop;
x-scheme-handler/snap&#61;snap-handle-link.desktop;
x-scheme-handler/spice&#61;org.remmina.Remmina.desktop;remmina-file.desktop;
x-scheme-handler/uvox&#61;org.gnome.Totem.desktop;
x-scheme-handler/vnc&#61;org.remmina.Remmina.desktop;remmina-file.desktop;
x-scheme-handler/vnd.libreoffice.cmis&#61;libreoffice-startcenter.desktop;</code></b>
</pre>
<p><span style="	font-weight: 400;
">While a URL starting with a custom scheme might seem suspicious to cautious users, the exact target of a link can be hidden in applications that allow embedding HTML markup. An example of such an application is the personal information management application </span><a href="https://wiki.gnome.org/Apps/Evolution"><span style="	font-weight: 400;
">Evolution</span></a><span style="	font-weight: 400;
">:</span></p>
<div class="wp-caption" id="attachment_48825"><img class="size-full wp-image-48825" src="/content/dam/crowdstrike/www/en-us/wp/2021/11/Picture1-1.png" alt="" width="1370" height="999"/>Figure 1. Example of how a link target can be hidden in the Evolution application</div>
<p><span style="	font-weight: 400;
">In the previous URL handler listing, we have observed that the remote administration tool </span><a href="https://remmina.org"><span style="	font-weight: 400;
">Remmina</span></a><span style="	font-weight: 400;
"> is registered for the URL schemes </span><code><b>rdp</b></code><span style="	font-weight: 400;
">, </span><code><b>remmina</b></code><span style="	font-weight: 400;
">, </span><code><b>spice</b></code><span style="	font-weight: 400;
"> and </span><code><b>vnc</b></code><span style="	font-weight: 400;
">. The corresponding </span><code><b>.desktop</b></code><span style="	font-weight: 400;
"> file specified as part of the handler definition contains a reference to the shell script </span><code><b>remmina-file-wrapper</b></code><span style="	font-weight: 400;
">, which receives the specified URL as its second argument (</span><code><b>%U</b></code><span style="	font-weight: 400;
"> placeholder):</span></p>
<pre>
<code><b>$ cat /usr/share/applications/remmina-file.desktop

&lt;...&gt;
Exec&#61;remmina-file-wrapper -c %U
Icon&#61;org.remmina.Remmina
MimeType&#61;application/x-remmina;x-scheme-handler/remmina;x-scheme-handler/rdp;x-scheme-handler/spice;x-scheme-handler/vnc;
&lt;...&gt;</b></code>
</pre>
<p><span style="	font-weight: 400;
">That shell script then starts the actual Remmina binary in case a matching URL is specified:</span></p>
<pre>
<code><b>$ cat /usr/bin/remmina-file-wrapper
#!/usr/bin/env bash
&lt;...&gt;

REMMINA&#61;&#34;/usr/bin/remmina&#34;

if &lt;&lt; ! -f &#34;$REMMINA&#34; &gt;&gt; ; then
    REMMINA&#61;&#34;${USRBIN}/remmina&#34;
else
    REMMINA&#61;&#34;remmina&#34;
fi

export GLADE_HOME&#61;&#34;$USRBIN/../share/remmina/ui.html&#34;

case &#34;$&#64;&#34; in
    *rdp:*)
        &#34;$REMMINA&#34; &#34;${&#64;#rdp:\/\/}&#34;
        ;;
    *spice:*)
        &#34;$REMMINA&#34; &#34;${&#64;#spice:\/\/}&#34;
        ;;
    *vnc:*)
        &#34;$REMMINA&#34; &#34;${&#64;#vnc:\/\/}&#34;
        ;;
    *remmina:*)
        &#34;$REMMINA&#34; &#34;${&#64;#remmina:\/\/}&#34;
        ;;
    *)
        &#34;$REMMINA&#34; &#34;${&#64;}&#34;
        ;;
esac</b></code>
</pre>
<p><span style="	font-weight: 400;
">These URL handlers therefore open up quite a large <a href="https://www.crowdstrike.com/cybersecurity-101/attack-surface/">attack surface</a> that is reachable with minimal user interaction. As an example, clicking on a URL such as </span><code><b>vnc://user:pass&#64;example.com/</b></code><span style="	font-weight: 400;
"> automatically starts Remmina and instantly establishes a VNC connection with the given credentials to the specified remote host. Any vulnerability affecting Remmina’s VNC protocol implementation would therefore potentially allow for one-click exploitation scenarios. We therefore decided to take a closer look at </span><a href="https://github.com/LibVNC/libvncserver"><span style="	font-weight: 400;
">libvncclient</span></a><span style="	font-weight: 400;
">, which is used by Remmina for its VNC support.</span></p>
<h2><span style="	font-weight: 400;
">Code Auditing</span></h2>
<p><a href="https://libvnc.github.io"><span style="	font-weight: 400;
">Libvncclient</span></a><span style="	font-weight: 400;
"> implements the </span><a href="https://datatracker.ietf.org/doc/html/rfc6143"><span style="	font-weight: 400;
">VNC protocol</span></a><span style="	font-weight: 400;
">, which is based on the concept of a remote frame buffer. It offers functionality to transfer keystrokes and mouse inputs from the client to the server while relaying the server’s graphical desktop as framebuffer updates to the client. The original specification of VNC’s underlying remote framebuffer protocol (RFB) was released in 1998 and was relatively simple. However, over the years, various implementations added further features to optimize different aspects of the protocol, many of them aimed at reducing the amount of data that needs to be transferred for framebuffer updates. On the flip side, these optimizations naturally also resulted in more complexity and therefore an increased </span><a href="https://cve.mitre.org/cgi-bin/cvekey.cgi?keyword&#61;vnc"><span style="	font-weight: 400;
">attack surface</span></a><span style="	font-weight: 400;
">.</span> <span style="	font-weight: 400;
">Two of these optimizations originate from the </span><a href="https://www.uvnc.com"><span style="	font-weight: 400;
">UltraVNC</span></a><span style="	font-weight: 400;
"> and </span><a href="https://www.tightvnc.com"><span style="	font-weight: 400;
">TightVNC</span></a><span style="	font-weight: 400;
"> protocol implementations</span></p>
<p> </p>
<p>and use custom encoding schemes. For compatibility reasons, both of these schemes are also supported by libvncclient. As outlined in the following sections, a manual code audit uncovered two distinct memory corruption vulnerabilities in the handling of these encoding schemes.</p>
<p> </p>
<h3><span style="	font-weight: 400;
">Heap Buffer Overflow in Ultra Encoding</span></h3>
<p><span style="	font-weight: 400;
">Messages from the VNC server to the client are handled by the </span><code><b>HandleRFBServerMessage()</b></code><span style="	font-weight: 400;
"> function. That function first reads the message type and then continues to process the rest of the message accordingly. Framebuffer updates are transmitted in messages of type </span><code><b>rfbFramebufferUpdate</b></code><span style="	font-weight: 400;
">. These messages will then contain information about the number of rectangular framebuffer updates contained in the message, and for each rectangle, the data encoding (e.g., </span><code><b>rfbEncodingUltra</b></code><span style="	font-weight: 400;
"> for UltraVNC encoding). The following source code excerpt from libvncclient shows the function </span><code><b>HandleUltraBPP()</b></code><span style="	font-weight: 400;
">, which implements the processing of UltraVNC-encoded framebuffer updates:</span></p>
<pre>
<code><b>static rfbBool
HandleUltraBPP (rfbClient* client, int rx, int ry, int rw, int rh)
{
  rfbZlibHeader hdr;
  int toRead&#61;0;
&lt;...&gt;
  lzo_uint uncompressedBytes &#61; (( rw * rh ) * ( BPP / 8 ));

  if (!ReadFromRFBServer(client, (char *)&amp;hdr, sz_rfbZlibHeader))
    return FALSE;

  toRead &#61; rfbClientSwap32IfLE(hdr.nBytes);
&lt;...&gt;</b></code>
</pre>
<p><span style="	font-weight: 400;
">The string </span><code><b>BPP</b></code><span style="	font-weight: 400;
"> in the function name will be expanded by the C preprocessor into three different functions for handling frame buffer updates with 8, 16 or 32 bits of color information per pixel. As shown in the listing, after reading the header from the socket via </span><code><b>ReadFromRFBServer()</b></code><span style="	font-weight: 400;
">, its field </span><code><b>nBytes</b></code><span style="	font-weight: 400;
"> is used to set the </span><i><span style="	font-weight: 400;
">signed</span></i><span style="	font-weight: 400;
"> integer variable </span><code><b>toRead</b></code><span style="	font-weight: 400;
">. Afterward, </span><code><b>toRead</b></code><span style="	font-weight: 400;
"> is used as the size to allocate a 4-byte aligned buffer (</span><code><b>ultra_buffer</b></code><span style="	font-weight: 400;
">) in case no previous one has been allocated in a prior invocation of the function, or if the previous one has an insufficient size:</span></p>
<pre>
<code><b>&lt;...&gt;
  /* allocate enough space to store the incoming compressed packet */
  if ( client-&gt;ultra_buffer_size &lt; toRead ) { if ( client-&gt;ultra_buffer !&#61; NULL ) {
      free( client-&gt;ultra_buffer );
    }
    client-&gt;ultra_buffer_size &#61; toRead;
    /* buffer needs to be aligned on 4-byte boundaries */
    if ((client-&gt;ultra_buffer_size % 4)!&#61;0)
      client-&gt;ultra_buffer_size &#43;&#61; (4-(client-&gt;ultra_buffer_size % 4));
    client-&gt;ultra_buffer &#61; (char*) malloc( client-&gt;ultra_buffer_size );
&lt;...&gt;</b></code>
</pre>
<p><span style="	font-weight: 400;
">Afterward, the code reads the amount of data specified by </span><code><b>toRead</b></code><span style="	font-weight: 400;
"> from the socket:</span></p>
<pre>
<code><b>&lt;...&gt;
 /* Fill the buffer, obtaining data from the server. */
  if (!ReadFromRFBServer(client, client-&gt;ultra_buffer, toRead))
      return FALSE;
&lt;...&gt;</b></code>
</pre>
<p><span style="	font-weight: 400;
">A malicious VNC server is able to fully control the value of the </span><code><b>toRead</b></code> <span style="	font-weight: 400;
">variable, including negative values. As a result, </span><code><b>ultra_buffer</b></code><span style="	font-weight: 400;
"> can be overflowed by first sending a frame buffer update that leads to an allocation of a certain size, followed by a second update that sets </span><code><b>toRead</b></code><span style="	font-weight: 400;
"> to a negative value. Then, the buffer allocated during the first frame buffer update can be overflowed by the second one, as the condition </span><code><b>client-&gt;ultra_buffer_size &lt; toRead</b></code> <span style="	font-weight: 400;
">will not be fulfilled, leading to a call to </span><code><b>ReadFromRFBServer()</b></code><span style="	font-weight: 400;
"> where </span><code><b>toRead</b></code> <span style="	font-weight: 400;
">is negative.</span></p>
<p> </p>
<p><span style="	font-weight: 400;
">As </span><code><b>ReadFromRFBServer()</b></code> <span style="	font-weight: 400;
">expects an unsigned integer value, the implicit type conversion ultimately leads to a </span><code><b>read()</b></code><span style="	font-weight: 400;
"> call with</span> <span style="	font-weight: 400;
">an overly large </span><code><b>count</b></code><span style="	font-weight: 400;
"> value. The exact amount of data that is overwritten can be controlled by closing the underlying socket connection early.</span></p>
<p> </p>
<h3><span style="	font-weight: 400;
">Intra-Struct/Heap Overflow in Tight Encoding</span></h3>
<p><span style="	font-weight: 400;
">In case Tight encoding is used for framebuffer updates, the function </span><code><b>HandleTightBPP()</b></code><span style="	font-weight: 400;
"> is invoked (similar to Ultra encoding, </span><code><b>BPP</b></code><span style="	font-weight: 400;
"> is a placeholder for the preprocessor). As shown in the following listing, the function reads the value </span><code><b>comp_ctl</b></code><span style="	font-weight: 400;
"> from the underlying socket:</span></p>
<pre>
<code><b>static rfbBool
HandleTightBPP(rfbClient *client, int rx, int ry, int rw, int rh)
{
  CARDBPP fill_colour;
  uint8_t comp_ctl;
  uint8_t filter_id;
  filterPtrBPP filterFn;
  z_streamp zs;
  int err, stream_id, compressedLen, bitsPixel;
  int bufferSize, rowSize, numRows, portionLen, rowsProcessed, extraBytes;
  rfbBool readUncompressed &#61; FALSE;

  if (client-&gt;frameBuffer &#61;&#61; NULL)
    return FALSE;

  if (rx &#43; rw &gt; client-&gt;width || ry &#43; rh &gt; client-&gt;height)
  {
    rfbClientLog(&#34;Rect out of bounds: %dx%d at (%d, %d)\n&#34;, rx, ry, rw, rh);
    return FALSE;
  }

  if (!ReadFromRFBServer(client, (char *)&amp;comp_ctl, 1))
    return FALSE;
&lt;...&gt;</b></code>
</pre>
<p><span style="	font-weight: 400;
">That value is used by the function to make a number of assumptions regarding the exact format — e.g., it is used to select a filter function, defines the type of rectangle, and signals whether compression is used. Later, the function </span><code><b>ReadCompactLen()</b></code><span style="	font-weight: 400;
"> is invoked to read the value of </span><code><b>compressedLen</b></code><span style="	font-weight: 400;
"> from the underlying socket:</span></p>
<pre>
<b><code>&lt;...&gt;
  /* Read the length (1..3 bytes) of compressed data following. */
  compressedLen &#61; (int)ReadCompactLen(client)
&lt;...&gt;</code></b>
</pre>
<p><code><b>ReadCompactLen()</b></code><span style="	font-weight: 400;
"> allows the server to specify a length value that uses up to three bytes. The most significant bits of the first one or two bytes specify whether another byte follows. Therefore, the specified size can consist of up to 22 bits (7 &#43; 7 &#43; 8). The maximum integer value that can be returned by the function is therefore 4,194,303 (2 ^ 22 - 1).</span> <span style="	font-weight: 400;
">In case the client expects uncompressed data (indicated by </span><code><b>comp_ctl</b></code><span style="	font-weight: 400;
">), </span><code><b>compressedLen</b></code><span style="	font-weight: 400;
"> is passed to </span><code><b>ReadFromRFBServer()</b></code><span style="	font-weight: 400;
"> as the amount of data that is read into </span><code><b>client-&gt;buffer</b></code><span style="	font-weight: 400;
">:</span></p>
<pre>
<b><code>&lt;...&gt;
 if (readUncompressed)
  {
    if (!ReadFromRFBServer(client, (char *)client-&gt;buffer, compressedLen))
      return FALSE;
&lt;...&gt;</code></b>
</pre>
<p><span style="	font-weight: 400;
">As shown below, </span><code><b>client-&gt;buffer</b></code><span style="	font-weight: 400;
"> is a fixed-size array (307,200 bytes) located in the central </span><code><b>rfbClient</b></code><span style="	font-weight: 400;
"> struct, which is heap-allocated for each connection:</span></p>
<pre>
<b><code>typedef struct _rfbClient {
	uint8_t* frameBuffer;
&lt;...&gt;
	int serverPort; /**&lt; if -1, then use file recorded by vncrec */ rfbBool listenSpecified; int listenPort, flashPort; struct { int x, y, w, h; } updateRect; /** Note that the CoRRE encoding uses this buffer and assumes it is big enough to hold 255 * 255 * 32 bits -&gt; 260100 bytes.  640*480 &#61; 307200 bytes.
	   Hextile also assumes it is big enough to hold 16 * 16 * 32 bits.
	   Tight encoding assumes BUFFER_SIZE is at least 16384 bytes. */

#define RFB_BUFFER_SIZE (640*480)
	char buffer;
	char *bufoutptr;
	unsigned int buffered;
&lt;...&gt;</code></b>
</pre>
<p><span style="	font-weight: 400;
">As the maximum size of </span><code><b>compressedLen</b></code><span style="	font-weight: 400;
"> can exceed the size of the </span><code><b>buffer</b></code><span style="	font-weight: 400;
"> struct member, the server is able to overflow said buffer and fully control the subsequent members of the </span><code><b>rfbClient</b></code><span style="	font-weight: 400;
"> struct as well as adjacent heap memory. This includes pointers through which an arbitrary write primitive can be achieved as well as function pointers that allow hijacking of</span></p>
<p> </p>
<p>the control flow.</p>
<p> </p>
<h2><span style="	font-weight: 400;
">Fuzzing</span></h2>
<p><span style="	font-weight: 400;
">In addition to manually auditing some of the libvncclient code base, we also made use of </span><a href="https://github.com/AFLplusplus/AFLplusplus"><span style="	font-weight: 400;
">afl&#43;&#43;</span></a><span style="	font-weight: 400;
"> to conduct fuzzing tests. To do so, we developed a thin wrapper around libvncclient’s main message dispatching function </span><code><b>HandleRFBServerMessage()</b></code><span style="	font-weight: 400;
">. The argument for this function is a client object that is configured to read framebuffer updates from </span><code><b>stdin</b></code><span style="	font-weight: 400;
">. An introduction to the general methodology we applied to fuzzing in this case will be presented in a subsequent blog post.</span> <span style="	font-weight: 400;
">Through fuzzing, we uncovered two additional bugs that are outlined below.</span></p>
<h3><span style="	font-weight: 400;
">Heap Buffer Overflow in TRLE</span></h3>
<p><span style="	font-weight: 400;
">In case a TRLE-encoded framebuffer update is received, the function </span><code><b>HandleTRLE()</b></code><span style="	font-weight: 400;
"> is invoked. As shown in the following source code excerpt, the function contains a loop that receives an arbitrary amount of </span><code><b>0xFF</b></code><span style="	font-weight: 400;
"> bytes from the underlying socket into the heap-allocated </span><code><b>client-&gt;raw_buffer</b></code><span style="	font-weight: 400;
">:</span></p>
<pre>
<code>static rfbBool HandleTRLE(rfbClient *client, int rx, int ry, int rw, int rh) {
&lt;...&gt;
uint8_t *buffer;
&lt;...&gt;
buffer &#61; (uint8_t*)(client-&gt;raw_buffer);
&lt;...&gt;
while (*buffer &#61;&#61; 0xff) {
if (!ReadFromRFBServer(client, (char*)buffer &#43; 1, 1))
return FALSE;
length &#43;&#61; *buffer;
buffer&#43;&#43;;
}
&lt;...&gt;
}</code>
</pre>
<p><span style="	font-weight: 400;
">As there is no bounds check present, the buffer can be overflown by a malicious server.</span></p>
<h3><span style="	font-weight: 400;
">Stack Buffer Overflow in ZRLE Encoding</span></h3>
<p><span style="	font-weight: 400;
">In case a ZRLE-encoded framebuffer update is received, the function </span><code><b>HandleZRLE()</b></code><span style="	font-weight: 400;
"> is invoked. As shown in the following source code excerpt, the function first reads a structure of type </span><code><b>rfbZRLEHeader</b></code><span style="	font-weight: 400;
"> from the server. Afterward, additional zlib-compressed data is received.</span></p>
<pre>
<b><code>static rfbBool HandleZRLE (rfbClient* client, int rx, int ry, int rw, int rh)
{

while (( remaining &gt; 0 ) &amp;&amp; ( inflateResult &#61;&#61; Z_OK )) {
&lt;...&gt;

		/* Fill the buffer, obtaining data from the server. */
		if (!ReadFromRFBServer(client, client-&gt;buffer,toRead))
			return FALSE;

		client-&gt;decompStream.next_in  &#61; ( Bytef * )client-&gt;buffer;
		client-&gt;decompStream.avail_in &#61; toRead;

		/* Need to uncompress buffer full. */
		inflateResult &#61; inflate( &amp;client-&gt;decompStream, Z_SYNC_FLUSH );
&lt;...&gt;
	}

	if ( inflateResult &#61;&#61; Z_OK ) {
&lt;...&gt;
		for(j&#61;0; j&lt;rh; j&#43;&#61;rfbZRLETileHeight)
			for(i&#61;0; i&lt;rw; i&#43;&#61;rfbZRLETileWidth) {
&lt;...&gt;
int result&#61;HandleZRLETile(
client,
(uint8_t *)buf,
remaining,
rx&#43;i,
ry&#43;j,
subWidth,
subHeight
);
&lt;...&gt;
}</code></b>
</pre>
<p><span style="	font-weight: 400;
">The decompressed additional data is then passed to the function </span><code><b>HandleZRLETILE()</b></code><span style="	font-weight: 400;
"> as the parameter </span><code><b>buf</b></code><span style="	font-weight: 400;
">. Its first byte determines the </span><code><b>type</b></code><span style="	font-weight: 400;
"> of tile that is to be processed:</span></p>
<pre>
<code>static int HandleZRLETile(rfbClient* client, uint8_t* buffer,size_t buffer_length, int x,int y,int w,int h)
{
&lt;...&gt;
uint8_t type;
&lt;...&gt;
	type &#61; *buffer;
	buffer&#43;&#43;;
&lt;...&gt;
	if( type &#61;&#61; 0 ) /* raw *
&lt;...&gt;
	else if( type &#61;&#61; 1 ) /* solid */
&lt;...&gt;
else if( type &lt;&#61; 127 ) { /* packed Palette */
		CARDBPP palette&lt;16&gt;;
		int i,j,shift,
&lt;...&gt;
		/* read palette */
		for(i&#61;0; i&lt;type; i&#43;&#43;,buffer&#43;&#61;REALBPP/8)
			palette<i> &#61; UncompressCPixel(buffer);
&lt;...&gt;
}</i></code>
</pre>
<p><i><span style="	font-weight: 400;
">As shown in the above source code excerpt, in case </span><code><b>type</b></code><span style="	font-weight: 400;
"> is neither 0 nor 1 but less than or equal to 127, a branch that handles a packed palette is taken. In this case, a stack-allocated array of 16 elements named </span><code><b>palette</b></code><span style="	font-weight: 400;
"> is declared. The exact value of </span><code><b>type</b></code><span style="	font-weight: 400;
"> then determines how many elements are read into the </span><code><b>palette</b></code><span style="	font-weight: 400;
"> array inside a for-loop. In case </span><code><b>type</b></code><span style="	font-weight: 400;
">’s value exceeds 16, an out-of-bounds write on the stack occurs.</span></i></p>
<h2><i><span style="	font-weight: 400;
">Exploitability</span></i></h2>
<p><i><span style="	font-weight: 400;
">While the outlined vulnerabilities allow to achieve an arbitrary write primitive and to fully control the instruction pointer, the presence of address space layout randomization (ASLR) makes exploitation difficult in a default Ubuntu x64 desktop environment. This is due to the fact that Remmina itself is compiled as a position-independent executable and is therefore similar to libraries loaded at a randomized location. Thus, an attacker has no fixed addresses to rely on. However, in case one of the vulnerabilities can be turned into an information leak, or by finding a separate bug allowing to disclose memory addresses, the outlined vulnerabilities would immediately allow for remote code execution.</span></i></p>
<h2><i><span style="	font-weight: 400;
">Conclusion</span></i></h2>
<p><i><span style="	font-weight: 400;
">While Linux distributions have begun to provide </span><a href="https://ubuntu.com/core/docs/security-and-sandboxing"><span style="	font-weight: 400;
">application-specific isolation mechanisms</span></a><span style="	font-weight: 400;
"> and have also started to integrate </span><a href="https://gitlab.gnome.org/GNOME/tracker-miners/-/blob/master/src/libtracker-miners-common/tracker-seccomp.c"><span style="	font-weight: 400;
">sandboxing techniques in critical components</span></a><span style="	font-weight: 400;
"> that expose complex attack surfaces, many other parts of typical desktop environments have not been hardened in a similar fashion and remain comparably easy targets for exploitation. In the case presented here, we were able to identify several memory corruption bugs that can be triggered with minimal user interaction due to vulnerable URL handlers.</span></i></p>
<p><i> </i></p>
<p><i> <span style="	font-weight: 400;
">Some of these were also identified through fuzzing, and we’ll document our general methodology in a subsequent blog post. The research is by far not exhaustive and should only be considered to serve as an example of the extensive attack surface exposed by custom URL handlers.</span> <span style="	font-weight: 400;
">It is worth noting that Linux isn’t the only platform that exposes this type of attack surface. In fact, the recently disclosed MSHTML remote code execution vulnerability (CVE-2021-40444 — for more information, see the </span><a href="/en-us/blog/patch-tuesday-analysis-september-2021/"><span style="	font-weight: 400;
">September 2021 Patch Tuesday</span></a><span style="	font-weight: 400;
"> update) on Windows operating systems was also primarily enabled by the platform supporting and automatically adding custom protocol handlers for known file types.</span> <span style="	font-weight: 400;
">Finally, we would like to thank Christian Beier of the libvnc project for working with us and developing patches for the bugs.</span><h4><span style="	font-weight: 400;
