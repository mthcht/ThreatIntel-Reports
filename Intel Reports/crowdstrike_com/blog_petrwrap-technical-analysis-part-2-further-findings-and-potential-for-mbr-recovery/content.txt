<div class="cmp-text" id="text-ff9ca030d9" data-cmp-name="cmp-text-" data-cmp-is="text" data-target-location="false">
    
       <span><h2><i>Update</i>:</h2>
<p>Due to naming convention consistency in the industry, CrowdStrike is now calling this variant of Petya - NotPetya.</p>
<hr />

<h2><b>Executive Summary</b></h2>
<p>This technical analysis is a continuation of the previous technical blog (<a href="/en-us/blog/petrwrap-ransomware-technical-analysis-triple-threat-file-encryption-mft-encryption-credential-theft/">NotPetya Technical Analysis – A Triple Threat: File Encryption, MFT Encryption, Credential Theft</a>) describing the threat of NotPetya, a destructive malware with self-propagation capabilities. After further analysis, CrowdStrike researchers discovered:</p>
<ul><li>How the NotPetya DLL loads functions to ensure proper cleanup from the victim machine</li><li>The order in which the infection process takes place</li><li>Potential for recovery of the victim machine’s MBR and boot manager when running one specific antivirus software</li></ul>
<h2><b>NotPetya DLL Loader</b></h2>
<p>The NotPetya DLL, seen in the wild with the filename “perfc.dat”, takes the following steps during the DLL loading process to ensure that there is no trace of the NotPetya DLL (perfc.dll) having being on the system:</p>
<ul><li>Copies file contents from disk into a buffer residing in process memory</li><li>Checks to determine if the DLL has already been loaded</li><li>Copies the NotPetya DLL already mapped into process memory to another buffer</li><li>Calculates and patches the relative offsets in the NotPetya copy with a new offset</li><li>The new relative offset is calculated by subtracting the base address original NotPetya DLL from the RVAs identified in the relocation table of the NotPetya copy</li><li>Calls the function at offset &#43;0x94A5 located in the NotPetya copy<ul><li>Calls FreeLibrary to unmap the original NotPetya DLL from process memory</li><li>Overwrites the original DLL file with null bytes</li><li>Deletes the original NotPetya DLL from the file system</li><li>Fixes the import table of the NotPetya DLL copy</li><li>Calls the perfc_1 (export ordinal 1) function to kick off the worm activity and file/MFT encryption</li></ul>
</li></ul>
<p>The file cleanup combined with the file and MFT encryption process make it difficult to recover the original NotPetya DLL.</p>
<h2><b>SMB Exploitation and Infection</b></h2>
<p>The EternalBlue and EternalRomance exploits are used for the MS17-010 vulnerability, and are subsequently used to propagate NotPetya to vulnerable hosts on the local subnet. The infection process occurs in the following order:</p>
<ol><li>Test for vulnerable condition</li><li>Check Windows version</li><li>Trigger MS17-010 vulnerability. For more information, please click <a href="https://technet.microsoft.com/en-us/library/security/ms17-010.aspx">here</a>.</li><li>Deploy an SMB backdoor</li><li>NotPetya infection</li></ol>
<h3>Test for Vulnerable Condition</h3>
<p>As can be seen in the control flow graph, core_MS17_010 is initially called to determine if the exploit condition exists. If it does, core_MS17_010 is called again with slightly different arguments to exploit the vulnerability and send the shellcode. <img class="alignnone size-full wp-image-19231" src="/content/dam/crowdstrike/www/en-us/wp/2017/07/pertrwrap-tech-blog-II-image-1a.png" alt="pertrwrap" width="975" height="421"/> The following is an image and explanation of the SMB network traffic used to determine if the victim is vulnerable (Victim: 172.16.1.132):<img class="alignnone size-full wp-image-19230" src="/content/dam/crowdstrike/www/en-us/wp/2017/07/pertrwrap-tech-blog-II-image-2.png" alt="pertrwrap" width="975" height="221"/></p>
<ol><li>An initial SMB_COM_NEGOTIATE request and response</li><li>Session setup</li><li>Tree connect</li><li>PeekNamedPipe SMB transaction with FID: 0x0000<ol><li>Exploitable condition:<ol><li>Trans response, error: STATUS_INSUFF_SERVER_RESOURCES</li><li>Error code: 0xC0000205</li><li>Server is out of resources</li></ol>
</li></ol>
</li><li>Close session: Tree disconnect and AndX logoff</li></ol>
<p>A check is then done to determine if the victim’s response is STATUS_INSUFF_SERVER_RESOURCES (0xC0000205). An example of this is shown below: <img class="alignnone size-full wp-image-19229" src="/content/dam/crowdstrike/www/en-us/wp/2017/07/pertrwrap-tech-blog-II-image-3.png" alt="pertrwrap" width="975" height="292"/> If the error code does not exist, the session is cleaned up and NotPetya iterates to the next IP address. If the error code exists, the core_MS17_010 returns 0, and the vulnerability is exploited.</p>
<h3>Check Windows Version</h3>
<p>Prior to executing the exploit code, a test ensues to determine which version of Microsoft Windows is running on the system. This is done by parsing the “Session Setup AndX Response” packet. Once the version of the OS is identified it is mapped to a numeric value:</p>
<ul><li>2 - Windows XP</li><li>3 - Windows XP Pro x64, Windows 2003, Windows 2003 R2</li><li>4 - Windows Vista</li><li>5 - Windows 7</li><li>6 - Windows 2008</li><li>7 - Windows 2008 R2</li></ul>
<p>NotPetya will only exploit one of the Windows versions listed above. If the version does not match one of the identified Windows versions, then the function to execute the exploit code returns without executing. The EternalBlue exploit is launched against only the following Windows versions:</p>
<ul><li>Windows 7</li><li>Windows 2008</li><li>Windows 2008 R2</li></ul>
<p>The EternalRomance exploit is launched against only the following Windows versions:</p>
<ul><li>Windows XP</li><li>XP Pro x64</li><li>Windows Server 2003</li><li>Windows Server 2003 R2</li><li>Vista</li></ul>
<p>The EternalRomance code path begins at 0x10005C4C.</p>
<h3>Trigger MS17-010 (EternalBlue)</h3>
<p>The following image and explanation refers to the SMB network traffic that starts the exploitation process:<img class="alignnone size-full wp-image-19228" src="/content/dam/crowdstrike/www/en-us/wp/2017/07/pertrwrap-tech-blog-II-image-4.png" alt="pertrwrap" width="975" height="248"/></p>
<ol><li>An initial SMB_COM_NEGOTIATE request and response</li><li>Session setup</li><li>Tree connect</li><li>PeekNamedPipe SMB transaction with FID: 0x0000<ol><li>Exploitable condition:<ol><li>Trans response, error: STATUS_INSUFF_SERVER_RESOURCES</li><li>Error code: 0xC0000205</li><li>Server is out of resources</li></ol>
</li></ol>
</li><li>NT trans request<ol><li>Followed by Trans2 secondary request, FID: 0x0000</li></ol>
</li></ol>
<p>This exploitation process is followed by a series of additional SMB packets. There are three types of packets that are identified based on size:</p>
<ul><li>TYPE 0:<ul><li>TCP segment data size: 132 bytes</li><li>The first 10 bytes of the TCP segment data contain:<ul><li>0x0 - Null byte</li><li>0x2 - 0xFFF7</li><li>0x4 - 2 byte value based on a timing check</li><li>0x6 - 2 byte value based on another timing check</li></ul>
</li><li>The rest of the packet contains null bytes</li></ul>
</li><li>TYPE 1:<ul><li>TCP segment data size: 2920 bytes</li><li>This packet sends the kernel shellcode</li><li>This packet is sent with the same data 13 times</li></ul>
</li></ul>
<p>This first burst of requests are TYPE 0 packets. To trigger the exploit, a 4207-byte &#34;SMB Trans2 Secondary Request, FID: 0x0000&#34; packet is sent to the vulnerable machine, as depicted below. This packet is built and sent in function sub_10003C0A. <img class="alignnone size-full wp-image-19227" src="/content/dam/crowdstrike/www/en-us/wp/2017/07/pertrwrap-tech-blog-II-image-5.png" alt="pertrwrap" width="975" height="79"/> The exploit code is contained within the 4096 byte &#34;Extra byte parameters&#34; section of the Trans2 packet. This 4096 buffer is filled with hex byte &#39;\x54&#39; (&#39;T&#39;). At offset 0xB within that buffer, the hex byte &#39;\x51&#39; is used to specify the start of the data that will trigger the overflow on the vulnerable machine. The specific code to trigger the overflow is the 175-byte chunk of binary data within the NotPetya binary, starting at address 0x10010B08. The code is not encoded.</p>
<h4>Sending the Ring 0 Shellcode</h4>
<p>Once the vulnerability has been exploited, the Ring 0 shellcode is sent using TYPE 1 packets. The ring 0 shellcode, which contains the SMB backdoor code, is the 2423 byte chunk for binary data starting at address 0x1000123B0 within the NotPetya binary. The ring 0 shellcode is encoded with a simple byte xor. The xor key is 0xCC. The shellcode starts at offset 0x1F1 within the TCP segment data.</p>
<h4> </h4>
<h3>Deploy SMB Backdoor</h3>
<p>The backdoor that gets deployed onto an exploited system appears to be a modified version (based on network traffic) of the DoublePulsar rootkit leaked by the Shadow Brokers actors.</p>
<h3>Network Traffic</h3>
<p>If there is a DoublePulsar knock feature, it is not used. Once the SMB backdoor is in place, the ‘Reserved’ field of the SMB Header for the response packet is set to 0x1100, as seen below. Normally this field is NULL. <img class="alignnone size-full wp-image-19226" src="/content/dam/crowdstrike/www/en-us/wp/2017/07/pertrwrap-tech-blog-II-image-6.png" alt="pertrwrap" width="380" height="362"/> Prior to triggering the code to exploit the system, the “Reserved” field in the SMB header is checked to see if it is set to 0x11. <img class="alignnone size-full wp-image-19225" src="/content/dam/crowdstrike/www/en-us/wp/2017/07/pertrwrap-tech-blog-II-image-7.png" alt="pertrwrap" width="975" height="273"/> If the “Reserved” field in the SMB header is set to 0x11, the staging code is built and sent to the targeted machine. <img class="alignnone size-full wp-image-19224" src="/content/dam/crowdstrike/www/en-us/wp/2017/07/pertrwrap-tech-blog-II-image-8.png" alt="" width="975" height="117"/> “Trans2 Request, SESSION_SETUP” packets are used to send the staging DLL to the target.</p>
<h3>NotPetya Infection</h3>
<p>To deploy NotPetya onto the system, a staging DLL is injected into “lsass.exe.” Once the DLL is injected into lsass, NotPetya is written to the system in c:\windows: <img class="alignnone size-full wp-image-19223" src="/content/dam/crowdstrike/www/en-us/wp/2017/07/pertrwrap-tech-blog-II-image-9.png" alt="pertrwrap" width="838" height="139"/> The NotPetya DLL keeps the same filename the file was called when executed on the attacking machine (in this case, olga.dll). Once the file is written, it can be launched. The staging DLL executes the file in the same way as psexec and wmic: rundll32.exe. As can be seen below, the same arguments are passed to the NotPetya DLL.<img class="alignnone size-full wp-image-19222" src="/content/dam/crowdstrike/www/en-us/wp/2017/07/pertrwrap-tech-blog-II-image-10.png" alt="pertrwrap" width="947" height="666"/></p>
<h2><b>MBR Restoration for Machines running Kaspersky Antivirus</b></h2>
<p>Through analysis, it was discovered that if the victim machine has avp.exe (associated with Kaspersky antivirus) process running, NotPetya will NOT encrypt the MFT. Victim machines that have avp.exe running when impacted by NotPetya will simply have the first 10 sectors of the physical disk overwritten with uninitialized data. For victim machines that do not have avp.exe running, NotPetya will overwrite the MBR with a custom boot loader, which will then load 16-bit code responsible for encrypting the MFT. The following displays “before and after” images of the MBR on a machine with avp.exe running. Notice the lack of the footer (0x55, 0xAA) in the MBR after the modification.<img class="alignnone size-full wp-image-19221" src="/content/dam/crowdstrike/www/en-us/wp/2017/07/pertrwrap-tech-blog-II-image-11.png" alt="pertrwrap" width="1037" height="584"/></p>
<p>Additionally, analysis also shows that the NotPetya malware overwrites the 2nd sector of the C:\ volume with uninitialized data as well. This sector contains the Boot Manager, which is loaded by the VBR (Volume Boot Record) and is needed for booting up the system. Overwriting this sector renders the machine unbootable. It should be noted that this change occurs regardless of whether the victim machine has avp.exe process running on the system or not. The following are the before and after images of the 2nd sector: <img class="alignnone size-full wp-image-19220" src="/content/dam/crowdstrike/www/en-us/wp/2017/07/pertrwrap-tech-blog-II-image-12.png" alt="pertrwrap" width="1026" height="577"/> If the avp.exe process exists on the system, upon reboot, the victim is prompted with an error message that an operating system cannot be found:<img class="alignnone size-full wp-image-19219" src="/content/dam/crowdstrike/www/en-us/wp/2017/07/pertrwrap-tech-blog-II-image-13.png" alt="pertrwrap" width="774" height="253"/></p>
<h2>Steps to Recover MBR</h2>
<p>Analysis has confirmed that in the case of the presence of avp.exe process, the MBR is actually recoverable via the Windows repair tools. For Windows 7 and higher, the following steps can be taken to recover the MBR:</p>
<ul><li>Boot from a Windows installation disk</li><li>Upon system reboot, click Next → Repair Your Computer</li></ul>
<p><img class="alignnone size-full wp-image-19218" src="/content/dam/crowdstrike/www/en-us/wp/2017/07/pertrwrap-tech-blog-II-image-14.png" alt="pertrwrap" width="709" height="526"/></p>
<ul><li>Under System Recovery Options, click on “Use Recovery Tools”</li><li>Click “Startup Repair”</li><li>The MBR should now be repaired. Users can confirm this by clicking on “Click here for diagnostic and repair details” and scrolling down to the MBR section</li></ul>
<p><img class="alignnone size-full wp-image-19217" src="/content/dam/crowdstrike/www/en-us/wp/2017/07/pertrwrap-tech-blog-II-image-15.png" alt="pertrwrap" width="573" height="307"/></p>
<ul><li>Click Close → Finish. This will reboot the machine</li><li>Some users might receive the following error message upon reboot. This error is due to the boot manager corruption</li></ul>
<p><img class="alignnone size-full wp-image-19216" src="/content/dam/crowdstrike/www/en-us/wp/2017/07/pertrwrap-tech-blog-II-image-16.png" alt="pertrwrap" width="621" height="325"/></p>
<ul><li>To rectify this issue, change the boot order to boot from the Windows disk image. This can be achieved by going into the BIOS settings</li><li>Upon system reboot, follow the first 2 steps and the user should then see the OS under System Recovery Options</li><li>Click on the OS, choose the “Use recovery tools” option, and click Next</li><li>To repair the boot manager, click on “Command Prompt” and enter the following command:</li></ul>
<p style="	text-align: left;
"><b>bootrec /fixboot</b></p>
<p><img class="alignnone size-full wp-image-19215" src="/content/dam/crowdstrike/www/en-us/wp/2017/07/pertrwrap-tech-blog-II-image-17.png" alt="pertrwrap" width="731" height="362"/></p>
<ul><li>Upon successful execution of the command, close the command prompt and click “Restart”</li><li>Upon repairing the MBR and the boot manager, the user can boot the system without the Windows installation disk and access the system</li></ul>
<p>The above steps will allow the user to access the system again, but certain files (that match the target extension list) on the system will still be encrypted and can only be decrypted with a private key possessed by the NotPety authors. Analysis also shows that the malware only encrypts the first 1MB of file contents, if the size of the file is greater than 1MB. The remaining contents are left untouched and intact. <img class="alignnone size-full wp-image-19214" src="/content/dam/crowdstrike/www/en-us/wp/2017/07/pertrwrap-tech-blog-II-image-18.png" alt="pertrwrap" width="975" height="358"/> As the above image shows, line 20 compares the file size with 0x100000 (1,048,576 bytes or 1MB). If the file size is greater than 1MB, line 26 declares variables FileSize_1 and FinalFileSize to be 0x100000. These values are then used in CreateFileMappingW, MapViewOfFile and CryptEncrypt to encode the first 1MB of the file contents. The rest of the file remains untouched.</p>
