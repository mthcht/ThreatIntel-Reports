        <h1>Protecting the Software Supply Chain: Deep Insights into the CCleaner Backdoor</h1>
        
        <div class="publish_info">
            <p>October 04, 2017</p>

<div class="cmp-text" id="text-c25987656a" data-cmp-name="cmp-text-" data-cmp-is="text" data-target-location="false">
    
       <span>The term “supply chain attacks” means different things to different people. To the general business community, it refers to attacks targeting vulnerable third-parties in a larger organization’s supply chain. A well-known retail chain’s massive breach in 2013 is a classic example: Adversaries used a poorly protected HVAC vendor as their gateway to hack into the giant retailer’s <p> </p>enterprise network. However, threat researchers have another definition: To them, supply chain attacks can also denote the growing phenomenon in which malicious code is injected into new releases and updates of legitimate software packages, effectively turning an organization’s own software supply infrastructure into a potent and hard-to-prevent attack vector. The recent backdoor that was discovered embedded in the legitimate, signed version of CCleaner 5.33, is just such an attack.

To help inform the user community and empower them to better defend against software supply chain attacks, the CrowdStrike® Security Response Team (SRT) conducted a thorough analysis of the CCleaner backdoor. A popular PC optimization tool, the 5.33 version of CCleaner has had widespread distribution across multiple industries, but the embedded code appeared to actually be targeted at specific groups in the technology sector. (More information on targeted industries is available for CrowdStrike customers in our Falcon Intelligence™ portal.) <a href="/en-us/endpoint-security-products/falcon-x-threat-intelligence/">CrowdStrike&#39;s threat intelligence team</a> had also previously reported on the malware’s C2 (command and control) infrastructure in a recent alert for CrowdStrike customers identifying possible links to Aurora Panda. The report also outlines the potential for additional adversary tactics, techniques and procedures (TTPs).
<h2>Technical Analysis</h2>
<h3><strong>CCleaner</strong></h3>
CCleaner is a PC cleaning utility developed by Piriform, which was recently acquired by antivirus (AV) provider Avast in June 2017. The affected version of the utility contains a modified __scrt_common_main_seh function that routes the execution flow to a custom function meant to decode and load the malware. This takes place even before the entry point (EP) of the utility is reached.The new execution flow leads to a function that decodes a blob of data, as reproduced in Python below:

<code><p> </p><p> </p><p> </p>dec &#61; &lt;&gt;</code>

<code><p> </p><p> </p><p> </p>for i in range(0, len(indata)):</code>

<code><p> </p><p> </p><p> </p><p> </p><p> </p><p> </p><p> </p>key &#61; ((key * 0x47a6547) &amp; 0xFFFFFFFF) &amp; 0xFF</code>

Next, the program creates a memory heap with the flag HEAP_CREATE_ENABLE_EXECUTE to allow for execution, and copies the shellcode on the heap, and executes it.
<h3><strong>ShellCode</strong></h3>
The shellcode is responsible for loading the payload in memory. It attains the PEB (Process Environment Block) of the malware process to load kernel32.dll and find the location of the function <strong>GetProcAddress</strong>. This function is used to retrieve the addresses of functions such as <strong>VirtualAlloc</strong>, <strong>memcpy</strong>, and <strong>LoadLibrary</strong>. Once the payload is copied to the newly allocated memory, the shellcode resolves the needed API&#39;s, and calls the OEP (original entry point) of the payload in memory.
<h3><strong>Payload</strong></h3>
<h3><strong>Environment Checks</strong></h3>
Once it’s loaded, the payload creates a thread that performs the core functionality of the malware. It performs a few checks at the onset of the environment and the user privileges. The malware employs the function <strong>msvcrt.time</strong> to record the current time of the malware. It then uses <strong>IcmpCreateFile</strong> and <strong>IcmpSendEcho </strong>to send an IPv4 ICMP echo to an invalid IP address, with a timeout of 601 seconds. This is meant to delay the execution of the malware by 601 seconds; this delay is then measured by calling <strong>msvcrt.time</strong> again, and ensuring that more than 600 seconds have elapsed between the first and second calls to the function. It should be noted that if the call to <strong>IcmpCreateFile</strong> fails, the malware will just sleep for 600 seconds.These steps are measures against debugging and/or sandboxing. It also invokes <strong>IsUserAnAdmin </strong>to ensure that the current user is member of the administrator&#39;s group. If either of these checks fails, the malware exits immediately.

It uses a decoding scheme as the one described above to decode strings during runtime in memory. It is important to note that these dynamically decoded strings are zeroed out in memory before each function using them exits. The strings dynamically decoded throughout the execution of the malware are listed in the Appendix section of this blog.

The malware also checks the privilege levels of its own process; if the process does not have administrative privileges, it uses <strong>AdjustTokenPrivileges</strong> to enable the <strong>SeDebugPrivilege</strong> value for the process. This enables the process to either debug or adjust memory for a process owned by another account.
<h3><strong>Registry Checks</strong></h3>
The malware checks for the following registry key:<strong> HKLM\SOFTWARE\Piriform\Agomo\TCID</strong>. The key value is supposed to hold a system time value; if the value is greater than the current time, the malware will terminate. It also checks the value of <strong>HKLM\SOFTWARE\Piriform\Agomo\MUID</strong>. If the key does not exist, the malware will set its value using a pseudo-random number derived in the following manner:
<div class="blog-post-code-wrapper decrypt"><code>// Pseudocode to calculate MUID </code>val
<code><strong>DWORD</strong> MUID;</code>
<code>unsigned <strong>int</strong> seed, rand1, rand2;</code>
<code>seed &#61; GetTickCount();</code>
<code> <strong>srand</strong>(seed);</code>
<code>rand1 &#61; <strong>rand</strong>();</code>
<code>rand2 &#61; <strong>rand</strong>() * rand1;</code>
<code>MUID &#61; GetTickCount() ^ rand2;</code></div>
<h3><strong>Gathering Victim Information</strong></h3>
Once the checks are completed, the malware gathers the following information about the victim machine:
<ul><li>OS major version</li><li>OS minor version</li><li>OS architecture</li><li>Computer name</li><li>Computer DNS domain</li><li>IPv4 addresses associated with the machine. This information is gathered by calling <strong>GetAdaptersInfo</strong>, and then enumerating through each adapter to search for the IP_ADAPTER_INFO<em> → </em>IpAddressList<em> → </em>IpAddress field.</li><li>Installed applications. The malware accesses the registry key HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\Uninstall, and enumerates through each key, and compares the <em>Publisher</em> value with &#34;Microsoft Corporation.&#34; If there is a match, it moves on to the next value. If not, it will attain the DisplayName value using <strong>SHGetValueA</strong>, and insert it into memory. Each name is prepended with an “S” in memory.</li><li>Full name of the executable image of each running process. The malware calls <strong>WTSEnumerateProcessA</strong> to get a pointer to an array of WTS_PROCESS_INFO structures, which are then used to get the <em>ProcessName</em> field for each process. Each process name is prepended with a “P” in memory.</li></ul>
This information is stored in a data structure in memory in the following manner:

<img class="alignnone wp-image-19776 size-full" src="/content/dam/crowdstrike/www/en-us/wp/2017/10/dataStructureMem2.png" alt="" width="975" height="226"/>

The MUID_Val is used as a unique identifier for the victim machine.

Next, the structure is encoded in memory in two steps:
<ul><li>Step 1: Aforementioned scheme</li><li>Step 2: Modified version of base64</li></ul>
The image below displays the data structure as it goes through each encoding step.

<img class="alignnone wp-image-19776 size-full" src="/content/dam/crowdstrike/www/en-us/wp/2017/10/data-structure.png" alt="" width="975" height="226"/>

The custom base64 encoding scheme uses a modified Base64 index table. Rather than the regular table that has the following values: <b>ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz0123456789&#43;/</b>; its table has the following values: <b>abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ0123456789!*</b>.
<h3><b>C2 Communication</b></h3>
Once the victim machine information has been encoded, the malware queries the registry key <b>HKLM\SOFTWARE\Piriform\Agomo\NID</b>. Upon the initial run, the registry key does not exist; however, the malware eventually inserts an IP address computed via a DGA (Domain Generating Algorithm) later in the execution flow. It is interesting to note that even if the registry key exists, the malware extracts the IP address from the registry value, but does not do anything with it. After the registry check, it decodes the hard-coded IP address <b>216.126.225&lt;.&gt;148</b>, and attempts to send the encoded data struct to it via an HTTP POST request on port 443. It uses <b>InternetSetOptionA</b> to set the following option flags on the HTTP handle:
<ul><li>SECURITY_FLAG_IGNORE_CERT_DATE_INVALID → Ignores bad or expired SSL certificates from the server</li><li>SECURITY_FLAG_IGNORE_CERT_CN_INVALID → Ignores incorrect SSL certificate common names</li><li>SECURITY_FLAG_IGNORE_WRONG_USAGE → Ignores incorrect usage problems</li><li>SECURITY_FLAG_IGNORE_UNKNOWN_CA → Ignores unknown certificate authority problems</li><li>SECURITY_FLAG_IGNORE_REVOCATION → Ignores certificate revocation problems</li></ul>
The malware also calls <b>HttpAddRequestHeadersA</b> to append the domain <b>speccy.piriform&lt;.&gt;com</b> to the POST request. This is performed to appear inconspicuous and make it harder to detect. It is also likely an attempt to confuse the analyst performing dynamic analysis of the malware. Once the information is sent to the C2, the malware expects to receive a stage 2, which it reads into a locally allocated memory block. Analysis shows that once stage 2 is received, it is decoded using the same custom Base64 and the decoding algorithm. Once decoded, the functions <b>GetProcAddress</b> and <b>LoadLibraryA</b> are pushed to the stack, and the EP of stage 2 is called. At the time of analysis, stage 2 was not available.
<h4><b>DGA</b></h4>
If the malware cannot connect to the C2, it employs a Domain Generating Algorithm, or DGA, to generate a domain. The DGA is dependent on the current year and month; therefore, it generates a new domain on a monthly basis. Below is the code, reproduced in C, displaying the DGA utilized by the malware.

<code>#include &lt;stdio.h&gt;</code>
<code>void main()</code>
<code>{<!-- --></code>
<code>    <span class="indent once">SYSTEMTIME st;</span></code>
<code>    <span class="indent once">DWORD r1, r2, r3, seed;</span></code>
<code>    <span class="indent once">char buf&lt;100&gt;;</span></code>
<code>    <span class="indent once">const char *format &#61; &#34;ab%x%x.com&#34;;</span></code>
<code>    <span class="indent once">GetLocalTime(&amp;st);</span></code>
<code>    <span class="indent once">seed &#61; st.wYear * 10000 &#43; st.wMonth;</span></code>

<code>    <span class="indent once">r2 &#61; rand();</span></code>
<code>    <span class="indent once">r3 &#61; rand() * r2;</span></code>

The list of domains calculated for all months in the years 2017 and 2018 are listed in the Appendix.

Once the DGA domain for the current month and year has been calculated, the malware calculates an IP address using that domain in the following steps:
<ul><li>Get a <i>hostent</i> structure by calling <b>gethostbyname</b> on the generated domain</li><li>Get the <i>h_addr_list</i>, which is a NULL terminated list of IP addresses associated with the domain</li></ul>
These A records (127.100.183&lt;.&gt;225 and 10.158.168&lt;.&gt;171) for the domain ab1145b758c30&lt;.&gt;com, as highlighted in the PCAP screenshot below will be used to calculate a new C2 IP address. If there are more than two A records, the malware will only utilize the first two on the list.

<img class="alignnone wp-image-19776 size-full" src="/content/dam/crowdstrike/www/en-us/wp/2017/10/c2-ip.png" alt="" width="975" height="226"/>

<p> </p>

The Python code below reproduces the algorithm to calculate the new C2 IP address from the A records of the newly generated domain.

<p> </p>
<div class="blog-post-code-wrapper decrypt"><code>import struct</code>
<code>import socket</code>
<code>a1 &#61; 0xE1B7647F <p> </p># Addresses are returned in network byte order</code>
<code>a2 &#61; 0xABA89E0A</code><code>def mod_record(rr):</code>
<code><span class="indent once"> <p> </p><p> </p><p> </p>rr1 &#61; (((rr &amp; 0xff000000) / 0x1000000) ^ (rr &amp; 0xff)) * 0x1000000</span></code>
<code><span class="indent once"> <p> </p><p> </p><p> </p>rr2 &#61; (((rr &amp; 0xff0000) / 0x10000) ^ ((rr &amp; 0xff00) / 0x100)) * 0x10000</span></code>
<code><span class="indent once"> <p> </p><p> </p><p> </p>rr3 &#61; rr &amp; 0xff00</span></code>
<code><span class="indent once"> <p> </p><p> </p><p> </p>rr4 &#61; rr &amp; 0xff</span></code>
<code><span class="indent once"> <p> </p><p> </p><p> </p>return (rr1 | rr2 | rr3 | rr4)</span></code>
<code>newa1 &#61; mod_record(a1)</code>
<code>newa2 &#61; mod_record(a2)</code><code>newIP &#61; <p> </p>(newa2 &amp; 0xffff0000) | (newa1 &gt;&gt; 0x10)  # newIP &#61; 0xA1369ED3</code><code>print socket.inet_ntoa(struct.pack(&#34;&lt;L&#34;, newIP)) <p> </p># Output is 211.158.54.161</code>

</div>
The new C2 IP address derived from the records of the domain ab1145b758c30&lt;.&gt;com is 211.158.54&lt;.&gt;161. The malware will attempt to connect to this C2 as shown below. If the connection is successful, it will subsequently send the encoded data structure and await stage 2.

<img class="alignnone wp-image-19776 size-full" src="/content/dam/crowdstrike/www/en-us/wp/2017/10/new-c2-ip.png" alt="" width="975" height="226"/>
<h3><b>Initial (Buggy) Registry Modifications </b></h3>
Once the C2 communication subroutine has ended, the malware makes two registry modifications:
<ul><li>Encodes the newly calculated C2 IP address and attempts to save it in <b>HKLM\SOFTWARE\Piriform\Agomo\NID</b>. The encoding scheme is the same as the one mentioned before. Analysis shows that before the registry key string is built, a function is called to change the endianness of <p> </p>0x44494E (DIN) to 0x4E4944 (NID). However, due to a bug in the code the function incorrectly changes it to 0x004E4944 (prepended with a NULL value). Subsequently, function <b>SHSetValueA</b> is called with the following parameters:
<ul><li><b>hKey</b> &#61; HKEY_LOCAL_MACHINE</li><li><b>Subkey</b> &#61; &#34;SOFTWARE\Piriform\Agomo&#34;</li><li><b>Value</b> &#61; &#34;&#34;</li><li><b>ValueType</b> &#61; REG_DWORD</li><li><b>Data</b> &#61; ...</li><li><b>DataLength</b> &#61; 0x4</li></ul>
</li></ul>
The parameter <b>Value</b> should be &#34;NID&#34;; however, since the string is incorrectly prepended with a NULL value, the function doesn&#39;t read the string at all. The C2 IP address is instead saved in<b> HKLM\SOFTWARE\Piriform\Agomo\Default </b>as shown below.

<img class="alignnone wp-image-19776 size-full" src="/content/dam/crowdstrike/www/en-us/wp/2017/10/c2-ip-save-location.png" alt="" width="975" height="226"/>
<ul><li>Takes the current time value as determined by the earlier call to <b>msvcrt.time</b> and adds 172,800 seconds (2 days) to the value. Saves the new value in HKLM\SOFTWARE\Piriform\Agomo\TCID.</li></ul>
<h3>Recommendations</h3>
Falcon Endpoint will notify you of any additional activity through our Falcon Intelligence detections. The intent behind the malicious packages was to collect an initial set of reconnaissance data; we urge you to block the known IP address and domains at your network perimeter to prevent any communication to the collection server. In addition, we recommend you update to the latest version of the Avast CCleaner software to ensure the embedded malicious code is removed.

For additional information on CrowdStrike’s threat intelligence offerings, visit the <a href="/en-us/endpoint-security-products/falcon-x-threat-intelligence/">Falcon Intelligence product page</a>.
