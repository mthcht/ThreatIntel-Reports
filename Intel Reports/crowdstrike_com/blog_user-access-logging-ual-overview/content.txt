        <h1>UAL Thank Us Later: Leveraging User Access Logging for Forensic Investigations</h1>
        
        <div class="publish_info">
            <p>June 08, 2021</p>
            <span>|</span>
            <a href="/en-us/blog/author.patrick-bennett/" title="Posts by Patrick Bennett" rel="author">
                Patrick Bennett</a>
            <span>|</span>
                
            <a href="/en-us/blog/category.endpoint-protection/" title="Endpoint Security &amp; XDR">Endpoint Security &amp; XDR</a>
        </div>
        <div class="post_image">
            <img width="1060" height="698" src="https://assets.crowdstrike.com/is/image/crowdstrikeinc/Blog_1060x698-3-1" class="attachment-post-thumbnail size-post-thumbnail wp-post-image" alt="" decoding="async" fetchpriority="high" srcset="/content/dam/crowdstrike/www/en-us/wp/2021/06/Blog_1060x698-3-1.jpeg 1060w,https://assets.crowdstrike.com/is/image/crowdstrikeinc/Blog_1060x698-3-1-768x506 768w,https://assets.crowdstrike.com/is/image/crowdstrikeinc/Blog_1060x698-3-1-1024x674 1024w,https://assets.crowdstrike.com/is/image/crowdstrikeinc/Blog_1060x698-3-1-300x198 300w" sizes="(max-width: 1060px) 100vw, 1060px"/>
        </div>
    

    
</div>

    <div style="display: none;">
        
    </div>


    
</div>
<div class="text text--blog-content aem-GridColumn aem-GridColumn--default--12">

  

<div class="cmp-text" id="text-4834189368" data-cmp-name="cmp-text-" data-cmp-is="text" data-target-location="false">
    
       <span><p><span style="	font-weight: 400;
">CrowdStrike analysts recently began researching and leveraging User Access Logging (UAL), a newer forensic artifact on Windows Server operating system that offers a wealth of data to support forensic investigations. UAL has proven beneficial to help correlate an account and the source IP address with actions performed remotely on systems.<br />
</span> <span style="	font-weight: 400;
">To help your investigations, this blog post provides an overview of UAL databases and offers examples of interpreting the treasure trove of data that they contain.</span></p>
<h2><span style="	font-weight: 400;
">What Is User Access Logging?</span></h2>
<p><span style="	font-weight: 400;
">UAL is a feature included by default in Server editions of Microsoft Windows, starting with Server 2012. As defined by </span><a href="https://docs.microsoft.com/en-us/windows-server/security/windows-services/security-guidelines-for-disabling-system-services-in-windows-server"><span style="	font-weight: 400;
">Microsoft</span></a><span style="	font-weight: 400;
">, UAL is a feature that “logs unique client access requests, in the form of IP addresses and user names, of installed products and roles on the local server.”</span> <span style="	font-weight: 400;
">This means that UAL records user access to various services running on a Windows Server. The access is logged to databases on disk that contain information on the type of service accessed, the user account that performed the access and the source IP address from which the access occurred. One key element of UAL is that each record is based on the combination of username, source IP and service accessed — so it’s naturally suited for identifying anomalous or rare access to a system.</span> <span style="	font-weight: 400;
">With default settings, this information is retained for up to three years. Naturally, this data can be extremely valuable in forensic investigations. Unfortunately, there’s a marked lack of awareness of this type of artifact in the digital forensic community. Many forensic solutions do not parse these databases, and therefore threat analysts could potentially miss data relevant to an investigation.</span></p>
<h2><span style="	font-weight: 400;
">Where to Find UAL Data</span></h2>
<p><span style="	font-weight: 400;
">UAL database files are stored under the directory C:\Windows\System32\LogFiles\Sum. Inside this directory, you’ll find up to five Extensible Storage Engine (ESE) database files with .mdb extensions. The screenshot in Figure 1 provides an example of what the contents might look like.</span></p>
<p> </p>
<div class="wp-caption" id="attachment_30526"><img class="size-full wp-image-30526" src="/content/dam/crowdstrike/www/en-us/wp/2021/06/Picture1-4.png" alt="" width="977" height="210"/>Figure 1. C:\Windows\System32\LogFiles\Sum sample contents</div>
<p><span style="	font-weight: 400;
">The files shown above include:</span></p>
<ul><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Current.mdb (UAL database — current year; active copy)</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">&lt;GUID&gt;.mdb (UAL database — current year)</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">&lt;GUID&gt;.mdb (UAL database — previous year)</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">&lt;GUID&gt;.mdb (UAL database — two years prior)</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Systemidentity.mdb (database containing information about the server, including a map of RoleGuid values to Role names – more on this below)</span></li></ul>
<p><span style="	font-weight: 400;
">The Current.mdb file contains UAL data for the current year, while the two previous years are stored in .mdb files with GUID-style filenames. Per Microsoft:<p> </p>
</span> <i><span style="	font-weight: 400;
">“UAL makes a copy of the active database file, current.mdb, to a file named GUID.mdb every 24 hours. On the first day of the year, UAL will create a new GUID.mdb. The old GUID.mdb is retained as an archive. After two years, the original GUID.mdb will be overwritten.”</span></i> <span style="	font-weight: 400;
">This means there can be up to three years of historical data stored on the UAL (i.e., data from the previous year, two years prior and the current year up to the present).<p> </p>
</span> <span style="	font-weight: 400;
">Following the above, Current.mdb and the GUID-style files contain the same set of tables. These files will include the CLIENTS table, where some of the juiciest forensic data is stored — this is where you’ll find the historical records of users accessing various services.</span> <span style="	font-weight: 400;
">Table 1 shows a sample record from the CLIENTS table. Please note that some of the fields are omitted for visibility (see Appendix for a full listing of all tables and fields in the UAL databases).</span></p>
<table class="orange"><tbody><tr class="top-row bg-orange"><td><b>RoleGuid</b></td><td><b>TotalAccesses</b></td><td><b>InsertDate</b></td><td><b>LastAccess</b></td><td><b>Address</b></td><td><b>AuthenticatedUserName</b></td></tr><tr><td>10a9226f-50ee-49d8-a393-9a501d47ce04</td><td>1</td><td>2019-03-12T18:06:56Z</td><td>2019-03-12T18:06:56Z</td><td>0a0a0cc8</td><td>DOMAIN\User1</td></tr></tbody></table>
<p><i><span style="	font-weight: 400;
">Table 1. Sample UAL CLIENTS table record</span></i> </p>
<p><span style="	font-weight: 400;
">In the above example, the UAL record indicates that the user DOMAIN\User1 accessed the system via SMB on 2019-03-12 at 18:06:56 UTC, coming from the source IP address 10.10.12.200.<p> </p>
</span> <span style="	font-weight: 400;
">The source IP address is stored in the </span><b>Address</b> <span style="	font-weight: 400;
">field in hexadecimal (0a 0a 0c c8 &#61; 10 10 12 200). The </span><b>InsertDate</b><span style="	font-weight: 400;
"> field contains the UTC timestamp of the first access for the year for the combination of user, RoleGuid and source IP. </span><b>LastAccess</b><span style="	font-weight: 400;
"> is similar but represents the most recent access for the year.</span> <span style="	font-weight: 400;
">The </span><b>TotalAccesses</b><span style="	font-weight: 400;
"> value of 1 indicates that this was the only access for the year (again, based on the combination of user, source IP and RoleGuid). If access occurred on additional days between the </span><b>InsertDate</b> <span style="	font-weight: 400;
">and </span><b>LastAccess</b><span style="	font-weight: 400;
">, the total count would be included in this field. In addition, a daily count of the number of accesses per day would be included in additional fields named </span><b>Day1</b><span style="	font-weight: 400;
"> up to </span><b>Day366</b><span style="	font-weight: 400;
">, which represent the day of the year the access occurred (see Appendix for more details). Unfortunately, a full timestamp is only included for </span><b>InsertDate</b> <span style="	font-weight: 400;
">and </span><b>LastAccess</b> <span style="	font-weight: 400;
">— nothing in between. But as will be shown, this is plenty<p> </p>
</span> <span style="	font-weight: 400;
">The </span><b>RoleGuid</b> <span style="	font-weight: 400;
">field represents the type of service that was accessed. In this case, it was 10a9226f-50ee-49d8-a393-9a501d47ce04, which corresponds with what is known as the File Server Role. This typically represents SMB access, though it’s possible other protocols may be logged here as well.</span> <span style="	font-weight: 400;
">RoleGuid values are mapped to human-readable Role Names in the SystemIdentity.mdb database, under the ROLE_IDS table. Table 2 shows a sample ROLE_IDS table.</span></p>
<table class="orange"><tbody><tr class="top-row bg-orange"><td><b>RoleGuid</b></td><td><b>ProductName</b></td><td><b>RoleName</b></td></tr><tr><td>c50fcc83-bc8d-4df5-8a3d-89d7f80f074b</td><td>0997dbd9-4db4-49aa-8ec5-8f5c6ae1c870</td><td>Active Directory Certificate Services</td></tr><tr><td>b4cdd739-089c-417e-878d-855f90081be7</td><td>0997dbd9-4db4-49aa-8ec5-8f5c6ae1c870</td><td>Active Directory Rights Management Service</td></tr><tr><td>48eed6b2-9cdc-4358-b5a5-8dea3b2f3f6a</td><td>0997dbd9-4db4-49aa-8ec5-8f5c6ae1c870</td><td>DHCP Server</td></tr><tr><td>7cc4b071-292c-4732-97a1-cf9a7301195d</td><td>0997dbd9-4db4-49aa-8ec5-8f5c6ae1c870</td><td>FAX Server</td></tr><tr><td>10a9226f-50ee-49d8-a393-9a501d47ce04</td><td>0997dbd9-4db4-49aa-8ec5-8f5c6ae1c870</td><td>File Server</td></tr><tr><td>bbd85b29-9dcc-4fd9-865d-3846dcba75c7</td><td>0997dbd9-4db4-49aa-8ec5-8f5c6ae1c870</td><td>Network Policy and Access Services</td></tr><tr><td>7fb09bd3-7fe6-435e-8348-7d8aefb6cea3</td><td>0997dbd9-4db4-49aa-8ec5-8f5c6ae1c870</td><td>Print and Document Services</td></tr><tr><td>d6256cf7-98fb-4eb4-aa18-303f1da1f770</td><td>0997dbd9-4db4-49aa-8ec5-8f5c6ae1c870</td><td>Web Server</td></tr><tr><td>4116a14d-3840-4f42-a67f-f2f9ff46eb4c</td><td>0997dbd9-4db4-49aa-8ec5-8f5c6ae1c870</td><td>Windows Deployment Services</td></tr><tr><td>d8dc1c8e-ea13-49ce-9a68-c9dca8db8b33</td><td>0997dbd9-4db4-49aa-8ec5-8f5c6ae1c870</td><td>Windows Server Update Services</td></tr><tr><td>c23f1c6a-30a8-41b6-bbf7-f266563dfcd6</td><td>0997dbd9-4db4-49aa-8ec5-8f5c6ae1c870</td><td>FTP Server</td></tr><tr><td>910cbaf9-b612-4782-a21f-f7c75105434a</td><td>0997dbd9-4db4-49aa-8ec5-8f5c6ae1c870</td><td>BranchCache</td></tr><tr><td>952285d9-edb7-4b6b-9d85-0c09e3da0bbd</td><td>0997dbd9-4db4-49aa-8ec5-8f5c6ae1c870</td><td>Remote Access</td></tr></tbody></table>
<p><i><span style="	font-weight: 400;
">Table 2. Sample ROLE_IDS table</span></i></p>
<h3><span style="	font-weight: 400;
">A Note on Roles</span></h3>
<p><span style="	font-weight: 400;
">The Roles referenced by UAL data are tied directly to Server Roles installed on Windows Server systems. This is done via the Server Manager application, by clicking on Manage → Add Roles and Features.</span></p>
<div class="wp-caption" id="attachment_30527"><img class="size-full wp-image-30527" src="/content/dam/crowdstrike/www/en-us/wp/2021/06/Picture2-2.png" alt="" width="977" height="177"/>Figure 2. Server Manager Roles and Features menu</div>
<p><span style="	font-weight: 400;
">This will bring up a menu that lists available Roles that can be installed, which will look similar to what’s shown in Figure 3.</span></p>
<div class="wp-caption" id="attachment_30528"><img class="size-full wp-image-30528" src="/content/dam/crowdstrike/www/en-us/wp/2021/06/Picture3-1.png" alt="" width="977" height="604"/>Figure 3. Server Manager Server Roles menu</div>
<p><span style="	font-weight: 400;
">Certain Roles are included in the ROLE_IDS table by default, regardless of whether or not they are enabled. Other Roles may get added to the bottom of the ROLE_IDS table when they are installed via the Server Manager. For example, when making a server into a Domain Controller, one would install the </span><b>Active Directory Domain Services</b><span style="	font-weight: 400;
"> Role, at which point this server would be added to the bottom of the ROLE_IDS table, and access under this Role would start being logged in the CLIENTS table. However, not every installed Role will necessarily end up being tracked by UAL.</span></p>
<h3><span style="	font-weight: 400;
">File Server Role</span></h3>
<p><span style="	font-weight: 400;
">From a forensic perspective, one of the most fruitful Roles in UAL analysis is the </span><b>File Server</b><span style="	font-weight: 400;
"> Role. It can be found as a subitem under File and Storage Services in the Server Manager menu.</span></p>
<div class="wp-caption" id="attachment_30529"><img class="size-full wp-image-30529" src="/content/dam/crowdstrike/www/en-us/wp/2021/06/Picture4.png" alt="" width="977" height="600"/>Figure 4. File and Storage Services</div>
<p><span style="	font-weight: 400;
">As shown in Figure 4, Microsoft notes that the File Server Role “manages shared folders and enables users to access files on <a>computer from the network.” The consequence is that SMB access is logged in UAL databases under the </a></span><a><b>File Server</b><span style="	font-weight: 400;
"> RoleGuid. This means UAL databases potentially contain up to three years of historical SMB access. This data can be extremely valuable during investigations, as we’ll demonstrate in the next section.<p> </p>
</span> <span style="	font-weight: 400;
">It’s important to note that this SMB logging includes when, for example, a user maps a file share and performs actions that use SMB under the hood, including SMB named pipes. For example, remotely interacting with a service using sc.exe will result in File Server UAL entries on the target system, because an SMB named pipe (\\.\PIPE\svcctl) is used. Similarly, a UAL File Server entry for a user doesn’t necessarily mean that the user purposefully used SMB.</span> <span style="	font-weight: 400;
">As a side note, even if the File Server Role is not explicitly enabled, SMB access will still be logged by UAL (as long as the firewall rules to allow SMB access are enabled). When the File Server Role is installed, these firewall rules are automatically enabled.</span><h2><span style="	font-weight: 400;
">Interpreting UAL Data</span></h2>
<span style="	font-weight: 400;
">Let’s step through some quick examples to demonstrate just how powerful UAL analysis can be. Please note that the following data is simulated, but this information is very similar to what you’d see in real-world scenarios when analyzing UAL data.</span> <span style="	font-weight: 400;

">. In image analysis, UAL databases can be parsed with any tool that supports parsing ESE databases, such as esedbexport, which is part of Joachim Metz’s </span><a href="https://github.com/libyal/libesedb"><span style="	font-weight: 400;
">libesedb</span></a><span style="	font-weight: 400;
"> project.<p> </p>
</span> <span style="	font-weight: 400;
">At least two recently developed solutions are used for parsing UAL data from a forensic perspective: Eric Zimmerman’s </span><a href="https://ericzimmerman.github.io"><span style="	font-weight: 400;

"> and Brian Moran’s </span><a href="https://github.com/brimorlabs/KStrike"><span style="	font-weight: 400;
">KStrike</span></a><span style="	font-weight: 400;
">. These tools add value by automatically converting RoleGuids to Role Names and automatically parsing the </span><b>Address</b> <span style="	font-weight: 400;
