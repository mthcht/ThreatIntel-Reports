        <h1>Weeding out WannaMine v4.0: Analyzing and Remediating This Mineware Nightmare</h1>
        
        <div class="publish_info">
            <p>November 12, 2019</p>
            <span>|</span>
            <a href="/en-us/blog/author.collin-montenegro-and-mark-robinson/" title="Posts by Collin Montenegro and Mark Robinson" rel="author">
                Collin Montenegro and Mark Robinson</a>
            <span>|</span>
                
            <a href="/en-us/blog/category.from-the-front-lines/" title="From The Front Lines">From The Front Lines</a>
        </div>
        <div class="post_image">
            <img width="1060" height="698" src="https://assets.crowdstrike.com/is/image/crowdstrikeinc/Wannamine-blog-1" class="attachment-post-thumbnail size-post-thumbnail wp-post-image" alt="" decoding="async" fetchpriority="high" srcset="/content/dam/crowdstrike/www/en-us/wp/2019/11/Wannamine-blog-1.jpg 530w,https://assets.crowdstrike.com/is/image/crowdstrikeinc/Wannamine-blog-1-300x198 300w" sizes="(max-width: 1060px) 100vw, 1060px"/>
        </div>
    

    
</div>

    <div style="display: none;">
        
    </div>


    
</div>
<div class="text text--blog-content aem-GridColumn aem-GridColumn--default--12">

  

<div class="cmp-text" id="text-6dd2e31f8d" data-cmp-name="cmp-text-" data-cmp-is="text" data-target-location="false">
    
       <span><span style="	font-weight: 400;
">Although the world of mineware is not new to the security industry, it continues to grow as adversaries develop new capabilities to compromise systems and turn them into bots used for mining cryptocurrency. In this blog, we hope to provide some deeper insight into the world of mineware. We will discuss in-depth one of the most notorious mineware malware variants out there, “WannaMine.”<p> </p></span>

<span style="	font-weight: 400;
">Our deep dive will analyze the latest WannaMine variant currently being used in the wild, dubbed &#34;WannaMine v4.0,” and outline how you can successfully identify and remediate a WannaMine v4.0 infected host.</span>
<h2><span style="	font-weight: 400;
">Cryptojacking and WannaMine</span></h2>
<span style="	font-weight: 400;
">In essence, cryptojacking is the unauthorized use of a computing device to mine cryptocurrency. It occurs when adversaries compromise an organization’s systems and use their resources to mine cryptocurrency, freeing them from having to purchase hardware and electricity (more detailed information can be found in previous blogs on </span><a href="/en-us/blog/cryptomining-harmless-nuisance-disruptive-threat/"><span style="	font-weight: 400;
">cryptomining</span></a><span style="	font-weight: 400;
"> and </span><a href="/en-us/blog/whats-in-your-wallet-resource-draining-cryptojacking-attacks-are-on-the-rise-2/"><span style="	font-weight: 400;
">cryptojacking</span></a><span style="	font-weight: 400;
">). Many times, this malicious mining occurs without the victim ever realizing it due to a lack of security monitoring.<p> </p></span>

<span style="	font-weight: 400;
">As adversaries and cybercriminals searched for better ways to compromise hosts en masse, the creation of a malware dubbed “WannaMine” was born. WannaMine is a mineware malware variant created for the sole purpose of installing and running Monero software on a victim’s system and using its processing power to mine Monero for the adversary. WannaMine plays on the naming convention used for the notorious </span><a href="/en-us/epp-101/what-is-ransomware/"><span style="	font-weight: 400;
">ransomware</span></a><span style="	font-weight: 400;
"> mentioned at the beginning of the article, </span><a href="/en-us/blog/falcon-endpoint-protection-blocks-the-ransomware-that-attacked-the-nhs/"><span style="	font-weight: 400;
">WannaCry</span></a><span style="	font-weight: 400;
">. This is likely because WannaMine leverages WannaCry&#39;s exploitation code, &#34;EternalBlue,&#34; to compromise hosts and propagate the Monero mining software.</span>
<h2><span style="	font-weight: 400;
">WannaMine v4.0 Analysis and Remediation Overview</span><span style="	font-weight: 400;
"><p> </p></span></h2>
<span style="	font-weight: 400;
">Like its predecessors, WannaMine v4.0 leverages the EternalBlue exploit to spread and compromise vulnerable hosts. Its design is similar to WannaMine v3.0 in that it stores the EternalBlue exploit binaries in a directory located in C:\Windows; however, the directory in version 4.0 has been renamed &#34;NetworkDistribution.” Instead of leveraging a single hard-coded service name like WannaMine v3.0, version 4.0 will randomly generate a .dll and service name based on a list of hard-coded strings. It does this in order to maintain persistence on the host.</span>

<span style="	font-weight: 400;
">We will start with a quick high-level overview of the remediation steps that are needed, and then follow with a more detailed step-by-step walk-through.</span>

<span style="	font-weight: 400;
">The remediation of WannaMine v4.0 can be broken into the following three steps:</span>
<ol><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Killing the malicious processes (newly spawned or injected)</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Locating and removing the persistence mechanism (e.g., service)</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Removing artifacts (e.g., NetworkDistribution).</span></li></ol>
<span style="	font-weight: 400;
">The following offers details on each step:</span>
<h2><span style="	font-weight: 400;
">WannaMine v4.0 Step-by-Step Remediation</span></h2>
<em><span style="	font-weight: 400;
">Note: there are 2 scenarios. Pre-infection (CrowdStrike Falcon®</span><span style="	font-weight: 400;
">®</span><span style="	font-weight: 400;
"> is already installed and preventions are on) and post-infection detections where Falcon has been installed on the client’s endpoints after infection, therefore blocking it.<p> </p> In some of the examples shown below we have turned on DETECTIONS ONLY and PREVENTIONS off for illustrative purposes.</span></em>
<h3><span style="	font-weight: 400;
">STEP 1. Killing the Malicious svchost.exe and dllhostex.exe Processes</span></h3>
<span style="	font-weight: 400;
">As you can see in </span><span style="	font-weight: 400;
">Figure 1 and 2. </span><span style="	font-weight: 400;
">, Falcon will immediately block the launch of WannaMine’s main XMRig mining module (dllhostex.exe) and then quarantine the binary. Since the process has been killed and the binary removed, we must find the svchost.exe process that is being used to run the malicious service and kill it.<p> </p></span>

<span style="	font-weight: 400;
">Using Falcon’s process explorer, you can see that the parent process of dllhostex.exe is svchost.exe.</span>

<div class="wp-caption" id="attachment_24470"><img class="size-full wp-image-24470" src="/content/dam/crowdstrike/www/en-us/wp/2019/11/Figure-1-1.png" alt="screenshot of a process tree" width="974" height="497"/> Figure 1. Process execution tree indicating svchost.exe as the parent process of dllhostex.exe</div>

<div class="wp-caption" id="attachment_24473"><img class="wp-image-24473 size-full" src="/content/dam/crowdstrike/www/en-us/wp/2019/11/Figure-1B-1.png" alt="screenshot with process details" width="974" height="460"/> Figure 2. Further detail of specific process information within the UIBy looking over the process details within Falcon, we can quickly grab the process ID associated with the svchost.exe that is running the malicious WannaMine DLL.</div>

<img class="alignnone size-full wp-image-24474" src="/content/dam/crowdstrike/www/en-us/wp/2019/11/Figure-3A-1.png" alt="" width="974" height="435"/>

<div class="wp-caption" id="attachment_24475"><img class="size-full wp-image-24475" src="/content/dam/crowdstrike/www/en-us/wp/2019/11/Figure-4A-1.png" alt="" width="974" height="249"/> Figure 3-4. Process execution tree provides process ID information</div>

<span style="	font-weight: 400;
">From there, we can query that specific svchost.exe process, via the process ID obtained from the Falcon UI, in order to gather the service group name being used for the malicious service, in this case the netsvcs service group.</span>

<div class="wp-caption" id="attachment_24530"><img class="wp-image-24530 size-full" src="/content/dam/crowdstrike/www/en-us/wp/2019/11/Figure-5-edited-1.png" alt="" width="974" height="158"/> Figure 5. PowerShell query to output svchost service group name. Note: This must be run within the &#34;EDIT &amp; RUN SCRIPTS&#34; tab</div>

<em><span style="	font-weight: 400;
">Note: Depending on whether the SVCHOST is grouped (Microsoft refactored the way SVCHOST groups services in Windows 10 1703; read about that </span><a href="https://docs.microsoft.com/en-us/windows/application-management/svchost-service-refactoring"><span style="	font-weight: 400;
">here</span></a><span style="	font-weight: 400;
">) or if it is a single process, the removal process will vary. Windows 10, by default, will spawn an individual SVCHOST process per module but Windows 7 will group. Killing the grouped PID is not an option here as we want to minimize downtime for the clients we work with.<p> </p></span><span style="	font-weight: 400;
">Review Appendix A.3 for further insight into this grouping.</span></em>

<span style="	font-weight: 400;
">To be more specific, we can actually query the SVCHOST process using “tasklist” to output the service name associated with it, which happens to be the exact name of the malicious WannaMine DLL.</span>

<div class="wp-caption" id="attachment_24532"><img class="wp-image-24532 size-full" src="/content/dam/crowdstrike/www/en-us/wp/2019/11/Figure-6-edited-1.png" alt="" width="974" height="137"/> Figure 6. Tasklist output to display associated service name. Note: This must be run within the &#34;EDIT &amp; RUN SCRIPTS&#34; tab.</div>

<span style="	font-weight: 400;
">As an extra step, you can also query the registry key that SVCHOST based on the service group name of “netsvcs” found in the image above.</span>

<span style="	font-weight: 400;
">From the output below, we can see the “MicrosoftNetBIOSManager” DLL module that was added to the netsvcs service group. This has the same name we found previously, using the commands above.</span>

<div class="wp-caption" id="attachment_24480"><img class="size-full wp-image-24480" src="/content/dam/crowdstrike/www/en-us/wp/2019/11/Figure-5-1.png" alt="" width="974" height="122"/> Figure 7. Registry query output showing newly added malicious dll module name</div>

<span style="	font-weight: 400;
">Based on that information we can pivot and check the registry key where Windows services are stored to see if we find an associated service named “MicrosoftNetBIOSManager.” As expected, we see that there is such an entry. Looking at the values stored within the Parameters key we find the exact path to the malicious .dll:</span>

<div class="wp-caption" id="attachment_24481"><img class="wp-image-24481 size-full" src="/content/dam/crowdstrike/www/en-us/wp/2019/11/Figure-6-1.png" alt="" width="974" height="70"/> Figure 8. Registry query command to output path location to .dll on disk</div>

<div class="wp-caption" id="attachment_24482"><img class="wp-image-24482 size-full" src="/content/dam/crowdstrike/www/en-us/wp/2019/11/Figure-7-1.png" alt="" width="974" height="210"/> Figure 9. Output of registry query command showing path location to the malicious .dllTo confirm that this is the malicious DLL we are looking for, we can calculate the hash for the binary</div>

<div class="wp-caption" id="attachment_24483"><img class="wp-image-24483 size-full" src="/content/dam/crowdstrike/www/en-us/wp/2019/11/Figure-8-1.png" alt="" width="974" height="54"/> Figure 10. Built-in RTR command to gather filehash information.</div>

<div class="wp-caption" id="attachment_24484"><img class="wp-image-24484 size-full" src="/content/dam/crowdstrike/www/en-us/wp/2019/11/Figure-9-1.png" alt="" width="974" height="233"/> Figure 11. Output of the filehash command for the malicious .dll</div>

<span style="	font-weight: 400;
">Once we have the hash of the DLL, we notice that this has not been seen in VirusTotal, which is abnormal for a legitimate Windows dll stored in the System32 or SysWow64 directories. </span>

<span style="	font-weight: 400;
">In our lab environment, we infected a Windows 10 host at a specific date and time. Once infected, we inspected the creation timestamp of the malicious DLL. The time stamp provided was invalid, stating the DLL was created months prior to the initial infection. This indicated timestomping techniques had been used.</span>

<img class="alignnone size-full wp-image-24485" src="/content/dam/crowdstrike/www/en-us/wp/2019/11/Figure-10-1.png" alt="" width="974" height="120"/>

<div class="wp-caption" id="attachment_24486"><img class=" wp-image-24486" src="/content/dam/crowdstrike/www/en-us/wp/2019/11/Figure-11-1.png" alt="" width="632" height="109"/> Figure 12-13. Shows a creation date that pre-dated the in-lab installation</div>

<span style="	font-weight: 400;
">A clearer indication is seen on a Windows 7 host where the timestomping goes back to 2009. (See A.2 Timestomping Example.)</span>

<div class="wp-caption" id="attachment_24487"><img class="size-full wp-image-24487" src="/content/dam/crowdstrike/www/en-us/wp/2019/11/Figure-12-1.png" alt="" width="974" height="116"/> Figure 14. Another image showing timestomping being used on a Windows 7 host</div>

<span style="	font-weight: 400;
">Reviewing the compiler timestamp for the binary, you can see that it was created recently — in 2019 and not 2009.</span>

<div class="wp-caption" id="attachment_24488"><img class=" wp-image-24488" src="/content/dam/crowdstrike/www/en-us/wp/2019/11/Figure-13-1.png" alt="" width="518" height="257"/> Figure 15. Reviewing compiler timestamp information that proves timestomping is in fact being used</div>

<span style="	font-weight: 400;
">Another method to highlight the malicious dll being loaded by SVCHOST comes from outlier analysis (</span><span style="	font-weight: 400;
">Figure 16.</span><span style="	font-weight: 400;
">). We see the hard-coded path for MicrosoftNetBIOSManager (</span><span style="	font-weight: 400;
">Figure 17.</span><span style="	font-weight: 400;
">) which is odd and adds context to the above indicating this isn’t native to the OS.</span>

<div class="wp-caption" id="attachment_24533"><img class="wp-image-24533 size-full" src="/content/dam/crowdstrike/www/en-us/wp/2019/11/Figure-16-edited-1.png" alt="" width="974" height="72"/> Figure 16. Registry query used to show further outlier information indicating the difference between the known legitimate and malicious .dll. Note: This must be run within the &#34;EDIT &amp; RUN SCRIPTS&#34; tab.</div>

<span style="	font-weight: 400;
">An example of the many ServiceDLL fields and what they look like before filtering again on the hard-coded path </span><b>C:\Windows\System32\ </b><span style="	font-weight: 400;
">as opposed to </span><b>%systemroot%</b><span style="	font-weight: 400;
">.</span>

<div class="wp-caption" id="attachment_24490"><img class="size-full wp-image-24490" src="/content/dam/crowdstrike/www/en-us/wp/2019/11/Figure-15-1.png" alt="" width="974" height="666"/> Figure 17. Output of the registry command indicating the differences</div>

<span style="	font-weight: 400;
">Now that we have confirmed the SVCHOST process is indeed the one associated with the malicious WannaMine service, let’s kill the process. Gracefully stopping the service will end the process.</span>

<div class="wp-caption" id="attachment_24534"><img class="wp-image-24534 size-full" src="/content/dam/crowdstrike/www/en-us/wp/2019/11/Figure-18-edited-1.png" alt="" width="479" height="49"/> Figure 18. PowerShell command to stop the malicious service. Note: This must be run within the &#34;EDIT &amp; RUN SCRIPTS&#34; tab.</div>
<h3><span style="	font-weight: 400;
">STEP 2. Removing the Persistence</span></h3>
<span style="	font-weight: 400;
">While discovering and killing the svchost.exe process being used to launch the WannaMine service, we found and confirmed the service name being used for persistence.</span>

<span style="	font-weight: 400;
">Now we remove the service so WannaMine v4.0 no longer has persistence in place.</span>

<div class="wp-caption" id="attachment_24535"><img class="wp-image-24535 size-full" src="/content/dam/crowdstrike/www/en-us/wp/2019/11/Figure-19-edited-1.png" alt="" width="874" height="40"/> Figure 19. Powershell command to remove the service after it has been stopped. Note: This must be run within the &#34;EDIT &amp; RUN SCRIPTS&#34; tab.</div>

<div class="wp-caption" id="attachment_24536"><img class="wp-image-24536 size-full" src="/content/dam/crowdstrike/www/en-us/wp/2019/11/Figure-20-edited-1.png" alt="" width="974" height="370"/> Figure 20. Output provided after running the service removal command</div>

<span style="	font-weight: 400;
">Just like that, we have removed the malicious service and relinquished WannaMine v4.0’s persistence!</span>
<h3><span style="	font-weight: 400;
">STEP 3. Removing Remaining Artifacts</span></h3>
<span style="	font-weight: 400;
">Now that we have killed the SVCHOST process and removed the persistence, it’s time to clean up and remove the remaining artifacts.</span>

<span style="	font-weight: 400;
">Based on our research, WannaMine v4.0 has a few specific artifacts that it places on the host. The first one is the NetworkDistribution folder located in C:\Windows. This folder contains all of the Equation Group binaries (e.g., EternalBlue, Double Pulsar, etc.) and needs to be removed.</span>

<div class="wp-caption" id="attachment_24494"><img class="size-full wp-image-24494" src="/content/dam/crowdstrike/www/en-us/wp/2019/11/Figure-19-1.png" alt="" width="974" height="645"/> Figure 21. Depicts the folder named “NetworkDistribution” and some of its contents</div>

<div class="wp-caption" id="attachment_24537"><img class="wp-image-24537 size-full" src="/content/dam/crowdstrike/www/en-us/wp/2019/11/Figure-22-edited-1.png" alt="" width="384" height="45"/> Figure 22. Command used to remove the entire directory</div>

<span style="	font-weight: 400;
">The next artifact to remove is the malicious DLL that we discovered in step one. This is located in C:\Windows\System32.</span>

<div class="wp-caption" id="attachment_24496"><img class="wp-image-24496 size-full" src="/content/dam/crowdstrike/www/en-us/wp/2019/11/Figure-21-1.png" alt="" width="974" height="50"/> Figure 23. Built-in RTR command used to remove the malicious .dll</div>

<span style="	font-weight: 400;
">Next, we have the dllhostex.exe that is the binary that WannaMine v4.0 uses to run the XMRig miner module. As seen Figure 1, Falcon quarantines this binary; however, if it was not quarantined you can find it in C:\Windows\System32.</span>

<div class="wp-caption" id="attachment_24497"><img class="size-full wp-image-24497" src="/content/dam/crowdstrike/www/en-us/wp/2019/11/Figure-22-1.png" alt="" width="974" height="54"/> Figure 24. Built-in RTR command used to remove the XMRig miner module binary</div>

<span style="	font-weight: 400;
">Lastly, a registry entry that contains the descriptive text for the service.</span>

<div class="wp-caption" id="attachment_24538"><img class="wp-image-24538 size-full" src="/content/dam/crowdstrike/www/en-us/wp/2019/11/Figure-25-edited-1.png" alt="" width="888" height="42"/> Figure 25. PowerShell command to remove the remaining registry artifact. Note: This must be run within the &#34;EDIT &amp; RUN SCRIPTS&#34; tab.</div>
<h3>Completion</h3>
<span style="	font-weight: 400;
">Congratulations! If you followed the above steps, you have successfully discovered and remediated the pesky WannaMine v4.0 malware.</span>

Using a “query first then kill” methodology, you can confirm a host is infected prior to running any remediation kill scripts. This helps our analysts quickly remediate systems at scale.</span>
