								<h1>Global outbreak of Log4Shell</h1>
								<div class="date">December 16, 2021 &nbsp;|&nbsp;
									<a href="/blogs/author/santiago-cortes">Santiago Cortes</a></div>
							</div>
							<div class="blog-body">
								<h2>Executive summary</h2>

<p>Log4Shell is a high severity vulnerability (CVE-2021-44228) impacting Apache Log4j versions 2.0 to 2.14.1. It was discovered by Chen Zhaojun of Alibaba Cloud Security Team and disclosed via the project&acute;s GitHub repository on December 9, 2021.</p>

<h3>Key takeaways:</h3>

<ul>
	<li>Prevalent utility Log4j across the industry allows unauthenticated remote code execution.</li>
	<li>The publicly available proof-of-concept and vulnerability&rsquo;s easy exploitability make this vulnerability particularly dangerous.</li>
	<li>Different opportunistic campaigns are taking advantage of the vulnerability to spread malware like botnets and miners.</li>
</ul>

<h2>Background</h2>

<p>Log4j is an open-source Java logging utility developed by the Apache Foundation. It is widely used as a prevalent dependency in many applications and services. If exploited, the vulnerability allows for unauthenticated remote code execution, leaving services particularly exposed .<br />
<br />
An attacker that can forge log messages or their parameters may manage to execute arbitrary code loaded from malicious LDAP servers if message lookup substitution is enabled. (LDAP, or lightweight directory access protocol, is a protocol that makes it possible for applications to query user information rapidly.) Log4j disabled this feature in version 2.15.0 in early December 2021.</p>

<h2>Analysis</h2>

<p>Log4j includes a lookup mechanism to retrieve information like &ldquo;${java:runtime}&rdquo; and &ldquo;${java:os}&rdquo; from the system, but also to make requests using Java Naming and Directory Interface (JNDI). The key issue is that many services may log user provided information without proper input validation. For example, URLs requested or any of its headers, such as the User-Agent used in a HTTP request, are commonly logged.</p>

<p>JNDI can use different service provider interfaces (SPIs) like LDAP to find and invoke objects, and as the logging information can be forged by an unauthenticated user, a vulnerable service may reach an arbitrary LDAP server under control of the attacker to invoke a malicious payload.<br />
<br />
We can observe the growth of JNDI related scans cross the internet:</p>

<p><img alt="bump in activity" src="https://cdn.levelblue.com/blog-content/bump_in_activity.png" /></p>

<p style="text-align:center">Figure 1. JNDI related scans across honeypots.</p>

<p>According to a Netlab <a href="https://blog.netlab.360.com/yi-jing-you-xxxge-jia-zu-de-botnetli-yong-log4shelllou-dong-chuan-bo-wei-da-bu-ding-de-gan-jin-liao/" target="_blank">blog</a> on December 13, 2021, Netlab identified 10 different implants using the vulnerability to spread:</p>

<ul>
	<li>Muhstik, DDoS+backdoor</li>
	<li>Mirai</li>
	<li>DDoS family Elknot</li>
	<li>Mining family m8220</li>
	<li>SitesLoader</li>
	<li>xmrig.pe / xmrig.ELF</li>
	<li>Meterpreter variants</li>
</ul>

<p>According to <a href="https://twitter.com/Adam_Cyber/status/1470152554310950917" target="_blank">Crowdstrike</a>, their research team has identified campaigns leveraging the vulnerability consistent with advanced attackers, such as deploying web shells and conducting lateral movement.</p>

<p>LevelBlue Labs has identified prevalent obfuscation techniques to avoid potential detection and protection mechanisms, like using the lookup keywords upper and lower and by using lookup arguments like &ldquo;${::-j}&rdquo; or, even with an extra tweak, the following lookup would be translated as a j: &ldquo;<i>${env:ENV_NAME:-j}&rdquo;</i>.</p>

<p>For example, a lookup like:</p>

<p style="text-align:center"><img alt="JNDI lookup" src="https://cdn.levelblue.com/blog-content/JNDI_lookup.jpg" /></p>

<p style="text-align:center">Figure 3. Exploitation example.</p>

<p>Could be obfuscated as:</p>

<p style="text-align:center"><img alt="obfuscated lookup" src="https://cdn.levelblue.com/blog-content/obfuscated_lookup.jpg" /></p>

<p style="text-align:center">Figure 4. Obfuscation example</p>

<p>We have also seen references of obfuscation using base64 by invoking &ldquo;/Basic/Command/Base64/&rdquo; in the destination, for example in the event:</p>

<p style="text-align:center"><img alt="Base64 obfuscation" src="https://cdn.levelblue.com/blog-content/base64_obfuscation.jpg" /></p>

<p style="text-align:center">Figure 5. Base64 obfuscation example.<br />
&nbsp;</p>

<p>The base64 deobfuscates to:</p>

<p style="text-align:center"><img alt="deobfuscation" src="https://cdn.levelblue.com/blog-content/deobfuscation.jpg" /></p>

<p style="text-align:center">Figure 6. Deobfuscated payload.<br />
&nbsp;</p>

<p>In addition to being leveraged for obfuscation, environmental variables are being used in other ways. <a href="https://news.sophos.com/en-us/2021/12/12/log4shell-hell-anatomy-of-an-exploit-outbreak/" target="_blank">Sophos</a> has reported on campaigns that are stealing AWS secrets by requesting environment variables in the lookup:</p>

<p style="text-align:center"><img alt="retrieving secrets" src="https://cdn.levelblue.com/blog-content/retrieving_secrets.jpg" /></p>

<p style="text-align:center">Figure 7. Retrieving secrets from environment variables.</p>

<p>To make sure the string is evaluated, attackers are injecting the lookups in every available field inside a HTTP request. For example:</p>

<p><img alt="exploitation log4j" src="https://cdn.levelblue.com/blog-content/exploitation_log4j.jpg" /></p>

<p style="text-align:center">Figure 8. Exploitation attempt leveraging all available fields.</p>

<h2>Recommended actions</h2>

<ol>
	<li>Identify if any of your servers use Log4j and patch or update Log4j to the latest version.</li>
	<li>If you are unable of updating or patching, there are some workarounds recommended by Apache:
	<ol>
		<li>Disable lookups when executing Java by adding the option:<br />
		&nbsp;<i>-Dlog4j2.formatMsgNoLookups=true</i></li>
		<li>Disable lookups by setting an environment variable:<br />
		<i>set LOG4J_FORMAT_MSG_NO_LOOKUPS=true</i></li>
		<li>Repackage your log4j-core-*.jar file by deleting the JNDI component:<br />
		<i>zip -q -d log4j-core-*.jar org/apache/logging/log4j/core/lookup/JndiLookup.class</i></li>
	</ol>
	</li>
	<li>Review your application logs for jndi lookups with the command:<br />
	<i>sudo egrep -i -r &#39;\$\{jndi:(ldap[s]?|rmi|dns):/[^\n]+&#39; /var/log</i></li>
	<li><i>Review detections of suspicious child processes spawned by Java</i></li>
</ol>

<h2>Conclusion</h2>

<p>Log4Shell can potentially have a very large impact at the end of 2021, based on the number of exposed and vulnerable devices and the facility of its exploitation. In fact, it will likely be remarked as one of the most significant vulnerabilities of 2021.</p>

<p>LevelBlue Labs will keep monitoring the situation and will update an <a href="https://otx.alienvault.com/pulse/61bb5295973e2f639b157c6b" target="_blank">OTX Pulse</a> to keep our customers protected.</p>
