    <div class='cylance-blog-container'><p>Apparently Locky always comes back.</p> <p>A persistent threat, Locky ransomware apparently has no plans of disappearing anytime soon. Locky has caused issues recently when it was used to attack Hollywood Presbyterian Medical Center during <a href="http://www.latimes.com/business/technology/la-me-ln-hollywood-hospital-bitcoin-20160217-story.html" target="_blank" adhocenable="false">February 2016</a> where it claimed nearly 400,000 victims in the very first week of its detection. It was about this time that we <a href="/locky-holds-worlds-critical-systems-to-ransom" target="_blank" adhocenable="false">first looked at Locky</a>.</p> <p>The largest publicly admitted ransom was paid by the Hollywood Hospital, $17,000 in Bitcoin. The second largest sum was $1,600 in Bitcoin paid by the <a href="https://www.trendmicro.com/vinfo/us/security/news/cyber-attacks/locky-ransomware-strain-led-kentucky-hospital-to-an-internal-state-of-emergency" target="_blank" adhocenable="false">Methodist Hospital</a>. Since then, we have seen a lot of variants of the ransomware, named for the file extensions given to the encrypted files. Other variants we’ve seen have included zepto, thor, and osiris. Now we have Diablo6.</p> <h2><b>Little Changes in Known Malware Make it Fresh and New Again</b></h2> <p>Endpoint security is improving by the day but so are cybercriminals; this makes detecting the criminals a never-ending game of cat and mouse. It is often found that malware authors proactively monitor the detection rates of their product, allowing them to stay one step ahead of AV vendors by making improvements to their code to avoid detection. In some cases, authors can make small changes in their code to keep their malware as dangerous to the end user as it was the day they released it.</p> <p>This appears to be the case the case with the Locky ransomware.&nbsp; This old malware didn’t need to have anything new, as the authors behind Locky just had to tweak the only part of the process that can never be fixed - the end user. The most recent change for Locky came as one of the most popular ways to spread malware: spear phishing emails.</p> <p>In this blog post, a VBS file archived via zip is dissected showing the techniques used by threat actors to avoid detection.</p> <h2><b>Our Technical Teardown</b></h2> <p>The original command-and-control server (C2) at the time of this writing is down, so I will be using <a href="http://www.inetsim.org/" target="_blank" adhocenable="false">INetSim</a> to simulate the C2. INetSim is a Linux tool used to emulate common Internet services such as HTTP, HTTPS, DNS, and many others. Implemented on a <a href="https://remnux.org/" target="_blank" adhocenable="false">REMnux</a> virtual machine in our lab environment, INetSim will serve as a fake DNS server. This allows for the VBS script to connect to its C2 at atesbocegianaokulu [dot] com/y872ff2f. Once the DNS request is made, INetSim will act as the C2 and serve the malicious payload.</p> <p>The attack happens in two stages: The first stage is the spear phishing email that has a zip archive attached, pictured in Figure 1.a. Inside the archive is a VBS file with the same name as the archive, pictured in the Figure 1.b.</p> <p><i style="text-align: center;"><img width="100%" src="/content/dam/blogs-blackberry-com/images/blogs/2017/11/Fig1a-Locky.png"></i></p> <p style="text-align: center;"><i>Figure 1.a - Phishing Email</i></p> <p style="text-align: center;"><i style="font-size: 15px;"><img src="/content/dam/blogs-blackberry-com/images/blogs/2017/11/Fig1b-Locky.png"></i><br> </p> <p style="text-align: center;"><i>Figure 1.b - Zip File and VBS File</i></p> <p>When the victim decompresses the archive, and clicks on the file, the VBS script starts to run. The script tries to connect to the C2 and download the ransomware payload as seen in the Figure 2.a and Figure 2.b:</p> <p style="text-align: center;"><img width="100%" style="text-align: center;" src="/content/dam/blogs-blackberry-com/images/blogs/2017/11/Fig2a-Locky.png"></p> <p style="text-align: center;"><i>Figure 2.a - VBS script code:<b>&nbsp;</b>the code has two different C2s separated with a dash and the name of the payload. If the VBS script can’t download the payload from the first C2 it tries to download it from the second one.</i></p> <p><i style="text-align: center;"><img width="100%" src="/content/dam/blogs-blackberry-com/images/blogs/2017/11/Fig2b-Locky-CROP.png"></i></p> <p style="text-align: center;"><i>Figure 2.b - VBS Script Code New Name</i></p> <p>The word “Enterprise” in Figure 2.b is used to avoid the malicious code detection by static analysis in an attempt to make the end user think that it is a valid component of some sort of enterprise utility. At the same time, it uses the string to split the malicious instructions, the real instructions are:</p> <p><b>Microsoft.XMLHTTP Adodb.streaM shell.Application Wscript.shell Process Get Temp Type Open Write ResponseBody Savetofile \GlNPcFUJR.exe http: //</b></p> <p>In Figure 3, the VBS script can be seen connecting to the C2 and downloading the payload.</p> <p><i style="text-align: center;"><img width="100%" src="/content/dam/blogs-blackberry-com/images/blogs/2017/11/Fig3-Locky.png"></i></p> <p style="text-align: center;"><i>Figure 3 - PCAP vbs Script</i></p> <p>The script saves the second stage payload in the AppData/Local/Temp folder with a different name, then runs the malware as seen in Figure 4 and Figure 5.</p> <p><i style="text-align: center;"><img width="100%" src="/content/dam/blogs-blackberry-com/images/blogs/2017/11/Fig4-Locky.png"></i></p> <p style="text-align: center;"><i>Figure 4 -Temp Folder</i></p> <p style="text-align: center;"><i><img src="/content/dam/blogs-blackberry-com/images/blogs/2017/11/Fig5-Locky.png"></i></p> <p style="text-align: center;"><i>Figure 5 - Stage 2 of the Malware is Running</i></p> <p>The attack ends after encryption with the ransom message, seen below in Figure 6, then deletes itself.</p> <p><i style="text-align: center;"><img width="100%" src="/content/dam/blogs-blackberry-com/images/blogs/2017/11/Fig6-Locky.png"></i></p> <p style="text-align: center;"><i>Figure 6 - Ransomware Window</i></p> <p>Now all files on the system are encrypted with Diablo6, which we already know as Locky ransomware. &nbsp;Figure 6.a shows the affected files, post-encryption, with the new Diablo6 file type.</p> <p style="text-align: center;"><i><img src="/content/dam/blogs-blackberry-com/images/blogs/2017/11/Fig6a-Locky.png"></i></p> <p style="text-align: center;"><i>Figure 6.a - DIABLO6 Encryption</i></p> <p>The Locky Diablo6 Ransomware will target the following file types:</p> <p><i>.</i>sql, .mp4, .7z, .rar, .m4a, .wma<i>, .</i>avi<i>, .</i>wmv, .csv, .d3dbsp, .zip, .sie, .sum, .ibank, .t13, .t12, .qdf<i>, .</i>gdb, .tax, .pkpass, .bc6, .bc7, .bkp<i>, .</i>qic<i>, .</i>bkf<i>, .</i>sidn<i>, .</i>sidd<i>, .</i>mddata<i>, .</i>itl<i>, .</i>itdb<i>, .</i>icxs<i>, .</i>hvpl<i>, .</i>hplg<i>, .</i>hkdb<i>, .</i>mdbackup<i>, </i>.backup, .backupdb, <i>.</i>syncdb<i>, .</i>gho<i>, .</i>cas<i>, .</i>svg, .map, .wmo<i>, .</i>itm<i>, .</i>sb<i>, .</i>fos<i>, .</i>mov<i>, .</i>vdf<i>, .</i>ztmp, .sis, .sid<i>, .</i>ncf, .menu, .layout, .dmp, .blob, .esm<i>, .</i>vcf<i>, .</i>vtf<i>, .</i>dazip<i>, .</i>fpk, .mlx, .kf<i>, .</i>iwd<i>, .</i>vpk, .tor, .psk, .rim, .w3x, .fsh<i>, .</i>ntl, .arch00, .lvl<i>, .</i>snx<i>, .</i>cfr<i>, .</i>ff<i>, .</i>vpp_pc, .lrf, .m2, .mcmeta, .vfs0, .mpqge, .kdb, .db0, .dba, .rofl, .hkx, .bar, .upk, .das, .iwi, .litemod, .asset, .forge, .ltx, .bsa, .apk, .re4, .sav, .lbf, .slm, .bik, .epk, .rgss3a, .pak, .big, wallet, .wotreplay, .xxx, .desc, .py, .m3u, .flv, .js, .css, .rb, .png, .jpeg, .txt, .p7c, .p7b, .p12, .pfx, .pem, .crt, .cer, .der, .x3f, .srw, .pef, .ptx, .r3d, .rw2, .rwl, .raw, .raf, .orf, .nrw, .mrwref, .mef, .erf, .kdc, .dcr, .cr2, .crw, .bay, .sr2, .srf, .arw, .3fr, .dng, .jpe, .jpg, .cdr, .indd, .ai, .eps, .pdf, .pdd<i>, .</i>psd, .dbf, .mdf, .wb<i>2, .rtf, .</i>wpd, .dxg, .xf, .dwg, .pst, .accdb, .mdb, .pptm<i>, .pptx, .ppt, .</i>xlk<i>, .xlsb</i>, .xlsm, .<i>xlsx</i>, .<i>xls, .</i>wps<i>, .</i>docm, .docx, .doc, .odb, .odc, .odm, .<i>odp</i>, .ods, .<i>odt</i>.</p> <p>The domain dbr663dnbssfrodison[dot]net was recently created on August 1st, 2017 using the registrant email: jenniemarc(at)mail.com. A reverse Whois Lookup on that account shows that 333 domains were registered by this email starting in 2016 and as recently as October 2017. Some of those domains are known to be serving other families of ransomware which can be seen in the appendix. We can use this registrant and these domains to proactively detect and prevent the download of the Locky payload.</p> <h2><b>How to Stop Locky and Attacks Like It</b></h2> <p>Phishing emails can be tricky, as they’re specifically designed to trick the end user into initiating an action that they believe to be legitimate. Whether it’s opening a malicious email attachment or entering credentials to a fake Gmail or Yahoo login page, mitigating social engineering attacks can be achieved with user education. Below are some simple actions to take to help prevent these types of attacks:</p> <p style="margin-left: 40px;">1.&nbsp; Never open email if you don’t know the sender.<br> 2.&nbsp; Never open a document if you don’t know where it came from.<br> 3.&nbsp; Backup your personal information constantly.</p> <p>If you use our endpoint protection product,&nbsp;<a href="https://www.cylance.com/en_us/products/our-products/protect.html">CylancePROTECT®,</a>&nbsp;you are&nbsp;<a href="https://www.cylance.com/en_us/blog/cylance-vs-locky-ransomware.html">already protected</a>&nbsp;from this attack.</p> <p>This is one form of many different attack patterns that we are seeing. The threat can be distributed via HTML attachments disguised as invoices, Word documents embedded with malicious macro code or Visual Basic scripts (VBS), malicious URLs in spam emails, VBS, JS files
