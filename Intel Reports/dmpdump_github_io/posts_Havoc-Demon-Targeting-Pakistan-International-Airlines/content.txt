data-toc-skip>Havoc Demon Targeting Pakistan International Airlines</h1><div class="post-meta text-muted"> <span> Posted <time data-ts="1751155200" data-df="ll" data-bs-toggle="tooltip" data-bs-placement="bottom" > Jun 29, 2025 </time> </span><div class="d-flex justify-content-between"> <span> By <em> <a href="https://github.com/dmpdump">dmpdump</a> </em> </span><div> <span class="readtime" data-bs-toggle="tooltip" data-bs-placement="bottom" title="911 words" > <em>5 min</em> read</span></div></div></div></header><div class="content"><p>Back in January 2025, I reviewed a <a href="https://dmpdump.github.io/posts/Havoc/">campaign</a> delivering Havoc Demon to targets in Bangladesh, Pakistan, and China via LNK files. While hunting for new threats this month, I came across an malicious Word document uploaded from Pakistan which leads to a very similar infection chain, very likely attributable to the same threat actor. It is rare to find malicious macro-enabled Office documents these days, but I felt nostalgic and decided to take a look.</p><ul><li>File names: <code class="language-plaintext highlighter-rouge">HTCL_Report.doc</code> and <code class="language-plaintext highlighter-rouge">Aircraft_Modification.doc</code><li>SHA2: a27f2936eb86674120cd54f293670362d51f4784cecb7cf60bf8b78752f24b70</ul><p>The Word document was uploaded to VirusTotal on June 25, 2025 from Pakistan. The document was uploaded 4 times by 2 distinct submitters.</p><p><a href="/assets/images/demonpia/vt.png" class="popup img-link shimmer"><img src="/assets/images/demonpia/vt.png" alt="sshot" loading="lazy"></a></p><p>The document metadata shows that it was created on May 5, 2025 and last modified on June 25,2025 (the same day of the VirusTotal upload) by user <code class="language-plaintext highlighter-rouge">kali</code>. Tooling is also revealed in the metadata. We can see that <code class="language-plaintext highlighter-rouge">WPS Office</code>, a Microsoft Office alternative, was likely used.</p><p><a href="/assets/images/demonpia/metadata.png" class="popup img-link shimmer"><img src="/assets/images/demonpia/metadata.png" alt="sshot" loading="lazy"></a></p><p>When the Word document is opened, we can see that it has macros and a lure that suggests targeting of Pakistan International Airlines (PIA). The document title is <code class="language-plaintext highlighter-rouge">TMS Data - June 2025</code>. It contains a lure to entice the victim into enabling macros to view the document correctly.</p><p><a href="/assets/images/demonpia/maldoc.png" class="popup img-link shimmer"><img src="/assets/images/demonpia/maldoc.png" alt="sshot" loading="lazy"></a></p><p>Anomalous activity stands out as soon as macros are enabled. A new <code class="language-plaintext highlighter-rouge">WINWORD.exe</code> process is created, with allocated RWX memory containing a portable executable. This is typically a good sign of process injection.</p><p><a href="/assets/images/demonpia/winword_inject.png" class="popup img-link shimmer"><img src="/assets/images/demonpia/winword_inject.png" alt="sshot" loading="lazy"></a></p><p>Extracting the Word macros, we can immediately see a common obfuscation technique consisting of concatenation of hex encoded data blobs, using the <code class="language-plaintext highlighter-rouge">ObjectExceed</code> function as a hex decoder.</p><p><a href="/assets/images/demonpia/badmacro1.png" class="popup img-link shimmer"><img src="/assets/images/demonpia/badmacro1.png" alt="sshot" loading="lazy"></a></p><p><a href="/assets/images/demonpia/objexceed.png" class="popup img-link shimmer"><img src="/assets/images/demonpia/objexceed.png" alt="sshot" loading="lazy"></a></p><p>Additionally, we can see that the concatenated hex blob is base64 decoded.</p><p><a href="/assets/images/demonpia/b64.png" class="popup img-link shimmer"><img src="/assets/images/demonpia/b64.png" alt="sshot" loading="lazy"></a></p><p>In order to speed up analysis, we can modify the macro so that it concatenates the hex-encoded blob for us and dumps it to a file for analysis.</p><p><a href="/assets/images/demonpia/dumper1.png" class="popup img-link shimmer"><img src="/assets/images/demonpia/dumper1.png" alt="sshot" loading="lazy"></a></p><p>After dumping the file, we can base64 decode it and analyze the malicious macro code. A quick review of the macro shows that it executes payload from memory. We can see this in a function named <code class="language-plaintext highlighter-rouge">RightAmex</code>, which modifies memory permissions to <code class="language-plaintext highlighter-rouge">PAGE_EXECUTE_READWRITE</code> and subsequently invokes payload in that executable memory with the low level Invoke API <code class="language-plaintext highlighter-rouge">DispCallFunc</code>.</p><p><a href="/assets/images/demonpia/rightamex.png" class="popup img-link shimmer"><img src="/assets/images/demonpia/rightamex.png" alt="sshot" loading="lazy"></a></p><p>Tracing the payload executed in memory, we can see that it is executed from function <code class="language-plaintext highlighter-rouge">ModelTransaction</code>.</p><p><a href="/assets/images/demonpia/modeltransaction.png" class="popup img-link shimmer"><img src="/assets/images/demonpia/modeltransaction.png" alt="sshot" loading="lazy"></a></p><p>Tracing back that function, we can see that it is executed from a series of functions that determine if the architecture is 64 bit or 32 bit. Specifically, the function <code class="language-plaintext highlighter-rouge">TheKeogh</code> is responsible for determining the system architecture.</p><p><a href="/assets/images/demonpia/thekeogh.png" class="popup img-link shimmer"><img src="/assets/images/demonpia/thekeogh.png" alt="sshot" loading="lazy"></a></p><p><code class="language-plaintext highlighter-rouge">TheKeogh</code> is used by <code class="language-plaintext highlighter-rouge">ThisCost</code>, which decides which payload will be executed via <code class="language-plaintext highlighter-rouge">ModelTransaction</code>.</p><p><a href="/assets/images/demonpia/archdecide.png" class="popup img-link shimmer"><img src="/assets/images/demonpia/archdecide.png" alt="sshot" loading="lazy"></a></p><p>Looking at the embedded 64 bit and 32 bit payloads, we can see that the same obfuscation technique used previously is also used here - concatenation of hex encoded blobs:</p><p><a href="/assets/images/demonpia/objblob2.png" class="popup img-link shimmer"><img src="/assets/images/demonpia/objblob2.png" alt="sshot" loading="lazy"></a></p><p>We can therefore use the same technique to dump the payload. We can modify the macro, removing the functions loading the payload, and adding a function to dump the concatenated blob. In this case, I chose the <code class="language-plaintext highlighter-rouge">ModelAsset</code> function to concatenate the payload before dumping it.</p><p><a href="/assets/images/demonpia/dumper2.png" class="popup img-link shimmer"><img src="/assets/images/demonpia/dumper2.png" alt="sshot" loading="lazy"></a></p><h1 id="final-payload-havoc-demon">Final payload: Havoc Demon</h1><p>Reviewing the dumped payload, we can see that it the same <a href="https://dmpdump.github.io/posts/Havoc/">shellcode</a> that was delivered in the activity that I reviewed in January 2025. The shellcode also loads an embedded Havoc Demon reflectively.</p><p>In the beginning of the dumped payload we see the start of the shellcode, which loads the embedded portable executable.</p><p><a href="/assets/images/demonpia/sc1.png" class="popup img-link shimmer"><img src="/assets/images/demonpia/sc1.png" alt="sshot" loading="lazy"></a></p><p>The portable executable embedded in the shellcode:</p><p><a href="/assets/images/demonpia/sc2.png" class="popup img-link shimmer"><img src="/assets/images/demonpia/sc2.png" alt="sshot" loading="lazy"></a></p><p>Like the previous shellcode, this one also uses the <code class="language-plaintext highlighter-rouge">djb2</code> algorithm to resolve APIs via hashes.</p><p><a href="/assets/images/demonpia/apihashing.png" class="popup img-link shimmer"><img src="/assets/images/demonpia/apihashing.png" alt="sshot" loading="lazy"></a></p><p><code class="language-plaintext highlighter-rouge">djb2</code> implementation for API hashing:</p><p><a href="/assets/images/demonpia/apihashing2.png" class="popup img-link shimmer"><img src="/assets/images/demonpia/apihashing2.png" alt="sshot" loading="lazy"></a></p><p>The shellcode is a typical reflective loader which performs the following actions:</p><ul><li>Searches for a portable executable (in memory) by locating the DOS Header (‘MZ’), validating that a PE header (‘PE’) exists<li>Resolves native APIs for memory allocation (<code class="language-plaintext highlighter-rouge">NtAllocateVirtualMemory</code>) and memory protection updates (<code class="language-plaintext highlighter-rouge">NtProtectVirtualMemory</code>)<li>Allocates memory<li>Copies sections of the portable executable to the newly allocated memory<li>Updates section permissions<li>Executes the portable executable</ul><p>The base address of the embedded PE is located with the following loop:</p><p><a href="/assets/images/demonpia/locatepe.png" class="popup img-link shimmer"><img src="/assets/images/demonpia/locatepe.png" alt="sshot" loading="lazy"></a></p><p>The following is an overview of the decompiled code with the overall shellcode functionality:</p><p>It locates the base address of the PE, resolves APIs, and allocates memory: <a href="/assets/images/demonpia/scdemon1.png" class="popup img-link shimmer"><img src="/assets/images/demonpia/scdemon1.png" alt="sshot" loading="lazy"></a></p><p>It copies the sections of the PE, updates permissions, and finally executes the PE: <a href="/assets/images/demonpia/scdemon2.png" class="popup img-link shimmer"><img src="/assets/images/demonpia/scdemon2.png" alt="sshot" loading="lazy"></a></p><p>A quick triage of the payload loaded by the shellcode reveals that its original name is <code class="language-plaintext highlighter-rouge">demon.x64.dll</code>, suggesting that we are dealing with a Havoc demon.</p><p><a href="/assets/images/demonpia/demon.png" class="popup img-link shimmer"><img src="/assets/images/demonpia/demon.png" alt="sshot" loading="lazy"></a></p><p>The configuration of the demon is not encrypted, so we can retrieve the C2 easily. In this case, the threat actor decided to use Microsoft’s <a href="https://learn.microsoft.com/en-us/azure/developer/dev-tunnels/overview">dev tunnels</a>, which allows developers to share local environments over the internet (via Microsoft’s infrastructure). This feature is supposed to be used for ad-hoc/testing use cases. The C2 configured in the demon is <code class="language-plaintext highlighter-rouge">hxxp://djlmwd9b-80.euw.devtunnels[.]ms/</code>. The main advantage of using a service such as dev tunnels for C2 redirection is that threat actors do not need to worry about setting up C2 domains that may end up being blocked. The C2 traffic will be generated towards legitimate Microsoft infrastructure, therefore increasing the chances of evading detection and blocks. The use of dev tunnels has been documented before, even for Havoc demons, such as in the <a href="https://medium.com/@manan07/microsoft-devtunnel-with-havoc-c2-f72de5fcd9ba">following</a> article.</p><p><a href="/assets/images/demonpia/demonconfig.png" class="popup img-link shimmer"><img src="/assets/images/demonpia/demonconfig.png" alt="sshot" loading="lazy"></a></p><p>Given the many overlaps with the previous activity (in targeting, shellcode, and payload), this new malicious Word document is very likely attributed to the same threat actor that I (and others) reviewed in January 2025.</p><h1 id="iocs">IOCs</h1><ul><li>Word document: a27f2936eb86674120cd54f293670362d51f4784cecb7cf60bf8b78752f24b70<li>Shellcode:
