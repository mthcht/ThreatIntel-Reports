class="px-1"><header><h1 data-toc-skip>Likely Chinese Threat Actor Uses Low Detection Linux Backdoor and NHAS Reverse SSH</h1><div class="post-meta text-muted"> <span> Posted <time data-ts="1746057600" data-df="ll" data-bs-toggle="tooltip" data-bs-placement="bottom" > May 1, 2025 </time> </span><div class="d-flex justify-content-between"> <span> By <em> <a href="https://github.com/dmpdump">dmpdump</a> </em> </span><div> <span class="readtime" data-bs-toggle="tooltip" data-bs-placement="bottom" title="754 words" > <em>4 min</em> read</span></div></div></div></header><div class="content"><p>On April 22, 2025, <a href="https://x.com/malwrhunterteam/status/1914632170129309952">MalwareHunterTeam</a> shared a hash for a low detection Linux ELF with 2 hard-coded IP addresses: 43.159.18[.]135 and 119.42.148[.]187. Upon review of the executable (ea41b2bf1064efcb6196bb79b40c5158fc339a36a3d3ddee68c822d797895b4e), I found an interesting backdoor written in C that uses a local proxy to execute payload from attacker infrastructure via an external SOCKS5 proxy. The ELF is a 64-bit executable, has stripped symbols and was compiled with GCC.</p><p><a href="/assets/images/elf_nhas/elfinfo.png" class="popup img-link shimmer"><img src="/assets/images/elf_nhas/elfinfo.png" alt="sshot" loading="lazy"></a></p><p>The ELF was uploaded from China and Singapore, with both submissions very close to each other.</p><p><a href="/assets/images/elf_nhas/submissions.png" class="popup img-link shimmer"><img src="/assets/images/elf_nhas/submissions.png" alt="sshot" loading="lazy"></a></p><h1 id="backdoor-analysis">Backdoor analysis</h1><p>The backdoor uses the environment variable <code class="language-plaintext highlighter-rouge">_PROXY_SERVICE_CHILD</code> to detect if it is running as a child process. If the environment variable is not set (itâ€™s not running as a child process), it starts the malicious logic. The malware forks a new process using <code class="language-plaintext highlighter-rouge">kworker/0:0</code> as an argument to look like a legitimate kernel worker thread (likely modifying the process name). If the fork suceeds, it sets the <code class="language-plaintext highlighter-rouge">_PROXY_SERVICE_CHILD</code> environment variable to 1 and executes itself with <code class="language-plaintext highlighter-rouge">/proc/self/exe</code>.</p><p><a href="/assets/images/elf_nhas/code1.png" class="popup img-link shimmer"><img src="/assets/images/elf_nhas/code1.png" alt="sshot" loading="lazy"></a></p><p>The malware checks if the environment variable <code class="language-plaintext highlighter-rouge">BUILD_VER</code> is set. This environment variable is used to keep track of a port number, subsequently created (if needed) to connect to a local proxy. If the environment variable is not set, it generates a random port number. The malware then creates a thread to establish a tunnel via a proxy. This proxy acts as intermediate infrastructure between the infected host and the attacker-controlled infrastructure, likely serving payloads. The malware binds a socket to localhost (127.0.0.1), connecting to it using the previously generated random port number. By making requests to the local proxy, the requests are forwarded to the external proxy, receiving payload from the target IP address and executing it in memory.</p><p><a href="/assets/images/elf_nhas/proxy_flow.png" class="popup img-link shimmer"><img src="/assets/images/elf_nhas/proxy_flow.png" alt="sshot" loading="lazy"></a></p><p>At a very high level, the architecture of the backdoor looks like this:</p><p><a href="/assets/images/elf_nhas/architecture.png" class="popup img-link shimmer"><img src="/assets/images/elf_nhas/architecture.png" alt="sshot" loading="lazy"></a></p><p>Connections to the remote proxy are implemented in a continuous loop:</p><p><a href="/assets/images/elf_nhas/code3.png" class="popup img-link shimmer"><img src="/assets/images/elf_nhas/code3.png" alt="sshot" loading="lazy"></a></p><p>The remote proxy is hosted at <code class="language-plaintext highlighter-rouge">43.159.18[.]135:2333</code> (a SOCKS5 proxy), and the target infrastructure, likely used as command and control, is <code class="language-plaintext highlighter-rouge">119.42.148[.]187:2443</code>. The use of the SOCKS5 proxy has the objective of hiding the command and control infrastructure behind â€˜benignâ€™ infrastructure.</p><p><a href="/assets/images/elf_nhas/code4.png" class="popup img-link shimmer"><img src="/assets/images/elf_nhas/code4.png" alt="sshot" loading="lazy"></a></p><p>Looking at the function that generates the user name for the proxy authentication on <code class="language-plaintext highlighter-rouge">43.159.18[.]135:2333</code>, we can see following user name format:</p><p><a href="/assets/images/elf_nhas/code5.png" class="popup img-link shimmer"><img src="/assets/images/elf_nhas/code5.png" alt="sshot" loading="lazy"></a></p><p>This code will generate a user name that looks like this: <code class="language-plaintext highlighter-rouge">"ipideatj10011_2952-zone-custom-region-HK-session-" + &lt;10 random characters&gt; + "-sessTime-60"</code>. This suggests that the threat actor is using the <code class="language-plaintext highlighter-rouge">ipidea[.]net</code> residential proxy service provider. In the proxy connection function, we can see that the generated user name is sent together with the <code class="language-plaintext highlighter-rouge">ipideatj10011_2952Aa1024</code> password for authentication purposes.</p><p><a href="/assets/images/elf_nhas/auth.png" class="popup img-link shimmer"><img src="/assets/images/elf_nhas/auth.png" alt="sshot" loading="lazy"></a></p><p>A machine translation of the â€˜Aboutâ€™ section of the <code class="language-plaintext highlighter-rouge">ipidea</code> website suggests that the company was founded in 2019 in Xuzhou, China.</p><p><a href="/assets/images/elf_nhas/ipidea.png" class="popup img-link shimmer"><img src="/assets/images/elf_nhas/ipidea.png" alt="sshot" loading="lazy"></a></p><p>Additional information on the proxy server from Censys confirms the <code class="language-plaintext highlighter-rouge">ipidea</code> suspicion, also showing that the proxy is hosted in Singapore and associated with Tencent ASN AS132203.</p><p><a href="/assets/images/elf_nhas/censys1.png" class="popup img-link shimmer"><img src="/assets/images/elf_nhas/censys1.png" alt="sshot" loading="lazy"></a></p><p><code class="language-plaintext highlighter-rouge">119.42.148[.]187</code>, the command and control, is accessible via SSH, and hosted in Hong Kong.</p><p><a href="/assets/images/elf_nhas/censys2.png" class="popup img-link shimmer"><img src="/assets/images/elf_nhas/censys2.png" alt="sshot" loading="lazy"></a></p><p>After the connection to the remote proxy, a connection to the localhost proxy will monitor if payload is received for execution. The function that connects to the local proxy takes 2 arguments, 127.0.0.1 and the random port number.</p><p><a href="/assets/images/elf_nhas/loc_prox_fcall.png" class="popup img-link shimmer"><img src="/assets/images/elf_nhas/loc_prox_fcall.png" alt="sshot" loading="lazy"></a></p><p>It then creates a socket and connects to it. A memory file descriptor is created, which is subsequently used for code execution in memory. A loop is implemented to receive payload, setting 104857600 bytes as the payload size limit. The received payload is written to the recently created memory file descriptor.</p><p><a href="/assets/images/elf_nhas/down_exec1.png" class="popup img-link shimmer"><img src="/assets/images/elf_nhas/down_exec1.png" alt="sshot" loading="lazy"></a></p><p>Once the complete payload has been received, the backdoor uses the <code class="language-plaintext highlighter-rouge">lseek</code> syscall to reposition the file offset in the memory file descriptor to the beginning of the file. The malware then creates a new fork, reusing the <code class="language-plaintext highlighter-rouge">kworker/0:0</code> process name for the child process, executing the payload in the memory file descriptor with the <code class="language-plaintext highlighter-rouge">execveat</code> syscall.</p><p><a href="/assets/images/elf_nhas/down_exec2.png" class="popup img-link shimmer"><img src="/assets/images/elf_nhas/down_exec2.png" alt="sshot" loading="lazy"></a></p><h1 id="nhas-reverse-ssh">NHAS Reverse SSH</h1><p>Pivoting on the command and control IP address, <code class="language-plaintext highlighter-rouge">119.42.148[.]187</code>, I found another ELF (28096799c02d198149a7de0e7d6001554fc0c0907a4cfff5fcfa29f8cd93a4c3) contacting the same IP address.</p><p><a href="/assets/images/elf_nhas/pivot.png" class="popup img-link shimmer"><img src="/assets/images/elf_nhas/pivot.png" alt="sshot" loading="lazy"></a></p><p>The first VirusTotal upload date of this file, April 14, 2025, is very close to the first upload time of the previous backdoor. Upload locations include the UK and Turkey.</p><p><a href="/assets/images/elf_nhas/upload_nhas.png" class="popup img-link shimmer"><img src="/assets/images/elf_nhas/upload_nhas.png" alt="sshot" loading="lazy"></a></p><p>A quick review of this executable shows that it was created in Golang, compiled for Linux, and it is a publicly available project that implements a reverse shell via native SSH syntax. The project is available in the following <a href="https://github.com/NHAS/reverse_ssh">GitHub</a> repository. This reverse_ssh client is configured with the same C2 and port as the backdoor described before: <code class="language-plaintext highlighter-rouge">119.42.148[.]187:2443</code></p><p><a href="/assets/images/elf_nhas/nhas_c2.png" class="popup img-link shimmer"><img src="/assets/images/elf_nhas/nhas_c2.png" alt="sshot" loading="lazy"></a></p><h1 id="iocs">IOCs:</h1><ul><li>ELF
