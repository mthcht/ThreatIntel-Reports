	<div class="entry-content">
		<h3 class="tags">
                            <a class="post-category" href="https://expel.com/blog/category/mdr/">
                    MDR</a>
                                        ·  14  MIN READ
                        				·  KYLE PELLETT AND ANDREW JERRY											·  AUG 25, 2022			                            ·  TAGS:
                                     <a href="https://expel.com/blog/tag/phishing/">
                        <span>Phishing</span>
                    </a>
                                    </h3>
		<p>The &#8220;Great Resignation&#8221; has recruiters working overtime scouring LinkedIn resumés for potential candidates. Unfortunately, some of these resumés are posted by bad actors taking advantage of the situation.</p>
<p>With a new twist on the <a href="https://malpedia.caad.fkie.fraunhofer.de/details/js.more_eggs" target="_blank" rel="noopener">MORE_EGGS</a> family of malware, attackers are throwing their names in the ring by <strong>submitting poisoned resumés to job recruiters</strong>. The Expel SOC recently spotted a deployment of this technique. The victim&#8217;s computer was infected and the malware payload tried to exfiltrate data within a few minutes.</p>
<h2>How we spotted our initial lead</h2>
<p>So, to be honest, malware sometimes acts so quickly that multiple alerts sound before one of our analysts can start the triage process. As you’d imagine, we’re automatically suspicious when we see multiple alerts fire for the same activity. It tells us that something strange is happening.</p>
<p>In this case, we received seven unique Microsoft Defender for Endpoint alerts within a few seconds for activity that clearly (for reasons explained below) resembled malicious code execution. This tipped our SOC analysts to an attack that was well under way — action to contain the host was needed urgently.</p>
<p>After this type of malware gains <a href="https://attack.mitre.org/tactics/TA0001/" target="_blank" rel="noopener">initial access</a> — even if partially blocked by existing security controls — the attack can spread quickly and deploy <a href="https://attack.mitre.org/techniques/T1059/" target="_blank" rel="noopener">code execution</a>, <a href="https://attack.mitre.org/tactics/TA0005/" target="_blank" rel="noopener">defense evasion</a>, and <a href="https://attack.mitre.org/tactics/TA0011/" target="_blank" rel="noopener">command and control</a> techniques (in this case the answer was D — all of the above).</p>
<p>This is why a detection strategy that covers all parts of the <a href="https://attack.mitre.org/" target="_blank" rel="noopener">MITRE ATT&amp;CK</a> framework is so important. In this case, Defender for Endpoint caught the use of <a href="https://attack.mitre.org/techniques/T1220/" target="_blank" rel="noopener">XSL Script Processing</a> first.</p>
<p>Cybersecurity is sometimes a battle of humans vs computers, and humans have the disadvantage with respect to time. A lot can happen in one “computer second,” and tech like the Expel Workbench™ and Ruxie™ help level the field by transforming alert data into intel our SOC analysts can quickly respond to while an attack is under way. (More on how we use Defender’s features to our advantage <a href="https://expel.com/blog/windows-defender-atp-our-two-cents/" target="_blank" rel="noopener">here</a>.)</p>
<p>Let&#8217;s take a look at one of several Microsoft Defender for Endpoint alerts we received, how the Expel Workbench helped guide our analysts to find important information quickly, and how we inferred that this attack was in progress.</p>
<p><img decoding="async" class="aligncenter wp-image-12309 size-full" src="data:image/svg+xml,%3Csvg%20xmlns='http://www.w3.org/2000/svg'%20viewBox='0%200%201308%201133'%3E%3C/svg%3E" alt="Microsoft Defender for Endpoint alerts on Expel Workbench; part 1." width="1308" height="1133" data-lazy-src="https://expel.com/wp-content/uploads/2022/08/ProcessArgs_1_page2.png" /><noscript><img decoding="async" class="aligncenter wp-image-12309 size-full" src="https://expel.com/wp-content/uploads/2022/08/ProcessArgs_1_page2.png" alt="Microsoft Defender for Endpoint alerts on Expel Workbench; part 1." width="1308" height="1133" /></noscript></p>
<p><img decoding="async" class="aligncenter wp-image-12310 size-full" src="data:image/svg+xml,%3Csvg%20xmlns='http://www.w3.org/2000/svg'%20viewBox='0%200%201308%20898'%3E%3C/svg%3E" alt="Microsoft Defender for Endpoint alerts on Expel Workbench; part 2." width="1308" height="898" data-lazy-src="https://expel.com/wp-content/uploads/2022/08/ProcessArgs_2_page2.png" /><noscript><img decoding="async" class="aligncenter wp-image-12310 size-full" src="https://expel.com/wp-content/uploads/2022/08/ProcessArgs_2_page2.png" alt="Microsoft Defender for Endpoint alerts on Expel Workbench; part 2." width="1308" height="898" /></noscript></p>
<p>Can you spot the evil here? Here’s what we saw in the recent process activity:</p>
<ul>
<li>We see <strong>regsvr32</strong> attempting to execute <strong>42981.ocx</strong>, which is similar to <a href="https://threatpost.com/cybercriminals-windows-utility-regsvr32-malware/178333/" target="_blank" rel="noopener">a technique used by malware (such as QBot and Lokibot</a>). This is a pretty good giveaway that some malicious code has been executed; it&#8217;s written this <strong>42981.ocx</strong> file to disk, and has now called <strong>regsvr32</strong> to run whatever code lies within this DLL file.</li>
<li>The process arguments of <strong>cmd.exe</strong> are <a href="https://attack.mitre.org/techniques/T1027/" target="_blank" rel="noopener">heavily obfuscated</a>, an indication of an attacker trying to evade detection. One thing that isn’t obfuscated is <strong>johndoe[.]com/kbvbskrvf</strong>, a likely suspect for a command and control IOC.</li>
<li>This alert is looking for <a href="https://attack.mitre.org/tactics/TA0007/" target="_blank" rel="noopener">discovery</a> activity or “Suspicious sequence of exploration activities.” We see this in the command <strong>cmd /v /c nltest /trusted_domains</strong> outputting to a text file in a temporary directory, which is consistent with identifying domains trusted by this host — quite unusual if you ask us.</li>
<li><strong>msxsl.exe</strong> is a deprecated XML parsing tool with a <a href="https://lolbas-project.github.io/lolbas/OtherMSBinaries/Msxsl/" target="_blank" rel="noopener">well documented</a> use case for <a href="https://attack.mitre.org/techniques/T1220/" target="_blank" rel="noopener">executing code and bypassing application controls</a> — here we see it trying to run an obscurely named text file.</li>
<li>We also observe <strong>wmic</strong> creating the process <strong>ie4uinit.exe -basesettings</strong>. This is another <a href="https://lolbas-project.github.io/" target="_blank" rel="noopener">LOLBAS</a> (living off the land binary, script, or library) like <strong>msxsl.exe</strong> that can <a href="https://lolbas-project.github.io/lolbas/Binaries/Ie4uinit/" target="_blank" rel="noopener">easily execute code</a> because it can execute commands from a specially prepared <strong>ie4uinit.inf</strong> file.</li>
</ul>
<p>Okay, so a lot of bad stuff going on…and so far, not a lot of answers to how this happened. At this point, we declared an incident, notified our customer, and sent them remediation actions to contain the host and block communications with <strong>johndoe[.]com</strong></p>
<p><em><span style="font-size: 0.8em;">(Side note: This is not the real C2 we observed, but in the interest of protecting the anonymity of the user the attackers impersonated, we refer to them as johndoe for this blog.)</span></em></p>
<h2>Identifying the root cause</h2>
<p>The next question we wanted to answer: How did this malware infection get here?</p>
<p>We used the customer’s EDR tool to review the timeline and walk back through the chain of events that ultimately led us to an event involving Outlook.exe.</p>
<p><code class="inline-code">OUTLOOK.EXE opened the http link hxxps://www.linkedin[.]com/e/v2?e=-1swgqb-l437ev7b-v3&amp;lipi=urn%3Ali%3Apage%3Aemail_email_jobs_new_applicant_01%3Bgo6DX7fyT96rJM8b2IE8Fw%3D%3D&amp;t=plh&amp;ek=email_jobs_new_applicant_01&amp;li=0&amp;m=email_jobs_new_applicant&amp;ts=job_posting_download_resume&amp;urlhash=quvr&amp;url=https%3A%2F%2Fwww%2Elinkedin%2Ecom%2Ftalent%2Fapi%2FtalentResumes%3FapplicationId%3D10266114276%26tokenId%3D2155098668%26checkSum%3DAQGkxf8BsoxNsmZdzes9P0qm-HMeqGo9oXk</code></p>
<p>The user clicked on a link in an email from a legitimate sender to a legitimate domain; based on the requested resource, it appears they were seeking a resumé for a job posting. This is interesting for a couple of reasons.</p>
<ul>
<li>The attackers evaded inbox malspam detection using a legitimate email sender</li>
<li>The document is likely expected, based on a job posting created by the targeted user</li>
<li>The link in the email also appears legitimate</li>
</ul>
<p>Unfortunately, our target still fell prey to a malicious phishing document. So what happens if the victim clicks through to download the resumé from LinkedIn?</p>
<p>To find out, we followed the trail and discovered a PDF crafted to present the viewer with an error. The error is actually an attempt to lure the victim to an unsafe site where they can download General-Manager-resumé.docx (the file is presented as a Word document).</p>
<p><img decoding="async" class="aligncenter wp-image-12307 size-full" src="data:image/svg+xml,%3Csvg%20xmlns='http://www.w3.org/2000/svg'%20viewBox='0%200%201308%20835'%3E%3C/svg%3E" alt="Error message phishing attempt example." width="1308" height="835" data-lazy-src="https://expel.com/wp-content/uploads/2022/08/Fakedoc_page4.png" /><noscript><img decoding="async" class="aligncenter wp-image-12307 size-full" src="https://expel.com/wp-content/uploads/2022/08/Fakedoc_page4.png" alt="Error message phishing attempt example." width="1308" height="835" /></noscript></p>
<p>Of course, this is suspicious to us because we know what happens. But an everyday user recruiting from LinkedIn has probably seen resumés that aren’t compatible with their software. This seems to be what the attackers are counting on. Notably, the domain johndoe[.]com aligns with what the recruiter expects to see based on the applicant&#8217;s name. (It was later discovered that the victim was in fact a recruiter and wasn&#8217;t aware of a problem with their host after following this funnel.)</p>
<h2>What happened to the host?</h2>
<p>So what happens when the user clicks on the .docx link? Well, as it turns out, a bunch of things (before the user is finally presented with a word document). First of all, the file that lands on the victim’s disk is actually a zip archive by the same name 一 General Manager Resume 1.zip. Once the zip is written to disk, we immediately see it create John Doe CV.lnk.</p>
<p>At this point we see a familiar code execution from one of our alerts:</p>
<p>Obfuscated<br />

<p>Unfortunately, we were unable to acquire any of the files we described. However, given the behaviors performed on the host we were able to tell the story of how a LinkedIn resumé phishing document resulted in a MORE_EGGS backdoor.</p>
<p>Even without acquiring the file, our analysis of the activity aligns with the financially motivated cybercrime gangs FIN6, Evilnum, or the Cobalt Group. It’s difficult to attribute activity to a specific group, but we saw LinkedIn used in 2021 to deliver MORE_EGGS ー with one key difference. The first iteration of threat groups harnessing LinkedIn for this purpose was an inverse of the victim-attacker relationship. Instead of recruiters expecting resumés, the FIN6 group was posing as employers and sending fake job offers to their victims over LinkedIn. Based on their prior use of LinkedIn, it&#8217;s quite possible this is the work of FIN6 or a copycat.</p>
<p>Either way, credit should be given where due. Financially motivated threat actors aren’t playing around and the victim user this article was based around wasn’t aware that downloading a resumés from LinkedIn left a backdoor on their machine.</p>

<p><strong>Resilience:</strong> Even though we detected and reported this incident quickly, the bottom line is that malicious code executed on one of our customer-managed devices on their network. Whenever we can directly point to environment controls to enable defenders or disrupt attackers, we include them in the incident findings report. In this incident we provided the customer with the following resilience actions:</p>
<p>Disrupt attackers:</p>
<ul>
<li>Phishing education for users, specifically from trusted sources (LinkedIn).</li>
<li>Configure Jscript (.js, .jse), Windows Scripting Files (.wsf, .wsh) and HTML for application (.hta) files to open with Notepad.
<ul>
<li>By associating these file extensions with Notepad you mitigate common remote code execution techniques. Note that PowerShell files (.ps1) already open by default in Notepad.</li>
</ul>
</li>
</ul>
<p>Enable Defenders:</p>
<ul>
<li>Increase visibility into PowerShell activity by taking advantage of logging capabilities. Module and ScriptBlock logging provide greater visibility into potential PowerShell attacks.
<ul>
<li>Good: Ensure PowerShell 3.0 (at least) is installed on all Windows systems and enable PowerShell Module logging.</li>
<li>Better: Ensure PowerShell 5.0 (at least) is installed on all Windows systems and enable PowerShell ScriptBlock logging and transcription logging.</li>
<li>Best: Ensure PowerShell 5.0 (at least) is installed on all Windows systems, enable PowerShell ScriptBlock logging and transcription logging; also make sure Microsoft-Windows-PowerShell%4Operational.evtx is at least 1 GB in size on all systems to aid in an investigation.</li>
</ul>
</li>
</ul>
