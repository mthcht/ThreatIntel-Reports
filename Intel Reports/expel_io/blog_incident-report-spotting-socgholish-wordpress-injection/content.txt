    			<h3>EXPEL BLOG</h3>
		<h1 class="entry-title">Incident report: Spotting SocGholish WordPress injection</h1>			</header>

	
			<div class="post-thumbnail">
				<img width="1100" height="244" src="https://expel.com/wp-content/uploads/2021/2021/07/hero-1100x244-expel-incident-report-072221.png" class="attachment-post-thumbnail size-post-thumbnail wp-post-image" alt="alt=&quot;&quot;" decoding="async" fetchpriority="high" srcset="https://expel.com/wp-content/uploads/2021/2021/07/hero-1100x244-expel-incident-report-072221.png 1100w, https://expel.com/wp-content/uploads/2021/2021/07/hero-1100x244-expel-incident-report-072221-300x67.png 300w, https://expel.com/wp-content/uploads/2021/2021/07/hero-1100x244-expel-incident-report-072221-1024x227.png 1024w, https://expel.com/wp-content/uploads/2021/2021/07/hero-1100x244-expel-incident-report-072221-768x170.png 768w, https://expel.com/wp-content/uploads/2021/2021/07/hero-1100x244-expel-incident-report-072221-295x65.png 295w, https://expel.com/wp-content/uploads/2021/2021/07/hero-1100x244-expel-incident-report-072221-500x111.png 500w" sizes="(max-width: 1100px) 100vw, 1100px" />			</div><!-- .post-thumbnail -->

		
	<div class="entry-content">
		<h3 class="tags">
                            <a class="post-category" href="https://expel.com/blog/category/rapid-response/">
                    Rapid response</a>
                                        ·  5  MIN READ
                        				·  TYLER FORNES, RYAN GOTT, KYLE PELLETT AND EVAN REICHARD											·  JUL 22, 2021			                            ·  TAGS:
                                     <a href="https://expel.com/blog/tag/tech-stack/">
                        <span>tech stack</span>
                    </a>
                                    / <a href="https://expel.com/blog/tag/threat-landscape/">
                        <span>threat landscape</span>
                    </a>
                                    </h3>
		<p>Earlier this week, our SOC stopped a ransomware attack at a large software and staffing company. The attackers compromised the company’s WordPress CMS and used the <a href="https://www.menlosecurity.com/blog/increase-in-attack-socgholish/">SocGholish framework</a> to trigger a drive-by download of a Remote Access Tool (RAT) disguised as a Google Chrome update.</p>
<p>In total, four hosts downloaded a malicious Zipped JScript file that was configured to deploy a RAT, but we were able to stop the attack before ransomware deployment and help the organization remediate its WordPress CMS.</p>
<p>We’ll walk you through what happened, how we caught it, and provide recommendations on how to secure your WordPress CMS. We also hope that this story is a good reminder of the power of <a href="https://expel.com/blog/triage-windows-endpoints-asking-right-questions/">asking the right investigative questions</a>.</p>
<h2>How we spotted our initial lead</h2>
<p>Around 07:00 UTC ( that’s 3:00 am ET), our SOC received an EDR alert for suspicious Windows Script Host (WSH) activity on one Windows 10 host.</p>
<p>The TL;DR is an employee double clicked a Zipped JScript file named “Chrome.Update.js” and EDR blocked execution.</p>
<p>Here’s what we were able to infer from our initial lead into the activity:</p>
<ul>
<li>This doesn’t look like legitimate Google Chrome update activity</li>
<li>Possible “fake update” activity delivered via Zipped JScript file</li>
<li>The activity is only on this host, not prevalent, and unlikely to be a false positive</li>
<li>The WSH process spawned from Windows Explorer, suggesting the employee double-clicked the JScript file versus part of an exploit-chain</li>
</ul>
<p style="padding-left: 40px;"><img decoding="async" class="alignnone wp-image-9078 size-full" src="data:image/svg+xml,%3Csvg%20xmlns='http://www.w3.org/2000/svg'%20viewBox='0%200%201308%20669'%3E%3C/svg%3E" alt="An EDR alert for suspicious Windows Script Host activity." width="1308" height="669" data-lazy-src="https://expel.com/wp-content/uploads/2021/2021/07/Initial-lead_-suspicious-WSH-process-activity_.png" /><noscript><img decoding="async" class="alignnone wp-image-9078 size-full" src="https://expel.com/wp-content/uploads/2021/2021/07/Initial-lead_-suspicious-WSH-process-activity_.png" alt="An EDR alert for suspicious Windows Script Host activity." width="1308" height="669" /></noscript><em>Initial lead: suspicious WSH process activity</em></p>
<p>As part of our alert triage process, we did a quick check to make sure we weren’t seeing the WSH activity anywhere else in the environment.</p>
<p>We weren’t. And EDR blocked the activity.</p>
<p>Case closed? Nope.</p>
<p>The quality of your SOC investigations is rooted in the questions you ask. There are two very important questions we needed to answer before calling it “case closed:”</p>
<ol>
<li>What does the JScript file do?</li>
<li>How did the Zipped JScript file get there?</li>
</ol>
<p>To figure out what the JScript file does, we grabbed a copy of the Zipped JScript file and submitted it to our internal sandbox (we use VMRay at Expel).</p>
<p>The JScript file did the following at runtime:</p>
<ul>
<li>Contacted command-and-control servers hosted at [.]services[.]accountabilitypartner[.]com (195.189.96.41) and [.]drpease[.]com</li>
<li>Opened an HTTP POST request for /pixel.png on TCP port 443</li>
<li>Delivery mechanism consistent with potential SocGholish framework activity</li>
</ul>
<p>Given this info, it’s our opinion that “/pixel.png” is likely a second stage payload. We were unable to acquire a copy of “/pixel.png” for further analysis, but &#8230;</p>
<p>Bottom line: it’s bad.</p>
<p>Now we needed to figure out how the Zipped JScript file got there.</p>
<p>This is where the story gets interesting.</p>
<h2>How we investigated the SocGholish WordPress injection</h2>
<p>We needed to know how the Zipped JScript file got onto the host computer. It would’ve been easy to assume, “Okay, the Zipped JScript file was likely delivered via phishing and EDR is blocking the activity, so we’ll block the C2 and move on.”</p>
<p>“<a href="https://giphy.com/gifs/sport-corso-3o6fIUOohyg4yegu1G">Not so fast, my friend.</a>” &#8211; Lee Corso</p>
<p>Using EDR live response features, we acquired a copy of the employee’s Google Chrome browser history as it could potentially contain evidence we needed to determine how the Zipped JScript file got there. The host in question is a Windows machine, so we grabbed a copy of C:Users\AppDataLocalGoogleChromeUser DataDefaultHistory” and reviewed it using internal tools. You can parse the History .db files using SQLite as well.</p>
<p>Sure enough, Google Chrome history recorded that “Chrome.Update.js” was downloaded after visiting a URL hosted on the company’s WordPress CMS. The company’s WordPress CMS was likely compromised, resulting in delivery of “fake updates” that deploy the SocGholish RAT. The company’s WordPress CMS is publicly accessible, so anyone visiting the site could potentially be compromised.</p>
<p>At this point in our investigation, we declared a critical incident, notified our customer, and in parallel escalated our on-call procedure to bring in additional cavalry to aid in the investigation.</p>
<h2>Our response and remediation efforts</h2>
<p>Google Chrome history contained evidence to suggest that the malicious Zipped JScript file was downloaded after visiting a webpage on the company’s WordPress site. We let our customer know that there was evidence to suggest their WordPress site was compromised and to invoke their internal Incident Response plan. We also armed the customer with information about command-and-control servers and advised them to implement blocks.</p>
<p>Adding to the excitement, as the late night hours turned into early morning hours on the East Coast, our SOC started to receive additional EDR alerts for deployment of the malicious Zipped JScript on additional Windows 10 hosts.</p>
<p>EDR blocked that activity as well, but we needed to get a handle on the WordPress situation quickly. Anytime we’d see a download of the Zipped JScript on a new host, we’d repeat our process to establish how the file got there. In each case the Zipped JScript file was downloaded after visiting the company’s WordPress site.</p>
<p>But it turned out that multiple pages on the site were compromised, not just one. This context was super important.</p>
<p>For situational awareness, we did a quick check and noticed the company was running an older version of WordPress, 5.5.3. We didn’t have endpoint visibility into the WordPress server as it was hosted by a third party. If we did, we would have wanted to establish when and how the site was compromised.</p>
<p>We inferred that the attacker likely exploited a vulnerability in a WordPress plugin or WordPress 5.5.3. We grabbed source code of any page that was recorded as triggering a drive-by-download and got to work.</p>
<p>We almost immediately spotted a malicious inline script on every page that triggered a drive-by download:</p>
<p style="padding-left: 40px;"><img decoding="async" class="alignnone wp-image-9079 size-full" src="data:image/svg+xml,%3Csvg%20xmlns='http://www.w3.org/2000/svg'%20viewBox='0%200%201308%20346'%3E%3C/svg%3E" alt="Malicious inline script deployed to multiple pages on the company’s WordPress site." width="1308" height="346" data-lazy-src="https://expel.com/wp-content/uploads/2021/2021/07/Malicious-inline-script-deployed-to-multiple-pages-on-the-companys-WordPress-site_.png" /><noscript><img decoding="async" class="alignnone wp-image-9079 size-full" src="https://expel.com/wp-content/uploads/2021/2021/07/Malicious-inline-script-deployed-to-multiple-pages-on-the-companys-WordPress-site_.png" alt="Malicious inline script deployed to multiple pages on the company’s WordPress site." width="1308" height="346" /></noscript><em>Malicious inline script deployed to multiple pages on the company’s WordPress site</em></p>
<p>We let the customer know about our findings and then turned our attention towards decoding and deobfuscating the script.</p>
<p>Most of the obfuscation consisted of base64 encoded functions and strings. With the help of the Chrome DevTools Console, we stepped through the obfuscated script and eventually landed on the following:</p>
<p style="padding-left: 40px;"><img decoding="async" class="alignnone wp-image-9080 size-full" src="data:image/svg+xml,%3Csvg%20xmlns='http://www.w3.org/2000/svg'%20viewBox='0%200%201308%201107'%3E%3C/svg%3E" alt="Decoded: malicious inline script deployed to multiple WordPress pages." width="1308" height="1107" data-lazy-src="https://expel.com/wp-content/uploads/2021/2021/07/Decoded_-malicious-inline-script-deployed-to-multiple-WordPress-pages_.png" /><noscript><img decoding="async" class="alignnone wp-image-9080 size-full" src="https://expel.com/wp-content/uploads/2021/2021/07/Decoded_-malicious-inline-script-deployed-to-multiple-WordPress-pages_.png" alt="Decoded: malicious inline script deployed to multiple WordPress pages." width="1308" height="1107" /></noscript><em>Decoded: malicious inline script deployed to multiple WordPress pages</em></p>
<p>From the decoded script above, you’ll see that to trigger the drive-by download, the user must be referred to the site (not referred from the same site) and be running Windows. The Zipped JScript was served from notify.aproposaussies[.]com (179.43.169.30). At the time of writing, only Kaspersky (1/87) has flagged the domain as malicious on VirusTotal.</p>
<p>Everything is not fine.</p>
<p>We now understand what’s happening.</p>
<p>At this point, the customer was already in the process of removing the malicious inline scripts and updating to the most recent version of WordPress.</p>
<p>We did one last check of the environment to make sure no additional hosts downloaded the evil Zipped JScript file, checked to make sure that no hosts were talking to known C2 servers, and that no other malicious processes had executed. We asked the right questions and in doing so, figured out what happened.</p>
<p>A quick recap:</p>
<p>The attackers likely used the “<a href="https://www.darkreading.com/attacks-breaches/-socgholish-attack-framework-powers-surge-in-drive-by-attacks">SocGholish</a>” framework to inject a malicious script into multiple pages on the company’s WordPress site by exploiting a vulnerability in a WordPress plugin or WordPress 5.5.3. If an employee navigated to a compromised web page from a device running a Windows OS, an obfuscated inline script triggered a drive-by download of a ZIP file with an embedded Windows JScript file. The malicious JScript file was configured to enable remote access to infected hosts by communicating with command-and-control (C2) servers hosted on legitimate compromised infrastructure. That remote access is then typically used to deploy variants of the WASTEDLOCKER family of ransomware.</p>
<h2>Lessons learned and tips to prevent similar incidents</h2>
<p>WordPress security and its ecosystem has improved over the years, but it’s still an attack vector. Keep up to date on patches, but also:</p>
<ul>
<li>Run trusted and well-known WordPress plugins. These tend to have had more scrutiny and more focus on security.</li>
<li>Follow a WordPress hardening guide or install a WordPress security plug-in. There are many, so choose one that is right for you.</li>
<li>Explore implementing or updating your website <a href="https://developer.mozilla.org/en-US/docs/Web/HTTP/CSP">Content Security Policy</a> to block malicious scripts.</li>
<li>MFA everything and all users.</li>
<li>Lock down your dev and staging instances, too (including adding MFA). You need to control the entire chain of the website, not just the final site.</li>
<li>If a third party hosts your WordPress site, have all the contact info and recovery info needed in case of an incident.</li>
<li>Run an <a href="https://info.expel.io/oh-noes">IR tabletop exercise</a> where the initial entry point is your WordPress site.</li>
</ul>
<p>Remember, the quality of your SOC investigations is rooted in the questions you ask.</p>
<p>If we didn’t answer, “How did it get there?” we would have missed a huge finding that the company’s WordPress site was compromised, resulting in drive-by downloads.</p>
