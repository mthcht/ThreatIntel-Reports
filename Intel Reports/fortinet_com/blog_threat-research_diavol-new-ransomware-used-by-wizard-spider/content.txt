  <p><b><a href="https://www.fortinet.com/fortiguard/labs.html?utm_source=blog&amp;utm_medium=campaign&amp;utm_campaign=FortiGuardLabs">FortiGuard Labs</a> Threat Research Report</b></p>
<p><b>Affected Platforms:               </b>Windows<br />
<b>Impact:</b>                                   Data encryption, Data destruction<br />
<b>Threat Severity:</b>                     Critical</p>
<h2>Diavol Introduction </h2>
<p>At the beginning of June, <a href="https://www.fortinet.com/products/endpoint-security/fortiedr.html?utm_source=blog&amp;utm_campaign=fortiedr">FortiEDR</a> prevented a ransomware attack that had targeted one of our customers. After successfully stopping the attack, we were able to isolate two suspicious files that, at the time, were not found on VirusTotal: <i>locker.exe</i> and <i>locker64.dll</i>. In the timeline of the attack, locker.exe was deployed a day before locker64.dll.</p>
<p>While we were able to identify locker64.dll to be a Conti (v3) ransomware, locker.exe appeared to be entirely different. So, let’s say hello to a new ransomware family.</p>
<p>In this blog, we’ll dive into the inner workings of Diavol and its possible attribution to the criminal group known as Wizard Spider.</p>
<h2>First Encounter with Diavol</h2>
<p>The ransomware drops a ransom note in a text format in every folder it goes over, as can be seen in figure 1.</p>

<div class="cmp cmp-text aem-GridColumn aem-GridColumn--default--12">
  <p>As part of a rather unique encryption procedure, Diavol operates using user-mode Asynchronous Procedure Calls (APCs) without a symmetric encryption algorithm. Usually, ransomware authors aim to complete the encryption operation in the shortest amount of time. Asymmetric encryption algorithms are not the obvious choice as they significantly slower than symmetric algorithms.</p>
<h2>Technical Analysis</h2>
<p><i>locker.exe</i> is a 32-bit executable compiled with Microsoft Visual C/C++ Compiler. The timestamp 2021-04-30 15:58:15 on the file supports the hypothesis that this ransomware is relatively new.</p>
<p>Upon execution, Diavol starts by checking the command line arguments:</p>
<p style="margin-left: 40.0px;">&quot;<i>-p</i>&quot;: path to a file with a list of paths to scan first for files.<br />
&quot;<i>-log</i>&quot;: path to a log file.<br />
&quot;<i>-m</i><span>&quot;: mode: </span><i>net</i><span> or </span><i>local</i>.<br />
             net<b> </b>- encrypt network shares only.<br />
             local<b> </b>- encrypt local drives only and ignore network shares.<br />
&quot;<i>-h</i>&quot;: path to a file that contains specific hosts (names and IPs) to enumerate for shares.<br />
&quot;<i>-s</i>&quot;: <a title="IP address" href="https://www.fortinet.com/resources/cyberglossary/what-is-ip-address?utm_source=blog&amp;utm_medium=blog&amp;utm_campaign=blog-what-is-ip-address">IP address</a> that the initial register message will be sent to. Overrides the hardcoded address.</p>
<p>The following command-line parameters were observed in the incident:</p>
<p style="margin-left: 40.0px;"><i>-p &quot;C:\b.txt&quot; -m local -log &quot;C:\programdata\log.txt&quot;</i></p>
<p>The log file lists files that were encrypted.</p>
<p>The authors didn’t remove the path to the debugging information file, unfortunately it didn’t divulge sensitive or incriminating details about the ransomware authors:</p>
<p style="margin-left: 40.0px;"><i>D:\Development\Master\onion\locker.divided\LockMainDIB\Release\LockMainDIB.pdb</i></p>
<p>While Diavol is not packed nor has any anti-disassembly tricks, it does use an interesting anti-analysis technique to obfuscate its code. Its main routines are kept in bitmap images, which are stored in the PE resource section. Before calling each routine, it copies the bytes from the bitmap to a global buffer that has execute permissions.</p>
<p style="margin-left: 40.0px;"> </p>

  <p>Diavol has 14 different routines stored as bitmaps. They are called in the following order:</p>

<div class="cmp cmp-image aem-GridColumn--default--none aem-GridColumn aem-GridColumn--default--10 aem-GridColumn--offset--default--1">

<div class="cmp cmp-text aem-GridColumn aem-GridColumn--default--12">
  <p><b>· Create an identifier for the victim:</b></p>
<p>The <i>GENBOTID routine </i>creates a unique identifier of the infected machine. It is composed of the following:</p>
<p style="margin-left: 40.0px;">&lt;NetBIOS_computer_name&gt; + &lt;username&gt; + “_W” + &lt;OS major version in hex&gt; + &lt;OS minor version in hex&gt; + &lt;OS build number in hex&gt; + “.” + &lt;random_GUID_bytes in hex&gt;</p>
<p><b>· Initialize configuration:</b></p>
<p>The <i>SHAPELISTS</i> routine copy the hardcoded configuration from the PE’s .data section. The configuration begins with the “<i>STATIC_DATA</i>” string and holds many unicode strings:</p>
<p style="margin-left: 40.0px;">- Base64 encoded RSA public key.<br />
- Server address for the initial registration.<br />
- Group ID for the initial registration.<br />
- List of excluded file extensions, file names and paths.<br />
- List of process names to terminate.<br />
- List of service names to stop.<br />
- List of paths to enumerate files.<br />
- List of filenames to delete.<br />
- Ransom note (in reverse).</p>
<p><b>· Register with the C&amp;C server and update configuration:</b></p>
<p>The <i>REGISTER</i> payload uses the WinINet API to send a request to a server and returns the response status code. Diavol issues the C&amp;C server a POST request to the <i>hxxp://&lt;server_address&gt;/BnpOnspQwtjCA/register</i> URL with the following headers:</p>
<p style="margin-left: 40.0px;"><i>User-Agent: &quot;Agent&quot;<br />
 Content-Type: application/x-www-form-urlencoded; charset=UTF-8</i></p>
<p>The body of the request is<i>:</i></p>
<p style="margin-left: 40.0px;"><i>cid=&lt;unique_victim_id&gt;&amp;group=&lt;group_id_from_config&gt;ip_local1=111.111.111.111&amp;ip_local2=222.222.222.222&amp;ip_external=2.16.7.12</i></p>
<p>If the server returns status success, the ransomware will attempt to get an updated configuration from the C&amp;C server. Since the “<i>-s</i>” command-line parameter was not provided, and as the configured address is 127.0.0.1 (localhost), the ransomware didn’t register itself or retrieve configuration updates.</p>
<p>The <i>FROMNET</i> routine sends a request to the server using the WinINet API as well and returns the data from the response. To get an updated configuration and override the hardcoded values, the ransomware sends HTTP GET requests using the same headers as before, to: </p>
<p style="margin-left: 40.0px;"><i>hxxp://173[.]232[.]146[.]118/Bnyar8RsK04ug/&lt;unique_victim_id&gt;/qqq123/&lt;object_name&gt;</i></p>
<p>The following objects are supported:</p>
<p style="margin-left: 40.0px;">- <i>/key</i> - base64 encoded RSA public key.<br />
- <i>/services</i> - list of services to stop.<br />
- <i>/priority</i> - list of paths to scan for files for the first time.<br />
- <i>/ignore</i> - list of file extensions, filenames and paths to exclude.<br />
- <i>/ext</i> - list of file extensions to include.<br />
- <i>/wipe</i> - list of filenames to delete if they are found while enumerating the filesystem.<br />
- <i>/landing</i><span> - ransomware note.</span></p>
<p>Inspecting the network traffic to 173.232.146.118 showed that the HTTP Cookie header contains the string “<i>diavol_session</i>”.</p>
<p> </p>

<div class="cmp cmp-text aem-GridColumn aem-GridColumn--default--12">
  <p><b>· Stop services and processes:</b></p>
<p>To maximize its effect on the target machine and to encrypt as many files as possible, the ransomware terminates running processes that can lock access to valuable files, such as databases, office applications, financial\accounting software, web servers, and virtual machines.</p>
<p><i>SERVPROC</i> terminates services using the Service Control Manager (SCM) API. This API requires administrator permissions, which suggests the attackers are aware of this requirement and have taken appropriate steps beforehand. </p>
<p>The sample attempts to stop the following services:</p>
<p style="margin-left: 40.0px;"><i>sqlservr.exe</i>, <i>sqlmangr.exe</i>, <i>RAgui.exe</i>, <i>QBCFMonitorService.exe</i>, <i>supervise.exe</i>, <i>fdhost.exe</i>, <i>Culture.exe</i>, <i>RTVscan.exe</i>, <i>Defwatch.exe</i>, <i>wxServerView.exe</i>, <i>sqlbrowser.exe</i>, <i>winword.exe</i>, <br />
<i>GDscan.exe</i><span>, </span><i>QBW32.exe</i><span>, </span><i>QBDBMgr.exe</i><span>, </span><i>qbupdate.exe,axlbridge.exe</i><span>, </span><i>360se.exe</i><span>, </span><i>360doctor.exe</i><span>, </span><i>QBIDPService.exe</i><span>, </span><i>wxServer.exe</i><span>, </span><i>httpd.exe</i><span>, </span><i>fdlauncher.exe</i><span>, </span><i>MsDtSrvr.exe</i><span>, </span><i>tomcat6.exe</i><span>, <br />
 </span><i>java.exe</i><span>, </span><i>wdswfsafe.exe</i><span>.</span></p>
<p>In addition, the malware developers don’t check to see if the API calls are successful.</p>

<div class="cmp cmp-text aem-GridColumn aem-GridColumn--default--12">
  <p><i>KILLPR</i> use <i>CreateToolhelp32Snapshot</i>, <i>Process32First</i> and <i>Process32Next</i> APIs to enumerate the running processes in the system.</p>
<p>The sample attempts to terminate the following list of processes:</p>
<p style="margin-left: 40.0px;"><i>DefWatch</i>, <i>ccEvtMgr</i>, <i>ccSetMgr</i>, <i>SavRoam</i>, <i>dbsrv12</i>, <i>sqlservr</i>, <i>sqlagent</i>, <i>Intuit.QuickBooks.FCS</i>, <i>dbeng8</i>, <i>sqladhlp</i>, <i>QBIDPService</i>, <i>Culserver</i>, <i>RTVscan</i>, <br />
<i style="">vmware-usbarbitator64</i><span style="">, </span><i style="">vmware-converter</i><span style="">, </span><i style="">VMAuthdService</i><span style="">, </span><i style="">VMnetDHCP</i><span style="">, </span><i style="">VMUSBArbService</i><span style="">, </span><i style="">VMwareHostd</i><span style="">, </span><i style="">sqlbrowser</i><span style="">, </span><i style="">SQLADHLP</i><span style="">, </span><i style="">sqlwriter</i><span style="">, </span><i style="">msmdsrv</i>,<br />
<i style="">tomcat6</i><span style="">, </span><i style="">QBCFMonitorServicechrome.exe</i><span style="">, </span><i style="">outlook.exe</i><span style="">, </span><i style="">chrome.exe</i><span style="">.</span></p>
<p>The authors of Diavol made some mistakes in their hardcoded configuration. For starters, <i>SERVPROC</i> gets the list of the process names to terminate <b>instead</b> of service names to stop, and vice versa. The services list appears to include items that have nothing to do with services, like &quot;winword.exe&quot;. In the processes list, only the last three items look like actual process names, while even one of them is wrong (&quot;<i>QBCFMonitorServicechrome.exe</i>&quot; looks like a concatenation of “<i>QBCFMonitorService</i>” and “<i>chrome.exe</i>”).</p>
<p><b>· Initialize encryption key:</b></p>
<p><i>RSAINIT</i> initializes the RSA public key used for encryption with the standard WinCrypt API.</p>
<p><b><b>· </b>Find all drives to encrypt:</b></p>
<p><i>ENMDSKS</i> gets all the local drives in the system using <i>GetLogicalDriveStrings</i> API and checks to ensure they are not in the exclusions list. This routine will be skipped if the mode command-line parameter (“<i>-m</i>”) is set to “net”.</p>
<p>The default list of the excluded file extensions, file names and paths is:</p>
<p style="margin-left: 40.0px;"><i>*.exe</i>, <i>*.sys</i>, <i>*.dll</i>, <i>*.lock64</i>, <i>*readme_for_decrypt.txt</i>, <i>*locker.txt</i>, <i>*unlocker.txt</i>, <i>%WINDIR%\</i>, <i>%PROGRAMFILES%\</i>, <i>%PROGRAMW6432%\</i>, <i>%TEMP%\</i></p>
<p><i>SMBFAST</i><b> </b>and <i>SMB</i> routines enumerate available network shares for access.</p>
<p><i>SMBFAST</i> will only be called if the “-h” parameter is present. This routine scans for accessible network shares on specific hosts.</p>
<p><i>SMB</i> scans for accessible network shares on hosts found in the ARP table. Note that this routine will be skipped if the mode command-line parameter (“<i>-m</i>”) is set to “local”.</p>
<p><b><b>· </b>Find files to encrypt:</b></p>
<p><i>FINDFILES</i> traverses the file and directories in a given path. This routine is invoked multiple times:</p>
<p style="margin-left: 40.0px;">- Depending on the “-p” command-line or hardcoded configuration.<br />
- According to <i>ENMDSKS</i> results.<br />
- According to <i>SMBFAST</i> results.<br />
- According to <i>SMB</i><span> results.</span></p>
<p>First, the routine checks to see if the given file or folder is not in the exclusion list. Then the routine determines whether the current item is in the list of files to delete. If not, it queues an APC on the current thread that gets the path to the item as an argument.</p>


</div>

  <p><b>· Prevent recovery by deleting shadow copies:</b></p>
<p><i>VSSMOD </i>drops and executes <i>wscpy.exe</i> in the current working directory, depending on the OS version (Windows Server 2003 or Vista and newer) and architecture (32 or 64-bit). Deleting the shadow copies snapshots is performed using the <i>IVssBackupComponents</i> COM object to call the <i>DeleteSnapshots</i> method.</p>
<p>The dropped binaries are also kept in plaintext in the resource section under &quot;<i>TEXT</i>”. These binaries contained a PDB reference source file that was used to compile the malware binary:<i></i></p>
<p style="margin-left: 40.0px;"><i>D:\Projects\Repository\LockCry.divided\WipeShadowCopies64\RelNoCRT\WipeShadowCopies64.pdb</i></p>
<p style="margin-left: 40.0px;"><i>D:\Projects\Repository\LockCry.divided\WipeShadowCopies64\x64\RelNoCRT\WipeShadowCopies64.pdb</i></p>
<p style="margin-left: 40.0px;"><i>D:\Development\Master\onion\locker.divided\WipeShadowCopies64\RelNoCRT_Win2003\WipeShadowStorageWin2003_32.pdb</i></p>
<p style="margin-left: 40.0px;"><i>D:\Development\Master\onion\locker.divided\WipeShadowCopies64\x64\RelNoCRT_Win2003\WipeShadowStorageWin2003_64.pdb</i></p>
<p><b>· Encryption:</b></p>
<p>As mentioned before, <i>FINDFILES</i> will queue an APC object with a file or directory path. For the APC to get executed, the main function calls <i>SleepEx</i> API in its final steps to set the thread to an alertable state.</p>
<p>The APC routine checks if the argument is directory and will create the “<i>README_FOR_DECRYPT.txt</i>” ransom note regardless of whether the files in the directory were encrypted. If the argument is a file, the <i>ENCDEFILE</i><b> </b>routine will be called.</p>

<div class="cmp cmp-text aem-GridColumn aem-GridColumn--default--12">
  <p>Unlike most other ransomwares, Diavol does not use any symmetric encryption but only RSA to encrypt the files.</p>
<p>ENCDEFILE checks the file size. If it is less than 2,000,000 bytes, then only up to the first 11,700 bytes will be encrypted. If the file size is equal or greater than 2,000,000 bytes, then just the first 1,170,000 bytes will be encrypted.</p>
<p>Each block of 11,700 bytes is split into ten chunks of 117 bytes each, and each chunk is encrypted using <i>CryptEncrypt </i>API. 117 plaintext bytes then become 128 bytes of ciphertext. Therefore, 11,700 bytes become 12,800 bytes following the encryption. Diavol overwrites the file and writes 11,700 encrypted bytes at the chunk’s original offset and appends the 1,100 remaining bytes to the end of the file.</p>
<p>Finally, <i>ENCDEFILE</i> calls <i>MoveFile</i> API and appends a “<i>.lock64</i>” extension to the filename.</p>
<p>Once all APCs are dequeued and completed, the thread returns following the call to <i>SleepEx</i>. </p>
<p><b>· Change the desktop wallpaper</b>:</p>
<p>Before the process terminates, <i>CHNGDESK</i> is invoked.</p>
<p>First, it captures the desktop window and sets the background color to black. It then writes &quot;<i>All your files are encrypted! For more information see “README-FOR-DECRYPT.txt</i>&quot; with <i>DrawText</i> API to a bitmap image and saves it as &quot;<i>encr.bmp</i>&quot; in the public pictures folder. Finally, it changes the desktop wallpaper to the new image using the <i>SystemParametersInfo</i>API with the SPI_SETDESKWALLPAPER flag.</p>

  <h2>Connections to Conti and Egregor</h2>
<p>As Diavol was deployed in conjunction with the Conti ransomware in this attack, albeit on different machines, we tried to see if there’s any correlation between them.</p>
<p>Starting with the command-line parameters, the ones used by Diavol are nearly identical to those of Conti and used for the same functionality: log file, encrypt local drives or network shares, and scan specific hosts for network shares. In addition, Diavol and Conti both operate similarly with asynchronous I/O operations when queuing the file paths for encryption.</p>
<p>There also might be a link between Diavol and Egregor ransomware. Some lines in the ransom note are identical, as can be seen in figure 12, although this is not reliable as it could simply be a red herring that Diavol’s authors planted.</p>
<p>Some have reported a link between Wizard Spider, the threat actor behind Conti, and Twisted Spider, the threat actor behind Egregor. Allegedly, these gangs cooperate on various operations. They are also both notoriously known for double ransoming their victims (data theft and encryption).</p>

  <h2>Summary</h2>
<p>In this blog, we presented a complete analysis of the new ransomware family - Diavol.</p>
<p>Currently, the source of the intrusion is unknown. The parameters used by the attackers, along with the errors in the hardcoded configuration, hint to the fact that Diavol is a new tool in the arsenal of its operators which they are not yet fully accustomed to. </p>
<p>As the attack progressed, we found more Conti payloads named <i>locker.exe</i> in the network, strengthening the possibility the threat actor is indeed Wizard Spider. Despite a few similarities between Diavol, Conti, and other related ransomware, it’s still unclear, however, whether there’s a direct link between them. And, there are a couple of major differences from attacks previously attributed to Wizard Spider &amp; Co., namely:</p>
<p style="margin-left: 40.0px;">- No checks and balances to ensure the payload will not execute on Russian victims.<br />
- No clear evidence of double extortion in the environment was found.</p>
