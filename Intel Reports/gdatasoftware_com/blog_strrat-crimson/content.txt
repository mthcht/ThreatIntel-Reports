href="/blog/bots-botnets" class="nm-article-image-category"> Bots &amp; Botnets </a> </div> <p class="nm-article-teaser">This Java based malware installs RDPWrap, steals credentials, logs keystrokes and remote controls Windows systems. It may soon be capable to infect without Java installed.</p> <hr> <div class="nm-article-content" id="blogpage-content-container"> <div id="c205685" class="fce column large-12 small-12 frame-layout-0 " ><p>Java is not commonly used for malware anymore and its runtime environment is not installed on as many systems as it was in the past. The more it seems surprising when new Java based malware families arise.</p><p>I am an active member of the forum <a href="https://malwaretips.com/" target="_blank" rel="noreferrer">MalwareTips.com</a>. A member of this forum, <a href="https://malwaretips.com/members/upnorth.38832/" target="_blank" rel="noreferrer">upnorth</a>, shared a sample<sup>[2]</sup> to be used for testing Antivirus products. This sample<sup>[2]</sup> caught my attention. It was a Java archive but described as WSHRat. I expected to see either a dropper for a known WSH based RAT or another Adwind variant. I was wrong. This sample<sup>[2]</sup> is a new breed of Java RAT. One that is prepared to not rely on a preinstalled Java Runtime Environment (JRE).</p></div> <div data-magellan-target="c205668" class="small-12 column"> <div class="row"> <div id="c205668" class="fce column large-12 small-12 frame-layout-0 " ><div class="ce-textpic ce-center ce-below"><div class="ce-bodytext"><header class=" "><h2 class=" "> Infection chain overview </h2></header><p>The following sections will describe the infection chain in detail. Here is an overview involving initial infection, intermediate files, unpacking layers and hardcoded downloads by the payload. The numbering of files in the image corresponds to numbers in the <a href="https://www.gdatasoftware.com/blog/strrat-crimson#c205508" target="_top">IOC listing</a> at the bottom of the article.</p></div><div class="ce-gallery row small-up-1 medium-up-1 large-up-1 nm-count-1 "><div class="column column-block"><figure class="image"><img class="image-embed-item " src="/fileadmin/user_upload/Presse/Deutschland/2020/06/JavaRAT_infectionchain2.png" width="1024" height="768" loading="lazy" alt="" /></figure></div></div></div></div> </div> </div> <div data-magellan-target="c205460" class="small-12 column"> <div class="row"> <div id="c205460" class="fce column large-12 small-12 frame-layout-0 " ><div class="ce-textpic ce-center ce-below"><div class="ce-bodytext"><header class=" "><h3 class=" "> Infection chain 1: Spam email with malicious Jar attachment </h3></header><p>The infection starts with a rather ordinary spam email<sup>[1]</sup> that has a malicious attachment named <strong>NEW ORDER.jar</strong><sup>[2]</sup>.</p><p>I found this email via VirusTotal graphs which shows a relationship to our Jar file. It is not clear if the uploader of the email redacted the email body or if the threat actors didn't want to take their time to add any content. It should be noted that Outlook prevents access to email attachments with<strong> .jar</strong> extension. In this case I applied a <a href="https://support.microsoft.com/en-us/help/829982/outlook-blocked-access-to-the-following-potentially-unsafe-attachments" target="_blank" rel="noreferrer">registry hack</a> to have it shown anways.</p></div><div class="ce-gallery row small-up-1 medium-up-1 large-up-1 nm-count-1 "><div class="column column-block"><figure class="image"><img class="image-embed-item thumbnail" src="/fileadmin/user_upload/Presse/Deutschland/2020/06/JavaRAT_emailcontent.png" width="833" height="646" loading="lazy" alt="" /></figure></div></div></div></div> <div id="c205461" class="fce column large-12 small-12 frame-layout-0 " ><div class="ce-textpic ce-center ce-below"><div class="ce-bodytext"><p>The <strong>NEW ORDER.jar</strong><sup>[2]</sup> is a simple dropper. It retrieves a VBScript<sup>[3]</sup> from the resources, saves the script as<strong> bqhoonmpho.vbs</strong><sup>[3]</sup> to the home directory of the user and executes it using <strong>wscript.exe</strong>.</p></div><div class="ce-gallery row small-up-1 medium-up-1 large-up-1 nm-count-1 "><div class="column column-block"><figure class="image"><img class="image-embed-item thumbnail" src="/fileadmin/user_upload/Presse/Deutschland/2020/06/JavaRAT_VBSLoaderJar.png" width="1097" height="640" loading="lazy" alt="" /></figure></div></div></div></div> </div> </div> <div data-magellan-target="c205514" class="small-12 column"> <div class="row"> <div id="c205514" class="fce column large-12 small-12 frame-layout-0 " ><div class="ce-textpic ce-center ce-below"><div class="ce-bodytext"><header class=" "><h3 class=" "> Infection chain 2: VBScript downloads and installs Java for the RAT </h3></header><p>The VBScript<sup>[3]</sup> has a large string in it and uses PowerShell to replace characters in this string. The resulting base64 string is subsequently decoded and executed by PowerShell.</p><p>The unpacked layer is again a VBScript. This script will copy the packed version of itself to <strong>%APPDATA%\edeKbMYRtr.vbs</strong>. It will also download a Java Runtime Environment (see picture below) and add it to the registry. That way it may be prepared to infect systems that don't have Java installed. It even has a built-in check that runs<strong> javaw.exe</strong> with the <strong>-version</strong> parameter to verify that the JRE has the version 1.6, 1.7 or 1.8.</p><p>The email attachment already requires a Java Runtime Environment (JRE) on the system, which means the current infection chain misses the opportunity to work regardless of the JRE installation. If this VBScript is ever shipped with a different initial infection step, it may enable the RAT to work on more systems.</p></div><div class="ce-gallery row small-up-1 medium-up-1 large-up-1 nm-count-1 "><div class="column column-block"><figure class="image"><img class="image-embed-item thumbnail" src="/fileadmin/user_upload/Presse/Deutschland/2020/06/JavaRAT_DownloadJRE.png" width="995" height="547" loading="lazy" alt="" /></figure></div></div></div></div> <div id="c205517" class="fce column large-12 small-12 frame-layout-0 " ><p>The VBScript continues to write the actual payload to <strong>%APPDATA%\ntfsmgr.jar</strong><sup>[4] </sup>and add a RUN key named <strong>ntfsmgr</strong> to the registry that will autorun the dropped Jar<sup>[4] </sup>. The RUN key will use the newly installed JRE if no JRE was present before.</p></div> </div> </div> <div data-magellan-target="c205523" class="small-12 column"> <div class="row"> <div id="c205523" class="fce column large-12 small-12 frame-layout-0 " ><div class="ce-textpic ce-center ce-below"><div class="ce-bodytext"><header class=" "><h3 class=" "> Infection chain 3: Initial payload analysis </h3></header><p>What striked me first about the Jar payload is the package name<strong> strpayload</strong> and one of the dependencies listed in the <strong>MANIFEST.MF</strong> named <strong>system-hook-3.5.jar</strong>.</p><p>A quick search for the library turns up a <a href="https://github.com/kristian/system-hook" target="_blank" rel="noreferrer">GitHub repository by user kristian</a> stating that "Java (low-level) System Hook provides a very light-weight global keyboard and mouse listener for Java". We can already estimate that the malware may use it to log keystrokes.</p><p>We see immediately that the Jar file is obfuscated by Allatori. Upon opening the Main.class we find a URL reference to <strong>hxxp://jbfrost.live/strigoi/lib.zip</strong><br> The URL provides a ZIP bundle of all the dependencies listed in the MANIFEST.MF. The malware will probably not work correctly if this site is down.</p><p>Due to the use of Allatori most of the strings in the Jar file are encrypted with AES. Method arguments and local variables mostly use the name 'a' which creates non-working decompiled Java code. Although Java bytecode saves variable names, it uses indices to reference them. So using the same name several times doesn't cause an error in the bytecode but causes one in decompiled code because it's not clear anymore which variable was referenced by 'a'.</p><p>An example is seen in the image below. The left side shows decompiled Java code with both arguments named 'a'. The right side shows the same method in Java bytecode.
