<h2 class="wp-block-heading" id="h-summary">Summary</h2>



<p>VBA purging is a recent office macro detection evasion technique. It removes the VBA macro&nbsp;<code>PerformanceCache</code>&nbsp;from malicious documents. While the VBA macro source code is only stored in compressed form in Office documents, this&nbsp;<code>PerformanceCache</code>&nbsp;caches the decompressed VBA source code in uncompressed plain text form. Because many security scanning solutions rely on this uncompressed plain text VBA macro source code to be present in order to detect malicious VBA macro code, their detection can be evaded by VBA purging.</p>



<p>This article details a recent spam campaign with malicious office attachments using VBA purging, how the detection evasion works and how Hornetsecurity protects against VBA purging. Evaluating the data of the observed campaign indicates that the campaign was not targeted towards a specific geographic region nor a specific industry sector.</p>



<div style="height:var(--wp--preset--spacing--20);width:100%" aria-hidden="true" class="wp-block-spacer"></div>



<h2 class="wp-block-heading" id="h-background">Background</h2>



<p>Recently, Hornetsecurity has observed malspam delivering Formbook<sup><a href="#h-references">1</a></sup>&nbsp;and AgentTesla<sup><a href="#h-references">2</a></sup>&nbsp;via malicious office attachments using VBA purging.</p>



<p>VBA purging is a technique first described by Didier Stevens<sup><a href="#h-references">3</a></sup>. VBA macros are stored in module streams<sup><a href="#h-references">4</a></sup>&nbsp;of CFBF<sup><a href="#h-references">5</a></sup>&nbsp;files. A stream storing VBA code has two parts, the&nbsp;<code>PerformanceCache</code>&nbsp;and&nbsp;<code>CompressedSourceCode</code>:</p>

<p>The&nbsp;<code>CompressedSourceCode</code>&nbsp;contains the VBA code in compressed form. While not specified in the official documentation, the&nbsp;<code>PerformanceCache</code>&nbsp;generally contains the VBA code in compiled form (P-code) and the uncompressed plain text VBA code.</p>



<p>VBA purging deletes the&nbsp;<code>PerformanceCache</code>&nbsp;by setting its size to 0. Then the VBA code will only be present in compressed form. Because many detection tools rely on the uncompressed plain text VBA code copy in the&nbsp;<code>PerformanceCache</code>, their detection can be evaded.&nbsp;<strong>One detection tool affected is ClamAV a popular open source anti-virus scanner often used on email gateways.</strong></p>



<p>This is interesting for multiple reasons. First, Formbook and AgentTesla are very common malware that can be obtained and operated easily. For example, in 2017 Formbook could be rented for US$ 29 per week, US$ 59 per month, or US$ 99 for three months and US$ 299 for buying it<sup><a href="#h-references">6</a></sup>, while AgentTesla is spread so far that tutorial videos for it exist on YouTube.</p>

<p>This low barrier of entry means that this type of malware is usually send via less sophisticated methods, such as ZIP’ing malware executable and attaching it directly to spam emails, or using publicly available exploits for old vulnerabilities (CVE-2017-0199, CVE-2017-8570, CVE-2017-8759, CVE-2018-8174). In case malicious VBA macros are used the code is usually not very advanced and technically far away from the skill level needed for VBA purging.</p>



<p>However, since September 2020 Hornetsecurity has observed multiple Formbook and AgentTesla malspam campaigns using VBA purging. While the observed malspam is similar to a previous VBA purging campaign observed in the wild by Didier Stevens<sup><a href="#h-references">7</a></sup>&nbsp;the hereafter reported malicious documents are not in the OOXML format and can, hence, not have been created by EPPlus. The hereafter reported documents have the OLE/CFB format.</p>

<p style="margin-right:var(--wp--preset--spacing--10);margin-left:var(--wp--preset--spacing--10)"><code>Loader"68 74 74 70 ..."</code></p>



<p>However, the main difference is that the herein observed malicious documents are not in the OOXML format as the documents outlined by Didier Stevens.</p>

<h3 class="wp-block-heading" id="h-vba-purging">VBA purging</h3>



<p>As&nbsp;<a href="#h-background">outlined earlier</a>, VBA purging is when the&nbsp;<code>PerfomanceCache</code>&nbsp;of a VBA macro document is deleted. You can use Didier Stevens’&nbsp;<code>oledump.py</code>&nbsp;tool to analyze the size of the streams, including the size of the&nbsp;<code>PerfomanceCache</code>. In the following, we see highlighted in red that the&nbsp;<code>PerfomanceCache</code>&nbsp;size of the VBA macro module streams is 0, meaning the&nbsp;<code>PerfomanceCache</code>&nbsp;was purged:</p>

<p>We currently do not know how these documents were created. The VBA purging documents outlined in the article by Didier Stevens<sup><a href="#h-references">7</a></sup>&nbsp;were allegedly created with EPPlus. However, EPPlus can only create documents in the OOXML format.</p>



<p>In one of the maldocs we found a reference to&nbsp;<code>Aviary for Android 4.8.4</code>, a photo editing app for Android. It could be possible that these documents were generated or converted by an Android app which performs VBA purging. Here it is important to note that VBA purging in itself is not malicious. It saves disk storage by deleting a cache, which can be recreated from the&nbsp;<code>CompressedSourceCode</code>&nbsp;data. So there likely exist office software that performs VBA purging as part of optimizing office document file sizes.</p>



<div style="height:var(--wp--preset--spacing--10);width:100%" aria-hidden="true" class="wp-block-spacer"></div>



<h3 class="wp-block-heading" id="h-why-is-this-dangerous">Why is this dangerous?</h3>



<p>Many email gateways use ClamAV as anti-virus scanner. When ClamAV scans an office document, it extracts the individual sub-files of the document. It does so for both modern OOXML documents (which are basically ZIP files) and OLE/CFB documents (which also feature a file system structure inside<sup><a href="#h-references">5</a></sup>). In this process it also extracts the module stream files we&nbsp;<a href="#h-background">mentioned earlier</a>.</p>



<p>It then searches the extracted files using byte patterns, e.g., to detect a malicious Emotet document variant they use the following signature:</p>



<p style="margin-right:0;margin-left:0"><code>Doc.Malware.Emotet-9769220-0;Engine:51-255,Target:2;0&amp;1&amp;2;4174747269627574652056425f4e616d65203d2022413179397a677337686679357a;446f63756d656e745f6f70656e;565f3837317874326e6f656a682e4d3333737137636d686574</code></p>



<p>For better understanding we convert the search patterns into a more readable YARA rule:</p>



<p><code>
rule Doc_Malware_Emotet_9769220_0 {<br>
&nbsp;&nbsp;&nbsp; strings:<br>
&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; $a0 = "Attribute VB_Name = \"A1y9zgs7hfy5z"<br>
&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; $a1 = "Document_open"<br>
&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; $a2 = "V_871xt2noejh.M33sq7cmhet"<br>
&nbsp;&nbsp;&nbsp; condition:<br>
&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; $a0 and $a1 and $a2<br>
}<br>
</code></p>



<p>From this we see that the rule matches on plain text VBA code (the attribute&nbsp;<code>VB_Name</code>&nbsp;that is automatically added by Office at the beginning of the macro, the sub routing name&nbsp;<code>Document_open</code>&nbsp;that instructs office to run this sub routine when the document is opened and a code snipped using two obfuscated variables&nbsp;<code>V_871xt2noejh</code>&nbsp;and&nbsp;<code>M33sq7cmhet</code>).</p>



<p>In a VBA purged document there (highly likely) is no&nbsp;<code>Attribute VB_Name</code>&nbsp;string (nor any of the other strings) in plain text, because only the&nbsp;<code>CompressedSourceCode</code>&nbsp;using a proprietary compression is available. Compression algorithms work by storing repeating patterns in a data stream more efficiently, e.g., the following attributes at the start of VBA code all contain repeating patterns:</p>



<p><code>
Attribute VB_Name = "Lhc713a6y33gome"<br>
Attribute VB_Base = "1Normal.ThisDocument"<br>
Attribute VB_GlobalNameSpace = False<br>
Attribute VB_Creatable = False<br>
[...]
</code></p>



<p>So in compressed form&nbsp;<code>Attribute</code>&nbsp;would only be stored in plain text form once and all further occurrences of it are replaced with a shorter reference, so the previous VBA code snippet would turn into something like:</p>



<p><code>
0001b0d0  41 74 74 72 69 62 75 74  00 65 20 56 42 5f 4e 61  |Attribut.e VB_Na|<br>
0001b0e0  6d 00 65 20 3d 20 22 4c  68 63 00 37 31 33 61 36  |m.e = "Lhc.713a6|<br>
0001b0f0  79 33 33 80 67 6f 6d 65  22 0d 0a 0a 98 08 42 61  |y33.gome".....Ba|<br>
0001b100  73 02 98 31 4e 6f 72 00  6d 61 6c 2e 54 68 69 73  |s..1Nor.mal.This|<br>
0001b110  00 44 6f 63 75 6d 65 6e  74 81 0d 56 47 6c 6f 62  |.Document..VGlob|<br>
0001b120  61 6c 01 b0 10 53 70 61  63 01 6c 46 61 6c 04 73  |al...Spac.lFal.s|<br>
0001b130  65 0c a2 43 72 65 61 74  08 61 62 6c 15 1f 50 72  |e..Creat.abl..Pr|
</code></p>



<p>Hence any rule matching on&nbsp;<code>Attribute</code>&nbsp;(or any other string) would not match anymore as the string is split up, in this example, into&nbsp;<code>Attribut</code>,&nbsp;<code>e VB_Nam</code>, etc., substrings, that are then referenced again later in the data stream in order to save storage space, as storing the reference is smaller than the original repeated string.</p>



<p>Hence, if the document for which the above ClamAV rule matches would have its&nbsp;<code>PerformanceCache</code>&nbsp;removed ClamAV would not detect it anymore. This also affects other security tools relying on plain text VBA code in the&nbsp;<code>PerformanceCache</code>&nbsp;for their detection of malicious content.</p>



<p>While VBA purging is not widely used, we predict that is only a matter of time before this technique catches on and is misused in more sophisticated attacks. This means ClamAV or similar working solutions can not provide adequate protection anymore.</p>

<h2 class="wp-block-heading" id="h-conclusion-and-countermeasure">Conclusion and Countermeasure</h2>



<p>As we have seen VBA purging can evade detection by&nbsp;<strong>some</strong>&nbsp;security solutions. Many solutions seem to fallback to a hash-based detection because presumably their detection signatures rely on plain text VBA code in the&nbsp;<code>PerformanceCache</code>.</p>



<p>Hornetsecurity’s&nbsp;<a href="https://www.hornetsecurity.com/en/services/advanced-threat-protection/">Advanced Threat Protection</a>&nbsp;includes sophisticated VBA macro analyzers that, unlike ClamAV or other solutions, extracts the VBA macro code from the&nbsp;<code>CompressedSourceCode</code>&nbsp;of office documents. This way Hornetsecurity’s multi-layer approach to email attachment scanning will continue working even against VBA purged documents.</p>
