                  <section class="block-paragraph_block">
                    
                    
                      <p data-block-key="k4lbf">In the fall of 2018, the CERT Coordination Center (CERT/CC) Reverse Engineering (RE) Team received a tip from a trusted source about a YARA rule that triggered an alert in VirusTotal. This YARA rule was found in the <a href="https://www.us-cert.gov/ncas/alerts/TA17-293A">Department of Homeland Security (DHS) Alert TA17-293A</a>, which describes nation state threat activity associated with Russian activity. I believed this information warranted further analysis.</p>
                    
                  </section>
                
                  <section class="block-paragraph_block">
                    
                    
                      <p data-block-key="2bjdj">The YARA rule, shown in <a href="#figure1">Figure 1</a>, is allegedly associated with the Energetic Bear group. The Energetic Bear group, named by security firm CrowdStrike, conducts global intelligence operations, primarily against the energy sector. It has been in operation since 2012. (For more information, see CrowdStrike Global Threat Report: 2013 Year in Review.) This group has also been referred to as Dragonfly (Symantec), Crouching Yeti (Kaspersky), Group 24 (Cisco), and Iron Liberty (SecureWorks), among others. (For more information, see APT Groups and Operations.)</p>
                    
                  </section>
                
                  <section class="block-code">
                    
                    
                      




  <pre class="line-numbers language-bash p-0 mx-0 my-8 first:mt-0">
    <code
      id="target-element-current-f6e458bc-1b0d-4ebf-a173-129456123a81"

                  <section class="block-paragraph_block">
                    
                    
                      <p data-block-key="v1oi0">Unfortunately, upon reviewing numerous public threat reports from the above vendors, I could not find further information tying this YARA rule or associated exemplars to the Energetic Bear group, but I still believed that the activity warranted further investigation and analysis.</p>
                    
                  </section>
                
                  <section class="block-heading_block">
                    
                    
                      

    <h2>Methodology</h2>

                    
                  </section>
                
                  <section class="block-paragraph_block">
                    
                    
                      <p data-block-key="8d9gk">I used the following methodology for this analysis:</p>
                    
                  </section>
                
                  <section class="block-paragraph_block">
                    
                    
                      <p data-block-key="ia129"></p><ul><li data-block-key="dkffz">analyzed the YARA rule and initial exemplar<ul><li data-block-key="kcqjn">analyzed exemplar with IDA</li><li data-block-key="zuhd1">researched and applied API hashing module routine findings to exemplar</li><li data-block-key="9r07t">mapped research findings and analysis to exemplar with IDA</li></ul></li><li data-block-key="xjlwj">created a tightly scoped YARA rule to discover new exemplars<ul><li data-block-key="3hcjx">created API hash YARA rules to discover more exemplars</li><li data-block-key="j5dgj">analyzed new exemplars with refined YARA rule</li><li data-block-key="8ee7b">created a tightly scoped YARA rule</li></ul></li><li data-block-key="pcooz">discovered API hashes found in new exemplars</li><li data-block-key="4t46u">questioned attribution</li><li data-block-key="ri7h7">identified future work</li><li data-block-key="4x5wd">reported results</li></ul>
                    
                  </section>
                
                  <section class="block-heading_block">
                    
                    
                      

    <h3>Analyzed the YARA Rule and Initial Exemplar</h3>

                    
                  </section>
                
                  <section class="block-paragraph_block">
                    
                    
                      <p data-block-key="tgegi">I was interested in understanding the string variables found in the YARA rule shown in <a href="#figure1">Figure 1</a>. Specifically, it was not immediately clear what the <i>$api_hash</i> variable represented, whereas the variables <i>$http_post</i> and <i>$http_push</i> appeared to be associated with Hypertext Transfer Protocol (HTTP) header fields. I focused my analysis on the <i>$api_hash</i> variable.</p>
                    
                  </section>
                
                  <section class="block-heading_block">
                    
                    
                      

    <h4>Analyzed Exemplar with IDA</h4>

                    
                  </section>
                
                  <section class="block-paragraph_block">
                    
                    
                      <p data-block-key="udrjh">After cursory analysis of the initial exemplar (SHA256: 1b17ce735512f3104557afe3becacd05ac802b2af79dab5bb1a7ac8d10dccffd), I determined that the <i>$api_hash</i> variable was alerting on the routine (highlighted in green in <a href="#figure2">Figure 2</a>).</p>
                    
                  </section>
                
                  <section class="block-image_block">
                    
                    
                      <a href="/media/images/api_hash.original.png" class="image_large_view cursor-pointer block bg-gray-25">

                        </div>
                      
                    
                    
                  </section>
                
                  <section class="block-heading_block">
                    
                    
                      

    <h4>Researched and Applied API Hashing Module Routine Findings to Exemplar</h4>

                    
                  </section>
                
                  <section class="block-paragraph_block">
                    
                    
                      <p data-block-key="5d08b">The key points to highlight in <a href="#figure2">Figure 2</a> are the <i>or</i> of 0x60, <i>shift logical left (shl) by 1</i>, followed by an <i>add,</i> and <i>jump</i>. Based on this information coupled with the variable name <i>$api_hash</i>, I was able to determine that this was a Windows application programming interface (API) hashing routine.</p>
                    
                  </section>
                
                  <section class="block-paragraph_block">
                    
                    
                      <p data-block-key="nv6yd">I wanted to find further information on any API hashing techniques. Through open source intelligence (OSINT) gathering, I discovered the FireEye Flare IDA Pro utilities Github page that mentioned a plug-in called <i>Shellcode Hashes</i> and an associated blog post from 2012 titled "Using Precalculated String Hashes when Reverse Engineering Shellcode," which further discussed API hashing. (For more information, see <a href="https://github.com/fireeye/flare-ida">FireEye Flare IDA Pro utilities Github</a> and <a href="https://www.fireeye.com/blog/threat-research/2012/11/precalculated-string-hashes-reverse-engineering-shellcode.html">Using Precalculated String Hashes when Reverse Engineering Shellcode</a>.) After I examined the FireEye Flare IDA plug-in script further, I found it contained 23 API hashing modules. I identified an API hashing module, shown in <a href="#figure3">Figure 3</a>, that was very similar to the routine found in the exemplar shown in <a href="#figure2">Figure 2</a>. This API hashing module is a function that contains a <i>for loop</i>, which contains an <i>or</i> of 0x60 followed by <i>add</i> and a <i>shift left</i> <i>by 1.</i></p>
                    
                  </section>
                
                  <section class="block-code">
                    
                    
                      




  <pre class="line-numbers language-bash p-0 mx-0 my-8 first:mt-0">
    <code
      id="target-element-current-55a6c31c-a42b-4d05-9694-f796c94d7264"
      data-codeblock="prism"
      data-lang="bash"
      data-line-numbers="True"
      data-copy="False"
      data-version="1.30.0"
      class="prism-target language-bash my-0 shadow-none"
    >




def sll1AddHash32(inString,fName):
    if inString is None:
        return 0
    val = 0
    for i in inString:
        b = ord(i)
        b = 0xff &amp; (b | 0x60)
        val = val + b
        val = val &lt;&lt; 1
        val = 0xffffffff &amp; val
return val

  </code>
</pre>

                  <section class="block-paragraph_block">
                    
                    
                      <p data-block-key="obymg">The CERT/CC has an API hashing tool that creates a set of YARA signatures of API hashes for a given set of dynamic link library (DLL) files. This API hashing tool contained 22 API hashing modules. One of these modules matched the routine from the exemplar shown in <a href="#figure2">Figure 2</a> and the FireEye API hashing module shown in <a href="#figure3">Figure 3</a>. I called this API hashing module <i>sll1Add</i>. I used the CERT/CC API hashing tool and a clean set of DLL files (see <a href="#table1">Table 1</a>), to create a set of YARA rules for the <i>sll1Add</i> routine. After running the entire set of YARA rules against the exemplar, I received an alert for kernel32.dll API hashes shown in <a href="#figure4">Figure 4</a>.</p>
                    
                  </section>

                  <section class="block-heading_block">
                    
                    
                      

    <h4>Mapped Research Findings and Analysis to Exemplar with IDA</h4>

                    
                  </section>
                
                  <section class="block-paragraph_block">
                    
                    
                      <p data-block-key="5ow6o">I used another CERT/CC tool called UberFLIRT. UberFLIRT calculates and stores position independent code (PIC) hashes of arbitrary functions, easily shares information via a central database, and allows for fewer false positives than IDA's Fast Library Identification and Recognition Technology (FLIRT). I labeled the function shown in <a href="#figure2">Figure 2</a> in IDA as <i>api_hash_func_slladd1</i> and saved it to the Uberflirt database to facilitate future analysis of similar exemplars.</p>
                    
                  </section>
                
                  <section class="block-paragraph_block">
                    
                    
                      <p data-block-key="m38zc">Examining the entry point of the exemplar, I found two values that are pushed onto the stack and passed as parameters to a function. These two values are 0x0038D13C and 0x000D4E88. The value 0x0038D13C is the hash of <i>VirtualProtect</i> shown in <a href="#figure4">Figure 4</a>. The other value, 0x000D4E88, is discussed below.</p>
                    
                  </section>
                
                  <section class="block-paragraph_block">
                    
                    
                      <p data-block-key="qj6ex">Examining this function, where the API hashes were passed as parameters, I determined that this exemplar uses manual symbol loading techniques, which are very similar to that of shellcode, to interact with the system through APIs.</p>
                    
                  </section>
                
                  <section class="block-paragraph_block">
                    
                    
                      <p data-block-key="t6vb3">This process reads the Thread Environment Block (TEB) to find the pointer to Process Environment Block (PEB) structure. The PEB structure is then parsed to find the DllBase of kernel32.dll. This exemplar also checks to ensure that it has the correct kernel32.dll by using 0x000D4E88 hash value to check for the kernel32.dll base name to the kernel32.dll that was found via manual symbol loading. The function then continues to parse the portable executable (PE) export data and passes the virtual protect hash (0x0038D13C) to the hashing algorithm. The same is done for the remaining hashes. This process is shown in <a href="#figure5">Figure 5</a> with my added comments. I labeled the function from <a href="#figure5">Figure 5</a> <i>manual_symbol_resolution</i> and saved it to the UberFLIRT database to aid in future analysis of similar exemplars.</p>
                    
                  </section>

    <h3>Questioned Attribution</h3>

                    
                  </section>
                
                  <section class="block-paragraph_block">
                    
                    
                      <p data-block-key="edjmb">I attempted to identify other public reporting or research related to this Energetic Bear group API hashing tool. I did not identify any public reporting or research.</p>
                    
                  </section>
                
                  <section class="block-paragraph_block">
                    
                    
                      <p data-block-key="it161">Because of the link to the Energetic Bear group, I thought the exemplars could be a remote access Trojan (RAT), such as Havex, which is also attributed to this particular group. I discovered research by Veronica Valeros on <a href="https://www.veronicavaleros.com/blog/2018/3/12/a-study-of-rats-third-timeline-iteration">A Study of RATs: Third Timeline Iteration</a>. I contacted her directly and ask if she recalled any of the RATs she researched using an API hashing technique. She could not recall, but mentioned that it could have been missed because she was not explicitly looking for this technique. I used her research to attempt to identify RATs that use this API hashing technique. I was unable to identify any publically reported RAT using this technique.</p>
                    
                  </section>

You can find the API Hashing listed as a Method under Anti-Static Analysis--<a href="https://github.com/MAECProject/malware-behaviors/blob/master/anti-static-analysis/exe-code-obfuscate.md">Executable Code Obfuscation</a>.</p>
                    
                  </section>
                
                  <section class="block-paragraph_block">
                    
                    
                      <p data-block-key="iv8wf">(March 27, 2019)<br/>The power of open source sharing has been positive. It was brought to my attention (thanks to Matt Brooks from Citizen Lab) that this API hashing tool is related to Trojan.Heriplor from Symantec's <a href="https://www.symantec.com/blogs/threat-intelligence/dragonfly-energy-sector-cyber-attacks">Dragonfly: Westion energy sector targeted by sophisticated attack group</a> report. The hash in Symantec's report is, in fact, one of the exemplars found shown in the <a href="#appendix">appendix</a>. Symantec provided this hash in the form of a picture, and I must have fat-fingered the hash when transcribing it. However, this specific API hashing technique isn't mentioned in their report.</p>
                    
                  </section>
                
                  <section class="block-paragraph_block">
                    
                    
                      <p data-block-key="98in4">This corroboration does help to answer my own question from the <a href="#identified-future-work">Identified Future Work</a> section. Symantec's Trojan.Heriplor analysis attributes my analysis of this API hashing tool to Energetic Bear. More importantly, this linkage also shows that this tool is still actively used.</p>
                    
                  </section>
