			<h1 class="h2">Fantastic payloads and where we find them</h1>
			<div id="blog-info">
				<p>March 30, 2020</p>
				<div class="author-byline">
										<div class="author-photo"><img src="https://intezer.com/wp-content/uploads/2025/03/headshot.png" alt=""></div>					<p class="quote-sm">Written by Michael Kajiloti</p>

			<main class="blog">
				<!-- The Content - TOC Setup -->
				<div class="content lead">
					<?xml encoding="utf-8" ?><p>Attackers have long used evasion features in their malware to avoid detection by security products and analysis systems. One of the most common anti-analysis tricks we have seen in today&rsquo;s Windows malware is the use of <strong>packers</strong>. Packers often complicate the analysis and detection of binary files by hiding the malware&rsquo;s real code and data; often referred to as the <strong>payload</strong>.</p><p>The most common solution to overcome packing is the use of <strong>dynamic analysis</strong> (sandboxes) and emulation systems. However, as these solutions have become more mainstream, packers have started to incorporate new <strong>anti-analysis</strong> methods to both detect and evade these systems.</p><p>One such anti-analysis method which is becoming increasingly popular is the malformation or complete removal of the Windows PE header from unpacked malicious payloads. This can make both finding the malware payload and properly analyzing its code more challenging.</p><p>While removal of Windows PE headers from malicious payloads is not an entirely new method, we are seeing some of the most popular and sophisticated crimeware families fully incorporate this approach into their packers. Below we will present some examples that demonstrate this trend through our Genetic Malware Analysis platform, Intezer Analyze.</p><p><strong>On packers and payloads</strong><br>
When we investigate a packed malware file, the malware (payload) itself is encrypted or encoded and therefore we can&rsquo;t access its real code and data. Typically, in order to analyze the malware&rsquo;s code, we must execute the file in a sandboxed environment, let it unpack itself, and then grab the payload from memory.</p><p>While various malware and packers operate in different ways, an abstraction of a typical unpacking flow looks like this:</p><p><img decoding="async" class="aligncenter wp-image-8427 size-large" src="https://intezer.com/wp-content/uploads/2025/03/intezer-1024x377.png" alt="Intezer" width="1024" height="377"></p><p>In this flow we can see that the packer will decode and execute a shellcode component that will in turn decode and execute the real payload. This is just a common scenario; some packers will not use a shellcode loader and instead they will unpack the payload directly. It&rsquo;s worth noting that unlike the packer&rsquo;s code, which is designed to be obfuscated and appear random, we have observed that these shellcode components are often reused. This means that these components generally have more value than the packer itself when investigating them with code reuse analysis. Therefore, detecting and analyzing this component&rsquo;s code can be worthwhile for us.</p><p>With that said, finding and analyzing the malware module itself is by far the most effective way to properly detect and classify packed malware. The most common way to identify malware payloads in memory is to search for PE header artifacts. Knowing this, many attackers try to stay undetected by removing these artifacts from the PE Header of the payload, or by using no PE header at all, as we&rsquo;ll see soon in the examples presented in this article.</p><p><strong>The many faces of Emotet<br>
</strong><a href="https://intezer.com/blog/emotet-evolves-but-code-remains-mostly-the-same/">Emotet is one of the most widespread malware</a> in the crimeware ecosystem which has undergone a number of updates since its inception. In particular, we have noticed that this malware is constantly evolving and adapting, both in the malware code itself, and in the packers it uses. While switching the packer is relatively inexpensive for the attacker and happens frequently, modifying the malware&rsquo;s code is much more costly and happens rarely.</p><p>An interesting example of this can be seen when we compare the following Emotet samples:</p><ul>
<li>Emotet sample from June 2019: <a href="https://analyze.intezer.com/#/analyses/f5619f98-3552-440e-b7f9-fdf0a5bc0111" target="_blank" rel="noopener noreferrer">8e5089260064a955819a92ebccc43d05520e32d234dd3c176bed5f6d0665ebdb</a></li>
<li>Emotet sample from December 2019: <a href="https://analyze.intezer.com/#/analyses/5edfe909-6c21-4d7a-badc-c6d95756cfe3" target="_blank" rel="noopener noreferrer">3e8a273f80575ab6079d6512008bb6355b59de978fabc2a9a66c0ed148b94fef</a></li>
</ul><p>When looking at the code of the packed files themselves, we can see just how different these samples are, showing no apparent connections to one another:</p><p><img decoding="async" class="aligncenter wp-image-8428 size-large" src="https://intezer.com/wp-content/uploads/2025/03/intezer1-1024x200.png" alt="Intezer" width="1024" height="200"></p><p><a href="https://analyze.intezer.com/#/analyses/f5619f98-3552-440e-b7f9-fdf0a5bc0111" target="_blank" rel="noopener noreferrer">Packer (June 2019) &ndash; 8e5089260064a955819a92ebccc43d05520e32d234dd3c176bed5f6d0665ebdb</a></p><p><img decoding="async" class="aligncenter wp-image-8429 size-large" src="https://intezer.com/wp-content/uploads/2025/03/intezer2-1024x479.png" alt="intezer platform" width="1024" height="479"></p><p><a href="https://analyze.intezer.com/#/analyses/5edfe909-6c21-4d7a-badc-c6d95756cfe3" target="_blank" rel="noopener noreferrer">Packer (December 2019) &ndash; 3e8a273f80575ab6079d6512008bb6355b59de978fabc2a9a66c0ed148b94fef</a></p><p>But if we look at the unpacked memory module payloads after undergoing dynamic analysis, we can see that they are nearly identical from a binary code perspective:</p><p><img decoding="async" class="aligncenter wp-image-8430 size-large" src="https://intezer.com/wp-content/uploads/2025/03/intezer3-1024x368.png" alt="intezer platform" width="1024" height="368"></p><p><a href="https://analyze.intezer.com/#/analyses/f5619f98-3552-440e-b7f9-fdf0a5bc0111/sub/7804422f-75c6-4904-b69d-25b6a2b38acb" target="_blank" rel="noopener noreferrer">Payload (June 2019) &ndash; 8e5089260064a955819a92ebccc43d05520e32d234dd3c176bed5f6d0665ebdb</a></p><p><img decoding="async" class="aligncenter wp-image-8431 size-large" src="https://intezer.com/wp-content/uploads/2025/03/intezer4-1024x373.png" alt="Intezer Platform" width="1024" height="373"></p><p><a href="https://analyze.intezer.com/#/analyses/5edfe909-6c21-4d7a-badc-c6d95756cfe3/sub/e4019d81-86b9-4b45-8e1e-a85645f0dd8b" target="_blank" rel="noopener noreferrer">Payload (December 2019) &ndash; 3e8a273f80575ab6079d6512008bb6355b59de978fabc2a9a66c0ed148b94fef</a></p><p>There is one notable exception: in the December 2019 sample, the payload memory module has a <strong>&ldquo;malformed header&rdquo;</strong> tag. In this case, this indicates that the packer loaded the module with a PE Header that&rsquo;s missing certain PE artifacts, such as the &ldquo;MZ&rdquo; magic or other PE Header constants. This is usually done in an attempt to make it more difficult to find these modules in memory, since the most common method of finding PE memory modules is by scanning for these artifacts.</p><p>It is worth mentioning that in Intezer Analyze we repair the malformed header as part of our analysis process. Downloading and examining the memory module will reveal that it has a full PE header.</p><p>It is also important to note that even in the instance of a constantly evolving threat such as Emotet, when the attacker modifies the code of the malware itself, there are still parts that remain unchanged. Therefore, when Intezer Analyze executes the malware, finds its payload, and then analyzes its code, we can still detect those traces to Emotet&rsquo;s code that have not been modified.</p><p>This can be observed when we investigate a recent sample of Emotet from March 2020. Looking at the packer&rsquo;s code, we can see that it is different yet again, and has no relations to any code we have seen previously:</p><p><img decoding="async" class="aligncenter wp-image-8432 size-large" src="https://intezer.com/wp-content/uploads/2025/03/intezer5-1024x237.png" alt="Intezer Platform" width="1024" height="237"></p><p><a href="https://analyze.intezer.com/#/analyses/48b36b5c-4cea-40bf-b3b5-a0d2dfec4cda" target="_blank" rel="noopener noreferrer">Packer (March 2020) &ndash; C6684aa4af41685ffc584c1b1bb78855c894b3dae7f6fae4fb572bc02525102f</a></p><p>However, when we look at the memory module payload itself, we can see that some of the code is new and has been modified from previous versions. Most importantly, a substantial part of the code has been reused from previous Emotet variants that we have seen before:</p><p><img decoding="async" class="aligncenter wp-image-8433 size-large" src="https://intezer.com/wp-content/uploads/2025/03/intezer6-1024x301.png" alt="Intezer Platform" width="1024" height="301"></p><p><a href="https://analyze.intezer.com/#/analyses/48b36b5c-4cea-40bf-b3b5-a0d2dfec4cda/sub/0d3943b2-eae2-496c-acce-20381f23b17b" target="_blank" rel="noopener noreferrer">Payload (March 2020) &ndash; C6684aa4af41685ffc584c1b1bb78855c894b3dae7f6fae4fb572bc02525102f</a></p><p><strong>When less is more</strong><br>
While using a malformed PE Header is a relatively simple trick, using no PE header at all is a more sophisticated and complicated affair. It takes more than just removing a few unused magic constants from the header in order to conceal it. This method requires loading and executing the payload just like shellcode.</p><p>When malware authors write and compile the malware payload itself, it is usually compiled and built into the form of standard PE files, not position independent shellcode. Directly converting a sophisticated malware&rsquo;s codebase such as Dridex or TrickBot to run like shellcode can be complicated.</p><p>Most malware authors choose a simpler option, using a packer that unpacks and loads the payload module without using its PE Header at load time. These packers typically store the relevant information from the PE header, such as API import and relocations data, in a custom structure during packing time. The packer then uses that structure during the unpacking process in order to properly load the malware code.</p><p><strong>Same Dridex, new tricks</strong><br>
An example of a popular malware that has recently adopted such a packer is the Dridex banking trojan. Let&rsquo;s compare two recent Dridex samples from February 2020:</p><ul>
<li><a href="https://analyze.intezer.com/#/analyses/9fdbb152-275f-4cb3-beaa-4acc33a28a52" target="_blank" rel="noopener noreferrer">9b6ffb05ec826b3fbff7db04ebd187e2f956a5730af1d36e10a9f56ee1bb767a</a></li>
<li><a href="https://analyze.intezer.com/#/analyses/66328685-405d-4445-9437-2942e125aed4" target="_blank" rel="noopener noreferrer">45e0408d3ae2647cb9be835a5277eb7de41c83d8e473d22d5b3772c2f65305fb</a></li>
</ul><p>Similar to Emotet, these packed files are very different from one another at the code level, showing no resemblance:</p><p><img decoding="async" class="aligncenter wp-image-8434 size-large" src="https://intezer.com/wp-content/uploads/2025/03/intezer7-1024x277.png" alt="Intezer Platform" width="1024" height="277"></p><p><a href="https://analyze.intezer.com/#/analyses/9fdbb152-275f-4cb3-beaa-4acc33a28a52" target="_blank" rel="noopener noreferrer">Packer &ndash; 9b6ffb05ec826b3fbff7db04ebd187e2f956a5730af1d36e10a9f56ee1bb767a</a></p><p><img decoding="async" class="aligncenter wp-image-8435 size-large" src="https://intezer.com/wp-content/uploads/2025/03/intezer8-1024x226.png" alt="Intezer Platform" width="1024" height="226"></p><p><a href="https://analyze.intezer.com/#/analyses/66328685-405d-4445-9437-2942e125aed4" target="_blank" rel="noopener noreferrer">Packer &ndash; 45e0408d3ae2647cb9be835a5277eb7de41c83d8e473d22d5b3772c2f65305fb</a></p><p>However, when we look at the payloads themselves, we can see again that they are actually very similar:</p><p><img decoding="async" class="aligncenter wp-image-8436 size-large" src="https://intezer.com/wp-content/uploads/2025/03/intezer9-1024x489.png" alt="Intezer Platform" width="1024" height="489"></p><p><a href="https://analyze.intezer.com/#/analyses/9fdbb152-275f-4cb3-beaa-4acc33a28a52/sub/d47a060e-c241-42c5-8f20-ea00a849a042" target="_blank" rel="noopener noreferrer">Payload &ndash; 9b6ffb05ec826b3fbff7db04ebd187e2f956a5730af1d36e10a9f56ee1bb767a</a></p><p><img decoding="async" class="aligncenter wp-image-8437 size-large" src="https://intezer.com/wp-content/uploads/2025/03/intezer10-1024x440.png" alt="Intezer Platform" width="1024" height="440"></p><p><a href="https://analyze.intezer.com/#/analyses/66328685-405d-4445-9437-2942e125aed4/sub/cd707392-6139-4d9a-bded-253719d943da" target="_blank" rel="noopener noreferrer">Payload &ndash; 45e0408d3ae2647cb9be835a5277eb7de41c83d8e473d22d5b3772c2f65305fb</a></p><p>There is one notable exception here as well, but in this case it is much more substantial. In the second payload we can see the <strong>&ldquo;fileless_code&rdquo; </strong>tag, which indicates this module has no PE header. Downloading and examining this module will make it known that it is indeed a binary file with no structure.</p><p>Intezer Analyze detects these modules during dynamic analysis and analyzes their code even though no PE Header is present. It will also detect any other shellcode pieces that are used by the malware.</p><p><strong>Look ma! No heads!<br>
</strong>To demonstrate how prevalent this trend is, let&rsquo;s look at the analyses of recent samples of the&nbsp;<a href="https://analyze.intezer.com/#/analyses/b166869f-beda-4010-bf28-d50717750df9" target="_blank" rel="noopener noreferrer">Ursnif</a> and <a href="https://analyze.intezer.com/#/files/26d61225f07e39d9e35880631c5043626eb8e3846a526a44015d3a5d03642d55" target="_blank" rel="noopener noreferrer">TrickBot</a> malware. When looking at the dynamic analysis modules, we see several different malicious modules that are detected and analyzed:</p><p><img decoding="async" class="aligncenter wp-image-8438 size-full" src="https://intezer.com/wp-content/uploads/2025/03/Intezer11.png" alt="Intezer Platform" width="830" height="703"></p><p>If we review them more closely, we will see that all of them&nbsp; &mdash; minus copies of the original packer file &mdash; have the <strong>&ldquo;fileless_code&rdquo; </strong>tag, indicating that they are all shellcode-style modules with no PE header:</p><p><img decoding="async" class="aligncenter wp-image-8439 size-large" src="https://intezer.com/wp-content/uploads/2025/03/intezer12-1024x209.png" alt="Intezer Platform" width="1024" height="209"></p><p><a href="https://analyze.intezer.com/#/files/26d61225f07e39d9e35880631c5043626eb8e3846a526a44015d3a5d03642d55/sub/3d431957-aae3-42b6-8582-0b214f0f824d" target="_blank" rel="noopener noreferrer">TrickBot &ndash; 26d61225f07e39d9e35880631c5043626eb8e3846a526a44015d3a5d03642d55</a></p><p><img decoding="async" class="aligncenter wp-image-8440 size-large" src="https://intezer.com/wp-content/uploads/2025/03/intezer13-1024x368.png" alt="Intezer Platform" width="1024" height="368"></p><p><strong>Conclusion</strong><br>
The use of pure shellcode and shellcode-style components in modern malware seems to be a growing trend, one that is being adopted by some of the most successful malware developers. At the same time, when we identify and analyze these components using code reuse analysis, we can see that they are mostly just a different version of the same thing.
