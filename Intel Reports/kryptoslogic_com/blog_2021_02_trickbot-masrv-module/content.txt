<time class=blog-post-date datetime="2021-02-01 14:00:00 +0000 UTC">Monday, February 1, 2021<br>Tags:
<a href=https://www.kryptoslogic.com/blog/tag/trickbot>TrickBot</a></time></header><div class=blog-post-content><p><h1 id=overview>Overview</h1><p>Active since 2016, TrickBot is one of the most prevalent modular banking trojans. The botnet&rsquo;s modules carry out objectives such as credential harvesting, propagating via the network, web injection and others. Being an actively developed botnet, we often come across updated modules and in some cases new tools that are added as part of its arsenal.</p><p>Recently we have discovered a relatively new module that goes by the name <code>masrv</code>. The module is a network scanner that incorporates the <a href=https://github.com/robertdavidgraham/masscan>Masscan</a> open-source tool. Additionally, the module contains an unreferenced Anchor C2 communication function and a list of hardcoded IPs which have previously been associated with Anchor and Bazar <sup id=fnref:1><a href=#fn:1 class=footnote-ref role=doc-noteref>1</a></sup><sup id=fnref:2><a href=#fn:2 class=footnote-ref role=doc-noteref>2</a></sup>.</p><p>We believe this module is used as one of TrickBot&rsquo;s network reconnaissance tools to gather more information about the victim&rsquo;s network.</p><h1 id=the-masrv-module>The masrv module</h1><p>The module arrives as either a 32-bit or 64-bit DLL, depending on the Windows OS version of the victim machine the bot is running on. Both DLLs we observed are debug builds and log their execution into standard output.</p><p>As with other TrickBot modules, the module is executed via its export functions <code>Start</code> and <code>Control</code><sup id=fnref:3><a href=#fn:3 class=footnote-ref role=doc-noteref>3</a></sup>.</p><h2 id=commands-for-the-modules-c2>Commands for the Module&rsquo;s C2</h2><p>The module makes requests to the C2 to receive information that it requires to pass as parameters to Masscan.</p><table><thead><tr><th style=text-align:left><strong>Command</strong></th><th style=text-align:left><strong>HTTP Method</strong></th><th><strong>Description</strong></th></tr></thead><tbody><tr><td style=text-align:left>81</td><td style=text-align:left>POST</td><td>send results</td></tr><tr><td style=text-align:left>freq</td><td style=text-align:left>GET</td><td>Get frequency for running Masscan</td></tr><tr><td style=text-align:left>domains</td><td style=text-align:left>GET</td><td>Get a List of IP address ranges followed by port range</td></tr><tr><td style=text-align:left>over</td><td style=text-align:left>GET</td><td>Signal to the C2 that scan is complete</td></tr><tr><td style=text-align:left>rate</td><td style=text-align:left>GET</td><td>Get rate value for transmitting packets</td></tr><tr><td style=text-align:left>npcap.exe</td><td style=text-align:left>GET</td><td>Get Nmap&rsquo;s packet sniffing library installer</td></tr></tbody></table><p>The URI construction for the GET requests follows this format:</p><pre><code>http://&lt;c2&gt;:&lt;port&gt;/&lt;gtag&gt;/&lt;botID&gt;/mass/&lt;command_string&gt;
</code></pre><ul><li><code>gtag</code> - The Campaign ID that is seen in the config<sup id=fnref:4><a href=#fn:4 class=footnote-ref role=doc-noteref>4</a></sup> present in the main bot.</li><li><code>botID</code> - The Bot ID created in the victim machine by the main bot.</li><li><code>command_string</code> - One of the string commands from the above table.</li></ul><p>At the time of researching this module, we were unable to pull down the config associated with <code>masrv</code>. So, in order to observe a dynamic run, we have implemented a mock server on <code>localhost</code> at port <code>8080</code>, to be able to feed responses back to the module. Below is an example of one of the GET request being made for the command <code>freq</code>.</p><figure><a href=images/pcap.png data-toggle=lightbox data-title="Network capture of the Module traffic"><img src=images/pcap.png loading=lazy></a><figcaption><h4>Network capture of the Module traffic</h4></figcaption></figure><h2 id=information-gathering>Information Gathering</h2><p>At first, the module makes GET requests for information from the commands <code>freq</code>, <code>domains</code> and <code>rate</code>. If successful, the module executes Masscan&rsquo;s main function routine which is compiled within the DLL. Below we can see the execution result of the log from standard output. The date mentioned in the logs is that of when the module was compiled.</p><pre><code>SendEvent(VERS, MASS scanner build Dec  4 2020 13:19:27 started)

Send cmd to server: freq

SendEvent(DBG, Successfully executed command: rate)
</code></pre><p>The Masscan tool has its own network stack and doesn&rsquo;t rely on that of the OS. In order for it to be able to retrieve the results, Masscan requires a low-level packet filter and on a Windows OS it attempts to load <code>NPcap\Packet.dll</code>. If <code>Packet.dll</code> doesn&rsquo;t exist, then the module makes a request to download the <code>NPcap</code> executable from the C2. <code>NPcap</code> is silently installed on the machine by passing the parameter <code>/S</code>. It gets executed by invoking <code>CreateProcessA</code> or <code>ShellExecuteExA</code> (if the first API is unsuccessful).</p><p>The Masscan tool also attempts to initialize the network adapter. If the tool fails to detect any interface, a module-specific function is called that tries to get a MAC address from the ARP table, to pass to Masscan as <code>--router-mac &lt;mac></code>. For each ARP entry in the <code>MIB_IPNETTABLE</code><sup id=fnref:5><a href=#fn:5 class=footnote-ref role=doc-noteref>5</a></sup>, the module finds the corresponding index of the IPv4 entry in the <code>MIB_IPADDRTABLE</code><sup id=fnref:6><a href=#fn:6 class=footnote-ref role=doc-noteref>6</a></sup>. It leverages the APIs <code>GetIpNetTable</code> and <code>GetIpAddrTable</code> respectively to retrieve this information. If successful, it gets the dotted-decimal format of the IPv4 address and logs the results of the <code>ping</code> command that is run on the target <code>8.8.8.8</code> from that IPv4 address. If the ping ran successfully, the module gathers the ARP type information and logs the ARP entry of the IPv4 address. Then it queries for the MAC address from the <code>MIB_IPNETROW</code> entry. Below is an example of the <code>ping</code> command.</p><pre><code>ping 8.8.8.8 -S 127.0.0.1
</code></pre><p>The module sends results from the Masscan run if it has discovered open ports on any of the IP ranges that were provided. Results are aggregated by calling a module-specific function from the Masscan function <code>output_report_status</code> which adds discovered ports to a global string. These results are posted back (via the <code>81</code> message) regularly, with the frequency, in seconds, determined by the <code>freq</code> value queried at the beginning.</p><figure><a href=images/postdata.PNG data-toggle=lightbox data-title="POST Request"><img src=images/postdata.PNG loading=lazy></a><figcaption><h4>POST Request</h4></figcaption></figure><h2 id=anchorbazar-reference>Anchor/Bazar reference</h2><p>Both the 32-bit and 64-bit DLLs have an unreferenced function that share similarities to Anchor&rsquo;s C2 communication subroutine. It is not uncommon for this actor to be seen sharing code between its toolset.
