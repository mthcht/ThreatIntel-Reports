// Helper function to get a cookie value by name
function getCookie(name) {
    const cookieArray = document.cookie.split("; ");
    for (const cookie of cookieArray) {
        const [key, value] = cookie.split("=");
        if (key === name) {
            return decodeURIComponent(value);
        }
    }
    return null;
}

// Helper function to set a cookie
function setCookie(name, value, days = 7) {
    const expires = new Date(Date.now() + days * 24 * 60 * 60 * 1000).toUTCString();
    document.cookie = `${name}=${encodeURIComponent(value)}; expires=${expires}; path=/`;
}

// Function to fetch and store the country in lowercase
async function fetchAndStoreCountry() {
    const cookieName = "lrqalocales";

    // Check if the country is already stored in the cookie
    const storedCountry = getCookie(cookieName);
    if (storedCountry) {
        return storedCountry; // Return the stored country code
    }

    // If not in cookie, fetch the country
    const apiUrl = "https://ipinfo.io/json?token=737607a82d617e";
    try {
        const ipinfoRequest= await fetch(apiUrl, {
  referrerPolicy: "origin-when-cross-origin" });
        if (!ipinfoRequest.ok) {
            return 'us'
        }
        const jsonResponse = await ipinfoRequest.json();
        const country = jsonResponse.country.toLowerCase();
        setCookie(cookieName, country); 
        return country;
    } catch (error) {
        return 'us';
    }
}

// Wrapper to allow synchronous retrieval
function getCountrySync() {
    let country = getCookie("lrqalocaler");
    if (country) {
        return country;
    }

    // Asynchronous call to fetch and store country
    fetchAndStoreCountry().catch((error) => {
            return 'us';
    });

    return null; // Return null while the async operation completes
}

<p>As a counterpart to the<span>&nbsp;</span><a href="https://labs.nettitude.com/blog/introducing-poshc2-v6-0/" target="_blank" rel="noopener noreferrer">release of PoshC2 version 6.0</a>&nbsp;we are providing a list of some of its Indicators of Compromise (IoCs), particularly as used out-of-the-box, as well as some other effective methods for detecting it in your environment.</p>
<p>We also introduce the new PoshC2 Detections GitHub repository at<span>&nbsp;</span><a href="https://github.com/nettitude/PoshC2_IOCs" target="_blank" rel="noopener noreferrer">https://github.com/nettitude/PoshC2_IOCs</a><span>&nbsp;</span>that will be continually updated as development continues, in order to assist blue teams with detecting PoshC2, particularly when used by less sophisticated attackers that do not alter or configure it to change the default IoCs. We encourage the community to contribute to and help update and improve this repository.</p>
<p>It is worth noting that PoshC2 is open source, so while these are IoCs for PoshC2 if used it its default state, ultimately it can be altered either through configuration or by changing the source code itself. The default configuration is subject to change, however where possible the location of that value is pointed out to the reader in order to allow these values to be monitored and updated, in addition to providing the GitHub repository.</p>
<h2>Communications</h2>
<p>One way to detect PoshC2 is to focus on the communications. Compromised hosts have to communicate with the C2 server somehow in order to pick up tasks and return task output. This is unavoidable and PoshC2 can currently only do this through the use of HTTP(S) requests.</p>
<p>That isn&rsquo;t to say this this makes it easy to detect; a huge amount of HTTP traffic is present in most environments and the flexibility of the protocol allows for traffic to be hidden and routed through legitimate websites using techniques such as Domain Fronting and reverse proxies.</p>
<div id="attachment_13933" class="wp-caption aligncenter"><img class="wp-image-13933" src="/globalassets/cyber-labs/detecting-poshc2--indicators-of-compromise/detecting-poshc2.png?epslanguage=en" alt="An example of HTTP comms when domain fronting." width="605" height="435" aria-describedby="caption-attachment-13933" />
<p id="caption-attachment-13933" class="wp-caption-text">An example of HTTP comms when domain fronting.</p>
</div>
<p>Something very helpful for catching C2 communications is SSL Inspection. By being able to inspect SSL traffic leaving the perimeter, the contents of that traffic can be checked and statistics acquired for detecting C2 communications. Without this, network defenders are largely blind, particularly against domain fronted communications which travel via legitimate third-party websites.</p>
<p>If SSL Inspection is implemented then the HTTP traffic can be viewed, and while PoshC2 encrypts the contents of the HTTP bodies, the HTTP URLs and headers can still be viewed.</p>
<h3>URLs</h3>
<p>PoshC2 has two different ways of generating URLs to use for communications. Operators can either use a static list of URLs or provide a wordlist from which random URLs will be generated, and these files are stored at<span>&nbsp;</span><code>resources/urls.txt</code><span>&nbsp;</span>and<span>&nbsp;</span><code>resources/wordlist.txt</code><span>&nbsp;</span>respectively, with the static URL list being the default option. These URLs are then loaded into the database when the project is first created, and a random URL is chosen by each implant each time it beacons. The default URL list is below:</p>

<p>While these URLs were originally copied from legitimate requests, if you see several of them being repeated to a site, particularly if they do not seem relevant to that site and if the response does not make sense, then it could be PoshC2 beacon traffic.</p>
<h3>HTTP Responses</h3>
<p>PoshC2 also has static HTML responses that it responds with. The default is six HTTP 200 responses and one 404 response. These are stored in files at<span>&nbsp;</span><code>resources/responses/</code><span>&nbsp;</span>and also loaded into the database when the server is first created. The server responds with a random 200 response to POST requests that do not error or require a specific response, and with the single 404 response to all unexpected URLs or when the C2 server errors. Other responses return context relevant data, such as tasks, implant code and so on.</p>
<pre class="lang:xhtml decode:true">&lt;!DOCTYPE HTML PUBLIC "-//IETF//DTD HTML 2.0//EN"&gt;
&lt;html&gt;&lt;head&gt;
&lt;title&gt;404 Not Found&lt;/title&gt;
&lt;/head&gt;&lt;body&gt;
&lt;h1&gt;Not Found&lt;/h1&gt;
&lt;p&gt;The requested URL was not found on this server.&lt;/p&gt;
&lt;hr&gt;
&lt;address&gt;Apache (Debian) Server&lt;/address&gt;
&lt;/body&gt;&lt;/html&gt;</pre>
<p>This static HTML file then at<span>&nbsp;</span><code>resources/responses/404_response.html</code><span>&nbsp;</span>is an IoC and if returned from a webserver that you are investigating is suggestive of PoshC2. Similarly, the<span>&nbsp;</span><em>200_response*</em><span>&nbsp;</span>files in the same directory are IoCs if returned by POST requests.</p>
<p>Note however, that it is recommended that operators change these files before creating a PoshC2 project, as is the use of a C2 proxy, so as with the other indicators the absence of this particular response is not unexpected for a PoshC2 installation.</p>
<h3>SSL Certificate</h3>
<p>PoshC2 by default creates a self-signed certificate for its HTTP server, the values for which are stored in<span>&nbsp;</span><code>poshc2/server/Config.py</code><span>&nbsp;</span>file. These values are not in the &lsquo;normal&rsquo; configuration file<span>&nbsp;</span><code>config.yml</code><span>&nbsp;</span>and are less documented and are therefore harder to change.</p>
<pre class="lang:yaml decode:true">Cert_C = "US"
Cert_ST = "Minnesota"
Cert_L = "Minnetonka"
Cert_O = "Pajfds"
Cert_OU = "Jethpro"
Cert_CN = "P18055077"
Cert_SerialNumber = 1000
Cert_NotBefore = 0
Cert_NotAfter = (10 * 365 * 24 * 60 * 60)</pre>
<p>An experienced operator will not expose their C2 server to the internet, but will instead use a proxy server with a valid certificate and filter firewall traffic to the C2 server that is not from that proxy, however if these steps are not taken and a certificate with the below values is presented then it is another strong indicator that PoshC2 is in use, likely by less sophisticated adversaries.</p>
<div id="attachment_13934" class="wp-caption aligncenter"><img class="wp-image-13934" src="/globalassets/cyber-labs/detecting-poshc2--indicators-of-compromise/word-image-1.png?epslanguage=en" alt="The issuer data can be viewed using Chrome and the default values suggest this is a PoshC2 server." width="403" height="509" aria-describedby="caption-attachment-13934" />
<p id="caption-attachment-13934" class="wp-caption-text">The issuer data can be viewed using Chrome and the default values suggest this is a PoshC2 server.</p>
</div>
<h3>JA3 Signatures</h3>
<p>Another method for signaturing C2 traffic without SSL Inspection is to fingerprint the<span>&nbsp;</span><em>Client Hello</em><span>&nbsp;</span>packet that initialises the TCP connection. The specific bytes that make up the packet are dependent upon the type of connection being formed and the underlying packages and methods used to do so.</p>
<p>As these packets are sent before encryption has been established they are cleartext and can be intercepted and read. It turns out that the packet contents are quite unique and can be fingerprinted. Salesforce have an<span>&nbsp;</span><a href="https://engineering.salesforce.com/tls-fingerprinting-with-ja3-and-ja3s-247362855967" target="_blank" rel="noopener noreferrer">excellent blog post on JA3 fingerprinting</a>, but essentially the details are extracted from the packets that make up this TCP handshake and hashed to create a quick and easy signature, that can be used to identify not only that PoshC2 is in use, but also the specific implant type.</p>
<p><img class="wp-image-13935 aligncenter" src="/globalassets/cyber-labs/detecting-poshc2--indicators-of-compromise/word-image-2.png?epslanguage=en" width="1200" height="324" /><br />The current Windows 10 PoshC2 signatures are below:</p>
<p><code>PowerShell: c12f54a3f91dc7bafd92cb59fe009a35</code><br /><code>Sharp: fc54e0d16d9764783542f0146a98b300</code></p>
<p>JA3 fingerprinting has been incorporated into security products such as Splunk and Darktrace to provide quick and easy identification of C2 traffic, and can be utilised with these fingerprints. These fingerprints are available and will be maintained in the<span>&nbsp;</span><a href="https://github.com/nettitude/PoshC2_IOCs" target="_blank" rel="noopener noreferrer">PoshC2 Detections GitHub repository</a>.</p>
<h3>Telemetry</h3>
<p>Another mechanism for detecting C2 traffic, with or without SSL Inspection, is through the use of telemetry.</p>
<p>PoshC2&rsquo;s default values for beacon time and jitter in the<span>&nbsp;</span><code>config.yml</code><span>&nbsp;</span>file are 5 seconds and a 20% jitter, as can be seen below, which is both a fast beacon rate and a small jitter, making it relatively easy to detect using this method.</p>
<pre class="lang:yaml decode:true ">DefaultSleep: "5s"
Jitter: 0.20</pre>
<p>With these values the implants are going to beacon every 4-6 seconds by default, with an average of 5 seconds.</p>
<p>After capturing some traffic, filtering by the C2 server host and only checking TLS Client Hello packets, we can see that the TLS connection is created roughly every five seconds, confirming what we expect.</p>
<div id="attachment_13936" class="wp-caption aligncenter"><img class="wp-image-13936" src="/globalassets/cyber-labs/detecting-poshc2--indicators-of-compromise/word-image-3.png?epslanguage=en" alt="We can see the difference between each session initialisation is between 4 and 6 seconds." width="1151" height="966" aria-describedby="caption-attachment-13936" />
<p id="caption-attachment-13936" class="wp-caption-text">We can see the difference between each session initialisation is between 4 and 6 seconds.</p>
</div>
<p>By exploring this small sample of data, we can determine that the average time between the requests is 5.05 seconds, with a standard deviation of 0.81 or 16%, close to the 5 second beacon with the 20% jitter we expect from PoshC2&rsquo;s defaults.</p>
<p>Security products that can provide telemetry data, e.g. Splunk can be used to detect C2 traffic in this way by checking for repetitive beacons at a relatively fixed frequency, whether it&rsquo;s at the above defaults for PoshC2 or at any frequency, given that it is configurable.</p>
<h2>PowerShell Implant</h2>
<p>PoshC2 has three implant types; PowerShell, C#, and Python, with the latter being a lightweight implant type mostly intended for compromising *nix hosts.<br />The PowerShell Implant runs by loading<span>&nbsp;</span><code>System.Management.Automation.dll</code>, the engine behind<span>&nbsp;</span><code>PowerShell.exe</code>, into the desired process and executing PowerShell commands directly using this DLL.</p>
<div id="attachment_13937" class="wp-caption aligncenter"><img class="wp-image-13937" src="/globalassets/cyber-labs/detecting-poshc2--indicators-of-compromise/word-image-4.png?epslanguage=en" alt="The PowerShell implant supports full PowerShell execution from any process (here netsh.exe) by loading System.Management.Automation.dll." width="1200" height="636" aria-describedby="caption-attachment-13937" />
<p id="caption-attachment-13937" class="wp-caption-text">The PowerShell implant supports full PowerShell execution from any process (here netsh.exe) by loading System.Management.Automation.dll.</p>
</div>
<p>While this does not utilise<span>&nbsp;</span><code>PowerShell.exe,</code><span>&nbsp;</span>bypassing many restrictions and controls, it is still subject to PowerShell specific constraints such as Constrained Language mode and ScriptBlock logging. The latter, in particular, can be used to detect PoshC2 and any other malicious PowerShell commands, whether via<span>&nbsp;</span><code>PowerShell.exe</code><span>&nbsp;</span>or otherwise.</p>
<p>Note, however, that ScriptBlock logging was only enabled in PowerShell v5.0 and above, so if PowerShell v2.0 is available on the target then a downgrade attack can be performed and PowerShell 2.0 used, bypassing Constrained Language mode, ScriptBlock logging, AMSI and other controls added in v5.0.</p>
<h3>PS Logging</h3>
<p>Enabling ScriptBlock logging and Transcript Logging for PowerShell allows logging of processes running PowerShell commands.</p>
<p>There are multiple IoCs here, not least that the Host Application is not<span>&nbsp;</span><code>PowerShell.exe</code>, as should be expected, but instead<span>&nbsp;</span><code>netsh.exe</code>.</p>
<div id="attachment_13938" class="wp-caption aligncenter"><img class="wp-image-13938" src="/globalassets/cyber-labs/detecting-poshc2--indicators-of-compromise/word-image-5.png?epslanguage=en" alt="PowerShell transcripts provide valuable information. Alerting on Host Applications that are not PowerShell.exe is a good way to find PowerShell implants from any C2 framework." width="507" height="412" aria-describedby="caption-attachment-13938" />
<p id="caption-attachment-13938" class="wp-caption-text">PowerShell transcripts provide valuable information. Alerting on Host Applications that are not PowerShell.exe is a good way to find PowerShell implants from any C2 framework.</p>
</div>
<p>Similarly in the Event Viewer for ScriptBlock logging:</p>
<div id="attachment_13939" class="wp-caption aligncenter"><img class="wp-image-13939" src="/globalassets/cyber-labs/detecting-poshc2--indicators-of-compromise/word-image-6.png?epslanguage=en" alt="PowerShell ScriptBlock logging can reveal a wealth of information, here including the beacon URLs for PoshC2 and again netsh.exe as a host process." width="992" height="458" aria-describedby="caption-attachment-13939" />
<p id="caption-attachment-13939" class="wp-caption-text">PowerShell ScriptBlock logging can reveal a wealth of information, here including the beacon URLs for PoshC2 and again netsh.exe as a host process.</p>
</div>
<p>This also includes the &lsquo;command line&rsquo; of the ScriptBlock being executed, which includes the beacon URLs from which a random element is being chosen, therefore listing all the C2 URLs that are in use by this implant.</p>
<p>Aside from the wealth of information ScriptBlock logging provides for any threat hunter, an event command line specific to PoshC2 that can be used to identify the PowerShell implant is below:</p>
<div id="attachment_13940" class="wp-caption aligncenter"><img class="wp-image-13940" src="/globalassets/cyber-labs/detecting-poshc2--indicators-of-compromise/word-image-7.png?epslanguage=en" alt="The $ServerClean variable is specific to PoshC2 and a clear IoC." width="671" height="450" aria-describedby="caption-attachment-13940" />
<p id="caption-attachment-13940" class="wp-caption-text">The $ServerClean variable is specific to PoshC2 and a clear IoC.</p>
</div>
<p><code>CommandLine= $ServerClean = Get-Random $ServerURLS</code></p>
<p>This ScriptBlock is part of the beaconing process and will be repeated frequently in the Event Viewer making it easy to identify.</p>
<h3>System.Management.Automation.dll</h3>
<p>As mentioned earlier, the PowerShell implant does not function by invoking<span>&nbsp;</span><code>PowerShell.exe</code>, but instead by loading<span>&nbsp;</span><code>System.Management.Automation.dll</code><span>&nbsp;</span>into the implant process.</p>
<p>This means that this is an IoC, and if we find a process that should not have this DLL loaded, particularly if it is an unmanaged code binary (so not .NET) then it is highly likely that this is a process that has been injected into by a PowerShell implant, PoshC2 or otherwise.</p>
<div id="attachment_13941" class="wp-caption aligncenter"><img class="wp-image-13941" src="/globalassets/cyber-labs/detecting-poshc2--indicators-of-compromise/word-image-8.png?epslanguage=en" alt="System.Management.Automation.dll, the engine behind PowerShell, is a .NET library." width="478" height="135" aria-describedby="caption-attachment-13941" />
<p id="caption-attachment-13941" class="wp-caption-text">System.Management.Automation.dll, the engine behind PowerShell, is a .NET library.</p>
</div>

<p id="caption-attachment-13942" class="wp-caption-text">netsh.exe is a C++ binary and should not be loading .NET libraries.</p>
</div>
<p>We can see this DLL loaded into an implant process using Process Hacker:</p>
<div id="attachment_13943" class="wp-caption aligncenter"><img class="wp-image-13943" src="/globalassets/cyber-labs/detecting-poshc2--indicators-of-compromise/word-image-10.png?epslanguage=en" alt="System.Management.Automation.dll loaded into netsh.exe." width="692" height="732" aria-describedby="caption-attachment-13943" />

<div id="attachment_13945" class="wp-caption aligncenter"><img class="wp-image-13945" src="/globalassets/cyber-labs/detecting-poshc2--indicators-of-compromise/word-image-12.png?epslanguage=en" alt="The module is not loaded in a genuine netsh process and the .NET tabs are not available." width="616" height="406" aria-describedby="caption-attachment-13945" />
<p id="caption-attachment-13945" class="wp-caption-text">The module is not loaded in a genuine netsh process and the .NET tabs are not available.</p>
</div>
<p>An operator can be smarter about their migration by injecting into .NET processes, however it is still unlikely that a legitimate process (that isn&rsquo;t<span>&nbsp;</span><code>PowerShell.exe</code>) would load<span>&nbsp;</span><code>System.Management.Automation.dll</code>, so this is a great IoC to look out for in your environment.</p>
<p>This can be implemented at scale by searching for loaded modules in an EDR product, for example in CarbonBlack with a query of:</p>
<p><code>modload:System.Management.Automation* AND -process_name:powershell.exe AND &ndash;process_name:powershell_ise.exe</code></p>
<p>This searches for process with<span>&nbsp;</span><code>System.Management.Automation.dll</code><span>&nbsp;</span>or<span>&nbsp;</span><code>System.Management.Automation.ni.dll</code><span>&nbsp;</span>(the Native Image version) loaded into a process when that process is<span>&nbsp;</span><em>not</em><span>&nbsp;</span><code>PowerShell.exe</code><span>&nbsp;</span>or<span>&nbsp;</span><code>PowerShell_ISE.exe</code>. Other legitimate exclusions may need to be added for your particular environment.</p>
<div id="attachment_13946" class="wp-caption aligncenter"><img class="wp-image-13946" src="/globalassets/cyber-labs/detecting-poshc2--indicators-of-compromise/word-image-13.png?epslanguage=en" alt="Here we can see various processes that have loaded System.Management.Automation.dll into memory that are likely implants." width="802" height="545" aria-describedby="caption-attachment-13946" />

</div>
<h2>C# Implant</h2>
<p>The C# or Sharp implant is PoshC2&rsquo;s &lsquo;next gen&rsquo; implant, written (unsurprisingly) in C#. The key difference from a detection perspective is that this implant does not require loading<span>&nbsp;</span><code>System.Management.Automation.dll</code><span>&nbsp;</span>in order to function. Most of the functionality of the C# implant is custom-written and while it<span>&nbsp;</span><em>can</em><span>&nbsp;</span>load<span>&nbsp;</span><code>System.Management.Automation.dll</code><span>&nbsp;</span>in order to execute PowerShell, this is an operator decision and is by no means necessary.</p>
<p>A similar process to the above can be applied, but is a little harder to implement. .NET processes load the<span>&nbsp;</span><code>mscoree.dll</code><span>&nbsp;</span>library which is the core library behind the .NET CLR (Command Language Runtime), so again any unmanaged code processes that are loading this library could have been migrated into. The issue here is finding these processes, as it&rsquo;s not as simple as searching for just &lsquo;module loaded and process name is not powershell.exe&rsquo;.</p>
<p>A blacklist can be created of common target processes that are unmanaged, always available and should not be loading the .NET runtime, and these can be monitored, as well as noting this during manual triage.</p>
<div id="attachment_13947" class="wp-caption aligncenter"><img class="wp-image-13947" src="/globalassets/cyber-labs/detecting-poshc2--indicators-of-compromise/word-image-14.png?epslanguage=en" alt="Implant loads mscoree.dll if it is not already present in the process &ndash; another IoC if the process is supposed to be unmanaged." width="667" height="322" aria-describedby="caption-attachment-13947" />
<p id="caption-attachment-13947" class="wp-caption-text">A C# implant loads mscoree.dll if it is not already present in the process &ndash; another IoC if the process is supposed to be unmanaged.</p>
</div>
<p>Note that the PowerShell implant will also load this library as it is also a .NET process, however the presence of<span>&nbsp;</span><code>System.Management.Automation.dll</code><span>&nbsp;</span>marks it as an implant that can run PowerShell.</p>
<p>The C# implant has the ability to load compiled binaries into memory over C2 and run them, which is extremely powerful. There are indicators of this however, as a new virtual runspace is created for each module and their namespace is listed in the AppDomain to which they are loaded.</p>
<p>Viewing the .NET Assemblies in Process Explorer or Process Hacker for example then reveals what modules have been run.</p>
<div id="attachment_13948" class="wp-caption aligncenter"><img class="wp-image-13948" src="/globalassets/cyber-labs/detecting-poshc2--indicators-of-compromise/word-image-15.png?epslanguage=en" alt="In the C# implant loaded modules have their namespace visible in the AppDomain, making it clear what has been loaded into the implant." width="667" height="404" aria-describedby="caption-attachment-13948" />
<p id="caption-attachment-13948" class="wp-caption-text">In the C# implant loaded modules have their namespace visible in the AppDomain, making it clear what has been loaded into the implant.</p>
</div>
<p>Above we can see the<span>&nbsp;</span><em>Core</em><span>&nbsp;</span>and<span>&nbsp;</span><em>dropper_cs</em><span>&nbsp;</span>modules that make up the core functionality of the C# implant, as well as some native modules from Microsoft that are required to run. These modules will always be present in the PoshC2 C# implant and are a clear IoC. We also see<span>&nbsp;</span><a href="https://github.com/GhostPack/Seatbelt" target="_blank" rel="noopener noreferrer">Seatbelt</a><span>&nbsp;</span>and<span>&nbsp;</span><a href="https://github.com/GhostPack/SharpUp" target="_blank" rel="noopener noreferrer">SharpUp</a>, two common C# offensive modules from SpectreOps, and we can surmise that they have been on the target.</p>
<h2>In General</h2>
<h3>Migration</h3>
<p>We have used<span>&nbsp;</span><code>netsh.exe</code><span>&nbsp;</span>as the example implant process in this post and that is for a good reason. The default migration process for PoshC2 in the C# and PowerShell implants is<span>&nbsp;</span><code>C:\Windows\System32\netsh.exe</code>, so when the<span>&nbsp;</span><code>migrate</code><span>&nbsp;</span>or<span>&nbsp;</span><code>inject-shellcode</code><span>&nbsp;</span>commands are used and a specific process ID or name is not set, then a new<span>&nbsp;</span><code>netsh.exe</code><span>&nbsp;</span>process is spawned and migrated into.</p>
<p>This itself is a common IoC for PoshC2, as<span>&nbsp;</span><code>netsh.exe</code><span>&nbsp;</span>is not typically frequently run in environments, and certainly not on most end user&rsquo;s hosts. Therefore, if there is a sudden uptick in the number of these processes being run in the environment or if several are running on a host then it could be worth investigating.</p>
<h3>Persistence</h3>
<p>PoshC2 has three quick persistence commands available to the PowerShell implant. Each of these installs a<span>&nbsp;</span><code>PowerShell.exe</code><span>&nbsp;</span>one liner payload to the registry in the key at<span>&nbsp;</span><code>HKCU\Software\Microsoft\Windows\CurrentVersion\themes</code><span>&nbsp;</span>with a name<span>&nbsp;</span><em>Wallpaper777</em>,<span>&nbsp;</span><em>Wallpaper555</em><span>&nbsp;</span>or<span>&nbsp;</span><em>Wallpaper666</em>, depending on the command being run.</p>
<p>This payload is then triggered by either:</p>
<ul>
<li>A registry key at<span>&nbsp;</span><code>HKCU\Software\Microsoft\Windows\CurrentVersion\run</code><span>&nbsp;</span>with the name<span>&nbsp;</span><code>IEUpdate</code></li>
<li>A Scheduled Task, also with the name<span>&nbsp;</span><code>IEUpdate</code></li>
<li>A shortcut file placed at<span>&nbsp;</span><code>%APPDATA%\Microsoft\Windows\Start Menu\Programs\Startup\IEUpdate.lnk</code></li>
</ul>
<p>All of these can be alerted upon and used to determine that the adversary using PoshC2, in addition to alerting on the invocation of<span>&nbsp;</span><code>PowerShell.exe</code><span>&nbsp;</span>with encoded parameters, and so on.</p>
<h3>Binary payloads</h3>
<p>Some of the more common payloads that are dropped on targets are the PoshC2 executables and DLLs that can be run using<span>&nbsp;</span><code>rundll32.exe</code>.</p>
<p>For the DLLs, there are different versions for PowerShell and Sharp implants across versions 2 and 4 of PowerShell and x86 and x64 bit architectures, however all the DLLs have a single entry-point common to all:<span>&nbsp;</span><code>VoidFunc</code>.</p>
<div id="attachment_13949" class="wp-caption aligncenter"><img class="wp-image-13949" src="/globalassets/cyber-labs/detecting-poshc2--indicators-of-compromise/word-image-16.png?epslanguage=en" alt="All the DLL payloads have the same single entry point of VoidFunc." width="583" height="406" aria-describedby="caption-attachment-13949" />
<p id="caption-attachment-13949" class="wp-caption-text">All the DLL payloads have the same single entry point of VoidFunc.</p>
</div>
<p>This entry-point is hard-coded in PoshC2 and cannot be changed without hacking the compiled binary itself.</p>
<p>For the common executable and DLL payloads we&rsquo;ve also added Yara rules for detecting them. These are based on signaturable parts of the binaries that will not change across different installs of PoshC2, for example with different comms options. These are also available in the new<span>&nbsp;</span><a href="https://github.com/nettitude/PoshC2_IOCs" target="_blank" rel="noopener noreferrer">PoshC2 Detections GitHub repository</a>.</p>
<h1>In Summary</h1>
<p>We&rsquo;ve looked at a few different detections for catching PoshC2 when used out-of-the-box. Using these in your environment will help protect against the less sophisticated users of PoshC2 in addition to further understanding how the tool works.</p>
<p>Any new detections or amendments can be added to the<span>&nbsp;</span><a href="https://github.com/nettitude/PoshC2_IOCs" target="_blank" rel="noopener noreferrer">https://github.com/nettitude/PoshC2_IOCs</a><span>&nbsp;</span>repository and we encourage the community to add their own detections or rules and configurations for other security tools to help build a centralised data store for everyone.</p>
