<h2><span style="font-weight:400">Key Points</span></h2>
<ul>
  <li style="font-weight:400" aria-level="1">
    <span style="font-weight:400"
      >Lacework Labs identified new samples and infrastructure associated with
      HCRootkit / Sutersu Linux rootkit activity, building-off its recent </span
    ><a
      href="https://twitter.com/AvastThreatLabs/status/1430527767855058949?s=20"
      ><span style="font-weight:400">initial identification</span></a
    ><span style="font-weight:400"> from our colleagues at Avast.&nbsp;</span>
  </li>
  <li style="font-weight:400" aria-level="1">
    <span style="font-weight:400"
      >Malicious droppers include and deliver additional files, a kernel module,
      and userland ELF. These files compromise a host with standard rootkit
      functionality.</span
    >
  </li>
  <li style="font-weight:400" aria-level="1">
    <span style="font-weight:400"
      >The main agent uses a unique custom protobuf based protocol for C2
      communication.&nbsp;</span
    >
  </li>
</ul>
<h2><span style="font-weight:400">Summary</span></h2>
<p>
  <span style="font-weight:400"
    >Lacework Labs recently examined a new publicly shared rootkit, identifying
    its core capabilities and level of threat it represents to Linux hosts. The
    rootkit was first shared by Avast, triggering us to confirm coverage and
    investigate further. Our analysis below provides insight into the installer
    (droppers), in addition to the Kernel module and userland samples dropped.
    Our objective with this blog is to build on top of the findings from Avast,
    share our analysis, and provide defenders with detection options in the form
    of Yara rules and IOCs.&nbsp;</span
  >
</p>

<h2><span style="font-weight:400">The Dropper</span></h2>
<p>
  <span style="font-weight:400">The ELF dropper (</span
  ><i
    ><span style="font-weight:400"
      >602c435834d796943b1e547316c18a9a64c68f032985e7a5a763339d82598915</span
    ></i
  ><span style="font-weight:400"
    >) is a modified version of the coreutils “kill” binary. The majority of the
    “kill” binary’s core functionality remains the same, but with the addition
    of writing two ELF files to disk during execution. One of these components
    is a userland binary and the other a kernel module (</span
  ><i
    ><span style="font-weight:400"
      >10c7e04d12647107e7abf29ae612c1d0e76a79447e03393fa8a44f8a164b723d</span
    ></i
  ><span style="font-weight:400">) identified by Avast as the </span
  ><a href="https://github.com/mncoppola/suterusu"
    ><span style="font-weight:400">Sutersu rookit</span></a
  ><span style="font-weight:400"
    >.&nbsp; Notably, figure – 0 and figure 1 show the the ELF dropper and
    kernel rootkit had low or non-existent detection rates on
    VirusTotal.&nbsp;</span
  >
</p>

<p></p>
<p>
  <span style="font-weight:400"
    >The rootkit is written to disk first after a temporary filename has been
    generated via the </span
  ><a href="https://man7.org/linux/man-pages/man3/mktemp.3.html"
    ><i><span style="font-weight:400">mktemp</span></i></a
  ><span style="font-weight:400"> system call. After writing 20224 bytes (</span
  ><i><span style="font-weight:400">0x4f00</span></i
  ><span style="font-weight:400"
    >) to the temporary file, the file descriptor is closed and then the </span
  ><a href="https://man7.org/linux/man-pages/man8/insmod.8.html"
    ><i><span style="font-weight:400">insmod</span></i></a
  ><span style="font-weight:400">
    utility is used to install this kernel module. Errors are subsequently&nbsp;
    ignored through stdout/stderr redirection to </span
  ><i><span style="font-weight:400">/dev/null</span></i
  ><span style="font-weight:400">. If </span
  ><a href="https://wiki.ubuntu.com/UEFI/SecureBoot"
    ><span style="font-weight:400">secure boot</span></a
  ><span style="font-weight:400"> is enabled on the underlying system (</span
  ><i
    ><span style="font-weight:400"
      >which would require signed kernel modules</span
    ></i
  ><span style="font-weight:400"
    >) or the kernel version is not what the kernel module was compiled for, </span
  ><i><span style="font-weight:400">insmod</span></i
  ><span style="font-weight:400"> will fail. Finally, the </span
  ><a href="https://man7.org/linux/man-pages/man1/dmesg.1.html"
    ><span style="font-weight:400">dmesg</span></a
  ><span style="font-weight:400">
    utility is used to clear the dmesg output (</span
  ><a href="https://attack.mitre.org/techniques/T1070/002/"
    ><span style="font-weight:400">T1070</span></a
  >,<span style="font-weight:400"
    >) which would contain forensic artifacts of a kernel module being installed
    as well as remove the underlying ELF binary via </span
  ><a href="https://man7.org/linux/man-pages/man2/unlink.2.html"
    ><i><span style="font-weight:400">unlink</span></i></a
  ><span style="font-weight:400">.</span>
</p>

<h2>&nbsp;</h2>
<p>
  <span style="font-weight:400"
    >After writing the kernel module to disk, the embedded userland component is
    written to either </span
  ><i><span style="font-weight:400">/proc/.inl</span></i
  ><span style="font-weight:400"> or</span
  ><i><span style="font-weight:400"> /tmp/.tmp_XXXXXX</span></i
  ><span style="font-weight:400">
    depending on whether or not the open command succeeded for </span
  ><i><span style="font-weight:400">/proc/.inl</span></i
  ><span style="font-weight:400">.&nbsp;</span>
</p>
<p>
  <span style="font-weight:400"
    >Given the underlying backdoor coreutils utility is kill, it is not uncommon
    for the legitimate usage of this utility to be executed via “</span
  ><i><span style="font-weight:400">sudo kill</span></i
  ><span style="font-weight:400"
    >” when terminating privileged processes. Executing with sudo results in
    appropriate permissions to both install the kernel module and write to the
    privileged location in </span
  ><i><span style="font-weight:400">/proc/</span></i
  ><span style="font-weight:400"
    >.&nbsp; After writing the file, the file descriptor is closed and the
    binary is executed via the </span
  ><a href="https://man7.org/linux/man-pages/man3/system.3.html"
    ><i><span style="font-weight:400">system</span></i></a
  ><span style="font-weight:400"> syscall followed by deletion via the </span
  ><a href="https://man7.org/linux/man-pages/man2/unlink.2.html"
    ><i><span style="font-weight:400">unlink</span></i></a
  ><span style="font-weight:400">
    syscall. This behavior can be seen in the following figure below.</span
  >
</p>

<h2><span style="font-weight:400">The Rootkit – Sutersu</span></h2>
<p>
  <span style="font-weight:400"
    >The kernel module as pointed out by Avast is the open-source rootkit
    “</span
  ><a href="https://github.com/mncoppola/suterusu"
    ><span style="font-weight:400">Sutersu</span></a
  ><span style="font-weight:400"
    >”. This rootkit has wide kernel version support, as well as supporting
    multiple architectures including x86, x86_64, and ARM. Sutersu supports
    file, port, and process hiding, as one would expect from a rootkit. Sutersu
    also supports functionality beyond process and file hiding in the form of
    additional modules that are specified during compile time.</span
  >
</p>
<p>
  <span style="font-weight:400"
    >At the time of this writing, these additional modules include a keylogger,
    a module to download and execute (DLEXEC) a binary upon a given event, and
    an ICMP module to monitor for specific “magic bytes” before triggering an
    event. The DLEXC and ICMP module can be used together to trigger the
    downloading and execution of a binary when a specific ICMP packet is
    received. They also can be used independently. Lacework labs identified
    multiple Sutersu kernel modules with various modules and external
    IPs.&nbsp;</span
  >
</p>
<p>
  <span style="font-weight:400"
    >One variant of Sutersu identified within a dropper ELF containing the ICMP
    module that watches for incoming ICMP packets to then trigger further
    actions is shown as hiding any outbound connections to a given address.
    Figure – 4 below shows hardcoded IPv4 addresses identified within the
    Sutersu KO file. The “127.0.0.1” corresponds to a sshd server setup command
    within the userland ELF binary shown in figure -X.&nbsp;</span
  >
</p>

    masquerade under the process name “</span
  ><i><span style="font-weight:400">[kthread]</span></i
