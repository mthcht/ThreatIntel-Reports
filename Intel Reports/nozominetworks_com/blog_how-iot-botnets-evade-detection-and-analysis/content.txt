src="https://cdn.prod.website-files.com/645a45d56fc4750d4edd96fe/646273c8f8f4cde2557792e3_2022-03-08-Blog-Botnet-Anti-Analysis-Techniques-BLOG.jpeg" loading="lazy" alt="How IoT Botnets Evade Detection and Analysis - Part 1" class="post-featured-image"/><h1 class="post-title">How IoT Botnets Evade Detection and Analysis - Part 1</h1><div class="post-author-date-wrapper"><div class="post-author-text-byline">by </div><div class="w-dyn-list"><div role="list" class="author-collection-list w-dyn-items"><div role="listitem" class="author-collection-list-item w-dyn-item"><div class="post-author-text-byline post-author-text-byline-author">Nozomi Networks Labs</div></div></div></div><div class="post-author-text-byline post-author-text-byline-separator"> | </div><div class="post-author-text-byline post-author-text-byline-date">March 1, 2022</div></div><div class="post-body-rich-text-block blog-tich-text w-richtext"><h3>On the Hunt for Elusive Malware</h3><p>Malware families targeting the IoT sector share multiple similarities with their Windows-based counterparts, such as trying to evade detection by both researchers and security products. Their aim is to stay below the radar as long as possible. One key technique to stymie reverse engineering botnet code is to obfuscate the code by compressing or encrypting the executable, called packing.</p><p>In this post, we explore the current status of the packers used by IoT malware, using data collected by Nozomi Networks honeypots. Weâll also dig deeper into various obstacles that researchers face when analyzing obfuscated samples and show how they can be handled.</p><h3>IoT Botnet Attack Statistics</h3><p>Honeypots are vulnerable systems that are deliberately made available to adversaries so that information on malicious activity can be collected for research, detection, and defensive purposes. Nozomi Networks Labs runs a network of passive honeypots across various regions that emulate multiple protocols commonly abused by attackers. The data they collect is aggregated and used for research and detection efforts.</p><p>Over the course of a seven-day period, Nozomi Networks honeypots collected 205 unique samples from attacks. The vast majority of the collected files are Linux executable files for 32-bit architectures.</p><figure class="w-richtext-align-center w-richtext-figure-type-image"><div><img src="https://cdn.prod.website-files.com/645a45d56fc4750d4edd96fe/646be1a5eb25554db1a89497_1-files-collected-by-honeypots-768x475.webp" loading="lazy" alt="Statistics of the different file types collected by our honeypots"/></div><figcaption>Statistics of the different file types collected by our honeypots.</figcaption></figure><p>Attackers use packers to obfuscate their code, concealing the original code with the intent of evading detection and making malware analysis more difficult and time consuming. Of the samples we collected, approximately one third were packed using multiple versions of UPX, a free and open-source packer widely used by both legitimate companies and malicious actors. UPX was the only packer used in this set of samples.</p><figure class="w-richtext-align-center w-richtext-figure-type-image"><div><img src="https://cdn.prod.website-files.com/645a45d56fc4750d4edd96fe/646be1c3ce622fc3c2ce18e4_2-samples-collected-by-honeypots.webp" loading="lazy" alt="Percentage breakdown of UPX-packed samples and unpacked samples collected by the honeypots"/></div><figcaption>Percentage breakdown of UPX-packed samples and unpacked samples collected by the honeypots.</figcaption></figure><p>At the time of the analysis, 49 of 205 samples were not present on <a href="https://www.virustotal.com/gui/home/upload" target="_blank">VirusTotal</a>, a share site for malware details and research, and thus we decided to focus on these potentially new threats. This subset of files follows a similar percentage distribution regarding the packer and the version employed. Most of the new filesâalmost 80%âwere not packed at all, with the remainder packed with UPX.</p><p>Based on initial research, one of these samples appears to belong to the same malware family that showed up in other internal Threat Intelligence research. It also stands out because of the particular UPX packing structure modifications.</p><figure class="w-richtext-align-center w-richtext-figure-type-image"><div><img src="https://cdn.prod.website-files.com/645a45d56fc4750d4edd96fe/646be1d842e1d8cf7da6c392_3-samples-not-present-virus-total.webp" loading="lazy" alt="Of new samples not present on VirusTotal, 80% were not packed, while the rest were packed with UPX"/></div><figcaption>Of new samples not present on VirusTotal, 80% were not packed, while the rest were packed with UPX.</figcaption></figure><h3>Unpacking Challenges</h3><p>When a sample is only packed with UPX, it is very easy to unpack with the standard UPX tool using the -d command line argument. Therefore, attackers will commonly modify the UPX structures of a packed sample so that it remains executable, but the standard UPX tool can no longer recognize and unpack it.</p><p>Since UPX is an open-source tool, we can check its source code on GitHub to understand its structures and what fields it uses. (You can see more of its file structure <a href="https://github.com/upx/upx/blob/d7ba31cab8ce8d95d2c10e88d2ec787ac52005ef/src/stub/src/include/linux.h#L721" target="_blank">here</a>.)</p><figure class="w-richtext-align-center w-richtext-figure-type-image"><div><img src="https://cdn.prod.website-files.com/645a45d56fc4750d4edd96fe/646be1e6286d785cdad63860_4-UPX-file-structures-620x506.webp" loading="lazy" alt="Sample UPX file structure."/></div><figcaption>Sample UPX file structure.</figcaption></figure><p>Most IoT samples packed with UPX modify the l_info and p_info structs in the header. For example, as we have seen before with the SBIDIOT malware, itâs common for malware authors to modify the l_magic value of the l_info struct in UPX-packed samples. In this case, unpacking the sample is as trivial as replacing the modified l_magic value with UPX! and executing upx -d.</p><p>In other cases, like the <a href="https://threatpost.com/mozi-botnet-majority-iot-traffic/159337/" target="_blank">Mozi </a>IoT malware family, the p_info struct is modified to set p_filesize and p_blocksize to zero. The solution then involves repairing the two zeroed values by replacing them with the filesize value that is available at the trailer of the binary.</p><p>However, when we tried to unpack the samples of interest, UPX returned an unusual error:</p><figure class="w-richtext-align-center w-richtext-figure-type-image"><div><img src="https://cdn.prod.website-files.com/645a45d56fc4750d4edd96fe/646be1f4d1fc4a0a19f4da09_5-malware-sample-fails.webp" loading="lazy" alt="Malware sample fails while testing to determine if it can be unpacked with UPX"/></div><figcaption>Malware sample fails while testing to determine if it can be unpacked with UPX.</figcaption></figure><p>In this case, UPX is telling us that there was a problem with the b_info structure. After some research, we concluded that this structure doesnât seem to be widely used by malware authors. b_info is a structure placed before each compressed block and contains information about its compressed and uncompressed size, along with the algorithm and parameters used to compress them. Once we checked the b_info structure in the file, we realized it hadnât been zeroed or modified in an obvious way.</p><p>Diving deeper into the internals of UPX, we found <a href="https://github.com/upx/upx/blob/devel/src/p_unix.cpp" target="_blank">the exact place in the code</a> that raises this exception. If the compressed size (sz_cpr) is bigger than the uncompressed size (sz_unc), UPX will fail. However, their values were coherent, so we discarded this modification as the source of the problem. In <a href="https://github.com/upx/upx/blob/devel/src/p_unix.cpp#L514" target="_blank">these lines of code</a>, we can see that the most promising reason could be a problem with a difference between the sum of the declared size of the uncompressed sections and the declared size of the uncompressed file. In our sample, the sum of the value of sz_unc was bigger than the value of p_filesize, so we modified the appropriate p_info structure to set its p_filesize field with a value that wouldnât trigger this exception.</p><p>After changing these values, a header checksum exception was raised. Calculating this checksum value was possible since we had its source code, but it would be time-consuming, so we temporarily moved to another research path. We decided to create packed samples that were as similar as possible to the malicious sample so it would be easier to spot the differences.</p><p>With the help of upx --fileinfo we got the parameters needed to pack another executable with almost the same compression and decompression algorithm.</p><figure class="w-richtext-align-center w-richtext-figure-type-image"><div><img src="https://cdn.prod.website-files.com/645a45d56fc4750d4edd96fe/646be2016d96e736552cc624_6-extracting-compression-information-malware-sample-768x128.webp" loading="lazy" alt="Extracting compression information from a malware sample"/></div><figcaption>Extracting compression information from a malware sample.</figcaption></figure><p>To compress the sample, the attackers used a command similar to upx --best --nrv2d &lt;elf_file&gt;. As a starting point to check differences, we used the rz-diff tool to compare the main decompression functions:</p><figure class="w-richtext-align-center w-richtext-figure-type-image"><div><img src="https://cdn.prod.website-files.com/645a45d56fc4750d4edd96fe/646be2126d96e736552cdc0f_7-comparing-unpacking-methods-768x405.webp" loading="lazy" alt="Creating an executable with similar packing and comparing unpacking functions"/></div><figcaption>Creating an executable with similar packing and comparing unpacking functions.</figcaption></figure><p>We started comparing the differences between the functions, looking for the code we were thinking the attackers added to the decompression process. An unexpected difference appeared:</p><figure class="w-richtext-align-center w-richtext-figure-type-image"><div><img src="https://cdn.prod.website-files.com/645a45d56fc4750d4edd96fe/646be2280db810b1412d84df_8-malware-sample-packed-UPX-4-0-0-620x245.webp" loading="lazy" alt="Malware sample packed using UPX 4.0.0"/></div><figcaption>Malware sample packed using UPX 4.0.0.</figcaption></figure><p>At the moment, the stable UPX version is v3.96, and version 4.0.0 is in development. Changelog doesnât seem to contain big changes in how ELF compression works, but there are a lot of commits that affect portions of the code involved in the calculation of these values.</p><p>We then checked how this new version handled the issues we were seeing by downloading the pre-release version of UPX and compiling it. After only fixing the UPX! strings headers (b_info and p_info structures were left untouched) and passing this executable to UPX version 4.0.0, the sample was accurately decompressed.</p><figure class="w-richtext-align-center w-richtext-figure-type-image"><div><img src="https://cdn.prod.website-files.com/645a45d56fc4750d4edd96fe/646be2360db810b1412dcecf_9-successful-extraction-620x142.webp" loading="lazy" alt="Successful extraction with UPX 4.0.0 (commit a46b63)"/></div><figcaption>Successful extraction with UPX 4.0.0 (commit a46b63).</figcaption></figure><p>It is possible that the attackers realized that this version of UPX (which is still in development) generates functional samples that cannot be extracted by standard production versions of UPX versions used by default by everyone.</p><h3>Universal Manual Unpacking</h3><p>Instead of digging deeper into the modifications introduced by attackers, there is another approach we can consider. The idea here is to stop the debugging process when the code and data are already unpacked but the unpacked code hasnât been executed yet, to prevent the subsequent possible data and code modifications, and write down the unpacked code and data back to the disk. This approach is widely used to unpack Windows samples, but what about IoT threats? From a high-level perspective, the same logic can certainly be applied to them.</p><p>Here are several universal techniques that allow us to circumvent packers regardless of what modifications are introduced. They generally involve relying on the steps that an unpacker has to do in order to achieve its goal, mainly:</p><ol role="list"><li>The packed code and data should be read and unpacked</li><li>A big memory block should be available to host the resulting unpacked sample</li><li>The result of the unpacking should be written into this big memory block</li><li>Depending on the existing protection flags for this block, they may need to be adjusted to make code execution possible</li><li>Finally, the control should be transferred to the first instruction of the unpacked code known as the Original Entry Point (OEP)</li></ol><p>So, how can these techniques help us unpack the samples?</p><ol role="list"><li>Generally, packed code and data have high entropy and can therefore be easily identified in hex editors. Finding these blocks and tracking their cross-references can help us find the decryption/decompression/decoding routine(s).</li></ol><figure class="w-richtext-align-center w-richtext-figure-type-image"><div><img src="https://cdn.prod.website-files.com/645a45d56fc4750d4edd96fe/646be2477cef76f1734e2f07_10-high-entropy-block-768x404.webp" loading="lazy" alt="Packed Data"/></div></figure><p>2. Keeping track of memory allocations (mmap syscall) may help us find the future virtual address of the unpacked code and data.</p><figure class="w-richtext-align-center w-richtext-figure-type-image"><div><img src="https://cdn.prod.website-files.com/645a45d56fc4750d4edd96fe/646be25e7cf308c7374d21a0_11-mmap-syscall-620x323%20(1).webp" loading="lazy" alt="mmap syscall (rax = 0x09) with a big memory block "/></div><figcaption>mmap syscall (rax = 0x09) with a big memory block length requested.</figcaption></figure><p>3. Memory or hardware breakpoint on write operation set on the allocated block will help us intercept the moment when the unpacked code and data of interest will be written there.</p><figure class="w-richtext-align-center w-richtext-figure-type-image"><div><img src="https://cdn.prod.website-files.com/645a45d56fc4750d4edd96fe/646be26d286d785cdad7068a_12-hardware-breakpoint-620x315.webp" loading="lazy" alt="Setting a hardware breakpoint on write operation in IDA"/></div><figcaption>Setting a hardware breakpoint on write operation in IDA.</figcaption></figure><p>4. Keeping an eye on the mprotect syscall, which is commonly used to change protection flags, can help identify the moment when this will happen.</p><figure class="w-richtext-align-center w-richtext-figure-type-image"><div><img src="https://cdn.prod.website-files.com/645a45d56fc4750d4edd96fe/646be285a3452ca60c927776_13-mprotect-syscall-620x184.webp" loading="lazy" alt="mprotect syscall followed by an unusual control flow instruction"/></div><figcaption>mprotect syscall followed by an unusual control flow instruction.</figcaption></figure><p>5. Looking for unusual control flow instructions can help identify the moment when the unpacker finished its job and is ready to transfer control to the first instruction of the freshly unpacked code (OEP).</p><figure class="w-richtext-align-center w-richtext-figure-type-image"><div><img src="https://cdn.prod.website-files.com/645a45d56fc4750d4edd96fe/646be2b3d1fc4a0a19f5a653_14-unusual-control-flow-instruction.webp" loading="lazy" alt="The final unusual control flow instruction leading to the OEP."/></div><figcaption>The final unusual control flow instruction leading to the OEP.</figcaption></figure><p>Following these approaches separately or in combination eventually helps intercept the moment when the unpacked code and data finally reside in memory readily available to be dumped to the disk for subsequent static analysis.</p><p>In addition to these techniques, calling munmap syscall next to transferring control to the OEP is another feature of UPX that allows researchers to quickly unpack such samples. They can simply intercept it and then follow the execution flow</p><figure class="w-richtext-align-center w-richtext-figure-type-image"><div><img src="https://cdn.prod.website-files.com/645a45d56fc4750d4edd96fe/646be2c2a3452ca60c92b6b2_15-munmap-syscall-768x256.webp" loading="lazy" alt="munmap syscall (rax = 0x0B) executed next to transferring control to the OEP"/></div><figcaption>munmap syscall (rax = 0x0B) executed next to transferring control to the OEP.</figcaption></figure><h2><strong>Conclusions</strong> </h2><p>The landscape of malware targeting the IoT sector keeps evolving and borrowing many features from the more well-known IT sector. Staying up-to-date with the latest trends in this area and being able to handle them helps the security community combat new threats more efficiently and reduces the potential impact of associated cyberattacks. </p><p>You can read additional research in <a href="https://www.nozominetworks.com/blog/how-iot-botnets-evade-detection-and-analysis-part-2/" target="_blank">Part 2</a> of this blog.</p><p><br/></p></div><div class="related-resources-wrapper related-blog-wrapper-1 w-condition-invisible"><div class="w-dyn-list"><div class="w-dyn-empty"><div>No items found.</div></div></div></div><div class="related-resources-wrapper related-resources-wrapper-1 w-condition-invisible"><div class="w-dyn-list"><div class="w-dyn-empty"><div>No items found.</div></div></div></div><div class="related-resources-wrapper related-blog-wrapper-1"><div class="w-dyn-list"><div role="list" class="w-dyn-items"><div role="listitem" class="w-dyn-item"><div class="related-resources-item-wrapper"><img alt="" loading="lazy" width="300" height="178" src="https://cdn.prod.website-files.com/645a45d56fc4750d4edd96fe/646279e97b4ceb91b269f982_NN-Security-Report-2021-July.jpeg" class="related-resources-image-thumbnail"/><div class="related-resources-title-link-wrapper"><h2 class="related-resource-item-header">OT/IoT Security Report -
