<h1 class="blog-banner__heading">
      DanaBot control panel revealed
  </h1>

    </div>
  </div>

  <div class="blog-content">
    <div class="blog-content__sharethis sharethis_toolbox sharethis_32x32_style">
      <div class="blog-content__sharethis_label sharethis__label">Share with your network!</div>
      <div class="blog-content__sharethis_buttons sharethis_buttons">
        <div class="sharethis-inline-share-buttons"></div>
        
<span class="addthis_button_subscribe at300b UNCONVERTED" title=Subscribe>
  <span class="at-icon-wrapper block-subscribe-button__trigger block-subscribe-button__addthis"></span>
</span>


      </div>
    </div>
        <div class="blog-content__metadata blog-content__metadata-author">
      <span class="blog-content__date">  <time datetime="2019-03-13T20:30:00Z">March 13, 2019</time>

</span>
        <span class="blog-content__author">
                Dennis Schwarz and Proofpoint Threat Insight Team
  
        </span>
    </div>
    <div class="node-full__body blog-content__body">  <p><strong>Overview</strong></p>

<p>Proofpoint researchers discovered and reported on the DanaBot banking malware in May 2018 [1]. In our October 2018 update [2], we speculated that DanaBot may be set up as a “malware as a service” in which one threat actor controls a global command and control (C&amp;C) panel and infrastructure system and then sells access to other threat actors known as affiliates. Affiliates then target and distribute DanaBot malware as they see fit. While analyzing a component of this infrastructure, we discovered an interesting graphical client application that we believe to be a control panel used by affiliates to access the global C&amp;C system. Once logged on to the system, they can configure and build their DanaBot malware; access infected devices; and sift through any stolen data including credentials, financial account information, and more.</p>

<p><strong>Control Panel Application</strong></p>

<p>Our current theory is that when an affiliate buys access to the DanaBot system, they are given the control panel application described here and a user account to the global C&amp;C system.</p>

<p>Like the malware, the control panel is written in the Delphi programming language. It has a compilation date of “2019-02-04 22:33:42” and an internal name of “Client.exe”. The application is mostly a graphical frontend in which inputs are formatted as commands that are sent to a backend C&amp;C server for processing. Once processed, the C&amp;C server sends back the results, which are then displayed by the application.</p>

<p>Figures 1 through 6 give a tour of the main components of the control panel. While a valid login is required to send and receive data to and from the backend C&amp;C server, the figures still illustrate some of the potential actions a DanaBot affiliate can execute via the control panel:</p>

<ul>
	<li>Login to a backend C&amp;C server (Figure 1)</li>
	<li>Build new DanaBot malware (Figure 2)</li>
	<li>See various statistics from infected devices (Figure 3)</li>
	<li>Configure various aspects of the malware (e.g., video recording of the screen, keylogging, and webinjects) (Figure 4)</li>
	<li>Search and view stolen information (e.g., credentials and financial account information) (Figure 5)</li>
	<li>Operate on infected devices (e.g., search for files, download files, execute commands, take a screenshot, and open a VNC session) (Figure 6)</li>
</ul>

<p><a href="/sites/default/files/dbf1big.png" target="_blank"><img src="/sites/default/files/dbf1small.png"></a></p>

<p><em>Figure 1: Control panel “Connect” tab</em></p>

<p><a href="/sites/default/files/dbf2big.png" target="_blank"><img src="/sites/default/files/dbf2small.png"></a></p>

<p><em>Figure 2: Control panel “Builds” button</em></p>

<p><a href="/sites/default/files/dbf3big.png" target="_blank"><img src="/sites/default/files/dbf3small.png"></a></p>

<p><em>Figure 3: Control panel “Stats” tab</em></p>

<p><a href="/sites/default/files/dbf4big.png" target="_blank"><img src="/sites/default/files/dbf4small.png"></a></p>

<p><em>Figure 4: Control panel “Config” tab</em></p>

<p><a href="/sites/default/files/dbf5_0.png" target="_blank"><img src="/sites/default/files/dbf5small.png"></a></p>

<p><em>Figure 5: Control panel “Logs” tab</em></p>

<p><a href="/sites/default/files/dbf6big.png" target="_blank"><img src="/sites/default/files/dbf6small.png"></a></p>

<p><em>Figure 6: Control panel “Online” tab</em></p>

<p><strong>Association with DanaBot Malware</strong></p>

<p>In addition to finding the control panel application on infrastructure closely tied to DanaBot, two other significant pieces of evidence tie this control panel application to the DanaBot malware:</p>

<ul>
	<li>C&amp;C protocol overlap</li>
	<li>Shared RSA public key</li>
</ul>

<p>In February 2019, a new version of the DanaBot malware was spotted in the wild that contained a new C&amp;C protocol. ESET researchers were the first to notice the update and published a blog post [3] detailing the changes. Since then all of the DanaBot affiliates into which we have visibility have switched to this new version.</p>

<p>Using ESET’s post as background, we can compare and contrast the network communications used in the control panel application (traffic generated when trying to login to a C&amp;C server - Figure 7) and the C&amp;C protocol used in the malware (initial beacon - Figure 8).</p>

<p><img src="/sites/default/files/dbf7small.png"></p>

<p><em>Figure 7: Control panel “login” request</em></p>

<p><img src="/sites/default/files/dbf8small.png"></p>

<p><em>Figure 8: DanaBot malware “initial beacon”</em></p>

<p>In both figures we can see two sets of communications each containing a 24-byte header (highlighted in red) followed by encrypted data (highlighted in blue).</p>

<p>The header contains:</p>

<ul>
	<li>Offset 0x0: length of data (QWORD)</li>
	<li>Offset 0x8: random value (QWORD)</li>
	<li>Offset 0x10: random value + length of data (QWORD)</li>
</ul>

<p>The encrypted data sections are composed of 3 pieces:</p>

<ul>
	<li>AES-256 encrypted data using a randomly generated key</li>
	<li>Padding length (DWORD)</li>
	<li>The randomly generated AES key that has been RSA encrypted using an embedded RSA public key</li>
</ul>

<p>In the first set of communications, the AES encrypted data contains a second RSA public key that is generated by the control panel application and malware. This second RSA key is used to decrypt data sent back from the C&amp;C server.</p>

<p>The second set of communications contains the initial commands “login command” for the control panel application and “initial beacon” for the malware. <p><code>-----BEGIN PUBLIC KEY-----</code></p>

<p><code>MIGfMA0GCSqGSIb3DQEBAQUAA4GNADCBiQKBgQCyJo2aXOQNP+KeAnWlpOiuMk5W</code></p>

<p><code>-----END PUBLIC KEY-----</code></p>

<p>This RSA public key has actually been used in all of the DanaBot malware samples we have observed since the upgrade in February. It is part of the reason we suspect that there is a single global C&amp;C panel with which all affiliate malware communicates.</p>

<p>In addition to the overlapping C&amp;C protocol and shared RSA key, the code in both the control panel and the malware share the same structure and style.</p>

<p><strong>Conclusion</strong></p>

<p>A stand-alone binary application through which affiliates access malware control panels is unusual, with malware developers generally opting for web-based control panels. Several factors, however, suggest that the application described here is used by DanaBot affiliates to build and configure their malware and then to access victim devices.</p>

<p>In either case, it is usually a careless OPSEC mistake by a threat actor or an intentional “leak” of the malware that exposes the control panel. Once exposed, however, they tend to provide useful insights into malware campaigns and a perspective usually hidden to defenders.</p>
