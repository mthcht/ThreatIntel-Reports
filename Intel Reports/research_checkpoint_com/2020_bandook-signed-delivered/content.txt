<h2 class="wp-block-heading"><br />Introduction</h2>



<p><style>
.wp-block-table td {
    font-family: monospace;
    padding-right: 5px
}
.wp-caption .wp-caption-text {
    text-align: center;
}
.wp-block-image .aligncenter>figcaption {
    text-align: center; 
}
</style></p>



<p class="wp-block-paragraph">Check Point Research recently observed a new wave of campaigns against various targets worldwide that utilizes a strain of a 13-year old backdoor Trojan named Bandook.</p>



<p class="wp-block-paragraph">Bandook, which had almost disappeared from the threat landscape, was featured in 2015 and 2017 campaigns, dubbed &#8220;<a href="https://www.eff.org/wp/operation-manul">Operation Manul</a>&#8221; and &#8220;<a href="https://www.eff.org/deeplinks/2018/01/dark-caracal-good-news-and-bad-news">Dark Caracal</a>&#8220;, respectively. These campaigns were presumed to be carried out by the Kazakh and the Lebanese governments, as uncovered by the Electronic Frontier Foundation (EFF) and Lookout.</p>



<p class="wp-block-paragraph">During this past year, dozens of digitally signed variants of this once commodity malware started to reappear in the threat landscape, reigniting interest in this old malware family.</p>



<p class="wp-block-paragraph">In the latest wave of attacks, we once again identified an unusually large variety of targeted sectors and locations. This further reinforces a previous hypothesis that the malware is not developed in-house and used by a single entity, but is part of an offensive infrastructure sold by a third party to governments and threat actors worldwide, to facilitate offensive cyber operations.</p>



<p class="wp-block-paragraph">In this publication, we showcase the latest evolution of the infection chain offered by this unknown third-party, compare the different Bandook variants, and share the various techniques its creators use to hinder analysis and detection of all the components in the attack flow.</p>



<h2 class="wp-block-heading"><br />Infection Chain</h2>



<p class="wp-block-paragraph">As the infection chain is constantly evolving, we describe the one used by the attackers from as early as July, to the present day.</p>



<p class="wp-block-paragraph">The full infection chain of the attack can be broken down into three main stages. The first stage starts, as in many other infection chains, with a malicious Microsoft Word document delivered inside a ZIP file. Once the document is opened, malicious macros are downloaded using the external template feature. The macros’ code in turn drops and executes the second stage of the attack, a PowerShell script encrypted inside the original Word document. Finally, the PowerShell script downloads and executes the last stage of the infection: the Bandook backdoor.</p>



<p class="has-text-align-center wp-block-paragraph">&#x1f4a1; <code>The names of the various artifacts described below may vary from one infection to the next.</code></p>



<div class="wp-block-image">
<figure class="aligncenter size-full"><img fetchpriority="high" decoding="async" width="1489" height="1146" class="wp-image-24406" src="//research.checkpoint.com/wp-content/uploads/2020/11/Capture2.png" alt="" srcset="https://research.checkpoint.com/wp-content/uploads/2020/11/Capture2.png 1489w, https://research.checkpoint.com/wp-content/uploads/2020/11/Capture2-300x231.png 300w, https://research.checkpoint.com/wp-content/uploads/2020/11/Capture2-1024x788.png 1024w, https://research.checkpoint.com/wp-content/uploads/2020/11/Capture2-768x591.png 768w" sizes="(max-width: 1489px) 100vw, 1489px" />
<figcaption>Figure 1: Full infection chain.</figcaption>
</figure>
</div>



<p class="wp-block-paragraph">&nbsp;</p>



<h2 class="wp-block-heading"><br />First Stage – Lure Documents</h2>



<p class="wp-block-paragraph">The first stage starts with a Microsoft Word document with embedded encrypted malicious script data and an external template that points to a document containing malicious VBA macros.</p>



<p class="wp-block-paragraph">The external template is downloaded via a URL shortening web service like <strong>TinyURL</strong> or <strong>Bitly</strong>, which redirects to another domain controlled by the attacker.</p>



<p class="wp-block-paragraph">The external template document contains a VBA code that runs automatically, decrypts the embedded data from the original lure document, and drops the decoded data into two files in the local user folder: <code>fmx.ps1</code> (the next stage PowerShell) and <code>sdmc.jpg</code> (base64 encoded PowerShell code).</p>



<p class="wp-block-paragraph">To allow this behavior, the attackers use a combination of two techniques: encrypted data is embedded inside a shape object within the original document (hidden from view by a small font size and white foreground), and is accessed from the external template code by using the following code:</p>



<p class="has-text-align-center wp-block-paragraph"><code>o4QQLW7zXjLbj = ActiveDocument.Shapes(1).TextFrame.TextRange.Text</code></p>



<p class="wp-block-paragraph">For proper analysis, both the original document and the external template must be located, which makes things a bit more difficult for investigators.</p>



<p class="wp-block-paragraph">We observed and analyzed multiple pairs of documents and external templates. Different lure images were used, alongside different encryption keys.</p>



<p class="wp-block-paragraph"><br /><strong>Examples of lure documents:</strong></p>



<div class="wp-block-image">
<figure class="aligncenter size-large"><img decoding="async" width="1024" height="296" class="wp-image-24407" src="//research.checkpoint.com/wp-content/uploads/2020/11/1-1-1024x296.png" alt="" srcset="https://research.checkpoint.com/wp-content/uploads/2020/11/1-1-1024x296.png 1024w, https://research.checkpoint.com/wp-content/uploads/2020/11/1-1-300x87.png 300w, https://research.checkpoint.com/wp-content/uploads/2020/11/1-1-768x222.png 768w, https://research.checkpoint.com/wp-content/uploads/2020/11/1-1.png 1537w" sizes="(max-width: 1024px) 100vw, 1024px" />

</figure>
</div>



<p class="wp-block-paragraph"><strong>Examples of external templates with macros:</strong></p>



<div class="wp-block-image">
<figure class="aligncenter size-large"><img decoding="async" width="1024" height="289" class="wp-image-24408" src="//research.checkpoint.com/wp-content/uploads/2020/11/2-2-1024x289.png" alt="" srcset="https://research.checkpoint.com/wp-content/uploads/2020/11/2-2-1024x289.png 1024w, https://research.checkpoint.com/wp-content/uploads/2020/11/2-2-300x85.png 300w, https://research.checkpoint.com/wp-content/uploads/2020/11/2-2-768x217.png 768w, https://research.checkpoint.com/wp-content/uploads/2020/11/2-2.png 1220w" sizes="(max-width: 1024px) 100vw, 1024px" />

<p class="wp-block-paragraph">The external templates are not visible to the victim. Their only purpose is to provide malicious macros.</p>



<p class="wp-block-paragraph">Interestingly, with each attack, after a certain amount of time, the attacker switched the malicious external template to a benign one, further muddying our analysis of the infection chain.</p>



<p class="wp-block-paragraph">Here again, the external templates look like random benign documents:</p>



<div class="wp-block-image">
<figure class="aligncenter size-large"><img loading="lazy" decoding="async" width="1024" height="306" class="wp-image-24409" src="//research.checkpoint.com/wp-content/uploads/2020/11/3-1-1024x306.png" alt="" srcset="https://research.checkpoint.com/wp-content/uploads/2020/11/3-1-1024x306.png 1024w, https://research.checkpoint.com/wp-content/uploads/2020/11/3-1-300x90.png 300w, https://research.checkpoint.com/wp-content/uploads/2020/11/3-1-768x230.png 768w, https://research.checkpoint.com/wp-content/uploads/2020/11/3-1.png 1200w" sizes="auto, (max-width: 1024px) 100vw, 1024px" />

<p class="wp-block-paragraph">The themes of the documents are often of cloud-based services like Office365, OneDrive and Azure that contain images of other documents supposedly available once the victim clicks &#8220;Enable Content.&#8221;</p>



<p class="wp-block-paragraph">For example, one of the documents that specifically got our attention depicts an Office365 logo and a preview of a certificate issued by the government of Dubai. <strong>JAFZA &#8211; Jebel Ali Free Zone</strong>, featured at the top of the document, is an industrial area surrounding the port of Jebel Ali in Dubai, where more than 7,000 global companies are based.</p>



<div class="wp-block-image">

<h2 class="wp-block-heading"><br />Second Stage – PowerShell Loader</h2>



<p class="wp-block-paragraph">After the VBA code drops the two files (<code>fmx.ps1</code> and <code>sdmc.jpg</code>), it invokes <code>fmx.ps1</code>.</p>



<p class="wp-block-paragraph"><code>fmx.ps1</code> is a short PowerShell script that decodes and executes a base64 encoded PowerShell stored in the second dropped file (<code>sdmc.jpg</code>).</p>



<p class="wp-block-paragraph">First, the decoded PowerShell script downloads a zip file containing four files from a cloud service such as Dropbox, Bitbucket or an S3 bucket. The zip file is stored in the user&#8217;s <code>Public</code> folder, and the four files are locally extracted.</p>



<div class="wp-block-image">
<figure class="aligncenter size-full"><img loading="lazy" decoding="async" width="545" height="242" class="wp-image-24398" src="//research.checkpoint.com/wp-content/uploads/2020/11/image-15.png" alt="" srcset="https://research.checkpoint.com/wp-content/uploads/2020/11/image-15.png 545w, https://research.checkpoint.com/wp-content/uploads/2020/11/image-15-300x133.png 300w" sizes="auto, (max-width: 545px) 100vw, 545px" />
<figcaption>Figure 6: Malware components stored on Dropbox.com.</figcaption>
</figure>
</div>



<div class="wp-block-image">
<figure class="aligncenter size-full"><img loading="lazy" decoding="async" width="549" height="236" class="wp-image-24399" src="//research.checkpoint.com/wp-content/uploads/2020/11/image-16.png" alt="" srcset="https://research.checkpoint.com/wp-content/uploads/2020/11/image-16.png 549w, https://research.checkpoint.com/wp-content/uploads/2020/11/image-16-300x129.png 300w" sizes="auto, (max-width: 549px) 100vw, 549px" />
<figcaption>Figure 7: Malware components after being extracted on the victim&#8217;s device.</figcaption>
</figure>

<p class="wp-block-paragraph">Three of the files, <code>a.png</code>, <code>b.png</code> and <code>untitled.png</code>, are used by the PowerShell script to generate the malware payload in the same folder. <code>untitled.png</code>, unlike the other two files, is in a valid image format. It contains a hidden RC4 function encoded in the RGB values of the pixels, created using a known tool named <a href="https://github.com/peewpw/Invoke-PSImage">invoke-PSImage</a>.</p>



<p class="wp-block-paragraph">The final executable payload is concatenated from the following files:</p>



<ul class="wp-block-list">
<li><code>a.png</code> &#8211; After it is decrypted using RC4 and stored as <code>aps.png</code>.</li>
<li><code>b.png</code> &#8211; As is.</li>
</ul>



<p class="wp-block-paragraph">Finally, the PowerShell script executes the malware, opens <code>draft.docx</code>, and deletes all previous artifacts from the <code>Public</code> folder.</p>



<p class="wp-block-paragraph"><code>draft.docx</code> is a benign document whose sole purpose is to convince the victim that the document is no longer available, and that the overall execution was successful.</p>



<div class="wp-block-image">
<figure class="aligncenter size-full"><img loading="lazy" decoding="async" width="343" height="441" class="wp-image-24400" src="//research.checkpoint.com/wp-content/uploads/2020/11/image-17.png" alt="" srcset="https://research.checkpoint.com/wp-content/uploads/2020/11/image-17.png 343w, https://research.checkpoint.com/wp-content/uploads/2020/11/image-17-233x300.png 233w" sizes="auto, (max-width: 343px) 100vw, 343px" />
<figcaption>Figure 8: Final document shown to the user post-infection.</figcaption>
</figure>
</div>



<h2 class="wp-block-heading">Third Stage – The Bandook Loader</h2>



<p class="wp-block-paragraph">The final payload in this infection chain is a variant of an old full-featured RAT named Bandook. Written in both Delphi and C++, Bandook has a long history, starting in 2007 as a commercially available RAT that was developed by a Lebanese individual nicknamed PrinceAli. Over time, several variants of the malware builder were leaked to the Web, and the malware became publicly available for download.</p>



<div class="wp-block-image">
<figure class="aligncenter size-full"><img loading="lazy" decoding="async" width="650" height="179" class="wp-image-24401" src="//research.checkpoint.com/wp-content/uploads/2020/11/image-18.png" alt="" srcset="https://research.checkpoint.com/wp-content/uploads/2020/11/image-18.png 650w, https://research.checkpoint.com/wp-content/uploads/2020/11/image-18-300x83.png 300w" sizes="auto, (max-width: 650px) 100vw, 650px" />
<figcaption>Figure 9: Bandook’s history described on a hacking forum.</figcaption>
</figure>
</div>



<p class="wp-block-paragraph">Bandook&#8217;s execution flow starts with a loader, written in Delphi, that uses the Process Hollowing technique to create a new instance of an Internet Explorer process and inject a malicious payload into it. The payload contacts the C&amp;C sever, sends basic information about the infected machine, and waits for additional commands from the server.</p>



<p class="wp-block-paragraph">The variant of the Bandook malware we observed in this attack was not one of the variants whose builder was previously leaked to the Web (which supported a range of more than 100 commands).</p>



<p class="wp-block-paragraph">In this attack, the threat actor utilized a custom, slimmed-down version of the malware with only 11 supported commands, including:</p>



<ul class="wp-block-list">
<li>File operations</li>
<li>Taking screenshots</li>
<li>File download</li>
<li>File upload</li>
<li>File execution</li>
</ul>



<p class="has-text-align-center wp-block-paragraph">&#x1f4a1; <code>For a full list of commands and their corresponding request codes, see Appendix A.</code></p>



<p class="wp-block-paragraph">In this version, the communication protocol with the C&amp;C server was also upgraded  to use AES encryption.</p>



<h2 class="wp-block-heading"><br />Bandook variants in the wild</h2>



<p class="wp-block-paragraph">After comparing the Bandook variant we observed in the attack with the ones created by different leaked builders, we began hunting for variants more similar to the ones we observed.</p>



<p class="wp-block-paragraph">Our search led us to tweets by the <a href="https://twitter.com/malwrhunterteam">MalwareHunterTeam</a> (MHT) from 2019-2020 that mention various Bandook samples — all of them digitally signed with certificates that were issued by <strong>Certum</strong>.</p>



<div class="wp-block-image">
<figure class="aligncenter size-full"><img loading="lazy" decoding="async" width="518" height="424" class="wp-image-24402" src="//research.checkpoint.com/wp-content/uploads/2020/11/image-19.png" alt="" srcset="https://research.checkpoint.com/wp-content/uploads/2020/11/image-19.png 518w, https://research.checkpoint.com/wp-content/uploads/2020/11/image-19-300x246.png 300w" sizes="auto, (max-width: 518px) 100vw, 518px" />
<figcaption>Figure 10: Signed Bandook samples discovered by MHT.</figcaption>
</figure>
</div>



<p class="wp-block-paragraph">In the newer attack flows we observed, we once again found valid <strong>Certum </strong>certificates were used to sign the Bandook malware executable.</p>



<div class="wp-block-image">
<figure class="aligncenter size-full"><img loading="lazy" decoding="async" width="390" height="450" class="wp-image-24403" src="//research.checkpoint.com/wp-content/uploads/2020/11/image-20.png" alt="" srcset="https://research.checkpoint.com/wp-content/uploads/2020/11/image-20.png 390w, https://research.checkpoint.com/wp-content/uploads/2020/11/image-20-260x300.png 260w" sizes="auto, (max-width: 390px) 100vw, 390px" />
<figcaption>Figure 11: Valid signature information of a newly discovered Bandook sample.</figcaption>
</figure>
</div>



<p class="wp-block-paragraph">Analyzing all Bandook samples noted by MHT, we discovered that the very first of the samples was compiled in March 2019 and supported around 120 commands. <strong><strong>A sample compiled a few days later &#8211; a different signed Bandook variant (with only 11 commands) utilized the very same C&amp;C server</strong>.</strong> Since then, all signed samples use only 11 basic commands. The shared C&amp;C provides clear evidence that both the slimmed-down and the fully-fledged variants of the malware are operated by a single attacker.</p>



<p class="wp-block-paragraph">In addition to the Bandook samples that were reported by MHT, we identified additional samples from the same time period (2019-2020) which were <strong>not digitally signed</strong> and contained about 120 commands. These were the only ITW Bandook samples we were able to locate from this time period.</p>



<p class="wp-block-paragraph">Several factors led us to believe that these <strong>signed and unsigned</strong> variants are specially crafted Bandook variants, used and developed by the same entity.</p>



<ul class="wp-block-list">
<li>Both use the same domain registration services for their C&amp;C domains: Porkbun or NameSilo.</li>
<li>They share a similar method of communication, using the <strong>AES</strong> encryption algorithm in <strong>CFB</strong> mode, with a hardcoded IV: 0123456789123456. This feature is not available in the public leaks of this malware.</li>
<li>They incorporated commands that we did not observe in any other public leak or report. Most notable are the commands to execute <strong>Python</strong> and <strong>Java</strong> payloads.</li>
</ul>



<div class="wp-block-image">
<figure class="aligncenter size-full"><img loading="lazy" decoding="async" width="701" height="322" class="wp-image-24404" src="//research.checkpoint.com/wp-content/uploads/2020/11/1.png" alt="" srcset="https://research.checkpoint.com/wp-content/uploads/2020/11/1.png 701w, https://research.checkpoint.com/wp-content/uploads/2020/11/1-300x138.png 300w" sizes="auto, (max-width: 701px) 100vw, 701px" />
<figcaption>Figure 12: Bandook subroutine to execute a precompiled Python from a file named &#8220;dpx.pyc&#8221;.</figcaption>
</figure>
</div>



<p class="wp-block-paragraph"><strong>At this point, we have three different variants of the malware, which we believe are operated and sold by a single entity, in accordance with their chronological appearance:</strong></p>



<ol class="wp-block-list" type="1">
<li>A full-fledged version with 120 commands (not signed).</li>
<li>A full-fledged version (single sample) with 120 commands (signed).</li>
<li>A slimmed-down version with 11 commands (signed).</li>
</ol>



<p class="wp-block-paragraph">The move to a slimmed-down version with only 11 commands for signed executables may indicate the operators&#8217; desire to reduce the malware’s footprint and maximize their chances for an undetectable campaign against high profile targets (and high paying customers), while continuing the use of the un-signed 120 commands variant for lower profile ones.</p>



<p class="wp-block-paragraph">Furthermore, such a minimized backdoor might indicate that the slimmed-down variant of Bandook is only utilized as a loader for an additional, more full-featured malware to be downloaded next.</p>



<h2 class="wp-block-heading"><br />Targeting</h2>



<p class="wp-block-paragraph">As mentioned previously, in this campaign we observed an unusually large variety of targeted sectors and locations. This strengthens a hypothesis made by researchers &#8211; that the malware is not being developed and used by a single entity, but an offensive infrastructure is being sold by a third-party, to governments and threat actors worldwide, to facilitate offensive cyber operations.</p>



<p class="wp-block-paragraph"><strong>The different targeted sectors include:</strong></p>



<p class="wp-block-paragraph">Government, financial, energy, food industry, healthcare, education, IT and legal institutions.</p>



<p class="wp-block-paragraph"><strong>In the following countries:</strong></p>



<p class="wp-block-paragraph">Singapore, Cyprus, Chile, Italy, USA, Turkey, Switzerland, Indonesia and Germany.</p>



<h2 class="wp-block-heading"><br />Connection to Dark Caracal</h2>



<p class="wp-block-paragraph">This campaign isn&#8217;t the first instance of the Bandook malware incorporated in a targeted attack. As mentioned, in a joint <a href="https://info.lookout.com/rs/051-ESQ-475/images/Lookout_Dark-Caracal_srr_20180118_us_v.1.0.pdf">report</a> from Lookout and the EFF, targeted attacks utilizing a Bandook variant, called “Dark Caracal”, were attributed to the Lebanese General Security Directorate.</p>



<p class="wp-block-paragraph">Some of this campaign&#8217;s characteristics and similarities to previous campaigns leads us to believe that the activity we describe in this report is indeed the continuation and evolution of the infrastructure used during the Dark Caracal operation:</p>



<ul class="wp-block-list">
<li>The use of the same certificate provider (<strong>Certum</strong>) throughout the various campaigns.</li>
<li>The use of the Bandook Trojan, in what appears to be a unique evolving fork from the same source code (which is not known to be publicly available). Samples from the Dark Caracal campaign (2017) utilized around 100 commands, compared to the current 120 command version we analyzed.</li>
<li>This wave of attacks shares the same anomalous characteristics for targeted attacks &#8211;  an extreme variance in the selected targets, both in their industry and their geographic spread.</li>
</ul>



<p class="wp-block-paragraph">Finally, EFF researchers who first disclosed the Dark Caracal operation also believe that the same attacker &#8220;<a href="https://twitter.com/cooperq/status/1123114038211170305">is back at it</a>&#8221; again.</p>



<h2 class="wp-block-heading"><br />Conclusion</h2>



<p class="wp-block-paragraph">All evidence points to our belief that the mysterious operators behind the malicious infrastructure of &#8220;<a href="https://www.eff.org/wp/operation-manul">Operation Manul</a>&#8221; and &#8220;<a href="https://www.eff.org/deeplinks/2018/01/dark-caracal-good-news-and-bad-news">Dark Caracal</a>&#8221; are still alive and operational, willing to assist in the offensive cyber operations to anyone who is willing to pay.</p>



<p class="wp-block-paragraph">Although not as capable, nor as practiced in operational security like some other offensive security companies, the group behind the infrastructure in these attacks seems to improve over time, adding several layers of security, valid certificates and other techniques, to hinder detection and analysis of its operations.</p>



<p class="wp-block-paragraph"><a href="https://www.checkpoint.com/products/advanced-endpoint-protection/">Check Point SandBlast Agent</a> protects against this APT attack, and prevents it from the very first steps.</p>

function loadImg(element){
    var bg = '';
	switch (screenW) {
		case  'mobile':
			bg = element.getAttribute('data-bg-mobile');
			if(bg)
				break; 
		case 'tablet':
			bg = element.getAttribute('data-bg-tablet');
			if(bg)
				break; 
		case 'laptop':
			bg = element.getAttribute('data-bg-laptop');
			if(bg)
				break; 
		default: 
			bg = element.getAttribute('data-bg');
	}
	if(bg)
		element.style.backgroundImage = 'url(' + bg + ')';	
}

function calcScreenW(){
	var screenC = false;
	curentW= jQuery(window).width();	
	if ( curentW > 1450 ) { 
		if (screenW != 'desktop'){
			screenW = 'desktop';	
			screenC = true;
		}
	} else if (curentW > 973){
		if (screenW != 'laptop'){
			screenW = 'laptop';	
			screenC = true;
		}		
	} else if (curentW >767){
		if (screenW != 'tablet'){
			screenW = 'tablet';	
			screenC = true;
		}		
	} else {
		if (screenW != 'mobile'){
			screenW = 'mobile';	
			screenC = true;
		}		
	}
	if (screenC) {
		jQuery('.bg-change-done').removeClass('bg-change-done');
		screenC = false;
		animate_elems();
	}
}

var $elems = jQuery('.responsive-bg');
var winheight = jQuery(window).height();
jQuery(window).scroll(function(){
	animate_elems();
});
function animate_elems() {
	wintop = jQuery(window).scrollTop(); // calculate distance from top of window
	// loop through each item to check when it animates
	$elems.each(function(){
		$elm = jQuery(this);
		if($elm.hasClass('lazyload')|| $elm.hasClass('bg-change-done')) { return true; } // if already animated skip to the next item

		topcoords = $elm.offset().top; // element's distance from top of page in pixels

		if(wintop > (topcoords - (winheight))) {			
			$elm.addClass('bg-change-done');		
			loadImg($elm[0]);
		} 
	});	
}
