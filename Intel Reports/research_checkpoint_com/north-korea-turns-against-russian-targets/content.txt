<p><strong>Introduction</strong></p>
<p>Over the past few weeks, we have been monitoring suspicious activity directed against Russian-based companies that exposed a predator-prey relationship that we had not seen before. For the first time we were observing what seemed to be a coordinated North Korean attack against Russian entities. While attributing attacks to a certain threat group or another is problematic, the analysis below reveals intrinsic connections to the tactics, techniques and tools used by the North Korean APT group &#8211; Lazarus.</p>
<p>This discovery came about as we were tracking multiple malicious Office documents that were designed and crafted specifically for Russian victims. Upon closer examination of these documents, we were able to discern that they belonged to the early stages of an infection chain which ultimately led to an updated variant of a versatile Lazarus backdoor, dubbed <a href="https://www.us-cert.gov/ncas/analysis-reports/AR18-221A">KEYMARBLE</a> by the US-CERT.</p>
<p>Sometimes referred to as Hidden Cobra, Lazarus is one of the most prevalent and active APT groups in the world today. The infamous group, which is known to be a North Korean sponsored threat actor, is believed to be behind some of the largest security breaches of the last decade.</p>
<p>This includes the <a href="https://en.wikipedia.org/wiki/Sony_Pictures_hack" target="_blank" rel="noopener">Sony Pictures Entertainment hack</a>, the <a href="https://www.wired.com/2016/05/insane-81m-bangladesh-bank-heist-heres-know/" target="_blank" rel="noopener">Bangladesh bank heist</a>, and numerous other high stakes operations, such as the theft of millions of dollars worth in cryptocurrencies from at least five different <a href="https://www.coindesk.com/north-korean-hacking-group-lazarus-stole-571-million-in-cryptos-report" target="_blank" rel="noopener">cryptocurrency exchange</a> services worldwide.</p>
<p>While our campaign&#8217;s timeline seems to overlap with last week&#8217;s ESTsecurity <a href="https://blog.alyac.co.kr/2105" target="_blank" rel="noopener">report</a> on the &#8220;Operation Extreme Job&#8221; campaign targeting South Korean security companies, we have observed different tactics, techniques and procedures (TTPs) employed in the two operations.</p>
<p>It is long believed among the security community that Lazarus is divided into at least two subdivisions: the first named Andariel which focuses primarily on attacking the South Korean government and organizations, and the second, Bluenoroff, whose main focus is monetization and global espionage campaigns.</p>
<p>The differences between the two campaigns, which were conducted at the same time, provides wind once again to the theory that multiple divisions are at work here.</p>
<p>This incident, however, represents an unusual choice of victim by the North Korean threat actor. Usually, these attacks reflect the geopolitical tensions between the DPRK and nations such as the U.S, Japan and South Korea. In this case, though, it is probably Russian organizations who are the targets.</p>
<p>&nbsp;</p>
<p><strong>Infection Chain</strong></p>
<p>During our analysis we encountered two different infection flows.</p>
<p>The main infection flow consists of the following three main steps:</p>
<ol>
<li>A ZIP file which contains two documents: a benign decoy PDF document and a malicious Word document with macros.</li>
<li>The malicious macro downloads a VBS script from a Dropbox URL, followed by the VBS script execution.</li>
<li>The VBS script downloads a CAB file from the dropzone sever, extracts the embedded EXE file (backdoor) using Windows&#8217; &#8220;expand.exe&#8221; utility, and finally executes it.</li>
</ol>
<p>At first, the infection chain consisted of all the above stages, but at a certain point, the attackers decided to skip on the second stage of the infection chain and the malicious Word macros were modified to directly &#8220;download and execute&#8221; the Lazarus Backdoor in stage three.</p>

<p><strong>Malicious Macros</strong></p>
<p>The campaign exhibits very similar macro code in both the XLS and DOC variants of the dropper.</p>
<p>The macros themselves are very simple and straightforward, but in this case, keeping the macros simple and without any advanced obfuscation tricks, resulted in malicious documents that were able to pass undetected by many reputable security vendors on Virus Total.</p>
<p>An interesting part of the download stage in one of the documents, is the unexplained usage of a Dropbox &#8220;Host&#8221; field in the HTTP request header.</p>

<p><strong>The Dropzone</strong></p>
<p>The Lazarus Group is known to utilize an array of compromised servers for its operations, and this time is no different.</p>

<p>A closer look at the compromised server shows an unconvincing website for the &#8220;Information Department&#8221; of the &#8220;South Oil Company&#8221;. The server is located in Iraq and hosted by EarthLink Ltd. Communications&amp;Internet Services.</p>

<p><strong>The KEYMARBLE Backdoor</strong></p>
<p>KEYMARBLE is as a general purpose backdoor that was described in a <a href="https://www.us-cert.gov/ncas/analysis-reports/AR18-221A" target="_blank" rel="noopener">report</a> by NCCIC last August. The malware is a remote administration tool (RAT) that provides its operators with basic functionality to retrieve information from the victim’s machine. Once executed, it conducts several initializations, contacts a C&amp;C server and waits indefinitely for new commands from it. Each received command is processed by the backdoor and handled within an appropriate function, which in turn collects a piece of information or conducts an action on the target machine.</p>
<p>&nbsp;</p>
<p><strong>AV Detection</strong></p>
<p>As part of the infection flow we previously described, all of the malicious documents mentioned downloaded KEYMARBLE, compressed inside a CAB file.</p>
<p>It is interesting to note, that by encapsulating the backdoor in a CAB file, the attackers were able to lower the detection rate of this sample from five vendors to a mere two vendors, who detected this file as malicious on VirusTotal:</p>
<p style="text-align: center;"><a href="/wp-content/uploads/2019/02/fig7.jpg"><img loading="lazy" decoding="async" class="aligncenter wp-image-20835" src="/wp-content/uploads/2019/02/fig7-1024x640.jpg" alt="" width="629" height="392" srcset="https://research.checkpoint.com/wp-content/uploads/2019/02/fig7-1024x640.jpg 1024w, https://research.checkpoint.com/wp-content/uploads/2019/02/fig7-300x188.jpg 300w, https://research.checkpoint.com/wp-content/uploads/2019/02/fig7-768x480.jpg 768w" sizes="auto, (max-width: 629px) 100vw, 629px" /></a></p>

<p><strong>Version Comparison</strong></p>
<p>This instance of the malware resembles its predecessor from last year in flow and functionality. Both operate in two main stages – an initialization phase that sets up necessary data structures and contacts the C&amp;C server, and the main command dispatch loop that receives commands from the server and passes them on to their corresponding handlers. Particular mechanisms within these stages also appear in other pieces of malware that originate from North Korea, a lot of which are attributed to the infamous Lazarus Group.</p>
<p>Having said that, there are particular differences in this variant from the previously reported sample of the same family. For one, the authors used <a href="https://github.com/wolfSSL/wolfssl" target="_blank" rel="noopener">wolfSSL</a>, an open source code repository used to authenticate the client’s identity to the C2 server and encrypt communication. This is not the first time this library is used in North Korean malware. Intezer <a href="https://www.intezer.com/lazarus-group-targets-more-cryptocurrency-exchanges-and-fintech-companies/" target="_blank" rel="noopener">described</a> a different RAT that leveraged it in an attack against cryptocurrency exchanges last year. Additionally, while most of the command codes handled by the backdoor overlap in both the new and old version, some of the codes were omitted from the recent sample and several others were modified, so as the functionality of their handlers.</p>
<p>In the upcoming paragraphs we will outline the key features of KEYMARBLE, focusing on both correlations and distinctions from the previous sample reported by the US-CERT.</p>
<p>&nbsp;</p>
<p><strong>Initialization</strong></p>
<p>Both backdoor variants start with an action of dynamic Win32 API functions resolution. This is a very typical initial stage that appears across multiple North Korean malwares, whereby a list of function names is decrypted during runtime and then resolved to a global table in memory. The addresses from that table will be used subsequently to invoke any calls to the desired API functions.  One of the features in this mechanism that distinguishes this malware family from others is perhaps the usage of the open source <a href="http://read.pudn.com/downloads198/sourcecode/crypt/ca/930917/McbDES2.hpp__.htm" target="_blank" rel="noopener">McbDES2</a> code to implement function name decryption with the DES algorithm.</p>
<p>&nbsp;</p>
<p><a href="/wp-content/uploads/2019/02/fig8-1.png"><img loading="lazy" decoding="async" class="aligncenter wp-image-20826" src="/wp-content/uploads/2019/02/fig8-1-1024x576.png" alt="" width="725" height="408" srcset="https://research.checkpoint.com/wp-content/uploads/2019/02/fig8-1-1024x576.png 1024w, https://research.checkpoint.com/wp-content/uploads/2019/02/fig8-1-300x169.png 300w, https://research.checkpoint.com/wp-content/uploads/2019/02/fig8-1-768x432.png 768w, https://research.checkpoint.com/wp-content/uploads/2019/02/fig8-1-117x67.png 117w, https://research.checkpoint.com/wp-content/uploads/2019/02/fig8-1.png 1325w" sizes="auto, (max-width: 725px) 100vw, 725px" /></a></p>
<p style="text-align: center;"><strong>Figure 8</strong>: Comparison of API resolution logic in both versions of KEYMARBLE</p>
<p>&nbsp;</p>
<p><a href="/wp-content/uploads/2019/02/fig9-1.png"><img loading="lazy" decoding="async" width="849" height="678" class="aligncenter size-full wp-image-20827" src="/wp-content/uploads/2019/02/fig9-1.png" alt="" srcset="https://research.checkpoint.com/wp-content/uploads/2019/02/fig9-1.png 849w, https://research.checkpoint.com/wp-content/uploads/2019/02/fig9-1-300x240.png 300w, https://research.checkpoint.com/wp-content/uploads/2019/02/fig9-1-768x613.png 768w" sizes="auto, (max-width: 849px) 100vw, 849px" /></a></p>
<p style="text-align: center;"><strong>Figure 9</strong>: API function name decryption using the open source McbDoDES template library</p>
<p>&nbsp;</p>
<p>Following this, KEYMARBLE will start preparing the data structures required for communicating with the C&amp;C server. This will include both initiation of WolfSSL related structures as well as initial contact with the server. For the former, the malware will drop a hardcoded PEM certificate to the disk under %TEMP% with the file name “Thumbss.db”, which will have its data read and passed to an internal WolfSSL function called <em>ProcessFile</em>. This will in turn parse it and assign data derived from the certificate to a global context structure used for communication. The used certificate in this sample can be found in the IOC section below.</p>
<p>&nbsp;</p>
<p><a href="/wp-content/uploads/2019/02/fig10.jpg"><img loading="lazy" decoding="async" class="aligncenter wp-image-20828" src="/wp-content/uploads/2019/02/fig10-1024x559.jpg" alt="" width="895" height="489" srcset="https://research.checkpoint.com/wp-content/uploads/2019/02/fig10-1024x559.jpg 1024w, https://research.checkpoint.com/wp-content/uploads/2019/02/fig10-300x164.jpg 300w, https://research.checkpoint.com/wp-content/uploads/2019/02/fig10-768x419.jpg 768w, https://research.checkpoint.com/wp-content/uploads/2019/02/fig10.jpg 1030w" sizes="auto, (max-width: 895px) 100vw, 895px" /></a></p>
<p style="text-align: center;"><strong>Figure 10</strong>: Initialization of communication using wolfSSL, and outline of a proprietary structure that comprises some of the key structures required for the malware’s communication.</p>
<p>&nbsp;</p>
<p>&nbsp;</p>
<p>As for initiating contact with the C2 server, the malware will create a socket, set it to be non-blocking by invoking <em>ioctlsocket </em>with the command argument set to 0x8004667E, and attempt to connect to the hardcoded IP address 194\.45\.8\.41 over port 443. This will happen indefinitely with 30 minute intervals between each connection attempt until success, at which point the malware will break from the loop and continue its execution.</p>
<p>&nbsp;</p>
<p><strong>Communication Protocol</strong></p>
<p>Each message exchanged between the malware and the server will have a predefined structure (as outlined in figure 4) which resembles a TLS application record. As mentioned before, the malware leverages SSL for communication, hence each such message will be encrypted with a key exchanged during the SSL handshake between client and server, and the action of sending or receiving data will be handled by wolfSSL functions designated for this purpose (<em>SendData</em> and <em>ReceiveData</em> accordingly).</p>
<p>&nbsp;</p>
<p><a href="/wp-content/uploads/2019/02/fig11-1.png"><img loading="lazy" decoding="async" width="413" height="235" class="aligncenter size-full wp-image-20829" src="/wp-content/uploads/2019/02/fig11-1.png" alt="" srcset="https://research.checkpoint.com/wp-content/uploads/2019/02/fig11-1.png 413w, https://research.checkpoint.com/wp-content/uploads/2019/02/fig11-1-300x171.png 300w, https://research.checkpoint.com/wp-content/uploads/2019/02/fig11-1-117x67.png 117w" sizes="auto, (max-width: 413px) 100vw, 413px" /></a></p>
<p style="text-align: center;"><strong>Figure 11:</strong> Custom protocol message structure. Resembles a TLS record.</p>
<p>&nbsp;</p>
<p>After initiating the first connection with the C2 server, KEYMARBLE will issue a beacon message. This message is meant to carry the machine’s UID, which is a result of the operation:<br />
MD5(<em>ProductID</em>|<em>MAC</em>), where the first field is obtained by querying the <em>SOFTWARE\Microsoft\Windows NT\CurrentVersion\ProductId</em> registry key, and the second is the MAC address obtained by invoking the function <em>GetAdaptersInfo</em>. However, this UID will be retrieved only after an explicit request from the server, and until that’s done the data field in the beacon will be left blank.</p>
<p><a href="/wp-content/uploads/2019/02/fig12.jpg"><img loading="lazy" decoding="async" width="605" height="287" class="aligncenter size-full wp-image-20830" src="/wp-content/uploads/2019/02/fig12.jpg" alt="" srcset="https://research.checkpoint.com/wp-content/uploads/2019/02/fig12.jpg 605w, https://research.checkpoint.com/wp-content/uploads/2019/02/fig12-300x142.jpg 300w" sizes="auto, (max-width: 605px) 100vw, 605px" /></a></p>
<p style="text-align: center;"><strong>Figure 12:</strong> calculation of UUID as a result of MD5 on ProductID and MAC address.</p>
<p>&nbsp;</p>
<p>After the initial beacon the malware will enter an infinite loop where it will anticipate to get command codes from the server. These will be passed on to a dispatcher function, where each command will be handled by an appropriate handler. The command is received in two parts – first the server will send a message carrying the command’s data length, and only then it will issue the actual command code.</p>
<p><a href="/wp-content/uploads/2019/02/fig13.jpg"><img loading="lazy" decoding="async" width="671" height="441" class="aligncenter size-full wp-image-20831" src="/wp-content/uploads/2019/02/fig13.jpg" alt="" srcset="https://research.checkpoint.com/wp-content/uploads/2019/02/fig13.jpg 671w, https://research.checkpoint.com/wp-content/uploads/2019/02/fig13-300x197.jpg 300w" sizes="auto, (max-width: 671px) 100vw, 671px" /></a></p>
<p style="text-align: center;"><strong>Figure 13</strong>: beacon and main message loop</p>
<p>&nbsp;</p>
<p><strong>Backdoor Commands</strong></p>
<p>The command dispatcher is a very basic mechanism that uses a switch case in order to pass control to the corresponding function. The command codes range from 0x1234556 to 0x1234578 and most overlap with commands that appeared in the older version of the backdoor. However, this version carries a smaller number of commands (18 vs. 22) and few of them differ in code and functionality from the older version. Also, much like with receiving the command code, each command argument sent (if such is required) will be preceded with a length message to indicate what buffer size should be allocated for the sent argument.</p>
<p><a href="/wp-content/uploads/2019/02/fig14-1.png"><img loading="lazy" decoding="async" width="1024" height="215" class="aligncenter size-large wp-image-20832" src="/wp-content/uploads/2019/02/fig14-1-1024x215.png" alt="" srcset="https://research.checkpoint.com/wp-content/uploads/2019/02/fig14-1-1024x215.png 1024w, https://research.checkpoint.com/wp-content/uploads/2019/02/fig14-1-300x63.png 300w, https://research.checkpoint.com/wp-content/uploads/2019/02/fig14-1-768x161.png 768w, https://research.checkpoint.com/wp-content/uploads/2019/02/fig14-1.png 1541w" sizes="auto, (max-width: 1024px) 100vw, 1024px" /></a></p>
<p style="text-align: center;"><strong>Figure 14:</strong> command dispatch function comparison between the old and new version of KEYMARBLE</p>

function calcScreenW(){
	var screenC = false;
	curentW= jQuery(window).width();	
	if ( curentW > 1450 ) { 
		if (screenW != 'desktop'){
			screenW = 'desktop';	
			screenC = true;
		}
	} else if (curentW > 973){
		if (screenW != 'laptop'){
			screenW = 'laptop';	
			screenC = true;
		}		
	} else if (curentW >767){
		if (screenW != 'tablet'){
			screenW = 'tablet';	
			screenC = true;
		}		
	} else {
		if (screenW != 'mobile'){
			screenW = 'mobile';	
			screenC = true;
		}		
	}
	if (screenC) {
		jQuery('.bg-change-done').removeClass('bg-change-done');
		screenC = false;
		animate_elems();
	}
}

function animate_elems() {
	wintop = jQuery(window).scrollTop(); // calculate distance from top of window
	// loop through each item to check when it animates
	$elems.each(function(){
		$elm = jQuery(this);
		if($elm.hasClass('lazyload')|| $elm.hasClass('bg-change-done')) { return true; } // if already animated skip to the next item

		topcoords = $elm.offset().top; // element's distance from top of page in pixels

		if(wintop > (topcoords - (winheight))) {			
			$elm.addClass('bg-change-done');		
			loadImg($elm[0]);
		} 
	});	
}

	  var name = cname + "=";
