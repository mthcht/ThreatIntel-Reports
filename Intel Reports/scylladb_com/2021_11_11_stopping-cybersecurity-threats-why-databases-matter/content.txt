                        <p>From intrusion detection, to threat analysis, to endpoint security, the effectiveness of cybersecurity efforts often boils down to how much data can be processed — in real time — with the most advanced algorithms and models.</p>
<p>Many factors are obviously involved in stopping cybersecurity threats effectively. However, the databases responsible for processing the billions or trillions of events per day (from millions of endpoints) play a particularly crucial role. High throughput and <a href="https://www.scylladb.com/glossary/low-latency-database/" target="_blank" rel="noopener">low latency</a> directly correlate with better insights as well as more threats discovered and mitigated in near real time. But cybersecurity data-intensive systems are incredibly complex; many span 4+ data centers with database clusters exceeding 1000 nodes and petabytes of heterogeneous data under active management.</p>
<p>How do expert engineers and architects at leading cybersecurity companies design, manage, and evolve <a href="https://www.scylladb.com/glossary/data-architect/" target="_blank" rel="noopener">data architectures</a> that are up to the task? Here’s a look at 3 specific examples.</p>
<h2>Accelerating real-time threat analysis by 1000% at FireEye</h2>
<h3>Cybersecurity use case</h3>
<p><a href="https://www.fireeye.com/" target="_blank" rel="noopener">FireEye</a>&#8216;s Threat Intelligence application centralizes, organizes, and processes threat intelligence data to support analysts. It does so by grouping threats using analytical correlation, and by processing and recording vast quantities of data, including DNS data, RSS feeds, domain names, and URLs. Using this array of billions of properties, FireEye’s cybersecurity analysts can explore trillions of questions to provide unparalleled visibility into the threats that matter most.</p>
<h3>Database challenge</h3>
<p>Their legacy system used <a href="https://en.wikipedia.org/wiki/PostgreSQL" target="_blank" rel="noopener">PostgreSQL</a> with a custom graph database system to store and facilitate the analysis of threat intelligence data. As the team of analysts grew into the hundreds, system limitations emerged. The graph size grew to 500M nodes, with around 1.5B edges connecting them. Each node had more than 100 associated properties accumulated over several years. The PostgreSQL-based system became slow, proved difficult to scale, and was not distributed or highly available. FireEye needed to re-architect their system on a new technology base to serve the growing number of analysts and the businesses that rely on them.</p>
<h3>Database strategy</h3>
<p>To start, the team evaluated several graph databases and selected <a href="https://github.com/JanusGraph/janusgraph" target="_blank" rel="noopener">JanusGraph</a>. FireEye’s functional criteria included traversing speed, full/free text search, and concurrent user support. Non-functional criteria included requirements for <a href="https://www.scylladb.com/glossary/high-availability-database/" target="_blank" rel="noopener">high availability</a> and disaster recovery, plus a pluggable storage backend for flexibility. The team felt that JanusGraph met these criteria well, and also appreciated its user-controllable indexing, schema management, triggers, and OLAP capabilities for distributed graph processing.</p>
<p>Next, they shifted focus to evaluating compatible backend storage solutions. With analysis speed top of mind, they looked past Java-based options. FireEye selected <a href="https://www.scylladb.com/product/" target="_blank" rel="noopener">ScyllaDB</a> based on its raw performance and manageability. The FireEye team chose to run ScyllaDB themselves within a secure enclave guarded by an NGINX gateway. Today, the ScyllaDB solution is deployed on AWS i3.8xlarge instances in 7-node clusters. Each node is provisioned with 32 CPUs, 244MB of memory, and 16TB SSD storage.</p>
<p><img fetchpriority="high" decoding="async" class="wp-image-21551 size-full" src="https://www.scylladb.com/wp-content/uploads/block-diagram-1.png" alt="FireEye's system architecture diagram" width="1600" height="676" srcset="https://www.scylladb.com/wp-content/uploads/block-diagram-1.png 1600w, https://www.scylladb.com/wp-content/uploads/block-diagram-1-300x127.png 300w, https://www.scylladb.com/wp-content/uploads/block-diagram-1-1100x465.png 1100w, https://www.scylladb.com/wp-content/uploads/block-diagram-1-768x324.png 768w, https://www.scylladb.com/wp-content/uploads/block-diagram-1-1536x649.png 1536w" sizes="(max-width: 1600px) 100vw, 1600px" /></p>
<p style="text-align: center;"><em>System architecture diagram &#8211; provided by FireEye</em></p>
<h3>Impact</h3>
<p>With the new JanusGraph + ScyllaDB system, FireEye achieved a performance improvement of 100X. For example, a query traversing 15,000 graph nodes now returns results in 300ms (vs 30 seconds to 3 minutes).</p>
<p><img decoding="async" class="wp-image-44220 size-full aligncenter" src="https://www.scylladb.com/wp-content/uploads/fireeye2.png" alt="Query execution comparison diagram - provided by FireEye" width="800" height="497" srcset="https://www.scylladb.com/wp-content/uploads/fireeye2.png 800w, https://www.scylladb.com/wp-content/uploads/fireeye2-300x186.png 300w, https://www.scylladb.com/wp-content/uploads/fireeye2-768x477.png 768w" sizes="(max-width: 800px) 100vw, 800px" /></p>
<p style="text-align: center;"><em>Query execution comparison diagram &#8211; provided by FireEye</em></p>
<p>Moreover, they were able to dramatically slash the storage footprint while preserving the 1000-2000% performance increase they had experienced by switching to ScyllaDB. Ultimately, they reduced AWS spend to 10% of the original cost.</p>

<h2>Scaling security from 1.5M to 100M devices at Lookout</h2>
<h3>Cybersecurity use case</h3>
<p><a href="https://www.lookout.com/" target="_blank" rel="noopener">Lookout</a> leverages artificial intelligence to provide visibility and protection from network threats, web-based threats, vulnerabilities, and other risks. To protect an enterprise’s mobile devices, they ingest device telemetry and feed it across a network of services to identify risks. Low-latency is key: if a user installs an app with malware and it takes minutes to detect, that device’s information is already compromised.</p>
<h3>Database challenge</h3>
<p>Lookout needed to scale from supporting 1.5 million devices to 100 million devices without substantially increasing costs. This required ingesting more and more telemetry as it came in. They needed a highly scalable and fault-tolerant streaming framework that could process device telemetry messages and persist these messages into a scalable, fault-tolerant persistent store with support for operational queries.</p>
<p>Their existing solution involved <a href="https://spark.apache.org/" target="_blank" rel="noopener">Spark</a>, <a href="https://aws.amazon.com/dynamodb/" target="_blank" rel="noopener">DynamoDB</a>, and <a href="https://www.elastic.co/" target="_blank" rel="noopener">ElasticSearch</a>. However, DynamoDB alone would cost them about $1M per month if they scaled it for 100M devices. Also, DynamoDB lacked the required level of sorting for <a href="https://www.scylladb.com/glossary/time-series-data/" target="_blank" rel="noopener">time series data</a>. With DynamoDB, they had to use a query API that introduced a significant performance hit, impacting latency.</p>
<h3>Database strategy</h3>
<p>Lookout replaced Spark with <a href="https://kafka.apache.org/" target="_blank" rel="noopener">Kafka</a> and migrated from DynamoDB to ScyllaDB. Kafka Connect servers send data from Kafka into ScyllaDB, then it’s pushed into ElasticSearch.</p>
<p><img decoding="async" class="wp-image-44228 size-full aligncenter" src="https://www.scylladb.com/wp-content/uploads/lookout1.png" alt="System architecture diagram - provided by Lookout" width="800" height="429" srcset="https://www.scylladb.com/wp-content/uploads/lookout1.png 800w, https://www.scylladb.com/wp-content/uploads/lookout1-300x161.png 300w, https://www.scylladb.com/wp-content/uploads/lookout1-768x412.png 768w" sizes="(max-width: 800px) 100vw, 800px" /></p>
<p style="text-align: center;"><em>System architecture diagram &#8211; provided by Lookout</em></p>
<p>They also discovered that Kafka’s default partitioner became less and less efficient with sharding as the number of partitions grew, so they replaced the default Kafka sharding function with a murmur3 hash and put it through a consistent hashing algorithm (jump hash) to get an even distribution across all partitions.</p>
<h3>Impact</h3>
<p>Lookout met its goal of growing from 1.5 million devices to 100 million devices while reining in costs. What would have cost them nearly $1M per month with DynamoDB cost them under $50K per month with ScyllaDB. They also achieved the low latency vital to effective threat detection. Their message latency is currently in the milliseconds, on average.</p>

<p>&nbsp;</p>
<h2>Achieving database independence at ReversingLabs</h2>
<h3>Cybersecurity use case</h3>
<p><a href="https://www.reversinglabs.com/" target="_blank" rel="noopener">ReversingLabs</a> is a complete advanced malware analysis platform that speeds destructive file detection through automated static analysis — enabling analysts to prioritize the highest risks with actionable detail in milliseconds. Their TitaniumCloud Reputation Services provides powerful threat intelligence solutions with up-to-date threat classification and rich context on over 20B goodware and malware files.</p>
<h3>Database challenge</h3>
<p>ReversingLabs’ database stores the outcome of multiple analyses on billions of files so that end users can perform malware detection “in the wild”: immediately checking if a file they encounter is known to be safe or suspected to be malware. If the file’s status is unknown, it needs to be analyzed then entered into the ReversingLabs database. 50+ APIs and specialized feeds connect end users to that database.</p>
<p><img loading="lazy" decoding="async" class="aligncenter wp-image-44229 size-full" src="https://www.scylladb.com/wp-content/uploads/reversinglabs-1.png" alt="System architecture diagram - provided by ReversingLabs" width="800" height="375" srcset="https://www.scylladb.com/wp-content/uploads/reversinglabs-1.png 800w, https://www.scylladb.com/wp-content/uploads/reversinglabs-1-300x141.png 300w, https://www.scylladb.com/wp-content/uploads/reversinglabs-1-768x360.png 768w" sizes="(max-width: 800px) 100vw, 800px" /></p>
<p style="text-align: center;"><em>System architecture diagram &#8211; provided by ReversingLabs</em></p>
<p>Years ago, ReversingLabs built their own highly available database for this application. It’s a key-value store, based on the LSM Tree architecture. The database needed to insert/update and read large amounts of data with low latency and stable response times, and it achieved those goals.</p>
<p>However, as their business expanded, they wanted “database independence” to eliminate dependencies on the aging legacy database. They built a library that enabled them to connect to other databases without modifying their application layer. But connecting OSS or COTS solutions to meet their specialized needs wasn’t exactly a plug-and-play endeavor. For example, they needed:</p>
<ul>
<li>Key-value native protobuf format</li>
<li>LZ4 compression for reduced storage size</li>
<li>Latency &lt;2 ms</li>
<li>Support for record sizes tanging from 1K to 500M</li>
</ul>
<h3>Database strategy</h3>
<p>The first test of their database independence strategy was adopting ScyllaDB, with an 8 node cluster, for their 2 large volumes APIs. They were able to meet their specialized needs by applying strategies like:</p>
<ul>
<li>Using blobs for keys as well as values</li>
<li>Tuning the chunk size to improve compression 49%</li>
<li>Improving performance by using NVMe disks in RAID 0 configuration and doing inserts/updates and reads at consistency level quorum</li>
</ul>
<h3>Impact</h3>
<p>ReversingLabs successfully moved beyond their legacy database, with a highly available and scalable system that meets their specialized needs. Within the ScyllaDB database, file reputation queries had average writes latencies of &lt;6 milliseconds and sub-millisecond average reads. For p99 (long-tail) latencies, they achieved &lt;12 millisecond writes, and &lt;7 millisecond reads.</p>
<p>In end-to-end workflow testing (including and beyond the database), average latencies were less than 120 milliseconds, and p99 latencies were 166 milliseconds. This included the user request, authentication process, request validation, the round-trip time to query the database, to format and then finally send the response to the application, scaled to 32 workers, all in parallel.</p>

<p style="text-align: center;"><a class="cta-blog-button" href="https://www.scylladb.com/2020/04/29/reversinglabs-serving-file-reputation-for-twenty-billion-files/" target="_blank" rel="noopener">Read more about the role of databases in ReversingLabs&#8217; cybersecurity efforts</a></p>
<h2>Databases: what’s essential for cybersecurity</h2>
<p>To wrap up, here’s a quick recap of the core database capabilities that have helped these and many other companies stop cybersecurity threats faster and more accurately:</p>
<ul>
<li>Consistent low-latency performance for real-time streaming analytics — to identify and prevent digital threats in real-time</li>
<li>Extreme throughput and rapid scaling — to ingest billions of events across threat analysis, malware protection, and intrusion detection</li>
<li>Lower total cost of ownership, reduced complexity, automated tuning, and DevOps-friendly operational capabilities — to free resources for strategic cybersecurity projects</li>
<li>The ability to run at extremely high levels of utilization — to ensure consistent performance and ultra-low latency without overprovisioning and waste</li>
<li>Cloud, on-premises, and a variety of hybrid topologies to support data distribution and geographic replication — for high availability, resilience, and instant access to critical data anywhere in the world</li>
</ul>

// Function to get a cookie by name
function getCookie(cname) {
    let name = cname + "=";
    let decodedCookie = decodeURIComponent(document.cookie);
    let ca = decodedCookie.split(';');
    for(let i = 0; i < ca.length; i++) {
        let c = ca[i];
        while (c.charAt(0) == ' ') {
            c = c.substring(1);
        }
        if (c.indexOf(name) == 0) {
            return c.substring(name.length, c.length);
        }
    }
    return "";
}
