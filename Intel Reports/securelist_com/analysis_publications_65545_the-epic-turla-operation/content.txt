<h2 id="executive-summary">Executive Summary</h2>
<p>Over the last 10 months, Kaspersky Lab  researchers have analyzed a massive cyber-espionage operation which we call  &#8220;Epic Turla&#8221;. The attackers behind Epic Turla have infected several hundred  computers in more than 45 countries, including government institutions,  embassies, military, education, research and pharmaceutical companies.</p>
<p>The attacks are known to have used at least  two zero-day exploits:</p>
<ul>
<li><strong><a href="https://nvd.nist.gov/vuln/detail/CVE-2013-5065">CVE-2013-5065</a></strong> &#8211; Privilege escalation vulnerability in Windows XP and Windows 2003</li>
<li><strong><a href="https://nvd.nist.gov/vuln/detail/CVE-2013-3346">CVE-2013-3346</a></strong> &#8211; Arbitrary  code-execution vulnerability in Adobe Reader</li>
</ul>
<p>We also observed exploits against older  (patched) vulnerabilities, social engineering techniques and watering hole  strategies in these attacks. The primary backdoor used in the Epic attacks is  also known as &#8220;WorldCupSec&#8221;, &#8220;TadjMakhal&#8221;, &#8220;Wipbot&#8221; or &#8220;Tavdig&#8221;.</p>
<p>When G-Data published on <a href="https://blog.gdatasoftware.com/blog/article/uroburos-highly-complex-espionage-software-with-russian-roots.html">Turla/Uroburos</a>  back in February,  several questions remained unanswered. One big unknown was the infection vector for Turla (aka Snake or  Uroburos). Our analysis indicates that victims are infected via a sophisticated  multi-stage attack, which begins with the Epic Turla. In time, as the attackers  gain confidence, this is upgraded to more sophisticated backdoors, such as the  Carbon/Cobra system. Sometimes, both backdoors are run in tandem, and used to  &#8220;rescue&#8221; each other if communications are lost with one of the backdoors.</p>
<p>Once the attackers obtain the necessary  credentials without the victim noticing, they deploy the rootkit and other  extreme persistence mechanisms.</p>
<p>The attacks are still ongoing as of July  2014, actively targeting users in Europe and the Middle East.</p>
<p><strong><em>Note: A full analysis of the Epic attacks is available  to the Kaspersky Intelligent Services subscribers. Contact:  <a href="mailto:intelreports@kaspersky.com">intelreports@kaspersky.com</a></em></strong></p>
<h2 id="the-epic-turla-attacks">The Epic Turla attacks</h2>
<p>The attacks in this campaign fall into  several different categories depending on the vector used in the initial compromise:</p>
<ul>
<li>Spearphishing e-mails with Adobe PDF exploits (CVE-2013-3346 +  CVE-2013-5065)</li>
<li>Social engineering to trick the user into running malware installers  with &#8220;.SCR&#8221; extension, sometimes packed with RAR</li>
<li>Watering hole attacks using Java exploits (CVE-2012-1723), Flash  exploits (unknown) or Internet Explorer 6,7,8 exploits (unknown)</li>
<li>Watering hole attacks that rely on social engineering to trick the  user into running fake &#8220;Flash Player&#8221; malware installers</li>
</ul>
<p>The attackers use both direct spearphishing  and watering hole attacks to infect their victims. Watering holes (waterholes)  are websites of interest to the victims that have been compromised by the  attackers and injected to serve malicious code.</p>
<p>So far we haven&#8217;t been able to locate any  e-mail used against the victims, only the attachments. The PDF attachments do  not show any &#8220;lure&#8221; to the victim when opened, however, the SCR packages  sometime show a clean PDF upon successful installation.</p>
<p class="c"><a href="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2014/08/08080259/Uroburos_1.jpg" class="magnificImage"><img decoding="async" border="0" alt="turla" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2014/08/08080255/Uroburos_1_sm.jpg" style="border:1px solid #000"></a></p>
<p>Some of known attachment names used in the  spearphishing attacks are:</p>
<ul>
<li><strong><span dir="RTL">ؤتمر جنيف</span>.rar</strong> (translation from Arabic:  &#8220;Geneva conference.rar&#8221;)</li>
<li><strong>NATO position on Syria.scr</strong></li>
<li><strong>Note_№107-41D.pdf</strong></li>
<li><strong>Talking Points.scr</strong></li>
<li><strong>border_security_protocol.rar</strong></li>
<li><strong>Security protocol.scr</strong></li>
<li><strong>Program.scr</strong></li>
</ul>
<p>In some cases, these filenames can provide  clues about the type of victims the attackers are targeting.</p>

<h2 id="the-epic-command-and-control-infrastructure">The Epic command-and-control infrastructure</h2>
<p>The Epic backdoors are commanded by a huge  network of hacked servers that deliver    command and control functionality.</p>
<p>The huge  network commanded by the Epic Turla attackers serves multiple purposes. For  instance, the motherships function as both exploitation sites and command and  control panels for the malware. </p>
<p>Here&#8217;s how the big picture looks  like:</p>
<p class="c"><a href="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2014/08/08075922/Uroburos_13.jpg" class="magnificImage"><img decoding="async" border="0" alt="turla" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2014/08/08075917/Uroburos_13_sm.jpg" style="border:1px solid #000"></a><br />
<em>Epic Turla lifecycle</em></p>
<p>The first level of command and control  proxies generally talk to a second level of proxies, which in turn, talk to the  &#8220;mothership&#8221; server. The mothership server is generally a VPS, which runs the  Control panel software used to interact with the victims. The attackers operate  the mothership using a network of proxies and VPN servers for anonymity  reasons. The mothership also work as the exploitation servers used in the  watering hole attacks, delivering Java, IE or fake applications to the victim.</p>
<p>We were able to get a copy of one of the  motherships, which provided some insight into the operation. </p>
<p>It runs a control panel which is password  protected:</p>
<p class="c"><img decoding="async" border="0" alt="turla" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2014/08/08075913/Uroburos_14.jpg" style="border:1px solid #000"><br />
<em>Epic mothership control panel login</em></p>
<p>Once logged into the Control panel, the  attackers can see a general overview of the system including the number of  interesting potential targets:</p>
<p class="c"><a href="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2014/08/08075909/Uroburos_15.jpg" class="magnificImage"><img decoding="async" border="0" alt="turla" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2014/08/08075904/Uroburos_15_sm.jpg" style="border:1px solid #000"></a><br />
<em>Epic control panel status overview</em></p>
<p>A very interesting file on the servers is <strong>task.css</strong>, where the attackers define  the IP ranges they are interested in. To change the file, they are using the  &#8220;Task editor&#8221; from the menu. Depending on the &#8220;tasks&#8221;, they will decide whether  to infect the visitors or not. In this case, we found they targeted two ranges  belonging to:</p>
<ul>
<li>&#8220;Country A&#8221; &#8211; Federal Government Network</li>
<li>&#8220;Country B&#8221; &#8211; Government Telecommunications and Informatics Services Network</li>
</ul>
<p>It should be noted though, the fact that  the attackers were targeting these ranges doesn&#8217;t necessarily mean they also  got infected. Some other unknown IPs were also observed in the targeting  schedules.</p>

<ul>
<li>TRY</li>
<li>DON&#8217;T TRY -> Version of the  browser and OS does not meet the conditions</li>

<h2 id="the-epic-tavdig-wipbot-backdoor">The Epic / Tavdig / Wipbot  backdoor</h2>
<p>For this first stage of the attack, the  threat actor uses a custom backdoor. In some cases, the backdoor is packaged  together with the CVE-2013-5065 EoP exploit and heavily obfuscated. This makes  the analysis more difficult.</p>
<p>The CVE-2013-5065 exploit allows the  backdoor to achieve administrator privileges on the system and run  unrestricted. This exploit only works on unpatched Microsoft Windows XP  systems.</p>
<p>Other known detection names for the  backdoor is Trojan.Wipbot (Symantec) or Tavdig.</p>
<p>The main backdoor is about 60KB in size and  implements a C&#038;C protocol on top of normal HTTP requests. The communication  protocol uses <img decoding="async" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2014/08/08080122/Uroburos_code_1.jpg" alt="xxx" /> requests in the C&#038;C replies, which the malware decrypts and processes. The  replies are sent back to the C&#038;C through the same channel.</p>
<p>The malware behavior is defined by a  configuration block. The configuration block usually contains two hard-coded  C&#038;C URLs. He have also seen one case where the configuration block contains  just one URL. The configuration can also be updated on the fly by the  attackers, via the C&#038;C. </p>
<p>The backdoor attempts to identify the  following processes and, if found, it will terminate itself:</p>
<ul>
<li>tcpdump.exe</li>
<li>windump.exe</li>
<li>ethereal.exe</li>
<li>wireshark.exe</li>
<li>ettercap.exe</li>
<li>snoop.exe</li>
<li>dsniff.exe</li>
</ul>
<p>It contains an internal unique ID, which is  used to identify the victim to the C&#038;C. Most samples, especially old ones,  have the ID <strong>1156fd22-3443-4344-c4ffff</strong>.  Once a victim is confirmed as &#8220;interesting&#8221;, the attackers upload another Epic  backdoor which has a unique ID used to control this specific victim.</p>
<p>During the first C&#038;C call, the backdoor  sends a pack with the victim&#8217;s system information. All further information sent  to the C&#038;C is encrypted with a public key framework, making decryption  impossible. The commands from the C&#038;C are encrypted in a simpler manner and  can be decrypted if intercepted because the secret key is hardcoded in the  malware.</p>
<p>Through monitoring, we were able to capture  a large amount of commands sent to the victims by the attackers, providing an  unique view into this operation. Here&#8217;s a look at one of the encrypted server  replies:</p>
<p class="c"><img decoding="async" border="0" alt="turla" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2014/08/08080117/Uroburos_code_2.jpg"></p>
<p>Once a victim is infected and &#8220;checks in&#8221;  with the server, the attackers send a template of commands:</p>
<p class="c"><a href="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2014/08/08075859/Uroburos_16.jpg" class="magnificImage"><img decoding="async" border="0" alt="turla" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2014/08/08075856/Uroburos_16_sm.jpg"></a></p>
<p>Next, the attackers try to move through the  victim&#8217;s network using pre-defined or collected passwords:</p>
<p class="c"><a href="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2014/08/08075850/Uroburos_17.jpg" class="magnificImage"><img decoding="async" border="0" alt="turla" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2014/08/08075846/Uroburos_17_sm.jpg"></a></p>
<p>Listing all .doc files recursively is also  a common &#8220;theme&#8221;:</p>
<p class="c"><img decoding="async" border="0" alt="turla" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2014/08/08075842/Uroburos_18.jpg"></p>
<p>In total, we have decoded several hundreds  of these command packages delivered to the victims, providing an unique insight  into the inner workings of the attackers.</p>
<p>In addition to generic searches, some very  specific lookups have been observed as well.   These include searches for:</p>
<ul>
<li><strong>*NATO*.msg</strong></li>
<li><strong>eu energy dialogue*.*</strong></li>
<li><strong>EU*.msg</strong></li>
<li><strong>Budapest*.msg</strong></li>
</ul>
<p>In this case, the attackers were interested  to find e-mails related to &#8220;NATO&#8221;, &#8220;Energy Dialogue within European Union&#8221;  and so on.</p>
<p>For some of the C&#038;C servers, the  attackers implemented RSA encryption for the C&#038;C logs, which makes it  impossible to decrypt them. This scheme was implemented in April 2014.</p>

<h2 id="lateral-movement-and-upgrade-to-more-sophisticated-backdoors">Lateral movement and upgrade to  more sophisticated backdoors</h2>
<p>Once a victim is compromised, the attackers  upload several tools that are used for lateral movement.</p>
<p>One such tool observed in the attacks and  saved as &#8220;<strong>C:Documents and SettingsAll  usersStart MenuProgramsStartupwinsvclg.exe</strong>&#8221; is:</p>

<h2 id="language-artifacts">Language artifacts</h2>
<p>The payload recovered from one of the  mothership servers (at <strong>newsforum.servehttp[.]com/wordpress/wp-includes/css/img/upload.php</strong>,  MD5: <strong>4dc22c1695d1f275c3b6e503a1b171f5</strong>,  Compiled: Thu Sep 06 14:09:55 2012) contains two modules, a loader/injector and  a backdoor. Internally, the backdoor is named &#8220;Zagruzchik.dll&#8221;:</p>
<p class="c"><img decoding="async" border="0" alt="turla" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2014/08/08075800/Uroburos_23.jpg"></p>
<p>The word &#8220;Zagruzchik&#8221; means &#8220;boot loader&#8221;  in Russian.</p>
<p>The Control panel for the Epic motherships  also sets the language to codepage &#8220;1251&#8221;: </p>
<p class="c"><img decoding="async" border="0" alt="turla" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2014/08/08080107/Uroburos_code_4.jpg"></p>
<p>Codepage 1251 is commonly used to render  Cyrillic characters.</p>
<p>There are other indications that the  attackers are not native English language speakers:</p>
<ul>
<li><em>Password it´s wrong!</em></li>
<li><em>Count successful more MAX</em></li>
<li><em>File is not exists</em></li>
<li><em>File is exists for edit</em></li>
</ul>
<p>The sample <strong>e9580b6b13822090db018c320e80865f</strong> that was delivered to several Epic  victims as an upgraded backdoor, has the compilation code page language set to  &#8220;LANG_RUSSIAN&#8221;.</p>
<p class="c"><a href="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2014/08/08075755/Uroburos_24.jpg" class="magnificImage"><img decoding="async" border="0" alt="turla" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2014/08/08075751/Uroburos_24_sm.jpg"></a></p>
<p>The threat actor behind the &#8220;Epic&#8221;  operation uses mainly hacked servers to host their proxies. The hacked servers  are controlled through the use of a PHP webshell. This shell is password  protected; the password is checked against an MD5 hash:</p>
<p class="c"><a href="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2014/08/08075745/Uroburos_25.jpg" class="magnificImage"><img decoding="async" border="0" alt="turla" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2014/08/08075740/Uroburos_25_sm.jpg"></a></p>
<p>The MD5 &#8220;<strong>af3e8be26c63c4dd066935629cf9bac8</strong>&#8221; has been solved by Kaspersky Lab  as the password &#8220;kenpachi&#8221;. In February 2014 we observed the <a href="https://securelist.com/the-miniduke-mystery-pdf-0-day-government-spy-assembler-0x29a-micro-backdoor/31112/">Miniduke</a> threat actor using the same backdoor on their  hacked servers, although using a much stronger password.</p>
<p>Once again, it is also interesting to point  out the usage of Codepage 1251 in the webshell, which is used to render  Cyrillic characters.</p>
<p>There appears to be several links between  Turla and Miniduke, but we will leave that for a future blogpost.</p>

<h2 id="summary">Summary</h2>
<p>When G-Data published their Turla paper,  there were few details publicly available on how victims get infected with this  malware campaign. Our analysis indicates this is a sophisticated multi-stage  infection; which begins with Epic Turla. This is used to gain a foothold and  validate the high profile victim. If the victim is interesting, they get  upgraded to the Turla Carbon system.</p>
<p>Most recently, we observed this attack  against a Kaspersky Lab user on August 5, 2014, indicating the operation  remains fresh and ongoing.</p>
