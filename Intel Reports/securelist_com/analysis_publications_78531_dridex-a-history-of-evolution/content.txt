												<div class="c-wysiwyg">
													<p>The Dridex banking Trojan, which has become a major financial cyberthreat in the past years (in 2015, the damage done by the Trojan was <a href="https://www.theregister.co.uk/2015/10/14/dridex_botnet_takedown/">estimated</a> at over $40 million), stands apart from other malware because it has continually evolved and become more sophisticated since it made its first appearance in 2011. Dridex has been able to escape justice for so long by hiding its main command-and-control (C&amp;C) servers behind proxying layers. Given that old versions stop working when new ones appear and that each new improvement is one more step forward in the systematic development of the malware, it can be concluded that the same people have been involved in the Trojan&#8217;s development this entire time. Below we provide a brief overview of the Trojan&#8217;s evolution over six years, as well as some technical details on its latest versions.</p>
<p><a href="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2017/05/07174533/dridex_en_0.png" class="magnificImage"><img loading="lazy" decoding="async" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2017/05/07174533/dridex_en_0.png" alt="Dridex: A History of Evolution" width="848" height="464" class="aligncenter size-full wp-image-78557" srcset="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2017/05/07174533/dridex_en_0.png 848w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2017/05/07174533/dridex_en_0-300x164.png 300w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2017/05/07174533/dridex_en_0-768x420.png 768w" sizes="auto, (max-width: 848px) 100vw, 848px" /></a></p>
<h2 id="how-it-all-began">How It All Began</h2>
<p>Dridex made its first appearance as an independent malicious program (under the name &#8220;Cridex&#8221;) around September 2011. An analysis of a Cridex sample (MD5: 78cc821b5acfc017c855bc7060479f84) demonstrated that, even in its early days, the malware could receive dynamic configuration files, use web injections to steal money, and was able to infect USB media. This ability influenced the name under which the &#8220;zero&#8221; version of Cridex was detected — Worm.Win32.Cridex. </p>
<p>That version had a binary configuration file:</p>
<p><a href="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2017/05/07174625/dridex_en_1.png" class="magnificImage"><img loading="lazy" decoding="async" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2017/05/07174625/dridex_en_1.png" alt="Dridex: A History of Evolution" width="991" height="436" class="aligncenter size-full wp-image-78543" srcset="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2017/05/07174625/dridex_en_1.png 991w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2017/05/07174625/dridex_en_1-300x132.png 300w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2017/05/07174625/dridex_en_1-768x338.png 768w" sizes="auto, (max-width: 991px) 100vw, 991px" /></a></p>
<p>Sections named databefore, datainject, and dataafter made the web injections themselves look similar to the widespread Zeus malware (there may have been a connection between this and the 2011 Zeus source code leak).</p>

<h2 id="dridex-1-10">Dridex 1.10</h2>
<p>The &#8220;zero&#8221; version was maintained until June 2014. A major operation (<a href="https://krebsonsecurity.com/2014/06/operation-tovar-targets-gameover-zeus-botnet-cryptolocker-scourge/">Operation Tovar</a>) to take down another widespread malicious program — Gameover Zeus — was carried out that month. Nearly as soon as Zeus was taken down, the &#8220;zero&#8221; version of Cridex stopped working and Dridex version 1.100 appeared almost exactly one month afterward (on June 22).</p>

<h2 id="dridex-from-version-2-to-version-3">Dridex: from Version 2 to Version 3</h2>
<p>By early 2015, Dridex implemented a kind of P2P network, which is also reminiscent of the Gameover Zeus Trojan. On that network, some peers (supernodes) had access to the C&amp;C and forwarded requests from other network nodes to it. The configuration file was still stored in XML format, but it got a new section, &lt;nodes&gt;, which contained an up-to-date peer list. Additionally, the protocol used for communication with the C&amp;C was encrypted.</p>
<h2 id="dridex-from-version-3-to-version-4">Dridex: from Version 3 to Version 4</h2>
<p>One of the administrators of the Dridex network was arrested on August 28, 2015. In the early days of September, networks with identifiers 120, 200, and 220 went offline. However, they came back online in October and new networks were added: 121, 122, 123, 301, 302, and 303.</p>
<p>Notably, the cybercriminals stepped up security measures at that time. Specifically, they introduced geo-filtering wherein an IP field appeared in C&amp;C request packets, which was then used to identify the peer&#8217;s country. If it was not on the list of target countries, the peer received an error message.</p>
<p>In 2016, the loader became more complicated and encryption methods were changed. A binary loader protocol was introduced, along with a &lt;settings&gt; section, which contained the configuration file in binary format.</p>
<h2 id="dridex-4-x-back-to-the-future">Dridex 4.x. Back to the Future</h2>
<p>The fourth version of Dridex <a href="https://securityintelligence.com/dridexs-cold-war-enter-atombombing/">was detected</a> in early 2017. It has capabilities similar to the third version, but the cybercriminals stopped using the XML format in the configuration file and packets and went back to binary. The analysis of new samples is rendered significantly more difficult by the fact that the loader now works for two days, at most. This is similar to Lurk, except that Lurk&#8217;s loader was only active for a couple of hours. </p>

<p>In the fourth version of Dridex, there are many places where the CRC32 hashing algorithm is used, including hashes used to search for function APIs and to check packet integrity. It would make sense for hashes used in packets to be none other than CRC32 of requested module names. This assumption can easily be verified by running the following Python code:</p>

<p>That&#8217;s right – the hashes obtained this way are the same as those in the program&#8217;s code.</p>
<p>With regards to encryption of the loader&#8217;s packets, nothing has changed. As in Dridex version 3, the RC4 algorithm is used, with a key stored in encrypted form in the malicious program&#8217;s body.</p>
<p>One more change introduced in the fourth version is that a much stricter loader authorization protocol is now used. A loader&#8217;s lifespan has been reduced to one day, after which encryption keys are changed and old loaders become useless. The server responds to requests from all outdated samples with error 404.</p>
<h3 id="analysis-of-the-bots-protocol-and-encryption">Analysis of the Bot&#8217;s Protocol and Encryption</h3>
<p>Essentially, the communication of Dridex version 4 with its C&amp;C is based on the same procedure as before, with peers still acting as proxy servers and exchanging modules. However, encryption and packet structure have changed significantly; now a packet looks like the &lt;settings&gt; section from the previous Dridex version. No more XML.</p>
<p><a href="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2017/05/07174555/dridex_en_9.png" class="magnificImage"><img loading="lazy" decoding="async" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2017/05/07174555/dridex_en_9.png" alt="Dridex: A History of Evolution" width="583" height="389" class="aligncenter size-full wp-image-78550" srcset="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2017/05/07174555/dridex_en_9.png 583w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2017/05/07174555/dridex_en_9-300x200.png 300w" sizes="auto, (max-width: 583px) 100vw, 583px" /></a></p>
<p>The Basic Packet Generation function is used to create packets for communication with the C&amp;C and with peers. There are two types of packets for the C&amp;C: </p>
<ol>
<li>Registration and transfer of the generated public key</li>
<li>Request for a configuration file</li>

<p>The packet begins with a 16-byte key, followed by 4 bytes of information about the size of data encrypted with the previous key using RC4. Next comes a 16-byte key and data that has been encrypted with that key using RC4. After decryption we get a packet compressed with gzip.</p>
<h3 id="peer-to-cc-encryption">Peer to C&amp;C Encryption</h3>
<p>As before, the malware uses a combination of RSA, RC4 encryption, and HTTPS to communicate with the C&amp;C. In this case, peers work as proxy servers. An encrypted packet has the following structure: 4-byte CRC, followed by RSA_BLOB. After decrypting RSA (request packets cannot be decrypted without the C&amp;C private key), we get a GZIP packet.</p>

<li>02h – Formgrabber</li>
<li>08h – Redirects</li>
<li>etc.</li>
</ul>
<p>The only thing that has changed is the encryption of strings in the configuration file – RC4 is now used.</p>

<h3 id="geographical-distribution">Geographical Distribution</h3>
<p><a href="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2017/05/07174529/dridex_en_14.png" class="magnificImage"><img loading="lazy" decoding="async" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2017/05/07174529/dridex_en_14.png" alt="Dridex: A History of Evolution" width="848" height="464" class="aligncenter size-full wp-image-78558" srcset="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2017/05/07174529/dridex_en_14.png 848w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2017/05/07174529/dridex_en_14-300x164.png 300w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2017/05/07174529/dridex_en_14-768x420.png 768w" sizes="auto, (max-width: 848px) 100vw, 848px" /></a></p>
<p>The developers of Dridex look for potential victims in Europe. Between January 1st and early April 2017, we detected Dridex activity in several European countries. The UK accounted for more than half (nearly 60%) of all detections, followed by Germany and France. At the same time, the malware never works in Russia, as the C&amp;Cs detect the country via IP address and do not respond if the country is Russia.</p>
<h2 id="conclusion">Conclusion</h2>
<p>In the several years that the Dridex family has existed, there have been numerous unsuccessful attempts to block the botnet&#8217;s activity. The ongoing evolution of the malware demonstrates that the cybercriminals are not about to bid farewell to their brainchild, which is providing them with a steady revenue stream. For example, Dridex developers continue to implement new techniques for evading the User Account Control (UAC) system. These techniques enable the malware to run its malicious components on Windows systems.</p>
<p>It can be surmised that the same people, possibly Russian speakers, are behind the Dridex and Zeus Gameover Trojans, but we do not know this for a fact. The damage done by the cybercriminals is also impossible to assess accurately. Based on a very rough estimate, it has reached hundreds of millions of dollars by now. Furthermore, given the way that the malware is evolving, it can be assumed that a significant part of the &#8220;earnings&#8221; is reinvested into the banking Trojan&#8217;s development.</p>
