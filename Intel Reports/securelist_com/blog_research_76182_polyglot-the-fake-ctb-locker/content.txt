												<div class="c-wysiwyg">
													<p>Cryptor malware programs currently pose a very real cybersecurity threat to users and companies. Clearly, organizing effective security requires the use of security solutions that incorporate a broad range of technologies capable of preventing a cryptor program from landing on a potential victim&#8217;s computer or reacting quickly to stop an ongoing data encryption process and roll back any malicious changes. However, what can be done if an infection does occur and important data has been encrypted? (Infection can occur on nodes that, for whatever reason, were not protected by a security solution, or if the solution was disabled by an administrator.) In this case, the victim&#8217;s only hope is that the attackers made some mistakes when implementing the cryptographic algorithm, or used a weak encryption algorithm.</p>
<h2 id="a-brief-description">A brief description</h2>
<p>The cryptor dubbed Polyglot emerged in late August. According to the information available to us, it is distributed in spam emails that contain a link to a malicious RAR archive. The archive contains the cryptor&#8217;s executable code.</p>
<p>Here are some examples of the links used:</p>
<p><i>hXXp://bank-info.gq/downloads/reshenie_suda.rar</i></p>
<p><i>hXXp://bank-info.gq/downloads/dogovor.rar</i></p>
<p>When the infected file is launched, nothing appears to happen. However, the cryptor copies itself under random names to a dozen or so places, writes itself to the autostart folder and to TaskScheduler. When the installation is complete, file encryption starts. The user&#8217;s files do not appear to change (their names remain the same), but the user is no longer able to open them. </p>
<p><a href="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2016/09/07184356/zipcryptor_eng_1.png" class="magnificImage"><img loading="lazy" decoding="async" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2016/09/07184356/zipcryptor_eng_1.png" alt="Polyglot - the fake CTB-locker" width="789" height="514" class="aligncenter size-full wp-image-76190" srcset="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2016/09/07184356/zipcryptor_eng_1.png 789w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2016/09/07184356/zipcryptor_eng_1-300x195.png 300w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2016/09/07184356/zipcryptor_eng_1-768x500.png 768w" sizes="auto, (max-width: 789px) 100vw, 789px" /></a></p>
<p>When encryption is complete, the cryptor changes the desktop wallpaper, (interestingly, the wallpaper image is unique to each victim) and displays the ransom message. </p>

<p>After this, the user is told to pay for file decryption in bitcoins. The cryptor contacts its C&amp;C, which is located on the Tor network, for the ransom sum and the bitcoin address where it should be sent.</p>

<p>From this moment on, the cryptor allows the user to check the ransom payment status on the C&amp;C. 
</p>

<p style="text-align:center;font-weight:bold"><em>Ransom payment details</em></p>
<p>If the ransom is not paid on time, the cryptor notifies the user that it&#8217;s no longer possible to decrypt their files, and that it is about to &#8216;self-delete&#8217;. </p>
<p><a href="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2016/09/07184333/zipcryptor_eng_6.png" class="magnificImage"><img loading="lazy" decoding="async" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2016/09/07184333/zipcryptor_eng_6.png" alt="Polyglot - the fake CTB-locker" width="785" height="515" class="aligncenter size-full wp-image-76195" srcset="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2016/09/07184333/zipcryptor_eng_6.png 785w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2016/09/07184333/zipcryptor_eng_6-300x197.png 300w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2016/09/07184333/zipcryptor_eng_6-768x504.png 768w" sizes="auto, (max-width: 785px) 100vw, 785px" /></a></p>

<h2 id="imitating-ctb-locker">Imitating CTB-Locker</h2>
<p>Initially, this cryptor caught our attention because it mimics all the features of another widespread cryptor – CTB-Locker (Trojan-Ransom.Win32.Onion). The graphical interface window, language switch, the sequence of actions for requesting the encryption key, the payment page, the desktop wallpapers – all of them are very similar to those used by CTB-Locker. The visual design has been copied very closely, while the messages in Polyglot&#8217;s windows have been copied word for word.</p>

<p>That said, we should note the following: a detailed analysis has revealed that Polyglot was developed independently from CTB-Locker; in other words, no shared code has been detected in the two Trojans (except the publicly available DLL code). Perhaps the creators of Polyglot wanted to disorient the victims and researchers, and created a near carbon copy of CTB-Locker from scratch to make it look like a CTB-Locker attack and that there was no hope of getting files decrypted for free.</p>
<h2 id="cc-communication">C&amp;C communication</h2>
<p>The Trojan contacts the C&amp;C server located on Tor via a public tor2web service, using the HTTP protocol.</p>
<p>Prior to each of the below data requests, a POST request is sent with the just one parameter: &#8220;live=1&#8221;.</p>
<p><b>Request 1.</b></p>
<p>At the start of operation, the Trojan reports the successful infection to the C&amp;C. The following data is sent to the C&amp;C:</p>
<p>{<br />&#8220;ip&#8221;:&#8221;xxx.xxx.xxx.xxx&#8221;, &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;//ip address of the infected computer<br />&#8220;method&#8221;:&#8221;register&#8221;, &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;//action type. &#8220;register&#8221; = Trojan informs C&amp;C of new infection<br />&#8220;uid&#8221;:&#8221;xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx&#8221;, &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;//Infected computer&#8217;s ID<br />&#8220;version&#8221;:&#8221;10f&#8221;, &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;//Trojan version contained in its body<br />&#8220;info&#8221;:&#8221;Microsoft (build xxxx), 64-bit&#8221;, &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;//OS version on the infected computer<br />&#8220;description&#8221;:&#8221; &#8220;, &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;//Always a whitespace (&#8221; &#8220;)<br />&#8220;start_time&#8221;:&#8221;14740xxxxx&#8221;, &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;//Trojan&#8217;s start time<br />&#8220;end_time&#8221;:&#8221;0&#8243;, &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;//Encryption finish time. 0 = no encryption has run yet<br />&#8220;user_id&#8221;:&#8221;5&#8243; &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;//Number hardwired in the sample<br />}</p>
<p>This data block is passed through a bitwise NOT operation, encoded into Base64 and sent to the C&amp;C in a POST request.</p>
<p><a href="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2016/09/07184245/zipcryptor_eng_17.png" class="magnificImage"><img loading="lazy" decoding="async" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2016/09/07184245/zipcryptor_eng_17.png" alt="Polyglot - the fake CTB-locker" width="642" height="178" class="aligncenter size-full wp-image-76206" srcset="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2016/09/07184245/zipcryptor_eng_17.png 642w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2016/09/07184245/zipcryptor_eng_17-300x83.png 300w" sizes="auto, (max-width: 642px) 100vw, 642px" /></a></p>
<p style="text-align:center;font-weight:bold"><em>Contents of the sent request</em></p>
<p>Parameters of the POST request:</p>
<p>signature – CRC32 from the sent data<br />ver – Trojan version<br />gcdata – data, with contents as described above.</p>
<p><a href="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2016/09/07184241/zipcryptor_eng_18.png" class="magnificImage"><img loading="lazy" decoding="async" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2016/09/07184241/zipcryptor_eng_18.png" alt="Polyglot - the fake CTB-locker" width="743" height="797" class="aligncenter size-full wp-image-76207" srcset="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2016/09/07184241/zipcryptor_eng_18.png 743w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2016/09/07184241/zipcryptor_eng_18-280x300.png 280w" sizes="auto, (max-width: 743px) 100vw, 743px" /></a></p>
<p style="text-align:center;font-weight:bold"><em>Request 1 and the reply received from the C&amp;C</em></p>
<p><b>Request 2.</b></p>
<p>When the Trojan has finished encrypting the user&#8217;s data, it sends another request to the C&amp;C. The content of the request is identical to that of request 1 except the field &#8220;end_time&#8221;, which now shows the time encryption was completed.</p>
<p><b>Request 3.</b></p>
<p>This is sent to the C&amp;C to request the bitcoin address for payment and the ransom sum to be paid.</p>
<p>{<br />&#8220;method&#8221;:&#8221;getbtcpay&#8221;<br />&#8220;uid&#8221;:&#8221;xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx&#8221;<br />}</p>
<p>The C&amp;C replies to this request with the following data:</p>
<p>{<br />&#8220;code&#8221;:&#8221;0&#8243;,<br />&#8220;text&#8221;:&#8221;OK&#8221;,<br />&#8220;address&#8221;:&#8221;xxxxxxxx&#8221;, &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;//bitcoin address (may vary)<br />&#8220;btc&#8221;:0.7, &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;//amount to be paid in BTC (may vary)<br />&#8220;usd&#8221;:319.98 &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;//amount to be paid in USD (may vary)<br />}</p>
<p><b>Request 4.</b></p>
<p>This is sent to request a file decryption key from the C&amp;C.</p>
<p>{<br />&#8220;method&#8221;:&#8221;getkeys&#8221;,<br />&#8220;key&#8221;:&#8221;&#8221;,<br />&#8220;uid&#8221;:&#8221;xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx&#8221;,<br />&#8220;info&#8221;:[&#8220;DYqbX3m9u0Pk9bE9Rg2Co3empC2M/yrnqgNS3r0AT2vwCw8Zas08bd4BNiO3XuAqi6/5WQ0VBiUkRUToo+YFL/QtPkiRIQ/D9RyKhzpBHlNpf2hPb9eloDzpkonQl7L6cQyJ2FipEG2ggZOdTDBcNAEAAAA=&#8221;]
<br />}</p>
<p><b>Request 5.</b></p>
<p>The Trojan reports that data decryption has been completed and states the number of decrypted files to the C&amp;C.</p>
<p>{<br />&#8220;method&#8221;:&#8221;setend&#8221;,<br />&#8220;uid&#8221;:&#8221;xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx&#8221;,<br />&#8220;decrypted&#8221;:&#8221;1&#8243;<br />}</p>
<h2 id="description-of-the-encryption-algorithm">Description of the encryption algorithm</h2>
<p>During our analysis of the malicious code, it became evident that the Trojan encrypts files in three stages, creating intermediate files:</p>
<ul style="margin-bottom:30px">
<li>First, the original file is placed in a password-protected ZIP archive. The archive has the same name as the original file plus the extension &#8220;a19&#8221;;</li>
<li>Polyglot encrypts the password-protected archive with the AES-256-ECB algorithm. The resulting file again uses the name of the original file, but the extension is now changed to &#8220;ap19&#8221;;</li>
<li>The Trojan deletes the original file and the file with the extension &#8220;a19&#8221;. The extension of the resulting file is changed from &#8220;ap19&#8221; to that of the original file.</li>
</ul>
<p><a href="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2016/09/07184236/zipcryptor_eng_19.png" class="magnificImage"><img loading="lazy" decoding="async" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2016/09/07184236/zipcryptor_eng_19-1024x382.png" alt="Polyglot - the fake CTB-locker" width="604" height="225" class="aligncenter size-large wp-image-76208" srcset="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2016/09/07184236/zipcryptor_eng_19-1024x382.png 1024w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2016/09/07184236/zipcryptor_eng_19-300x112.png 300w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2016/09/07184236/zipcryptor_eng_19-768x287.png 768w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2016/09/07184236/zipcryptor_eng_19.png 1171w" sizes="auto, (max-width: 604px) 100vw, 604px" /></a></p>
<p style="text-align:center;font-weight:bold"><em>Flowchart of the search and file encryption actions performed by Polyglot</em></p>
<p>A separate AES key is generated for each file, and is nothing more than a &#8216;shared secret&#8217; generated according to the Diffie-Hellman protocol on an elliptic curve. However, first things first.</p>
<p>Before encrypting any files, the Trojan generates two random sequences, each 32 bytes long. The <i>SHA256 </i>digests of each sequence become the private keys <b>s_ec_priv_1 </b>and <b>s_ec_priv_2</b>. Then, the Bernstein elliptic curve (Curve25519) is used to obtain public keys <b>s_ec_pub_1</b> and <b>s_ec_pub_2 </b>(respectively) from each private key.</p>
<p>The Trojan creates the structure <b>decryption_info</b> and writes the following to it: a random sequence used as the basis for creating the key<b> s_ec_priv_1</b>, the string <b>machine_guid</b> taken from the registry, and a few zero bytes.</p>
<div align="center" style="margin-bottom:15px">
<table border="1" cellspacing="0" cellpadding="0" style="width: 90%">
<tbody>
<tr>
<td style="padding: 30px">struct decryption_info</p>
<p>{<br />
&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;char s_rand_str_1[32];<br />
&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;char machine_guid[36];<br />
&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;char zeroes[12];<br />
};
</p>
</td>
</tr>

<p>Using the private key <b>s_ec_priv_2 </b>and the cybercriminal&#8217;s public key<b> mal_pub_key </b>produces the shared secret <b>mal_shared_secret</b> = ECDH(<i>s_ec_priv_2</i>, <i>mal_pub_key</i>). The structure <i>decryption_info </i>is encrypted with algorithm <i>AES-256-ECB </i>using a key that is the <i>SHA256</i> digest of this secret. For convenience, we shall call the obtained 80 bytes of the encrypted structure <b>encrypted_info</b>.</p>
<p>Only when Polyglot obtains the <i>encrypted_info</i> value does it proceed to generate the session key AES for the file. Using the above method, a new pair of keys is generated, <b>f_priv_key</b> and <b>f_pub_key</b>. Using <b>f_priv_key</b> and <b>s_ec_pub_1 </b>produces the shared secret <b>f_shared_secret</b> = ECDH(<i>f_priv_key</i>, <i>s_ec_pub_1</i>). </p>
<p>The SHA256 digest of this secret will be the AES key with which the file is encrypted.</p>
<p>To specify that the file has already been encrypted and that it&#8217;s possible to decrypt the file, the cybercriminals write the structure <b>file_info</b> to the start of each encrypted file:</p>
<div align="center" style="margin-bottom:15px">

<tbody>
<tr>
<td style="padding: 30px">struct file_info</p>

<p>The elliptic curve, the Diffie-Hellman protocol, AES-256, a password-protected archive – it was almost flawless. But not quite, because the creator of Polyglot made a few mistakes during implementation. This gave us the opportunity to help the victims and restore files that had been encrypted by Polyglot.</p>
<h2 id="mistakes-made-by-the-creators">Mistakes made by the creators</h2>
<p>As was mentioned earlier, all the created keys are based on a randomly generated array of characters. Therefore, the strength of the keys is determined by the generator&#8217;s strength. And we were surprised to see the implementation of this generator:</p>
<p><a href="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2016/09/07184232/zipcryptor_eng_20.png" class="magnificImage"><img loading="lazy" decoding="async" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2016/09/07184232/zipcryptor_eng_20.png" alt="Polyglot - the fake CTB-locker" width="629" height="780" class="aligncenter size-full wp-image-76209" srcset="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2016/09/07184232/zipcryptor_eng_20.png 629w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2016/09/07184232/zipcryptor_eng_20-242x300.png 242w" sizes="auto, (max-width: 629px) 100vw, 629px" /></a></p>
<p style="text-align:center;font-weight:bold"><em>A graphical representation of the random sequence generation procedure</em></p>
<p>Let&#8217;s convert this function into pseudocode so it&#8217;s easier to follow:</p>
<p><a href="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2016/09/07184229/zipcryptor_eng_21.png" class="magnificImage"><img loading="lazy" decoding="async" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2016/09/07184229/zipcryptor_eng_21.png" alt="Polyglot - the fake CTB-locker" width="320" height="158" class="aligncenter size-full wp-image-76210" srcset="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2016/09/07184229/zipcryptor_eng_21.png 320w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2016/09/07184229/zipcryptor_eng_21-300x148.png 300w" sizes="auto, (max-width: 320px) 100vw, 320px" /></a></p>
<p>Please note that when another random byte is selected, the entire result of the function <i>rand()</i> is not used, just the remainder of dividing the result by 32. Only the cybercriminal knows why they decided to make the random string this much weaker – an exhaustive search of the entire set of the possible keys produced by such a pseudo-random number generator will only take a few minutes on a standard PC. </p>
<p>Taking advantage of this mistake, we were able to calculate the AES key for an encrypted file. Although there was a password-protected archive below the layer of symmetric encryption, we already knew that the cybercriminal had made another mistake.</p>
<p>Let&#8217;s look at how the archive key is generated:</p>
<p><a href="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2016/09/07184226/zipcryptor_eng_22.png" class="magnificImage"><img loading="lazy" decoding="async" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2016/09/07184226/zipcryptor_eng_22.png" alt="Polyglot - the fake CTB-locker" width="247" height="86" class="aligncenter size-full wp-image-76211" /></a></p>
<p>We can see that the key length is only 4 bytes; moreover, these are specific bytes from the string MachineGuid, the unique ID assigned to the computer by the operating system. Furthermore, a slightly modified MachineGuid string is displayed in the requirements text displayed to the victim; this means that if we know the positions in which the 4 characters of the ZIP archive password are located, we can easily unpack the archive.</p>
<p><a href="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2016/09/07184221/zipcryptor_eng_23.png" class="magnificImage"><img loading="lazy" decoding="async" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2016/09/07184221/zipcryptor_eng_23-1024x491.png" alt="Polyglot - the fake CTB-locker" width="604" height="290" class="aligncenter size-large wp-image-76212" srcset="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2016/09/07184221/zipcryptor_eng_23-1024x491.png 1024w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2016/09/07184221/zipcryptor_eng_23-300x144.png 300w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2016/09/07184221/zipcryptor_eng_23-768x368.png 768w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2016/09/07184221/zipcryptor_eng_23.png 1253w" sizes="auto, (max-width: 604px) 100vw, 604px" /></a></p>
