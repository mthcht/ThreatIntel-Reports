<h3><a name="1"></a>Executive Summary</h3>
<p>In October 2012, Kaspersky Lab&#8217;s Global Research &amp; Analysis Team initiated a new threat research after a series of attacks against computer networks of various international diplomatic service agencies. A large scale cyber-espionage network was revealed and analyzed during the investigation, which we called &#8220;Red October&#8221; (after famous novel &#8220;The Hunt For The Red October&#8221;).</p>
<p>This report is based on detailed technical analysis of a series of targeted attacks against diplomatic, governmental and scientific research organizations in different countries, mostly related to the region of Eastern Europe, former USSR members and countries in Central Asia.</p>
<p>The main objective of the attackers was to gather intelligence from the compromised organizations, which included computer systems, personal mobile devices and network equipment.</p>
<p>The earliest evidence indicates that the cyber-espionage campaign was active since 2007 and is still active at the time of writing (January 2013).<br />
Besides that, registration data used for the purchase of several Command &amp; Control (C&amp;C) servers and unique malware filenames related to the current attackers hints at even earlier time of activity dating back to May 2007.</p>
<h3 id="main-findings">Main Findings</h3>
<p><strong>Advanced Cyber-espionage Network</strong>: The attackers have been active for at least several years, focusing on diplomatic and governmental agencies of various countries across the world.</p>
<p>Information harvested from infected networks was reused in later attacks. For example, stolen credentials were compiled in a list and used when the attackers needed to guess secret phrase in other locations. To control the network of infected machines, the attackers created more than 60 domain names and several server hosting locations in different countries (mainly Germany and Russia). The C&amp;C infrastructure is actually a chain of servers working as proxies and hiding the location of the &#8216;mothership&#8217; control server.</p>
<p><strong>Unique architecture</strong>: The attackers created a multi-functional kit which has a capability of quick extension of the features that gather intelligence. The system is resistant to C&amp;C server takeover and allows the attack to recover access to infected machines using alternative communication channels.</p>
<p><strong>Broad variety of targets</strong>: Beside traditional attack targets (workstations), the system is capable of stealing data from mobile devices, such as smartphones (iPhone, Nokia, Windows Mobile), enterprise network equipment (Cisco), removable disk drives (including already deleted files via a custom file recovery procedure).</p>
<p><strong>Importation of exploits</strong>: The samples we managed to find were using exploit code for vulnerabilities in Microsoft Word and Microsoft Excel that were created by other attackers and employed during different cyber attacks. The attackers left the imported exploit code untouched, perhaps to harden the identification process.</p>
<p><strong>Attacker identification</strong>: Basing on registration data of C&amp;C servers and numerous artifacts left in executables of the malware, we strongly believe that the attackers have Russian-speaking origins. Current attackers and executables developed by them have been unknown until recently, they have never related to any other targeted cyberattacks.</p>
<h3><a name="2"></a>Anatomy of the attack</h3>
<h4><a name="3"></a>General description</h4>
<p>These attacks comprised of the classical scenario of specific targeted attacks, consisting of two major stages:</p>
<ol>
<li>Initial infection</li>
<li>Additional modules deployed for intelligence gathering</li>
</ol>
<p>The malicious code was delivered via e-mail as attachments (Microsoft Excel, Word and, probably PDF documents) which were rigged with exploit code for known security vulnerabilities in the mentioned applications. In addition to Office documents (CVE-2009-3129, CVE-2010-3333, CVE-2012-0158), it appears that the attackers also infiltrated victim network(s) via Java exploitation (known as the &#8216;Rhino&#8217; exploit (CVE-2011-3544).</p>
<p>Right after the victim opened the malicious document or visit malicious URL on a vulnerable system, the embedded malicious code initiated the setup of the main component which in turn handled further communication with the C&amp;C servers.</p>
<p>Next, the system receives a number of additional spy modules from the C&amp;C server, including modules to handle infection of smartphones.</p>
<p>The main purpose of the spying modules is to steal information. This includes files from different cryptographic systems, such as &#8220;Acid Cryptofiler&#8221;, (see <a href="https://fr.wikipedia.org/wiki/Acid_Cryptofiler">https://fr.wikipedia.org/wiki/Acid_Cryptofiler</a>) which is known to be used in organizations of European Union/European Parliament/European Commission since the summer of 2011. All gathered information is packed, encrypted and only then transferred to the C&amp;C server.</p>
<h4><a name="4"></a>Step-by-step description (1st stage)</h4>
<p>During our investigation we couldn&#8217;t find any e-mails used in the attacks, only top level dropper documents. Nevertheless, based on indirect evidence, we know that the e-mails can be sent using one of the following methods:</p>
<ul>
<li>Using an anonymous mailbox from a free public email service provider</li>
<li>Using mailboxes from already infected organizations</li>
</ul>
<p>E-mail subject lines as well as the text in e-mail bodies varied depending on the target (recipient). The attached file contained the exploit code which activated a Trojan dropper in the system.</p>
<p>We have observed the use of at least three different exploits for previously known vulnerabilities: CVE-2009-3129 (MS Excel), CVE-2010-3333 (MS Word) and CVE-2012-0158 (MS Word). The first attacks that used the exploit for MS Excel started in 2010, while attacks targeting the MS Word vulnerabilities appeared in the summer of 2012.</p>
<p>As a notable fact, the attackers used exploit code that was made public and originally came from a previously known targeted attack campaign with Chinese origins. The only thing that was changed is the executable which was embedded in the document; the attackers replaced it with their own code.</p>
<p>The embedded executable is a file-dropper, which extracts and runs three additional files.</p>
<p><small><br />
%TEMP%MSC.BAT<br />
%ProgramFiles%WINDOWS NTLHAFD.GCP (&lt;- This file name varies)<br />
%ProgramFiles%WINDOWS NTSVCHOST.EXE<br />
</small></p>
<p>MSC.BAT file has the following contents:</p>
<p><small>chcp 1251<br />
:Repeat<br />
attrib -a -s -h -r &#8220;%DROPPER_FILE%&#8221;<br />
del &#8220;%DROPPER_FILE%&#8221;<br />
if exist &#8220;%DROPPER_FILE%&#8221; goto Repeat<br />
del &#8220;%TEMP%msc.bat&#8221;</small></p>
<p>Another noteworthy fact is in the first line of this file, which is a command to switch the codepage of an infected system to 1251. This is required to address files and directories that contain Cyrillic characters in their names.</p>
<p align="center"><img decoding="async" title="" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2013/01/08212505/red_october_1.jpg" alt="" border="1" /></p>
<p>The &#8220;LHAFD.GCP&#8221; file is encrypted with RC4 and compressed with the &#8220;Zlib&#8221; library. This file is essentially a backdoor, which is decoded by the loader module (svchost.exe). The decrypted file is injected into system memory and is responsible for communication with the C&amp;C server.</p>
<p align="center"><img decoding="async" title="" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2013/01/08212501/red_october_2.jpg" alt="" border="1" /></p>
<p>On any infected system, every major task is performed by the main backdoor component. The main component is started only after its loader (&#8220;svchost.exe&#8221;) checks if the internet connection is available. It does so by connecting to three Microsoft hosts:</p>
<ul>
<li>update.microsoft.com</li>
<li>www.microsoft.com</li>
<li>support.microsoft.com</li>
</ul>
<p align="center"><img decoding="async" title="" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2013/01/08212457/red_october_3.jpg" alt="" border="1" /><br />
<small>Figure &#8211; Hosts used to validate internet connection</small></p>
<p>After the Internet connection is validated, the loader executes the main backdoor component that connects to its C&amp;C servers:</p>
<p align="center"><img decoding="async" title="" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2013/01/08212453/red_october_4.png" alt="" border="1" /><br />
<small>Capture of malware&#8217;s communication with the C2</small></p>
<p>The connections with the C&amp;C are encrypted &#8211; different encryption algorithms are used to send and receive data.</p>
<p align="center"><img decoding="async" title="" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2013/01/08212449/red_october_5.jpg" alt="" border="1" /><br />
<small>Encrypted communication with the C2</small></p>
<p>During our investigation, we found more than 60 different command-and-control domains. Each malware sample contains three such domains, which are hardcoded inside the main backdoor component:</p>
<p align="center"><img decoding="async" title="" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2013/01/08212445/red_october_6.jpg" alt="" border="1" /><br />
<small>Hardcoded C2 domains inside backdoor</small></p>
<h4><a name="5"></a>Step-by-step description (2nd stage)</h4>
<p>After a connection with the C&amp;C server is established, the backdoor starts the communication process, which leads to the loading of additional modules. These modules can be split into two categories: &#8220;offline&#8221; and &#8220;online&#8221;.<br />
The main difference between these categories is their behavior on the infected system:</p>
<ul>
<li>&#8220;Offline&#8221;: exists as files on local disk, capable of creating its own system registry keys, local disk log files, and may communicate with C&amp;C servers on their own.</li>
<li>&#8220;Online&#8221;: exists only in system memory and is never saved to local disk, do not create registry keys, all logs are also kept in memory instead of local disk and sends the result of work to the C&amp;C server using own code.</li>
</ul>
<p align="center"><img decoding="async" title="" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2013/01/08212441/red_october_7.jpg" alt="" border="1" /></p>
<p>There is a notable module among all others, which is essentially created to be embedded into Adobe Reader and Microsoft Office applications. The main purpose of its code is to create a foolproof way to regain access to the target system. The module expects a specially crafted document with attached executable code and special tags. The document may be sent to the victim via e-mail. It will not have an exploit code and will safely pass all security checks. However, like with exploit case, the document will be instantly processed by the module and the module will start a malicious application attached to the document.</p>
<p>This trick can be used to regain access to the infected machines in case of unexpected C&amp;C servers shutdown/takeover.</p>

<h3><a name="11"></a>С&amp;C information</h3>
<p>A list of the most popular domains used for command and control can be found below:</p>
<p class="c"><img decoding="async" title="" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2013/01/08212423/red_october_11.jpg" alt="" border="1" /></p>
<p>Interestingly, although the domain &#8220;dll-host-update.com&#8221; appears in one of the malware configurations, it had not been registered by the attackers. The domain has since been registered by Kaspersky Lab on Nov 2nd, 2012 to monitor the attacker&#8217;s activities.</p>
<p>Another interesting example is &#8220;dll-host-udate.com&#8221; &#8211; the &#8220;udate&#8221; part appears to be a typo.</p>
<p>All the domains used by attackers appear to have been registered between 2007-2012. The oldest known domain was registered in Nov 2007; the newest on May 2012.</p>
<p>Most of the domains have been registered using the service &#8220;reg.ru&#8221;, but other services such as &#8220;webdrive.ru&#8221;, &#8220;webnames.ru&#8221; or &#8220;timeweb.ru&#8221; have been used as well.</p>
<p>During our monitoring, we observed the domains pointing to several malicious webservers. A list of servers with confirmed malicious behavior can be found below.</p>
<p>In total, we have identified 10 different servers which exhibited confirmed malicious behavior. Most of these severs are located in Germany, at Hetzner Online Ag.</p>
<p class="c"><img decoding="async" title="" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2013/01/08212420/red_october_12.jpg" alt="" border="1" /></p>
<p>During our analysis, we were able to obtain an image of one of the command-and-control servers. The server itself proved to be a proxy, which was forwarding the request to another server on port 40080. The script responsible for redirections was found in /root/scp.pl and relies on the &#8220;socat&#8221; tool for stream redirection.</p>
<p>By scanning the Internet for computers with port 40080 open, we were able to identify three such servers in total, which we call &#8220;mini-motherships&#8221;:</p>
<p align="center"><img decoding="async" title="" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2013/01/08212416/red_october_13.jpg" alt="" border="1" /></p>

<p align="center"><img decoding="async" title="" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2013/01/08212412/red_october_14.jpg" alt="" border="1" /></p>

<p>For instance, the script &#8220;/cgi-bin/nt/th&#8221; is being used to receive commands from the command-and-control server, usually in the form of new plugins to run on the victim&#8217;s computer. The &#8220;/cgi-bin/nt/sk&#8221; script is called by the running plugins to upload stolen data and information about the victim.</p>
<p>When connecting to the C&amp;C, the backdoor identifies itself with a specific string which includes a hexadecimal value that appears to be the victim&#8217;s unique ID. Different variants of the backdoor contain different victim IDs. Presumably, this allows the attackers to distinguish between the multitudes of connections and perform specific operations for each victim individually.</p>
<p>For instance, a top level XLS dropper presumably used against a Polish target, named &#8220;Katyn_-_opinia_Rosjan.xls&#8221; contains the hardcoded victim ID &#8220;F50D0B17F870EB38026F&#8221;. A similar XLS named &#8220;tactlist_05-05-2011_.8634.xls / EEAS New contact list (05-05-2011).xls&#8221; possibly used in Moldova contains a victim ID &#8220;FCF5E48A0AE558F4B859&#8221;.</p>
<p><strong>Part 2 of this paper will cover malware modules and provide more technical details about their operation.</strong></p>
