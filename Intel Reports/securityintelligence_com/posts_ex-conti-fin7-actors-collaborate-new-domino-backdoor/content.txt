&lt;strong&gt;April 27, 2023 Update&lt;/strong&gt;
&lt;p class=&quot;xxmsonormal0&quot;&gt;&lt;span lang=&quot;EN-US&quot; data-ogsc=&quot;black&quot;&gt;This article is being republished with modifications from the original that was published on April 14, 2023, to change the name of the family of malware from Domino to Minodo. This is being done to avoid any possible confusion with the HCL Domino brand. The family of malware that is described in this article is unrelated to, does not impact, nor uses HCL Domino or any of its components in any way. The malware is not associated with HCL or its Domino product suite in any way.&lt;/span&gt;&lt;/p&gt;


&lt;hr /&gt;

&lt;em&gt;This blog was made possible through contributions from Christopher Caridi. &lt;/em&gt;

IBM Security X-Force recently discovered a new malware family we have called “Minodo,” which we assess was created by developers associated with the cybercriminal group that X-Force tracks as ITG14, also known as FIN7. Former members of the Trickbot/Conti syndicate which X-Force tracks as ITG23 have been using Minodo since at least late February 2023 to deliver either the Project Nemesis information stealer or more capable backdoors such as Cobalt Strike.
&lt;h2&gt;Background&lt;/h2&gt;
This discovery highlights the intricate nature of cooperation among cybercriminal groups and their members:
&lt;ul type=&quot;disc&quot;&gt;
 	&lt;li&gt;Since late February 2023, Minodo Backdoor campaigns have been observed using the &lt;a href=&quot;https://securityintelligence.com/posts/itg23-crypters-cooperation-between-cybercriminal-groups/&quot;&gt;Dave Loader&lt;/a&gt;, which we have linked to the Trickbot/Conti syndicate and its former members.&lt;/li&gt;
 	&lt;li&gt;Minodo’s code shows overlap with the Lizar (aka Tirion, Diceloader) malware family, leading us to suspect that it was created by current or former ITG14 developers.&lt;/li&gt;
 	&lt;li&gt;One of Minodo’s final payloads is the Project Nemesis infostealer. Project Nemesis was first advertised on the dark web in December 2021, though has been rarely used since then.&lt;/li&gt;
&lt;/ul&gt;
&lt;h2&gt;Analysis&lt;/h2&gt;
&lt;h3&gt;Ex-Conti members deploy Minodo in recent campaigns&lt;/h3&gt;
Former members of ITG23 (aka the Trickbot/Conti syndicate) are likely behind recent campaigns using the Dave Loader to load Minodo Backdoor and probably collaborated with current or former ITG14 developers to purchase or use the new malware family. X-Force &lt;a href=&quot;https://securityintelligence.com/posts/itg23-crypters-cooperation-between-cybercriminal-groups/&quot;&gt;previously assessed&lt;/a&gt; that Dave is one of several loaders or crypters developed by members of the Trickbot/Conti group. Although the group has fractured, many of its loaders/crypters — including Dave — have been maintained and continue to be used by factions composed of former Trickbot/Conti members, including Quantum, Royal, BlackBasta, and Zeon.
&lt;ul type=&quot;disc&quot;&gt;
 	&lt;li&gt;The Dave Loader has been used recently with several Cobalt Strike samples with the watermark “206546002,” which X-Force and other security researchers — &lt;a href=&quot;https://www.trendmicro.com/en_us/research/22/i/play-ransomware-s-attack-playbook-unmasks-it-as-another-hive-aff.html&quot;&gt;here&lt;/a&gt; and &lt;a href=&quot;https://medium.com/walmartglobaltech/from-royal-with-love-88fa05ff7f65&quot;&gt;here&lt;/a&gt; — have associated with groups composed of former members of the Trickbot/Conti syndicate, including Quantum and Royal. X-Force observed Dave-loaded Cobalt Strike samples using this watermark in suspected Royal attacks in fall 2022.&lt;/li&gt;
 	&lt;li&gt;Dave Loader has also been used this year to load IcedID and Emotet, both of which serve as initial access vectors for ransomware attacks from former Trickbot/Conti-affiliated factions.&lt;/li&gt;
&lt;/ul&gt;
Recently observed Dave samples were discovered loading a new malware, which we have named Minodo Backdoor. This new backdoor gathers basic system information, which it then sends to the C2, and in return receives an AES encrypted payload.

In most instances, the received payload was a second loader that was found to have code overlap with Minodo Backdoor, and as such we have dubbed it Minodo Loader. This loader contains an encrypted payload within its resources, which it decrypts using AES. The decrypted payload is a .NET infostealer, which identifies itself as &#039;Nemesis Project.&#039;

The Minodo Backdoor is designed to contact a different C2 address for domain-joined systems, suggesting a more capable backdoor, such as Cobalt Strike, will be downloaded on higher value targets instead of Project Nemesis.

&lt;a href=&quot;https://securityintelligence.com/wp-content/uploads/2023/04/Minodo-Chain.jpg&quot;&gt;&lt;img class=&quot;alignnone size-full wp-image-441835&quot; src=&quot;https://securityintelligence.com/wp-content/uploads/2023/04/Minodo-Chain.jpg&quot; alt=&quot;Chart depicting the Minodo backdoor attack chain.&quot; width=&quot;995&quot; height=&quot;601&quot; /&gt;&lt;/a&gt;
&lt;h3&gt;Minodo’s connections to ITG14&lt;/h3&gt;
Analysis revealed that both the Minodo Backdoor and Loader share code overlap with the &lt;a href=&quot;https://bi-zone.medium.com/from-pentest-to-apt-attack-cybercriminal-group-fin7-disguises-its-malware-as-an-ethical-hackers-c23c9a75e319&quot;&gt;Lizar&lt;/a&gt; Malware, also known as Tirion and DiceLoader, which is attributed to the threat group ITG14 (FIN7). In addition to having similar coding styles and functionality, both Minodo and DiceLoader share the same configuration structure and have similar bot ID formats. Lizar was &lt;a href=&quot;https://www.prodaft.com/blog/detail/opblueraven-unveiling-fin7carbanak-part-i-tirion&quot;&gt;reportedly first used in March 2020&lt;/a&gt;, when it was originally named Tirion, and has been observed in numerous ITG14 campaigns up to the end of 2022. Minodo has been active in the wild since at least October 2022, which notably is when Lizar observations began to decrease.

X-Force researchers found additional evidence connecting Minodo Backdoor to ITG14’s Carbanak Backdoor. We identified Minodo Backdoor samples from December 2022, which used a different loader that we have dubbed NewWorldOrder Loader. The samples used the name &lt;em&gt;ThunderboltService.exe&lt;/em&gt; and downloaded and loaded the Project Nemesis Stealer.

Around the same time, we also uncovered NewWorldOrder Loader samples, with the same filename &lt;em&gt;ThunderboltService.exe&lt;/em&gt;, used to load the Carbanak Backdoor. &lt;a href=&quot;https://www.mandiant.com/resources/blog/fin7-pursuing-an-enigmatic-and-evasive-global-criminal-operation&quot;&gt;Carbanak&lt;/a&gt; has been used by ITG14 since late 2015.

&lt;a href=&quot;https://securityintelligence.com/wp-content/uploads/2023/04/NWO-Loader.jpg&quot;&gt;&lt;img class=&quot;alignnone size-full wp-image-441836&quot; src=&quot;https://securityintelligence.com/wp-content/uploads/2023/04/NWO-Loader.jpg&quot; alt=&quot;Chart depicting the two loader paths&quot; width=&quot;551&quot; height=&quot;577&quot; /&gt;&lt;/a&gt;

Finally, X-Force analysts also observed that two of the Minodo Backdoor C2 addresses belonging to MivoCloud: 94.158.247[.]72 and 185.225.17[.]202 are very close to C2 addresses ITG14 has used previously for SSH-based backdoors such as 94.158.247[.]23 and 185.225.17[.]220. While such overlap is insufficient for attribution, it is nonetheless consistent with other evidence linking Minodo to an ITG14 developer and may indicate the latter&#039;s preference for MivoCloud for at least some of their C2 addresses. MivoCloud has also been used to host Diceloader and Carbanak C2 servers.

Collaboration between members of ITG14 and former members of ITG23 is not without precedent. Other researchers — &lt;a href=&quot;https://www.mandiant.com/resources/blog/evolution-of-fin7&quot;&gt;here&lt;/a&gt; and &lt;a href=&quot;https://www.truesec.com/hub/blog/collaboration-between-fin7-and-the-ryuk-group-a-truesec-investigation&quot;&gt;here&lt;/a&gt; — observed ITG14 attacks using Ryuk ransomware in 2020, which has been attributed to the Trickbot/Conti cybercrime syndicate, while &lt;a href=&quot;https://www.sentinelone.com/labs/black-basta-ransomware-attacks-deploy-custom-edr-evasion-tools-tied-to-fin7-threat-actor/&quot; target=&quot;_blank&quot; rel=&quot;noopener&quot;&gt;others&lt;/a&gt; have also identified connections between a FIN7 developer and tools used by the BlackBasta group, whose members also have ties to the former Trickbot/Conti syndicate.
&lt;h3&gt;Project Nemesis&lt;/h3&gt;
The Project Nemesis infostealer has been active in the wild since December 2021, when it was offered for sale on several Dark Web forums. It can collect data from a range of web browsers, as well as applications including Steam, Telegram, Discord, cryptowallets, and VPN providers.

Minodo has been used to install Project Nemesis since at least October 2022 — prior to its use in late February 2023 by ex-Conti actors. This leads us to assess that the ITG14 members responsible for developing Minodo probably had a relationship with Project Nemesis and offered Minodo and the infostealer to the ex-Conti threat actors as a package. The ex-Conti members in turn likely used the Project Nemesis infostealer against lower value targets.

The use of infostealers by former Trickbot/Conti members and their distributors is not without precedent — other security researchers observed the Vidar infostealer delivered during &lt;a href=&quot;https://www.microsoft.com/en-us/security/blog/2022/11/17/dev-0569-finds-new-ways-to-deliver-royal-ransomware-various-payloads/&quot;&gt;DEV-0569&lt;/a&gt; Batloader campaigns that also led in some cases to Cobalt Strike and Royal ransomware. We assess that infostealers may often be deployed during lower priority infections, e.g., on personal computers or those not belonging to an Active Directory domain, while higher priority infections receive other backdoors such as Cobalt Strike.
&lt;h2&gt;Malware deep dive&lt;/h2&gt;
This section contains a deeper look at the Dave Loader, Minodo Backdoor, Minodo Loader, and Project Nemesis Infostealer malware.
&lt;h3&gt;Dave loader&lt;/h3&gt;
The sample analyzed for the purpose of this report is a 64-bit executable with MD5 hash &lt;strong&gt;2CC79806701F1A6E877C29B93F06F1BB&lt;/strong&gt; and a reported compile date of 28 February 2023. This sample is identified as a variant of Dave Loader, a crypter linked to threat group ITG23 and more commonly observed with payloads such as Emotet.

This sample has two encrypted resources within a resource directory named &quot;&lt;strong&gt;XKLKLCRTE&lt;/strong&gt;.&quot; Dave Loader loads the resources using the API calls &lt;strong&gt;LdrFindResource_U&lt;/strong&gt; and &lt;strong&gt;LdrAccessResource&lt;/strong&gt;, and decrypts them using XOR and the key &lt;strong&gt;mh8ZqMlTsaDYBZe7ma\x00&lt;/strong&gt;.

The smaller payload, with resource ID 3412, is a shellcode blob that contains code designed to load a PE file. The second payload, with resource ID 6732, is a PE file. Dave Loader executes the decrypted shellcode and passes the PE payload to it, which the shellcode then loads and executes.
&lt;h3&gt;Minodo backdoor&lt;/h3&gt;
The loaded payload is a 64-bit DLL written in Visual C++, with MD5 hash &lt;strong&gt;CDBE0FEB82B1CAF164C7DA42CB9A20BE&lt;/strong&gt;. The payload was found to be a hitherto unknown backdoor, which will be referred to as Minodo Backdoor.

Upon execution, Minodo Backdoor starts by generating a Bot ID for the infected system by gathering the username and hostname and creating a hash of the collected data, to which it then appends its current process id. For example, &lt;strong&gt;a648628c13d928dc-4480&lt;/strong&gt;. This bot ID format is similar to that generated by Lizar, which also used a checksum of system information, combined with the process ID. However, Lizar used a CRC algorithm to create its hash, whereas Minodo uses a custom algorithm which XOR&#039;s multiple values together. Interestingly, Lizar used a similar XOR-based algorithm as part of its encryption mechanisms during communication with the C2.

The malware first sends 4 bytes to the C2 that contain a XOR-encrypted value, which is the size of the data to follow, and then sends the AES-encrypted system data.

The malware then begins a loop of checking in with the C2 by first sending a 4-byte encrypted size, followed by a 16-byte AES-encrypted block, which decrypts to the following 5 bytes &lt;strong&gt;{01 00 00 00 ff}&lt;/strong&gt;. It continues this every second until it receives a response from the C2.

When it receives a response, the malware expects the C2 to send 4 bytes containing the payload size, followed by the payload itself. The malware then decrypts the received payload using AES and the shared key.

The decrypted data consists of a 4 byte size, followed by a single byte indicating either a command or the load method that the malware should use for the payload, followed by the payload data itself, if applicable.

If the command byte is &lt;strong&gt;0x3&lt;/strong&gt;, then the malware exits.

If the command byte is &lt;strong&gt;0x7&lt;/strong&gt;, then the malware enumerates the running processes on the system and compiles a list of process names and IDs. This data is then encrypted using AES and returned to the C2.

Otherwise, the malware proceeds to load and execute the received payload using the method indicated by the load/command byte. The following methods are supported:
&lt;ul type=&quot;disc&quot;&gt;
 	&lt;li&gt;&lt;strong&gt;0x1&lt;/strong&gt; — Copy the payload into allocated memory within the current process and create a new thread to execute an export named &lt;strong&gt;ReflectiveLoader&lt;/strong&gt; within the payload DLL.&lt;/li&gt;
 	&lt;li&gt;&lt;strong&gt;0x4&lt;/strong&gt; — Save the payload to the &lt;strong&gt;%Temp%&lt;/strong&gt; directory with a filename generated via &lt;strong&gt;GetTempFileNameA&lt;/strong&gt;, and execute the file using &lt;strong&gt;CreateProcessA&lt;/strong&gt;.&lt;/li&gt;
 	&lt;li&gt;&lt;strong&gt;0x5&lt;/strong&gt;, &lt;strong&gt;0x6&lt;/strong&gt; — Open process with process id provided with the downloaded data. Allocate space within the process and write the payload data to the process memory. Then create a new thread in the remote process to execute an export named &lt;strong&gt;ReflectiveLoader&lt;/strong&gt; within the copied payload DLL. This method is likely intended to be used following command &lt;strong&gt;0x7&lt;/strong&gt;.&lt;/li&gt;
&lt;/ul&gt;
In the case of the analyzed sample, the received payload contained a DLL binary, and command 0x1 was specified.

&lt;h3&gt;Overlap with Lizar toolkit&lt;/h3&gt;
Minodo Backdoor shares a lot of overlap with malware associated with the Lizar Toolkit, which is attributed to threat group ITG14. The Lizar Toolkit consists of a backdoor/loader, a collection of modules/plugins which can be executed by the loader, and C2 client and server applications. A detailed report on the Lizar Toolkit can be found &lt;a href=&quot;https://bi-zone.medium.com/from-pentest-to-apt-attack-cybercriminal-group-fin7-disguises-its-malware-as-an-ethical-hackers-c23c9a75e319&quot;&gt;here&lt;/a&gt;.

In addition to having similar coding styles and utilizing similar API calls, Minodo Backdoor uses the same configuration structure as the Lizar Loader. In both cases, a configuration block containing a pipe-delimited list of IP addresses is decrypted using XOR and a 16-byte key which is stored immediately before the encrypted config.

Both malware families also generate system IDs in very similar manners. In the case of Minodo Backdoor a hash is generated from the system username and hostname and the process ID is appended. For example, &lt;strong&gt;a648628c13d928dc-4480&lt;/strong&gt;. The hashing algorithm is custom and appears to be based on an encryption algorithm used previously in the Lizar Loader.  Lizar Loader also generates a system ID which consists of a hash of system data followed by the current process ID, however, it uses a CRC algorithm to create its hash.

Minodo Backdoor also incorporates elements that appear in some of Lizar&#039;s plugins. For example, the system information gathered by Minodo Backdoor and sent to the C2 matches the reported functionality of Lizar&#039;s Info32/Info64.dll plugin. Also, the function within Minodo Backdoor to enumerate running processes and send a list of process names and IDs back to the C2 is very similar to the functionality of Lizar&#039;s ListProcesses32/ListProcesses64.dll plugin.

Finally, Lizar&#039;s Jumper32/Jumper64.dll plugin resembles the code within Minodo Backdoor, which is able to load a payload using several different methods, with the required one specified using a command ID number.
&lt;h3&gt;Minodo loader&lt;/h3&gt;
The payload received by the Minodo backdoor from the C2 during this analysis was a 64-bit DLL, written in C++, with MD5 hash &lt;strong&gt;2373BE26018075847AEA51636B739F66&lt;/strong&gt; and an internal filename of &lt;strong&gt;MultiRunDll64.dll&lt;/strong&gt;. The payload was found to be a Loader with similarities to Minodo Backdoor, and will be referred to as Minodo Loader.

Minodo Loader has one export named &lt;strong&gt;ReflectiveLoader&lt;/strong&gt; that contains code taken from the well-known &lt;a href=&quot;https://github.com/stephenfewer/ReflectiveDLLInjection/blob/master/dll/src/ReflectiveLoader.c&quot;&gt;ReflectiveDLLInjection project&lt;/a&gt;. When run, this code performs the steps needed to properly load the DLL into memory, and allows for DLL payloads to load themselves directly from memory into a host process.

Once loaded, Minodo Loader starts by loading an encrypted payload from its resources and decrypting it using AES-256-CBC and a hardcoded key. The Microsoft WinCrypt library is used for AES encryption and decryption by both Minodo Backdoor and Loader.

The decrypted data consists of 4 bytes containing the payload size, followed by a further 4 bytes containing either the entry point offset for the payload or a value indicating the load method which should be used, followed by the payload itself.

If the entry point/load field contains a positive value, then the loader allocates memory within the current process space, copies the payload data to it, and then executes it at the specified offset indicated by the field value. This option is most likely used for shellcode payloads or further DLLs with ReflectiveLoader exports.

If the value of the load field is -1 then the loader allocates memory within the current process and then loads the PE payload into it using the full PE loading procedure. It copies the headers, maps the individual PE sections,  processes any relocations,  loads the PE&#039;s imports, and then executes the PE from its internally specified entry point.

Finally, if the load field is equal to zero or another negative number, the payload is loaded as a .NET assembly.

&lt;h2&gt;A tangled web provides opportunities&lt;/h2&gt;
This analysis highlights the intricate relationships between cyber criminal groups and their members. Minodo Backdoor shares code overlap with the Lizar (aka Tirion, Diceloader) family of malware, developed by ITG14. Since late February 2023, former members of ITG23 have been observed using Dave Loader with Minodo Backdoor during their campaigns, suggesting at least some level of collaboration between the two groups. The use of malware with ties to multiple groups in a single campaign — such as Dave Loader, Minodo Backdoor and Project Nemesis Infostealer — highlights the complexity involved in tracking threat actors but also provides insight into how and with whom they operate.

<html><body><hr>
<p>&nbsp;</p>
<p><strong>April 27, 2023 Update</strong></p>
<p class="xxmsonormal0"><span lang="EN-US" data-ogsc="black">This article is being republished with modifications from the original that was published on April 14, 2023, to change the name of the family of malware from Domino to Minodo. This is being done to avoid any possible confusion with the HCL Domino brand. The family of malware that is described in this article is unrelated to, does not impact, nor uses HCL Domino or any of its components in any way. The malware is not associated with HCL or its Domino product suite in any way.</span></p>
<hr>
<p><em>This blog was made possible through contributions from Christopher Caridi.&nbsp;</em></p>
<p>IBM Security X-Force recently discovered a new malware family we have called &ldquo;Minodo,&rdquo; which we assess was created by developers associated with the cybercriminal group that X-Force tracks as ITG14, also known as FIN7. Former members of the Trickbot/Conti syndicate which X-Force tracks as ITG23 have been using Minodo since at least late February 2023 to deliver either the Project Nemesis information stealer or more capable backdoors such as Cobalt Strike.</p>
<h2>Background</h2>
<p>This discovery highlights the intricate nature of cooperation among cybercriminal groups and their members:</p>
<ul type="disc">
<li>Since late February 2023, Minodo Backdoor campaigns have been observed using the <a href="https://securityintelligence.com/posts/itg23-crypters-cooperation-between-cybercriminal-groups/" >Dave Loader</a>, which we have linked to the Trickbot/Conti syndicate and its former members.</li>
<li>Minodo&rsquo;s code shows overlap with the Lizar (aka Tirion, Diceloader) malware family, leading us to suspect that it was created by current or former ITG14 developers.</li>
<li>One of Minodo&rsquo;s final payloads is the Project Nemesis infostealer. Project Nemesis was first advertised on the dark web in December 2021, though has been rarely used since then.</li>
</ul>
<h2>Analysis</h2>
<h3>Ex-Conti members deploy Minodo in recent campaigns</h3>
<p>Former members of ITG23 (aka the Trickbot/Conti syndicate) are likely behind recent campaigns using the Dave Loader to load Minodo Backdoor and probably collaborated with current or former ITG14 developers to purchase or use the new malware family. X-Force <a href="https://securityintelligence.com/posts/itg23-crypters-cooperation-between-cybercriminal-groups/" >previously assessed</a> that Dave is one of several loaders or crypters developed by members of the Trickbot/Conti group. Although the group has fractured, many of its loaders/crypters &mdash; including Dave &mdash; have been maintained and continue to be used by factions composed of former Trickbot/Conti members, including Quantum, Royal, BlackBasta, and Zeon.</p>
<ul type="disc">
<li>The Dave Loader has been used recently with several Cobalt Strike samples with the watermark &ldquo;206546002,&rdquo; which X-Force and other security researchers &mdash; <a href="https://www.trendmicro.com/en_us/research/22/i/play-ransomware-s-attack-playbook-unmasks-it-as-another-hive-aff.html" target="_blank" rel="noopener nofollow" >here</a> and <a href="https://medium.com/walmartglobaltech/from-royal-with-love-88fa05ff7f65" target="_blank" rel="noopener nofollow" >here</a> &mdash; have associated with groups composed of former members of the Trickbot/Conti syndicate, including Quantum and Royal. X-Force observed Dave-loaded Cobalt Strike samples using this watermark in suspected Royal attacks in fall 2022.</li>
<li>Dave Loader has also been used this year to load IcedID and Emotet, both of which serve as initial access vectors for ransomware attacks from former Trickbot/Conti-affiliated factions.</li>
</ul>
<p>Recently observed Dave samples were discovered loading a new malware, which we have named Minodo Backdoor. This new backdoor gathers basic system information, which it then sends to the C2, and in return receives an AES encrypted payload.</p>
<p>In most instances, the received payload was a second loader that was found to have code overlap with Minodo Backdoor, and as such we have dubbed it Minodo Loader. This loader contains an encrypted payload within its resources, which it decrypts using AES. The decrypted payload is a .NET infostealer, which identifies itself as &lsquo;Nemesis Project.&rsquo;</p>
<p>The Minodo Backdoor is designed to contact a different C2 address for domain-joined systems, suggesting a more capable backdoor, such as Cobalt Strike, will be downloaded on higher value targets instead of Project Nemesis.</p>
<p><amp-img src="https://securityintelligence.com/wp-content/uploads/2023/04/Minodo-Chain.jpg" layout="intrinsic" class="alignnone size-full wp-image-441835" alt="Chart depicting the Minodo backdoor attack chain." srcset="https://securityintelligence.com/wp-content/uploads/2023/04/Minodo-Chain.jpg 995w, https://securityintelligence.com/wp-content/uploads/2023/04/Minodo-Chain-150x91.jpg 150w" width="995" height="601" lightbox="lightbox"></amp-img></p>
<h3>Minodo&rsquo;s connections to ITG14</h3>
<p>Analysis revealed that both the Minodo Backdoor and Loader share code overlap with the <a href="https://bi-zone.medium.com/from-pentest-to-apt-attack-cybercriminal-group-fin7-disguises-its-malware-as-an-ethical-hackers-c23c9a75e319" target="_blank" rel="noopener nofollow" >Lizar</a> Malware, also known as Tirion and DiceLoader, which is attributed to the threat group ITG14 (FIN7). In addition to having similar coding styles and functionality, both Minodo and DiceLoader share the same configuration structure and have similar bot ID formats. Lizar was <a href="https://www.prodaft.com/blog/detail/opblueraven-unveiling-fin7carbanak-part-i-tirion" target="_blank" rel="noopener nofollow" >reportedly first used in March 2020</a>, when it was originally named Tirion, and has been observed in numerous ITG14 campaigns up to the end of 2022. Minodo has been active in the wild since at least October 2022, which notably is when Lizar observations began to decrease.</p>
<p>X-Force researchers found additional evidence connecting Minodo Backdoor to ITG14&rsquo;s Carbanak Backdoor. We identified Minodo Backdoor samples from December 2022, which used a different loader that we have dubbed NewWorldOrder Loader. The samples used the name <em>ThunderboltService.exe</em> and downloaded and loaded the Project Nemesis Stealer.</p>
<p>Around the same time, we also uncovered NewWorldOrder Loader samples, with the same filename <em>ThunderboltService.exe</em>, used to load the Carbanak Backdoor. <a href="https://www.mandiant.com/resources/blog/fin7-pursuing-an-enigmatic-and-evasive-global-criminal-operation" target="_blank" rel="noopener nofollow" >Carbanak</a> has been used by ITG14 since late 2015.</p>
<p><amp-img src="https://securityintelligence.com/wp-content/uploads/2023/04/NWO-Loader.jpg" layout="intrinsic" class="alignnone size-full wp-image-441836" alt="Chart depicting the two loader paths" srcset="https://securityintelligence.com/wp-content/uploads/2023/04/NWO-Loader.jpg 551w, https://securityintelligence.com/wp-content/uploads/2023/04/NWO-Loader-150x157.jpg 150w" width="551" height="577" lightbox="lightbox"></amp-img></p>
<p>Finally, X-Force analysts also observed that two of the Minodo Backdoor C2 addresses belonging to MivoCloud: 94.158.247[.]72 and 185.225.17[.]202 are very close to C2 addresses ITG14 has used previously for SSH-based backdoors such as 94.158.247[.]23 and 185.225.17[.]220. While such overlap is insufficient for attribution, it is nonetheless consistent with other evidence linking Minodo to an ITG14 developer and may indicate the latter&rsquo;s preference for MivoCloud for at least some of their C2 addresses. MivoCloud has also been used to host Diceloader and Carbanak C2 servers.</p>
<p>Collaboration between members of ITG14 and former members of ITG23 is not without precedent. Other researchers &mdash; <a href="https://www.mandiant.com/resources/blog/evolution-of-fin7" target="_blank" rel="noopener nofollow" >here</a> and <a href="https://www.truesec.com/hub/blog/collaboration-between-fin7-and-the-ryuk-group-a-truesec-investigation" target="_blank" rel="noopener nofollow" >here</a> &mdash; observed ITG14 attacks using Ryuk ransomware in 2020, which has been attributed to the Trickbot/Conti cybercrime syndicate, while <a href="https://www.sentinelone.com/labs/black-basta-ransomware-attacks-deploy-custom-edr-evasion-tools-tied-to-fin7-threat-actor/" target="_blank" rel="noopener nofollow"  target="_blank" rel="noopener nofollow" >others</a> have also identified connections between a FIN7 developer and tools used by the BlackBasta group, whose members also have ties to the former Trickbot/Conti syndicate.</p>
<h3>Project Nemesis</h3>
<p>The Project Nemesis infostealer has been active in the wild since December 2021, when it was offered for sale on several Dark Web forums. It can collect data from a range of web browsers, as well as applications including Steam, Telegram, Discord, cryptowallets, and VPN providers.</p>
<p>Minodo has been used to install Project Nemesis since at least October 2022 &mdash; prior to its use in late February 2023 by ex-Conti actors. This leads us to assess that the ITG14 members responsible for developing Minodo probably had a relationship with Project Nemesis and offered Minodo and the infostealer to the ex-Conti threat actors as a package. The ex-Conti members in turn likely used the Project Nemesis infostealer against lower value targets.</p>
<p>The use of infostealers by former Trickbot/Conti members and their distributors is not without precedent &mdash; other security researchers observed the Vidar infostealer delivered during <a href="https://www.microsoft.com/en-us/security/blog/2022/11/17/dev-0569-finds-new-ways-to-deliver-royal-ransomware-various-payloads/" target="_blank" rel="noopener nofollow" >DEV-0569</a> Batloader campaigns that also led in some cases to Cobalt Strike and Royal ransomware. We assess that infostealers may often be deployed during lower priority infections, e.g., on personal computers or those not belonging to an Active Directory domain, while higher priority infections receive other backdoors such as Cobalt Strike.</p>
<h2>Malware deep dive</h2>
<p>This section contains a deeper look at the Dave Loader, Minodo Backdoor, Minodo Loader, and Project Nemesis Infostealer malware.</p>
<h3>Dave loader</h3>
<p>The sample analyzed for the purpose of this report is a 64-bit executable with MD5 hash <strong>2CC79806701F1A6E877C29B93F06F1BB</strong> and a reported compile date of 28 February 2023. This sample is identified as a variant of Dave Loader, a crypter linked to threat group ITG23 and more commonly observed with payloads such as Emotet.</p>
<p>This sample has two encrypted resources within a resource directory named &ldquo;<strong>XKLKLCRTE</strong>.&rdquo; Dave Loader loads the resources using the API calls <strong>LdrFindResource_U</strong> and <strong>LdrAccessResource</strong>, and decrypts them using XOR and the key <strong>mh8ZqMlTsaDYBZe7ma\x00</strong>.</p>
<p>The smaller payload, with resource ID 3412, is a shellcode blob that contains code designed to load a PE file. The second payload, with resource ID 6732, is a PE file. Dave Loader executes the decrypted shellcode and passes the PE payload to it, which the shellcode then loads and executes.</p>
<h3>Minodo backdoor</h3>
<p>The loaded payload is a 64-bit DLL written in Visual C++, with MD5 hash <strong>CDBE0FEB82B1CAF164C7DA42CB9A20BE</strong>. The payload was found to be a hitherto unknown backdoor, which will be referred to as Minodo Backdoor.</p>
<p>Upon execution, Minodo Backdoor starts by generating a Bot ID for the infected system by gathering the username and hostname and creating a hash of the collected data, to which it then appends its current process id. For example, <strong>a648628c13d928dc-4480</strong>. This bot ID format is similar to that generated by Lizar, which also used a checksum of system information, combined with the process ID. However, Lizar used a CRC algorithm to create its hash, whereas Minodo uses a custom algorithm which XOR&rsquo;s multiple values together. Interestingly, Lizar used a similar XOR-based algorithm as part of its encryption mechanisms during communication with the C2.</p>

<p>The malware first sends 4 bytes to the C2 that contain a XOR-encrypted value, which is the size of the data to follow, and then sends the AES-encrypted system data.</p>
<p>The malware then begins a loop of checking in with the C2 by first sending a 4-byte encrypted size, followed by a 16-byte AES-encrypted block, which decrypts to the following 5 bytes <strong>{01 00 00 00 ff}</strong>. It continues this every second until it receives a response from the C2.</p>
<p>When it receives a response, the malware expects the C2 to send 4 bytes containing the payload size, followed by the payload itself. The malware then decrypts the received payload using AES and the shared key.</p>
<p>The decrypted data consists of a 4 byte size, followed by a single byte indicating either a command or the load method that the malware should use for the payload, followed by the payload data itself, if applicable.</p>
<p>If the command byte is&nbsp;<strong>0x3</strong>, then the malware exits.</p>
<p>If the command byte is&nbsp;<strong>0x7</strong>, then the malware enumerates the running processes on the system and compiles a list of process names and IDs. This data is then encrypted using AES and returned to the C2.</p>
<p>Otherwise, the malware proceeds to load and execute the received payload using the method indicated by the load/command byte. The following methods are supported:</p>
<ul type="disc">
<li><strong>0x1</strong> &mdash; Copy the payload into allocated memory within the current process and create a new thread to execute an export named <strong>ReflectiveLoader</strong>&nbsp;within the payload DLL.</li>
<li><strong>0x4</strong> &mdash; Save the payload to the <strong>%Temp%</strong> directory with a filename generated via <strong>GetTempFileNameA</strong>, and execute the file using <strong>CreateProcessA</strong>.</li>
<li><strong>0x5</strong>, <strong>0x6</strong> &mdash; Open process with process id provided with the downloaded data. Allocate space within the process and write the payload data to the process memory. Then create a new thread in the remote process to execute an export named <strong>ReflectiveLoader</strong>&nbsp;within the copied payload DLL. This method is likely intended to be used following command <strong>0x7</strong>.</li>
</ul>
<p>In the case of the analyzed sample, the received payload contained a DLL binary, and command 0x1 was specified.</p>

<h3>Overlap with Lizar toolkit</h3>
<p>Minodo Backdoor shares a lot of overlap with malware associated with the Lizar Toolkit, which is attributed to threat group ITG14. The Lizar Toolkit consists of a backdoor/loader, a collection of modules/plugins which can be executed by the loader, and C2 client and server applications. A detailed report on the Lizar Toolkit can be found <a href="https://bi-zone.medium.com/from-pentest-to-apt-attack-cybercriminal-group-fin7-disguises-its-malware-as-an-ethical-hackers-c23c9a75e319" target="_blank" rel="noopener nofollow" >here</a>.</p>
<p>In addition to having similar coding styles and utilizing similar API calls, Minodo Backdoor uses the same configuration structure as the Lizar Loader. In both cases, a configuration block containing a pipe-delimited list of IP addresses is decrypted using XOR and a 16-byte key which is stored immediately before the encrypted config.</p>
<p>Both malware families also generate system IDs in very similar manners. In the case of Minodo Backdoor a hash is generated from the system username and hostname and the process ID is appended. For example, <strong>a648628c13d928dc-4480</strong>. The hashing algorithm is custom and appears to be based on an encryption algorithm used previously in the Lizar Loader.&nbsp; Lizar Loader also generates a system ID which consists of a hash of system data followed by the current process ID, however, it uses a CRC algorithm to create its hash.</p>
<p>Minodo Backdoor also incorporates elements that appear in some of Lizar&rsquo;s plugins. For example, the system information gathered by Minodo Backdoor and sent to the C2 matches the reported functionality of Lizar&rsquo;s Info32/Info64.dll plugin. Also, the function within Minodo Backdoor to enumerate running processes and send a list of process names and IDs back to the C2 is very similar to the functionality of Lizar&rsquo;s ListProcesses32/ListProcesses64.dll plugin.</p>
<p>Finally, Lizar&rsquo;s Jumper32/Jumper64.dll plugin resembles the code within Minodo Backdoor, which is able to load a payload using several different methods, with the required one specified using a command ID number.</p>
<h3>Minodo loader</h3>
<p>The payload received by the Minodo backdoor from the C2 during this analysis was a 64-bit DLL, written in C++, with MD5 hash <strong>2373BE26018075847AEA51636B739F66</strong> and an internal filename of <strong>MultiRunDll64.dll</strong>. The payload was found to be a Loader with similarities to Minodo Backdoor, and will be referred to as Minodo Loader.</p>
<p>Minodo Loader has one export named <strong>ReflectiveLoader</strong> that contains code taken from the well-known <a href="https://github.com/stephenfewer/ReflectiveDLLInjection/blob/master/dll/src/ReflectiveLoader.c" target="_blank" rel="noopener nofollow" >ReflectiveDLLInjection project</a>. When run, this code performs the steps needed to properly load the DLL into memory, and allows for DLL payloads to load themselves directly from memory into a host process.</p>
<p>Once loaded, Minodo Loader starts by loading an encrypted payload from its resources and decrypting it using AES-256-CBC and a hardcoded key. The Microsoft WinCrypt library is used for AES encryption and decryption by both Minodo Backdoor and Loader.</p>
<p>The decrypted data consists of 4 bytes containing the payload size, followed by a further 4 bytes containing either the entry point offset for the payload or a value indicating the load method which should be used, followed by the payload itself.</p>
<p>If the entry point/load field contains a positive value, then the loader allocates memory within the current process space, copies the payload data to it, and then executes it at the specified offset indicated by the field value. This option is most likely used for shellcode payloads or further DLLs with ReflectiveLoader exports.</p>
<p>If the value of the load field is -1 then the loader allocates memory within the current process and then loads the PE payload into it using the full PE loading procedure. It copies the headers, maps the individual PE sections,&nbsp; processes any relocations,&nbsp; loads the PE&rsquo;s imports, and then executes the PE from its internally specified entry point.</p>
<p>Finally, if the load field is equal to zero or another negative number, the payload is loaded as a .NET assembly.</p>

<h2>A tangled web provides opportunities</h2>
<p>This analysis highlights the intricate relationships between cyber criminal groups and their members. Minodo Backdoor shares code overlap with the Lizar (aka Tirion, Diceloader) family of malware, developed by ITG14. Since late February 2023, former members of ITG23 have been observed using Dave Loader with Minodo Backdoor during their campaigns, suggesting at least some level of collaboration between the two groups. The use of malware with ties to multiple groups in a single campaign &mdash; such as Dave Loader, Minodo Backdoor and Project Nemesis Infostealer &mdash; highlights the complexity involved in tracking threat actors but also provides insight into how and with whom they operate.</p>
