      <h1 style="position:absolute;width:1px;height:1px;padding:0;margin:-1px;overflow:hidden;clip:rect(0,0,0,0);white-space:nowrap;border-width:0">AppLocker Rules as Defense Evasion: Complete Analysis</h1>
      <div>
        <p><em>Splunk is committed to using inclusive and unbiased language. This blog post might contain terminology that we no longer use. For more information on our updated terminology and our stance on biased language, please visit <a href="/en_us/blog/leadership/biased-language-has-no-place-in-tech-a-follow-up.html">our blog post</a>. We appreciate your understanding as we work towards making our community more inclusive for everyone.</em></p>
        <p>Microsoft continues to develop, update and improve features to monitor and prevent the execution of malicious code on the Windows opearting system. One of these features is <a href="https://docs.microsoft.com/en_us/windows/security/threat-protection/windows-defender-application-control/applocker/applocker-overview">AppLocker</a>. This feature advances the functionality of software restriction policies and enables administrators to create rules to allow or deny applications from running based on their unique identities (e.g., files) and to specify which users or groups can run those applications.</p>
        <p>AppLocker has the ability to control the execution of executables (“.exe” and “.com”), scripts (“.js”, “ps1”, “vbs”, “.cmd” and “.bat”), windows installer (“.msi, “.mst”, “.msp”), dll modules, packaged apps, and app installer.</p>
        <p>This software restriction policy may be abused by adversaries, like the “Azorult loader,” a payload that imports its own AppLocker policy to deny the execution of several antivirus components as part of its defense evasion.</p>
        <p>In this blog, the Splunk Threat Research Team will do a deep dive analysis on “Azorult loader” and its several components to understand tactics and techniques that may help SOC analysts and blue teamers defend against these types of threats.</p>

        <h2 id="azorult-loader">Azorult Loader</h2>
        <p>Azorult loader is a classic “Trojan Horse” that contains several components including the Azorult malware itself and additional embedded files to enable remote access and data collection. This loader is an <a href="https://en.wikipedia.org/wiki/AutoIt">autoit</a> compiled executable that contains a self-extracting stream in its resource sections along with several files.</p>
        <h3 id="defense-evasion">Defense Evasion</h3>
        <p>Azorult implements a hardcoded sandbox evasion checklist: It looks for specific usernames, files on the desktop, hostnames and processes running on the targeted host. If identified, it will exit. It will also terminate its execution if the OS version of the compromised host is “winxp”.</p>
