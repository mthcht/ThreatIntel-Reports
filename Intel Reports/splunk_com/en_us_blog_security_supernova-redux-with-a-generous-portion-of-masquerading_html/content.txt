      <h1 style="position:absolute;width:1px;height:1px;padding:0;margin:-1px;overflow:hidden;clip:rect(0,0,0,0);white-space:nowrap;border-width:0">SUPERNOVA Redux, with a Generous Portion of Masquerading</h1>
      <div>
        <p>C <em>ontributors: <a href="/en_us/blog/author/mbaccio.html">Mick Baccio</a>, Katie Brown, <a href="/en_us/blog/author/jbrodsky.html">James Brodsky</a>, <a href="/en_us/blog/author/andrewc.html">Drew Church</a>, <a href="/en_us/blog/author/dherrald.html">Dave Herrald</a>, <a href="/en_us/blog/author/rkovar.html">Ryan Kovar,</a> <a href="/en_us/blog/author/mlaferrera.html">Marcus LaFerrera</a>, Michael Natkin and <a href="/en_us/blog/author/jstoner.html">John Stoner</a></em></p>
        <p>If you want just to see how to find masquerading, skip down to the “<em>detections</em>” sections. Otherwise, read on for a quick breakdown of what happened, how to detect it, and MITRE ATT&#x26;CK mappings.</p>

        <h2 id="recent-supernova-attack-now-with-masquerading">Recent SUPERNOVA Attack, Now with Masquerading</h2>
        <p>The Cybersecurity and Infrastructure Security Agency (CISA) issued an <a href="https://us-cert.cisa.gov/ncas/analysis-reports/ar21-112a">analyst report</a> (AR21-112A) on April 22, 2021 that discussed a recent incident that they supported. As you read the first paragraph, it hits recent hot buttons: Pulse Secure and SolarWinds. Then you start wondering where is this going?</p>
        <div class="quote simple">
          <div>
            <div>“The threat actor connected to the entity’s network via a Pulse Secure virtual private network (VPN) appliance, moved laterally to its SolarWinds Orion server, installed malware referred to by security researchers as SUPERNOVA (a .NET webshell), and collected credentials.” — Analysis Report (AR21-112A)</div>
          </div>
        </div>
        <p>All of a sudden, we see SUPERNOVA and we breathe a sigh of relief; we’ve got this, in fact, we blogged about this back in January 2021 in <strong><a href="/en_us/blog/security/detecting-supernova-malware-solarwinds-continued.html">Detecting Supernova Malware Solarwinds Continued</a></strong>. But as we read farther, we come to find out that the adversary lovingly decided to take their copy of <a href="https://docs.microsoft.com/en_us/sysinternals/downloads/procdump">procdump.exe</a> — a command line tool that is used to create dumps of processes and has been used by various actors to dump credentials — renamed it Splunklogger.exe, and placed it on the compromised SolarWinds server.</p>
        <p><strong>Great.</strong></p>
        <p>This blog will highlight some new detections that were seen in this attack along with a discussion around masquerading. We will also provide some detections that you can take advantage of in your own environment. As mentioned before, we won’t go deep into SUPERNOVA itself as we already have covered that in a previous <a href="/en_us/blog/security/detecting-supernova-malware-solarwinds-continued.html">blog</a>, but these actions on objective are important to call out.</p>
        <h2 id="what-you-need-to-know">What You Need to Know</h2>
        <p>This specific attack has a few interesting traits. The first is that the adversary is using residential IP addresses based in the United States (US) to make them appear as US-based employees and then leveraging valid accounts to gain access via the VPN.</p>
        <p>From there, the adversary used a virtual machine and obfuscated PowerShell scripts to move laterally to the SolarWinds server. At this point, the SUPERNOVA webshell is installed. Due to logs being cleared during the attack, CISA was not able to determine if the adversary exploited <a href="https://nvd.nist.gov/vuln/detail/CVE-2020-10148">CVE-2020-10148</a>, an authentication bypass vulnerability of SolarWinds Orion or another method to gain access.</p>
        <div class="column float-right">
          <div>
            <div>
              <img src="about:error" alt="" loading="lazy">
            </div>
            <div>At this point, the adversary is collecting credentials as well as deploying tools to maintain persistence, evade defenses and other activities. A common tool that certain adversaries use is procdump.exe. Procdump.exe is a Microsoft command line utility that is used to monitor applications and can create crash dumps. Adversaries have been observed using procdump to dump credentials. To obfuscate the existence of procdump.exe on the SolarWinds server, the adversary renamed their copy of procdump.exe to splunklogger.exe. This masquerading technique is fairly common with certain utilities because the existence of that utility on certain systems may trigger alarms for organizations, whereas a tool like Splunk is used in many organizations and would raise less concern when seen.</div>
          </div>
        </div>
        <p>After credentials were dumped from LSASS memory, the adversary used the organization’s web server to exfiltrate the credentials and then deleted the web server logs in an effort to cover their tracks.</p>
        <p>Additional access occurred after the initial attack by leveraging credentials that were likely cracked offline from the initial credential dump. A final identified access event occurred where both procdump.exe and winrar.exe were seen masquerading as wininit.exe and the adversary made an attempt to archive credentials, probably before they were exfiltrated.</p>
        <h2 id="detecting-masquerading-as-well-as-indicators-of-the-supernova-attack-in-splunk">Detecting Masquerading As Well as Indicators of the SUPERNOVA Attack in Splunk</h2>
        <p>Here we will give you some hot-off-the-press searches to help find some of the badness derived from the <a href="https://us-cert.cisa.gov/ncas/analysis-reports/ar21-112a">CISA Analysis Report</a> on this recent SUPERNOVA attack. If we have coverage for these searches in Splunk security content, we call them out further below in the MITRE ATT&#x26;CK section.</p>
        <p>We covered some thoughts on detecting the SUPERNOVA webshell in our previous <a href="/en_us/blog/security/detecting-supernova-malware-solarwinds-continued.html">post</a> on the subject as well as the associated vulnerabilities, so today we will focus on the activities that took place after the webshell was established, specifically around masquerading and file integrity of the files manipulated.</p>
        <h4 id="indicators-of-compromise-iocs">Indicators of Compromise (IOCs)</h4>
        <p>CISA published IOCs, including file names, hashes and IPs, in their <a href="https://us-cert.cisa.gov/ncas/analysis-reports/ar21-112a">blog post</a>. So we collected the common hashes for procdump.exe, along with the IOCs that CISA identified and converted these indicators into simple CSV format so that you may use them as <a href="https://docs.splunk.com/Documentation/Splunk/8.1.2/Knowledge/Aboutlookupsandfieldactions">lookup tables</a> — they are posted <a href="https://github.com/mnatkin-splunk/supernova-splunk-csvs">here</a>. But what’s a lookup table, and how does it help with security detection in Splunk? <a href="/en_us/blog/security/lookup-before-you-go-go-hunting.html">Got you covered there, too</a>.</p>
        <h4 id="process-monitoring">Process Monitoring</h4>
        <p>We frequently are asked "why should we use Sysmon for process monitoring instead of native Windows capability via Event ID 4688?" This situation is a perfect example as to why the rich data gathered by Sysmon is so valuable to defenders and threat hunters: hashes.</p>
        <p>Microsoft's documentation for 4688 is available <a href="https://docs.microsoft.com/en_us/windows/security/threat-protection/auditing/event-4688">here</a>. If we look in the example Event XML, we will only see information such as the NewProcessId or NewProcessName. If we've turned on CommandLine tracking, we'll be provided that information as well.</p>
        <p>Comparatively, Sysmon <a href="https://www.ultimatewindowssecurity.com/securitylog/encyclopedia/event.aspx?eventid=90001">Event Code 1</a> has a number of other fields including multiple types of hashes, the Company, and even the parent process id to better contextualize the process that was created.</p>
        <p>If we compare files being executed based on the name of the original file and the process, we can use Sysmon data with a search like this to get a side by side comparison and use the match function with the eval command to get a comparative.</p>
        <pre><code>EventCode=1 OriginalFileName=* process_name=*
</code></pre>
        <pre><code>| eval OriginalFileName=upper(OriginalFileName) | eval process_name=upper(process_name)
</code></pre>
        <pre><code>| eval match=if(OriginalFileName=process_name,"Match","No Match")
</code></pre>
        <pre><code>| search match="No Match"
</code></pre>
        <pre><code>| table _time host OriginalFileName process_name match
</code></pre>
        <p>
          <img src="about:error" alt="" loading="lazy">
        </p>
        <p>As you can see above, we can see that psexec.c is running under the name of smb.exe. It is important to note that without additional filtering, the search above is a bit noisy. However, for our purposes, we could easily swap out the OriginalFileName value of * in this search to look for just procdump.exe or add the string “<em>splunklogger</em>” like the search below.</p>
        <p>
          <img src="about:error" alt="" loading="lazy">
        </p>
        <p>Another technique that we can use to identify masquerading is the use of file hashes. To do this, we can again use a favored capability of Splunk, that is the lookup!</p>
        <p>We mentioned them above, and just to make sure this blog is super long, let’s cover them in greater detail. A very effective way of determining whether or not a process executing on one of your production servers is legit is to use lookups sourced with “known good” metadata about processes normally found in your environment. Assuming you have tight control over upgrades that will change legitimate binaries (e.g. a change control process/board), any executing binaries on your production servers that are “unknown” when compared to “known good” can be flagged.</p>
        <p>Here’s one way of accomplishing that in Splunk. First, we leverage Sysmon’s Event Code 1, which provides hash values and a lot of other interesting process metadata, to harvest this data from a known-good “golden image” server in the environment, and we pipe it to a lookup called UFKnownLookup.csv. Note, there’s all sorts of interesting metadata in these events, like the company name, the version number of the file, a description, and so forth:</p>
        <pre><code>index=endpoint process_name=splunk* EventCode=1 host=frostbite
| eval knowngood=1
| stats values(process_name) as process_name values(Company) as vendor values(Description) as description values(FileVersion) as version values(knowngood) as known_good by SHA256,MD5
| outputlookup UFKnownGood.csv
</code></pre>
        <p>
          <img src="about:error" alt="" loading="lazy">
        </p>
        <p>Now, we can leverage it in subsequent searches. For example, let’s rip a page from the CISA SUPERNOVA report, take a copy of procdump.exe (which is a Sysinternals binary) and rename it “splunklogger.exe” and then put it on our desktop and run it. It will look like this in a Sysmon event.</p>
        <p>
          <img src="about:error" alt="" loading="lazy">
        </p>
        <p>Now, let’s run a search to find Splunk binaries that are “unknown” to us, leveraging the lookup:</p>
        <pre><code>index=endpoint process_name=splunk* EventCode=1 host=frostbite 
</code></pre>
        <pre><code>| lookup UFKnownGood.csv SHA256 OUTPUT known_good 
</code></pre>
        <pre><code>| eval known_good = case(known_good == 1, "1", 1=1, "0") 
</code></pre>
        <pre><code>| search known_good=0 
</code></pre>
        <pre><code>| stats values(process_name) as process_name values(Company) as vendor values(Description) as description values(FileVersion) as version values(known_good) as known_good by SHA256,MD5
</code></pre>
        <p>
          <img src="about:error" alt="" loading="lazy">
        </p>
        <p>And <strong>voila</strong>! We can see that a binary called “splunklogger.exe” executed, but it wasn’t in our approved lookup list, and oh, by the way, even though it is called “splunklogger.exe” it certainly isn’t a real Splunk binary, based on the Company and Description metadata.</p>
        <p>In production, generating the lookup itself against raw data is a reasonable thing to do on an occasional basis. But for matching voluminous Sysmon data in Splunk, a tstats search against an accelerated data model from the Common Information Model is more optimal, like this one against Endpoint.Processes:</p>
        <pre><code>| tstats summariesonly=t prestats=t count,values(Processes.process),values(host) as host from datamodel=Endpoint.Processes where (Processes.process="*splunk*.exe*" AND Processes.process!="*cmd.exe*") by Processes.process_hash 
</code></pre>
        <pre><code>| rename Processes.process_hash as hashes 
</code></pre>
        <pre><code>| stats values(*) by hashes 
</code></pre>
        <pre><code>| rex field=hashes "SHA256=(?&#x3C;SHA256>.*)," 
</code></pre>
        <pre><code>| lookup UFKnownGood.csv SHA256 
</code></pre>
        <pre><code>| eval known_good = case(known_good == 1, "1", 1=1, "0") 
</code></pre>
        <pre><code>| search known_good=0
</code></pre>
        <p>
          <img src="about:error" alt="" loading="lazy">
        </p>
        <h3 id="fee-fi-fo-fim">Fee Fi Fo FIM</h3>
        <p>We also figured file integrity monitoring (FIM) was a super appropriate topic to cover since, ya know, procdump.exe went all splunklogger.exe on us. The goal we’re trying to accomplish with FIM is simple: detect unauthorized changes made to files, directories, network devices, OS and more. This can be accomplished by establishing a “baseline” for a file state, and monitoring for changes made to that state. It’s a great way to quickly identify file discrepancies, modifications, and additions.</p>
        <p>Need a FIM solution now? There are multiple software solutions that are designated for file integrity monitoring like <a href="https://www.tripwire.com/">Tripwire</a> and <a href="https://www.qualys.com/apps/file-integrity-monitoring/">Qualys FIM</a>.</p>
        <p>More of an open source kind of person? Not a problem, there are several solutions available depending on your requirements. Some of the more popular open source FIM solutions include <a href="https://www.ossec.net/">OSSEC</a> and <a href="https://www.osquery.io/">osquery</a>.</p>
        <p>Got none of that? You can always use “native” file activity monitoring, from things like Sysmon’s FileCreate and FileDelete events, or <a href="https://www.ultimatewindowssecurity.com/securitylog/encyclopedia/event.aspx?eventID=4663">even Windows 4663 events</a>. All of the solutions we mention integrate well with Splunk, and many populate our Change data model for ease of use.</p>

        <h2 id="conclusion">Conclusion</h2>
        <p>This blog is a little bit of an outlier.
