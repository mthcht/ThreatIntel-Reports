      <h1 style="position:absolute;width:1px;height:1px;padding:0;margin:-1px;overflow:hidden;clip:rect(0,0,0,0);white-space:nowrap;border-width:0">You Bet Your Lsass: Hunting LSASS Access</h1>
      <div>
        <p>One of the most commonly used techniques is to dump credentials after gaining initial access. Adversaries will use one of many ways, but most commonly <a href="https://github.com/gentilkiwi/mimikatz/wiki">Mimikatz</a> is used. Whether it be with PowerShell Invoke-Mimikatz, Cobalt Strike’s Mimikatz implementation, or a custom version. All of these methods have a commonality: targeting LSASS. The Local Security Authority Subsystem Service (LSASS) is a process in Microsoft Windows operating systems that is responsible for enforcing the security policy on the system. It verifies users logging on to a Windows computer or server, handles password changes, and creates access tokens (per <a href="https://en.wikipedia.org/wiki/Local_Security_Authority_Subsystem_Service">Wikipedia</a>).</p>
        <p>With that, the Splunk Threat Research Team dug into how Mimikatz, and a few other tools found in Atomic Red Team, access credentials via LSASS memory, <a href="https://attack.mitre.org/techniques/T1003/001/">T1003.001</a>. Part of this process for the Splunk Threat Research Team is to continuously update older analytics to ensure we are providing up to date coverage on latest techniques and behaviors.</p>
        <p>To begin, we’ll look at our current analytics related to LSASS memory dumping. We will then simulate T1003.001, OS Credential Dumping: LSASS Memory, by using <a href="https://github.com/gentilkiwi/mimikatz">Mimikatz</a>, Cobalt Strike, Atomic Red Team <a href="https://github.com/redcanaryco/atomic-red-team/blob/master/atomics/T1003.001/T1003.001.md">T1003.001</a>, and <a href="https://raw.githubusercontent.com/PowerShellMafia/PowerSploit/master/Exfiltration/Invoke-Mimikatz.ps1">Invoke-Mimikatz</a>. Last, we will update our current analytics or create new ones.</p>
        <h2 id="current-content-review">Current Content Review</h2>
        <h3 id="access-lsass-memory-for-dump-creation"><a href="https://splunkresearch.com/endpoint/access_lsass_memory_for_dump_creation/">Access LSASS Memory for Dump Creation</a></h3>
        <p>Our <a href="https://splunkresearch.com/endpoint/access_lsass_memory_for_dump_creation/">first analytic</a> identifies the image load dbgcore.dll or dbghelp.dll and a TargetImage of lsass.exe. Dbgcore.dll or dbghelp.dll are two core Windows debug DLLs that have minidump functions which provide a way for applications to produce crashdump files that contain a useful subset of the entire process context.</p>
        <p>This analytic focuses on the CallTrace and identifies whether dbgcore.dll or dbghelp.dll are loaded to dump credentials.</p>
        <pre><code>`sysmon` EventCode=10 TargetImage=*lsass.exe CallTrace=*dbgcore.dll* OR CallTrace=*dbghelp.dll*
</code></pre>
        <pre><code>| stats count min(_time) as firstTime max(_time) as lastTime by Computer, TargetImage,TargetProcessId, SourceImage, SourceProcessId
| rename Computer as dest
| `security_content_ctime(firstTime)`
| `security_content_ctime(lastTime)`
</code></pre>
        <p>We found, as we’ll show in our simulation, that dbgcore.dll and dbghelp.dll are no longer utilized with the latest version of Mimikatz or Cobalt Strike. However, it still does capture the more basic utilities that access LSASS memory.</p>
        <h3 id="detect-credential-dumping-through-lsass-access"><a href="https://splunkresearch.com/endpoint/detect_credential_dumping_through_lsass_access/">Detect Credential Dumping through LSASS access</a></h3>
        <p>This analytic looks for GrantedAccess of 0x1010 or 0x1410 against lsass.exe. These are common access types and it's probably a good time to understand what they are and the common values.</p>
        <pre><code>`sysmon` EventCode=10 TargetImage=*lsass.exe (GrantedAccess=0x1010 OR GrantedAccess=0x1410)
| stats count min(_time) as firstTime max(_time) as lastTime by Computer, SourceImage, SourceProcessId, TargetImage, TargetProcessId, EventCode, GrantedAccess
| rename Computer as dest
| `security_content_ctime(firstTime)`
| `security_content_ctime(lastTime)`
</code></pre>
        <p>This query is limited to two GrantedAcess rights that are familiar to older versions of Mimikatz, but does not stand the test of time in capturing the latest rights request.</p>
        <p>GrantedAccess is the requested permissions by the SourceImage into the TargetImage. Microsoft details out the process specific access rights <a href="https://docs.microsoft.com/en_us/windows/win32/procthread/process-security-and-access-rights?redirectedfrom=MSDN">here</a>. You may notice some other common values related to process command-line spoofing (PROCESS_SUSPEND_RESUME).</p>

        <p>Now that we have a basic understanding of how these two current analytics work, let's capture data and begin to test them out further.</p>
        <h2 id="capture-data">Capture Data</h2>
        <p>To get started with capturing process access event data with Sysmon, we have provided a simple config that identifies TargetImage of lsass.exe.         <h2 id="continuous-improvement">Continuous Improvement</h2>
        <h3 id="access-lsass-memory-for-dump-creation-1"><a href="https://splunkresearch.com/endpoint/access_lsass_memory_for_dump_creation/">Access LSASS Memory for Dump Creation</a></h3>
        <p>For our first analytic that focuses on CallTrace image load dbgcore.dll or dbghelp.dll, we found that over time Mimikatz moved away from these two DLLs (dbgcore.dll or dbghelp.dll). The main DLL used by Mimikatz is now ntdll.dll. Ntdll.dll is a native Windows binary that provides similar native API paths to perform credential dumping. For example in the <a href="https://github.com/gentilkiwi/mimikatz/blob/master/mimikatz/modules/sekurlsa/kuhl_m_sekurlsa.c#L1395">sekurlsa</a> module there are many ntdll exported api’s, but what stands out is <a href="https://docs.microsoft.com/en_us/windows-hardware/drivers/ddi/wdm/nf-wdm-rtlcopymemory">RtlCopyMemory</a> which is used to execute the module related to credential dumping.</p>
        <p>After simulating the behavior we needed, we get some results</p>
        <pre><code>`sysmon` EventCode=10 TargetImage=*lsass.exe CallTrace=*dbgcore.dll* OR CallTrace=*dbghelp.dll*
</code></pre>

</code></pre>
        <pre><code>TargetProcessId, SourceImage, SourceProcessId

</code></pre>

        <p>With all this testing, our updated Sysmon query combines the two analytics we set out to enhance by focusing on specific GrantedAccess rights and CallTrace DLLs. Will this catch everything? Probably not, but it will at least catch the majority of tools used and allow us to filter out known good in environments and focus on the rare.</p>
        <pre><code>`sysmon` EventCode=10 TargetImage=*lsass.exe GrantedAccess IN ("0x01000", "0x1010", "0x1038", "0x40", "0x1400", "0x1fffff", "0x1410", "0x143a", "0x1438", "0x1000") CallTrace IN ("*dbgcore.dll*", "*dbghelp.dll*", "*ntdll.dll*")| stats count min(_time) as firstTime max(_time) as lastTime by Computer, TargetImage, GrantedAccess, SourceImage, SourceProcessId, SourceUser, TargetUser | rename Computer as dest | `security_content_ctime(firstTime)`|`security_content_ctime(lastTime)`
</code></pre>

        <h2 id="additional-observations">Additional Observations</h2>
        <p>During our simulations we identified behaviors that may assist teams in identifying suspicious SourceUser accessing LSASS. Typically, we will see source NT AUTHORITY\SYSTEM and TargetUser NT AUTHORITY\SYSTEM for normal system process behavior. However, seeing source ATTACKRANGE\administrator and Target NT AUTHORITY\SYSTEM is suspicious.</p>
