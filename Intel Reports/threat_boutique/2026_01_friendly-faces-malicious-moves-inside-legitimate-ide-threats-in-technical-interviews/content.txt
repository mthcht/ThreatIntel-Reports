                            <h1 class="title-post" itemprop="name">Friendly Faces, Malicious Moves: Inside Legitimate IDE Threats in Technical Interviews</h1>
                    <div class="date">
            <span itemprop="datePublished"><a href="https://threat.boutique/archive/2026-01" title="Show all posts made on this month">25 January 2026</a></span>
            - Posted in            <span itemprop="articleSection"><a href="https://threat.boutique/category/hardenings">Hardenings</a></span> by            <span itemprop="author"><a href="https://threat.boutique/author/b3vSclAuunBMsdt">Jok3r</a></span> -
            <span><a href="https://threat.boutique/2026/01/friendly-faces-malicious-moves-inside-legitimate-ide-threats-in-technical-interviews" rel="permalink">Permalink</a></span>
        </div>
                    <div class="featured-image">
                <a href="https://threat.boutique/2026/01/friendly-faces-malicious-moves-inside-legitimate-ide-threats-in-technical-interviews"><img src="https://threat.boutique/content/images/20260125161109-IT_FAke_IDE.jpg" alt="Friendly Faces, Malicious Moves: Inside Legitimate IDE Threats in Technical Interviews"/></a>
            </div>
                                        <div class="post-body post-1769344630" itemprop="articleBody">
            <p>Background:
While reading an article about how threat actors abuse legitimate VS Code functionality to run malicious code on a target machine when a project is opened by the victim, I concluded that it might be possible to find a universal method that would work across all IDEs without any issues.</p>

<p>The way of abusing:
To run any code smoothly on all operating systems, an attacker can incorporate Husky into their solution. Husky is designed to execute commands during Git hooks such as commit, branch checkout, and more. While the tool itself helps automate workflows, it can also be exploited as the core of an attack. 
From our experience, it’s possible to include such commands so that, for example, during a Git branch checkout, arbitrary code can be executed on a machine. This effect can be achieved if an attacker places a malicious scenario inside the .husky folder.
<img src="https://threat.boutique/content/images/20260125162311-TI.SEND.png" alt="enter image description here" /></p>

<p>As a result, as shown in the screenshot, we successfully collected the information and used a LOLBin to send the data to our exfiltration server.
<img src="https://threat.boutique/content/images/20260125163140-ti.png" alt="enter image description here" /></p>

<p>Protection:</p>

<ul>
<li>During such activities, always use sandboxed systems or disposable environments.</li>
<li>Never run an IDE with SYSTEM or ROOT privileges.</li>
<li>Always inspect the project for suspicious configurations, especially in hidden folders.</li>
<li>Whenever possible, use online platforms for these tasks.</li>
<li>Monitor DNS requests to unknown addresses, and check for the creation of malicious files using Sysmon Event ID 11.</li>
</ul>

<p>Conclusion:
Abusing automation tools like Husky highlights how even legitimate development workflows can be weaponized by attackers. Staying vigilant about your development environment, auditing project configurations, and following best security practices is essential to reducing the risk of compromise.</p>
