    <div class='cylance-blog-container'><h2 style="text-align: center;"><b>Inside the APT28 DLL Backdoor Blitz<br> </b></h2> <p>This blog post is a follow-up to <a href="https://threatvector.cylance.com/en_us/home/flirting-with-ida-and-apt28.html" target="_blank"><i>Flirting With IDA and APT28</i></a>, where we covered generating IDA Pro signatures to identify and eliminate benign library code. In doing so, we can focus our attention on the bespoke code responsible for defining the implant’s behavior.</p> <p>This time around we perform a deep dive into the same APT28 sample by analyzing its capabilities and providing insight into its features.</p> <h2><b>Overview</b></h2> <table border="0" cellspacing="0" cellpadding="0" width="608"> <tbody><tr><td width="60" valign="top"><p><img src="/content/dam/blogs-blackberry-com/images/blogs/2019/08/fig1-apt28-dll.png" align="left" style="padding: 15.0px;"></p> </td> <td width="549" valign="top"><b><br> Hash: </b>B40909AC0B70B7BD82465DFC7761A6B4E0DF55B894DD42290E3F72CB4280FA44<br> <b>File Type: </b>Windows x64 DLL<br> <b>PE Compilation Timestamp: </b>4<sup>th</sup> July 2018 14:38:54<br> &nbsp;</td> </tr></tbody></table> <p style="text-align: center;"><i>Figure 1: APT28 sample details</i></p> <p>Analysis reveals the implant is a multi-threaded DLL backdoor that gives the threat actor (TA) full access to, and control of, the target host. When commanded by C2, the implant can upload or download files, create processes, interact with the host via a command shell and connect to C2 according to a defined sleep/activity schedule.</p> <p>As covered in <a href="https://threatvector.cylance.com/en_us/home/flirting-with-ida-and-apt28.html" target="_blank">Part 1</a>, the implant is written in C++ and statically linked against OpenSSL and the Poco C++ framework. Since the file is packaged as a DLL, the intention would be to inject it into a long-running process that is granted Internet access (such as a NetSvc service group) or one having local firewall permissions. We do not believe this DLL is intended to operate as a module for a larger tool.</p> <h2><b>Entry Point</b></h2> <p>Malicious code entry occurs via the DllMain export. The implant’s first task is to load the legitimate Microsoft npmproxy.dll found in “c:\windows\system32”. With this loaded, the addresses of the five implant exports, “DllCanUnloadNow”, “DllGetClassObject”, “DllRegisterServer”, “DllUnregisterServer” and “GetProxyDllInfo” are set to the addresses of the same, benign exports found in the Microsoft DLL.</p> <p>The reason for this is unconfirmed, but most likely serves a defensive measure to evade end point protection that scans export addresses looking for code changes (when compared to a predetermined signature database), or against a database of known-malicious code. Perhaps by setting its own exports to those of the benign copy, the implant is attempting to remain undetected:</p> <p style="text-align: center;"><img src="/content/dam/blogs-blackberry-com/images/blogs/2019/08/fig2-apt28-dll.png"></p> <p style="text-align: center;"><i>Figure 2: Substitution of PE export addresses</i></p> <p>With export addresses replaced, a thread is launched to execute the main code path. Within this, a second thread is launched responsible for send and receive operations with C2. A global variable forms the basis of inter-thread synchronization for processing of C2 messages.</p> <h2><b>Mutex</b></h2> <p>The primary thread’s first task is the creation of a Globally Unique ID (GUID) mutex “1b8232f6-6806-4733-901d-62bf3ef33e6c”. The GUID string can be found as plain text within the binary, offering a potential (albeit trivially replaceable) IoC. As is customary, if mutex creation fails the sample terminates with no further action taken.</p> <h2><b>Machine Fingerprint</b></h2> <p>Following mutex creation, the implant generates a unique CRC-32 host fingerprint using the primary network interface’s ethernet (MAC) address, the host name, and Windows version string. The final CRC-32 result is then XOR’d with fixed constant 0x64113. The generated host fingerprint is used later to build the C2 beacon URL, and presumably identifies the implant’s malicious use on each infected machine.</p> <h2><b>Command-and-Control (C2)</b></h2> <p>The main communication protocol with the C2 server is RSA encrypted, Base64 encoded JSON exchanged over HTTPS (TLS, port 443), or as fallback, on plain HTTP using port 80. During our analysis the implant did not attempt to use any other application protocols.</p> <p>The user-agent string for HTTP requests is populated from either the return value of the “﻿ObtainUserAgentString” API call, or if that fails, a hard-coded alternate (see IoCs below).</p> <p>Support for egress proxy servers is included through a call to “﻿WinHttpGetIEProxyConfigForCurrentUser”. The result is parsed and supplied to the Poco framework’s “﻿HTTPClientSession::setProxyConfig” prior to C2 activity.</p> <p>A 1024-bit RSA key pair is embedded within the implant and used to encrypt and decrypt communication between host and C2. The private key decrypts inbound traffic while the public key encrypts outbound. No stream cipher(s) or session keys are negotiated.</p> <p>The initial HTTPS beacon to “malaytravelgroup[.]com” is a GET request consisting of a randomly selected URL path concatenation, together with a query string containing the parameter split count, the XOR constant 0x64113 (also used when generating the host CRC-32), and the computed CRC-32 fingerprint. The query string is padded with random data to prevent ad-hoc analysis and finally Base64 encoded:</p> <table border="1" cellspacing="0" cellpadding="0" width="601"> <tbody><tr><td width="53" valign="top" bgcolor="#000000" style="color: rgb(51,204,51);"><p><b>Field</b></p> </td> <td width="62" valign="top" bgcolor="#000000" style="color: rgb(51,204,51);"><p><b>Name</b></p> </td> <td width="54" valign="top" bgcolor="#000000" style="color: rgb(51,204,51);"><p><b>Bytes</b></p> </td> <td width="87" valign="top" bgcolor="#000000" style="color: rgb(51,204,51);"><p><b>Value</b></p> </td> <td width="344" valign="top" bgcolor="#000000" style="color: rgb(51,204,51);"><p><b>Comment</b></p> </td> </tr><tr><td width="53" valign="top"><p>1</p> </td> <td width="62" valign="top"><p>-</p> </td> <td width="54" valign="top"><p>1</p> </td> <td width="87" valign="top"><p>1-254</p> </td> <td width="344" valign="top"><p>Parameter split count as ASCII value.</p> </td> </tr><tr><td width="53" valign="top"><p>2</p> </td> <td width="62" valign="top"><p>“aid”</p> </td> <td width="54" valign="top"><p>4</p> </td> <td width="87" valign="top"><p>0x64113</p> </td> <td width="344" valign="top"><p>XOR constant used when generating host CRC-32. The first byte of the decoded result is then discarded to arrive at the final value:</p> <p>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;CYlp=Tj<b>9RNBBg==</b> &gt;&gt; b64_decode(<b>9RNBBg==) </b>&gt;&gt; F5 13 41 06<br> &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;JOM=uJ<b>fgAz9Zpw</b> &gt;&gt; b64_decode(<b>fgAz9Zpw) &nbsp;</b>&gt;&gt; 7E 00 33 F5 9A 70</p> <table border="0" cellspacing="0" cellpadding="0"> <tbody><tr><td width="601" valign="top"><p style="text-align: center;"><img src="/content/dam/blogs-blackberry-com/images/blogs/2019/08/fig3xx-apt28-dll.PNG"><br> <br> 0x00064113 &gt;&gt; XOR constant/implant ID<br> 0x709AF533&gt;&gt; Host CRC-32 fingerprint</p> </td> </tr></tbody></table> <p>Once the beacon is received and decoded, the next phase of interaction is to issue commands that drive the implant’s functions, such as the provision of an activity schedule/sleep interval limiting implant activity.</p> <h2><b>Domain Generation Algorithm (DGA)</b></h2> <p>The primary C2 domain of “malaytravelgroup[.]com” is used for the initial beacon. Like all sensitive strings, it is XOR encoded and revealed only when needed.</p> <p>The implant also contains a Linear Congruential Generator (LCG) based algorithm for generating what appear to be backup C2 domain names. Curiously the implementation is 100% deterministic owing to the use of a fixed seed value (0xC31) as shown in Figure 4. Every invocation of the DGA will produce the same five domains:<br> </p> <div style="max-width: 1100.0px; border: 1.0px solid; padding: 10.0px;"><p>schooltillhungryprocess[.]com<br> reasonwithusefulpolicy[.]com<br> streetunderrelevantpeople[.]com<br> experiencewithweakkid[.]com<br> systembeforeniceparent[.]com</p> </div> <p style="text-align: center;"><i>Figure 4: Encoded C2 servers</i></p> <p>Once the domains are generated, they are XOR encoded. The LCG implementation can be seen here, together with its constant seed value:</p> <p style="text-align: center;"><img src="/content/dam/blogs-blackberry-com/images/blogs/2019/08/fig5-apt28-dll.png"></p> <p style="text-align: center;"><i>Figure 5: DGA LCG using fixed seed value</i></p> <p>&nbsp;Domain names are generated as the concatenation of 4 string-table lookups from 3 encrypted string tables. Table 3 is referenced twice during generation. The Top-Level Domain (TLD) suffix is hardcoded to “.com”.</p> <table border="1" cellspacing="0" cellpadding="0"> <tbody><tr><td width="76" valign="top" bgcolor="#000000" style="color: rgb(51,204,51);"><p><b>Table</b></p> </td> <td width="189" valign="top" bgcolor="#000000" style="color: rgb(51,204,51);"><p><b>Number of strings</b></p> </td> </tr><tr><td width="76" valign="top"><p>1</p> </td> <td width="189" valign="top"><p>99</p> </td> </tr><tr><td width="76" valign="top"><p>2</p> </td> <td width="189" valign="top"><p>31</p> </td> </tr><tr><td width="76" valign="top"><p>3</p> </td> <td width="189" valign="top"><p>149</p> </td> </tr></tbody></table> <p style="text-align: center;"><i>Table 3: String table sizes for domain generation</i></p> <p>Assuming the continued adoption of a four-part domain name, the LCG would be capable of generating millions of unique permutations. The reason for limiting its output to the same five, irrespective of date, time, host identity, or other input variable is unclear. A copy of the decrypted string tables is provided in the Appendix.</p> <p>By manually decreasing the delay between check-in attempts, we accelerated the amount of C2 activity generated by the implant. Curiously, during a 24-hour observation window, no attempts were made to communicate with the generated domains. Different reachability scenarios were tested, ranging from malformed responses to unreachable IP addresses. At no point did the implant generate any DNS or HTTP traffic relating to the generated domains.</p> <p>It’s conceivable the backup domains are accessed via HTTP 302 redirects issued by the primary site, but this would seemingly defeat the purpose of having primary and backup domains to cater for failure, migration, or shutdown. It’s also possible the generated C2 domains are active only for certain implant commands such as file upload or download.</p> <p>Lacking any evidence for their direct use, we can speculate the generated domains may be operating as a honeypot. Access to them from any Internet addresses would immediately be considered “suspicious” by virtue of their hidden existence only within the implant code. The source of such requests could be directly attributable to threat researchers or analysts investigating the implant’s inner workings.</p> <h2><b>Functions</b></h2> <p>The implant lacks any modularity and is therefore limited to eight functions. Each function is invoked from C2 by supplying a value from which a known CRC-32 is calculated.
