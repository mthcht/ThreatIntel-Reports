    <div class='cylance-blog-container'><h2><b>Introduction</b></h2> <p>BlackBerry Cylance researchers have recently discovered a previously unnamed Python RAT we’re calling PyXie. PyXie has been observed in the wild since at least 2018 without much attention from the cybersecurity industry.</p> <p>PyXie has been deployed in an ongoing campaign that targets a wide range of industries. It has been seen in conjunction with Cobalt Strike beacons as well as a downloader that has similarities to the <a href="https://securityintelligence.com/shifu-masterful-new-banking-trojan-is-attacking-14-japanese-banks/" target="_blank"><i>Shifu</i></a> banking Trojan. Analysts have observed evidence&nbsp;of the threat actors attempting to deliver ransomware to the healthcare and education industries with PyXie.</p> <p>BlackBerry Cylance has conducted multiple incident response (IR) engagements in which PyXie was identified on hosts in the victim environment.</p> <p>Key highlights of the PyXie campaign include:</p> <ul> <li>Legitimate LogMeIn and Google binaries used to sideload payloads.</li> <li>A Trojanized Tetris app to load and execute Cobalt Strike stagers from internal network shares.</li> <li>Use of a downloader with similarities to <i>Shifu</i> named &quot;Cobalt Mode&quot;.</li> <li>Use of <a href="https://github.com/BloodHoundAD/SharpHound" target="_blank"><i>Sharphound</i></a> to collect active directory information from victims.</li> <li>A custom compiled Python interpreter that uses scrambled opcodes to hinder analysis.</li> <li>Use of a modified RC4 algorithm to encrypt payloads with a unique key per infected host.</li> </ul> <p>The main focus of this blog will be on PyXie RAT, but IOCs for other parts of the campaign will be listed in the Appendix.</p> <h2><b>Attack Overview</b></h2> <p style="text-align: 16:29</p> </td> </tr></tbody></table> <p style="text-align: center;"><i>Table 1: Malicious side-loaded DLLs</i></p> <p>Once loaded by the LogMeIn or Google binary, the malicious DLL will locate its corresponding encrypted payload. It does this by taking the full path it was loaded from and appending a <i>.dat</i> extension to it.</p> <p>For example, if the malicious DLL is named <i>LmiGuardianDLL.dll</i>, the payload filename would be <i>LmiGuardianDLL.dll.dat</i>.<br> </p> <p>The encrypted payload is read from disk and then <i>AES-128</i> decrypted in <i>CBC</i> mode by the loader. The 16-byte initialization vector (IV) and symmetric key are hardcoded into the DLL and can vary from sample to sample. An example of a loader using hardcoded IV and Key is shown in Figure 2:<br> </p> <p style="text-align: center;"><img src="/content/dam/blogs-blackberry-com/images/blogs/2019/12/fig2-pyxie-rat.jpg"></p> <p style="text-align: right;"><i>Figure 2: Payload hardcoded decryption details</i></p> <p>The result of the decryption is the second stage payload, which is mapped into the loader process address space and executed.</p> <h2>Second Stage – Installation and Persistence<b></b></h2> <p>The second stage malware is primarily responsible for installing itself, setting up persistence, and spawning a new process to inject the third stage payload. The second stage component fingerprints the victim machine by generating a Hardware ID hash. This hash is later used as a seed for various functions as well as an encryption key in the subsequent stages.</p> <h3><b>Hardware ID</b></h3> <p>The Hardware ID is generated by calculating the MD5 hash of an 80-byte buffer containing information collected from the system, including:</p> <ul> <li>A 64-byte zero-padded Process Brand string obtained through a sequence of calls to the <i>CPUID</i> instruction (with EAX set to <i>0x80000002</i>, <i>0x80000003</i> and <i>0x80000004</i> values).</li> <li>The output values of <i>dwNumberOfProcessors</i> and <i>dwProcessorType</i> returned by a call to <i>GetSystemInfo.</i></li> <li>The volume serial number of the Operating System logical drive obtained through a call to <i>GetVolumeInformationA.</i></li> <li>The CRC32 hash of the current user obtained through a call to <i>GetUserNameA</i> using the ZLIB-CRC32 implementation with a seed of -1 (0xFFFFFFFF).<br> &nbsp;</li> </ul> <p style="text-align: center;"><img src="/content/dam/blogs-blackberry-com/images/blogs/2019/12/fig3a-pyxie-rat.jpg"><a name="_Ref23950296" id="_Ref23950296"></a></p> <p style="text-align: right;"><img src="/content/dam/blogs-blackberry-com/images/blogs/2019/12/fig3b-pyxie-rat.JPG"></p> <p style="text-align: right;"><i>Figure 3: Buffer used to compute the Hardware ID</i></p> <p>Given the buffer in Figure 3, the Hardware ID would be:&nbsp;<br> <i>66aafbed7c63b1a1d02899969d97e06f.</i></p> <h3><b>Mutexes</b></h3> <p>Two mutexes are created to prevent multiple payload instances from running at the same time.&nbsp;</p> <p>The first mutex name is generated using the following algorithm:</p> <ul> <li>The process infected with the second stage payload is obtained by calling the <i>GetModuleFileNameA</i> API, and the CRC32 of the corresponding binary is computed.&nbsp;</li> <li>The computed hash is XORed with the CRC32 of the currently logged-on user.</li> <li>The hex string representation of the last hash represents the mutex name.&nbsp;</li> </ul> <p>The second mutex is the last 28 characters of the hex representation of Hardware ID MD5 hash:</p> <p style="text-align: center;"><img src="/content/dam/blogs-blackberry-com/images/blogs/2019/12/fig4-pyxie-rat.jpg"></p> <p style="text-align: right;"><i>Figure 4: Mutex generated from the Hardware ID</i></p> <h3><b>Privilege Escalation</b></h3> <p>If the process infected with the second stage payload is running with administrator privileges, the malware will attempt to escalate its own privileges. It does so by creating and starting a temporary service, thus respawning and running as a LOCAL SYSTEM process. To remain stealthy, the malware deletes the temporary service from the Service Control Manager.</p> <h3><b>Installation</b></h3> <p>The loader and its corresponding payload are copied to a subdirectory within the <b>%APPDATA%</b> folder. The directory is chosen from the list of strings in Figure 5. The registry value is set to point to the path of the loader executable.</p> <h3><b>Injection of The Next Stage Payload</b></h3> <p>The third stage payload is decompressed with a call to <i>RtlDecompressBuffer</i> API and subsequently injected into a newly spawned process.</p> <p>The executable targeted for injection is chosen by enumerating over the executables in the <b>%SYSTEMROOT%</b>\System32 directory and searching for the first one that meets the following criteria:</p> <ul> <li>It is not running</li> <li>It has a GUI subsystem</li> <li>It has “Microsoft” in the version information</li> <li>It is signed</li> </ul> <p>The new process is spawned by using the <i>CreateProcessA</i> API and passing the path to the executable that loaded the seconded stage as the <i>lpCommandLine</i> parameter.<br> </p> <h2>Third Stage - “Cobalt Mode” Downloader<b></b></h2> <p>The third stage is a downloader that has been designated <i>Cobalt Mode </i>based on the debug information left in some of the analyzed samples. An example of the PDB path is shown in Figure 6:<br> </p> <table border="1" cellspacing="0" cellpadding="0"> <tbody><tr><td width="623" valign="top"><p style="text-align: center;"><b><i><br> Z:\coding\pyproject\compiled\cobalt_mode\cobalt_mode.pdb</i></b></p> </td> </tr></tbody></table> <p style="text-align: center;"><i>Figure 6: Third stage debug information</i></p> <p>The primary function of Cobalt Mode is:</p> <ul> <li>Connecting to a command and control (C&amp;C) server</li> <li>Downloading an encrypted payload</li> <li>Decrypting the payload</li> <li>Mapping and executing the payload in the address space of the current process</li> <li>Spawning a new process for code injection</li> </ul> <p><i>Cobalt Mode</i> malware can carry out a series of environmental checks. <p style="text-align: center;"><i>Table 3: RAT bytecode files</i></p> <h3><b>Analysis of .pyx files</b></h3> <p>At first glance, the <i>.pyx</i> files appear to be renamed <i>.pyc</i> files. However, after attempting to decompile them with <a href="https://github.com/wibiti/uncompyle2" target="_blank">uncompyle2</a> researchers recognized that this was not the case:</p> <p style="text-align: center;"><img src="/content/dam/blogs-blackberry-com/images/blogs/2019/12/fig9-pyxie-rat.jpg"></p> <p style="text-align: right;"><i>Figure 9: Uncompyle2 error</i></p> <p>Upon closer examination, we discovered these <i>.pyx</i> files have had the first 8 bytes stripped. These bytes typically would contain a 4-byte magic number, which denotes the version of the Python that generated them along with a 4-byte timestamp. The removal of these bytes is a well-known copy-protection trick aimed at preventing some tools from decompiling the files and getting access to the code.<br> <br> The workaround for this is simple. All that is needed is to add the appropriate magic number and a timestamp. A quick look at the strings from the malicious interpreter indicates that it may be based on Python 2.7.15:<br> </p> <table border="1" cellspacing="0" cellpadding="0"> <tbody><tr><td width="623" valign="top"><p style="text-align: left;"><br> &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; z:\coding\pyproject\python_static_2.7.15\pyuv-1.x\src\common.c<br> &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; z:\coding\pyproject\python_static_2.7.15\pyuv-1.x\deps\libuv\src\win\handle-inl.h<br> &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; z:\coding\pyproject\python_static_2.7.15\pyuv-1.x\deps\libuv\src\win\req-inl.h</p> </td> </tr></tbody></table> <p style="text-align: center;"><i><br> Figure 10: Strings identifying the Python version</i></p> <p>Once the appropriate header for Python 2.7.15 was added along with an arbitrary timestamp, researchers attempted to decompile the <i>.pyx</i> bytecode files again. This worked better than our first attempt, but something still wasn't right. Disassembly still failed when using a clean install of Python 2.7.15:<br> </p> <p style="text-align: center;"><img src="/content/dam/blogs-blackberry-com/images/blogs/2019/12/fig11-pyxie-rat.png"></p> <p style="text-align: right;"><i>Figure 11: Disassembler error</i></p> <p>Our exercise led researchers to believe that the interpreter may have remapped opcodes. This would prevent other interpreters from correctly parsing the bytecode generated by it.</p> <p>Remapping opcodes is not new. It is a technique often used by commercial applications such as Dropbox. The workaround for this is well known and documented in <a href="https://media.blackhat.com/bh-us-10/whitepapers/Smith/BlackHat-USA-2010-Smith-pyREtic-Reversing-wp.pdf" target="_blank">'</a><i><a href="https://media.blackhat.com/bh-us-10/whitepapers/Smith/BlackHat-USA-2010-Smith-pyREtic-Reversing-wp.pdf" target="_blank">pyREticʼ – In memory reverse engineering for obfuscated Python bytecode</a> </i>and<i> <a href="https://www.usenix.org/system/files/conference/woot13/woot13-kholia.pdf" target="_blank">Looking inside the (Drop) box</a></i>. The basic premise covers compiling the same Python script in both a standard and modified interpreter. The generated bytecode can then be compared to derive the remapped opcodes. This is typically straightforward if arbitrary code can be executed in the modified interpreter.<br> </p> <p>Unfortunately, PyXie is not a friendly Python interpreter and does not provide a mechanism to trivially run arbitrary code. Our solution was to analyze the modified interpreter in IDA Pro and compare it to the Python 2.7.15 source code. By using strings for reference, it was possible to locate the native <i>PyRun_SimpleStringFlags()</i> function. This function can be used to run Python code from an in-memory string.</p> <p>Once the location of <i>PyRun_SimpleStringFlags()</i> was found, a debugger was used to attach to the process running the PyXie interpreter.<br> </p> <p>Prior to redirecting the execution to a code cave, the script we wanted compiled is dropped in the working directory. For this controlled execution we used <a href="https://github.com/kholia/dedrop/blob/master/src/dedrop/all.py" target="_blank">all.py</a> from <a href="https://github.com/kholia/dedrop" target="_blank">DeDrop</a>. Then:</p> <ul> <li>Write the “import all.py” string into the memory.</li> <li>Push the address of the “import all.py” string to the stack.</li> <li>Call <i>PyRun_SimpleStringFlags() </i>which caused the interpreter to execute<i> all.py </i>and in turn emit compiled bytecode to<i> all.pyx.<br>  &nbsp;</i></li> </ul> <p style="text-align: center;"><img src="/content/dam/blogs-blackberry-com/images/blogs/2019/12/fig12-pyxie-rat.jpg"></p> <p style="text-align: right;"><i>Figure 12: Code cave importing all.py</i></p> <p>The <i>all.py</i> file was then executed within the standard Python 2.7.15 interpreter to output compiled bytecode as <i>all.pyc</i>.
