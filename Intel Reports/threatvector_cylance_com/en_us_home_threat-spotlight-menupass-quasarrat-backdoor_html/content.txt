    <div class="cylance-blog-container"><h2><b>Introduction</b></h2>
<p>During the latter half of 2018, BlackBerry Cylance threat researchers tracked a campaign targeting companies from several verticals across the EMEA region. The campaign seemed to be related to the MenuPass (a.k.a. APT10/Stone Panda/Red Apollo) threat actor, and utilized an open-source backdoor named QuasarRAT to achieve persistence within an organization. We identified several distinct loader variants tailored to specific targets by leveraging machine learning (ML) to analyse our malware corpus. We have not observed new QuasarRAT samples in the wild since late 2018, roughly coinciding with when the FBI indicted several members of the MenuPass group.</p>
<p>QuasarRAT is a lightweight remote administration tool written in C#. It can collect system information, download and execute applications, upload files, log keystrokes, grab screenshots/camera captures, retrieve system passwords and run shell commands. The <a title="Remote Access Trojan" href="https://www.blackberry.com/us/en/solutions/endpoint-security/ransomware-protection/remote-access-trojan">remote access Trojan (RAT)</a> is loaded by a bespoke loader (a.k.a. DILLWEED). The encrypted QuasarRAT payload is stored in the Microsoft.NET directory, decrypted into memory, and instantiated using a CLR host application. In later variants an additional component is also used to install the RAT as a service (a.k.a DILLJUICE).</p>
<p>The following technical analysis focuses on the bespoke QuasarRAT loader developed by MenuPass and modifications made to the QuasarRAT backdoor.</p>
<h2><b>Introducing the QuasarRAT Loader<br>
</b></h2>
<h3><b>Overview</b></h3>
<p>The QuasarRAT loader typically arrives as a 64-bit service DLL. Its primary purpose is to decrypt, load and invoke an embedded .NET assembly in-memory using the CppHostCLR technique. This technique is based on code snippets from Microsoft DevCentre examples. The assembly, obfuscated with ConfuserEx, is subsequently responsible for finding, decrypting, and executing a separate malicious .NET module. The encrypted module is stored in the %WINDOWS%\Microsoft.NET directory.&nbsp;</p>
<p>During our investigation we encountered several variants of the loader which indicated a development path lasting over a year; we were also able to locate some (but not all) of the encrypted payload files belonging to these loader variants. After decryption, we discovered that the payloads are backdoors based on <a href="https://github.com/quasar/QuasarRAT" target="_blank">the open-source code of QuasarRAT</a><sup>[1]</sup>, version 2.0.0.0 and 1.3.0.0.<br>
</p>
<h3><b>Features</b></h3>
<ul>
<li>Several layers of obfuscation</li>
<li>Payload and its immediate loader are .NET assemblies</li>
<li>Initial loader uses the <a href="https://code.msdn.microsoft.com/windowsdesktop/CppHostCLR-e6581ee0" target="_blank">CppHostCLR</a><sup>[2]</sup> technique to inject and execute the .NET loader assembly</li>
<li>Payload encrypted and stored under Microsoft.NET directory</li>
<li>Known to load QuasarRAT, but may work with any other .NET payload<br>
</li>
</ul>
<h3><b>Initial Loader and AntiLib</b></h3>
<p>The initial loader binary is a 64-bit PE DLL, intended to run as a service. The DllMain function is empty, while the malicious code is contained in the ServiceMain export. Some variants include an additional randomly named export that creates the malicious service. In newer versions this functionality was shifted to a standalone module.</p>
<p>The malware starts by deobfuscating an embedded next-stage executable. In the earliest variant, this is performed using simple XOR with a hardcoded 8-byte key composed of random letters. Later variants use a slightly more advanced XOR based algorithm that requires two single-byte keys. It’s possible that this approach was implemented to thwart XOR bruteforcing attempts:</p>
<p style="text-align: right;"><img src="/content/dam/blogs-blackberry-com/images/blogs/2019/06/fig1-quasarrat.png"><br>
<i>Figure 1: Second stage decryption loop</i></p>
<p>Starting with variant 3, the .NET injection mechanism is implemented inside a second stage DLL, which according to debugging strings seems to be part of a project called “AntiLib”:</p>
<p style="text-align: right;"><img src="/content/dam/blogs-blackberry-com/images/blogs/2019/06/fig2-quasarrat.png"><br>
<i>Figure 2: Debugging strings from variant 3</i></p>
<p>This DLL is reflectively loaded into memory by an obfuscated shellcode-like routine and invoked by executing an export bearing the unambiguous name: “FuckYouAnti”. Older samples do not contain this second stage library, and the .NET loading functionality is implemented directly in the initial loader:<br>
</p>
<p style="text-align: right;"><img src="/content/dam/blogs-blackberry-com/images/blogs/2019/06/fig3-quasarrat.png"><br>
<i>Figure 3: FuckYouAnti string in the code and in 2<sup>nd</sup> stage DLL export table</i></p>
<p>Once executed, the &quot;FuckYouAnti&quot; function will decrypt the .NET loader binary using the same XOR based algorithm with a different pair of hardcoded keys.<br>
</p>
<p>To load the assembly directly into memory, the malware makes use of a technique called &quot;CppHostCLR&quot; which is described in detail in Microsoft DevCentre. The code looks like the example code provided by Microsoft. It invokes the loader entry point using hardcoded class and method names, that are random and differ for each sample:<br>
</p>
<p style="text-align: right;"><img src="/content/dam/blogs-blackberry-com/images/blogs/2019/06/fig4-quasarrat.png"><br>
<i>Figure 4: Use of CppHostCLR technique</i></p>
<p style="text-align: right;"><img src="/content/dam/blogs-blackberry-com/images/blogs/2019/06/fig5-quasarrat.png"><br>
<i>Figure 5: Invoking .NET assembly loader</i></p>
<h2><b>String Encryption</b></h2>
<p>Hardcoded .NET version strings and several persistence related strings (in earlier variants) are encrypted using a custom algorithm. This algorithm is based on a single unit T-box implementation of AES-256, combined with 16-byte XOR. Both keys are hardcoded and differ for each sample, except for the oldest variant. The oldest variant set keys to “1234567890ABCDEF1234567890ABCDEF” and “1234567890ABCDEF” respectively and did not change between samples:<br>
</p>
<p style="text-align: right;"><img src="/content/dam/blogs-blackberry-com/images/blogs/2019/06/fig6-quasarrat.png"><br>
<i>Figure 6: Example AES and XOR decryption keys</i></p>
<p style="text-align: right;"><img src="/content/dam/blogs-blackberry-com/images/blogs/2019/06/fig7-quasarrat.png"><br>
<i>Figure 7: String decryption routine</i></p>
<h2><b>Digital Certificates</b></h2>
<p>Samples belonging to variant 3 of the loader present a valid digital signature from CONVENTION DIGITAL LTD (serial number 52 25 B8 E2 2D 3B BC 97 3F DD 24 2F 2C 2E 70 0C) countersigned by Symantec:</p>
<p style="text-align: right;"><img src="/content/dam/blogs-blackberry-com/images/blogs/2019/06/fig8-quasarrat.png"><i><br>
Figure 8: Digital certificate from variant 3</i></p>
<h2><b>The .NET loader</b></h2>
<p>Once executed, the malicious assembly will iterate through all files under %WINDOWS%\Microsoft.NET and attempt to decrypt files matching a specified size. It uses an implementation of RijndaelManaged algorithm in CBC mode:<br>
</p>
<p style="text-align: right;"><img src="/content/dam/blogs-blackberry-com/images/blogs/2019/06/fig9-quasarrat.png"><br>
<i>Figure 9: Finding encrypted payload</i></p>
<p style="text-align: right;"><img src="/content/dam/blogs-blackberry-com/images/blogs/2019/06/fig10-quasarrat.png"><br>
<i>Figure 10: Final payload decryption</i></p>
<p>If the decryption succeeds, the malware will attempt to load the decrypted assembly and invoke the specified method:</p>
<p style="text-align: right;"><img src="/content/dam/blogs-blackberry-com/images/blogs/2019/06/fig11-quasarrat.png"><br>
<i>Figure 11. Invoking backdoor payload</i></p>
<p>The final payload assembly is stored as an encrypted file somewhere under the Microsoft.NET Framework directory. The framework version is hardcoded in the loader binary in an encrypted form, and in most samples set to “v4.0.30319”<i>. </i>The location is different per sample and the file name imitates one of other the legitimate files found in the same directory. Example paths:</p>
<ul>
<li><i>%WINDOWS%\Microsoft.NET\Framework\v4.0.30319\WPF\Fonts\GlobalSerif.CompositeFont.rsp</i></li>
<li><i>%WINDOWS%\Microsoft.NET \Framework\v4.0.30319\Microsoft.Build.Engine.dll.uninstall</i></li>
</ul>
<p>The payload is decrypted and loaded in-memory as &quot;Client&quot;. We have encountered two versions of the Client: 2.0.0.0 and 1.3.0.0. They are similar, both having a version string in their configuration section set to “2.0.0.0”:<br>
</p>
<p style="text-align: right;"><img src="/content/dam/blogs-blackberry-com/images/blogs/2019/06/fig12-quasarrat.png"><br>
<i>Figure 12. Backdoor assembly in memory (version 2.0.0.0)</i></p>
<p style="text-align: right;"><img src="/content/dam/blogs-blackberry-com/images/blogs/2019/06/fig13-quasarrat.png"><br>
<i>Figure 13. Backdoor assembly in memory (version 1.3.0.0)</i></p>
<h2><b>QuasarRAT Backdoor</b></h2>
<p>QuasarRAT is an open-source project that proclaims to be designed for legitimate system administration and employee monitoring. Its code, together with documentation, can be found on GitHub.<b><br>
</b></p>
<h3><b>Features:</b></h3>
<p style="text-align: center;"><img src="/content/dam/blogs-blackberry-com/images/blogs/2019/06/fig14-quasarrat.png"></p>
<p style="text-align: right;"><i>Figure 14. README.md from Quasar GitHub repository</i></p>
<h2><b>Behaviour</b></h2>
<p>The .NET payload is a heavily obfuscated backdoor based on an open-source remote administration tool called <a href="https://github.com/quasar/QuasarRAT" target="_blank">QuasarRAT</a><sup>[3].</sup> The configuration is stored in a class called Settings, with sensitive string values encrypted with AES-128 in CBF mode and base64 encoded. The string’s decryption key is derived from the ENCRYPTIONKEY value inside Settings and is the same for all strings:</p>
<p style="text-align: right;"><img src="/content/dam/blogs-blackberry-com/images/blogs/2019/06/fig15-quasarrat.png"><br>
<i>Figure 15. Partially encrypted config (after deobfuscation)</i></p>
<p>The threat actor modified the original backdoor, adding their own field in the configuration, and code for checking the Internet connectivity. If a valid URL address is specified in the last value of config, the malware will try to download the content of that URL. It will proceed with connecting to the command and control (C2) server only once the download is successful:</p>
<p style="text-align: right;"><img src="/content/dam/blogs-blackberry-com/images/blogs/2019/06/fig16-quasarrat.png"><br>
<i>Figure 16: Custom connectivity check</i></p>
<p>The backdoor communicates with the C2 server whose IP address is provided in the HOSTS value of the configuration. All communication is encrypted with AES-128 in CBF mode using KEY and AUTHKEY values from configuration:</p>
<p style="text-align: right;"><img src="/content/dam/blogs-blackberry-com/images/blogs/2019/06/fig17-quasarrat.png"><br>
<i>Figure 17. C2 IP address decrypted in memory</i></p>
<p>Decrypted configuration examples:<br>
</p>
<p style="text-align: center;"><img src="/content/dam/blogs-blackberry-com/images/blogs/2019/06/fig17b-quasarrat.png"></p>
<h2><b>Additional Observations</b></h2>
<h3><b>Loader Variant Differences</b><br>
</h3>
<p><b><i>Features common for all variants:</i></b></p>
<ul>
<li>Most of the samples we collected seem to be compiled with VisualStudio 2010 RTM build 30319, with the exception of variant 4, which uses a different/unknown compiler signature</li>
<li>Some strings are encrypted with an algorithm based on a custom implementation of AES256 combined with XOR</li>
<li>The .NET loader is always injected using the Microsoft CPPHostCLR method; its entry point class/method names are random and differ for each sample</li>
<li>The .NET loader is obfuscated with ConfuserEx v1.0.0</li>
</ul>
<p><b><i>Features common for variants 2 and newer:</i></b></p>
<ul>
<li>The .NET loader size is 65,536 bytes&nbsp;&nbsp;&nbsp;</li>
<li>The .NET loader internal name imitates a random valid file name from the .NET runtime directory&nbsp;</li>
<li>The second stage is encrypted using an XOR-based algorithm with two hardcoded 1-byte keys, differing for each sample&nbsp;</li>
<li>AES and XOR keys for string decryption are stored hardcoded as randomly generated strings, differing for each sample</li>
</ul>
<p><b><i>Variant 1:</i></b></p>
<ul>
<li>Assumed development timeline: June 2017 – December 2017</li>
<li>Size of the initial loader binary: ~150 KB</li>
<li>.NET loader size: 56,832 bytes</li>
<li>.NET loader internal name: loader.dat/loader2.dat</li>
<li>Contains only one layer of obfuscation</li>
<li>Second stage encrypted with simple XOR, using a hardcoded key composed of 8 random upper/lowercase letters</li>
<li>Contains a randomly named export that creates a service as persistence mechanism</li>
<li>Hardcoded string decryption keys<ul>
<li>AES = 1234567890ABCDEF1234567890ABCDEF</li>
<li>XOR = 1234567890ABCDEF</li>
</ul>
</li>
</ul>
<p><b><i>Variant 2:</i></b></p>
<ul>
<li>Assumed development timeline: January 2018</li>
<li>Size of the initial loader binary: 163 - 169 KB</li>
</ul>
<p><b><i>Variant 3:</i></b></p>
<ul>
<li>Assumed development timeline: February 2018</li>
<li>Size of the initial loader binary: 262 KB</li>
<li>A second layer of obfuscation has been added</li>
<li>A function inside ServiceMain decrypts the second stage DLL (SvcDll.dll) and shellcode-like routine that injects this DLL into memory and calls the &quot;FuckYouAnti&quot; export</li>
<li>2nd stage + loader size: 163,840 bytes</li>
<li>Some samples of this version contain debugging strings</li>
<li>Some samples of this version are signed with a valid certificate from CONVENTION DIGITAL LTD issued by Symantec<ul>
<li>Serial number 52 25 B8 E2 2D 3B BC 97 3F DD 24 2F 2C 2E 70 0C</li>
</ul>
</li>
</ul>
<p><b><i>Variant 4:</i></b></p>
<ul>
<li>Assumed development timeline: April 2018</li>
<li>Size of the initial loader: 439 KB</li>
<li>2nd stage + loader size: 236,532 bytes; there is additional ~72kb of static buffers comparing to previous versions</li>
<li>Setting persistence mechanism has now been shifted to a standalone module (DILLJUICE)<i><sup>[4]&nbsp;</sup></i></li>
<li>This version uses a different/unknown compiler</li>
</ul>
<p><b><i>Variant 5:</i></b></p>
<ul>
<li>Assumed development timeline: April – May 2018</li>
<li>Size of the initial loader: 291 – 293 KB</li>
<li>2nd stage + loader size: 236,532 bytes&nbsp;</li>
<li>Second stage decryption functionality moved to separate subroutine</li>
<li>Added printing of a random base64 string of a random length between 2,000 and 5,000 bytes, possibly as a simple polymorphic measure (only version 5)</li>
<li>In several later samples from that variant the FuckYouAnti function from AntiLib creates an additional mutex &quot;ABCDEFGHIGKLMNOPQRSTUVWXYZ”</li>
</ul>
<p><b><i>Variant 6:</i></b></p>
<ul>
<li>Assumed development timeline: July – August 2018</li>
<li>Size of the initial loader: 341 – 394 KB</li>
<li>2nd stage + loader size: 236,532 bytes</li>
<li>Second stage decryption moved back to ServiceMain</li>
</ul>
