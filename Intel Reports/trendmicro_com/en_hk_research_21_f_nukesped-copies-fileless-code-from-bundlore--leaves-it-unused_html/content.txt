	<h1 class="article-details__title">NukeSped Copies Fileless Code From Bundlore, Leaves It Unused</h1>
	<p class="article-details__description">While investigating samples of NukeSped, a remote access trojan (RAT), Trend Micro came across several Bundlore adware samples using the same fileless routine that was spotted in NukeSped.</p>
	<p class="article-details__author-by">By: Luis Magisa, Ariel Neimond Lazaro 
		
			<time class="article-details__date">2021/06/22</time>

            <div class="richText">
	
    


	
		<div class="responsive-table-wrap">
			<p>While investigating samples of <a href="/en_gb/research/19/k/mac-backdoor-linked-to-lazarus-targets-korean-users.html">NukeSped</a>, a remote access trojan (RAT), Trend Micro came across several <a href="https://documents.trendmicro.com/assets/wp/wp-automatic-classifying-of-mac-os-x-samples.pdf" target="_blank">Bundlore</a> adware samples using the same fileless routine that was spotted in NukeSped. The backdoor has been attributed to the cybercriminal group Lazarus, <a href="https://www.trendmicro.com/vinfo/tmr/?/us/security/news/cybercrime-and-digital-threats/a-look-into-the-lazarus-groups-operations">which has been active since at least 2014</a>. There are <a href="https://www.trendmicro.com/vinfo/tmr/?/us/threat-encyclopedia/malware/trojan.macos.nukesped.yxbbs/">multiple variants</a> of NukeSped, which is designed to run on 32-bit systems and uses encrypted strings to evade detection. Recently, <a href="https://threatpost.com/lazarus-targets-defense-threatneedle-malware/164321/" target="_blank">a more sophisticated form of this trojan called ThreatNeedle</a> surfaced as part of a cyberespionage campaign by Lazarus.  </p>
<p>The encrypted Mach-O file discovered in these samples has upgraded Bundlore — a malware family that installs adware in a target’s device under the guise of downloading legitimate applications — to a stealthier and memory-resident threat. Bundlore has also been known to target macOS devices and was linked to <a href="https://news.sophos.com/en-us/2020/06/18/new-bundlore-adware-targets-macos-with-updated-safari-extensions/" target="_blank">an attack on macOS Catalina users</a> last year.    </p>
<p>Our analysis of the file <i>Ants2WhaleHelper</i> used by Lazarus led us to detect it as NukeSped. Another file with NukeSped detection, <i>unioncryptoupdater</i>, was also found in VirusTotal. Both contained a routine that looks to be based on <a href="https://github.com/ytlvy/CTest/blob/9039ccf993c478a29d77018135d992d79c1b1a0a/C/RunLib/run_bin.c" target="_blank">a GitHub submission</a>. Curiously, however, neither of these files seems to make use of this routine.  </p>
<p>Using Interactive Disassembler Pro (IDA Pro) on the <i>Ants2WhaleHelper</i> file revealed its main payload as <i>_mapBuffer</i> (Figure 1), which appears to be a modified version of the <i>_memory_exec</i> function (Figure 2). This function looks like it was based on code from the GitHub post; however, there were no references that point to the <i>_memory_exec</i> function.</p>
<p style="text-align: center;"><img src="/content/dam/trendmicro/global/en/research/21/f/nukesped-copies-fileless-code-from-bundlore,-leaves-it-unused-/filelessbundlore-figure01.png"/><br />
Figure 1. The <i>_mapBuffer</i> function </p>
<p style="text-align: center;"><img src="/content/dam/trendmicro/global/en/research/21/f/nukesped-copies-fileless-code-from-bundlore,-leaves-it-unused-/filelessbundlore-figure02.png"/><br />
Figure 2. The <i>_memory_exec</i> function copied from the GitHub post </p>
<p>Moreover, the payload has a <i>_resolve_symbol</i> function that does not seem to be used. It also does not appear to be necessary, as evidenced in Figure 3. NukeSped typically retrieves and launches its payload from a web server, so it does not need the superfulous <i>_resolve_symbol </i>function, which locates data internally. As Figure 4 shows, searching for the operation codes of this function on VirusTotal led to its detection in 201 files. The results yielded only two NukeSped samples while the rest were Bundlore samples.  </p>
<p style="text-align: center;"><img src="/content/dam/trendmicro/global/en/research/21/f/nukesped-copies-fileless-code-from-bundlore,-leaves-it-unused-/filelessbundlore-figure03.png"/><br />
Figure 3. The <i>_resolve_symbol</i> functions of NukeSped (left) vs. Bundlore (right) </p>
<p style="text-align: center;"><img src="/content/dam/trendmicro/global/en/research/21/f/nukesped-copies-fileless-code-from-bundlore,-leaves-it-unused-/filelessbundlore-figure04.png"/><br />
Figure 4. The searched operation codes </p>
<p>Similarly, a search using VirusTotal's Retrohunt yielded 273 results; most of these were Bundlore files and only three were Nukesped files. However, one of these Nukesped samples was verified as the parent of a Nukesped file from the previous search. Among the Bundlore samples discovered, the oldest one dates back to May of last year. Further investigation of these Bundlore samples from the VirusTotal query revealed that these were indeed using fileless routines, enabling Bundlore to execute a payload directly from memory.  </p>
<p><span class="body-subhead-title">Bundlore’s fileless routine</span></p>
<p>Our study of the Bundlore samples showed that these utilize the same functions that were found unused in the NukeSped samples. As seen in Figure 5, these were obfuscated, as they were under random names when disassembled in IDA Pro. While the functions have some differences, the routine for in-memory file execution remains the same (Figure 6 and 8).  </p>
<p style="text-align: center;"><img src="/content/dam/trendmicro/global/en/research/21/f/nukesped-copies-fileless-code-from-bundlore,-leaves-it-unused-/filelessbundlore-figure05.png"/><br />
Figure 5. The obfuscated functions </p>
<p style="text-align: center;"><img src="/content/dam/trendmicro/global/en/research/21/f/nukesped-copies-fileless-code-from-bundlore,-leaves-it-unused-/filelessbundlore-figure06.png"/><br />
Figure 6. The disassembly of NukeSped (left column) vs. Bundlore (right column) samples </p>
<p>The main routines of one of the Bundlore samples (sha256:0a3a5854d1ae3f5712774a4eebd819f9e4e3946f36488b4e342f2dd32c8e5db2) are as follows: </p>
<ol>
<li>Decrypt the <i>__DATA.__data</i> section to reveal the embedded Mach-O file, as shown in Figure 7. The decryption uses an XOR key that is incremented per cycle: for example, a 0xDD increment by 0x2A, 0xDD, 0x00, 0x2A, 0x54, 0x7E, 0xA8, 0xD2, 0xFC, 0x00, and so on.</li>
<br />
<p style="text-align: center;"> <img src="/content/dam/trendmicro/global/en/research/21/f/nukesped-copies-fileless-code-from-bundlore,-leaves-it-unused-/filelessbundlore-figure07.png"/><br />
Figure 7. The decryption routine of the <i>__DATA.__data</i> section </p>
<li>Invoke a function called <i>NSCreateObjectFileImageFromMemory</i> to create an adware image from the Mach-O file in memory. Afterward, <i>NSLinkModule</i> is called to link the malicious image to the main executable's image library. The Mach-O file format is changed from an executable (0x02) to a bundle (0x08) before it can call <i>NSCreateObjectFileImageFromMemory</i>, as was shown in Figure 6.  </li>
<br />
<li>Parse the Mach-O file's header structure in memory for <i>value(LC_MAIN)</i>, a load command that has the value 0x80000028. This command contains data such as the offset of the Mach-O file's entry point (Figure 8). Afterward, the adware retrieves the offset and goes to the entry point. </li>
</ol>
<p style="text-align: center;"><img src="/content/dam/trendmicro/global/en/research/21/f/nukesped-copies-fileless-code-from-bundlore,-leaves-it-unused-/filelessbundlore-figure08.png"/><br />
Figure 8. Finding the entry point of the malicious image in NukeSped (left column) vs. Bundlore (right column) </p>
<p><span class="body-subhead-title">Bundlore’s Mach-O file runs in memory</span></p>
<p>The decryption keys and increment values differ across the Bundlore samples. To gain a better understanding of the embedded file, we created a Python script to decrypt and extract their embedded Mach-O files. By doing so, we were able to observe one such decrypted Mach-O file (sha256: a7b6639d9fcdb13ae5444818e1c35fba4ffed90d9f33849d3e6f9b3ba8443bea) with the routines shown in Figure 9. It connects to a target URL (13636337101185210173363631[.]cloudfront[.]net/?cc-00&amp;), but the address varies among the samples. An app bundle called <i>Player.app</i>, which poses as Flash Player,<i> </i>is then downloaded and extracted into a /tmp directory. The <i>chmod 777</i> command is used on the extracted app bundle, after which the fake application is launched. While it performs these routines, Bundlore displays a fraudulent error message (Figure 10). Upon completion, it goes dormant by calling the sleep function and looping it repeatedly. </p>
<p>There were no significant differences seen when running the Bundlore samples in macOS Big Sur and macOS Catalina. However, our researchers found that with the default settings of macOS, in which the System Integrity Protection (SIP) and Gatekeeper security features are enabled, the Bundlore samples are blocked and are unable to run. This was observed in both macOS Catalina and macOS Big Sur environments; similarly, the Bundlore samples were also blocked and unable to run under the default settings of macOS Monterey, Apple's recently released operating system.</p>
<p style="text-align: center;"><img src="/content/dam/trendmicro/global/en/research/21/f/nukesped-copies-fileless-code-from-bundlore,-leaves-it-unused-/filelessbundlore-figure09.png"/><br />
Figure 9. The decrypted Mach-O file’s main routines </p>
<p style="text-align: center;"><img src="/content/dam/trendmicro/global/en/research/21/f/nukesped-copies-fileless-code-from-bundlore,-leaves-it-unused-/filelessbundlore-figure10.png"/><br />
Figure 10. The fake error message displayed by <i>Player.app</i></p>

This ensures that no significant incidents go unnoticed and allows faster response to threats before they can do any real damage to the system. </p>
