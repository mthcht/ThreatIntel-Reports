	<h1 class="article-details__title">Investigating the Gootkit Loader</h1>
	<p class="article-details__description">Gootkit has been tied to Cobalt Strike as well as other ransomware attacks in the past. Some of these recent victims later suffered SunCrypt ransomware attacks, although it is unclear if this was because of the Gootkit threat actor or if access was sold to other threat actors.</p>
	<p class="article-details__author-by">By: Marc Lanzendorfer
		
			<time class="article-details__date">December 11, 2020</time>
		
		
		<span>Read time:&nbsp;</span><span class="eta"></span> (<span class="words"></span> words)
	</p>

	<div class="article-details__icons">

</div>
            <div class="richText">
	
    


	
		<div>
			<p>Since October 2020, we saw an increase in the number of Gootkit cases targeting users in Germany. We investigated this development and found that the Gootkit loader was now capable of sophisticated behavior that enabled it to surreptitiously load itself onto an affected system and make analysis and detection more difficult.</p>
<p>This capability was used to deploy a DLL file. Gootkit has, in the past, been tied to Cobalt Strike as well as other ransomware attacks. Some of these recent victims later suffered SunCrypt ransomware attacks, although it is unclear if this was because of the Gootkit threat actor or if access was sold to other threat actors. We've also discovered in recent weeks that the Gootkit loader is being used in combination with REvil/Sodinokibi ransomware.</p>
<p><span class="body-subhead-title">Infection vector: Malicious search engine results</span></p>
<p>In the cases we saw, the Gootkit loader initially arrives via a ZIP archive downloaded from a website. These malicious websites can be found in malicious search engine results, like this:</p>

		</div>
	

</div>
            <div class="image">
	
    


	<figure class="image-figure">
		
			<img src="/content/dam/trendmicro/global/en/research/20/l/investigating-the-gootkit-loader/fig1%20-%20gootkit.png" alt="Malicious search engine results"/>
		
		<div class="caption-image-container ">
			<figcaption>Figure 1. Malicious search engine results</figcaption>
			
		</div>
	</figure>

</div>
            <div>




    
    
    <div class="richText">
	
    


	
		<div>
			<p>In this particular instance, <i>fifa manager kostenlos</i> can be translated as <i>fifa manager free</i>. Note that the search term used can vary significantly; we just used this search term as an example. We have encountered other cases where the search terms were <i>Aldi Talk postident coupon</i> and <i>Control Center 4 download</i>.</p>
<p>Clicking the link leads to a page on a legitimate site; however, the site has been compromised and used to host a malicious page. The aforementioned page looks legitimate:</p>

		</div>
	

</div>


    
    
    <div class="image">
	
    


	<figure class="image-figure">
		
			<img src="/content/dam/trendmicro/global/en/research/20/l/investigating-the-gootkit-loader/fig2%20-%20gootkit.png" alt="Malicious page"/>
		
		<div class="caption-image-container ">
			<figcaption>Figure 2. Malicious page</figcaption>
			
		</div>
	</figure>

</div>


    
    
    <div class="richText">
	
    


	
		<div>
			<p>It is meant to look like a legitimate forum, with a post containing a link to a file relevant to the search engine query. This particular link is more sophisticated than it looks, however. Attempting to redownload the same file from the same URL from the same host/machine fails; however, doing so from a different one succeeds, but the downloaded file has a different hash than the original file. This suggests that the server generates this file as it is needed, uniquely for each download attempt.</p>
<p><span class="body-subhead-title">Analysis of the downloaded file</span></p>
<p>The downloaded file is a ZIP file that contains a heavily encoded JS file (which shares the same filename as the ZIP file, save for the extension). We were able to use <a href="http://jsnice.org/" target="_blank">JSNice</a> to produce human-readable code:</p>

		</div>
	

</div>


    
    
    <div class="image">
	
    


	<figure class="image-figure">
		
			<img src="/content/dam/trendmicro/global/en/research/20/l/investigating-the-gootkit-loader/fig3%20-%20gootkit.png" alt="Deobfuscated code"/>
		
		<div class="caption-image-container ">
			<figcaption>Figure 3. Deobfuscated code</figcaption>
			
		</div>
	</figure>

</div>


    
    
    <div class="richText">
	
    


	
		<div>
			<p>Of interest here is the function “MT71,” which contains a variable with very long content. Trying to run the script with online runtimes such as at <a href="https://ideone.com/" target="_blank">Ideone</a> fails with the following error:</p>

		</div>
	

</div>


    
    
    <div class="image">
	
    


	<figure class="image-figure">
		
			<img src="/content/dam/trendmicro/global/en/research/20/l/investigating-the-gootkit-loader/fig4%20-%20gootkit.png" alt="Error message"/>
		
		<div class="caption-image-container ">
			<figcaption>Figure 4. Error message</figcaption>
			
		</div>
	</figure>

</div>


    
    
    <div class="richText">
	
    


	
		<div>
			<p>What becomes apparent through the error is what the WScript.Shell object is trying to do:</p>

		</div>
	

</div>


    
    
    <div class="image">
	
    


	<figure class="image-figure">
		
			<img src="/content/dam/trendmicro/global/en/research/20/l/investigating-the-gootkit-loader/fig5%20-%20gootkit.png" alt="Partially deobfucated and beautified code"/>
		
		<div class="caption-image-container ">
			<figcaption>Figure 5. Partially deobfucated and beautified code</figcaption>
			
		</div>
	</figure>

</div>


    
    
    <div class="richText">
	
    


	
		<div>
			<p>A new object is created (“WScript.Shell”), which tries to read the registry key “HKCU\SOFTWARE\nTpm\”. In case this registry key does not exist, it performs the following actions:</p>
<ul>
<li><span class="rte-red-bullet">A key with an empty value will be written at HKCU\SOFTWARE\nTpm</span></li>
<li><span class="rte-red-bullet">The value of “bE50” will be set to 32</span></li>
</ul>
<p>If the key already exists, the execution of the script will fail since “bE50” is not set. It becomes clear that this key is being used as a marker to check if the initial loader was already executed on an infected host.</p>
<p>To sandbox this script, we used <a href="https://github.com/HynekPetrak/malware-jail" target="_blank">malware-jail</a> from HynekPetrak. A line had to be added to define the variable of bE50 as 32; otherwise, the script will fail (due to the requirement of accessing the registry key). </p>

		</div>
	

</div>


    
    
    <div class="image">
	
    


	<figure class="image-figure">
		
			<img src="/content/dam/trendmicro/global/en/research/20/l/investigating-the-gootkit-loader/fig6%20-%20gootkit.png" alt="Modified script"/>
		
		<div class="caption-image-container ">
			<figcaption>Figure 6. Modified script</figcaption>
			
		</div>
	</figure>

</div>


    
    
    <div class="richText">
	
    


	
		<div>
			<p>The following command was then used to run the JS file in the jail:</p>
<p><span class="blockquote">node jailme.js -c ./config_wscript_only.json --t404 tr_input/fifa.js tr_output/test -o tr_output/fifa_out.json --trace</span></p>
<p>This command would return a 404 error whenever the Javascript file sends a request upon execution. It created the following output files:</p>

		</div>
	

</div>


    
    
    <div class="image">
	
    


	<figure class="image-figure">
		
			<img src="/content/dam/trendmicro/global/en/research/20/l/investigating-the-gootkit-loader/fig7%20-%20gootkit.png" alt="Contents of testurls.json"/>
		
		<div class="caption-image-container ">
			<figcaption>Figure 7. Contents of testurls.json, showing URLs that the malicious code tried to access</figcaption>
			
		</div>
	</figure>

</div>


    
    
    <div class="image">
	
    


	<figure class="image-figure">
		
			<img src="/content/dam/trendmicro/global/en/research/20/l/investigating-the-gootkit-loader/fig8%20-%20gootkit.png" alt="Partial contents of fifa_out.json"/>
		
		<div class="caption-image-container ">
			<figcaption>Figure 8. Partial contents of fifa_out.json; contains some interesting artifacts</figcaption>
			
		</div>
	</figure>

</div>


    
    
    <div class="richText">
	
    


	
		<div>
			<p>Looking at the output JSON file shows that the variable “qI27” is an array of three domains:</p>
<ul>
<li><span class="rte-red-bullet">www.adpm.com[.]br</span></li>
<li><span class="rte-red-bullet">windowp[.]org</span></li>
<li><span class="rte-red-bullet">www.ai-tech[.]paris</span></li>
</ul>
<p>Converting the whole line of “qI61” into a readable format reveals the following code block:</p>

		</div>
	

</div>


    
    
    <div class="image">
	
    


	<figure class="image-figure">
		
			<img src="/content/dam/trendmicro/global/en/research/20/l/investigating-the-gootkit-loader/fig9%20-%20gootkit.png" alt="Contents of the line qI61"/>
		
		<div class="caption-image-container ">
			<figcaption>Figure 9. Contents of the line qI61, showing the connection attempts to one of the three destination domains sequentially</figcaption>
			
		</div>
	</figure>

</div>


    
    
    <div class="richText">
	
    


	
		<div>
			<p>The snippet defines the following flow, running the following code against at least one of the URLs stored in QI27:</p>
<ol>
<li>“Og13” will be set to a random number after the point with a maximum length of 100</li>
<li>The local user’s DNS domain is queried<ol>
<li>if the machine is joined to a domain, “Og13” will have 278146 added at the end</li>
</ol>
</li>
<li>A web request to the URL selected in step 1 will be initiated, using sub-parameters<ol>
<li>search.php?gqhncrqossifzp={the number in the variable of Og13}</li>
</ol>
</li>
<li>It checks for the return code of the web request a. if not 200 (okay), the script goes to sleep and then tries the next URL.</li>
<li>If the web request is 200 (answer received), it stores the response in variable “zI11”</li>
<li>It checks whether the response text from the server contains the “Og13” value<ol>
<li>If it does not, it goes to sleep and then tries the next URL.</li>
</ol>
</li>
<li>If the value is in the response, it removes the “Og13” value from “zI11”</li>
<li>It replaces a double-digit number in brackets, e.g. (12), with a response from a function using the variable “KT44”, which is unknown at this stage.</li>
<li>It then calls another function “Nm34” (also currently unknown), passing the new “va67” variable on.</li>
</ol>
<p>With the above information, we now know that this loader makes a difference between domain and non-domain hosts (by adding “278146” at the end of the search parameter).</p>
<p>We can change environmental variables via the wscript.js file. As the malicious script is looking for the environmental variable “UserDNSDomain,” this was added to the configuration; we also changed the default username:</p>

		</div>
	

</div>


    
    
    <div class="image">
	
    


	<figure class="image-figure">
		
			<img src="/content/dam/trendmicro/global/en/research/20/l/investigating-the-gootkit-loader/fig10%20-%20gootkit.png" alt="Modified script"/>
		
		<div class="caption-image-container ">
			<figcaption>Figure 10. Modified script</figcaption>
			
		</div>
	</figure>

</div>


    
    
    <div class="richText">
	
    


	
		<div>
			<p>Rerunning the script — after changing the wscript.js parameter to provide a domain — reveals the request of the following URL:</p>

		</div>
	

</div>


    
    
    <div class="image">
	
    


	<figure class="image-figure">
		
			<img src="/content/dam/trendmicro/global/en/research/20/l/investigating-the-gootkit-loader/fig11%20-%20gootkit.png" alt="Requested URL"/>
		
		<div class="caption-image-container ">
			<figcaption>Figure 11. Requested URL</figcaption>
			
		</div>
	</figure>

</div>


    
    
    <div class="richText">
	
    


	
		<div>
			<p>Based on this observation, we can now run jailme.js without the --t404 option, but with the --down=y option. This allowed us to send the queries and download any requested files. By default, the jailme.js will stop executing the script after 60 seconds. The result of the queried URL now includes all three identified domains, shown below and including the responses:</p>

		</div>
	

</div>


    
    
    <div class="image">
	
    


	<figure class="image-figure">
		
			<img src="/content/dam/trendmicro/global/en/research/20/l/investigating-the-gootkit-loader/fig12%20-%20gootkit.png" alt="URLs and responses"/>
		
		<div class="caption-image-container ">
			<figcaption>Figure 12. URLs and responses</figcaption>
			
		</div>
	</figure>

</div>


    
    
    <div class="richText">
	
    


	
		<div>
			<p>While we received HTTP 200 responses, none of these included the random strings needed for the script to proceed. This was true using both samples we received. We are unsure why this is the case; all we can say for sure is that the servers are currently not providing the files to be downloaded by Gootkit if they are analyzed in this manner.</p>
<p><span class="body-subhead-title">Registry Analysis</span></p>
<p>We can use the presence of registry keys known to Gootkit to test if it has been deployed on an affected system.</p>
<p>On test machines, we were able to verify that the created registry entries were present:</p>

		<div>
			<p>The registry values in the last key can be merged into a PowerShell script:</p>

		</div>
	

</div>


    
    
    <div class="image">
	
    


	<figure class="image-figure">

		<div>
			<p>Most of this script is encoded; decoding it results in the following:</p>

		</div>
	

</div>


    
    
    <div class="image">
	
    


	<figure class="image-figure">
		
			<img src="/content/dam/trendmicro/global/en/research/20/l/investigating-the-gootkit-loader/fig18%20-%20gootkit.png" alt="Decoded code"/>

			<figcaption>Figure 18. Decoded code</figcaption>

</div>


    
    
    <div class="richText">
	
    


	
		<div>
			<p>This code is, by default, loaded into memory. If this code is instead saved to a file, this turns out to be a .NET DLL file. (This particular file is detected as Trojan.Win32.DELF.WLDT).</p>
<p>Opening this particular file in a .NET decompiler shows that it also contains more encoded code. Using a similar technique to dump the contents into a file reveals that this is also an executable file. This one is detected as Trojan.Win32.MALREP.THJBGBO, which we believe is the payload that this loader delivered to the affected system.</p>
<p><span class="body-subhead-title">Conclusions and Trend Micro solutions</span></p>
<p>This particular threat highlights the sophistication of today’s malware-delivering loaders. In a system without any security solutions enabled, there would be barely any sign of the infection, making analysis and removal more difficult.</p>
<p>With the appropriate Trend Micro solutions, the user would have been protected from this threat. <a href="https://www.trendmicro.com/en_us/business/products/network/advanced-threat-protection/analyzer.html">Deep Discovery Analyzer</a> would have proactively detected the script as a backdoor and classified it as malicious; <a href="https://www.trendmicro.com/en_us/business/products/user-protection/sps/endpoint.html">Apex One</a> would also have been capable of blocking the threat once it was executed. </p>
