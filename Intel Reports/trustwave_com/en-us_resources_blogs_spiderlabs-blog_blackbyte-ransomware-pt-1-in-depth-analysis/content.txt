          <h1>BlackByte Ransomware – Pt. 1 In-depth Analysis</h1>
          
        </div>
        <div class="blog-details-header-right">
          
            
            <a href="#" class="theme-changer">Change theme to light</a>
            
            
            

            
            <div class="hero-spider" role="img" aria-label="BlackByte Ransomware – Pt. 1 In-depth Analysis"></div>
            

            
          
        </div>
      </div>
      <div class="toc-content content-wrapper">
        
        <div class="wrapper-main_content">
          
          <div class="blog-details-metadata ">
            <span class="date-time">
              <time datetime="2021-10-15 14:00:00">October 15, 2021</time>
            </span>
            <span class="minread">10 Minute Read</span>
            
              <span class="blog-author">by Rodel Mendrez, Lloyd Macrohon</span>
            
          </div> 
          
          
          <div class="wrapper-blog-content">
            <span id="hs_cos_wrapper_post_body" class="hs_cos_wrapper hs_cos_wrapper_meta_field hs_cos_wrapper_type_rich_text" style="" data-hs-cos-general-type="meta_field" data-hs-cos-type="rich_text"><p><a href="https://www.trustwave.com/en-us/resources/blogs/spiderlabs-blog/blackbyte-ransomware-pt-2-code-obfuscation-analysis/" title="BlackByte Ransomware Pt 2 Code Obfuscation Analysis" data-udi="umb://document/6ed7918ad73b4f3f940a2459b2a495db">Please click here for Part 2</a></p>
<!--more--><p><strong>UPDATE 19.October.2021<span>&nbsp;</span></strong>- Based on some reactions and responses to our BlackByte analysis, and specifically, the included decryptor, we wanted to provide an update and some clarification.</p>
<p>First off, we’ve updated the decryptor on github to include two new files. One is the compiled build of the executable to make the tool more accessible and the second is a sample encrypted file “spider.png.blackbyte”&nbsp;that can be used to test the decryptor.</p>
<p>The decryptor takes a file (or files) encrypted by the ransomware as well as the raw encryption key in the fake image file “forest.png”. For instance:</p>
<p>BlackByteDecryptor forest.png spider.png.blackbyte&nbsp;</p>
<p>In the example the file “spider.png.blackbyte” has been encrypted by BlackByte. By using the key in “forest.png”, we can decrypt the file and retrieve “spider.png”. At no time is the original encrypted file, “spider.png.blackbyte”,&nbsp; modified.</p>
<p>As mentioned in the blog post, we know that “forest.png” was used as a key across multiple systems infected with BlackByte. Based on the retrieval of that key from a hardcoded web server (<em>hxxp://45.9.148.114/forest.png</em>), we believe that it’s likely that this key was reused across many, if not all, current victims.</p>
<p>However, it’s certainly possible that the actors behind BlackByte might have changed that key file at some point. While it was taken down and no longer accessible, they may have used different “forest.png” files for different victims.</p>
<p>In cases where the key might be different, the decryptor and key we provide<span>&nbsp;</span><a href="https://github.com/SpiderLabs/BlackByteDecryptor">here</a><span>&nbsp;</span>will not work to recover the victim’s data. That said, it will also cause no harm or damage your data (at least not any further than it already was by BlackByte). When the victim attempts to run the decryptor as in the previous example:</p>
<p>BlackByteDecryptor forest.png spider.png.blackbyte&nbsp;</p>
<p>The resulting recovered file “spider.png” will simply be garbage data as the incorrect key was used. Additionally, if the file “spider.png” already existed on the system for any reason, perhaps recovered from a backup, or copied from another system, the decryptor will exit out with the error:</p>
<p>Warning: Target file<span>&nbsp;</span><em>spiderlabs.png</em><span>&nbsp;</span>exists. Not overwriting.</p>
<p>The important part here is that the original, encrypted file, “spider.png.blackbyte”, will still be on the system and unmodified. If the victim can access the correct key file originally used to encrypt their data, then recovery is still possible.</p>
<p>The key itself however is only downloaded to the victim system’s memory and not stored on disk. This means victims would need to pull the key using memory dump tools on a fresh victim system. Because of this, and since we believe that the key we spotted and archived is likely reused by many victims, we provide that key with the decryptor in the hopes of helping those affected to recover.</p>
<hr>
<h3>Introduction</h3>
<p>During a recent malware incident response case, we encountered an interesting piece of ransomware that goes by the name of BlackByte.</p>
<p>We thought that this ransomware was not only interesting but also quite odd:</p>
<ol>
<li>Same as other notorious ransomware variants like REvil, BlackByte also avoids systems with Russian and ex-USSR languages.</li>
<li>It has a worm functionality similar to RYUK ransomware.</li>
<li>It creates a wake-on-LAN magic packet and sends it to the target host - making sure they are alive when infecting them.&nbsp;</li>
<li>The author hosted the encryption key in a remote HTTP server and in a hidden file with .PNG extension.</li>
<li>The author lets the program crash if it fails to download the encryption key.</li>
<li>The RSA public key embedded in the body is only used once, to encrypt the raw key to display in the ransom note – that’s it.</li>
<li>The ransomware uses only one symmetric key to encrypt the files.</li>
</ol>
<p>The auction site that is linked in the ransom note is also quite odd, see below. The site claims that it has exfiltrated data from its victims, but the ransomware itself does not have any exfiltration functionality. So this claim is probably designed to scare their victims into complying.</p>

<h3>File Decryption</h3>
<p>Unlike other ransomware that may have a unique key in each session, BlackByte uses the same raw key (which it downloads) to encrypt files and it uses a symmetric-key algorithm – AES.&nbsp; To decrypt a file, one only needs the raw key to be downloaded from the host. As long as the .PNG file it downloaded remains the same, we can use the same key to decrypt the encrypted files.</p>
<p>So, we wrote a file decryptor that is available at this link:<span>&nbsp;</span><a href="https://github.com/SpiderLabs/BlackByteDecryptor">https://github.com/SpiderLabs/BlackByteDecryptor</a></p>

<h3>BlackByte: Preparing the Infected System</h3>
<p>Before encrypting, BlackByte first prepares the system so that nothing may hamper it from its file encryption routine. <h3>BlackByte: Encryption Routine</h3>
<p>What we found interesting about this ransomware, is that it initially downloads a .PNG file from the link<span>&nbsp;</span><span>hxxp://45.9.148.114/forest.png</span><span>&nbsp;</span>which contains a key to be used later to encrypt the files. If the ransomware fails to download the key, it will crash and will save the infected system from getting its files encrypted.</p>

<p>The file it downloaded is not actually a PNG image file, instead:</p>
<ul>
<li>The first 40 bytes is the raw key used to encrypt the user's files, this is encrypted with 3DES.</li>
<li>The last 32 bytes contain the 3DES key used to decrypt the first 40 bytes raw key.</li>
<li>The raw key then goes through a PBKDF2 derivation function to derive the AES 128-bit key and Initialization Vector for the AES algorithm used to encrypt the user files.</li>
<li>This raw key is also re-encrypted using RSA with a public key embedded in the module and displayed in the ransom note.</li>
<li>The attacker can decrypt this key using his private key, but this key is the same - provided the user always gets the same “forest.png” file. Presumably, the file forest.png is replaced periodically.</li>
</ul>
<p>The first 40 bytes of the PNG file is a key (encrypted in TripleDES) used later for the ransomware’s file encryption.</p>

<p>A target file to be encrypted undergoes file size filtering:</p>
<ul>
<li>If the file is greater than 150MB
<ul>
<li>encrypt the first 50MB and the last 50MB of the file</li>
</ul>
</li>
<li>If the file is greater than 15MB
<ul>
<li>encrypt the first 5MB and the last 5MB of the file</li>
</ul>
</li>
<li>If the file is greater than 3MB
<ul>
<li>encrypt the first 1MB and the last 1MB of the file</li>
</ul>
</li>
<li>If the file is less than 3MB
<ul>
<li>encrypt the whole file</li>
</ul>
</li>
</ul>
