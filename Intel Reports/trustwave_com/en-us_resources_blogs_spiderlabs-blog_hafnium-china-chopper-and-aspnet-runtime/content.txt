          <h1>HAFNIUM, China Chopper and ASP.NET Runtime</h1>
          
        </div>
        <div class="blog-details-header-right">
          
            
            <a href="#" class="theme-changer">Change theme to light</a>
            
            
            

            
            <div class="hero-spider" role="img" aria-label="HAFNIUM, China Chopper and ASP.NET Runtime"></div>
            

            
          
        </div>
      </div>
      <div class="toc-content content-wrapper">
        
        <div class="wrapper-main_content">
          
          <div class="blog-details-metadata ">
            <span class="date-time">
              <time datetime="2021-03-15 05:00:00">March 15, 2021</time>
            </span>
            <span class="minread">4 Minute Read</span>
            
              <span class="blog-author">by Joshua Deacon</span>
            
          </div> 
          
          
          <div class="wrapper-blog-content">
            <span id="hs_cos_wrapper_post_body" class="hs_cos_wrapper hs_cos_wrapper_meta_field hs_cos_wrapper_type_rich_text" style="" data-hs-cos-general-type="meta_field" data-hs-cos-type="rich_text"><p>The recent Microsoft Exchange Server zero-day exploits (<span><a href="https://msrc.microsoft.com/update-guide/vulnerability/CVE-2021-26855">CVE-2021-26855</a></span>, <span><a href="https://msrc.microsoft.com/update-guide/vulnerability/CVE-2021-26857">CVE-2021-26857</a></span>, <span><a href="https://msrc.microsoft.com/update-guide/vulnerability/CVE-2021-26858">CVE-2021-26858</a></span>, <span><a href="https://msrc.microsoft.com/update-guide/vulnerability/CVE-2021-27065">CVE-2021-27065</a>)</span>&nbsp; have seen tens of thousands of organizations compromised by <span><a href="https://www.microsoft.com/security/blog/2021/03/02/hafnium-targeting-exchange-servers/">HAFNIUM</a></span> and numerous other threat actor groups. Working closely with our customers across the globe, we have quickly been able to identify and isolate attributes of those attacks – particularly the China Chopper web shell being uploaded to compromised Microsoft Exchange servers with a publicly facing Internet Information Services (IIS) web server. Although the China Chopper web shell <span><a href="https://www.fireeye.com/blog/threat-research/2013/08/breaking-down-the-china-chopper-web-shell-part-i.html">has been around for years</a></span>, in the interest of providing more information to the security community during this time,&nbsp; we decided to dig even deeper into how the China Chopper web shell works as well as how the ASP.NET runtime serves these web shells.</p> 
<!--more--> 
<p>China Chopper is an Active Server Page Extended (ASPX) web shell that is typically planted on an Internet Information Services (IIS) server through an exploit. China Chopper is used for post-exploitation by giving attackers access to execute any code they want on the server.</p> 
<p>The China Chopper server-side ASPX web shell is extremely small and typically, the entire thing is just one line. There are multiple versions of this web shell for executing code in different languages such as ASP, ASPX, PHP, JSP, and CFM. In this blog, we will cover the JScript version; however, they all are very similar aside from the language used.</p> 
<p>&nbsp;</p> 
<p><img loading="lazy" src="https://static.hsstatic.net/BlogImporterAssetsUI/ex/missing-image.png" border="0" alt="01" title="01" style="margin-left: auto; margin-right: auto; display: block;"></p> 
<p style="text-align: center;">Figure 1 - China Chopper ASPX Script</p> 
<p style="text-align: left;">&nbsp;</p> 
<p>This script is essentially a page where when an HTTP POST request is made to the page, and the script will call the JScript “eval” function to execute the string inside a given POST request variable. In the above script, the POST request variable is named “secret”, meaning any JScript contained in the “secret” variable will be executed on the server.</p> 
<p>JScript is implemented as an active scripting engine allowing the language to use ActiveX objects on the client it is running on. This can be and is abused by attackers to achieve reverse shells, file management, process execution, and much more.</p> 
<p>After setting up a test IIS server and placing the web shell on the server, we can now test our own payloads against it. To do this, we used Python to send HTTP POST requests to the China Chopper page and put our malicious JScript in an HTTP POST “secret” variable.</p> 
<p>Here is our example payload, which starts a command prompt and pings itself. This demonstrates the possibility of process execution.</p> 
<p>&nbsp;</p> 
<p><img loading="lazy" src="https://levelblue.com/hubfs/Imported_Blog_Media/6a0133f264aa62970b0263e995d573200b-800wi" border="0" alt="02" title="02" style="margin-left: auto; margin-right: auto; display: block;"></p> 
<p style="text-align: center;">Figure 2 - JScript Payload 01</p> 
<h3>&nbsp;</h3> 
<h3>Attackers Viewpoint</h3> 
<p>For the attacker, typically a client component of the China Chopper web shell is used on the attacker’s systems. This client is a C binary file.</p> 
<p>This client allows the attacker to perform many nefarious tasks such as downloading and uploading files, running a virtual terminal to execute anything you normally could using cmd.exe, modifying file times, executing custom JScript, file browsing, and more. All this is made available just from the one line of code running on the server.</p> 
<p><img loading="lazy" src="https://static.hsstatic.net/BlogImporterAssetsUI/ex/missing-image.png" border="0" alt="03" title="03" style="margin-left: auto; margin-right: auto; display: block;"></p> 
<p style="text-align: center;">Figure 3 - Custom Script Execution</p> 
<p>&nbsp;</p> 
<p><img loading="lazy" src="https://levelblue.com/hubfs/Imported_Blog_Media/6a0133f264aa62970b0263e995d5ad200b-800wi" border="0" alt="04" title="04" style="margin-left: auto; margin-right: auto; display: block;"></p> 
<p style="text-align: center;">Figure 4 - Virtual Terminal</p> 
<p>&nbsp;</p> 
<p><img loading="lazy" src="https://static.hsstatic.net/BlogImporterAssetsUI/ex/missing-image.png" border="0" alt="05" title="05"></p> 
<p style="text-align: center;">Figure 5 - File Manager</p> 
<p style="text-align: left;">&nbsp;</p> 
<p>To see exactly what the client is sending to the web shell, we captured the HTTP request for executing the following custom JScript:</p> 
<p style="text-align: center;"><strong><em>Response.Write(“Hello World”);</em></strong></p> 
<p>This script was expected to be sent as an HTTP POST request from the client to the server, with the custom JScript to be sent in the “secret” POST field. The following code is the request which was sent:</p> 
<p><img loading="lazy" src="https://static.hsstatic.net/BlogImporterAssetsUI/ex/missing-image.png" border="0" alt="06" title="06" style="margin-left: auto; margin-right: auto; display: block;"></p> 
<p style="text-align: center;">Figure 6 - Custom JScript Response</p> 
<p style="text-align: left;">&nbsp;</p> 
<p>We can see that the client encodes the custom JScript in Base64 and uses the markers <strong><em>-&gt;|</em></strong> and <strong><em>&lt;-|</em></strong> to help the client identify the portion of the response relating to the web shell.</p> 
<p>Executing a more complex command such as the virtual terminal and running “ipconfig” yields the same result; however, the base64 encoded command is automatically generated from the client and decodes to the following code:</p> 
<p><img loading="lazy" src="https://static.hsstatic.net/BlogImporterAssetsUI/ex/missing-image.png" border="0" alt="07" title="07" style="margin-left: auto; margin-right: auto; display: block;"></p> 
<p style="text-align: center;">Figure 7 - Virtual Terminal ipconfig Response</p> 
<p>&nbsp;</p> 
<p>This request introduces two new POST variables containing Base64 encoded strings:</p> 
<p><img loading="lazy" src="https://static.hsstatic.net/BlogImporterAssetsUI/ex/missing-image.png" border="0" alt="08" title="08" style="margin-left: auto; margin-right: auto; display: block;"></p> 
<p><span style="font-size: 1rem; letter-spacing: 0px;">According to this code, the Virtual Terminal feature will start the CMD process silently and execute the command sent from the client. The output is then captured and sent back to the client.</span></p> 
<p>&nbsp;</p> 
<h3>ASP.NET Runtime and .NET DLLs</h3> 
<p>Some of the artifacts found on the compromised IIS servers were DLLs.&nbsp; When an ASPX script is seen by the ASP.NET runtime for the first time, the ASPX script is parsed and transformed into a C# or VB.NET class file. This class file is then either compiled into its own .NET assembly or, depending on the IIS settings, combined with other converted ASPX scripts to form one larger .NET assembly. This .NET assembly is what is served to an end-user rather than the ASPX script itself. These .NET DLLs are stored in a temporary location along with an XML file specifically crafted for that .NET DLL called a preservation file.</p> 
<p><a href="https://npercoco.typepad.com/.a/6a0133f264aa62970b0263e995d699200b-pi"><img loading="lazy" src="https://static.hsstatic.net/BlogImporterAssetsUI/ex/missing-image.png" border="0" alt="09" title="09" style="margin-left: auto; margin-right: auto; display: block;"></a></p> 
<p style="text-align: center;">Figure 8 - ASP.NET Runtime Flow</p> 
<p style="text-align: left;">&nbsp;</p> 
<p>For the China Chopper ASPX file, a .NET Assembly was compiled along with a preservation file and stored in a temporary directory for compiled ASPX files. For our IIS server, the locations were:&nbsp;</p> 
<p style="text-align: left;"><a href="https://npercoco.typepad.com/.a/6a0133f264aa62970b0278801b116e200d-pi" class="asset-img-link"></a><a href="https://npercoco.typepad.com/.a/6a0133f264aa62970b0278801b116e200d-pi"></a><img loading="lazy" src="https://static.hsstatic.net/BlogImporterAssetsUI/ex/missing-image.png" border="0" alt="10" title="10"><a href="https://npercoco.typepad.com/.a/6a0133f264aa62970b0278801b116e200d-pi" class="asset-img-link"></a></p> 
<p>&nbsp;</p> 
<p>The suspicious looking random strings are just hashes of the file names and paths for internal use with the ASP.NET runtime.</p> 
<p>Opening the App_Web_bvbfecjk.dll file in DNSpy, we can see several methods inside of the DLL. Only one of these methods contains the C# .NET version of the China Chopper ASPX script, and the other methods are boilerplate code for the ASP.NET Runtime to execute before getting to the compiled ASPX script.</p> 
<p><img loading="lazy" src="https://levelblue.com/hubfs/Imported_Blog_Media/6a0133f264aa62970b0263e995d6bd200b-800wi" border="0" alt="11" title="11" style="margin-left: auto; margin-right: auto; display: block;"></p> 
<p style="text-align: center;">Figure 9 - C# Converted ASPX Script</p> 
<p style="text-align: left;">&nbsp;</p> 
<p>In this method, we can see a strong resemblance to the ASPX China Chopper web shell in a compiled C# .NET assembly format. This is what is served to a user when the page is requested.</p> 
<p>Something interesting to note here is the JScript stack frame. We can see that two local variables have been pushed onto the JScript stack. These variables are “<strong><em>__w</em></strong>” and “<strong><em>parameterContainer</em></strong>”. In theory, in the JScript we send to be executed, we should be able to access these variables. Let's try it out.</p> 
<p><img loading="lazy" src="https://static.hsstatic.net/BlogImporterAssetsUI/ex/missing-image.png" border="0" alt="12" title="12" style="margin-left: auto; margin-right: auto; display: block;"></p> 
<p style="text-align: center;">Figure 10 - JScript Payload 02</p> 
<p style="text-align: left;">&nbsp;</p> 
<p>We can see that the JScript was executed successfully and made use of the “<strong><em>__w</em></strong>” variable pushed on the stack to render the text “Hello World” in bold on the page. This can be useful for the attacker to render the output of their script.</p> 
<p>The preserve file for the generated assembly is an XML file and has a “<strong><em>.compiled</em></strong>” extension. The content in this file is not very interesting. It primarily contains metadata for the ASP.NET Runtime to use for identifying if the origin ASPX file needs recompiling as well as to know which assembly file to serve a page request for.</p> 
<p><img loading="lazy" src="https://static.hsstatic.net/BlogImporterAssetsUI/ex/missing-image.png" border="0" alt="13" title="13"></p> 
<p style="text-align: center;">Figure 11 - Preserve File</p> 
<p style="text-align: center;">&nbsp;</p> 
<p>To wrap up, when examining servers for signs of compromise, in addition to ASPX scripts, be aware also of the corresponding DLLs generated by ASP.NET runtime. As we are likely to see many systems compromised with web shells in the future due to the zero-day exploits by HAFNIUM and multiple other threat actor groups, it does not hurt to know a little more about how these ASPX web shell scripts work behind the scenes with the ASP.NET runtime and what the attack looks like from the attacker’s perspective.</p>
