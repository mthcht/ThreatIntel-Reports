          <article class="blog-post">
            
            <div class="blog-post__tags">
              Tags

            <div class="blog-post__body">
              <span id="hs_cos_wrapper_post_body" class="hs_cos_wrapper hs_cos_wrapper_meta_field hs_cos_wrapper_type_rich_text" style="" data-hs-cos-general-type="meta_field" data-hs-cos-type="rich_text"><p><em>Original research by Siddarth Sharma</em></p>
<p>&nbsp;</p>
<p>The Uptycs <a href="/products/use-cases/threat-hunting" rel="noopener" target="_blank">Threat Research</a> Team recently observed Golang-based worm dropping cryptominer binaries which use the MSR (Model Specific Register) driver to disable hardware prefetchers and increase the speed of the mining process by 15%.<!--more--></p>
<p>&nbsp;</p>
<p>The Golang-based worm which targets vulnerable *nix servers exploit known vulnerabilities in the popular web servers in order to spread itself and the embedded miner. The new variants of the worm were identified in June 2021 by our threat intelligence systems. Though some of the functionalities were similar to the malware discussed by the security firm <a href="https://intezer.com/blog/research/new-golang-worm-drops-xmrig-miner-on-servers/" rel="noopener" target="_blank">Intezer</a> last year, the newer variants of this malware had a bunch of activities up its sleeve.</p>
<p>&nbsp;</p>
<p>In this blog, we will detail the usage of MSR to disable the hardware prefetcher in the cryptomining malwares. We will also cover certain new techniques employed by the attackers in the attack kill chain for the persistence and dropping of the worm into certain sensitive directories on the vulnerable servers.</p>
<p>&nbsp;</p>
<h2>Hardware Prefetcher &amp; the MSR</h2>
<p>Hardware prefetcher is a technique in which the processors prefetch data based on the past access behaviour by the core&gt;, by using hardware prefetcher, stores instructions from the main memory into the L2 cache. However, on multicore processors, the use of aggressive hardware prefetching causes hampering and results in overall degradation of system performance.</p>
<p>&nbsp;</p>
<p>MSR registers in processor architecture are used to toggle certain CPU features and computer performance monitoring. By manipulating the MSR registers, hardware prefetchers can be disabled.</p>
<p>&nbsp;</p>
<h2>Miners Using MSR to Disable Hardware Prefetcher</h2>
<p>A miner running with root privileges can disable the prefetcher. This is done to boost the miner execution performance, thereby increasing the speed of the mining process. We have seen Xmrig miners in our threat intelligence systems using MSR to disable the hardware prefetcher.</p>
<p>&nbsp;</p>
<p>Xmrig miners use the <a href="https://github.com/tevador/RandomX">RandomX</a> algorithm which generates multiple unique programs that are generated by data selected from the dataset generated from the hash of a key block. &gt;The code to be run inside the VM is generated randomly and the resultant hash of its outcome is used as proof of work.</p>
<p>&nbsp;</p>
<p>As RandomX programs are run in a VM, this operation is generally memory intensive. Hence, the miner disables the hardware prefetcher using the MSR. According to the documentation of <a href="https://github.com/xmrig/xmrig/blob/master/doc/CPU.md#wrmsr">Xmrig</a>, disabling the hardware prefetcher increases the speed upto 15%.</p>
<p>&nbsp;</p>
<p>The miner uses the modprobe msr command to load the msr driver (see Figure 1).</p>
<p style="text-align: center;"><a href="https://f.hubspotusercontent00.net/hubfs/2617658/fig-1.png" rel="noopener" target="_blank"><img src="https://www.uptycs.com/hs-fs/hubfs/fig-1.png?width=750&amp;name=fig-1.png" alt="fig-1" width="750" loading="lazy" style="width: 750px; margin-left: auto; margin-right: auto; display: block;" srcset="https://www.uptycs.com/hs-fs/hubfs/fig-1.png?width=375&amp;name=fig-1.png 375w, https://www.uptycs.com/hs-fs/hubfs/fig-1.png?width=750&amp;name=fig-1.png 750w, https://www.uptycs.com/hs-fs/hubfs/fig-1.png?width=1125&amp;name=fig-1.png 1125w, https://www.uptycs.com/hs-fs/hubfs/fig-1.png?width=1500&amp;name=fig-1.png 1500w, https://www.uptycs.com/hs-fs/hubfs/fig-1.png?width=1875&amp;name=fig-1.png 1875w, https://www.uptycs.com/hs-fs/hubfs/fig-1.png?width=2250&amp;name=fig-1.png 2250w" sizes="(max-width: 750px) 100vw, 750px"></a><span style="font-style: italic;">Figure 1: Command used to load msr driver.</span></p>
<p>&nbsp;</p>
<p>This is done because in modular kernels the msr driver is not automatically loaded. Once the msr driver gets loaded, a pseudo file is created in /dev/cpu/ (/dev/cpu/CPUNUM/msr). This provides an interface to read and write the model-specific registers (MSRs) of an x86 CPU. The miner accesses /dev/cpu/CPUNUM/msr to modify the existing value of the msr with the new value as shown below (see Figure 2).</p>
<p style="text-align: center;"><img src="https://www.uptycs.com/hs-fs/hubfs/fig-2.png?width=700&amp;name=fig-2.png" alt="fig-2" width="700" loading="lazy" style="width: 700px; margin-left: auto; margin-right: auto; display: block;" srcset="https://www.uptycs.com/hs-fs/hubfs/fig-2.png?width=350&amp;name=fig-2.png 350w, https://www.uptycs.com/hs-fs/hubfs/fig-2.png?width=700&amp;name=fig-2.png 700w, https://www.uptycs.com/hs-fs/hubfs/fig-2.png?width=1050&amp;name=fig-2.png 1050w, https://www.uptycs.com/hs-fs/hubfs/fig-2.png?width=1400&amp;name=fig-2.png 1400w, https://www.uptycs.com/hs-fs/hubfs/fig-2.png?width=1750&amp;name=fig-2.png 1750w, https://www.uptycs.com/hs-fs/hubfs/fig-2.png?width=2100&amp;name=fig-2.png 2100w" sizes="(max-width: 700px) 100vw, 700px"><em>Figure 2: MSR file modification</em></p>
<p style="text-align: left;">&nbsp;</p>
<p>For disabling hardware prefetcher, the miner accesses the /dev/CPU/CPUNUM/msr special character file to read the old value of msr and then modifies it using pwrite system call in chunks of 8 bytes. The pseudo-code of this activity is shown below (see Figure 3).</p>
<p style="text-align: center;"><img src="https://www.uptycs.com/hs-fs/hubfs/fig-3.png?width=700&amp;name=fig-3.png" alt="fig-3" width="700" loading="lazy" style="width: 700px; margin-left: auto; margin-right: auto; display: block;" srcset="https://www.uptycs.com/hs-fs/hubfs/fig-3.png?width=350&amp;name=fig-3.png 350w, https://www.uptycs.com/hs-fs/hubfs/fig-3.png?width=700&amp;name=fig-3.png 700w, https://www.uptycs.com/hs-fs/hubfs/fig-3.png?width=1050&amp;name=fig-3.png 1050w, https://www.uptycs.com/hs-fs/hubfs/fig-3.png?width=1400&amp;name=fig-3.png 1400w, https://www.uptycs.com/hs-fs/hubfs/fig-3.png?width=1750&amp;name=fig-3.png 1750w, https://www.uptycs.com/hs-fs/hubfs/fig-3.png?width=2100&amp;name=fig-3.png 2100w" sizes="(max-width: 700px) 100vw, 700px"><em>Figure 3: Pseudo-code</em></p>

<h2>Wormed Cyptominer: Attack Kill Chain</h2>
<ol>
<li>The attack kill chain of the wormed cryptominer starts with a Shell script which downloads the Golang worm using curl utility.</li>
<li>The worm scans and exploits existing server based vulnerabilities like <a href="https://nvd.nist.gov/vuln/detail/CVE-2020-14882">CVE-2020-14882</a> and <a href="https://cve.mitre.org/cgi-bin/cvename.cgi?name=CVE-2017-11610#:~:text=The%20XML%2DRPC%20server%20in,1%2C%203.1.&amp;text=3%20allows%20remote%20authenticated%20users,to%20nested%20supervisord%20namespace%20lookups.">CVE-2017-11610</a>from the victim machine.</li>
<li>After having access to a vulnerable server, the worm downloads another shell script which downloads a copy of the same Golang worm.</li>
<li>The worm also writes multiple copies of itself to various sensitive directories like /boot,/efi,/grub and later drops Xmrig miner ELF in /tmp location.</li>
<li>The miner disables the hardware prefetcher by using MSR to boost the mining process.</li>
</ol>
<p>The shell-script we analysed (hash: 28e9b06e5a4606c9d806092a8ad78ce2ea7aa1077a08bcf3ec1d8e3d19714f08) involved several defense evasive techniques like firewall altering, disabling monitoring agents which we have detailed in our previous <a href="https://www.uptycs.com/blog/evasive-techniques-used-by-malicious-linux-shell-scripts">blog</a>a&gt;. Alongside this, the script also used the ‘sed -i’ command to modify the /etc/hosts file with the nanopool URL as shown in the below figure (see Figure 5).</p>

<p>With the rise and sky-high valuation of Bitcoin and several other cryptocurrencies, cryptomining-based attacks have continued to dominate the threat landscape. Wormed cyptominer attacks have a greater threshold as they write multiple copies and also spread across endpoints in a corporate network.</p>
<p>&nbsp;</p>
<p>&gt;Alongside the mining process, modification of the MSR registers can lead to fatal performance issues of the corporate resources. The Uptycs EDR solution offers the added benefit of taking a deep dive into the events logged, providing more insights of an attack.</p>
