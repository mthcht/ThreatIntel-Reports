                  <div class="c-page-title__meta">Analysis Report</div>
                <h1 class="c-page-title__title">
<span>Federal Agency Compromised by Malicious Cyber Actor</span>
</h1>
                                                          <div class="c-page-title__fields">  




<div  class="c-field c-field--name-field-last-updated c-field--type-datetime c-field--label-above">
  <div  class="c-field__label">Last Revised</div><div class="c-field__content"><time datetime="2020-10-24T12:00:00Z">October 24, 2020</time></div></div>

  




<div  class="c-field c-field--name-field-alert-code c-field--type-string c-field--label-above">
  <div  class="c-field__label">Alert Code</div><div class="c-field__content">AR20-268A</div></div>

</div>
                        
        
      </div>
          </div>
    <div class="c-page-title__decoration"></div>
  </div>
</div>
    </div>
    <div class="l-full__main">
                      

<div  class="l-page-section l-page-section--rich-text">
      <div class="l-constrain">
  
  
  <div class="l-page-section__content">
          <p>CISA became aware—via EINSTEIN, CISA’s intrusion detection system that monitors federal civilian networks—of a potential compromise of a federal agency’s network. In coordination with the affected agency, CISA conducted an incident response engagement, confirming malicious activity. The following information is derived exclusively from the incident response engagement and provides the threat actor’s tactics, techniques, and procedures as well as indicators of compromise that CISA observed as part of the engagement.</p>
<h4>Threat Actor Activity</h4>
<p>The cyber threat actor had valid access credentials for multiple users’ Microsoft Office 365 (O365) accounts and domain administrator accounts, which they leveraged for <em>Initial Access</em> [<a href="https://attack.mitre.org/versions/v7/tactics/TA0001/">TA0001</a>] to the agency's network (<em>Valid Accounts</em> [<a href="https://attack.mitre.org/versions/v7/techniques/T1078/">T1078</a>]). First the threat actor logged into a user’s O365 account from Internet Protocol (IP) address <code>91.219.236[.]166</code> and then browsed pages on a SharePoint site and downloaded a file (<em>Data from Information Repositories: SharePoint</em> [<a href="https://attack.mitre.org/versions/v7/techniques/T1213/002/">T1213.002</a>]). The cyber threat actor connected multiple times by Transmission Control Protocol (TCP) from IP address <code>185.86.151[.]223</code> to the victim organization’s virtual private network (VPN) server (<em>Exploit Public-Facing Application</em> [<a href="https://attack.mitre.org/versions/v7/techniques/T1190/">T1190</a>]).</p>
<p>CISA analysts were not able to determine how the cyber threat actor initially obtained the credentials. It is possible the cyber actor obtained the credentials from an unpatched agency VPN server by exploiting a known vulnerability—CVE-2019-11510—in Pulse Secure (<em>Exploitation for Credential Access</em> [<a href="https://attack.mitre.org/versions/v7/techniques/T1212/">T1212</a>]). In April 2019, Pulse Secure released patches for several critical vulnerabilities—including CVE-2019-11510, which allows the remote, unauthenticated retrieval of files, including passwords.[<a href="https://kb.pulsesecure.net/articles/Pulse_Security_Advisories/SA44101">1</a>] CISA has observed wide exploitation of CVE-2019-11510 across the federal government.[<a href="https://www.us-cert.gov/ncas/alerts/aa20-010a">2</a>]</p>
<p>After initial access, the threat actor performed <em>Discovery</em> [<a href="https://attack.mitre.org/versions/v7/tactics/TA0007/">TA0007</a>] by logging into an agency O365 email account from <code>91.219.236[.]166</code> and viewing and downloading help desk email attachments with “Intranet access” and “VPN passwords” in the subject line, despite already having privileged access <em>(Email Collection</em> [<a href="https://attack.mitre.org/versions/v7/techniques/T1114/">T1114</a>], <em>Unsecured Credentials: Credentials In Files</em> [<a href="https://attack.mitre.org/versions/v7/techniques/T1552/001/">T1552.001</a>]). (Note: these emails did not contain any passwords.) The actor logged into the same email account via Remote Desktop Protocol (RDP) from IP address <code>207.220.1[.]3</code> (<em>External Remote Services</em> [<a href="https://attack.mitre.org/versions/v7/techniques/T1133/">T1133</a>]). The actor enumerated the Active Directory and Group Policy key and changed a registry key for the Group Policy (<em>Account Manipulation</em> [<a href="https://attack.mitre.org/versions/v7/techniques/T1098/">T1098</a>]). Immediately afterward, the threat actor used common Microsoft Windows command line processes—<code>conhost</code>, <code>ipconfig</code>, <code>net</code>, <code>query</code>, <code>netstat</code>, <code>ping</code>, and <code>whoami</code>, <code>plink.exe</code>—to enumerate the compromised system and network (<em>Command and Scripting Interpreter</em> [<a href="https://attack.mitre.org/versions/v7/techniques/T1059/">T1059</a>], <em>System Network Configuration Discovery</em> [<a href="https://attack.mitre.org/versions/v7/techniques/T1016/">T1016</a>]).</p>
<p>The cyber threat actor then attempted multiple times to connect to virtual private server (VPS) IP <code>185.86.151[.]223</code> through a Windows Server Message Block (SMB) client. Although they connected and disconnected multiple times, the connections were ultimately successful. During the same period, the actor used an alias secure identifier account they had previously created to log into VPS <code>185.86.151[.]223</code> via an SMB share. The attacker then executed <code>plink.exe</code> on a victim file server (<em>Command and Scripting Interpreter</em> [<a href="https://attack.mitre.org/versions/v7/techniques/T1059/">T1059</a>]). (<code>plink.exe</code> is a command-line version of PuTTy that is used for remote administration.)</p>
<p>The cyber threat actor established <em>Persistence</em> [<a href="https://attack.mitre.org/versions/v7/tactics/TA0003/">TA0003</a>] and <em>Command and Control</em> [<a href="https://attack.mitre.org/versions/v7/tactics/TA0011/">TA0011</a>] on the victim network by (1) creating a persistent Secure Socket Shell (SSH) tunnel/reverse SOCKS proxy, (2) running <code>inetinfo.exe</code> (a unique, multi-stage malware used to drop files), and (3) setting up a locally mounted remote share on IP address <code>78.27.70[.]237</code> (<em>Proxy</em> [<a href="https://attack.mitre.org/versions/v7/techniques/T1090/">T1090</a>]). The mounted file share allowed the actor to freely move during its operations while leaving fewer artifacts for forensic analysis. Refer to Threat Actor Malware section for more information about the SSH Tunnel/reverse SOCKS proxy and <code>inetinfo.exe</code>.</p>
<p>The cyber threat actor created a local account, which they used for data <em>Collection</em> [<a href="https://attack.mitre.org/versions/v7/tactics/TA0009/">TA0009</a>], <em>Exfiltration</em> [<a href="https://attack.mitre.org/versions/v7/tactics/TA0010/">TA0010</a>], <em>Persistence</em> [<a href="https://attack.mitre.org/versions/v7/tactics/TA0003/">TA0003</a>], and <em>Command and Control</em> [<a href="https://attack.mitre.org/versions/v7/tactics/TA0011/">TA0011</a>] (<em>Create Account</em> [<a href="https://attack.mitre.org/versions/v7/techniques/T1136/">T1136</a>]). The cyber threat actor used the local account to:</p>
<ul>
<li>Browse directories on a victim file server (<em>Data from Shared Network Drive</em> [<a href="https://attack.mitre.org/versions/v7/techniques/T1039/">T1039</a>]).</li>
<li>Copy a file from a user’s home directory to their locally mounted remote share (<em>Data Staged</em> [<a href="https://attack.mitre.org/versions/v7/techniques/T1074/">T1074</a>]).
<ul>
<li>CISA analysts detected the cyber threat actor interacting with other files on users’ home directories but could not confirm whether they were exfiltrated.</li>
</ul>
</li>
<li>Create a reverse SMB SOCKS proxy that allowed connection between an cyber threat actor-controlled VPS and the victim organization’s file server (refer to Threat Actor Malware section for more information) (<em>Proxy</em> [<a href="https://attack.mitre.org/versions/v7/techniques/T1090/">T1090</a>]).</li>
<li>Interact with PowerShell module <code>Invoke-TmpDavFS.psm</code> (refer to Threat Actor Malware section for more information).</li>
<li>Exfiltrate data from an account directory and file server directory using <code>tsclient</code> (<code>tsclient</code> is a Microsoft Windows Terminal Services client) (<em>Data from Local System</em> [<a href="https://attack.mitre.org/versions/v7/techniques/T1005/">T1005</a>], <em>Data from Network Shared Drive</em> [<a href="https://attack.mitre.org/versions/v7/techniques/T1039/">T1039</a>]).</li>
<li>Create two compressed Zip files with several files and directories on them (<em>Archive Collected Data</em> [<a href="https://attack.mitre.org/versions/v7/techniques/T1560/">T1560</a>]); it is likely that the cyber threat actor exfiltrated these Zip files, but this cannot be confirmed because the actor masked their activity.</li>
</ul>
<p>See figure 1 for the sequence of the cyber threat actor’s tactics and techniques.</p>
<p><img>​</p>
<p><em>Figure 1: Cyber threat actor tactics and techniques</em></p>
<h4>Threat Actor Malware</h4>
<h5>Persistent SSH Tunnel/Reverse SOCKS Proxy</h5>
<p>While logged in as “Administrator,” the cyber threat actor created two Scheduled Tasks (see table 1) that worked in concert to establish a persistent SSH tunnel and reverse SOCKS proxy. The proxy allowed connections between an attacker-controlled remote server and one of the victim organization’s file servers (<em>Scheduled Task/Job</em> [<a href="https://attack.mitre.org/versions/v7/techniques/T1053/">T1053</a>], <em>Proxy</em> [<a href="https://attack.mitre.org/versions/v7/techniques/T1090/">T1090</a>]). The Reverse SOCKS Proxy communicated through port 8100 (<em>Non-Standard Port</em> [<a href="https://attack.mitre.org/versions/v7/techniques/T1571/">T1571</a>]). This port is normally closed, but the attacker’s malware opened it.</p>
<p><em>Table 1: Scheduled Tasks composing SSH tunnel and reverse SOCKS proxy</em></p>
<table class="tablesaw tablesaw-stack" data-tablesaw-mode="stack" data-tablesaw-minimap>
<thead>
<tr>
<th scope="col" role="columnheader" data-tablesaw-priority="persist">Scheduled Task</th>
<th scope="col" role="columnheader">Description</th>
</tr>
</thead>
<tbody>
<tr>
<td><code>ShellExperienceHost.exe</code></td>
<td>
<p>This task created a persistent SSH tunnel to attacker-controlled remote server <code>206.189.18[.]189</code> and employed port forwarding to allow connections from the remote server port 39999 to the victim file server through port 8100. This task was run daily.</p>
<p><code>ShellExperienceHost.exe</code> is a version of <code>plink.exe</code>, a command-line version of PuTTy that is used for remote administration.</p>
</td>
</tr>
<tr>
<td><code>WinDiag.exe</code></td>
<td>
<p>This task is a reverse SOCKS proxy that is preconfigured to bind to and listen on TCP port 8100. <code>WinDiag.exe </code>received responses through the SSH tunnel and forwarded the responses through port 8100 to the VPS IP address <code>185.193.127[.]17</code> over port 443. This task was run on boot.</p>
<p><code>WinDiag.exe</code> had compile information that matched the VPS login name</p>
</td>
</tr>
</tbody>
</table>
<h4>Dropper Malware: inetinfo.exe</h4>
<p>The threat actor created a Scheduled Task to run inetinfo.exe (<em>Scheduled Task/Job</em> [<a href="https://attack.mitre.org/versions/v7/techniques/T1053/">T1053</a>]). <code>inetinfo.exe</code> is a unique, multi-stage malware used to drop files (figure 2). It dropped <code>system.dll</code> and <code>363691858</code> files and a second instance of <code>inetinfo.exe</code>. The <code>system.dll</code> from the second instance of <code>inetinfo.exe</code> decrypted <code>363691858</code> as binary from the first instance of <code>inetinfo.exe</code>. The decrypted <code>363691858</code> binary was injected into the second instance of <code>inetinfo.exe</code> to create and connect to a locally named tunnel. The injected binary then executed shellcode in memory that connected to IP address <code>185.142.236[.]198</code>, which resulted in download and execution of a payload.</p>
<p><img>​</p>
<p><em>Figure 2: Dropper malware </em><code>inetinfo.exe</code></p>
<p>The cyber threat actor was able to overcome the agency’s anti-malware protection, and <code>inetinfo.exe</code> escaped quarantine. CISA analysts determined that the cyber threat actor accessed the anti-malware product’s software license key and installation guide and then visited a directory used by the product for temporary file analysis. After accessing this directory, the cyber threat actor was able to run <code>inetinfo.exe</code> (<em>Impair Defenses: Disable or Modify Tools</em> [<a href="https://attack.mitre.org/versions/v7/techniques/T1562/001/">T1562.001</a>]).</p>
<h4>Reverse SMB SOCKS Proxy</h4>
<p>PowerShell script <code>HardwareEnumeration.ps1</code> created a reverse SMB SOCKS proxy that allowed connection between attacker-controlled VPS IP <code>185.193.127[.]18</code> and the victim organization’s file server over port 443 (<em>Command and Scripting Interpreter: Power Shell</em> [<a href="https://attack.mitre.org/versions/v7/techniques/T1059/001/">T1059.001</a>], <em>Proxy</em> [<a href="https://attack.mitre.org/versions/v7/techniques/T1090/">T1090</a>]). PowerShell script <code>HardwareEnumeration.ps1</code> was executed daily via a Scheduled Task (<em>Scheduled Task/Job</em> [<a href="https://attack.mitre.org/versions/v7/techniques/T1053/">T1053</a>]).</p>
<p><code>HardwareEnumeration.ps1</code> is a copy of <code>Invoke-SocksProxy.ps1</code>, a free tool created and distributed by a security researcher on GitHub.[<a href="https://github.com/p3nt4/Invoke-SocksProxy">3</a>] <code>Invoke-SocksProxy.ps1</code> creates a reverse proxy from the local machine to attacker infrastructure through SMB TCP port 445 (<em>Non-Standard Port</em> [<a href="https://attack.mitre.org/versions/v7/techniques/T1571/">T1571</a>]). The script was likely altered with the cyber threat actor’s configuration needs.</p>
<h4>PowerShell Module: <code>invoke-TmpDavFS.psm</code></h4>
<p><code>invoke-TmpDavFS.psm</code> is a PowerShell module that creates a Web Distributed Authoring and Versioning (WebDAV) server that can be mounted as a file system and communicates over TCP port 443 and TCP port 80. <code>invoke-TmpDavFS.psm</code> is distributed on GitHub.[<a href="https://github.com/p3nt4/Invoke-TmpDavFS">4</a>]</p>
<div>
<h3>Summary</h3>
</div>
<p><em>This Analysis Report uses the MITRE Adversarial Tactics, Techniques, and Common Knowledge (ATT&amp;CK®) framework. See the <a href="https://attack.mitre.org/versions/v7/matrices/enterprise/">ATT&amp;CK for Enterprise</a> framework for all referenced threat actor tactics and techniques.</em></p>
<p>The Cybersecurity and Infrastructure Security Agency (CISA) responded to a recent threat actor’s cyberattack on a federal agency’s enterprise network. By leveraging compromised credentials, the cyber threat actor implanted sophisticated malware—including multi-stage malware that evaded the affected agency’s anti-malware protection—and gained persistent access through two reverse Socket Secure (SOCKS) proxies that exploited weaknesses in the agency’s firewall.</p>
<p>For a downloadable copy of IOCs, see: <a href="https://us-cert.cisa.gov/sites/default/files/publications/AR20-268A.stix.xml">AA20-268A.stix</a>.</p>
<div>
<h3>Solution</h3>
</div>
<h4>Indicators of Compromise</h4>
<p>CISA analysts identified several IP addresses involved in the multiple stages of the outlined attack.</p>
<ul>
<li><code>185.86.151[.]223</code> – Command and Control (C2)</li>
<li><code>91.219.236[.]166</code> – C2</li>
<li><code>207.220.1[.]3</code> – C2</li>
<li><code>78.27.70[.]237</code> – Data Exfiltration</li>
<li><code>185.193.127[.]18</code> – Persistence</li>
</ul>
<h4>Monitor Network Traffic for Unusual Activity</h4>
<p>CISA recommends organizations monitor network traffic for the following unusual activity.</p>
<ul>
<li>Unusual open ports (e.g., port 8100)</li>
<li>Large outbound files</li>
<li>Unexpected and unapproved protocols, especially outbound to the internet (e.g., SSH, SMB, RDP)</li>
</ul>
<p>If network defenders note any of the above activity, they should investigate.</p>
<h3><strong>Prevention</strong></h3>
<p>CISA recommends organizations implement the following recommendations to protect against activity identified in this report.</p>
<h4>Deploy an Enterprise Firewall</h4>
<p>Organizations should deploy an enterprise firewall to control what is allowed in and out of their network.</p>
<p>If the organization chooses not to deploy an enterprise firewall, they should work with their internet service provider to ensure the firewall is configured properly.</p>
<h4>Block Unused Ports</h4>
<p>Organizations should conduct a survey of the traffic in and out of their enterprise to determine the ports needed for organizational functions. They should then configure their firewall to block unnecessary ports. Organization should develop a change control process to make control changes to those rules. Of special note, unused SMB, SSH, and FTP ports should be blocked.</p>
<h4>Additional Recommendations</h4>
<p>CISA recommends organizations implement the following best practices.</p>
<ul>
<li>Implement multi-factor authentication, especially for privileged accounts.</li>
<li>Use separate administrative accounts on separate administration workstations.</li>
<li>Implement the principle of least privilege on data access.</li>
<li>Secure RDP and other remote access solutions using multifactor authentication and “jump boxes” for access.</li>
<li>Deploy and maintain endpoint defense tools on all endpoints.</li>
<li>Keep software up to date.</li>
</ul>
