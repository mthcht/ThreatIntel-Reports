							<h1 class="entry-title">Linux Restricted Shell Bypass</h1>

						<p class="post-meta"> by <span class="author vcard"><a href="https://vk9-sec.com/author/vry4n_mail/" title="Posts by Vry4n_" rel="author">Vry4n_</a></span> | <span class="published">Feb 26, 2021</span> | <a href="https://vk9-sec.com/category/red-team/post-exploitation/linux-post-exploitation/" rel="category tag">Linux Post-Exploitation</a> | <span class="comments-number"><a href="https://vk9-sec.com/linux-restricted-shell-bypass/#respond">0 comments</a></span></p>
												</div>
				
					<div class="entry-content">
					<p>Restricted shells are conceptually shells with restricted permissions, with features and commands working under a very peculiar environment, built to keep users in a secure and controlled environment, allowing them just the minimum necessary to perform their daily operations.</p>
<p>Once hackers get a low privileged shell, even a restricted one, it’s time to try to escape normal restrictions and get more features and privileges to play with. This is where restricted shell escaping techniques come into play. Escaping shell restrictions is just a small part of Penetration Testing Post Exploitation phase, designed to escalate privileges.</p>
<p>Sometimes a restricted shell can block the commands with / or the redirecting outputs like &gt;,&gt;&gt;</p>
<h3><strong>Common Restricted Shells</strong></h3>
<p>There is a lot of different restricted shells to choose from. Some of them are just normal shells with some simple common restrictions not actually configurable, such as rbash (restricted Bash), rzsh and rksh (Korn Shell in restricted mode), which are really trivial to bypass.</p>
<p>Others have a complete configuration set that can be redesigned to fit administrator’s needs such as lshell (Limited Shell) and rssh (Restricted Secure Shell).</p>
<h3><strong>Gathering Environment Information</strong></h3>
<p>Once we have access to a restricted shell, before we can go any further on all techniques, the first step is to gather as much information as possible about our current shell environment.</p>
<ul>
<li>Check available commands either by trying them out by hand, hitting TAB key twice or listing files and directories;</li>
<li>Check for commands configured with SUID permissions, especially if they are owned by root user. If these commands have escapes, they can be run with root permissions and will be our way out, or in.</li>
<li>Check variables ‘env’ or ‘printenv’</li>
<li>Check the list of commands you can use with sudo. This will let us execute commands with other user’s permissions by using our own password. This is especially good when configured for commands with escape features. (sudo -l)</li>
<li>Check what languages are at your disposal, such as python, expect, perl, ruby, etc. They will come in handy later on;</li>
<li>Check if redirect operators are available, such as &#8216;|&#8217; (pipe), “&gt;”, “&gt;&gt;”, “&lt;”;</li>
<li>Check for escape characters and execution tags such as: “;” (colon), “&amp;” (background support), “’” (single quotes), “” (double-quotes), “$(“ (shell execution tag), “${“</li>
<li>You must to check in what shell you are : echo $SHELL you will be in rbash by 90%</li>
</ul>
<p>Try to determine what kind of shell you are in. This is not easy depending on the configuration in place, but can be performed by issuing some commands and checking for general error messages.</p>
<ul>
<li>If some available command is unknown to you, install them in your own test Linux box and analyze its features, manual, etc.</li>
<li>Try to determine what kind of shell you are in. This is not easy depending on the configuration in place, but can be performed by issuing some commands and checking for general error messages.</li>
</ul>

<h2><strong>Best Practices &amp; Conclusion</strong></h2>
<ul>
<li>Prefer to work with “Allowed commands” instead of “Disallowed commands”. The amount of commands with escapes you don’t know are far superior than the ones you do.</li>
<li>Keep “Allowed Commands” list to a minimum necessary.</li>
<li>Inspect your allowed commands for escaping features on a regular basis, either by studying the manual or search in the security community.</li>
<li>Check allowed commands that could interact with Linux system variables and restrict their access.</li>
<li>Scripts that invoke other scripts can be a security risk specially when they are running with other user’s privileges and software that allow escape or third party command execution. Try to avoid this.</li>
<li>If any command allowed has escapes or command execution features, avoid using it. If not possible try to enforce restrictions to block certain functions or use restricted versions. Some commands have restricted versions with no command execution support.</li>
<li>If providing Linux editors is inevitable, use restricted versions, such as:</li>
</ul>
<p>vim = rvim (Restricted Vim)</p>
<p>ed = red (Restricted ED)</p>
<p>nano = rnano (Restricted Nano)</p>
<ul>
<li>A nice hint for restricted software would be to provide them as a symbolic link. For all purposes your user might think it’s using vim, for example, while it’s just a symbolic link to rvim.</li>
<li>If providing pagers is necessary avoid less and more, and use pages that don’t provide command execution escape like most.</li>
<li>When using any software that has built-in third party editors support that rely on $EDITOR and $VISUAL Linux variables, make these variables read-only to avoid users changing it’s content to software containing escapes.</li>
<li>Try to avoid allowing programming languages. If not possible ensure that configuration is hardened and dangerous functions such as pty(), system(), exec(), etc, are blocked. Some programming languages are easy to harden simply defining functions that are disabled, others are trickier and sometimes the only way to do it is either uninstalling certain functions or not providing the language itself.</li>
</ul>
