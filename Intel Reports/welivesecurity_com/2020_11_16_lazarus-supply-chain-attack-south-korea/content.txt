    <div class="article-tags mb-2 dark big"><p class="article-tag text-capitalize">ESET Research</p></div>    <h1 class="page-headline">Lazarus supply-chain attack in South Korea</h1>    <p class="sub-title">ESET researchers uncover a novel Lazarus supply-chain attack leveraging WIZVERA VeraPort software</p>
    
    <div class="article-authors d-flex flex-wrap"><div class="article-author d-flex"><a href="/en/our-experts/anton-cherepanov/" title="Anton Cherepanov"><picture><source srcset="https://web-assets.esetstatic.com/tn/-x45/wls/2020/03/Anton_Cherepanov.jpg" media="(max-width: 768px)" /><img class="author-image me-3"  src="https://web-assets.esetstatic.com/tn/-x45/wls/2020/03/Anton_Cherepanov.jpg" alt="Anton Cherepanov"  /></picture></a><div class="author-text"><p><a href="/en/our-experts/anton-cherepanov/" title="Anton Cherepanov"><b>Anton Cherepanov</b></a></p></div></div><div class="article-author d-flex"><a href="/en/our-experts/peter-kalnai/" title="Peter Kálnai"><picture><source srcset="https://web-assets.esetstatic.com/tn/-x45/wls/2023/07/kalnai-peter-photo.jpeg" media="(max-width: 768px)" /><img class="author-image me-3"  src="https://web-assets.esetstatic.com/tn/-x45/wls/2023/07/kalnai-peter-photo.jpeg" alt="Peter Kálnai"  /></picture></a><div class="author-text"><p><a href="/en/our-experts/peter-kalnai/" title="Peter Kálnai"><b>Peter Kálnai</b></a></p></div></div></div>

                    <div class="article-body">
    <p>ESET telemetry data recently led our researchers to discover attempts to deploy Lazarus malware via a supply-chain attack in South Korea. In order to deliver its malware, the attackers used an unusual supply-chain mechanism, abusing legitimate South Korean security software and digital certificates stolen from two different companies.</p>
<h2>Lazarus toolset</h2>
<p>The Lazarus group was first identified in Novetta&rsquo;s report <a href="https://www.operationblockbuster.comhttps://web-assets.esetstatic.com/wls/2016/02/Operation-Blockbuster-Report.pdf" target="_blank" rel="noopener">Operation Blockbuster</a> in February 2016; US-CERT and the FBI call this group <a href="https://us-cert.cisa.gov/northkorea" target="_blank" rel="noopener">HIDDEN COBRA</a>. These cybercriminals rose to prominence with the infamous case of <a href="https://www.welivesecurity.com/2014/12/08/sony-pictures-hacking-traced-thai-hotel-north-korea-denies-involvement/" target="_blank" rel="noopener">cybersabotage against Sony Pictures Entertainment</a>.</p>
<p>Some of the past attacks attributed to the Lazarus group attracted the interest of security researchers who relied on Novetta et al.&rsquo;s white papers with hundreds of pages describing the tools used in the attacks &ndash; <a href="https://www.welivesecurity.com/2017/02/16/demystifying-targeted-malware-used-polish-banks/" target="_blank" rel="noopener">the Polish and Mexican banks</a>, <a href="https://www.malwaretech.com/2017/05/how-to-accidentally-stop-a-global-cyber-attacks.html">the WannaCryptor outbreak</a>, <a href="https://researchcenter.paloaltonetworks.com/2017/08/unit42-blockbuster-saga-continues/" target="_blank" rel="noopener">phishing campaigns against US defense contractors</a>, <a href="https://www.welivesecurity.com/2018/04/03/lazarus-killdisk-central-american-casino/" target="_blank" rel="noopener">Lazarus KillDisk attack against Central American casino</a>, etc. &ndash; and provides grounds for the attribution of these attacks to the Lazarus group.</p>
<p>Note that the Lazarus toolset (i.e., the collection of all files that are considered by the security industry as fingerprints of the group&rsquo;s activity) is extremely broad, and we believe there are numerous subgroups. Unlike toolsets used by some other cybercriminal groups, none of the source code of any Lazarus tools has ever been disclosed in a public leak.</p>
<h2>Latest Lazarus supply-chain attack</h2>
<p>To understand this novel supply-chain attack, you should be aware that South Korean internet users are often asked to install additional security software when visiting government or internet banking websites.</p>
<p>WIZVERA VeraPort, referred to as an integration installation program, is a South Korean application that helps manage such additional security software. With WIZVERA VeraPort installed on their devices, users receive and install all necessarily software required by a specific website with VeraPort (e.g., browser plug-ins, security software, identity verification software, etc.). Minimal user interaction is required to start such a software installation from a website that supports WIZVERA VeraPort. Usually, this software is used by government and banking websites in South Korea. For some of these websites it is mandatory to have WIZVERA VeraPort installed for users to be able to access the sites&rsquo; services.</p>
<div class="caption center"><a href="https://web-assets.esetstatic.com/wls/2020/11/Figure-1.-A-WIZVERA-VeraPort-window-displayed-to-the-user-when-installing-additional-software.png"><img class="wp-image-145202" src="https://web-assets.esetstatic.com/wls/2020/11/Figure-1.-A-WIZVERA-VeraPort-window-displayed-to-the-user-when-installing-additional-software.png" alt="" width="700" height="422" /></a>
<p class="caption-text"><em>Figure 1. A WIZVERA VeraPort window displayed to the user when installing additional software</em></p>
</div>
<p>The Lazarus attackers abused the above-mentioned mechanism of installing security software in order to deliver Lazarus malware from a legitimate but compromised website. However, it should be noted that a successful malware deployment using this method requires a number of preconditions; that&rsquo;s why it was used in limited Lazarus campaigns. To make this attack possible:</p>
<ul>
<li>the victim must have WIZVERA VeraPort software installed</li>
<li>the victim must visit a compromised website that already has support for WIZVERA VeraPort</li>
<li>this website must have specific entries in its VeraPort configuration file that allows attackers to replace regular software in its VeraPort software bundle with their malware.</li>
</ul>
<p>It is important to note that, based on our analysis, we believe that these supply-chain attacks happen at websites that use WIZVERA VeraPort, rather than at WIZVERA itself.</p>
<p>Websites that support WIZVERA VeraPort software contain a server-side component, specifically some JavaScripts and a WIZVERA configuration file. The configuration file is base64-encoded XML containing the website address, a list of software to install, download URLs, and other parameters.</p>
<div class="caption center"><a href="https://web-assets.esetstatic.com/wls/2020/11/Figure-2.-An-example-of-a-WIZVERA-VeraPort-configuration-redacted-by-ESET.png"><img class="wp-image-145203" src="https://web-assets.esetstatic.com/wls/2020/11/Figure-2.-An-example-of-a-WIZVERA-VeraPort-configuration-redacted-by-ESET.png" alt="" width="700" height="198" /></a>
<p class="caption-text"><em>Figure 2. An example of a WIZVERA VeraPort configuration (redacted by ESET)</em></p>
</div>
<p>These configuration files are digitally signed by WIZVERA. Once downloaded, they are verified using a strong cryptographic algorithm (RSA), which is why attackers can&rsquo;t easily modify the content of these configuration files or set up their own fake website. However, the attackers can replace the software to be delivered to WIZVERA VeraPort users from a legitimate but compromised website. We believe this is the scenario the Lazarus attackers used.</p>
<div class="caption center"><a href="https://web-assets.esetstatic.com/wls/2020/11/Figure-3.-Simplified-scheme-of-the-WIZVERA-supply-chain-attack-conducted-by-the-Lazarus-group.png"><img class="wp-image-145204" src="https://web-assets.esetstatic.com/wls/2020/11/Figure-3.-Simplified-scheme-of-the-WIZVERA-supply-chain-attack-conducted-by-the-Lazarus-group.png" alt="" width="500" height="546" /></a>
<p class="caption-text"><em>Figure 3. Simplified scheme of the WIZVERA supply-chain attack conducted by the Lazarus group</em></p>

However, VeraPort only verifies that the digital signature is valid, without checking to whom it belongs. Thus, to abuse WIZVERA VeraPort, attackers must have any valid code-signing certificate in order to push their payload via this method or get lucky and find a VeraPort configuration that does not require code-signing verification.</p>

<h2>Attribution</h2>
<p>We strongly attribute this supply-chain attack to the Lazarus group, based on the following aspects:</p>
<ol>
<li>Community agreement: The current attack is a continuation of what KrCERT has called Operation Bookcodes. While KrCERT hasn&rsquo;t attributed that campaign to the Lazarus group, Kaspersky did in their report about <a href="https://securelist.com/apt-trends-report-q2-2020/97937/" target="_blank" rel="noopener">Q2 2020 APT trends</a>.</li>
<li>Toolset characteristics and detection:
<ol style="list-style-type: lower-alpha;">
<li>The initial dropper is a console application that requires parameters, executing the next stages in a cascade and utilizes an encryption, cf.&nbsp;<a href="https://www.welivesecurity.com/2017/02/16/demystifying-targeted-malware-used-polish-banks/" target="_blank" rel="noopener">the watering hole attacks against Polish and Mexican banks</a></li>

<h2>Malware analysis</h2>
<p>It is a common characteristic of many APT groups, especially Lazarus, that they unleash their arsenal within several stages that execute as a cascade: from the dropper to intermediate products (the Loader, serving as an injector) up to the final payloads (the Downloader, the Module). The same is true for this campaign.</p>
<p>During our analysis we found similarities in code and architecture between Lazarus malware delivered via this WIZVERA supply-chain attack and the malware described in the Operation BookCodes report (<a href="https://www.boho.or.kr/filedownload.do?attach_file_seq=2452&amp;attach_file_id=EpF2452.pdf" target="_blank" rel="noopener">part one</a>, <a href="https://www.boho.or.kr/filedownload.do?attach_file_seq=2455&amp;attach_file_id=EpF2455.pdf" target="_blank" rel="noopener">part two</a>) published by Korea Internet &amp; Security Agency this year.</p>

<h2>Signed initial downloader</h2>

<h3>Dropper</h3>
<p>This is the initial stage of the cascade. While one can&rsquo;t see any polymorphism or obfuscation in the code, it encapsulates three encrypted files in its resources. Moreover, it&rsquo;s a console application expecting three parameters in an encrypted state: the name of the first file (the Loader, <span style="font-family: courier new, courier, monospace;">Btserv.dll</span>), the name of the second file (the Downloader, <span style="font-family: courier new, courier, monospace;">bcyp655.tlb</span>), and the necessary decryption key for the previous values (542).</p>
<p><span style="font-family: courier new, courier, monospace;">BIT388293.tmp oJaRh5CUzIaOjg== aGlzejw/PyR+Zmg= 542</span></p>
<p>The extraction of resources is one of two main roles of the dropper; it does so in the <span style="font-family: courier new, courier, monospace;">%WINDOWS%\SYSTEM32</span> folder, decrypting the Loader and preserving the encrypted state of the Downloader that will be decrypted just before being injected into another process. It also drops the configuration file <span style="font-family: courier new, courier, monospace;">assocnet.in</span><span style="font-family: 'courier new', courier, monospace;">f</span> that will later be leveraged by the final payloads, namely the Downloader and the Module. Then it chooses a service by checking the following list of three legitimate service names <span style="font-family: courier new, courier, monospace;">Winmgmt;ProfSvc;wmiApSrv;</span> and injects the Downloader into the matched service using reflective DLL injection.</p>
<p>The file name of the Loader is stored in the following Windows registry value:</p>
<p><span style="font-family: courier new, courier, monospace;">HKLM\SYSTEM\CurrentControlSet\Control\Lsa\Security Packages</span></p>

<h3>Loader</h3>
<p>This component is a Themida-protected file. We estimate the version of Themida to be 2.0-2.5, which agrees with KrCERT&rsquo;s report (page 20). The Loader serves as a simple injector that is looking for its injection parameters in the resources: the name of the encrypted file and the decryption key, which is the string &ldquo;542&rdquo;. The instance delivered by the dropper looks for the file <span style="font-family: courier new, courier, monospace;">bcyp655.tlb</span> (the Downloader). It creates a mutex <span style="font-family: courier new, courier, monospace;">Global\RRfreshRA_Mutex_Object</span>. The choice of the targeted service and the injection method are the same as in the dropper.</p>
<p>Let us talk for a while about the encryption method used by the dropper and by this loader. The common key is the string &ldquo;542&rdquo;, which is initially provided as a command-line parameter to the Dropper and subsequently as a 3-byte encrypted resource for the Loader. To expand a short master key to a larger expanded key (so-called key scheduling), the MD5 hash of the string is computed, which is <span style="font-family: courier new, courier, monospace;">7DCD340D84F762EBA80AA538B0C527F7</span>. Then it takes first three double words, let's denote them A := 0x7DCD340D, B := 0x84f762EB, C:= 0xA80aa538. The length of an encrypted buffer is divided by 3, and this is the number of iterations that transforms the initial sequence (A,B,C) into the proper key. In every iteration (X,Y,Z) becomes (X^Y, Y^Z, X^Y^Z). Because the XOR operation (denoted ^) is commutative and transitive, and its square is zero, which leaves everything unchanged, we can compute that after 8 iterations we get the identity, so the key could reach just 7 pairwise different states and is equal to the first 12 characters of the MD5 hash of "542" if the length is a multiple of 24.</p>
<p>What is interesting is how the remainder of the length division by 3 is treated. If the number of iterations was increased by this remainder, then we would reach just another of the 7 states of the key. However, the twist is in the change of operation: ^ is replaced with the OR operation in the code for the remainder. For example, the key with the remainder 1 becomes {FE F7 3A F9 F7 D7 FF FD FF F7 FF FD} for one of the states (of (C, A^B, B^C) to be precise), so we get new possible transformations of the key that tend to be more likely to be ones than zeroes.</p>
<p>That was the part preparing the key. The encryption algorithm itself looks like <a href="https://en.wikipedia.org/wiki/A5/1" target="_blank" rel="noopener">A5/1</a> at first glance. It was a secret technology developed in 1987 and used in over-the-air communication privacy in the GSM cellular telephone standard until reverse-engineered in 1999. The crucial part of the algorithm is <em>three</em> linear feedback shift registers (LFSRs). However, only the lengths of LFSRs in the malware code coincide with the official implementation, not the constants.</p>

<h2>Conclusion</h2>
<p>Attackers are constantly trying to find new ways to deliver malware to target computers. Attackers are particularly interested in supply-chain attacks, because they allow them to covertly deploy malware on many computers at the same time. In recent years ESET researchers analyzed such cases as <a href="https://www.welivesecurity.com/2017/07/04/analysis-of-telebots-cunning-backdoor/" target="_blank" rel="noopener">M.E.Doc</a>, <a href="https://www.welivesecurity.com/2017/10/20/osx-proton-supply-chain-attack-elmedia/" target="_blank" rel="noopener">Elmedia Player</a>, <a href="https://www.welivesecurity.com/2018/10/18/new-linux-chachaddos-malware-distributed-servers-vestacp-installed/" target="_blank" rel="noopener">VestaCP</a>, <a href="https://www.welivesecurity.com/2018/11/06/supply-chain-attack-cryptocurrency-exchange-gate-io/" target="_blank" rel="noopener">Statcounter</a>, and the <a href="https://www.welivesecurity.com/2019/03/11/gaming-industry-scope-attackers-asia/" target="_blank" rel="noopener">gaming industry</a>. We can safely predict that the number of supply-chain attacks will increase in the future, especially against companies whose services are popular in specific regions or in specific industry verticals.</p>
<p>This time we analyzed how the Lazarus group used a very interesting approach to target South Korean users of WIZVERA VeraPort software. As mentioned in our analysis, it&rsquo;s the combination of compromised websites with WIZVERA VeraPort support and specific VeraPort configuration options that allow attackers to perform this attack. Owners of such websites could decrease the possibility of such attacks, even if their sites are compromised, by enabling specific options (e.g. by specifying hashes of binaries in the VeraPort configuration).</p>
