            <h1>Hijacking DLLs in Windows</h1>
            <p>DLL Hijacking is a popular technique for executing malicious payloads. This post lists nearly 300 executables vulnerable to relative path DLL Hijacking on Windows 10 (1909), and shows how with a few lines of VBScript some of the DLL hijacks can be executed with elevated privileges, bypassing UAC.</p>
            <hr>
        </header>
        <article class="container content">
            <h2 id="dll-hijacking">DLL Hijacking</h2>

<p>First of all, let’s get the definition out of the way. DLL hijacking is, in the broadest sense, tricking a legitimate/trusted application into loading an arbitrary DLL. Terms such as <em>DLL Search Order Hijacking</em>, <em>DLL Load Order Hijacking</em>, <em>DLL Spoofing</em>, <em>DLL Injection</em> and <em>DLL Side-Loading</em> are often -mistakenly- used to say the same. At best such terms describe specific cases of DLL hijacking, but are often used interchangeably and therefore incorrectly. As an umbrella term, DLL hijacking is more accurate, as DLL hijacking always involves a DLL taking over from a legitimate DLL.</p>

<p>Attackers have been seen to use DLL hijacking in different ways and for different reasons. Motives include <strong>execution</strong> (executing malicious code through a trusted executable may be less likely to set off alarm bells, and in some cases even bypasses application whitelist features such as AppLocker [<a href="https://docs.microsoft.com/en-us/windows/security/threat-protection/windows-defender-application-control/applocker/applocker-overview">1</a>]), obtaining <strong>persistence</strong> (if the target application is pre-installed and runs regularly, so will the malicious code) and <strong>privilege escalation</strong> (if the target application runs under elevated permissions, so will the malicious code).</p>

<p>There is a variety of approaches to choose from, with success depending on how the application is configured to load its required DLLs. Possible approaches include:</p>

<ol>
  <li><strong>DLL replacement</strong>: replace a legitimate DLL with an evil DLL. This can be combined with <em>DLL Proxying</em> [<a href="https://kevinalmansa.github.io/application%20security/DLL-Proxying/">2</a>], which ensures all functionality of the original DLL remains intact.</li>
  <li><strong>DLL search order hijacking</strong>: DLLs specified by an application without a path are searched for in fixed locations in a specific order [<a href="https://docs.microsoft.com/en-us/windows/win32/dlls/dynamic-link-library-search-order">3</a>]. Hijacking the search order takes place by putting the evil DLL in a location that is searched in before the actual DLL. This sometimes includes the working directory of the target application.</li>
  <li><strong>Phantom DLL hijacking</strong>: drop an evil DLL in place of a missing/non-existing DLL that a legitimate application tries to load [<a href="http://www.hexacorn.com/blog/2013/12/08/beyond-good-ol-run-key-part-5/">4</a>].</li>
  <li><strong>DLL redirection</strong>: change the location in which the DLL is searched for, e.g. by editing the <code class="language-plaintext highlighter-rouge">%PATH%</code> environment variable, or <code class="language-plaintext highlighter-rouge">.exe.manifest</code> / <code class="language-plaintext highlighter-rouge">.exe.local</code> files to include the folder containing the evil DLL [<a href="https://docs.microsoft.com/en-gb/windows/win32/sbscs/application-manifests">5</a>, <a href="https://docs.microsoft.com/en-gb/windows/win32/dlls/dynamic-link-library-redirection">6</a>] .</li>
  <li><strong>WinSxS DLL replacement</strong>: replace the legitimate DLL with the evil DLL in the relevant WinSxS folder of the targeted DLL. Often referred to as DLL side-loading [<a href="https://www.fireeye.com/content/dam/fireeye-www/global/en/current-threats/pdfs/rpt-dll-sideloading.pdf">7</a>].</li>
  <li><strong>Relative path DLL Hijacking:</strong> copy (and optionally rename) the legitimate application to a user-writeable folder, alongside the evil DLL. In the way this is used, it has similarities with (Signed) Binary Proxy Execution [<a href="https://attack.mitre.org/techniques/T1218/">8</a>]. A variation of this is (somewhat oxymoronically called) ‘<em>bring your own LOLbin</em>’ [<a href="https://www.microsoft.com/security/blog/2019/09/26/bring-your-own-lolbin-multi-stage-fileless-nodersok-campaign-delivers-rare-node-js-based-malware/">9</a>] in which the legitimate application is brought with the evil DLL (rather than copied from the legitimate location on the victim’s machine).</li>
</ol>

<h2 id="finding-vulnerable-executables">Finding vulnerable executables</h2>

<p>The biggest challenge is to find a vulnerable executable that can be exploited under default user permissions. When targeting pre-installed system executables on Windows, that typically excludes the first option, whilst any folders eligible in options 2 and 3 have to be user writeable, as should the the files and folder in options 4 and 5. This is usually not the case.</p>

<p>That leaves us with option six, the weakest variant, which the remainder of this post will focus on. Although usually unsuitable to obtain persistence or privilege escalation, it is often seen in the wild. Take OceanLotus/APT32, who at the end of 2019 have been observed to use a legitimate <code class="language-plaintext highlighter-rouge">rekeywiz.exe</code> alongside a malicious <code class="language-plaintext highlighter-rouge">duser.dll</code> [<a href="https://twitter.com/0xCARNAGE/status/1203882560176218113">10</a>, <a href="https://any.run/report/f08ccc040c8d8db60f30a6d1026aa6523e97c6cf52b1b30f083a830a0a65a3a9/3abfc241-3ab0-4016-acbb-040b44199d52">11</a>]. In this case, the malware embedded the legitimate software and dropped it to disk, adopting the ‘<em>bring your own LOLbin</em>’ approach (another way of achieving the same would have been to copy the legitimate executable from the <code class="language-plaintext highlighter-rouge">\system32\</code> folder, assuming the executable hasn’t been patched yet).</p>

<p>To prevent new versions of this technique to be successful, it is worthwhile identifying executables that are vulnerable to this kind of DLL hijacking. This will provide red teamers with new means for execution, but more importantly, it will allow threat hunters and defenders to take appropriate measures to detect and prevent.</p>

<h2 id="approach">Approach</h2>

<p>To keep things focussed, let’s limit ourselves to the executables present by default in <code class="language-plaintext highlighter-rouge">c:\windows\system32\</code>. On the tested Windows 10 v1909 instance, this comprised a total of 616 executables, or 613 if you only consider signed applications.</p>

<p>To monitor which DLLs each process attempts to load, we’ll use the well-known Procmon [<a href="https://docs.microsoft.com/en-us/sysinternals/downloads/procmon">12</a>] tool. The approach taken is therefore: (1) copy trusted executable to a user-writable location; (2) run copied executable; (3) use Procmon to identify DLLs looked for in user writable location.</p>

<p><a href="/assets/2020-06-22-procmon.png"><img src="/assets/2020-06-22-procmon.png" alt="Screenshot of Procmon capturing DLL queries by winsat.exe." /></a>
<em>Procmon capturing DLL queries by a copy of winsat.exe, located in <code class="language-plaintext highlighter-rouge">c:\users\wietze\downloads\</code>.</em></p>

<p>This allows us to identify all DLLs queried by each application, which will be all <em>potential</em> hijackable DLL candidates. But it does not automatically follow that all of these are also loaded (and therefore executed). The most reliable way to find out which DLLs are properly loaded, is to compile our own version of the DLL, and make it write to a unique file upon successfully loading. If we then repeat the above approach for all target executables and DLLs, it will result in a collection of files that tells us which DLLs are <em>confirmed</em> vulnerable to DLL hijacking.</p>

<p>Compiling custom versions of existing DLLs is more challenging than it may sound, as a lot of executables will not load such DLLs if procedures or entry points are missing. Tools such as DLL Export Viewer [<a href="https://www.nirsoft.net/utils/dll_export_viewer.html">13</a>] can be used to enumerate all external function names and ordinals of the legitimate DLLs. Ensuring that our compiled DLL follows the same format will maximise the chances of it being loaded successfully.</p>

<em>Sample C code for our own version of dxgi.dll, which showed up in the Procmon recording of winsat.exe.</em></p>

<h2 id="combining-with-uac-bypass">Combining with UAC bypass</h2>

<p>Having found all these executables, at most this allows us to execute code through trusted programs. However, it is also possible to gain elevated rights if used in conjunction with UAC Bypass techniques.</p>

<p><em>User Account Control</em> (UAC) [<a href="https://docs.microsoft.com/en-us/windows/security/identity-protection/user-account-control/how-user-account-control-works">16</a>] was introduced in Windows Vista as a security feature, asking users for confirmation through a prompt before a process running under normal privileges is elevated to higher privileges. After users complained about getting flooded with UAC prompts when doing arbitrary tasks, Microsoft introduced <em>auto elevation</em> in Windows 7, which automatically elevates certain processes if they are located in trusted directories (such as <code class="language-plaintext highlighter-rouge">c:\windows\system32</code> ).</p>

<p>With this in mind, you could try running arbitrary code with elevated privileges by using an executable that is marked for auto elevation that is also vulnerable to DLL hijacking. There are about 35 of such executables, as can be seen in the previous section. The problem to overcome is that of the trusted directory: both the auto-elevate executable and the custom DLL need to be located in a trusted directory, but none of theses are user writeable.</p>

<p>There is some excellent research about bypassing UAC out there - one of my favourite techniques is the mocking of trusted directories using trailing spaces [<a href="https://medium.com/tenable-techblog/uac-bypass-by-mocking-trusted-directories-24a96675f6e">17</a>]. I would recommend reading the full blog post, but it boils down to users being able to create <code class="language-plaintext highlighter-rouge">c:\windows \system32\</code> (note the space after the first folder), and auto-elevate executables placed in this folder consider this a trusted location.</p>

<p>It is debatable whether this is a proper security vulnerability - Microsoft argue it is not [<a href="https://msdn.microsoft.com/en-us/library/cc751383.aspx">18</a>], but it is at least a flaw, given that most (non-enterprise) Windows computers are using ‘administrator accounts’ by default.</p>

<p>Either way, this provides us with an excellent means through which DLL hijacking can be made much more powerful. Note that folders with trailing spaces cannot be created through traditional means on Windows. You could compile some lines of C to do this, as is done by the original researcher, but it turns out VBScript can actually do this for us too. The following proof-of-concept shows that with only a few lines of code you can get this to work:</p>

<h2 id="prevention-and-detection">Prevention and detection</h2>

<p>A simple way to prevent DLL hijacking from happening would be for applications to always use absolute paths instead of relative ones. Although some applications (notably portable ones) will not always be able to do so, applications located in <code class="language-plaintext highlighter-rouge">\system32\</code> and relying on DLLs in the same folder have no excuse for doing otherwise. The better option, which only very few Windows executables seem to do, is to verify all DLLs before loading them (e.g. by checking their signatures) - this would largely eliminate the problem.</p>

<p>Nevertheless, as we have seen, attackers will still be able to bring older versions of legitimate/trusted applications that can be exploited. So even if every application starts checking their DLLs before loading them from now on, we would still have to deal with this problem.</p>

<p>Let’s therefore focus on detection. You could hunt for the creation or loading of any of the DLLs mentioned before from unexpected paths, particularly in temp locations such as <code class="language-plaintext highlighter-rouge">%appdata%</code>. After all, the name of the (legitimate) application loading the DLLs can be changed, but the filenames of DLLs are always fixed. A sample Sigma rule for this can be found here [<a href="https://github.com/wietze/windows-dll-hijacking/blob/master/possible_windows_dll_hijacking.yml">19</a>] - it successfully detects our DLL hijacking, although as you can see, it doesn’t scale very well and is likely to be prone to false positives. You could take a more generic approach by looking for the presence of Microsoft-signed binaries in unexpected locations, of the loading of DLLs from unexpected locations by such Microsoft-signed binaries (regardless of location).</p>

<p>Finally, the demonstrated UAC bypass technique can be detected easily and reliably by looking for any activity in the <code class="language-plaintext highlighter-rouge">/windows /</code> folder, or in any folders ending in a space for that matter. As described before, Windows folders with trailing spaces cannot be created through normal means and should therefore be rare, and always suspicious. Setting your UAC mode to ‘Always notify’, one level higher than the default, will prevent this and other similar UAC bypass techniques from succeeding.</p>
